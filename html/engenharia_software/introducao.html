Engenharia de Software

A disciplina que transforma a programação em um processo sistemático, confiável e escalável

Engenharia de Software é a aplicação sistemática de princípios de engenharia ao desenvolvimento de software. Vai além de "escrever código": abrange elicitação de requisitos, design de arquitetura, testes, manutenção e gerenciamento de projetos. Surgiu formalmente na conferência NATO de 1968 em resposta à "crise do software" — projetos atrasados, acima do orçamento e cheios de defeitos.

Engenharia de Software vs Programação

Programação é uma das atividades da engenharia de software, mas as duas não são sinônimos. A diferença está no escopo, no produto esperado e nas disciplinas envolvidas.

Aspecto Programação Engenharia de Software
Escopo Implementação de soluções Processo completo: requisitos → entrega → manutenção
Foco Algoritmos e código Qualidade, processo, equipe, custo e prazo
Escala Código funcional individualmente Sistemas grandes com múltiplos desenvolvedores por anos
Produto Programa executável Sistema com documentação, testes e processo definido
Disciplinas Lógica e estruturas de dados Requisitos, design, testes, métricas, gestão de projetos

Principais Áreas

A engenharia de software engloba diversas disciplinas complementares. Projetos bem-sucedidos costumam cobrir todas elas — mesmo que com profundidades diferentes dependendo do contexto.

Requisitos

Elicitação, análise, especificação e validação do que o sistema deve fazer. O objetivo é evitar construir o produto errado — a fonte mais comum de falha em projetos.

Design e Arquitetura

Estrutura do sistema, decisões de alto nível, padrões de projeto, princípios SOLID e Clean Architecture. Definir “como” antes de “quanto” reduz retrabalho futuro.

Implementação

Boas práticas de código, Clean Code, revisão de código, refatoração e controle de versão. Código é lido muito mais do que escrito — legibilidade é produtividade.

Testes e Qualidade

Pirâmide de testes (unitário, integração, sistema), TDD, BDD, métricas de qualidade. Qualidade não é uma fase — é uma prática contínua ao longo de todo o ciclo.

Metodologias

Como o trabalho é organizado: Scrum, Kanban, XP, SAFe, Cascata, Espiral. A metodologia certa depende do contexto, do time e do tipo de projeto.

Manutenção e Evolução

Correção de bugs, refatoração, gestão de débito técnico, controle de dependências. A maioria dos sistemas passa mais tempo em manutenção do que em desenvolvimento inicial.

Crise do Software

A conferência NATO de 1968 formalizou o termo "crise do software" para descrever um padrão sistemático de fracasso em projetos. Os mesmos problemas continuam aparecendo hoje quando os princípios da engenharia são ignorados.

Problema Consequência Resposta da Área
Sistemas entregues com defeitos graves Danos financeiros e reputacionais Testes formais, QA, TDD
Projetos atrasados e acima do orçamento Clientes insatisfeitos, empresas falidas Métricas, estimativas, Agile
Código impossível de manter Reescrita total após poucos anos Clean Code, refatoração, arquitetura
Requisitos mal compreendidos Sistema que não atende o usuário Engenharia de requisitos, User Stories
Falta de documentação Perda de conhecimento, onboarding lento Documentação técnica, wikis, ADRs

Modelo de Custos do Software

O custo de corrigir um problema cresce exponencialmente conforme ele avança no ciclo de vida. Esse é o principal argumento para investir em requisitos, design e testes desde o início.

Distribuição de Custos no Ciclo de Vida
Distribuição típica de custos em um ciclo de vida:

Requirements & Design  ████████░░░░░░░░░  ~25%
Implementation         █████████████░░░░  ~35%
Testing & QA           █████████░░░░░░░░  ~25%
Maintenance            ███████████████░░  ~15% por ano acumulado

Custo de corrigir um bug por fase:
  Requisitos    : 1x
  Design        : 5x
  Implementação : 10x
  Testes        : 25x
  Produção      : 100x

→ Investir em qualidade cedo é sempre mais barato.
