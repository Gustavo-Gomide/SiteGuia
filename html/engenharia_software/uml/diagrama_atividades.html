Diagrama de Atividades

Fluxo de trabalho com decisões, paralelismo e responsabilidades separadas por swimlanes.

O diagrama de atividades responde "qual o passo a passo e onde pode dar errado?" — perfeito para alinhar regras de negócio antes de qualquer implementação.

Elementos do diagrama

O conjunto mínimo de elementos cobre a maioria dos fluxos reais de negócio e sistema.

Elemento Símbolo textual Propósito Quando usar
Início ● (círculo cheio) Ponto de entrada do fluxo Todo diagrama tem exatamente um
Ação [ Ação ] Passo executável do processo Cada etapa com responsável claro
Decisão ◇ com guardas [condição] Bifurcação do fluxo (if/else) Validações e regras condicionais
Merge ◇ unindo setas Reune caminhos divergentes Após um bloco de decisão
Fork ━━━ (barra grossa horizontal) Inicia atividades paralelas Etapas que podem ocorrer simultaneamente
Join ━━━ (barra grossa horizontal) Sincroniza atividades paralelas Aguardar todas as paralelas terminarem
Swimlane Coluna com nome do ator Separa responsabilidades visualmente Múltiplos atores no mesmo fluxo
Fim ⊙ (círculo com ponto) Término do fluxo Pode haver múltiplos fins (sucesso/erro)

Exemplo: processo de checkout

Um fluxo com swimlanes torna explícito quem é responsável por cada etapa — e onde estão os pontos de falha.

Swimlanes: | Cliente | Sistema | Gateway Pagto |

● Início

[Cliente] Adicionar itens ao carrinho
[Cliente] Confirmar endereço de entrega

◇ Endereço válido?
  [não] → [Cliente] Corrigir endereço → voltar
  [sim] → continua

[Sistema] Calcular frete e total
[Cliente] Selecionar forma de pagamento

━━━ Fork ━━━
  [Sistema] Reservar estoque
  [Gateway] Processar pagamento
━━━ Join ━━━

◇ Pagamento aprovado?
  [não] → [Sistema] Liberar estoque reservado
           [Sistema] Notificar falha → ⊙ Fim (falha)
  [sim] → [Sistema] Criar pedido confirmado
            [Sistema] Enviar e-mail de confirmação
            → ⊙ Fim (sucesso)

Armadilhas comuns

O diagrama de atividades fica inútil quando tenta modelar código em vez de processo.

Modelar cliques de UI

"Clicar no botão", "preencher campo" são passos de interface, não de processo. Foque nas etapas de negócio.

Esquecer exceções

Modelar só o happy path omite onde o processo quebra. Adicione as exceções mais comuns após ter o fluxo principal.

Sem swimlanes em fluxos multi-ator

Sem separar responsabilidades visualmente, o diagrama vira "o sistema faz tudo" — perfeito para criar silos depois.

Fork sem Join

Paralelo sem ponto de sincronização cria ambiguidade: quando o fluxo continua? Todo fork precisa de um join correspondente.

Aprofunde-se em: Diagrama de Sequência para detalhar chamadas técnicas, e Diagrama de Estados para ciclo de vida de entidades.

"Swimlanes tornam visível o que reuniões tentam resolver em horas: quem é responsável por quê."
