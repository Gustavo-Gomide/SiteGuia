Diagrama de Casos de Uso

Atores, objetivos e fronteira do sistema — o mapa de "o que o sistema faz e para quem".

Casos de uso não modelam telas nem código: modelam objetivos. É o primeiro diagrama a criar em um projeto — antes do backlog, antes do banco, antes do endpoint.

Elementos do diagrama

Quatro elementos constroem qualquer diagrama de casos de uso — simplicidade intencional para focar no negócio.

Elemento Notação Descrição Exemplo
Ator Boneco palito Papel externo que interage com o sistema Cliente, Admin, Gateway de Pagamento
Caso de uso Elipse com nome Objetivo que o sistema satisfaz (verbo + substantivo) "Comprar produto", "Emitir relatório"
Fronteira Retângulo ao redor dos casos Separa o que é sistema do que é externo Sistema: "Loja Online"
<<include>> Seta tracejada com rótulo Caso B sempre é executado como parte de A "Comprar" inclui "Autenticar"
<<extend>> Seta tracejada com rótulo Caso B pode estender A opcionalmente "Aplicar cupom" estende "Comprar"

Exemplo: Loja Online

Um diagrama textual já orienta backlog, endpoints e fronteiras de responsabilidade sem entrar em detalhes de implementação.

Sistema: Loja Online
Fronteira:
  [Comprar produto]
  [Acompanhar pedido]
  [Cadastrar produto]
  [Gerenciar estoque]
  [Processar pagamento]
  [Cancelar pedido]

Atores:
  Cliente       -- Comprar produto, Acompanhar pedido, Cancelar pedido
  Administrador -- Cadastrar produto, Gerenciar estoque
  Gateway Pagto -- Processar pagamento (sistema externo, ator secundário)

Relacionamentos:
  "Comprar produto"   <<include>>  "Autenticar usuário"
  "Comprar produto"   <<include>>  "Processar pagamento"
  "Aplicar cupom"     <<extend>>   "Comprar produto"   [condição: cupom informado]

Pré/pós condições do caso "Comprar produto":
  Pré:  cliente autenticado, produto disponível em estoque
  Pós:  pedido criado, pagamento iniciado, estoque reservado
  Exceção: pagamento recusado → pedido cancelado, estoque liberado

Armadilhas comuns

O diagrama de casos de uso é simples demais para fazer errado — e justamente por isso é tão mal-usado.

Casos de uso como telas

"Exibir lista", "Renderizar formulário" não são casos de uso — são passos de UI. Caso de uso é objetivo: "Comprar produto".

Esquecer atores externos

Sistemas de pagamento, e-mail, autenticação OAuth e APIs externas são atores secundários. Ignorá-los esconde dependências críticas.

Incluir detalhes técnicos

Passos de implementação ("chamar endpoint X", "inserir no banco") pertencem ao diagrama de sequência, não aos casos de uso.

Sem pré/pós condições

Um caso de uso sem condições é incompleto. Elas definem quando o caso é válido e o que garante após a execução.

"Um caso de uso sem pré e pós condições é um título sem contrato."
