Diagrama de Classes

Estrutura estática: entidades, atributos, métodos e relacionamentos do sistema.

O diagrama de classes transforma discussões vagas sobre domínio em contratos explícitos — antes de qualquer banco de dados, endpoint ou código.

Tipos de relacionamento

Escolher o relacionamento certo evita modelos que parecem certos no papel mas geram acoplamento errado no código.

Tipo Notação textual Significado Exemplo
Associação A — B (com multiplicidade) A conhece B; relação sem controle de ciclo de vida Cliente — Pedido (1..*)
Composição A ♦— B (diamante preenchido) "Parte de" forte: B morre quando A morre Pedido ♦— ItemPedido
Agregação A ◇— B (diamante vazio) "Parte de" fraca: B existe sem A Curso ◇— Aluno
Herança B ——▷ A (seta fechada) B é um tipo de A (generalização) PessoaFísica ——▷ Pessoa
Implementação C ---▷ I (seta fechada tracejada) C cumpre o contrato da interface I EmailService ---▷ Notificador
Dependência A ---> B (seta aberta tracejada) A usa B temporariamente (parâmetro, retorno) Controlador ---> DTO

Exemplo: domínio de e-commerce

Um modelo mínimo já orienta tabelas, validações e endpoints — sem precisar de código ainda.

// Domínio: Loja Online — classes e relacionamentos

Cliente (1) ————————— (*) Pedido
  - id: Long              - id: Long
  - nome: String          - dataCriacao: DateTime
  - email: String         - status: StatusPedido
                          - calcularTotal(): Decimal

Pedido (1) ♦——————— (*) ItemPedido
                          - quantidade: Int
                          - precoUnitario: Decimal   ← congelado na compra
                          - subtotal(): Decimal

ItemPedido (*) ————— (1) Produto
                          - id: Long
                          - nome: String
                          - preco: Decimal
                          - estoque: Int

// Interface
Pedido usa ——▷ Notificador (interface)
  + enviarConfirmacao(pedido: Pedido): void
EmailNotificador implementa Notificador

// Invariantes (documentar no diagrama!)
- Pedido.status: [CRIADO, PAGO, ENVIADO, ENTREGUE, CANCELADO]
- ItemPedido.quantidade >= 1
- Pedido deve ter ao menos 1 ItemPedido

Armadilhas comuns

O diagrama de classes é um dos mais mal-usados: ou detalhado demais, ou genérico demais para ter valor.

Confundir com schema SQL

Classes modelam comportamento e estrutura do domínio. Tabelas são detalhes de persistência — um não é cópia do outro.

Métodos demais

Listar todos os getters e setters transforma o diagrama num índice da IDE. Mostre só os métodos que expressam regra de negócio.

Esquecer multiplicidade

Uma linha sem multiplicidade (1, 0..1, 1..*, 0..*) é ambígua. Esse detalhe elimina discussões sobre "possível ter zero?"

Sem invariantes

O diagrama fica "bonito" mas inútil se não registrar as regras (ex.: pedido cancelado não pode ser enviado).

Explore mais: Diagrama de Objetos para validar com instâncias reais, e Diagrama de Sequência para as interações.

"Um diagrama de classes sem invariantes é uma planta baixa sem as medidas."
