Diagrama de Sequência

Interações entre objetos ao longo do tempo: chamadas, retornos e fluxos condicionais.

Quando alguém não entende um fluxo de API ou uma orquestração entre serviços, um diagrama de sequência comunica em minutos o que parágrafos de texto não conseguem.

Elementos essenciais

Poucos elementos compõem o diagrama; a maioria dos fluxos reais usa apenas os quatro primeiros.

Elemento Notação Propósito Exemplo
Lifeline Caixa + linha vertical tracejada Participante da interação :Controller, :Service, :DB
Mensagem síncrona → (seta fechada) Chamada que aguarda retorno authenticate(email, senha)
Mensagem de retorno - - → (seta tracejada) Resposta a uma chamada anterior return token | return erro
Mensagem assíncrona → (seta aberta) Disparo sem aguardar resposta publicarEvento(OrderPlaced)
Bloco alt [alt] com guardas Fluxo condicional (if/else) [credenciais válidas] / [inválidas]
Bloco loop [loop] Repetição com condição de parada [loop] tentar pagamento (max 3x)

Exemplo: fluxo de autenticação

Um caso real com fluxo principal e caminho de erro: o nível certo de detalhe para guiar código, logs e métricas.

Usuário -> UI: submete email + senha
UI -> AuthController: POST /auth/login {email, senha}
AuthController -> AuthService: authenticate(email, senha)
AuthService -> UserRepo: findByEmail(email)
UserRepo -> DB: SELECT * FROM users WHERE email = ?
DB --> UserRepo: linha do usuário (ou vazio)

alt usuário não encontrado
  AuthService --> AuthController: UserNotFoundException
  AuthController --> UI: 401 Unauthorized
  UI --> Usuário: "Credenciais inválidas"
else usuário encontrado
  AuthService -> PasswordHasher: verify(senhaPlain, hash)
  PasswordHasher --> AuthService: ok | falhou

  alt senha incorreta
    AuthService --> AuthController: InvalidCredentialsException
    AuthController --> UI: 401 Unauthorized
  else senha correta
    AuthService -> TokenService: gerarJWT(userId, roles)
    TokenService --> AuthService: token (exp 1h)
    AuthService --> AuthController: token
    AuthController --> UI: 200 OK {token}
    UI --> Usuário: redireciona para /dashboard
  end
end

Armadilhas comuns

O diagrama de sequência é fácil de criar errado: ou mostra o óbvio ou tenta modelar o sistema inteiro em um só diagrama.

Um diagrama por cenário

Misturar sucesso e todos os erros em um único diagrama gera ilegibilidade. Crie um para cada fluxo relevante.

Modelar o framework

Mostrar middleware, ORM e lifecycle de framework não traz valor. Foque nas mensagens do domínio.

Mensagens sem nome

Setas sem rótulo ("chama", "retorna") são inúteis. Nomeie com a intenção: authenticate, createOrder.

Não mostrar erros

Modelar só o happy path omite exatamente o que o time precisa discutir: como tratar falhas e timeouts.

Veja também: Diagrama de Atividades para fluxos de negócio, e Diagrama de Classes para a estrutura.

"Nomear as mensagens com intenção revela responsabilidades que o código ainda não tem."
