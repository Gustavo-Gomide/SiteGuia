Diagrama de Componentes

Módulos, interfaces e dependências — a arquitetura de alto nível que guia limites de responsabilidade.

O diagrama de componentes revela quem depende de quem. Se ninguém vê as dependências, ninguém as controla — e o acoplamento cresce silencioso até travar a manutenção.

Componentes e suas relações

Cada componente expõe interfaces e consome outras — a direção das dependências é a decisão arquitetural central.

Componente Expõe (interface) Consome Restrição arquitetural
Web/API Endpoints HTTP Casos de Uso (Aplicação) Só conhece DTOs e Use Cases
Aplicação IUseCaseX, IUseCaseY Interfaces do Domínio Orquestra sem regra de negócio própria
Domínio IRepositório, IServiço Nada externo Zero dependências de infra ou framework
Infra/Persistência Implementa IRepositório ORM, banco, storage Aponta para dentro (inversão de dependência)
Infra/Mensageria Implementa IEventBus Broker (Kafka, RabbitMQ) Desacoplado do Domínio via interface

Exemplo: arquitetura hexagonal

A inversão de dependência é o princípio fundamental — toda seta aponta para dentro, em direção ao domínio. 

// Arquitetura Hexagonal / Clean Architecture
// Regra: dependências apontam para o centro

[Web/API] ──depende──→ [Aplicação]
[CLI / Workers] ──depende──→ [Aplicação]
[Aplicação] ──depende──→ [Domínio]  ← núcleo puro
[Infra/DB] ──implementa──→ [IOrderRepo]   (porta do Domínio)
[Infra/Email] ──implementa──→ [INotifier]  (porta do Domínio)
[Infra/Pagto] ──implementa──→ [IPaymentGateway]

// Interfaces (portas) definidas NO Domínio:
interface IOrderRepo {
  save(order: Order): void
  findById(id: OrderId): Order | null
}
interface INotifier {
  sendConfirmation(order: Order): void
}

// Resultado:
// - Domínio testável sem banco, sem HTTP, sem broker
// - Trocar PostgreSQL por DynamoDB → só mudar Infra/DB
// - Adicionar CLI → só criar novo adaptador de entrada

Armadilhas comuns

A maioria dos sistemas começa limpo e vai acumulando dependências erradas silenciosamente até o diagrama não refletir mais a realidade.

Componentes genéricos

"Utils", "Common", "Shared" viram lixões. Se um módulo não tem nome preciso, é porque ainda não tem uma responsabilidade clara.

Domínio dependendo de Infra

Importar ORM, HTTP client ou banco diretamente no domínio destrói a testabilidade e engessa a arquitetura.

Diagrama ≠ código

Modelar dependências desejadas em vez das reais cria falsa segurança. Gere o diagrama do código — não escreva à mão.

Interfaces sem contrato

Interface vazia ou com método genérico não define nada. Um bom contrato tem nome de intenção: IOrderRepository, não IRepo.
