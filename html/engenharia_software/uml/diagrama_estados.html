Diagrama de Estados

Máquinas de estado: ciclo de vida, transições, guardas e ações de entidades do domínio.

O diagrama de estados é o mais eficiente para modelar entidades com regras de transição — Pedido, Assinatura, Ticket, Tarefa. Ele torna visível o que "entregue sem pago" e outros bugs de estado impossível revelam em produção.

Elementos e transições

Cada transição tem um evento que a dispara, uma guarda opcional e uma ação que executa — esse triplo é o que entra no código.

Elemento Notação Descrição Exemplo (Pedido)
Estado inicial ● (ponto de entrada) Ponto onde o objeto nasce Pedido criado
Estado [ NomeEstado ] Condição estável do objeto AGUARDANDO_PAGAMENTO
Transição A ——evento——> B Mudança de estado ao receber evento CRIADO ——confirmar——> AGUARDANDO
Guarda [condição] Transição só ocorre se condição for verdadeira [pagamento aprovado]
Ação on entry entry / ação Executa ao entrar no estado entry / enviarEmailConfirmação()
Estado final ⊙ (círculo duplo) Estado terminal — objeto não muda mais ENTREGUE, CANCELADO

Exemplo: máquina de estados de Pedido

Escrever a máquina de estados antes de codificar revela validações, ações assíncronas e transições proibidas.

// Entidade: Pedido
// Estados terminais: ENTREGUE, CANCELADO

● → [CRIADO]
  entry / reservarEstoque(), notificarVendedor()

[CRIADO] ——confirmar——→ [AGUARDANDO_PAGAMENTO]
  [sem guarda; sempre possível ao criar]

[AGUARDANDO_PAGAMENTO] ——aprovarPagamento [aprovado]——→ [PAGO]
  entry / enviarConfirmacaoPagamento()

[AGUARDANDO_PAGAMENTO] ——reprovarPagamento——→ [CANCELADO]
  entry / liberarEstoque(), notificarFalha()

[AGUARDANDO_PAGAMENTO] ——timeout 30min——→ [CANCELADO]
  entry / liberarEstoque(), notificarTimeout()

[PAGO] ——postar——→ [ENVIADO]
  entry / gerarCodigoRastreio(), notificarCliente()

[ENVIADO] ——confirmarEntrega——→ [ENTREGUE] ⊙
  entry / registrarDataEntrega(), liberarPagamentoVendedor()

[ENVIADO] ——solicitarDevolucao [dentro prazo]——→ [EM_DEVOLUCAO]
[EM_DEVOLUCAO] ——devolucaoConfirmada——→ [CANCELADO] ⊙

// Transições PROIBIDAS (documentar explicitamente):
// CANCELADO → ENVIADO  ✗
// ENTREGUE → qualquer estado  ✗
// AGUARDANDO → ENTREGUE (pular estados)  ✗

Armadilhas comuns

Sem máquina de estados, as regras ficam espalhadas em ifs por todo o codebase — e ninguém sabe quais transições são realmente válidas.

Estados implícitos

Guardar status como boolean (pago: true) esconde a máquina de estados. Use enum explícito desde o início.

Não documentar transições proibidas

Listar só o que é permitido é insuficiente. Explicitar o que é proibido evita "alguém achou que era válido" semanas depois.

Esquecer ações de entry/exit

A transição não muda só o estado — dispara side effects (e-mails, filas, reservas). Modelar isso evita bugs de sincronização.

Guardas vazias

Uma transição sem guarda significa "sempre é possível". Se não é, adicione a guarda — ela vai virar validação no código.

Combine com: Diagrama de Atividades para fluxos de processo, e Diagrama de Classes para modelar a entidade.

"Toda entidade com status no banco é uma máquina de estados esperando para ser documentada."
