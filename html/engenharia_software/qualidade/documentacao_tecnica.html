Documentação Técnica

O que documentar, como manter vivo e o que evitar

Documentação técnica existe para uma razão: reduzir dependência de pessoas específicas e acelerar manutenção. A regra prática é simples — documente o que evita perguntas repetidas e previne incidentes.

O que documentar — artefatos de alto ROI

Não documente tudo — documente o que gera valor real. Documentação desatualizada é pior que não ter.

ArtefatoObjetivoQuando atualizar
READMEComo rodar, testar, buildar e debugarToda mudança no setup ou comportamento
ADRDecisões arquiteturais e trade-offsSempre que tomar decisão arquitetural relevante
RunbookComo agir em incidentes e operaçãoApós cada incidente novo
API ReferenceContratos, parâmetros, erros, exemplosToda mudança de interface
Diagrama de arquiteturaComponentes, fluxos e limites do sistemaMudanças de estrutura relevantes

ADR — modelo mínimo

Architecture Decision Records (ADRs) documentam o "porquê" das decisões, não apenas o "o quê". São curtos, versionados no repositório e não precisam ser perfeitos.

ADR-005: Usar PostgreSQL em vez de MongoDB

Contexto:
  Sistema precisa de joins complexos entre entidades
  e garantia de consistência transacional (ex: pagamentos).

Decisão:
  Usar PostgreSQL como banco principal.

Alternativas consideradas:
  MongoDB (flexível mas sem transações ACID reais)
  MySQL (boa opção mas menor suporte a JSON avancado)

Consequências positivas:
  - transações ACID completas
  - queries relacionais sem overhead de código

Consequências negativas:
  - schema fixo exige migrations
  - escalabilidade horizontal mais complexa

Como manter a documentação viva

Documentação que não é mantida no fluxo de trabalho inevitavelmente fica desatualizada.

Doc perto do código

Store no mesmo repositório. Review de PR inclui verificar se doc precisa ser atualizada. "Sem doc = não passou."

Seções pequenas e focadas

Doc gigante ninguém lê nem atualiza. Prefira arquivos pequenos com escopo claro — um runbook por serviço.

Atualizar após incidente

Cada incidente novo que não estava no runbook é uma oportunidade. Documente o diagnóstico e a solução.

Diagrama como código

Use Mermaid, PlantUML ou C4 no repositório. Diagramas em slides/Drive ficam desatualizados em semanas.
