Code Review

Detectar bugs cedo, alinhar padrões e reduzir riscos antes do merge

Code review não é auditoria nem burocracia — é o principal mecanismo de qualidade embutida no fluxo. Uma revisão bem feita detecta bugs, difunde conhecimento e melhora o design do sistema continuamente.

O que um review deve verificar

Um review focado em risco real é mais valioso que uma revisão exaustiva de estilo. Priorize pelos itens de maior impacto.

DimensãoO que verificarSinal de problema
CorreçãoLógica, edge cases, validações, concorrênciaCaso não coberto por teste
SegurançaValidação de entrada, permissões, segredos expostosInput não sanitizado, log com dado sensível
PerformanceQueries N+1, alocações desnecessárias, IO síncronoLoop com query, serialização excessiva
ManutenibilidadeNomes, responsabilidade única, duplicaçãoFunção com 3+ responsabilidades
ConsistênciaPadrões do projeto, dependências, arquiteturaAbordagem diferente sem justificativa

Checklist prático de PR

Quanto menor o PR, melhor o review. PR gigante vira "aprovado no escuro".

Autor — antes de abrir o PR:
  [ ] mudança tem escopo claro e pequeno?
  [ ] há testes para o comportamento novo/alterado?
  [ ] descrição explica "por quê", não só "o quê"?
  [ ] README/docs precisam ser atualizados?

Revisor — durante o review:
  [ ] entendi o objetivo antes de comentar detalhes?
  [ ] meus comentários indicam risco e sugerem alternativa?
  [ ] estilo/formatacao está automatizado (nao comento manual)?
  [ ] aprovei apenas o que realmente li e entendi?

PR pequeno (< 400 linhas):
  - revisao mais cuidadosa e rapida
  - menos contexto para carregar
  - merge mais seguro

Armadilhas comuns

A maioria dos problemas de review não está no processo — está nos incentivos e comportamentos do time.

Nitpicks manuais

Se é estilo, use lint/prettier/formatter automatizado. Comentários de estilo manual desperdiçam tempo e geram atrito sem valor.

Crítica pessoal

"Você fez errado" em vez de "essa abordagem tem risco X". Review é sobre o código, não sobre quem escreveu.

Aprovação sem leitura

LGTM sem contexto vira "merge sem responsabilidade". Se não deu para ler, peça para dividir o PR.

Comentário sem alternativa

"Isso está errado" sem sugerir o que fazer correto não ajuda. Explique o risco e proponha a alternativa.
