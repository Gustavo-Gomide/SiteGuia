Melhorar estrutura sem alterar comportamento

Refatoração

Pagar dívida técnica com segurança e intenção

Refatorar é mudar a estrutura interna do código — nomes, funções, módulos — sem alterar o comportamento externo observável. É a prática que mantém a base de código sustentável ao longo do tempo, e diferente de reescrita: acontece em pequenos passos contínuos.

Quando refatorar

Refatoração é mais eficaz como hábito do que como projeto isolado. Estes são os sinais que indicam o momento certo:

Situação Sinal de alerta ROI esperado
Antes de adicionar uma feature Área confusa, difícil de entender Alto — a feature fica mais fácil de implementar
Bugs recorrentes na mesma região Mesmo módulo aparece em vários tickets Alto — reduz reincidência e tempo de debug
Mudança simples virou "cirurgia" Uma linha exige tocar 10 arquivos Alto — reduz acoplamento e risco de regressão
Code review aponta confusão frequente Revisores pedem explicação do mesmo trecho Médio — melhora onboarding e velocidade do time
Código morto ou nunca alterado Nenhum deploy toca aquela área Baixo — delete em vez de refatorar

Refatorar com segurança

Sem uma rede de segurança, refatoração vira reescrita cega. O processo seguro segue quatro etapas:

# Safety net — antes de qualquer refatoração

1) Criar ou ajustar testes que cubram o comportamento atual
   └── Testes unitários + testes de integração críticos

2) Refatoração em passos pequenos e reversíveis
   └── Uma responsabilidade por vez; nunca mudar comportamento

3) Rodar todos os testes a cada passo
   └── Se falhar → reverter imediatamente, não "consertar na raça"

4) Commits pequenos e descritivos
   └── git commit -m "refactor: extrai função calcularDesconto"
   └── PRs menores são revisados melhor e integrados mais rápido

# Sinais de que você saiu do escopo de refatoração
  - Testou e funcionou diferente de antes  → você mudou comportamento
  - O PR cresceu para centenas de linhas   → quebre em partes menores
  - Você adicionou lógica nova no meio     → separe em PR dedicado

Técnicas comuns

As técnicas a seguir cobrem a maioria dos code smells encontrados em bases de código reais:

Técnica Problema que resolve Exemplo prático
Extrair função Função grande demais, duplicação Bloco de cálculo de imposto vira calcularImposto()
Renomear Nomes genéricos que exigem comentário x → quantidadeItens; handle() → processarPedido()
Separar responsabilidades Classe/módulo faz coisas demais (God Object) OrderService que valida + persiste + envia e-mail → 3 classes
Introduzir interface / abstração Alto acoplamento, dificulta testes Dependência concreta de MySQLRepo → injetar IRepository
Mover lógica de negócio Regras espalhadas em camadas erradas Validação no controller movida para a entidade de domínio

Armadilhas comuns

A maioria dos problemas com refatoração vem de escopo mal definido ou pressa para "melhorar tudo de uma vez":

Refatorar tudo de uma vez

PRs gigantes são impossíveis de revisar e arriscados de integrar. Prefira refatorações incrementais ligadas a uma entrega.

Trocar comportamento sem perceber

A linha entre refatorar e "melhorar" é tênue. Se a lógica mudou, você saiu da refatoração — e pode ter quebrado regras de negócio.

Refatorar sem objetivo

Refatoração estética (renomear por gosto pessoal, reorganizar pastas) sem ganho real gera ruído no histórico e atrito no time.

Sem testes como rede de segurança

Mudar código sem cobertura de testes é aposta. Se não há testes, escreva-os antes — ou você não está refatorando, está arriscando.

Refatoração contínua reduz o acúmulo de dívida técnica e é um pilar central em Extreme Programming (XP).

"Refatoração não é um projeto — é um hábito."
