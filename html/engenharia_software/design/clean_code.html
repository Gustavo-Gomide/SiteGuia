Clean Code

Práticas para código legível, previsível e fácil de mudar — Robert C. Martin

Clean Code não é código bonito. É código cujo comportamento você prevê ao ler, que é fácil de alterar com segurança, e que minimiza surpresas. A métrica real é o custo de mudança ao longo do tempo.

Princípios práticos

Cada categoria aborda um tipo diferente de problema de legibilidade e manutenibilidade:

Categoria Princípio Violação comum
Nomes Nomes revelam intenção; evite abreviações e variáveis genéricas. x, data, temp, calc(a, b, t)
Funções Funções pequenas, fazem uma coisa; nome descreve exatamente o que fazem. Função de 200 linhas com múltiplos efeitos colaterais
Comentários Código deve se auto-documentar; comentários explicam por que, não o quê. Comentário que décrit o que o código já diz claramente
Erros Falhe cedo, com mensagens úteis; prefira exceções a códigos de erro. Returns null em vez de lançar erro; erros silenciosos
Testes Testes sustentam refatoração; sem testes, nenhum código permanece limpo. Zero testes, ou testes frágeis que quebram com refactorings simples
Formatação Consistência de estilo reduz carga cognitiva; use linter/formatter automático. Mix de estilos, indentação inconsistente por desenvolvedor

Antes e depois — nomes e funções

A diferença prática entre código com intenção obscura e código auto-explicativo:

// ANTES — intenção obscura
function calc(a, b, t) {
  if (t === 1) return a - b;
  return a + b;
}

// DEPOIS — intenção explícita
function calcularTotalComDesconto(precoBase, desconto) {
  return precoBase - desconto;
}

function calcularTotalComFrete(precoBase, frete) {
  return precoBase + frete;
}

// ANTES — função com múltiplas responsabilidades
function processarPedido(pedido) {
  validarPedido(pedido);
  calcularDesconto(pedido);
  salvarNoBanco(pedido);
  enviarEmail(pedido.cliente);
  registrarLog(pedido);
}

// DEPOIS — responsabilidades separadas
function processarPedido(pedido) {
  const pedidoValidado = validarPedido(pedido);
  const pedidoComDesconto = aplicarDesconto(pedidoValidado);
  return pedidoComDesconto;
}

Checklist de code review

Use estas perguntas durante revisões de código para identificar problemas de legibilidade:

Nomes e intenção

Você entende o que faz cada variável/função sem ler a implementação?

Baixa surpresa

O código se comporta exatamente como o nome sugere? Sem efeitos colaterais inesperados?

Cobertura de borda

Casos de borda estão cobertos por testes ou validações explícitas?

Sem duplicação óbvia

Há lógica repetida que deveria ser extraída? Cada conhecimento tem um único lugar?

Clean Code é sustentado por SOLID e reforçado por DRY, KISS e YAGNI. Sem testes automatizados, nenhuma prática de clean code dura.

"Se você precisa de comentário para explicar o que o código faz, talvez o nome esteja errado."
