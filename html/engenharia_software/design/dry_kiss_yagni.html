DRY, KISS e YAGNI

Três princípios para resistir à complexidade desnecessária

DRY, KISS e YAGNI são heurísticas complementares para manter o código simples e focado. Juntos, eles ajudam a decidir quando abstrair, quando simplificar e quando não construir algo.

Os 3 princípios comparados

Cada princípio resolve um tipo diferente de excesso de complexidade:

Princípio Objetivo Violação clássica Exagero típico
DRY Evitar duplicação de conhecimento (regras de negócio). A mesma regra existe em 5 lugares; muda em 3 deles. Abstração genérica criada para evitar 2 linhas repetidas.
KISS Escolher a solução mais simples que resolve o problema real. Solução de 500 linhas para um problema de 20. Simplismo que ignora casos de borda reais.
YAGNI Não construir o que não tem caso de uso validado hoje. Sistema de plugins construído antes do 1º caso real. Resistir até a coisas que claramente serão necessárias logo.

DRY na prática — duplicação vs abstração prematura

DRY não significa zero linhas repetidas. Significa que cada conhecimento tem um lugar único:

// VIOLAÇÃO DRY: a regra de desconto existe em 3 lugares
function calcularDescontoVIP(preco) {
  return preco * 0.85; // 15% off
}

function exibirPrecoVIP(preco) {
  return preco * 0.85; // 15% off (duplicado)
}

function aplicarDescontoCarrinho(item) {
  item.preco = item.preco * 0.85; // 15% off (duplicado)
}

// CORRETO: a regra tem um lugar
const DESCONTO_VIP = 0.15;

function aplicarDescontoVIP(preco) {
  return preco * (1 - DESCONTO_VIP);
}

// Todos os outros pontos chamam aplicarDescontoVIP()

// EXAGERO DRY: cada trecho de 2 linhas similar vira abstraction
const createGenericEntity = (config) => new Entity(config.name, config.fields);
// vs. apenas escrever new ProductEntity() e new UserEntity() diretamente

Quando aplicar cada princípio

Use estas perguntas para decidir entre abstrair, simplificar ou adiar:

A duplicação é real?

DRY só se aplica a duplicação de conhecimento. Código similar não é o mesmo que regra duplicada.

A solução tem o tamanho certo?

KISS: se outra pessoa não entende em 5 minutos sem ajuda, provavelmente está complexo demais.

Existe caso real hoje?

YAGNI: construa quando houver usuário/caso concreto. Previsões de funcionalidade geram 80% de desperdício.

A abstração tem um nome óbvio?

Se você não consegue nomear a abstração claramente, provavelmente ela está cedo demais (YAGNI) ou errada (KISS).

DRY, KISS e YAGNI são complementares a SOLID e Clean Code. Juntos, eles criam código que evolui sem acumular dívida técnica.

"Complexidade é o maior inimigo da manutenibilidade. Cada abstração precisa se pagar."
