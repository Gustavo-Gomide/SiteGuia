Clean Architecture

Arquitetura de sistemas com domínio isolado — Robert C. Martin

Clean Architecture organiza o sistema em camadas concêntricas onde as dependências apontam sempre para dentro — detalhes (frameworks, banco, UI) dependem do domínio, nunca o contrário. O objetivo é poder substituir banco, framework ou interface sem reescrever regras de negócio.

As camadas concêntricas

Cada camada só pode depender das camadas mais internas. A regra da dependência é inviolável:

Camada Responsabilidade Exemplos Pode conhecer
Entidades Regras de negócio corporativas — mudam raramente. Pedido, Usuario, Produto Ninguém — é o núcleo
Casos de Uso Regras de negócio da aplicação — orquestram entidades. CriarPedidoUseCase, AutenticarUsuario Entidades
Adaptadores Convertem dados entre casos de uso e mundo externo. Controllers, Presenters, Gateways Casos de Uso + Entidades
Frameworks e Drivers Detalhes: banco, web, UI, serviços externos. Express, Sequelize, React, Kafka Tudo — é a camada mais externa

Portas e Adaptadores — exemplo

A camada de domínio define interfaces (portas); os detalhes fornecem implementações (adaptadores):

// Porta (definida no núcleo — casos de uso)
interface NotificacaoGateway {
  enviar(destinatario: string, mensagem: string): Promise;
}

// Adaptador (detalhe — framework externo)
class EmailNotificacaoAdapter implements NotificacaoGateway {
  async enviar(destinatario: string, mensagem: string) {
    await this.smtpClient.send({ to: destinatario, body: mensagem });
  }
}

// Caso de uso (não conhece SMTP, só a interface)
class ConfirmarPedidoUseCase {
  constructor(private notificacao: NotificacaoGateway) {}

  async executar(pedidoId: string) {
    const pedido = await this.pedidoRepo.buscar(pedidoId);
    pedido.confirmar();
    await this.notificacao.enviar(pedido.clienteEmail, 'Pedido confirmado!');
  }
}

Sinais que indicam a necessidade

Clean Architecture tem custo real de setup — use quando os sinais abaixo aparecem:

Domínio testável

Você quer testar regras de negócio sem subir banco, servidor ou mocks de framework.

Troca de framework

Há planos de mudar banco, ORM ou biblioteca de HTTP sem reescrever regras de negócio.

Equipe grande

Fronteiras explícitas reduzem acoplamento acidental entre times trabalhando em paralelo.

Longevidade

O sistema precisará evoluir por anos com requisitos que mudam — domínio deve sobreviver a modas de framework.

A regra da dependência é o coração da Clean Architecture. Combine com SOLID e Design Patterns para maximizar a manutenibilidade.

"Dependências devem apontar para dentro. Detalhes dependem de políticas, nunca o contrário."
