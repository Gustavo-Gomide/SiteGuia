Padrões para atribuir responsabilidades em OO

GRASP

General Responsibility Assignment Software Patterns — Craig Larman

GRASP é um conjunto de padrões que orientam como distribuir responsabilidades entre classes e objetos em OO. O objetivo é maximizar a coesão, minimizar o acoplamento e tornar as decisões de design explícitas e defensaveis.

Os padrões GRASP

Cada padrão responde a uma questão de design sobre onde colocar responsabilidade:

Padrão Questão respondida Resposta / Heurística
Information Expert Quem deve ter a responsabilidade? A classe que possui a informação necessária.
Creator Quem deve criar instâncias de X? Quem agrega, contém, usa muito ou inicializa X.
Controller Quem coordena o caso de uso? Um objeto dedicado que orquestra sem concentrar regras.
Low Coupling Como minimizar dependências? Distribua responsabilidades para reduzir acoplamento.
High Cohesion Como manter o foco? Uma classe deve ter responsabilidades relacionadas.
Polymorphism Como tratar alternativas baseadas em tipo? Defina operações polimórficas na classe base.
Protected Variations Como proteger de mudanças em pontos instáveis? Envolva o ponto de variação com uma interface/adaptador.

Exemplo — Expert e Controller

Na prática, Expert e Controller trabalham juntos: o Controller orquestra, o Expert calcula:

// EXPERT: Pedido sabe calcular seu total (tem os dados)
class Pedido {
  constructor(private itens: ItemPedido[]) {}

  calcularTotal(): number {
    return this.itens.reduce((acc, item) => acc + item.preco * item.quantidade, 0);
  }
}

// CONTROLLER: checkout orquestra o caso de uso
class CheckoutController {
  constructor(
    private pedidoRepo: PedidoRepositorio,
    private pagamentoService: PagamentoService,
  ) {}

  async finalizarCompra(pedidoId: string, cartao: Cartao): Promise<void> {
    const pedido = await this.pedidoRepo.buscar(pedidoId);
    const total = pedido.calcularTotal(); // delega ao Expert
    await this.pagamentoService.cobrar(cartao, total);
    pedido.confirmar();
    await this.pedidoRepo.salvar(pedido);
  }
}

// EVITE: Controller calculando o total (viola Expert + cria acoplamento)

Armadilhas comuns

Os erros mais frequentes ao aplicar GRASP em projetos reais:

Controller gordo

O Controller concentra toda a lógica de negócio em vez de delegar aos Experts. Resultado: classe com 1000 linhas.

Expert mal aplicado

Coloca-se tudo na entidade por pureza, criando acoplamento entre domínio e detalhes técnicos.

Indirection demais

Camadas extra de coordenação sem ganho real de coesão ou desacoplamento.

Low Coupling dogmático

Tentar zero acoplamento resulta em over-engineering com interfaces para tudo sem necessidade real.

GRASP é complementar a SOLID e aos Design Patterns. Use-os juntos para tomar decisões de design fundamentadas.

"Onde colocar responsabilidade é a decisão de design mais frequente. GRASP a torna explícita."
