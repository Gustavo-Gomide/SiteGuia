Princípios de Design

Coesão, acoplamento e responsabilidades bem distribuidas

Princípios de design são heurísticas para tomar boas decisões sobre onde colocar lógica e como separar responsabilidades. Não são regras absolutas — são ferramentas de análise para reduzir o custo de mudança ao longo do tempo.

Conceitos fundamentais

Antes de aplicar qualquer princípio, é essencial compreender as métricas que eles otimizam:

Conceito Definição Indicador de problema
Coesão Grau em que os elementos de um módulo pertencem ao mesmo conceito. Classe com métodos sem relação óbvia entre si.
Acoplamento Nível de dependência entre módulos — quanto um conhece os detalhes do outro. Mudar uma classe requer mudar várias outras.
Separação de responsabilidades Cada parte do sistema tem um motivo principal para mudar. Mudar regra de negócio exige alterar código de persistência.
DRY Cada conhecimento tem uma representação única e authoritative. A mesma regra de negócio existe em múltiplos lugares e diverge.
SOLID 5 princípios para código OO coeso, extensível e testável. Classes fat-model, dependências hardcoded, interfaces inchadas.

Separação de responsabilidades — exemplo

A separação de responsabilidades é o princípio mais impactante no desenho de sistemas:

// MISTURA de responsabilidades (comum em projetos iniciais)
class PedidoController {
  async criarPedido(req: Request, res: Response) {
    // 1. parsing HTTP (responsabilidade de infraestrutura)
    const dados = req.body;

    // 2. validação de negócio (responsabilidade de domínio)
    if (!dados.clienteId) throw new Error('Cliente obrigatório');

    // 3. persistência (responsabilidade de infraestrutura)
    const pedido = await db.query('INSERT INTO pedidos...');

    // 4. envio de e-mail (responsabilidade de notificação)
    await emailService.send(dados.email, 'Pedido criado');

    res.json(pedido);
  }
}

// SEPARADO: cada responsabilidade no lugar certo
class CriarPedidoUseCase {  // domínio
  async executar(cmd: CriarPedidoCmd): Promise<PedidoId> {
    const pedido = Pedido.criar(cmd.clienteId, cmd.itens); // regra de negócio
    await this.repo.salvar(pedido);
    this.eventos.publicar(new PedidoCriado(pedido.id));
    return pedido.id;
  }
}
// PedidoController cuida apenas do HTTP
// EmailListener escuta PedidoCriado e envia e-mail

Como decidir

Use estas perguntas como guia ao tomar decisões de design:

Quantos motivos para mudar?

Se um arquivo muda por vários motivos diferentes, a coesão está baixa. Separe por motivo de mudança.

Quantos arquivos mudam juntos?

Se mudar X implica mudar Y, Z e W, o acoplamento está alto. Introduza uma abstração entre eles.

Consigo testar isso sem infraestrutura?

Se testes precisam de banco e servidor, as responsabilidades estão misturadas. Separe domínio de detalhes.

A abstração paga seu custo?

Toda abstração tem custo cognitivo. Só crie quando o benefício mensurado supera esse custo.

Princípios de design não podem ser aplicados cegamente. Use SOLID, GRASP e DRY/KISS/YAGNI como check-lists, não dogmas.

"A métrica real do bom design é o custo de mudança, não a pureza arquitetural."
