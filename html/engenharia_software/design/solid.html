5 princípios para código OO coeso e extensível

SOLID

Fundamentos de design orientado a objetos — Robert C. Martin

SOLID reúne cinco princípios que tornam o código OO coeso, testável e fácil de evoluir. Cada princípio isola um problema recorrente: responsabilidade difusa, rigidez a mudanças, polimorfismo incorreto, interfaces inchadas e dependência em detalhes.

Os 5 princípios

Cada letra endereça uma categoria específica de problema no design orientado a objetos:

Letra Nome Enunciado Violação típica
S Single Responsibility Uma classe deve ter apenas um motivo para mudar. Classe com lógica de negócio + persistência + envio de e-mail.
O Open/Closed Aberto para extensão, fechado para modificação. if/else crescente para cada novo tipo de desconto.
L Liskov Substitution Subtipos substituem seus tipos base sem surpresas. Subclasse lança exceção em método que apenas retorna valor.
I Interface Segregation Prefira interfaces pequenas e específicas. Interface com 15 métodos; classes implementam apenas 3.
D Dependency Inversion Módulos de alto nível não dependem de detalhes concretos. new MySQLRepositorio() hardcoded dentro do serviço.

Exemplo — Dependency Inversion na prática

DIP é o princípio mais impactante para testabilidade: ao depender de abstrações, você substitui implementações sem tocar no domínio.

// ANTES (acoplado ao detalhe)
class PedidoService {
  private repo = new PedidoRepositorioSQL(); // concreto!

  processar(id: string) {
    const pedido = this.repo.buscar(id);
    pedido.confirmar();
    this.repo.salvar(pedido);
  }
}

// DEPOIS (depende da abstração)
interface PedidoRepositorio {
  buscar(id: string): Pedido;
  salvar(pedido: Pedido): void;
}

class PedidoService {
  constructor(private repo: PedidoRepositorio) {} // injetado

  processar(id: string) {
    const pedido = this.repo.buscar(id);
    pedido.confirmar();
    this.repo.salvar(pedido);
  }
}

// No teste: new PedidoService(new FakePedidoRepositorio())

Quando priorizar cada princípio

Na prática, diferentes princípios são mais urgentes em diferentes situações de código:

SRP

Arquivo cresce demais; testes requerem mocks de subsistemas não relacionados.

OCP

Há um if/switch crescente para cada novo tipo ou estratégia de negócio.

LSP

Hierarquias de herança ou coleções polimórficas com comportamentos que diferem inesperadamente.

ISP

Classes implementam métodos que nunca usam (retornam null ou lançam NotImplemented).

DIP

Testes unitários são difíceis por dependências hardcoded de banco, rede ou serviços externos.

SOLID é guia, não lei. Excesso de abstrações "por princípio" contradiz YAGNI e KISS. Aplique onde o benefício concreto é visível.

"O objetivo do SOLID é reduzir o custo de mudança, não aumentar o número de arquivos."
