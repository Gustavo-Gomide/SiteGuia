Microserviços

Autonomia de times ao custo de complexidade distribuída

Microserviços são serviços pequenos com deploy independente que se comunicam por rede. Eles aumentam a autonomia de times, mas trazem custo real: observabilidade distribuída, consistência eventual, latencia de rede e complexidade operacional que não existem no monólito.

Microserviços vs Monólito

Cada abordagem tem vantagens e custos reais — a escolha depende de maturidade e contexto:

Dimensão Monólito Microserviços
Deploy Uma única unidade — simples e previsível. Serviços independentes — requer CI/CD maduro.
Observabilidade Logs e traces centralizados — debug simples. Tracing distribuído necessário (Jaeger, OpenTelemetry).
Consistência Transações ACID entre módulos. Consistência eventual ou sagas distribuídas.
Escala Escala toda a aplicação. Escala serviços individualmente.
Times Um time pode dominar tudo — coordenação necessária. Times autônomos por serviço/domínio.
Custo inicial Baixo — ideal para MVP e produtos em validação. Alto — exige infraestrutura madura desde o início.

Padrões de comunicação

Serviços comunicam por dois padrões principais com trade-offs distintos:

// PADRÃO 1: SÍNCRONO (REST/gRPC) — resposta imediata
// Vantagem: fluxo simples. Desvantagem: acoplamento temporal.

// POST /api/pedidos -> Serviço Pedidos
//   -> GET /api/estoque/{produtoId}  (síncrono, aguarda)
//   -> POST /api/pagamentos          (síncrono, aguarda)
//   -> resposta final ao cliente

// PADRÃO 2: ASSÍNCRONO (eventos) — desacoplado
// Vantagem: resiliência e desacoplamento. Desvantagem: consistência eventual.

// POST /api/pedidos -> Serviço Pedidos
//   -> publica PedidoCriado no broker
//   -> resposta imediata ao cliente (202 Accepted)

// Serviço Estoque consome PedidoCriado
//   -> reserva estoque
//   -> publica EstoqueReservado

// Serviço Pagamentos consome EstoqueReservado
//   -> processa pagamento
//   -> publica PedidoPago ou PagamentoRecusado

Quando e como usar

A decisão de adotar microserviços deve ser baseada em maturidade operacional, não em moda:

Use quando

Múltiplos times com ritmo próprio. Bounded contexts claros. Necessidade real de escala/isolamento de falha.

Evite quando

Sem CI/CD confiável. Sem monitoramento/alertas. Domínio ainda mudando e fronteiras indefinidas.

Checklist operacional

Tracing distribuído, alertas por serviço, rollback individual, health checks, circuit breakers.

Comece pelo monólito

Um monólito modular é muitas vezes a etapa correta — extraía serviços quando a fronteira próvar seu valor.

Microserviços são naturalmente alinhados a Bounded Contexts do DDD. Combine com Event-Driven para desacoplar comunicação entre eles.

"Microserviços resolvem problemas de org. Antes do problema de org existir, eles são o problema."
