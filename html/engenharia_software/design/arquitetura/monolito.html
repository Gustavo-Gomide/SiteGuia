Monólito

A escolha racional para muitos contextos — especialmente produtos iniciais

Monólito não significa código bagunçado. É uma única unidade de deploy que pode ser altamente modular por dentro. Para a maioria dos produtos, um monólito bem estruturado entrega mais rápido e com menos custo operacional do que uma arquitetura distribuída prematura.

Módulos vs camadas genéricas

A diferença entre um monólito saudável e um que vira um ball-of-mud está na organização interna:

Abordagem Organização Resultado
Camadas genéricas controllers/, services/, repositories/ com todos os domínios misturados. Acoplamento alto, mudanças cascata, difícil de extrair para microserviço.
Módulos por domínio pedidos/, usuarios/, estoque/ — cada um com seus controllers, services, repos. Fronteiras explícitas, mudanças locais, fácil de extraír para microserviço no futuro.
Monólito modular Módulos por domínio com interfaces explícitas entre eles (sem acesso direto ao banco alheio). Máximo benefício: simplicidade do monólito + fronteiras claras como microserviços.

Estrutura recomendada

Um monólito bem organizado facilita a extração futura de serviços se necessário:

src/
  pedidos/
    pedido.entity.ts      # entidade de domínio
    pedido.repo.ts        # interface de repositório
    pedido.repo.sql.ts    # implementação SQL
    criar-pedido.usecase.ts
    pedidos.controller.ts
    pedidos.module.ts     # exporta apenas a interface pública

  usuarios/
    usuario.entity.ts
    usuarios.module.ts
    # ... mesma estrutura

  estoque/
    estoque.module.ts
    # ...

  app.module.ts  # agrupa módulos

# Regra: pedidos/ não importa diretamente de usuarios/
# Comunicação via interface explícita ou evento interno

Quando manter e quando migrar

Sinais para cada decisão:

Continue no monólito

Produto em validação. Time pequeno. Fronteiras ainda incertas. Custo operacional de dist. não justificado.

Modularize primeiro

Antes de extrair microserviço, isole o módulo com fronteiras limpas dentro do monólito.

Sinal de alerta

Build/deploy lentos e frágeis. Acoplamento impedindo times de ir rápido. Necessidade real de escala isolada.

Monólito modular

É muitas vezes a etapa final correta — não um passo intermediário. Não force a distribuição.

Mudar de monólito para Microserviços é mais simples quando o monólito já tem fronteiras limpas (baseadas em Bounded Contexts).

"Um monólito mal escrito não é o problema da arquitetura monolítica — é o problema do código."
