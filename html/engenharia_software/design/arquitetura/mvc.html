MVC

Model-View-Controller — padrão clássico de separação de responsabilidades

MVC separa o sistema em três partes: o Model representa dados e regras, a View renderiza e apresenta, e o Controller coordena entradas do usuário e seleciona a resposta. Na prática web, o Controller orquestra chamadas a serviços/casos de uso — não contém regras de negócio.

Papel de cada parte

Cada componente tem uma responsabilidade clara e violatões conhecidas:

Componente Responsabilidade Não deve Armadilha comum
Model Estado, validações e invariantes do domínio. Conhecer HTTP, renderização ou ORM. Model anêmico (apenas getters/setters, regras no Controller).
View Apresentar dados ao usuário (HTML, JSON, etc.). Decidir regras de negócio ou acessar BD. Lógica de negócio escondida em templates/componentes.
Controller Receber input, chamar serviço/caso de uso, selecionar View. Conter regras de negócio ou manipular banco diretamente. Controller gordo com 500+ linhas de lógica.

Fluxo request-response

Um request HTTP fluindo por MVC com separação correta de responsabilidades:

// CONTROLLER — coordena, não decide
export class PedidosController {
  constructor(private criarPedidoUseCase: CriarPedidoUseCase) {}

  async criarPedido(req: Request, res: Response): Promise<void> {
    // 1. Parsing e validação de input (responsabilidade do Controller)
    const { clienteId, itens } = PedidoSchema.parse(req.body);

    // 2. Delegar a lógica ao caso de uso (NUNCA aqui)
    const pedidoId = await this.criarPedidoUseCase.executar({
      clienteId,
      itens,
    });

    // 3. Selecionar resposta (responsabilidade do Controller)
    res.status(201).json({ pedidoId });
  }
}

// CASO DE USO (fora do MVC, orquestra domínio)
class CriarPedidoUseCase {
  async executar(cmd: CriarPedidoCmd): Promise<string> {
    const pedido = Pedido.criar(cmd.clienteId, cmd.itens);
    await this.repo.salvar(pedido);
    return pedido.id;
  }
}

Corrigiões práticas

Os anti-padrões mais comuns e como corrigi-los:

Controller gordo

Mova regras para Service ou Use Case. O Controller deve ter no máximo 20-30 linhas por action.

Model anêmico

Consolidar invarântes dentro da entidade. Se o estado só muda via métodos, o model defende-se.

View com lógica

Extraia cálculos e decisões para Presenter/ViewModel. A View apenas renderiza dados prontos.

Testabilidade

Controllers magros são fáceis de testar. Casos de uso testam-se sem HTTP. Models testam invariantes.

MVC é um ponto de partida. Em sistemas maiores, combine com MVVM (para UI rica) ou Clean Architecture (para domínio complexo).

"Um Controller que faz mais do que coordenar é um Controller que roubou responsabilidade de outra camada."
