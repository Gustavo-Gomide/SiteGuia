MVVM

Model-View-ViewModel — padrão para UIs reativas e testáveis

MVVM é muito usado em UIs ricas (mobile, desktop, SPAs). O ViewModel expõe estado e ações prontas para a View consumir via data binding. A View ouve o ViewModel, mas o ViewModel não conhece a View — isso torna a lógica de UI testável sem renderização real.

Componentes e responsabilidades

Cada parte tem responsabilidade clara e testabilidade distinta:

Componente Responsabilidade Testabilidade
Model Dados e regras do domínio. Independente de UI e ViewModel. Unit tests simples sem UI.
ViewModel Estado derivado para a View + ações/comandos. Não conhece a View. Unit tests; injete fakes dos serviços de domínio.
View Renderização e captura de eventos. Bind ao ViewModel state. Testes de UI; verificação de binding e interações.

ViewModel reativo — exemplo

Um ViewModel de login expondo estado e ação sem saber como a View renderiza:

// VIEWMODEL (não conhece a View)
class LoginViewModel {
  // Estado observável
  email = '';
  senha = '';
  carregando = false;
  erro: string | null = null;

  // Estado derivado
  get formValido(): boolean {
    return this.email.includes('@') && this.senha.length >= 6;
  }

  // Ação
  async login(): Promise<void> {
    if (!this.formValido) return;

    this.carregando = true;
    this.erro = null;

    try {
      await this.authService.login(this.email, this.senha);
      this.router.navigate('/dashboard');
    } catch (e) {
      this.erro = 'Email ou senha incorretos.';
    } finally {
      this.carregando = false;
    }
  }
}

// VIEW (React/Vue/Angular — apenas bind e render)
// <button disabled={!vm.formValido} onClick={vm.login}>Entrar</button>
// <span>{vm.erro}</span>
// <input value={vm.email} onChange={e => vm.email = e.target.value} />

// TESTE do ViewModel (sem renderizar UI)
test('formValido false quando email inválido', () => {
  const vm = new LoginViewModel(fakeAuth, fakeRouter);
  vm.email = 'invalido';
  vm.senha = '123456';
  expect(vm.formValido).toBe(false);
});

Quando usar e armadilhas

MVVM brilha em UIs com muito estado interativo, mas tem anti-padrões conhecidos:

Use quando

UIs ricas com muito estado. Testes de lógica de UI importantes. Data binding reduz boilerplate.

ViewModel gigante

Separe por tela ou funcionalidade. Extraia casos de uso para serviços de domínio testados separadamente.

Estado global demais

Estado global dificulta debugging e testes. Prefira estado local ao ViewModel de cada tela.

Binding confuso

Regras de UI escondidas em bindings. Derive estado explicitamente no ViewModel — a View é estúpida.

MVVM é uma especialização de MVC para UIs reativas. Combine com Hexagonal para separar ViewModel de detalhes de infraestrutura.

"O ViewModel é o modelo da UI, não do domínio. Esses conceitos devem ser distintos."
