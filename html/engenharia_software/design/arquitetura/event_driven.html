Arquitetura Event-Driven

Desacoplamento por publicação e consumo de eventos

Em arquitetura event-driven, serviços publicam eventos imutáveis (fatos do passado) e outros serviços reagem de forma assíncrona. Isso desacopla produtores de consumidores, mas exige disciplina: idempotência, observabilidade e consistência eventual são problemas reais que precisam de estratégia.

Conceitos fundamentais

Os blocos básicos de uma arquitetura event-driven:

Conceito Definição Exemplo
Evento Fato imutável que ocorreu no passado. PedidoPago, UsuarioCriado, EstoqueBaixado
Publisher Serviço que emite o evento. Serviço de Pedidos publica PedidoConfirmado
Consumer Serviço que processa o evento de forma independente. Faturamento, Estoque, Notificações
Broker Infraestrutura de entrega de mensagens (fila/stream). Kafka, RabbitMQ, AWS SQS/SNS
Dead-letter Queue Destino para mensagens que falharam após retries. Fila de dead-letters com alertas e reprocessamento manual

Exemplo — publicação e consumo

Um evento de pagamento confirmado sendo publicado e consumido por múltiplos serviços:

// Evento (imutável, fato do passado)
type PedidoPagoEvento = {
  tipo: 'PedidoPago';
  pedidoId: string;
  valor: number;
  dataPagamento: string;
  clienteId: string;
};

// Publisher (Serviço de Pagamentos)
async function confirmarPagamento(pagamentoId: string): Promise {
  const pagamento = await pagamentoRepo.buscar(pagamentoId);
  pagamento.confirmar();
  await pagamentoRepo.salvar(pagamento);

  // Publica evento DEPOIS de persistir (at-least-once)
  await broker.publicar('PedidoPago', {
    tipo: 'PedidoPago',
    pedidoId: pagamento.pedidoId,
    valor: pagamento.valor,
    dataPagamento: new Date().toISOString(),
    clienteId: pagamento.clienteId,
  });
}

// Consumer (Faturamento) — DEVE ser idempotente
async function handlePedidoPago(evento: PedidoPagoEvento): Promise {
  const jaProcessado = await faturaRepo.existe(evento.pedidoId);
  if (jaProcessado) return; // idempotência

  await faturaRepo.criar({
    pedidoId: evento.pedidoId,
    valor: evento.valor,
    data: evento.dataPagamento,
  });
}

Cuidados essenciais

Os problemas que surgem ao não considerar esses pontos são difíceis de depurar em produção:

Idempotência

O mesmo evento pode ser entregue 2x. O consumer deve lidar com isso sem duplicar dados ou afeitos.

Observabilidade

Rastrear um fluxo entre serviços exige correlation ID, tracing distribuído e logs estruturados.

Retry e Dead-letter

Falhas nos consumers precisam de retry automático e destino para mensagens irrecuperáveis.

Consistência eventual

Dados entre serviços convergem ao longo do tempo, não imediatamente. Projeto deve aceitar isso.

Arquitetura event-driven combina bem com Microserviços e DDD (Domain Events como fronteira de bounded contexts).

"Eventos desacoplam coordenação, mas acoplam significado. Nomeie-os com cuidado."
