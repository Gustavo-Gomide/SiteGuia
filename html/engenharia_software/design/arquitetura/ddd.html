DDD

Domain-Driven Design — Eric Evans

DDD é uma abordagem de design onde o modelo de software reflete a linguagem e as regras reais do domínio. O objetivo é criar sistemas com fronteiras explícitas, termos compartilhados entre time e negócio, e código que evolui conforme o entendimento cresce.

Blocos fundamentais

DDD organiza o domínio em blocos táticos (construções em código) e estratégicos (fronteiras e termos):

Bloco Tipo Função
Ubiquitous Language Estratégico Linguagem única compartilhada entre time e domínio.
Bounded Context Estratégico Fronteira onde um modelo de domínio faz sentido e é consistente.
Entity Tático Objeto com identidade única que persiste ao longo do tempo.
Value Object Tático Objeto definido por seus atributos, sem identidade própria (imutável).
Aggregate Tático Cluster de Entities/Value Objects com uma raiz única que garante invariantes.
Domain Event Tático Fato relevante ocorrido no domínio (ex: PedidoCriado, PagamentoRecusado).
Repository Tático Interface de persistência definida pelo domínio (não pelo banco).

Aggregate — exemplo de invariante

O Aggregate garante consistência interna: não há forma de colocar o sistema em estado inválido via API externa:

class Pedido { // Aggregate Root
  private status: 'rascunho' | 'confirmado' | 'cancelado' = 'rascunho';
  private itens: ItemPedido[] = [];

  adicionarItem(produto: Produto, quantidade: number): void {
    if (this.status !== 'rascunho') {
      throw new Error('Não é possível alterar pedido confirmado.');
    }
    this.itens.push(new ItemPedido(produto, quantidade));
  }

  confirmar(): DomainEvent[] {
    if (this.itens.length === 0) {
      throw new Error('Pedido sem itens não pode ser confirmado.');
    }
    this.status = 'confirmado';
    return [new PedidoConfirmado(this.id, this.calcularTotal())];
  }

  // itens só são acessíveis via Pedido — não há repositório de ItemPedido
}

Quando aplicar (e quando evitar)

DDD tem custo de implementação real — aplique onde o retorno justifica:

Aplique quando

Domínio complexo com muitas regras e exceções. Múltiplos times com cadencia própria. Termos ambiguos entre áreas.

Evite quando

CRUD simples sem regras de negócio. Produto em validação (fronteiras mudam rápido). Time pequeno sem maturidade no domínio.

Como começar

Mapeie termos reais com o negócio. Escolha um fluxo crítico. Modele as invariantes. Isole infraestrutura trás de portas.

Armadilhas

Excesso de camadas sem clareza. Linguagem técnica em vez da do negócio. DDD como pasta de código, não como modelo mental.

DDD se complementa com Clean Architecture (fronteiras e dependências) e Microserviços (bounded contexts como limite de serviço).

"O modelo de domínio não é o banco de dados. É a linguagem executavél do negócio."
