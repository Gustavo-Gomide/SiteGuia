Uma única instância com acesso global — use com cautela

Singleton

Criacional — GoF — o padrão mais duvidoso do catálogo

Singleton garante que uma classe tenha exatamente uma instância e fornece um ponto global de acesso. É frequentemente mal aplicado como variável global com estado compartilhado. Prefira inversão de dependência — injete a dependência única em vez de acessá-la globalmente.

Singleton válido vs anti-padrão

Caso Singleton válido? Alternativa melhor
Logger de aplicação Sim — sem estado mutavél, thread-safe. Injetar logger via DI ainda é melhor para testes.
Pool de conexões Sim — recurso único gerenciado. Gerenciado pelo container de DI.
Cache de configuração Condicional — se imutável após inicialização. Objeto de configuração imutável injetado.
Estado global mutável Não — acoplamento oculto, difícil de testar. Inversão de dependência + escopo gerenciado.

Implementação e alternativa recomendada

// SINGLETON clássico (use apenas quando justificado)
class DatabasePool {
  private static instancia: DatabasePool | null = null;
  private conexoes: Connection[] = [];

  private constructor() {
    // inicializa pool
  }

  static getInstance(): DatabasePool {
    if (!DatabasePool.instancia) {
      DatabasePool.instancia = new DatabasePool();
    }
    return DatabasePool.instancia;
  }

  buscarConexao(): Connection {
    return this.conexoes.pop() ?? this.criarConexao();
  }
}

// ALTERNATIVA PREFERIDA: inversão de dependência
// Não acesse DatabasePool.getInstance() em todo lugar.
// Configure uma única instância no composition root:

const pool = new DatabasePool(); // única vez, na inicialização

// Injete onde necessário:
class PedidoRepository {
  constructor(private pool: DatabasePool) {} // injetável e testável
}

Quando usar e armadilhas

Use apenas quando

A unicidade é uma restrição real do domínio (não uma conferência). Exemplo: pool de recursos.

Problema em testes

Singleton com estado global afeta outros testes. Substitua por objeto injetado mesmo que seja único em produção.

Thread safety

Em ambientes multi-thread (Java, Go), a inicialização deve ser sincronizada ou usar padrões como Holder.

Substitua por DI

Frameworks de DI (Spring, NestJS, Guice) gerenciam escopos singleton sem os problemas do acesso global.

Singleton é o padrão GoF mais discutido negativamente. Prefira inversão de dependência: injete a instância única em vez de acessá-la globalmente. Veja DIP no SOLID.

"O problema do Singleton não é a unicidade — é o acesso global. A unicidade é gerenciável; o estado global escondido não."
