Design Patterns

Vocabulário comum para comunicar intenção de design

Design patterns não são código para copiar — são soluções nomeadas para problemas de design que se repetem. Eles reduzem acoplamento, comunicam intenção ("use um Strategy aqui") e facilitam manutenção quando aplicados com contexto real.

Três categorias

Os 23 padrões do Gang of Four (GoF) se dividem em três categorias por intenção:

Categoria Objetivo Padrões principais
Criacional Controlar como objetos são criados. Factory Method, Abstract Factory, Builder, Prototype, Singleton
Estrutural Compor objetos em estruturas maiores. Adapter, Bridge, Composite, Decorator, Facade, Flyweight, Proxy
Comportamental Definir como objetos se comunicam e distribuem responsabilidade. Strategy, Observer, Command, State, Template Method, Iterator, Visitor...

Pattern como intenção, não como código

O value dos patterns é a comunicação de intenção. Compara o problema sem e com pattern:

// SEM PATTERN — lógica de desconto espalhada
function calcularDesconto(tipo: string, valor: number): number {
  if (tipo === 'vip') return valor * 0.2;
  if (tipo === 'bulk') return valor * 0.15;
  if (tipo === 'promo') return valor * 0.1;
  return 0;
} // adicionar novo tipo = mexer nessa função

// COM STRATEGY — cada regra é uma classe isolada
interface DescontoStrategy {
  calcular(valor: number): number;
}

class DescontoVip implements DescontoStrategy {
  calcular(valor: number) { return valor * 0.2; }
}

class Pedido {
  constructor(private desconto: DescontoStrategy) {}
  calcularTotal(valor: number) { return valor - this.desconto.calcular(valor); }
} // adicionar novo tipo = nova classe, código existente não muda (OCP)

Quando aplicar e quando evitar

Patterns mal aplicados aumentam complexidade sem benefício real:

Use quando

Há variação real que precisa ser isolada. Existão múltiplos consumidores do mesmo comportamento. O nome do pattern comunica intenção claramente.

Evite quando

Você está criando abstração "porque sim". Há apenas uma implementação possível. O código simples já resolve o problema.

Antes de aplicar

Qual problema concreto ele resolve? Que mudança futura está sendo isolada? O time entende e consegue manter?

Pattern não é objetivo

O objetivo é reduzir acoplamento e facilitar mudança. O pattern é apenas um meio nomeado para isso.

Comece pelos padrões mais comuns: Strategy, Observer e Adapter aparecem na maioria dos sistemas.

"Pattern é vocabulário, não receita. Use quando o problema já existe, não para prevenir futuros imaginários."
