TDD (Test-Driven Development)

Ciclo Red-Green-Refactor, benefícios e quando vale a pena

TDD é uma técnica de desenvolvimento onde o teste é escrito antes da implementação. O ciclo Red → Green → Refactor força clareza na interface, documenta comportamento esperado e gera uma rede de segurança para refatoração contínua.

O ciclo Red → Green → Refactor

TDD não é "escrever mais testes" — é um ritmo de trabalho em passos pequenos e verificáveis:

# Exemplo completo: regra de desconto

# === RED — escreva um teste que FALHA ===
def deve_aplicar_desconto_10_porcento_para_pedido_100_ou_mais():
    assert calcular_total(100) == 90.0  # função ainda não existe → FALHA

# === GREEN — implemente o mínimo para PASSAR ===
def calcular_total(valor):
    if valor >= 100:
        return valor * 0.9
    return valor

# teste passa agora ✓

# === REFACTOR — melhore sem mudar comportamento ===
LIMITE_DESCONTO = 100
TAXA_DESCONTO = 0.10

def calcular_total(valor: float) -> float:
    if valor >= LIMITE_DESCONTO:
        return valor * (1 - TAXA_DESCONTO)
    return valor

# testes ainda passam ✓

# Próximo teste — borda
def deve_nao_aplicar_desconto_para_pedido_abaixo_de_100():
    assert calcular_total(99) == 99.0  # ← RED → GREEN → REFACTOR

Quando TDD ajuda (e quando não)

TDD tem custo. Os benefícios se materializam em contextos específicos — força conhecer onde aplicar:

Benefício Como TDD o produz Quando o benefício é menor
Design emergente melhor Código testável tende a ser modular e coeso Protótipos e exploração inicial de domínio desconhecido
Especificação executável Testes documentam comportamento com exemplos concretos UI e código de apresentação (difícil de testar antes)
Segurança para refatorar Rede de testes gerada naturalmente ao longo do desenvolvimento Código com dependências difíceis de controlar (I/O pesado)
Foco em interface pública Teste define contrato antes de pensar em implementação Algoritmos de exploração onde a interface muda com frequency
Feedback imediato Ciclos curtos detectam erros na hora, não depois Infraestrutura e scripts de automação de ambiente

Armadilhas comuns

TDD aplicado mecanicamente sem reflexão gera os mesmos problemas que a falta de testes:

Testes acoplados à implementação

Testar detalhes internos faz o teste quebrar em toda refatoração. O teste deve validar o comportamento da interface — não como foi implementado.

Testar frameworks em vez de regras

Verificar que o ORM persiste um objeto não é TDD útil. Foque nas regras de negócio onde o valor real está.

Confundir com "teste depois"

TDD é uma técnica de design, não de verificação. Escrever testes depois da implementação pode ter valor, mas é uma prática diferente com resultados diferentes.

Aplicar em todo tipo de código

TDD funciona melhor onde a interface é estável e o comportamento é definível antes do código. Para exploração ou UI, outras abordagens são mais práticas.

BDD é uma extensão do TDD com foco em linguagem de negócio — explore BDD. Para refatoração segura guiada por testes, veja Refatoração.

"TDD não é sobre testes — é sobre design guiado por exemplos."
