Testes de Aceitação

Critérios de aceite, linguagem ubíqua e cenários executáveis

Testes de aceitação verificam se uma feature atende os critérios de aceite definidos pelo negócio. O foco é "o que o sistema deve fazer", em linguagem de domínio — não como foi implementado. Quando automatizados, viram documentação viva que protege o comportamento ao longo do tempo.

Escrevendo bons critérios de aceite

Critérios de aceite são a fronteira entre "done" e "não done". Precisam ser objetivos, verificáveis e redigidos na linguagem do domínio:

Característica Como aplicar Exemplo
Objetivo e verificável Evite "deve ser rápido"; use "responde em < 200ms no p95" Reembolso liberado em até 5 dias úteis
Linguagem do domínio Use termos do produto, não de código (tabela, endpoint, classe) Comprador vê o pedido em "Aguardando envio"
Caso feliz + borda Pelo menos 1 cenário positivo e 1–2 cenários de erro Pedido válido é criado; pedido sem estoque retorna erro
Independente de UI Validar comportamento, não como a tela ficou Sistema registra o evento — não "o botão ficou verde"

Estrutura Given / When / Then

A estrutura GWT força clareza: pré-condição, ação e resultado esperado. É a base do BDD e de ferramentas como Cucumber e Behave:

# Cenário: solicitar reembolso dentro do prazo

Dado que o usuário autenticado tem um pedido entregue há 3 dias
Quando ele solicita reembolso com motivo "produto com defeito"
Então o sistema registra a solicitação com status "em análise"
  E envia notificação por e-mail ao comprador
  E cria tarefa para a equipe de suporte

# Cenário: solicitar reembolso fora do prazo

Dado que o usuário tem um pedido entregue há 35 dias
Quando ele tenta solicitar reembolso
Então o sistema recusa com a mensagem "prazo de 30 dias encerrado"
  E não cria nenhuma tarefa de suporte

# Boas práticas para cenários:
#   ✓ Uma ação por cenário
#   ✓ Dados concretos (3 dias, 35 dias)
#   ✓ Resultado de negócio, não técnico
#   ✗ Evite múltiplas ações encadeadas no mesmo "Quando"
#   ✗ Não mencione IDs internos, tabelas ou endpoints

Armadilhas comuns

Testes de aceitação bem-intencionados que viram fardo são o anti-padrão mais comum nesse nível:

Roteiro gigante e lento

Um cenário de aceitação que percorre 20 etapas é difícil de debugar quando falha. Quebre em cenários menores e independentes.

Cenários duplicados

Testar o mesmo comportamento em múltiplos cenários cria redundância cara. Um cenário bem escrito cobre o caso — o resto é ruído.

Critérios vagos

"Deve ser rápido", "deve funcionar bem" não são critérios — são desejos. Sem número ou condição verificável, não há como automatizar nem validar.

Não rodar no CI

Cenários que só existem em documentação viram stale rapidamente. Se não roda no CI, não é documentação viva — é texto desatualizado.

Para a técnica de desenvolvimento guiada por comportamento, veja BDD. Para onde rodar esses testes no sistema real, explore testes de sistema (E2E).

"Critério de aceite vago é convite para debate depois da entrega."
