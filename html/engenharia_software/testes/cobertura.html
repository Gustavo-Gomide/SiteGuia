Cobertura de Testes

O que coverage mede, seus limites e como usá-la de forma saudável

Cobertura de testes mede quanto do código foi executado pelos testes. É um sinal útil — mas não uma garantia. É possível ter 90% de coverage e bugs críticos não cobertos. A pergunta certa não é "quanto cobrimos?" mas sim "cobrimos o que pode quebrar e custa caro quando quebra?"

O que cada tipo de coverage mede

Ferramentas de coverage reportam métricas diferentes. Cada uma captura um ângulo distinto do risco não coberto:

Tipo O que mede Valor prático Limitação
Line coverage Quais linhas de código foram executadas Detecta arquivos completamente sem teste Linha executada ≠ comportamento validado
Branch coverage Quais caminhos de if/else/switch foram tomados Revela branches não testados (edge cases) Não valida que o assert cobre a regra
Function coverage Quais funções foram chamadas Detecta código morto ou não testado Função chamada sem assert significativo
Statement coverage Quais statements individuais foram executados Mais granular que line coverage Mesmo problema: execução ≠ validação
Mutation coverage Testes detectam mutações no código? Revela asserts fracos com alta cobertura Caro computacionalmente — raro no CI

Limites e armadilhas da cobertura

Alta cobertura com testes ruins é o pior cenário: gera confiança falsa e esconde riscos reais:

# Exemplos de testes que inflam coverage sem valor real

# 1. Teste sem assert significativo
def test_calcular_desconto():
    resultado = calcular_desconto(100)
    assert resultado is not None  # ← inflama coverage, valida nada

# 2. Teste acoplado à implementação
def test_processar_pedido():
    mock_repo.save.assert_called_once()  # ← verifica chamada, não comportamento

# 3. Teste que executa mas não valida regra
def test_aplicar_promo():
    carrinho.aplicar_promocao("PROMO10")
    # sem assert — coverage 100%, proteção 0%

# Uso saudável de coverage:
#   ✓ Detectar arquivos sem nenhum teste
#   ✓ Identificar branches não testados (if/else críticos)
#   ✓ Monitorar quedas bruscas no PR (coverage caiu 5% → revisão)
#   ✓ Guiar onde adicionar testes antes de refatoração

# Uso ruim de coverage:
#   ✗ Meta rígida de 80% sem critério de valor
#   ✗ Usar coverage para medir "qualidade do desenvolvedor"
#   ✗ Ignorar mutation coverage e confiar só em line/branch

Onde focar a cobertura

A cobertura que protege de verdade está nas áreas de maior risco e maior frequência de mudança:

Regras de negócio

Cálculos, descontos, permissões, validações. Erros aqui têm impacto direto — financeiro, segurança, dados do usuário.

Edge cases e limites

Zeros, negativos, valores máximos, strings vazias, datas em virada de mês. São onde os bugs vivem em produção.

Integrações críticas

Serialização de API, schema de banco, contratos entre serviços. Cobertura aqui evita incompatibilidades silenciosas.

Áreas de alto churn

Código alterado frequentemente tem maior probabilidade de regredir. Coverage alta aqui reduz o custo de cada mudança futura.

Para métricas de engenharia além da cobertura, veja métricas de software. Para como escrever testes que realmente protegem, explore testes unitários.

"100% de coverage com asserts fracos é pior que 60% com testes sólidos."
