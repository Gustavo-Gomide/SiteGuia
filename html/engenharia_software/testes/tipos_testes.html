Cada tipo de teste cobre um risco diferente

Tipos de Testes

Comparativo de custo, velocidade e quando usar cada um

Não existe um tipo de teste que resolva tudo. Unitários são rápidos mas não pegam integração real; E2E pega tudo mas é caro e frágil. A estratégia certa combina os tipos no nível correto — e o ponto de partida é entender o que cada um cobre.

Comparativo dos tipos principais

A tabela abaixo resume o perfil de cada tipo de teste considerando os trade-offs mais relevantes para decisão:

Tipo O que testa Velocidade Custo Quando usar
Unitário Função/classe em isolamento Muito rápido Baixo Regras, cálculos, validações, edge cases
Integração Código + dependências reais (DB, fila, HTTP) Médio Médio Queries, serialização, contratos entre camadas
Sistema / E2E Jornada completa do usuário Lento Alto Fluxos críticos: login, pagamento, cadastro
Aceitação Critérios de aceite de uma story Variável Alto Quando negócio precisa validar entrega
Regressão Bugs que não podem voltar Variável Baixo–médio Após incidentes e áreas instáveis
Performance Latência, throughput, limites de carga Lento Alto Antes de eventos de alta carga, lançamentos

Pirâmide e variações

A pirâmide clássica sugere mais testes unitários e menos E2E. Dependendo do contexto, o formato muda:

# Pirâmide clássica (aplicações com lógica de negócio rica)
    /E2E\         ← poucos, fluxos críticos
   /------\
  /Integra\       ← médios, contratos e dependências
 /----------\
/  Unitários \    ← muitos, rápidos, baratos

# Troféu de testes (APIs e frontends modernos)
     /    E2E   \
    /------------\
   /  Integração  \   ← maior peso aqui
  /----------------\
 /     Unitários    \

# Anti-padrão: pirâmide invertida
     /   Unitários  \   ← raros
    /----------------\
   /     Integração   \
  /--------------------\
 /        E2E           \  ← suíte frágil, lenta e cara

Armadilhas comuns por tipo

Cada tipo de teste tem seu conjunto específico de erros que silenciosamente corroem a qualidade da suíte:

Unitários acoplados à implementação

Testar detalhes internos (métodos privados, estrutura interna) faz a suíte quebrar em toda refatoração, mesmo sem mudar comportamento.

Integração sem ambiente controlado

Usar banco compartilhado ou dados externos gera flakiness. Use containers efêmeros, seeds determinísticos e cleanup após cada teste.

E2E para tudo

Validar regras de negócio via E2E é 100× mais lento e frágil que unitários. Reserve E2E para jornadas completas onde o caminho integrado importa.

Ignorar testes de performance

Performance sem baseline e sem SLO definido não protege nada. Degradações graduais só aparecem em incidentes de produção.
