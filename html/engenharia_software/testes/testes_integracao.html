Testes de Integração

Dependências reais, estratégias de ambiente e estabilidade

Testes de integração verificam se o código funciona corretamente com dependências reais: banco de dados, filas, cache, filesystem e APIs externas. Eles capturam problemas que testes unitários jamais veem — schema errado, query malformada, serialização incorreta, configuração faltando.

Quando e onde usar

Não é toda camada que precisa de teste de integração, mas estas são as que mais se beneficiam:

Camada / Componente O que validar Estratégia recomendada
Repositórios / ORM Queries corretas, índices, migrations, constraints DB real em container efêmero com seed pequeno
Serialização / Deserialização JSON correto, campos opcionais, tipos, enums Testes com payloads reais (não mocks)
Contratos de API Status codes, schema de resposta, headers Chamada real ao endpoint em ambiente de teste
Mensageria (fila / tópico) Publicação, consumo, ack/nack, ordering Broker em container; consumer real no teste
Cache Expiração, invalidação, fallback ao DB Redis/Memcached em container

Setup seguro e reprodutível

Testes de integração instáveis são piores que inúteis — geram alertas falsos e desconfiança no CI. O segredo está no controle do ambiente:

# Padrão recomendado: container + seed + cleanup

# 1. Levante dependências com Testcontainers ou docker-compose
db = PostgreSQLContainer("postgres:15")
db.start()

# 2. Aplique migração e seed determinístico
run_migrations(db.connection_url())
seed_test_data(db.connection_url(), fixtures="base_pequeno")

# 3. Execute o teste com a dependência real
resultado = UserRepository(db.connection_url()).find_active_users()
assert len(resultado) == 3

# 4. Cleanup após cada teste (ou use transação com rollback)
db.stop()  # ou rollback() para isolar sem re-criar

# Regras de ouro:
#   - Dados pequenos e explícitos (não "carregar o dump de produção")
#   - Cada teste isola seu estado (sem dependência de ordem de execução)
#   - Versão da dependência explícita (não "latest")
#   - Timeouts e retries controlados (não mascarar falhas reais)

Armadilhas comuns

Testes de integração mal escritos geram mais custo do que valor — e são abandonados rapidamente:

Ambiente compartilhado e mutável

Estado compartilhado entre testes causa falhas dependentes da ordem de execução. Cada teste deve criar e limpar seus próprios dados.

Testes lentos sem necessidade

Subir um banco para testar lógica de cálculo que seria coberta por unitário é desperdício. Use integração apenas para validar a dependência real.

Versão indefinida da dependência

Usar :latest nos containers gera resultados diferentes em máquinas diferentes. Fixe versões explicitamente.

Substituir tudo por integração

Resolver falta de confiança nos unitários com mais integração eleva o custo total da suíte sem ganho proporcional.

Para o nível acima da integração, veja testes de sistema (E2E). Para dublês de dependências, explore Mocks e Stubs.

"Integração instável é ruído — não é sinal de qualidade."
