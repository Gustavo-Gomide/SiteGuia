Testes Unitários

Regras, isolamento, estrutura AAA e armadilhas comuns

Testes unitários verificam o comportamento de uma função ou classe com dependências controladas. São o tipo mais rápido e barato — ideais para regras de negócio, validações e edge cases. Mas a eficácia depende de testar o comportamento externo, não os detalhes de implementação.

O que vale testar

Priorize o que tem lógica e pode quebrar com impacto real — não teste o que apenas delega para frameworks:

O que testar Por quê é importante Exemplos de casos
Regras de negócio e cálculos Erros aqui custam caro (financeiro, dados) Desconto 10% para pedido ≥ 100; arredondamento de moeda
Validações e normalizações Entrada inválida causa bugs silenciosos E-mail malformado, CPF inválido, string vazia
Fluxos com branching (if/else) Todo caminho pode ter comportamento diferente Usuário ativo vs. inativo; pedido com/sem frete grátis
Edge cases Os limites são onde os bugs vivem 0, negativo, máximo; virada de mês/ano; null
Permissões e autorizações Falha de segurança com impacto direto Admin pode deletar; guest não pode criar; owner vê só os seus

Estrutura AAA e nomes descritivos

Testes bem estruturados facilitam diagnóstico quando falham. O padrão AAA (Arrange / Act / Assert) é a base:

# Padrão AAA — estrutura recomendada

def deve_aplicar_desconto_10_porcento_quando_total_maior_igual_100():
    # Arrange — preparar dados e dependências
    carrinho = Carrinho(items=[Item("prod-1", preco=60), Item("prod-2", preco=50)])

    # Act — executar a unidade testada
    total = carrinho.calcular_total()

    # Assert — verificar comportamento
    assert total == 99.0  # 110 * 0.90

def deve_nao_aplicar_desconto_quando_total_menor_que_100():
    carrinho = Carrinho(items=[Item("prod-1", preco=30)])
    assert carrinho.calcular_total() == 30

# Modelo de nomes descritivos:
#   deve_[comportamento]_quando_[condição]
#   deve_rejeitar_email_invalido
#   deve_negar_acao_sem_permissao
#   deve_retornar_erro_quando_usuario_nao_encontrado

# Por que nomes importam:
#   Quando o teste quebra, o nome vira o diagnóstico imediato.
#   Evite: test1, testeCalcular, checkDesconto

Armadilhas comuns

A maioria dos problemas com testes unitários vem de acoplamento entre teste e implementação, ou isolamento insuficiente:

Acoplado à implementação

Testar métodos privados ou a estrutura interna faz o teste quebrar em toda refatoração — mesmo sem mudar comportamento. Teste a interface pública.

"Unitário" com I/O real

Testes que fazem chamadas ao banco, filesystem ou HTTP não são unitários — são de integração lentos. Controle dependências com stubs ou fakes.

Asserts fracos

assert result is not None não protege regra nenhuma. Valide o valor, o estado, ou o efeito exato esperado pelo comportamento.

Mock excessivo

Mockar tudo deixa o teste testando apenas a simulação, não o código real. Use o mínimo de dublê necessário para isolar a unidade.

Para isolamento com dublês de teste, veja Mocks e Stubs. Para a técnica que guia o design pelo teste, explore o TDD.

"Unitário ruim é pior que nenhum — cria confiança falsa e quebra a cada refatoração."
