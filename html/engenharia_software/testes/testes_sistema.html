Testes de Sistema (E2E)

Fluxos de usuário, suíte enxuta e combate à flakiness

Testes end-to-end validam fluxos críticos do usuário com o sistema rodando de verdade: UI ou API, banco de dados e integrações reais. São os mais caros de manter — por isso precisam ser poucos, bem escolhidos e extremamente estáveis.

O que E2E cobre bem

Reserve E2E para cenários onde a relevância do teste justifica o custo — onde a integração entre todas as camadas é o risco real:

Fluxo crítico Por que E2E é o nível certo Frequência ideal no CI
Login / autenticação Auth envolve UI, API, token, sessão e DB em conjunto A cada deploy
Checkout / pagamento Fluxo financeiro conecta múltiplos serviços A cada deploy
Cadastro de usuário Validações + persistência + envio de e-mail juntos A cada deploy
Permissões críticas Admin vê mais que guest — risco de segurança alto A cada deploy
Migrações de dados Após migration, fluxo principal ainda funciona? A cada release

Como desenhar uma suíte enxuta

E2E deve ser pequeno, rápido o suficiente para o CI e com dados completamente controlados:

# Princípios de uma suíte E2E saudável

# 1. Escolha apenas o que vale
    Mapeie os 5–10 fluxos mais críticos do produto
    Para tudo abaixo disso, prefira integração ou unitário

# 2. Dados controlados
    Cada teste cria os próprios dados via API ou seed
    Nunca dependa de dados deixados por outro teste
    Seeds explícitos e pequenos (não "carregar banco de produção")

# 3. Asserções de negócio, não de UI
    Valide: "pedido foi criado com status pendente"
    Evite: "o botão verde está visível em (x=230, y=450)"

# 4. Ambientes separados
    Staging isolado com dados sintéticos
    URLs, credenciais e integrações externas em modo sandbox

# 5. Timeouts explícitos
    Prefira espera por condição (elemento renderizado, request completo)
    Nunca use sleep() — é flakey por design

Armadilhas comuns em E2E

A maioria dos problemas com suítes E2E vem de escopo mal definido e ausência de isolamento de dados:

Suíte grande e lenta

Quando o E2E demora mais de 30 minutos, desenvolvedores passam a ignorar falhas ou desativar a suíte. Tamanho tem custo direto de confiança.

Quebras por mudanças visuais

Selectors baseados em classe CSS ou posição absoluta quebram a cada redesign. Prefira atributos de dados específicos para testes (data-testid).

Ambiente instável

Serviços externos não controlados (e-mail real, gateway de pagamento, terceiros) geram falhas esporádicas impossíveis de debugar.

E2E como cobertura principal

Substituir unitários e integração por E2E multiplica o custo de manutenção sem ganho proporcional em confiança nas regras de negócio.

Para o nível anterior (com dependências reais mas sem UI), veja testes de integração. Para validação por critérios de negócio, explore testes de aceitação.

"E2E que ninguém confia é pior que nenhum E2E."
