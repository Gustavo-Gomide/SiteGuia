Mocks e Stubs

Dublês de teste: tipos, quando usar e como evitar acoplamento

Dependências externas — banco de dados, HTTP, filas, relógio — tornam testes lentos e instáveis. Stubs, mocks e fakes permitem isolar a unidade testada sem depender da infra real. Mas usados em excesso, acoplam o teste à implementação e criam confiança falsa.

Tipos de dublê de teste

Cada tipo de dublê tem um papel específico. Escolher o errado torna o teste frágil ou inútil:

Dublê O que faz Quando usar Cuidado
Stub Retorna valores fixos pré-definidos Controlar cenário sem verificar interação Não valida se foi chamado ou com quais argumentos
Mock Retorna valores + verifica chamadas e argumentos Quando a interação em si é o comportamento a testar Excesso de mock acopla teste à implementação
Fake Implementação funcional simplificada Repositório em memória, relógio controlável, e-mail fake Precisa manter paridade de comportamento com o real
Spy Chama implementação real + registra chamadas Verificar comportamento sem substituir toda a dependência Introduz dependência na real — lento se o real faz I/O
Dummy Valor vazio passado para satisfazer assinatura Parâmetro obrigatório que não tem impacto no teste Não use se o valor pode impactar o comportamento

Padrão AAA com dublês

O padrão Arrange-Act-Assert organiza como configurar e usar dublês sem transformar o teste em código confuso:

# Exemplo: testar serviço de pedido com repositório stub

class PedidoRepositorioStub:
    """Fake — implementação em memória para teste"""
    def __init__(self):
        self._pedidos = []

    def salvar(self, pedido):
        self._pedidos.append(pedido)

    def buscar(self, pedido_id):
        return next((p for p in self._pedidos if p.id == pedido_id), None)

def deve_registrar_pedido_com_status_pendente():
    # ARRANGE — preparar dados e configurar dublê
    repositorio = PedidoRepositorioStub()
    servico = PedidoService(repositorio=repositorio)
    carrinho = Carrinho(items=[Item("prod-1", preco=50)])

    # ACT — executar comportamento
    pedido_id = servico.criar_pedido(carrinho, usuario_id="u-42")

    # ASSERT — validar resultado de negócio
    pedido = repositorio.buscar(pedido_id)
    assert pedido is not None
    assert pedido.status == "pendente"
    assert pedido.usuario_id == "u-42"

# Regras de uso:
#   - Assert no COMPORTAMENTO (status, valor, evento)
#   - NÃO no detalhe (quantas vezes salvar() foi chamado)
#   - Preferir Fake a Mock quando possível — menos frágil

Armadilhas comuns

Dublês mal usados geram uma suíte que passa no CI mas não detecta regressões reais:

Mock de tudo (over-mocking)

Quando cada dependência é mockada, o teste valida a simulação — não o código real. Qualquer mudança de implementação quebra o teste sem que o comportamento mude.

Mock de detalhes internos

Verificar que um método privado foi chamado 2 vezes acopla o teste à implementação. Qualquer refatoração quebrará o teste, mesmo sem mudar comportamento externo.

Esquecer serialização e timeouts

Stub de HTTP que retorna objeto já parseado esconde problemas reais de serialização, encoding e timeout. Use contrato real ou teste de integração para isso.

Não controlar tempo e aleatoriedade

Codigo que usa new Date() ou Math.random() sem injeção de dependência é impossível de testar de forma determinista — o teste fica flakey.

Para contexto sobre onde aplicar dublês, veja testes unitários. Para a técnica que usa testes para guiar o design, explore TDD.

"Mock o suficiente para isolar — não tanto que você esteja testando o mock."
