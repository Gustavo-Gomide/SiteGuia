Introdução a Testes de Software

Por que testar, pirâmide de testes e estratégia de cobertura

Testes existem para reduzir risco: impedindo regressões, permitindo refatoração segura e aumentando confiança em releases. O objetivo não é atingir "100% de coverage" — é cobrir o que pode quebrar e fazer mais diferença quando quebra.

Pirâmide de testes

A pirâmide de testes é uma heurística para equilibrar velocidade, custo e confiança. Não é uma lei — é um ponto de partida para pensar a estratégia:

Nível Velocidade Custo de manutenção Quando usar Exemplo
Unitário Muito rápido (ms) Baixo Regras, cálculos, validações, branching Desconto aplicado para pedido ≥ 100
Integração Médio (segundos) Médio DB, filas, serialização, contratos entre camadas Query retorna usuários ativos corretamente
Sistema / E2E Lento (min) Alto Jornadas críticas do usuário com sistema real Fluxo de checkout: login → carrinho → pagamento
Aceitação Variável Alto Critérios de aceite de stories e regras de negócio Reembolso validado dentro do prazo de 7 dias

Onde focar o esforço

Coverage alta em código irrelevante não agrega valor. Teste o que pode quebrar e custa caro quando quebra:

# Regra prática de onde investir

Alta prioridade (sempre testar):
  Regras de negócio: cálculos, descontos, limites, permissões
  Validações e transformações de dados
  Integrações externas (pagamento, notificações, auth)
  Edge cases: nulos, limites, datas, formatos inválidos

Média prioridade (testar em fluxos críticos):
  Camadas de controle (controllers, handlers)
  Serialização e contratos de API
  Migrações e mudanças de schema

Baixa prioridade (raramente vale o esforço):
  Código gerado automaticamente (ORM, scaffolding)
  Configurações triviais sem lógica
  Adaptadores que apenas delegam

# Sinais de estratégia saudável
  Testes rodam em < 2 min no CI           → unitários rápidos predominam
  Falhas apontam para causa real          → testes bem isolados
  Desenvolvedores rodam local antes do PR → feedback rápido

Boas práticas fundamentais

Independentemente do nível ou ferramenta, estas práticas determinam a qualidade da suíte de testes:

Nomes descrevem comportamento

deve_negar_acesso_usuario_sem_permissao é infinitamente melhor que teste1. O nome é documentação executável.

Determinismo acima de tudo

Controle de tempo, aleatoriedade e dependências externas. Teste flakey é pior que teste inexistente — cria desconfiança na suíte inteira.

Um motivo para falhar

Cada teste deve validar um comportamento específico. Múltiplos asserts em contextos diferentes dificultam diagnóstico quando o teste quebra.

Manutenção barata

Teste que quebra em toda refatoração (por estar acoplado à implementação) vira dívida. Teste o comportamento externo, não os detalhes internos.
