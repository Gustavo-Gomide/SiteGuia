Testes de Performance

Carga, stress, métricas de cauda e definição de SLOs

Testes de performance avaliam como o sistema se comporta sob carga: latência, throughput, consumo de recursos e limites. Sem baseline e sem SLO definido, qualquer resultado parece aceitável — até o incidente de produção provar o contrário.

Tipos de teste de performance

Cada tipo responde a uma pergunta diferente sobre o comportamento do sistema sob pressão:

Tipo Pergunta que responde Quando usar Ferramenta comum
Load O sistema aguenta a carga esperada de usuários? Antes de lançamentos, em cada release crítico k6, Locust, JMeter
Stress Qual é o limite real antes de degradar ou quebrar? Para definir capacidade e planejar infra k6, Gatling
Spike O sistema recupera após pico repentino de carga? Antes de eventos programados (Black Friday, campanha) k6, Locust
Soak O sistema degrada ao longo do tempo (memory leak, pool)? Antes de releases com suspeita de vazamento k6, Gatling
Baseline Como o sistema está hoje, sem carga extra? Sempre — referência para comparar regressões Qualquer ferramenta de load

Métricas e SLOs

Média de latência esconde experiências ruins. As métricas de cauda (p95, p99) revelam o que uma parcela real dos usuários vivencia:

# Métricas essenciais de performance

Latência (mais importante):
  p50: metade dos usuários espera até este valor
  p95: 95% dos usuários esperam até este valor (alvo mais comum de SLO)
  p99: 99% dos usuários esperam até este valor (fluxos críticos)

  Exemplo de SLO: "checkout deve responder em < 500ms p95"

Throughput:
  req/s ou transações/min que o sistema processa sem erro

Taxa de erro:
  % de requisições que falharam com 5xx ou timeout

Recursos:
  CPU%, memória, pool de conexões DB, lag de filas

# Por que p95/p99 importam mais que a média:
  Supondo 1000 req:
    900 respondem em 100ms  → média parece ótima
     90 respondem em 500ms
     10 respondem em 3000ms  → 1% dos usuários → 10 por minuto em escala

# Definindo SLO para um endpoint:
  1. Meça a baseline atual (sem carga)
  2. Execute load test com carga esperada
  3. Defina thresholds aceitáveis (ex.: p95 < 400ms, erro < 0.5%)
  4. Adicione ao CI para detectar regressões

Armadilhas comuns

A maioria dos problemas com testes de performance vem de ambiente inadequado ou métricas mal interpretadas:

Ambiente diferente de produção

Testar em máquina local ou ambiente sub-dimensionado gera resultados irreais. A infra precisa ser comparável à produção para o teste valer.

Sem baseline

Sem um ponto de referência anterior, não há como saber se uma release degradou a performance. Baseline é pré-requisito para qualquer análise.

Usar só a média

Média mascara caudas longas. Um sistema com p50=100ms e p99=5000ms faz 1% dos usuários sofrerem — invisível na média.

Testar tarde demais

Performance testada só em staging antes do launch geralmente resulta em arquitetura difícil de otimizar. Inclua no ciclo de desenvolvimento.

"Performance sem baseline é opinião. Com baseline, é dado."
