Especificação de Software

Transformando necessidades em descrição testável e implementável

Especificação é o processo de descrever o comportamento esperado do sistema com precisão suficiente para que alguém possa implementar e testar. Uma especificação bem escrita elimina ambiguidade, reduz retrabalho e serve como contrato entre desenvolvimento, negócio e QA.

Níveis de Especificação

Especificação não é tudo-ou-nada. Diferentes contextos exigem diferentes níveis de formalidade — o segredo é calibrar a profundidade de acordo com o risco e a volatilidade do requisito.

Nível Foco Conteúdo Típico Quando Usar
Visão / Contexto Objetivo e restrições de alto nível Problema a resolver, stakeholders, escopo, restrições legais Início do projeto; alinhamento estratégico
Funcional (Comportamental) O que o sistema deve fazer em cada cenário Fluxos, cenários BDD, regras de negócio, casos alternativos Antes do desenvolvimento; refinamento de backlog
Detalhe / Contrato Como o sistema se comporta em borda e falhas Validações, mensagens de erro, estados, integrações, SLAs Sistemas críticos; APIs externas; contratos de aceite
Operacional Como o sistema opera em produção Runbooks, SLOs, alertas, procedure de rollback Antes do deploy; para operação e suporte

Estrutura do SRS (IEEE 830)

O Software Requirements Specification (SRS) é o padrão IEEE 830 para documentar requisitos. Em ambientes ágeis, usa-se uma versão enxuta — mas o esqueleto é útil mesmo para documentos menos formais.

Estrutura SRS IEEE 830 (adaptada)
1. Introdução
   1.1 Propósito do documento
   1.2 Escopo do sistema
   1.3 Definições, acrônimos e abreviações (Glossário)
   1.4 Visão geral do documento

2. Descrição Geral
   2.1 Perspectiva do produto (contexto no ecossistema)
   2.2 Funções do produto
   2.3 Características dos usuários
   2.4 Restrições gerais (legais, técnicas, orçamento)
   2.5 Suposições e dependências

3. Requisitos Específicos
   3.1 Requisitos funcionais (por caso de uso ou feature)
     3.1.1 [RF001] Login com OAuth2
       - Descrição: ...
       - Pré-condição: ...
       - Fluxo principal: ...
       - Fluxos alternativos: ...
       - Critérios de aceitação: ...
   3.2 Requisitos não-funcionais
     - Performance: P95 < 300ms
     - Disponibilidade: 99.9% mensal
     - Segurança: autenticação 2FA para roles admin
   3.3 Restrições de design

4. Apêndices
   - Modelos de dados
   - Diagramas de fluxo

Rastreabilidade de Requisitos (RTM)

A Requirements Traceability Matrix (RTM) liga cada requisito à sua origem, aos casos de teste que o verificam e à funcionalidade que o implementa. Essencial para impacto de mudanças e auditorias.

Requirements Traceability Matrix (RTM)
ID       | Requisito                 | Origem        | Caso de Teste | Componente     | Status
---------|---------------------------|---------------|---------------|----------------|--------
RF-001   | Login OAuth2              | US-010, LGPD  | TC-021, TC-022| AuthService    | Done
RF-002   | Exportar CSV              | US-034        | TC-041..044   | ReportService  | Done
RF-003   | Autenticar 2FA admin      | SEC-007       | TC-060..063   | AuthService    | In Dev
RNF-001  | P95 < 300ms em /busca     | SLA contrato  | PERF-001      | SearchService  | Pending
RNF-002  | Disponib. 99.9% mensal    | SLA contrato  | SOAK-001      | Infraestrutura | Pending

Uso prático da RTM:
- Impacto de mudança: quais testes re-executar ao alterar RF-001?
- Cobertura: requisitos sem caso de teste (lacunas)
- Auditoria: origem de cada decisão de compliance

Antes de especificar, certifique-se de ter elicitado corretamente em Elicitação de Requisitos. Para comunicar visualmente as especificações, veja Diagrama de Casos de Uso.

"Uma boa especificação não é a mais longa — é aquela que elimina a interpretação."
