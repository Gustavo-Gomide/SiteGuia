Requisitos de Software

O que o sistema deve fazer — e o que não pode deixar de fazer

Requisito é qualquer condição ou capacidade que um sistema deve satisfazer. A diferença entre um projeto bem-sucedido e um projeto que entrega a coisa errada está, em grande parte, na qualidade do processo de requisitos: quanto mais tarde um requisito é descoberto ou corrigido, maior o custo.

Tipos de Requisitos

Os requisitos de software são tipicamente classificados em três categorias, cada uma com um papel diferente na especificação do sistema.

Tipo Definição Exemplos
Funcionais (RF) Descrevem o que o sistema deve fazer: comportamentos, funções e dados que o sistema manipula Gerar boleto; cadastrar cliente; autenticar via OAuth2; enviar e-mail de confirmação
Não-Funcionais (RNF) Definem como o sistema deve se comportar: restrições de qualidade, desempenho e conformidade P95 < 300ms; disponibilidade 99,9%; dados cifrados em repouso; compatível com LGPD
Regras de Negócio Políticas e restrições do domínio do problema, independentes da tecnologia Desconto de 10% para pedidos acima de R$500; prazo de pagamento de 30 dias; elegibilidade por CPF
Restrições Limitações impostas externamente (tecnologia, leis, prazos, orçamento) Deve rodar em AWS GovCloud; precisa de aprovação do setor jurídico; orçamento fixo

User Stories e Critérios de Aceitação

User stories são uma forma ágil de capturar requisitos do ponto de vista do usuário. Os critérios de aceitação em formato Gherkin (Given/When/Then) tornam os requisitos verificáveis e automatizáveis.

Formato Gherkin — Critérios de Aceitação
# User Story
Como gestor financeiro
Quero exportar o extrato mensal em CSV
Para reconciliar pagamentos no sistema contábil

# Criterios de Aceitacao (BDD)
Cenario: Exportacao bem-sucedida
  Given usuario autenticado com papel "gestor"
  And extrato do mes corrente possui pelo menos 1 lancamento
  When solicita exportacao CSV
  Then o sistema gera arquivo com colunas: data, descricao, valor, tipo
  And notifica via e-mail quando o arquivo estiver disponivel
  And o arquivo expira em 24h

Cenario: Acesso nao autorizado
  Given usuario autenticado com papel "operador"
  When tenta exportar extrato de outra empresa
  Then o sistema retorna HTTP 403
  And nenhum dado e exposto

Atributos de Qualidade de Requisitos

Requisitos mal escritos geram retrabalho, rejeição em testes e conflitos entre stakeholders. Os atributos abaixo definem o que significa um requisito ser "bom".

Atributo Definição Exemplo de Violação
Correto Representa fielmente a necessidade do stakeholder Capturar "calcular desconto" sem entender as regras de elegibilidade
Completo Cobre todos os cenários relevantes, incluindo de erro Descrever o fluxo feliz e ignorar o que acontece se o pagamento falhar
Não ambíguo Tem uma única interpretação possível "O sistema deve ser rápido" — o que é rápido?
Verificável É possível testar se o requisito foi atendido "Interface amigável ao usuário" — como medir?
Rastreável Pode ser ligado à sua origem e a casos de teste Requisito sem ID, sem origem (quem pediu?), sem caso de teste associado
Prioritizado Tem importância relativa definida (MoSCoW, valor de negócio) Lista de 200 requisitos todos marcados como "críticos"

Aprenda a elicitar requisitos de stakeholders em Elicitação de Requisitos e veja como formalizá-los em Especificação de Software.
