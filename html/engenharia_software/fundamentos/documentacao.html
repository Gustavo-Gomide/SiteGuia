Documentação de Software

Conhecimento que sobrevive à rotatividade e ao tempo

Documentação não é burocracia — é a forma mais eficiente de multiplicar o conhecimento de quem sabe para quem precisa saber. O objetivo não é cobrir tudo, mas ter o documento certo disponível no momento certo, reduzindo dependência de pessoas e prevenindo retrabalho.

Tipos de Documentação

Diferentes documentos servem audiências diferentes com propósitos diferentes. Tentar escrever um único documento que atenda todos é receita para um doc longo que ninguém lê.

Tipo Audiência Formato Exemplos
README Desenvolvedores entrando no projeto Markdown no repositório Como instalar, rodar, testar, buildar, debugar
Especificação / SRS Dev, QA, PO, stakeholders Wiki, Confluence, Notion, Markdown Requisitos, critérios de aceite, fluxos, cenários
ADR (Architecture Decision Record) Dev, arquitetos, futuros mantenedores Markdown no repositório Decisão arquitetural, contexto, alternativas, consequências
API Reference Desenvolvedores consumidores OpenAPI / Swagger, Redoc Endpoints, payloads, autenticação, erros, exemplos
Runbook / Playbook Operação, SRE, suporte Wiki interna, Confluence Como lidar com alertas, incidentes, deploys, rollbacks
Diagramas Qualquer audiência técnica Mermaid, PlantUML, draw.io Arquitetura C4, fluxo de dados, diagrama de sequência

ADRs — Architecture Decision Records

ADRs registram decisões arquiteturais importantes junto ao código. Quando alguém perguntar "por que usamos X em vez de Y?", a resposta deve estar num ADR — não na memória de quem já saiu do time.

ADR-001 — Uso de PostgreSQL em vez de MongoDB
# ADR-001: Uso de PostgreSQL em vez de MongoDB

## Status
Aceito

## Contexto
O sistema precisa de transacoes ACID para gerenciar pagamentos e transferencias.
Estudamos MongoDB pelo modelo de documento e escalabilidade horizontal.

## Decisao
Adotamos PostgreSQL como banco de dados primario.

## Justificativa
- Transacoes distribuidas com ACID sao requisito nao-negociavel (compliance PCI)
- Esquema relativamente estavel; ganhos de schema-less nao compensam
- Time tem mais expertise em SQL e tooling (migrations, backups)
- JSONB no Postgres cobre casos de dados semi-estruturados

## Alternativas Consideradas
- MongoDB: descartado pela falta de transacoes multi-documento robustas na epoca
- CockroachDB: complexidade operacional excessiva para o tamanho do time

## Consequencias
- Escalabilidade vertical prioritaria (sharding requer planejamento futuro)
- Migrações precisam de plano de rollback
- Rever essa decisao se o volume de dados semi-estruturados crescer > 30% do total

Documentação como Código

Documentar como código significa que docs vivem no repositório, passam por revisão (PR), têm histórico (git log) e estão sempre próximos do código que descrevem. Docs fora do repo ficam desatualizadas.

Docs no Repositório

README, ADRs e runbooks vivem em /docs no repositório. Parte do Definition of Done: toda PR que muda comportamento deve atualizar a doc correspondente.

OpenAPI / AsyncAPI

APIs documentadas em YAML/JSON geradas ou validadas automaticamente no pipeline. Consumers têm uma fonte única de verdade. Evita divergência entre código e documentação.

Diagramas como Código

Mermaid, PlantUML e D2 geram diagramas a partir de texto. Sem ferramenta proprietária, diff no PR mostra o que mudou na arquitetura, e a doc fica junto do código.

Testes como Documentação

Testes BDD (Cucumber, Behave) e testes de integração bem nomeados documentam o comportamento esperado do sistema. "O teste falhou" = "a doc mudou".

Aprenda a escrever especificações testáveis em Especificação de Software e veja boas práticas de documentação de qualidade em Requisitos de Software.

"Documentação desatualizada é pior do que nenhuma documentação — mentiras bem escritas são perigosas."
