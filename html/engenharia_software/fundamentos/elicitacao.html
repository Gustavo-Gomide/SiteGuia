Elicitação de Requisitos

Como descobrir o que realmente precisa ser construído

Elicitar requisitos não é simplesmente perguntar "o que você quer". É um processo ativo de descoberta que envolve entender o contexto, o domínio, as restrições e os objetivos de negócio — muitas vezes extraindo conhecimento tácito que o próprio stakeholder não sabe articular.

Técnicas de Elicitação

Cada técnica tem contextos ideais de aplicação. Times maduros combinam múltiplas técnicas ao longo do processo, usando dados quantitativos para validar hipóteses qualitativas.

Técnica Quando Usar Vantagens Limitações
Entrevistas Para aprofundar regras de negócio, exceções e contexto de uso Rica em detalhes; revela motivações implícitas Consome tempo; depende da disponibilidade do stakeholder
Workshops (JAD) Para alinhar múltiplas áreas e resolver conflitos de requisitos Convergência rápida; todos em sincronia Exige facilitador experiente; difícil de escalar para muitas pessoas
Observação / Etnografia Para sistemas que substituem processos manuais Captura o trabalho real (não o "oficial"); revela exceções não documentadas Invasiva; demora; pode alterar o comportamento observado
Prototipação Quando os requisitos são vagos ou difíceis de articular Reduz ambiguidade com algo tangível; gera feedback concreto Risco de o protótipo ser confundido com o produto final
Análise de Dados Para refinar prioridades com evidência quantitativa Objetivo; valida hipóteses; revela padrões de uso real Requer dados existentes e interpretação cuidadosa
Story Mapping Para construir visão de produto e priorizar incrementos Visual; facilita corte de MVP; alinha times Pode desviar para detalhes de UX antes de validar premissas

Desafios Comuns

Requisitos raramente chegam bem formados. Os problemas a seguir são a norma, não a exceção — reconhecê-los cedo determina se o projeto terá soluções sólidas ou retrabalho caro.

Conhecimento Tácito

Stakeholders conhecem seus processos mas não sabem descrevê-los. Use observação, cenários e perguntas "e se?" para trazer à tona o que é óbvio para quem trabalha na área.

Conflitos de Interesse

Áreas diferentes têm prioridades diferentes. Facilite workshops com representação de todas as partes e faça trade-offs explícitos. Requisitos sem prioridade clara são fonte de conflito em produção.

Volatilidade de Requisitos

Negócio muda. Design para mudança: prefira requisitos rastreáveis, critérios de aceite verificáveis e arquitetura que isola pontos de variação. Documente hipóteses para saber o que validar.

Gap Domínio/Tecnologia

Engenheiros não entendem o domínio; especialistas não entendem limitações técnicas. Invista em glossário compartilhado (Ubiquitous Language do DDD) e sessões de refinamento mistas.

Mapa de Stakeholders

Antes de iniciar a elicitação, identifique quem são os stakeholders e como gerenciá-los. A matriz Poder × Interesse é um ponto de partida clássico e eficaz.

Matriz Poder x Interesse
                    INTERESSE
                 Baixo  |  Alto
             ___________|___________
    Alto     |  Manter  | Gerenciar |
PODER        |  Satisf. | de Perto  |
             |__________|___________|
    Baixo    | Monitorar| Manter    |
             |          | Informado |
             |__________|___________|

Exemplos:
- Manter Satisfeito  : C-level com alto poder mas não usa o sistema
- Gerenciar de Perto : Product Owner, usuários-chave, compliance
- Manter Informado   : time de suporte, clientes externos
- Monitorar          : stakeholders periféricos

Boas práticas:
- Revise o mapa a cada ciclo (stakeholders mudam)
- Identifique possiveis defensores e resistentes
- Para cada grupo: frequencia e formato de comunicacao adequados

Entenda os tipos de requisitos antes de elicitá-los em Requisitos de Software e aprenda a formalizá-los em Especificação de Software.
