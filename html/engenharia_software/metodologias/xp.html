XP (Extreme Programming)

Práticas técnicas para aumentar qualidade e absorver mudança

XP é um conjunto de práticas de engenharia para software de alta qualidade com alta capacidade de mudança. Diferente de Scrum (que foca em processo), XP foca em práticas técnicas — TDD, CI, pair programming, refatoração e small releases.

Práticas técnicas centrais

Cada prática do XP atua sobre um problema técnico específico. Elas se potencializam quando combinadas.

PráticaProblema que resolveResultado esperado
TDDCódigo sem cobertura de testes, medo de refatorarDesign mais simples, regressão detectada cedo
Continuous IntegrationIntegração tardia, conflitos grandesBuild sempre verde, feedback em minutos
Pair ProgrammingSiloing de conhecimento, revisão tardiaRevisão contínua, difusão de conhecimento
RefatoraçãoDívida técnica acumulada, complexidade crescenteCódigo simples e adaptável ao longo do tempo
Small ReleasesFeedback tardio, risco de integraçãoValidação frequente, correção rápida
Collective Code OwnershipDependência de pessoas-chaveQualquer dev pode melhorar qualquer parte

Ciclo TDD — Red / Green / Refactor

TDD inverte a ordem: você escreve o teste antes do código. Isso força clareza sobre o comportamento esperado antes de implementar.

1. RED — escrever o teste que FALHA
   test("soma 2 + 3 = 5", () => {
     expect(soma(2, 3)).toBe(5)
   })
   -- teste quebra porque soma() ainda nao existe

2. GREEN — escrever o MÍNIMO para passar
   function soma(a, b) { return a + b }
   -- apenas o suficiente para o teste passar

3. REFACTOR — melhorar sem quebrar
   -- renomear, extrair, simplificar
   -- todos os testes devem continuar verdes

Repita! Ciclo curto (minutos, nao horas)

Quando usar — e quanto usar

XP tem overhead de aprendizado. Aplique as práticas onde o ROI é maior para o contexto do time.

Alta mudança de requisitos

XP é ideal quando o produto evolui rápido e o custo de regressão é alto. TDD + CI dão confiança para mudar.

Base de código legada

Pair programming e refatoração constante são mais valiosos em código com alta dívida técnica.

Pair não precisa ser constante

Pair em tudo tem custo cognitivo. Use onde dá mais valor: código crítico, onboarding, decisões de design.

Sem direção, XP não basta

XP precisa de backlog e PO. Disciplina técnica sem produto bem definido é eficiência na direção errada.

XP complementa: Scrum (processo) e Agile (valores).

"TDD não é sobre testes — é sobre design. Código testável é código bem estruturado."
