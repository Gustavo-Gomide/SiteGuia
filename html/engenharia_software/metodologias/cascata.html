Modelo Cascata

Fases sequenciais, escopo fixo e feedback no final

Cascata organiza o trabalho em fases sequenciais e lineares. É útil quando requisitos são estáveis, o custo de mudança é baixo e a validação formal é obrigatória — não quando o produto evolui continuamente.

Fases e artefatos

Cada fase deve estar completa e aprovada antes de a próxima começar. O artefato de saída de uma fase é a entrada da próxima.

FaseArtefato de entradaArtefato de saída
1. RequisitosNecessidade de negócioSRS / documento de requisitos
2. DesignSRS aprovadoDocumento de arquitetura / HLD
3. ImplementaçãoDesign aprovadoCódigo-fonte
4. VerificaçãoCódigo-fonteRelatório de testes / defeitos
5. ManutençãoSistema em produçãoPatches e atualizações

Quando usar e quando não usar

Cascata não é obsoleto — é inadequado para muitos contextos modernos. Saber diferenciá-los evita adoção por descuido ou rejeição por modismo.

USE cascata quando:
  - requisitos sao estáveis e bem definidos
  - ambiente regulado exige documentacao formal completa
  - contrato fixo com escopo fechado
  - sistemas embarcados ou hardware com mudanca cara

NAO USE cascata quando:
  - requisitos tendem a mudar ao longo do projeto
  - usuario precisa ver o produto cedo para validar
  - time pequeno com ciclos curtos de entrega
  - produto ainda nao tem product-market fit

Riscos e sinais de alerta

O risco central de Cascata é o feedback tardio: erros de requisito ou design só aparecem na fase de teste ou entrega.

Custo de mudança tardio

Mudar um requisito na fase de teste custa 10–100× mais que mudar na fase de requisitos.

Usuário vê produto só no final

Sem validação intermediária, o produto pode estar tecnicamente correto mas funcionalmente errado.

Requisitos "congelados"

Requisitos nunca são estáveis de verdade. Congelar cedo pressupõe conhecimento que não existe.

Integração tardia

Integrar componentes só no final revela incompatibilidades que custam caro para resolver.

Compare com abordagens incrementais: Iterativo e Incremental e Agile.

"Cascata funciona melhor quando você sabe exatamente o que vai construir antes de começar. Isso raramente é verdade em software de produto."
