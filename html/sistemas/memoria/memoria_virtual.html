Sistemas Operacionais

Memória Virtual

A ilusão de memória infinita — executando programas maiores que a RAM física com total isolamento

Memória virtual permite executar programas maiores que a RAM física disponível, isolando processos com espaços de endereçamento completamente independentes. Cada processo tem a ilusão de possuir toda a memória para si, enquanto o kernel gerencia silenciosamente o que realmente reside na RAM e o que está no disco.

Demand Paging

Demand paging (paginação sob demanda) é o mecanismo pelo qual páginas só são carregadas na RAM quando efetivamente referenciadas. Quando um processo é criado, o kernel mapeia seu espaço virtual mas não carrega nada — as páginas são carregadas conforme o programa as acessa.

Page Fault Minor

A página existe (está no page cache ou foi previamente alocada) mas não está mapeada no page table do processo. O kernel apenas atualiza o mapeamento — sem I/O de disco. Custo: ~100 ns.

Page Fault Major

A página precisa ser carregada do disco (swap ou arquivo). Requer I/O real — o processo fica bloqueado enquanto espera. Custo: ~5-15 ms (HDD) ou ~100 µs (NVMe).

Fluxo do Page Fault Handler (pseudocódigo do kernel)
// Chamado quando CPU gera exceção de page fault
void do_page_fault(struct pt_regs *regs, unsigned long error_code) {
    address = read_cr2();          // CR2 contém o endereço que causou o fault

    vma = find_vma(mm, address);   // Busca VMA (Virtual Memory Area) que contém o endereço
    if (!vma || address < vma->vm_start) {
        send_signal(SIGSEGV);      // Endereço inválido → segmentation fault
        return;
    }

    if (error_code & WRITE && !(vma->vm_flags & VM_WRITE)) {
        send_signal(SIGSEGV);      // Tentativa de escrita em área somente leitura
        return;
    }

    // Página válida — determinar onde ela está
    if (page_in_swap(address)) {
        page = swap_in(address);   // Major fault: carrega do swap (I/O)
    } else if (page_in_cache(address)) {
        page = map_from_cache(address); // Minor fault: apenas atualiza mapeamento
    } else {
        page = alloc_zero_page();  // Primeira vez: aloca página zerada (lazy alloc)
    }

    update_page_table(mm, address, page);  // Instala PTE
    return;                        // CPU re-executa a instrução que causou o fault
}

Swap

O swap é um espaço em disco reservado para extensão virtual da RAM. Quando a RAM física fica cheia, o kernel move páginas inativas para o swap, liberando RAM para dados mais urgentes. No Linux, pode ser uma partição dedicada ou um arquivo regular.

swappiness

Parâmetro do kernel (0-200) que controla agressividade de swap. Valor 0 = evitar swap ao máximo (preferir dropar page cache); 100 = comportamento equilibrado; 200 = swapear agressivamente. Default Linux: 60.

zswap

Cache comprimido em RAM para páginas destinadas ao swap. Páginas são comprimidas (LZ4/zstd) antes de ir ao disco. Reduz I/O de swap até 80% para cargas compressíveis. Ativado via /sys/module/zswap/parameters/enabled.

zram

Dispositivo de bloco comprimido inteiramente em RAM. Usado como swap para sistemas com pouca RAM (Chromebooks, Android). Evita totalmente o I/O de disco ao custo de CPU para compressão/descompressão.

Quando O Swap Vira Problema

Swap em HDD com latência de 10 ms: se o kernel swap-in/swap-out continuamente (thrashing), o sistema torna-se irresponsivo. Detectável via vmstat 1 vendo campos si e so constantemente altos.

Working Set e Thrashing

O working set de um processo é o conjunto de páginas que ele usa ativamente em uma janela de tempo. É a quantidade mínima de RAM necessária para o processo progredir sem gerar page faults continuamente.

Thrashing ocorre quando a RAM total é menor que a soma dos working sets de todos os processos em execução. O SO fica em um ciclo vicioso: página A entra → página B sai → processo que precisava de B gera page fault → B entra → A sai → e assim por diante. A CPU fica quase ociosa enquanto o disco opera no limite.

Copy-on-Write (CoW)

Copy-on-Write é uma otimização crucial onde páginas de memória são compartilhadas entre processo pai e filho após um fork(). Nenhuma cópia real é feita inicialmente — ambos referenciam as mesmas páginas físicas, marcadas como somente leitura.

Quando qualquer um dos processos escreve em uma página compartilhada, a MMU gera uma exceção de proteção. O kernel então duplica apenas essa página específica, atualiza os mapeamentos e retoma a execução. Páginas nunca escritas nunca são copiadas.

fork() com CoW

Um processo com 1 GB de dados pode ser "copiado" via fork() em microssegundos. Apenas as estruturas de controle (page tables, task_struct) são realmente duplicadas. Páginas físicas são compartilhadas até o primeiro write.

Redis RDB Snapshots

Redis usa fork() + CoW para salvar snapshots sem bloquear operações de escrita. O processo filho lê os dados do snapshot enquanto o pai continua aceitando escritas que copiam apenas as páginas modificadas.

CoW em Linguagens

PHP e Python usam CoW para módulos importados — o interpretador compartilha páginas de código entre workers. Ruby on Rails com Unicorn usa o mesmo princípio: um processo master que faz fork para workers que compartilham código da aplicação.

Custo do CoW

Cada page fault de CoW custa ~1 µs (minor fault). Se um processo escreve em muitas páginas rapidamente após o fork (alta taxa de "dirtying"), o benefício do CoW diminui. Monitorável via /proc/PID/stat campo minflt.
