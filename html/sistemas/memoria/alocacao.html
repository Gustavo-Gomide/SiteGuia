Sistemas Operacionais

Alocação de Memória

Estratégias para distribuir memória eficientemente — do kernel ao heap de aplicação

Alocar memória eficientemente é crítico para o desempenho de qualquer sistema. Fragmentação desperdiça RAM disponível, e lentidão no alocador impacta a performance de toda a aplicação — um alocador ruim pode ser responsável por 10-30% do tempo de execução em aplicações multi-thread intensivas.

Estratégias de Alocação Contígua

Antes da paginação generalizada, sistemas operacionais gerenciavam regiões contíguas de RAM livre usando estratégias simples de seleção de blocos:

Estratégia Algoritmo de Seleção Velocidade Fragmentação Resultante Uso Prático
First Fit Primeiro bloco livre suficientemente grande Rápida (para na primeira opção) Fragmentação no início da lista Alocadores simples, slab allocator interno
Next Fit Como First Fit mas começa de onde parou Rápida e mais uniforme Distribuída pelo heap Variante usada em alguns sistemas embarcados
Best Fit Menor bloco que satisfaz o pedido Lenta (varre toda a lista) Muitos fragmentos minúsculos inutilizáveis Raramente — fragmentação pior que First Fit
Worst Fit Maior bloco disponível Lenta (varre toda a lista) Destrói blocos grandes, poucos blocos médios sobram Quase nunca — geralmente inferior aos outros

Buddy System

O Buddy System é usado pelo kernel Linux para alocar páginas físicas contíguas. Divide a memória em blocos de tamanhos que são potências de 2 (1, 2, 4, 8, 16... páginas). É eficiente em alocação e coalescência — ao liberar, blocos irmãos ("buddies") adjacentes são fundidos automaticamente.

Buddy System — Split e Merge de Blocos
Estado inicial: um bloco de 16 páginas
[████████████████]  ← 16 páginas livres

Alocação de 3 páginas → bloco mínimo = 4 páginas (próxima potência de 2)
Processo de split:
  [████████████████] → [████████] [████████]  (split de 16 em dois de 8)
  [████████] [████████] → [████] [████] [████████]  (split de 8 em dois de 4)
  [████] = alocado   [████] = livre   [████████] = livre

Estado: [ALOCADO-4][LIVRE-4][LIVRE-8]

Alocação de 1 página → bloco mínimo = 1 página
  [LIVRE-4] → [LIVRE-2][LIVRE-2] → [LIVRE-1][LIVRE-1]
  → [ALOCADO-1][LIVRE-1][LIVRE-2][LIVRE-8]

Liberação do bloco de 4 páginas:
  Verifica se "buddy" (bloco irmão de mesmo tamanho) está livre
  Buddy (segundo bloco de 4) está LIVRE → merge!
  → [LIVRE-8][LIVRE-1][LIVRE-2]  (buddies de 4 fundiram em 8)

Liberação do bloco de 1 página:
  Buddy (segundo bloco de 1) está livre → merge!
  → [LIVRE-8][LIVRE-2]  (buddies de 1 fundiram em 2)
  Buddy (segundo bloco de 2) está livre → merge!
  → [LIVRE-8][LIVRE-4]  ... e assim por diante até restaurar

Complexidade: O(log N) para alocação e liberação

O kernel Linux expõe o estado do buddy system via /proc/buddyinfo, mostrando quantos blocos livres existem para cada ordem (tamanho em potência de 2 de páginas).

malloc() — Alocação em Espaço de Usuário

A função malloc() da biblioteca C (glibc) é o principal alocador de heap para programas de usuário. Internamente usa sbrk() para expandir o segmento de dados (alocações pequenas) e mmap() para alocações grandes (acima de ~128 KB no glibc).

Fastbins

Cache de chunks pequenos (16-80 bytes) recém-liberados. Alocação e liberação em O(1) — não une com vizinhos. Acelera padrões onde objetos pequenos são criados e destruídos frequentemente.

Smallbins e Largebins

Listas duplamente encadeadas organizadas por tamanho. Smallbins para objetos até ~512 bytes; largebins para objetos maiores com busca por best-fit aproximado. Chunks são unidos ao liberar (coalescing).

Arena Multi-thread

glibc mantém arenas separadas por thread para reduzir contenção de lock. Cada arena gerencia seu próprio heap com bins independentes — evita que threads fiquem bloqueadas esperando o lock global.

Double-free e Use-after-free

Chamar free() duas vezes ou usar ponteiro após freed corrompe silenciosamente os metadados do alocador (size, fd/bk pointers nos bins). Pode tornar-se vulnerabilidade de segurança exploitável via heap overflow.

Detectando Vazamentos com Valgrind
// Código C com memory leak
#include <stdlib.h>
int main() {
    char *buf = malloc(1024);  // Alocado mas nunca liberado
    buf[0] = 'A';
    return 0;  // free(buf) ausente — leak de 1024 bytes
}

// Compilar e executar com Valgrind
$ gcc -g -o programa programa.c
$ valgrind --leak-check=full ./programa

Alocadores Modernos

Para aplicações de alta performance, substituir o alocador padrão (glibc ptmalloc2) por um alocador especializado pode trazer ganhos significativos em throughput e redução de fragmentação:

Alocador Criador / Usado em Thread Safety Fragmentação Throughput Multi-thread Destaque
ptmalloc2 GNU / glibc (padrão Linux) Arenas por thread Média Médio Ubíquo, compatível, mas não ideal para alto paralelismo
jemalloc Facebook / Firefox, FreeBSD, Redis Excelente (per-CPU caches) Baixa Alto Profiling embutido, controle fino de fragmentação, usado em produção pesada
tcmalloc Google / Chrome, Abseil Excelente (thread-local caches) Baixa a média Muito alto Thread-local caches eliminam lock na maioria das alocações, heap profiler integrado
mimalloc Microsoft Research Excelente Muito baixa Muito alto Segurança aprimorada, randomização de heap, desempenho competitivo com tcmalloc
snmalloc Microsoft Research Lock-free Baixa Excepcional Arquitetura message-passing para devoluções entre threads — zero sync no caminho comum

Substituir ptmalloc2 por jemalloc ou tcmalloc em aplicações multi-thread intensivas (servidores web, bancos de dados) frequentemente resulta em ganhos de 10-30% em throughput devido à melhor gestão de arenas e menor contenção de locks.
