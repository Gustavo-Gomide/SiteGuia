Sistemas Operacionais

Page Fault

A interrupção que viabiliza a memória virtual — o mecanismo central por trás de toda paginação

Page fault é uma interrupção de hardware gerada pela MMU quando a CPU tenta acessar uma página que não está presente na RAM. Longe de ser apenas um erro, é o mecanismo central que viabiliza toda a memória virtual — é por causa dos page faults que processos podem usar mais memória do que existe fisicamente e que o Copy-on-Write do fork() funciona.

Tipos de Page Fault

Tipo Causa I/O de Disco Custo Típico Resultado
Minor Fault (Soft) Página no page cache ou alocada mas não mapeada (lazy alloc, CoW, mmap) Nenhum ~100-500 ns Kernel atualiza PTE e retoma execução
Major Fault (Hard) Página no swap ou em arquivo não carregado — requer leitura de disco Sim — bloqueia processo ~5-15 ms (HDD), ~100-500 µs (NVMe) Kernel carrega página, atualiza PTE, retoma
Invalid Fault (Segfault) Acesso a endereço fora de qualquer VMA válida, ou violação de proteção (escrita em R/O) Nenhum Instantâneo Kernel envia SIGSEGV — processo termina com core dump

Fluxo de Tratamento do Page Fault

O tratamento de um page fault envolve cooperação entre hardware (MMU, CPU) e software (kernel). O processo ocorre em microssegundos para minor faults, mas pode levar milissegundos para major faults:

Page Fault Handler — Fluxo Detalhado (x86-64 Linux)
Passo 1 — Detecção pelo Hardware:
  CPU acessa endereço virtual → MMU verifica TLB → TLB miss
  MMU verifica page table → bit Present = 0 (ou violação de proteção)
  CPU salva registradores, escreve endereço faltante em CR2
  CPU chama interrupt vector 14 (page fault exception)

Passo 2 — Entrada no Kernel:
  CPU entra em modo kernel (ring 0)
  Handler: exc_page_fault() → do_user_addr_fault()
  Lê endereço de CR2 e error_code (read/write, user/kernel, present)

Passo 3 — Verificação da VMA:
  mm = current->mm                      // Estrutura de memória do processo
  vma = find_vma(mm, fault_address)     // Busca VMA que contém o endereço
  Se não existe VMA → SIGSEGV (acesso inválido)
  Se escrita em VMA somente leitura → SIGSEGV (proteção)

Passo 4 — Resolução (caso válido):
  Se CoW (write em página shared R/O):
    page = alloc_page()                  // Aloca nova página
    copy_page(page, old_page)            // Copia conteúdo
    atualiza PTE com nova página R/W
  Se página no swap:
    swap_readpage()                      // Lê do disco (major fault — bloqueia)
    atualiza PTE com frame carregado
  Se lazy alloc (primeira vez):
    page = alloc_zeroed_page()           // Página zerada (segurança — nunca lixo)
    atualiza PTE

Passo 5 — Retorno:
  CPU re-executa a instrução que causou o fault
  Desta vez a página está presente → sucesso transparente

Causas Comuns e Diagnóstico

Lazy Allocation

malloc() retorna um ponteiro sem alocar memória real. A primeira escrita em cada página gera um minor fault. Razão pela qual malloc(1 GB) retorna instantaneamente mas o uso gradual leva tempo.

fork() e CoW

Após fork(), cada write em qualquer página compartilhada gera um minor fault de CoW. Aplicações que escrevem em muitas páginas logo após fork (ex: Redis com muitas keys) têm alto minflt.

mmap() de Arquivo

Arquivos mapeados em memória geram major faults na primeira leitura de cada página (carregamento do disco). Leituras subsequentes são minor faults (página no page cache).

Stack Crescimento

A stack cresce automaticamente. O kernel mapeia somente a região inicial; cada página nova de stack gera um minor fault. O limite de crescimento é controlado por ulimit -s (default 8 MB no Linux).

Medir Page Faults com perf e /proc
# Contar page faults durante execução de um programa
$ perf stat -e page-faults,minor-faults,major-faults ./meu_programa

Otimização para Reduzir Page Faults

Huge Pages (2 MB)

Reduz o número de page faults em até 512× para alocações grandes (1 huge page cobre o que 512 páginas normais cobriam). Fundamental para bancos de dados e aplicações com grandes buffer pools.

mlock() e mlockall()

Trava páginas na RAM impedindo que sejam enviadas ao swap. Gera todos os page faults imediatamente ao chamar (prefault). Essencial para aplicações de tempo real e baixa latência.

madvise(MADV_WILLNEED)

Dica ao kernel de que certas páginas serão usadas em breve. O kernel faz prefetch assíncrono, reduzindo major faults quando a aplicação realmente acessar as páginas.

Transparent Huge Pages (THP)

O kernel automaticamente promove regiões de memória anonima a huge pages quando possível. Habilitado por padrão no Linux. Pode ser configurado por processo via madvise(MADV_HUGEPAGE).

Prefault e mlock em C
#include <sys/mman.h>
#include <string.h>
#include <stdlib.h>

// Alocar e travar 256 MB na RAM (sem page faults futuros)
void* buf = malloc(256 * 1024 * 1024);

// Prefault: toca cada página para forçar alocação agora
memset(buf, 0, 256 * 1024 * 1024);

// Trava na RAM — impossível ser swapado
mlock(buf, 256 * 1024 * 1024);

// Alternativa: travar TODA memória do processo (aplicações RTOS)
mlockall(MCL_CURRENT | MCL_FUTURE);

// Dica ao kernel: páginas serão necessárias em breve
madvise(arquivo_mapeado, tamanho, MADV_WILLNEED);
