Fragmentação de Memória

Como a memória disponível torna-se inutilizável — causas, consequências e soluções

Fragmentação desperdiça memória disponível tornando-a inacessível. Há dois tipos fundamentais: fragmentação externa, onde existem blocos livres suficientes no total mas nenhum bloco contíguo é grande o suficiente para atender uma requisição; e fragmentação interna, onde espaço é desperdiçado dentro de uma alocação porque o bloco fornecido é maior que o necessário.

Fragmentação Interna vs Externa

Critério Fragmentação Interna Fragmentação Externa
Definição Espaço desperdiçado dentro de um bloco alocado Espaço livre total suficiente mas não contíguo para atender a requisição
Causa Principal Alocação em blocos de tamanho fixo ou alinhados por potência de 2 Alocações e liberações frequentes de tamanhos variáveis deixam "buracos"
Exemplo Concreto Buddy system: requisição de 33 KB aloca bloco de 64 KB — 31 KB perdidos Alocação variável: livre 10+10+10 KB = 30 KB total, mas requisição de 20 KB falha
Como Medir (bloco_alocado - bytes_requisitados) / bloco_alocado Maior bloco livre / total de memória livre
Impacto Desperdício previsível e limitado (no máximo 50% no buddy system) Pode tornar sistema incapaz de alocar memória mesmo com RAM "livre"
Solução Principal Slab/SLUB allocator para objetos de tamanho fixo Paginação (elimina fragmentação externa), compactação (custosa)

Compactação

Compactação resolve a fragmentação externa realocando processos ou blocos na memória para reunir os espaços livres dispersos em um único bloco contíguo. É a abordagem mais direta — mas de custo altíssimo.

Custo da Compactação

Copiar gigabytes de dados na RAM leva dezenas de milissegundos a segundos. Durante a compactação, todos os endereços de ponteiros devem ser atualizados — impossível em C/C++ com ponteiros diretos sem reescrever todo o código.

Compactação em GC

Java (G1GC, ZGC), Go, e .NET usam compactação durante a coleta de lixo. O GC controla todos os ponteiros, podendo atualizá-los atomicamente durante a compactação. ZGC compacta concorrentemente sem pausar a aplicação.

Compactação em Discos

Sistemas de arquivos como ext4, Btrfs, e ZFS enfrentam fragmentação em disco. O e4defrag compacta arquivos fragmentados inativos. SSDs sofrem menos com fragmentação mas beneficiam do TRIM para reorganização interna.

Por Que Paginação Resolve

Paginação elimina fragmentação externa para alocações baseadas em páginas: não importa onde os frames estão na RAM física — o mapeamento virtual cria a ilusão de contiguidade. Sobra apenas a fragmentação interna da última página.

Slab Allocator

O Slab Allocator é a solução definitiva para fragmentação de objetos do kernel. Em vez de alocar e liberar individualmente cada inode, PCB (task_struct) ou dentry, o slab mantém caches de objetos pré-alocados e pré-inicializados.

Princípio de Funcionamento

Um slab é um conjunto de uma ou mais páginas contendo objetos do mesmo tipo. Objetos liberados voltam ao cache em estado já construído (sem zerar memória). Próxima alocação é O(1) — sem inicialização, sem fragmentação.

Objetos Comuns no Kernel

task_struct (PCB de cada processo), inode, dentry (cache de nomes de arquivos), socket, file, pipe — todos gerenciados pelo slab. Em um sistema com milhares de processos e arquivos, o benefício é enorme.

SLUB — Padrão no Linux

SLUB (Simplified SLUB) é o allocator padrão do Linux desde 2.6.23. Metadados inline no objeto, sem arrays separados de descritores. Mais simples, melhor debugging via CONFIG_SLUB_DEBUG. Detecta corrupção de objetos automaticamente.

SLOB — Para Embarcados

SLOB (Simple List Of Blocks) é a versão minimalista para sistemas com pouquíssima RAM (roteadores, microcontroladores). Menor footprint de código e memória às custas de performance — aceita mais fragmentação.

Monitoramento e Diagnóstico

O Linux expõe informações detalhadas sobre fragmentação via /proc. Monitorar esses arquivos é essencial em sistemas de longa duração ou com alta alocação dinâmica:

/proc/buddyinfo — Fragmentação de Páginas por Ordem
$ cat /proc/buddyinfo
Node 0, zone      DMA      1      0      1      0      2      1      1      0      1      1      3
Node 0, zone   DMA32    759    572    440    243    115     34      6      2      2      1    208
Node 0, zone  Normal   1122    847    612    348    180     72     28      8      3      2    156

Interpretação das colunas — número de blocos livres para cada ordem:
  Ordem 0 = 1  página  (4 KB)   → 1122 blocos livres na zona Normal
  Ordem 1 = 2  páginas (8 KB)   → 847  blocos livres
  Ordem 2 = 4  páginas (16 KB)  → 612  blocos livres
  Ordem 3 = 8  páginas (32 KB)  → 348  blocos livres
  ...
  Ordem 10= 1024 páginas (4 MB) → 156  blocos livres

Sistema saudável: bastantes blocos em ordens altas (5+)
Sistema fragmentado: muitos blocos em ordem 0 e 1, poucos em ordens altas
Sintoma: alocações de DMA ou huge pages falham mesmo com memória "livre"

Colunas:
  active_objs  = objetos atualmente alocados
  num_objs     = capacidade total do cache
  objsize      = tamanho de cada objeto em bytes
  objperslab   = quantos objetos cabem em um slab

Quando o sistema está sob pressão de memória extrema, o kernel ativa o OOM Killer (Out-of-Memory Killer). Ele pontua processos por uso de memória e sacrifica o processo com maior pontuação para liberar RAM. A pontuação pode ser ajustada por processo via /proc/PID/oom_score_adj (de -1000 a +1000).
