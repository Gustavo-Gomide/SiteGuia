Sistemas Operacionais

Estados de Processo

O ciclo de vida de um processo: das cinco transições que o escalonador monitora ao impacto no desempenho do sistema.

O escalonador precisa saber a qualquer momento o que cada processo está fazendo para decidir quem recebe a CPU a seguir. Para isso, o kernel mantém o estado do processo como um campo no PCB. A máquina de estados define exatamente quais transições são possíveis e qual evento as dispara, permitindo ao SO tomar decisões eficientes de agendamento.

Diagrama de Estados

Os cinco estados principais e as transições entre eles:

EstadoNome em inglêsDescriçãoTransição de saídaEvento que causa a transição
Novo / Criado New Processo foi criado (fork), PCB alocado, mas ainda não está na fila de prontos Pronto Admissão pelo escalonador de longo prazo
Pronto Ready Processo aguarda na fila para receber a CPU; todos os recursos estão disponíveis exceto o processador Executando Dispatch pelo escalonador (curto prazo)
Executando Running Processo ocupa a CPU e executa instrucoes Pronto / Bloqueado / Terminado Preempção (quantum expirado) / chamada bloqueante / exit()
Bloqueado Blocked / Waiting Aguarda evento externo (leitura de disco, rede, sleep, lock) — não pode usar CPU mesmo que disponível Pronto Evento concluído (I/O completou, lock liberado, sinal recebido)
Terminado Terminated / Zombie Executou exit(); recursos de memória liberados, mas entrada na tabela de processos permanece até o pai chamar wait() Removido Pai executa wait() / waitpid()

Estados Especiais

Zumbi (Z)

O processo executou exit(), mas o pai ainda não chamou wait(). O PCB permanece na tabela de processos para preservar o código de saída. Se o pai nunca chama wait(), os PIDs vazam gradualmente.

Orfao

O pai terminou antes do filho. O kernel re-parent o filho para o init / systemd (PID 1), que periodicamente chama wait() impedindo zumbis.

Suspenso (swapped out)

O escalonador de médio prazo move o processo para disco (swap) quando há pressão de memória. As páginas são restauradas quando o processo retorna à ready queue.

Verificando Estados no Linux

O comando ps exibe o estado de cada processo na coluna STAT:

Shell — ps aux e interpretação da coluna STAT
# Listar todos os processos com estado
ps aux

# Exibir apenas PID, estado e comando
ps -eo pid,stat,comm

# Exemplo de saída:
#   PID STAT COMMAND
#     1 Ss   systemd
#   872 Sl   NetworkManager
#  1234 R+   meu_programa
#  5678 D    kworker/u4:2
#  9999 Z    processo_zumbi

# Significado das letras na coluna STAT:
#   R   Running / Runnable (pronto para executar)
#   S   Sleeping interrompível (aguardando evento, pode ser acordado por sinal)
#   D   Sleeping NÃO-interrompível (aguardando I/O; NÃO responde a sinais)
#   T   Stopped (parado por sinal SIGSTOP ou em debug)
#   Z   Zombie (terminou, aguarda wait() do pai)
#
# Modificadores adicionais:
#   s   Session leader
#   l   Multi-threaded
#   +   No foreground do terminal
#   N   Prioridade reduzida (nice > 0)
#   <   Prioridade elevada (nice < 0)

Impacto no Desempenho

SituacaoEstadoImpactoComo resolver
Processos em estado D acumulando D (uninterruptible sleep) Bloqueiam shutdown; aumentam load average mesmo sem usar CPU; indicam gargalo de I/O ou driver travado Identificar dispositivo de I/O lento; verificar driver com dmesg
Zumbis acumulando Z Vazamento de PID; tabela de processos se esgota; processo pai com bug Corrigir código do pai para chamar wait(); ou matar o pai (filho é adotado pelo init)
Processo I/O-bound Alterna S e R rapidamente Baixo uso de CPU; alta espera por I/O; responsivo a sinais Otimizar I/O (async I/O, buffers maiores, SSD)
Processo CPU-bound Majoritariamente R Alta utilização de CPU; pode monopolizar núcleo sem preempção Ajustar prioridade com nice; usar múltiplas threads ou processos
