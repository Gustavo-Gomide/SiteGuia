Sistemas Operacionais

Mutex e Semáforos

As duas primitivas fundamentais de sincronização — diferenças de ownership, semântica e quando usar cada uma.

Mutex (mutual exclusion lock) e semáforo são as ferramentas mais usadas para coordenar acesso a recursos compartilhados entre threads e processos. Embora relacionados, possuem semânticas distintas: o mutex é uma trava com ownership (quem travou deve destravar), enquanto o semáforo é um contador de sinalização sem proprietário.

Mutex

O mutex garante que apenas uma thread execute a seção crítica por vez. A thread que chama lock() adquire o mutex; se já estiver travado, a thread dorme até ser acordada pelo kernel quando o mutex for liberado:

C — pthread_mutex_t: uso básico e inicialização
#include <pthread.h>
#include <stdio.h>

static int saldo = 1000;
static pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;

void depositar(int valor) {
    pthread_mutex_lock(&mtx);      /* adquire — bloqueia se ocupado */
    saldo += valor;
    pthread_mutex_unlock(&mtx);    /* libera — acorda threads esperando */
}

void sacar(int valor) {
    pthread_mutex_lock(&mtx);
    if (saldo >= valor)
        saldo -= valor;
    pthread_mutex_unlock(&mtx);
}

/* Propriedades do mutex POSIX padrão:
   - Não reentrante: chamar lock() duas vezes na mesma thread causa deadlock
   - Use PTHREAD_MUTEX_RECURSIVE se precisar de reentrada
   - Ownership: somente quem chamou lock() pode chamar unlock()           */
C++ — std::unique_lock com timeout (timed_mutex)
#include <mutex>
#include <chrono>
#include <stdexcept>

std::timed_mutex mtx;

void operacao_critica() {
    using namespace std::chrono_literals;
    std::unique_lock<std::timed_mutex> lock(mtx, 100ms);
    if (!lock.owns_lock())
        throw std::runtime_error("timeout ao adquirir mutex");
    /* ... seção crítica ... */
}   /* unlock() automático ao sair do escopo — RAII */

Semáforo

Um semáforo mantém um contador inteiro não-negativo. A operação P (wait/down) decrementa; se chegar a zero, bloqueia. A operação V (signal/up) incrementa e acorda uma thread bloqueada:

C — sem_t POSIX: semáforo binário e de contagem
#include <semaphore.h>
#include <pthread.h>
#include <stdio.h>

/* Semáforo BINÁRIO (equivalente a mutex, mas sem ownership) */
static sem_t bin;

void thread_a(void) {
    sem_wait(&bin);   /* P(): decrementa; bloqueia se == 0 */
    /* seção crítica */
    sem_post(&bin);   /* V(): incrementa; acorda thread esperando */
}

/* Semáforo de CONTAGEM: controla acesso a N conexões de BD */
static sem_t pool_conexoes;

void init(void) {
    sem_init(&pool_conexoes, 0, 10);  /* máximo 10 conexões simultâneas */
}

void usar_conexao(void) {
    sem_wait(&pool_conexoes);   /* adquire uma conexão (decrementa) */
    /* ... usa conexão ... */
    sem_post(&pool_conexoes);   /* libera a conexão (incrementa) */
}

/* Nota: sem_t não tem owner — qualquer thread pode chamar sem_post()
   Isso permite usar semáforos para SINALIZAÇÃO entre threads distintas */

Mutex vs Semáforo: Diferenças

CritérioMutexSemáforo
Ownership Sim — somente quem fez lock() pode fazer unlock() Não — qualquer thread pode chamar post()/signal()
Valor inicial Sempre binário (travado/destravado) Inteiro N (binário quando N=1, contagem quando N>1)
Uso principal Proteger seção crítica (lock/unlock no mesmo escopo) Sinalização entre threads, controle de recursos (N instâncias)
Sinalização cruzada Não foi projetado para isso Sim — thread A faz sem_post(), thread B acorda do sem_wait()
Priority inheritance Suportado (PTHREAD_PRIO_INHERIT) Geralmente não suportado
Reentrada Não por padrão (use PTHREAD_MUTEX_RECURSIVE) N/A (sem ownership)

Armadilhas e Boas Práticas

Deadlock por Ordem Errada

Thread A adquire mutex1 depois mutex2; thread B adquire mutex2 depois mutex1 — impasse garantido. Solução: estabelecer uma ordem global de aquisição (ex.: por endereço de memória do mutex) e seguir sempre essa ordem.

Priority Inversion

Thread de alta prioridade espera mutex mantido por thread de baixa prioridade, que por sua vez é preemptada por thread de média prioridade. Solução: priority inheritance — o kernel eleva temporariamente a prioridade do holder para o nível do waiter.

RAII em C++

Usar std::lock_guard ou std::unique_lock garante que o mutex seja liberado automaticamente ao sair do escopo — mesmo em caso de exceção. Nunca gerencie mutex manualmente em C++ moderno.

Double-Lock e Esquecimento

Chamar lock() duas vezes na mesma thread com mutex não-recursivo causa deadlock imediato. Esquecer unlock() (ex.: retornar antecipadamente) deixa o mutex preso para sempre — corrija com RAII.
