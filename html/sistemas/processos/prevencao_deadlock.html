Sistemas Operacionais

Prevenção de Deadlocks

Técnicas práticas: lock ordering, try-lock com timeout, wait-die / wound-wait e ferramentas de detecção como TSan.

Na prática, a prevenção de deadlocks é mais pragmática que o modelo teórico do Algoritmo do Banqueiro. As duas técnicas que resolvem 99% dos casos em código real são lock ordering (adquirir mutexes sempre na mesma ordem global) e timeouts (desistir se o lock não for obtido em tempo razoável). O restante cobre cenários distribuídos e ferramentas de diagnóstico.

Lock Ordering

Adquirir mutexes sempre na mesma ordem global elimina a condição de Circular Wait — a mais fácil das condições de Coffman de quebrar na prática:

C — versão ERRADA (deadlock) vs versão CORRETA (lock ordering)
#include <pthread.h>
#include <stdint.h>

pthread_mutex_t m1 = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_t m2 = PTHREAD_MUTEX_INITIALIZER;

/* ---- VERSÃO ERRADA: ordens diferentes → deadlock ---- */
void *thread_a_errada(void *arg) {
    pthread_mutex_lock(&m1);   /* adquire m1 primeiro */
    pthread_mutex_lock(&m2);   /* depois m2           */
    /* ... */
    pthread_mutex_unlock(&m2);
    pthread_mutex_unlock(&m1);
    return NULL;
}

void *thread_b_errada(void *arg) {
    pthread_mutex_lock(&m2);   /* adquire m2 primeiro ← PROBLEMA */
    pthread_mutex_lock(&m1);   /* depois m1           */
    /* ... */
    pthread_mutex_unlock(&m1);
    pthread_mutex_unlock(&m2);
    return NULL;
}

/* ---- VERSÃO CORRETA: ordem por endereço de memória ---- */
void lock_ordered(pthread_mutex_t *a, pthread_mutex_t *b) {
    /* Sempre travar o mutex de endereço menor primeiro */
    if ((uintptr_t)a < (uintptr_t)b) {
        pthread_mutex_lock(a);
        pthread_mutex_lock(b);
    } else {
        pthread_mutex_lock(b);
        pthread_mutex_lock(a);
    }
}

void *thread_a_correta(void *arg) {
    lock_ordered(&m1, &m2);
    /* ... seção crítica ... */
    pthread_mutex_unlock(&m1);
    pthread_mutex_unlock(&m2);
    return NULL;
}

void *thread_b_correta(void *arg) {
    lock_ordered(&m2, &m1);   /* mesma função — ordem será m1 depois m2 */
    /* ... seção crítica ... */
    pthread_mutex_unlock(&m1);
    pthread_mutex_unlock(&m2);
    return NULL;
}

Timeouts e Try-Lock

Quando lock ordering não é viável (ex.: locks de terceiros, ordem dinâmica), try-lock com backoff exponencial evita espera infinita:

C — pthread_mutex_trylock com backoff exponencial
#include <pthread.h>
#include <time.h>
#include <errno.h>
#include <stdio.h>

/* Tenta adquirir m1 e m2 sem deadlock usando backoff exponencial */
int lock_com_backoff(pthread_mutex_t *m1, pthread_mutex_t *m2) {
    struct timespec intervalo = { .tv_sec = 0, .tv_nsec = 1000 }; /* 1 µs */
    int tentativas = 0;

    while (tentativas < 20) {
        if (pthread_mutex_trylock(m1) == 0) {
            if (pthread_mutex_trylock(m2) == 0) {
                return 0;  /* ambos adquiridos com sucesso */
            }
            pthread_mutex_unlock(m1);  /* não conseguiu m2; libera m1 */
        }

        /* Backoff exponencial: 1µs, 2µs, 4µs, 8µs, ... */
        nanosleep(&intervalo, NULL);
        intervalo.tv_nsec *= 2;
        tentativas++;
    }
    return EDEADLK;  /* desistiu após 20 tentativas */
}

/* Quando usar try-lock:
   - Sistemas onde starvation é preferível ao deadlock
   - Locks de diferentes subsistemas sem acesso ao código-fonte
   - Threads de tempo real com deadline strict
   CUIDADO: backoff puro sem aleatoriedade pode causar livelock  */

Wait-Die e Wound-Wait

Algoritmos baseados em timestamp lógico para bancos de dados distribuídos — evitam deadlock sem ordenação global de locks:

AlgoritmoTransação mais antigaTransação mais novaComportamentoRollbacks
Wait-Die Aguarda (wait) Aborta e reinicia (die) Antiga > Nova: antiga espera; Nova > Antiga: nova morre. Não-preemptivo no titular do lock Somente mais novas
Wound-Wait Preempta (wound) a mais nova Aguarda (wait) Antiga > Nova: antiga preempta a nova; Nova > Antiga: nova espera. Preemptivo no titular Somente mais novas (preemptadas)

Ambos garantem ausência de deadlock pois a relação de espera segue a ordem de timestamp — nunca circular. Wait-Die tem menos preempções mas mais rollbacks de novas transações; Wound-Wait tem mais preempções imediatas mas menos espera redundante.

Ferramentas de Detecção

ThreadSanitizer (TSan)

Instrumentação em tempo de compilação que detecta data races, deadlocks potenciais e uso incorreto de mutex. Overhead de 5–15x de tempo e 5–10x de memória — use apenas em testes e CI. Ative com -fsanitize=thread no GCC/Clang.

Valgrind Helgrind

Detector de erros de threading baseado em Valgrind. Rastreia a ordem de aquisição de locks e detecta violações. Mais lento que TSan (~20x), mas não requer recompilação — pode analisar binários existentes.

/proc e wchan

No Linux, cat /proc/PID/wchan mostra em qual função do kernel uma thread está dormindo. Threads em estado D ou S esperando por mutex aparecem em funções como futex_wait, revelando contenção de lock.

Shell — saída do ThreadSanitizer detectando potencial deadlock
$ gcc -fsanitize=thread -g -o demo demo.c -lpthread && ./demo
