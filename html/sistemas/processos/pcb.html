PCB — Bloco de Controle de Processo

A estrutura de dados que representa um processo no kernel: identidade, estado, memória, arquivos e credenciais em um único registro.

O PCB (Process Control Block) é o "cartão de identidade" de um processo dentro do kernel. Sem ele, o escalonador não saberia preservar o estado de um processo ao tirá-lo da CPU, nem restaurá-lo ao colocá-lo de volta. Cada processo ativo no sistema possui exatamente um PCB — no Linux, implementado pela estrutura task_struct, que contém aproximadamente 700 campos.

Troca de Contexto (Context Switch)

Quando o escalonador decide substituir o processo atual por outro, a CPU executa uma sequência precisa de passos para salvar e restaurar o estado:

Pseudocode — sequência do context switch
/* Chamado pelo escalonador quando o quantum expira ou
   o processo bloqueia voluntariamente */
void context_switch(Process *atual, Process *proximo) {

    /* 1. Salvar estado do processo atual no seu PCB */
    atual->pc         = cpu.program_counter;
    atual->registradores = cpu.registradores;   /* RAX..R15, flags */
    atual->sp         = cpu.stack_pointer;
    atual->estado     = PRONTO;  /* ou BLOQUEADO, se esperando I/O */

    /* 2. Salvar ponteiro para tabela de páginas atual */
    atual->mm         = cpu.cr3;  /* registrador CR3 no x86 */

    /* 3. Carregar tabela de páginas do próximo processo → flush de TLB */
    cpu.cr3           = proximo->mm;

    /* 4. Restaurar estado do próximo processo */
    cpu.program_counter = proximo->pc;
    cpu.registradores   = proximo->registradores;
    cpu.stack_pointer   = proximo->sp;
    proximo->estado     = EXECUTANDO;

    /* 5. Atualizar PCB do próximo com CPU atual */
    proximo->cpu_id   = this_cpu_id();
}

/* Custo típico: 1–10 µs por troca
   Fontes de overhead:
     - Flush de TLB (Translation Lookaside Buffer)
     - Cache pollution (dados do processo anterior no L1/L2)
     - Execução das instruções de save/restore            */

PCB no Linux: task_struct

No Linux, o PCB é a estrutura task_struct definida em include/linux/sched.h. O trecho a seguir mostra os campos mais relevantes de forma simplificada:

C — task_struct simplificada (Linux kernel)
struct task_struct {
    /* Identificação */
    pid_t              pid;       /* identificador único do processo/thread */
    pid_t              tgid;      /* thread group ID = PID do processo principal */

    /* Estado */
    volatile long      state;     /* TASK_RUNNING, TASK_INTERRUPTIBLE, TASK_ZOMBIE … */

    /* Memória */
    struct mm_struct  *mm;        /* descritor do espaço de endereçamento virtual */
    struct mm_struct  *active_mm; /* mm ativo (NULL para threads de kernel) */

    /* Arquivos */
    struct files_struct *files;   /* tabela de file descriptors abertos */
    struct fs_struct    *fs;      /* contexto de filesystem (cwd, root) */

    /* Credenciais */
    const struct cred  *cred;     /* UID, GID, capabilities efetivos */

    /* Escalonamento */
    int                prio;      /* prioridade dinâmica */
    int                static_prio;
    u64                se.vruntime; /* virtual runtime para o CFS */

    /* Sinais */
    struct signal_struct *signal;
    sigset_t           blocked;   /* máscara de sinais bloqueados */

    /* Parentes */
    struct task_struct *parent;   /* ponteiro para o processo pai */
    struct list_head    children; /* lista de filhos */

    /* Estatísticas */
    u64                utime;    /* tempo de CPU em modo usuário */
    u64                stime;    /* tempo de CPU em modo kernel */
    /* ... ~700 campos no total ... */
};

File Descriptors e Tabela de Arquivos

Cada processo possui uma tabela de file descriptors (FDs) — um array de índices inteiros. Os primeiros três são padrão:

FDNome simbólicoDestino padrãoComo redirecionar
0stdinTerminal (teclado)execve + dup2(fd, 0)
1stdoutTerminal (tela)dup2(fd, 1) ou > arquivo no shell
2stderrTerminal (tela)dup2(fd, 2) ou 2> arquivo
3+LivreQualquer arquivo, socket, pipeRetornado por open(), socket(), pipe(), accept()

Quando fork() é chamado, a tabela de FDs do filho é uma cópia da do pai — ambos apontam para as mesmas entradas na tabela global de arquivos abertos do kernel (struct file), com o contador de referências incrementado. Fechar um FD no filho não afeta o pai, mas ambos compartilham o mesmo offset de arquivo.
