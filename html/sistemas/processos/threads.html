Sistemas Operacionais

Threads

Linhas de execução concorrentes dentro de um processo — stack própria, memória compartilhada e o preço da sincronização.

Uma thread é a menor unidade de execução escalonável pelo kernel. Diferentemente de processos, threads de um mesmo processo compartilham heap, segmento de texto e descritores de arquivos, mas cada uma possui sua própria stack, program counter e conjunto de registradores. Isso torna a criação muito mais barata que um fork(), ao custo de exigir sincronização explícita sempre que dados compartilhados são acessados.

Modelos de Threading

A relação entre threads de usuário e threads do kernel define o modelo de threading e impacta diretamente desempenho e portabilidade:

ModeloMapeamentoVantagensDesvantagensExemplos
User-Level (N:1) N threads de usuário para 1 thread de kernel Troca de contexto rapidíssima (sem syscall), portável Bloqueio de uma bloqueia todo o processo; sem paralelismo real em multicore Green threads antigas (Python 2, Ruby MRI com GIL)
Kernel-Level (1:1) 1 thread de usuário para 1 thread de kernel Paralelismo real em multicore, bloqueio independente Overhead de syscall na criação e troca de contexto POSIX pthreads, Java threads (OpenJDK HotSpot), .NET CLR
Thread Pool (M:N) M threads de usuário para N threads de kernel (M > N) Equilíbrio entre custo de criação e paralelismo; ajuste fino Complexidade de implementação; difícil de depurar Go goroutines (runtime agenda M goroutines em N OS threads), Erlang

Python e o GIL: O CPython possui um Global Interpreter Lock que permite apenas uma thread executar bytecode por vez, tornando-o efetivamente 1:CPU para código puro Python — threads são úteis apenas para I/O-bound.

Criação e Uso — pthreads

A API POSIX threads é a interface padrão para threads em C/C++ em sistemas Unix/Linux:

C — pthread_create, mutex, contador compartilhado
#include <stdio.h>
#include <pthread.h>

#define N_THREADS 4
#define INCREMENTOS 100000

static long contador = 0;
static pthread_mutex_t mtx = PTHREAD_MUTEX_INITIALIZER;

void *incrementar(void *arg) {
    for (int i = 0; i < INCREMENTOS; i++) {
        pthread_mutex_lock(&mtx);
        contador++;
        pthread_mutex_unlock(&mtx);
    }
    return NULL;
}

int main(void) {
    pthread_t tids[N_THREADS];

    for (int i = 0; i < N_THREADS; i++)
        pthread_create(&tids[i], NULL, incrementar, NULL);

    for (int i = 0; i < N_THREADS; i++)
        pthread_join(tids[i], NULL);

    printf("contador = %ld (esperado %d)\n",
           contador, N_THREADS * INCREMENTOS);
    return 0;
}
/* Compilar: gcc -o demo demo.c -lpthread */

std::thread em C++

O C++11 padronizou threads na biblioteca padrão, eliminando dependência direta de pthreads e oferecendo RAII com std::lock_guard:

C++11 — std::thread, std::mutex, std::lock_guard
#include <iostream>
#include <thread>
#include <mutex>
#include <vector>

constexpr int N_THREADS   = 4;
constexpr int INCREMENTOS = 100000;

static long       contador = 0;
static std::mutex mtx;

void incrementar() {
    for (int i = 0; i < INCREMENTOS; i++) {
        std::lock_guard<std::mutex> lock(mtx); /* RAII: unlock automático */
        contador++;
    }
}

int main() {
    std::vector<std::thread> threads;
    threads.reserve(N_THREADS);

    for (int i = 0; i < N_THREADS; i++)
        threads.emplace_back(incrementar);

    for (auto &t : threads)
        t.join();

    std::cout << "contador = " << contador
              << " (esperado " << N_THREADS * INCREMENTOS << ")\n";
}
/* Compilar: g++ -std=c++17 -o demo demo.cpp */

Problemas Clássicos de Concorrência

ProblemaDefiniçãoComo se manifestaComo evitar
Race Condition Resultado depende da ordem não determinística de execução de threads Contador com resultado errado; leitura de dado parcialmente escrito Mutex, operações atômicas (std::atomic)
Deadlock Duas ou mais threads esperam indefinidamente por recursos que a outra segura Programa congela; uso de CPU cai a zero Lock ordering, try-lock com timeout, algoritmo do Banqueiro
Livelock Threads continuam executando (sem progresso real) respondendo uma à outra CPU alta, mas nenhuma tarefa avança; similar a dois humanos num corredor Backoff aleatório, prioridade diferenciada
Starvation Uma thread nunca obtém o recurso por ser constantemente preterida Thread de baixa prioridade nunca executa enquanto outras estão ativas Fairness no lock (fila FIFO), aging de prioridade
