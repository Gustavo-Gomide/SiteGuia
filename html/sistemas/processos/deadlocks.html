Sistemas Operacionais

Deadlocks

Impasse circular entre processos: condições de Coffman, grafos de alocação, estratégias de prevenção e recuperação.

Um deadlock ocorre quando dois ou mais processos estão bloqueados indefinidamente, cada um esperando por um recurso mantido pelo outro. Em sistemas de produção o custo é real: o processo trava silenciosamente, serviços param de responder e a única saída é reiniciar — perdendo estado e transações em andamento.

Condições de Coffman

As quatro condições são necessárias e suficientes para que um deadlock ocorra. Quebrar qualquer uma delas previne o deadlock:

CondiçãoDefiniçãoComo quebrarCusto da quebra
Exclusão Mútua Pelo menos um recurso é não-compartilhável — apenas um processo o usa por vez Tornar recursos compartilháveis (ex.: read-only, spooling de impressora) Nem sempre possível (ex.: arquivo para escrita)
Hold and Wait Processo mantém pelo menos um recurso e espera por recursos adicionais mantidos por outros Exigir que processos requisitem todos os recursos de uma vez antes de iniciar; ou liberar todos antes de pedir novos Baixa utilização de recursos; possível starvation
No Preemption Recursos não podem ser tomados à força de um processo Permitir preempção — se processo não obtém todos recursos, libera os que tem Só funciona para recursos cujo estado pode ser salvo (CPU, memória); impraticável para locks de BD
Circular Wait Existe uma cadeia circular de processos, cada um esperando recurso do próximo Impor ordenação total sobre todos os tipos de recurso; processos sempre adquirem na ordem crescente Restringe ordem de codificação; mais prático com lock ordering

Grafo de Alocação de Recursos

No grafo de alocação, processos são representados por círculos e recursos por quadrados. Uma aresta de processo para recurso indica requisição; de recurso para processo indica alocação. Um ciclo nesse grafo indica possível deadlock — se cada recurso tem uma única instância, o ciclo garante deadlock.

O código abaixo reproduz o cenário clássico de deadlock com dois mutexes e duas threads:

C — deadlock clássico com dois mutexes
#include <pthread.h>
#include <stdio.h>
#include <unistd.h>

pthread_mutex_t m1 = PTHREAD_MUTEX_INITIALIZER;
pthread_mutex_t m2 = PTHREAD_MUTEX_INITIALIZER;

/* Thread A: adquire m1 depois m2 */
void *thread_a(void *arg) {
    pthread_mutex_lock(&m1);
    printf("A: adquiriu m1, tentando m2...\n");
    sleep(1);                    /* cede tempo para thread B adquirir m2 */
    pthread_mutex_lock(&m2);   /* BLOQUEIA AQUI — m2 está com B */
    printf("A: adquiriu m2 (nunca impresso)\n");
    pthread_mutex_unlock(&m2);
    pthread_mutex_unlock(&m1);
    return NULL;
}

/* Thread B: adquire m2 depois m1 — ORDEM INVERSA → DEADLOCK */
void *thread_b(void *arg) {
    pthread_mutex_lock(&m2);
    printf("B: adquiriu m2, tentando m1...\n");
    sleep(1);
    pthread_mutex_lock(&m1);   /* BLOQUEIA AQUI — m1 está com A */
    printf("B: adquiriu m1 (nunca impresso)\n");
    pthread_mutex_unlock(&m1);
    pthread_mutex_unlock(&m2);
    return NULL;
}

int main(void) {
    pthread_t ta, tb;
    pthread_create(&ta, NULL, thread_a, NULL);
    pthread_create(&tb, NULL, thread_b, NULL);
    pthread_join(ta, NULL);   /* programa trava aqui forever */
    pthread_join(tb, NULL);
}

Estratégias de Tratamento

EstratégiaAbordagemPrósContrasCusto runtimeOnde usado
Prevenção Garantir que pelo menos uma condição de Coffman nunca seja satisfeita Deadlock impossível por construção Pode desperdiçar recursos; lock ordering pode ser difícil de manter Zero (decisão de design) Sistemas embarcados, drivers, sistemas críticos
Evasão (Banqueiro) Verificar antes de cada alocação se o sistema permanece em estado seguro Usa recursos eficientemente; deadlock nunca ocorre Requer declaração prévia de demanda máxima; O(n²) por alocação Alto (verificação constante) Sistemas de tempo real com requisitos bem definidos
Detecção e Recuperação Permitir deadlock, detectar periodicamente com grafo, resolver terminando processos Maior utilização de recursos no estado normal Custo da detecção; perda de trabalho ao terminar processos Periódico (varredura do grafo) BDs (lock manager detecta ciclos), alguns SOs
Ignorar (Avestruz) Fingir que deadlock não existe e reiniciar quando ocorrer Zero overhead; implementação simples Possível perda de dados; não adequado para sistemas críticos Zero Unix/Linux tradicionais para alguns recursos do kernel

Algoritmo do Banqueiro

Proposto por Dijkstra em 1965, o Algoritmo do Banqueiro determina se conceder uma requisição de recurso mantém o sistema em estado seguro — aquele em que existe pelo menos uma sequência de execução que permite todos os processos terminarem:

Pseudocode — Algoritmo do Banqueiro (verificação de estado seguro)
/* Dados:
   n = número de processos
   m = tipos de recurso
   Available[m]       = instâncias disponíveis de cada recurso
   Max[n][m]          = demanda máxima de cada processo
   Allocation[n][m]   = recursos atualmente alocados
   Need[i][j]         = Max[i][j] - Allocation[i][j]
*/

function estado_seguro(Available, Need, Allocation):
    Work    = cópia de Available
    Finish  = [false] * n           /* nenhum processo terminou ainda */
    sequencia_segura = []

    loop:
        /* Encontrar processo que pode terminar com recursos disponíveis */
        encontrou = false
        para i de 0 a n-1:
            se Finish[i] == false e Need[i] <= Work:
                Work     += Allocation[i]  /* processo termina, libera recursos */
                Finish[i]  = true
                sequencia_segura.append(i)
                encontrou  = true

        se não encontrou:
            sair do loop

    se todos Finish[i] == true:
        retornar SEGURO, sequencia_segura
    senão:
        retornar INSEGURO   /* deadlock potencial */

/* Por que SOs gerais não usam o Banqueiro:
   1. Processos raramente declaram demanda máxima antecipadamente
   2. Número de processos e tipos de recurso variam dinamicamente
   3. Overhead O(n²) por requisição é inaceitável em sistemas de propósito geral */
