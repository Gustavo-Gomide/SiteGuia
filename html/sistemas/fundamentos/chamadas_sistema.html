Sistemas Operacionais

Chamadas de Sistema (Syscalls)

A única porta de entrada ao kernel — mecanismo, categorias e exemplos

Chamadas de sistema são o mecanismo pelo qual todo programa de usuário solicita serviços do kernel: abrir arquivos, alocar memória, criar processos, enviar dados pela rede. Essa fronteira existe porque o kernel precisa ser protegido — uma aplicação não pode ler a memória de outra nem invadir o hardware. A syscall é a única passagem controlada de ring 3 para ring 0, onde o kernel valida a requisição, executa o serviço com privilégio completo e devolve o resultado ao solicitante.

Mecanismo de Execução em x86-64

Em x86-64 Linux, a instrução syscall é o ponto de entrada. O número da syscall é colocado em RAX; os argumentos vão em RDI, RSI, RDX, R10, R8, R9 (até seis). O processador salva o estado do processo, troca para ring 0 e desvia para o handler registrado via MSR_LSTAR. O retorno ocorre via instrução sysret.

Assembly x86-64 — write() manual sem libc
section .data
    msg db "ola, mundo", 0x0a   ; string + newline
    msg_len equ $ - msg

section .text
    global _start

_start:
    ; syscall write(1, msg, msg_len)
    mov rax, 1          ; syscall number: write
    mov rdi, 1          ; fd = stdout
    mov rsi, msg        ; ponteiro para o buffer
    mov rdx, msg_len    ; quantidade de bytes
    syscall             ; trap para ring 0

    ; syscall exit(0)
    mov rax, 60         ; syscall number: exit
    xor rdi, rdi        ; status = 0
    syscall

A libc não é obrigatória — qualquer programa pode invocar syscalls diretamente em assembly ou via wrappers de baixo nível. A libc apenas fornece wrappers convenientes que traduzem convenções POSIX para números de syscall específicos do kernel.

Categorias de Syscalls

Categoria Syscalls principais Função
Processos fork, execve, exit, wait4, getpid, clone Criar, substituir, encerrar e aguardar processos; criar threads (clone)
Memória mmap, munmap, brk, mprotect, madvise Mapear/desalocar regiões de memória virtual; ajustar heap; definir permissões
Arquivos open, read, write, close, stat, lseek, dup2 Abrir, ler, escrever, fechar arquivos e consultar metadados
Rede socket, bind, listen, accept, connect, sendto, recvfrom Criar sockets, estabelecer conexões TCP/UDP, enviar e receber dados
Sinais kill, signal, sigaction, sigprocmask, pause Enviar sinais entre processos; registrar handlers de sinal
Tempo gettimeofday, clock_gettime, nanosleep, timer_create Obter hora do sistema; suspender processo por intervalo; criar timers

Exemplos em C e Python

C — open / read / write / close
#include <fcntl.h>
#include <unistd.h>
#include <stdio.h>

int main(void) {
    // open(path, flags) → retorna file descriptor
    int fd = open("/etc/hostname", O_RDONLY);
    if (fd == -1) { perror("open"); return 1; }

    char buf[128];
    // read(fd, buf, n) → bytes lidos
    ssize_t n = read(fd, buf, sizeof(buf) - 1);
    if (n > 0) {
        buf[n] = '\0';
        // write(fd, buf, n) → escreve em stdout (fd=1)
        write(1, buf, n);
    }

    close(fd);   // libera o file descriptor
    return 0;
}
Python — os.open / os.read / os.write / os.close
import os

# os.open chama a syscall open diretamente (diferente do built-in open())
fd = os.open("/etc/hostname", os.O_RDONLY)

# os.read chama a syscall read
data = os.read(fd, 128)

# os.write chama a syscall write (fd=1 = stdout)
os.write(1, data)

os.close(fd)

# Para inspecionar as syscalls em tempo real:
# $ strace python3 script.py

Custo e Otimização de Syscalls

Cada syscall envolve um mode switch: o processador salva o estado do processo, muda para ring 0 e executa o handler — depois reverte. Esse ciclo custa tipicamente 100 a 1000 ns dependendo do processador, do tipo de syscall e se há mitigações ativas para vulnerabilidades (Spectre/Meltdown aumentaram o custo com PTI).

vDSO — Virtual Dynamic Shared Object

Para syscalls de leitura intensiva como gettimeofday e clock_gettime, o kernel mapeia uma região de memória compartilhada no espaço de usuário com dados atualizados periodicamente. A aplicação lê diretamente sem entrar no kernel — latência cai de ~100 ns para ~5 ns.

io_uring

Introduzido no Linux 5.1, io_uring implementa uma fila de submissão e uma fila de completude compartilhadas entre userspace e kernel. Centenas de operações de I/O podem ser submetidas e coletadas com zero ou uma syscall, eliminando o overhead por operação de I/O de alta frequência.

Batching e Polling

Técnicas como epoll (aguardar múltiplos fds de uma vez), sendmmsg/recvmmsg (enviar/receber múltiplos pacotes em uma syscall) e o modo SQPOLL do io_uring (um thread do kernel faz polling sem que o userspace precise fazer syscall) reduzem o número total de mode switches.

eBPF

eBPF permite injetar programas verificados no kernel que executam em ring 0 sem syscall por chamada. Usado para observabilidade (bpftrace), filtragem de pacotes (XDP) e segurança (seccomp-bpf): movem lógica para dentro do kernel eliminando round-trips.
