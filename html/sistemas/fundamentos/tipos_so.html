Sistemas Operacionais

Tipos de Sistemas Operacionais

Batch, time-sharing, RTOS, distribuído, embarcado e móvel

Não existe um SO universal: cada domínio de aplicação impõe requisitos distintos de latência, consumo de energia, isolamento, robustez e interatividade. Um sistema de tempo real para freios ABS precisa de garantias determinísticas que um SO de desktop jamais precisaria oferecer. Um SO distribuído precisa de transparência de localização que não faz sentido em um dispositivo embarcado com 64 KB de RAM. Classificar os SOs por tipo ajuda a entender quais mecanismos internos cada categoria prioriza.

Classificação por Uso

Tipo Definição Exemplos Característica-chave
Batch Executa jobs em lote sem interação do usuário IBM OS/360 JCL, PBS/Torque em clusters HPC Throughput máximo; sem tempo de resposta interativa
Time-Sharing / Interativo Múltiplos usuários ou processos compartilham a CPU com fatias de tempo Linux, macOS, Windows Tempo de resposta baixo percebido; preempção de processos
RTOS (Hard) Garante resposta dentro de prazo absoluto (deadline) FreeRTOS, VxWorks, QNX, Zephyr Escalonamento determinístico; latência de interrupção garantida
RTOS (Soft) Tenta cumprir deadlines; perdas ocasionais são toleráveis Linux com PREEMPT_RT, Android (camada RT) Preferência por baixa latência; não garante matematicamente
Distribuído Coordena múltiplas máquinas como sistema único Plan 9, Amoeba; modernamente Kubernetes Transparência de localização, tolerância a falhas
Embarcado Projetado para hardware com recursos limitados e função específica FreeRTOS, Contiki, TinyOS, RIOT Footprint mínimo; ROM/RAM escassos; sem MMU em muitos casos
Móvel Otimizado para dispositivos portativos com tela sensível ao toque e bateria Android, iOS, HarmonyOS Gerenciamento agressivo de energia; sandbox de aplicativos
SO de Rede Fornece serviços de rede compartilhados (arquivos, impressoras, diretório) Novell NetWare, Windows Server, Samba Protocolos de autenticação e compartilhamento centralizados

RTOS — Sistemas de Tempo Real

Em um RTOS hard real-time, perder um deadline não é apenas indesejável — pode ser catastrófico. O escalonador usa algoritmos com garantias matemáticas de temporização, como Rate Monotonic Scheduling (RMS) ou Earliest Deadline First (EDF). A latência de interrupção é medida em microssegundos e deve ser determinística, não estatística.

FreeRTOS

Kernel open-source de domínio público, amplamente utilizado em microcontroladores ARM Cortex-M. Suporta preempção com prioridades fixas, semáforos, filas e timers de software. Footprint menor que 10 KB de ROM.

VxWorks

RTOS comercial da Wind River, presente em sondas espaciais da NASA (Mars rovers), aviação e equipamentos militares. Certificado para DO-178C (aviação) e IEC 61508 (segurança funcional industrial).

QNX Neutrino

Microkernel RTOS usado em automóveis (BlackBerry QNX), sistemas médicos e infraestrutura crítica. Sua arquitetura microkernel permite reiniciar drivers com falha sem parar o sistema operacional.

Zephyr

RTOS open-source da Linux Foundation, projetado para IoT e dispositivos com restrições severas. Suporta mais de 500 placas diferentes, com footprint de menos de 8 KB para aplicações simples.

Aplicações críticas que dependem de RTOS: freios ABS, airbags, marcapassos, sistemas fly-by-wire, controle de reatores nucleares. Em todos esses casos, a falha de timing tem consequências físicas imediatas.

SOs Distribuídos

Um SO distribuído apresenta ao usuário uma visão unificada de múltiplas máquinas, com transparência de localização (o usuário não sabe nem precisa saber em qual máquina um recurso está). Isso difere de um cluster, onde cada nó tem seu SO independente e a coordenação é explícita.

Aspecto SO Distribuído Cluster / Grid Cloud
Transparência Total — usuário vê sistema único Parcial — ciente dos nós Parcial — abstraída por API
Exemplos Plan 9, Amoeba Beowulf, HTCondor AWS, GCP, Azure
Tolerância a falhas Embutida no SO Via middleware Via replicação de serviço
"SO de cluster moderno" Kubernetes pode ser visto como um SO de cluster: agenda pods (processos), gerencia recursos, abstrai hardware, tem controle de acesso — mas orquestra containers, não processos bare-metal

SOs Embarcados vs Móveis

Embora ambos rodem em hardware portátil, SOs embarcados tipicamente controlam um função específica (lavadora, termostato, sensor industrial) com recursos extremamente limitados, enquanto SOs móveis como Android e iOS são plataformas de propósito geral com interfaces ricas e ecossistemas de milhões de aplicativos.

Android Architecture

Android empilha: kernel Linux modificado (Binder, Wakelocks) + HAL (Hardware Abstraction Layer) + Android Runtime (ART, compilação AOT) + framework Java/Kotlin. O co-processador Titan M atua como RTOS isolado para operações de segurança.

iOS Architecture

iOS é baseado em Darwin/XNU (Mach + BSD). O Secure Enclave Processor (SEP) é um RTOS separado — isolado da CPU principal — responsável por chaves criptográficas, Touch ID e Face ID. O kernel principal é o XNU da Apple.

Restrições em Embarcados

Muitos microcontroladores embarcados não têm MMU, impossibilitando a separação user/kernel. O SO roda em memória flat com acesso irrestrito. A proteção vem do design cuidadoso do software, não do hardware. Memória pode ser de apenas 2–256 KB.
