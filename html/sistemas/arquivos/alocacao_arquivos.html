Alocação de Arquivos

Como sistemas de arquivos mapeiam dados para blocos físicos em disco

A forma como o sistema de arquivos mapeia um arquivo para blocos físicos no disco determina diretamente a performance de leitura e escrita, o grau de fragmentação ao longo do tempo e a complexidade do acesso aleatório. Cada método — contíguo, encadeado e indexado — apresenta compromissos distintos de desempenho e confiabilidade.

Alocação Contígua

Na alocação contígua, um arquivo ocupa um conjunto sequencial de blocos consecutivos no disco. A entrada de diretório armazena apenas o bloco inicial e o tamanho do arquivo.

Vantagens

Acesso sequencial muito rápido — um único seek de cabeça em HDDs. Implementação simples sem estruturas extras. Acesso aleatório direto: bloco de offset X = bloco_inicial + X/bloco_size.

Desvantagens

Fragmentação externa: ao deletar e recriar arquivos, surgem buracos no disco que podem não comportar novos arquivos. O tamanho deve ser conhecido na criação. Impossível crescer um arquivo sem realocar.

Onde é usado

CD-ROM e ISO 9660: o tamanho de todos os arquivos é conhecido antes da gravação, tornando alocação contígua perfeita. Também usado em sistemas embarcados com armazenamento estático.

Alocação Encadeada e FAT

Na alocação encadeada, cada bloco contém um ponteiro para o próximo bloco do arquivo. Não há fragmentação externa — qualquer bloco livre pode ser usado. O problema é o acesso aleatório: para ler o bloco N, é preciso percorrer todos os N-1 ponteiros anteriores.

O FAT centraliza todos os ponteiros em uma tabela separada (File Allocation Table), evitando o overhead de um ponteiro por bloco no disco de dados. A tabela inteira pode ficar em cache na RAM, acelerando a travessia.

pseudocódigo — travessia de cadeia FAT para arquivo fragmentado
// FAT (simplificada):
// índice:  0    1    2    3    4    5    6    7    8
// valor:  FREE FREE EOF  FREE FREE  8   FREE FREE  3

// Arquivo "doc.txt" começa no cluster 5
cluster = 5
while (FAT[cluster] != EOF):
    ler_cluster(cluster)   // cluster 5: dados bytes 0-511
    cluster = FAT[cluster] // FAT[5] = 8 → FAT[8] = 3 → FAT[3] = EOF
// Acesso aleatório ao byte 1024: deve percorrer clusters 5 → 8 antes de chegar ao 3

Alocação Indexada — Inode

O inode Unix contém um array de ponteiros para blocos de dados. Para suportar arquivos grandes com overhead mínimo em arquivos pequenos, usa-se uma estrutura multinível:

Para arquivos pequenos (a maioria), o acesso é O(1): basta ler os 12 ponteiros diretos do inode. Para arquivos grandes, até 3 leituras extras de blocos de ponteiro são necessárias — ainda constante.

Extents e Fragmentação

Extents (ext4, NTFS, btrfs, XFS) superam o modelo de ponteiro-por-bloco: descrevem faixas contíguas com (bloco inicial, comprimento). Um arquivo de 1 GB perfeitamente contíguo usa 1 extent, vs 262.144 entradas de bloco no modelo clássico.

bash — filefrag para ver fragmentação
# Verificar fragmentação de um arquivo
filefrag -v /home/usuario/video.mkv

# Saída típica sem fragmentação (SSD novo):
# ext:     0  logical:        0-   262143  physical: 1048576- 1310719  length: 262144  flags: last
# Total: 1 extent
# Não fragmentado!

# Saída típica com fragmentação (HDD velho):
# ext:     0  logical:        0-     4095  physical:  524288-  528383  length:   4096
# ext:     1  logical:     4096-     8191  physical: 1048576- 1052671  length:   4096
# ext:     2  logical:     8192-    12287  physical:  786432-  790527  length:   4096
# Total: 3 extents (fragmentado)

Fragmentação em HDDs

Fragmentação causa seeks adicionais de cabeça de leitura, aumentando latência de sequencial de 10x ou mais. Desfragmentação periódica (e4defrag, xfs_fsr) restaura a localidade.

Fragmentação em SSDs

Em SSDs, fragmentação tem impacto desprezível na leitura sequencial (sem seek mecânico). Desfragmentar SSDs é desnecessário e prejudicial ao desgaste das células NAND. O kernel usa TRIM/discard para gestão de espaço livre.
