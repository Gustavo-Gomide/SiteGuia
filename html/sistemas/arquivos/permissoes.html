Permissões de Arquivo

rwx, bits especiais, ACLs, umask e o modelo de segurança Unix

Permissões controlam quem pode ler, escrever ou executar arquivos — elas são a base do modelo de segurança Unix. Cada arquivo tem um dono (owner), um grupo associado e um conjunto de permissões para três classes: dono, grupo e outros (others). Entender esse modelo é pré-requisito para qualquer administração de sistema Linux.

Modelo rwx do Unix

Cada arquivo tem 9 bits de permissão divididos em três trios: rwx para o dono, rwx para o grupo e rwx para outros. Na representação octal, r=4, w=2, x=1. O comando ls -l exibe os bits como string de 10 caracteres (tipo + 9 bits).

Modo octal Representacao simbolica Significado tipico Riscos
644rw-r--r--Arquivo de configuracao ou textoNenhum significativo
755rwxr-xr-xBinario ou diretorio publicoNenhum significativo
700rwx------Script privado do usuarioNenhum
600rw-------Chave SSH privada, arquivo de senhaNenhum (protegido)
666rw-rw-rw-Arquivo gravavel por todosQualquer usuario pode sobrescrever
777rwxrwxrwxTudo liberadoAlto risco; evitar em producao
bash — chmod simbólico e octal
# Setar permissões com modo octal
chmod 755 script.sh

# Adicionar bit de execução para todos
chmod +x script.sh

# Remover escrita para outros
chmod o-w arquivo.txt

# Recursivo em diretório
chmod -R 750 /var/app/

Bits Especiais — setuid, setgid e Sticky Bit

setuid (bit 4000)

Quando ativo em um executavel, o processo roda com o UID do dono do arquivo, nao do usuario que executou. Permite que usuarios comuns executem operacoes privilegiadas de forma controlada. Exibido como s no bit x do dono.

setgid (bit 2000)

Em executavel: processo assume GID do grupo dono. Em diretorio: novos arquivos criados herdam o grupo do diretorio, nao o grupo primario do usuario. Util para diretorios compartilhados de equipe.

Sticky Bit (bit 1000)

Em diretorio: apenas o dono do arquivo (ou root) pode deletar ou renomear arquivos ali contidos, mesmo que o diretorio seja gravavel por todos. Usado em /tmp para evitar que usuarios apaguem arquivos uns dos outros.

bash — setuid no /usr/bin/passwd
# passwd precisa escrever em /etc/shadow (dono: root)
# Por isso tem setuid: qualquer usuário executa como root
ls -la /usr/bin/passwd
# -rwsr-xr-x 1 root root 68208 ...  /usr/bin/passwd
#    ^ 's' indica setuid ativo

# Setar setuid via chmod
chmod u+s /usr/bin/meuapp

# Sticky bit em /tmp
ls -ld /tmp
# drwxrwxrwt 15 root root ...  /tmp
#          ^ 't' indica sticky bit

# Setar sticky bit
chmod +t /shared/projeto/

ACLs — Access Control Lists

O modelo rwx so permite tres classes. ACLs POSIX permitem atribuir permissoes individuais para usuarios ou grupos adicionais, alem do trio padrao. A mascara (mask) ACL define o teto de permissoes para entradas nao-owner.

bash — getfacl e setfacl
# Ver ACL de um arquivo
getfacl relatorio.pdf
# # file: relatorio.pdf
# # owner: alice
# # group: financeiro
# user::rw-
# user:bob:r--        <-- permissão extra para bob
# group::r--
# mask::r--
# other::---

# Adicionar leitura+escrita para o usuário bob
setfacl -m u:bob:rw relatorio.pdf

# Adicionar ACL padrão para diretório (novos arquivos herdam)
setfacl -d -m g:devs:rwx /var/projeto/

# Remover ACL de um usuário
setfacl -x u:bob relatorio.pdf

# Remover todas as ACLs extras
setfacl -b relatorio.pdf

umask, chmod e chown

O umask define quais bits de permissao serao subtraidos das permissoes padrao ao criar novos arquivos e diretorios. Permissao padrao para arquivos e 0666, para diretorios e 0777. Com umask 022: arquivo criado com 644, diretorio com 755.

Comando Exemplo Efeito
umaskumask 022Novos arquivos: 644; novos dirs: 755
umaskumask 027Novos arquivos: 640; novos dirs: 750 (outros sem acesso)

chownchown alice:devs arquivoMuda dono e grupo
chownchown -R www-data:www-data /var/www/Recursivo em diretorio
