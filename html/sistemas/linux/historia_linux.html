História do Linux

De um projeto de hobby finlandês em 1991 ao software que roda o mundo

Em agosto de 1991, um estudante finlandês de 21 anos publicou uma mensagem informal no grupo comp.os.minix dizendo que estava criando um kernel de hobby — "just a hobby, won't be big and professional". Esse projeto se tornou o kernel mais usado do planeta, rodando desde smartwatches até os maiores supercomputadores do mundo.

GNU e a Parceria Essencial

Em 1984, Richard Stallman lançou o projeto GNU ("GNU's Not Unix") com o objetivo de criar um sistema operacional completamente livre. Em uma década, o projeto produziu ferramentas essenciais:

GCC

GNU Compiler Collection — compilador C/C++/Fortran/Ada que permitiu compilar o próprio kernel Linux e qualquer software livre.

Bash

Bourne Again Shell — shell padrão de praticamente todas as distribuições Linux; linha de comando que os usuários interagem diariamente.

glibc

GNU C Library — biblioteca C padrão usada por quase todos os programas Linux; interface entre aplicacoes e syscalls do kernel.

GPL v2

A licença que protegeu o Linux: qualquer modificação distribuída deve ter o código aberto. Impediu que empresas fechassem o projeto.

Quando Linus criou o kernel, ele se apoiou nas ferramentas GNU para torná-lo utilizável. O sistema completo é tecnicamente GNU/Linux — debate que persiste: Richard Stallman insiste no nome composto; a maioria do mundo usa apenas "Linux".

Desenvolvimento do Kernel Hoje

O kernel Linux é o maior projeto de software colaborativo da história. Alguns números da era atual:

Tamanho

Aproximadamente 35 milhões de linhas de código em C, Assembly e, crescentemente, Rust.

Desenvolvedores

Cerca de 4.500 desenvolvedores contribuem para cada release; representam mais de 200 empresas.

Cadência

Novo release a cada 8 semanas; janela de merge de 2 semanas seguida de RCs semanais.

Maiores Contribuidoras

Intel, AMD, Red Hat, Google, Meta, Microsoft, Samsung — empresas pagam engenheiros para trabalhar no kernel upstream.

O processo é distribuído: patches chegam via mailing list, subsystem maintainers aprovam e encaminham para árvores de staging, Linus Torvalds faz o merge final no repositório principal do git.
