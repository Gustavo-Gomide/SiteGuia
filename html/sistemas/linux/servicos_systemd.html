Servicos e systemd

Gerenciamento moderno de servicos, boot, logs e timers no Linux

O systemd substituiu o SysV init na maioria das distribuicoes a partir de 2012 e se tornou o sistema de init padrao no Linux moderno. Ele nao gerencia apenas servicos: controla toda a sequencia de boot, sistema de log (journald), montagem de sistemas de arquivos, timers, sockets, e muito mais — tudo com paralelizacao e dependencias declarativas.

Comandos systemctl

O systemctl e a interface principal para interagir com o systemd. Com ele voce gerencia o ciclo de vida de qualquer unit (servico, socket, timer, mount, etc.).

systemctl — referencia de comandos
# Ciclo de vida de um servico
systemctl start   nginx
systemctl stop    nginx
systemctl restart nginx
systemctl reload  nginx     # recarrega config sem reiniciar (se suportado)
systemctl status  nginx     # exibe estado, PID, log recente e ativacao

# Habilitar/desabilitar na inicializacao
systemctl enable  nginx     # cria symlink em /etc/systemd/system/
systemctl disable nginx
systemctl enable --now nginx  # habilita E inicia imediatamente

# Inspecao
systemctl is-active  nginx   # retorna "active" ou "inactive"
systemctl is-enabled nginx   # retorna "enabled" ou "disabled"
systemctl list-units --type=service
systemctl list-units --failed
systemctl list-unit-files --state=enabled

# Recarregar definicoes apos alterar arquivos de unit
systemctl daemon-reload

# Editar unit sem alterar o original (cria override em /etc/systemd/system/nginx.service.d/)
systemctl edit nginx

# Exibir arvore de dependencias
systemctl list-dependencies nginx

Arquivo de Unit (.service)

Uma unit de servico e um arquivo de texto simples que declara como o systemd deve gerenciar aquele processo. Arquivos do sistema ficam em /usr/lib/systemd/system/; customizacoes em /etc/systemd/system/ tem precedencia.

/etc/systemd/system/webserver.service — exemplo completo
[Unit]
Description=Servidor Web Hipotetico
Documentation=https://exemplo.com/docs
After=network-online.target
Wants=network-online.target
Requires=postgresql.service

[Service]
Type=simple
User=webuser
Group=webgroup
WorkingDirectory=/opt/webserver
EnvironmentFile=/etc/webserver/env
ExecStart=/opt/webserver/bin/webserver --config /etc/webserver/config.yaml
ExecStop=/bin/kill -SIGTERM $MAINPID
ExecReload=/bin/kill -SIGHUP $MAINPID
Restart=on-failure
RestartSec=5s
TimeoutStartSec=30s
LimitNOFILE=65536

[Install]
WantedBy=multi-user.target

[Unit] — Metadados e Dependencias

After: inicia so apos essas units estarem ativas. Requires: falha se a dependencia nao estiver ativa. Wants: tenta iniciar a dependencia mas nao falha se ela nao estiver disponivel.

[Service] — Execucao

Type=simple: o processo principal e o ExecStart. Type=forking: servico estilo daemon que faz fork. Type=notify: daemon envia sd_notify quando pronto. Restart=on-failure: reinicia automaticamente em caso de erro.

[Install] — Ativacao

WantedBy=multi-user.target significa que o servico sera iniciado no runlevel 3 (sem interface grafica). Use graphical.target para servicos que precisam de X11/Wayland.

Targets e Ordem de Boot

Targets sao agrupamentos de units que representam estados do sistema, equivalentes aos runlevels do SysV. O boot segue uma cadeia de dependencias declarativas ate atingir o target padrao.

Target SysV Runlevel Descricao
poweroff.target0Desligar o sistema
rescue.target1Modo de recuperacao — root shell, servicos minimos
multi-user.target3Multiusuario em modo texto — sem interface grafica
graphical.target5Multiusuario com interface grafica (inclui multi-user)
sleep.target—Suspender o sistema (sleep/hibernate)
reboot.target6Reiniciar o sistema
Gerenciando targets
# Ver o target padrao atual
systemctl get-default

# Definir novo target padrao
systemctl set-default multi-user.target

# Trocar de target imediatamente (sessao atual)
systemctl isolate rescue.target

# Sequencia de boot resumida:
# 1. Firmware (UEFI/BIOS) — POST, selecao de disco
# 2. Bootloader (GRUB2) — carrega kernel e initramfs
# 3. Kernel — inicializa hardware, monta /
# 4. initramfs — monta filesystems criticos
# 5. PID 1 = /sbin/init (systemd) — lanca primeiro processo
# 6. default.target — ativa todos os units necessarios
# 7. Login manager ou getty

journald e Logs

O journald e o sistema de log do systemd. Armazena mensagens de todos os servicos, o kernel e o boot em formato binario indexado, permitindo filtros poderosos.

journalctl — exemplos de consulta
# Todos os logs (paginado)
journalctl

# Logs de um servico especifico
journalctl -u nginx
journalctl -u nginx -u postgresql

# Seguir logs em tempo real (como tail -f)
journalctl -f
journalctl -fu nginx

# Logs do boot atual
journalctl -b

# Logs de dois boots atras
journalctl -b -2

# Apenas mensagens do kernel
journalctl -k

# Filtrar por prioridade
journalctl -p err      # apenas erros e acima
journalctl -p warning

# Filtrar por tempo
journalctl --since "2024-01-15 10:00:00"
journalctl --since "2 hours ago" --until "1 hour ago"

# Saida sem paginacao (para scripts)
journalctl -u nginx --no-pager

# Formato JSON para processamento
journalctl -u nginx -o json-pretty | head -50

# Espaco em disco utilizado pelos logs
journalctl --disk-usage

Por padrao os logs sao volateis (em /run/log/journal) e perdidos no reboot. Para tornar persistentes, crie o diretorio e defina o armazenamento em /etc/systemd/journald.conf:

/etc/systemd/journald.conf — persistencia e limites
[Journal]
Storage=persistent         # auto | volatile | persistent | none
Compress=yes
SystemMaxUse=500M          # limite total em disco
SystemKeepFree=100M        # espaco minimo livre a manter
MaxRetentionSec=1month     # apagar logs mais antigos que 1 mes
MaxFileSec=1week           # rotacionar arquivos apos 1 semana
