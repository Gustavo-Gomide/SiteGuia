Sistemas Operacionais

Permissões no Linux

rwx, sudo, PAM, capabilities e Mandatory Access Control com SELinux e AppArmor

O modelo de permissões do Linux vai muito além do rwx básico. sudo delega privilege sem expor a senha do root; PAM controla autenticação de forma modular; capabilities fragmentam os poderes do root; SELinux e AppArmor adicionam uma camada de Mandatory Access Control obrigatório, formando uma defesa em profundidade real.

sudo e /etc/sudoers

sudo permite executar comandos como root (ou outro usuário) sem trocar de sessão, registrando cada uso em log (/var/log/auth.log ou /var/log/secure). A configuração fica em /etc/sudoers e deve ser editada exclusivamente com visudo, que valida a sintaxe antes de salvar.

/etc/sudoers — sintaxe e exemplos comuns
# Sintaxe: USUARIO HOST=(USUARIO_DESTINO) COMANDOS

# Dar acesso total ao usuario joao
joao ALL=(ALL:ALL) ALL

# Grupo sudo/wheel: todos os membros viram root
%sudo  ALL=(ALL:ALL) ALL
%wheel ALL=(ALL:ALL) ALL    # convencao RHEL/Fedora

# Executar comandos especificos sem senha
deply ALL=(ALL) NOPASSWD: /usr/bin/systemctl restart nginx, \
                           /usr/bin/systemctl status nginx

# Incluir arquivos de configuracao adicional
#includedir /etc/sudoers.d

# Verificar o que um usuario pode fazer
sudo -l -U joao

# Listar proprias permissoes sudo
sudo -l

PAM — Pluggable Authentication Modules

PAM é a camada de autenticação do Linux. Cada serviço (login, ssh, sudo, su) tem um arquivo em /etc/pam.d/ que define a pilha de módulos a executar. Isso permite configurar 2FA, limites de sessão, restrições de terminal — tudo sem modificar os programas.

pam_unix

Autenticação por senha local via /etc/shadow. Módulo base presente em quase toda configuração PAM.

pam_google_authenticator

Adiciona 2FA (OTP) ao SSH ou login local. Instalado e configurado por usuário com google-authenticator.

pam_limits

Aplica limites de recursos (ulimits) por usuário: número de processos, tamanho de arquivos, memória. Configuração em /etc/security/limits.conf.

pam_securetty

Restringe login do root a terminais listados em /etc/securetty. Bloqueia root via SSH por padrão.

/etc/pam.d/sshd — configuração típica
# Tipos de linha: auth | account | password | session
# Controles: required | requisite | sufficient | optional

auth    required  pam_securetty.so
auth    required  pam_unix.so
auth    required  pam_nologin.so
auth    optional  pam_google_authenticator.so   # 2FA (se instalado)

account required  pam_unix.so
account required  pam_access.so

password required pam_unix.so sha512 shadow

session required  pam_unix.so
session required  pam_limits.so
session optional  pam_lastlog.so

Linux Capabilities

Tradicionalmente, root era tudo ou nada. As capabilities dividem os privilégios do root em unidades independentes, permitindo conceder apenas o necessário a um processo sem dar root completo.

CapabilityPrivilégio ConcedidoUso Comum
CAP_NET_BIND_SERVICEAbrir portas abaixo de 1024 sem rootnginx/Apache rodando como www-data na porta 80/443
CAP_NET_ADMINConfigurar interfaces de rede, rotas, firewallFerramentas de rede que precisam alterar configurações
CAP_SYS_TIMEAjustar o relógio do sistemantpd, chrony
CAP_KILLEnviar sinais a qualquer processo de qualquer UIDMonitores de processo, supervisores
CAP_SYS_PTRACEFazer ptrace em qualquer processoDebuggers (gdb, strace) em modo privilegiado
CAP_DAC_OVERRIDEIgnorar permissões de arquivo (DAC)Agentes de backup que precisam ler todos os arquivos
bash — getcap e setcap
# Ver capabilities de um binario
getcap /usr/bin/ping
# /usr/bin/ping cap_net_raw=ep

# Conceder cap_net_bind_service ao node.js (porta 80 sem root)
sudo setcap cap_net_bind_service=+ep /usr/bin/node

# Remover todas as capabilities de um binario
sudo setcap -r /usr/bin/node

# Listar capabilities do processo atual
cat /proc/$$/status | grep Cap

SELinux e AppArmor: Mandatory Access Control

O modelo DAC (Discretionary Access Control) tradicional deixa o dono do arquivo decidir as permissões. O MAC (Mandatory Access Control) adiciona políticas do sistema que nem o root pode contornar sem alterar a política explícita.

AspectoDAC Clássico (rwx)SELinuxAppArmor
ModeloBaseado em UID/GID + bits rwxContexts (user:role:type:level) + policyPerfis por caminho de programa
GranularidadePor arquivo/diretórioPor tipo de objeto e processo (MLS/MCS)Por programa e caminho de arquivo
ComplexidadeSimplesAlta — curva de aprendizado íngremeMédia — mais simples de escrever perfis
DistribuiçõesTodasRHEL, Fedora, CentOS (padrão)Ubuntu, Debian, openSUSE (padrão)
Modo permissivoN/ASim (loga sem bloquear)Sim (complain mode)
bash — SELinux e AppArmor comandos essenciais
# ==================== SELINUX ====================
# Ver status
gestatus
sestatus -v

# Ver contexto de um arquivo
ls -Z /var/www/html/

# Ver negações recentes de politica
ausearch -m avc -ts recent

# Gerar politica a partir de negacoes (cuidado em producao)
audit2allow -a -M minha_politica
semodule -i minha_politica.pp

# Mudar contexto de arquivo
chcon -t httpd_sys_content_t /var/www/html/

# Modo permissivo temporario
setenforce 0   # permissivo (so loga)
setenforce 1   # enforcing (bloqueia)

# ==================== APPARMOR ====================
# Ver status dos perfis
aa-status

# Colocar perfil em modo reclamacao (nao bloqueia)
aa-complain /etc/apparmor.d/usr.bin.nginx

# Perfis em /etc/apparmor.d/
ls /etc/apparmor.d/

# Carregar perfil
apparmor_parser -r /etc/apparmor.d/usr.sbin.nginx
