HTTPS e TLS

HTTP sobre TLS: confidencialidade, integridade e autenticação para toda a web

O HTTPS é HTTP encapsulado em TLS (Transport Layer Security — RFC 8446 para TLS 1.3). O TLS garante: confidencialidade (criptografia AES-GCM/ChaCha20), integridade (HMAC/AEAD) e autenticação do servidor (certificado X.509). TLS 1.3 eliminou algoritmos fracos (RSA key exchange, SHA-1, RC4) e reduziu o handshake para 1-RTT (0-RTT possível).

Handshake TLS 1.3 (1-RTT)

Fluxo simplificado do TLS 1.3
Cliente                                    Servidor
  |                                           |
  |-- ClientHello (TLS 1.3, key_share) -----> |  # suportes de suites e chave Diffie-Hellman
  |                                           |
  |<-- ServerHello + EncryptedExtensions ---  |  # suite escolhida + chave DH do servidor
  |     + Certificate + CertificateVerify     |  # certificado X.509 do servidor
  |     + Finished (tudo criptografado!)      |  # MAC sobre o handshake
  |                                           |
  |-- Finished -----------------------------> |  # cliente confirma
  |                                           |
  |<===== Dados da aplicação criptografados ==|

Em TLS 1.3, o servidor começa a criptografar após o ServerHello. Em TLS 1.2, eram necessários 2-RTT e o Certificate vinha em texto plano.

PKI e Certificados X.509

Hierarquia PKI

  • Root CA: Autoridade Certificadora raiz; certificado auto-assinado; embutida nos SOs e browsers; lista de confiabilidade (Mozilla NSS, Microsoft, Apple)
  • Intermediate CA: CA intermediária assinada pelo root; emite certificados de servidores. Separar root do Intermediate limita o impacto de comprometi-mento
  • Certificado de servidor: emitido pela Intermediate CA; validado pela chain of trust até o Root
  • Tipos de validação: DV (Domain Validation — apenas domínio, automatável via ACME/Let's Encrypt); OV (Organization Validation); EV (Extended Validation — valida legal)

Renovação e Revogação

  • Let's Encrypt + ACME: certificados DV gratuitos; renovados automaticamente a cada 90 dias via certbot ou acme.sh
  • OCSP (Online Certificate Status Protocol): consulta em tempo real se certificado foi revogado
  • OCSP Stapling: servidor anexa a resposta OCSP ao handshake TLS; cliente não precisa consultar o OCSP diretamente; melhora privacidade e perfomance
  • CRL (Certificate Revocation List): lista publicada pela CA de certificados revogados; mais pesada que OCSP; download periódico

HSTS e Segurança Web

  • HSTS (HTTP Strict Transport Security): header que informa o browser para só acessar via HTTPS; com includeSubDomains; preload é incluído na lista global do Chrome/Firefox
  • Certificate Pinning: aplicações móveis fixam o certificado esperado; protege contra CAs comprometidas; difícil de manter em escala
  • CT (Certificate Transparency): todos os certificados emitidos devem ser registrados em logs públícos auditados; detecta emissão não-autorizada
