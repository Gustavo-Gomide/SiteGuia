FTP — File Transfer Protocol

Transferência de arquivos via dois canais TCP: controle (21) e dados (20 ou efemero)

O FTP (RFC 959) usa dois canais TCP separados: um para controle (comandos/respostas) e outro para dados (transferência de arquivos). Em texto plano por padrão — usuário/senha transitável em claro. Para produção, usar FTPS (FTP + TLS) ou SFTP (SSH File Transfer Protocol).

Modo Ativo vs Passivo

Modo Ativo (PORT)

O cliente abre uma porta alta e diz ao servidor qual é (PORT command). O servidor conecta de volta do porto 20 para a porta informada pelo cliente. Problema: firewalls e NAT bloqueiam conexões iniciadas pelo servidor para o cliente — modo ativo raramente funciona atrás de NAT.

Modo Ativo
Cliente:21 → Servidor:21  (controle / CLIENT → SERVER)
Servidor:20 → Cliente:PORT  (dados / SERVER → CLIENT)
                              ^ Problema: NAT bloqueia esta direção

Modo Passivo (PASV)

O servidor abre uma porta alta e informa ao cliente (PASV command). O cliente conecta para essa porta. Funciona atrás de NAT pois o cliente sempre inicia. Modo padrão em clientes modernos.

Modo Passivo
Cliente:21 → Servidor:21  (controle / CLIENT → SERVER)
Cliente:NNNN → Servidor:PASV  (dados / CLIENT → SERVER)
                              ^ OK: NAT permite saída do cliente

FTPS vs SFTP

FTPS (FTP Secure)

FTP com TLS adicionado. Duas variantes:

  • Explícito (FTPES): coneta na porta 21 e upgrade com AUTH TLS; caí para FTP sem criptografia se não suportado
  • Implícito (FTPS): porta 990; TLS imediato; considera-se o padrão mais seguro
  • Mantém a arquitetura de dois canais — ainda problemático com firewalls stateful

SFTP (via SSH)

Não é FTP + SSH — é um protocolo independente (SSH File Transfer Protocol) que roda sobre SSH (porta 22). Um único canal TCP criptografado. Não tem os problemas de dois canais/firewall do FTP. É o protocolo recomendado para transferência segura de arquivos hoje.

Uso básico SFTP
# Conectar
sftp user@remote.server

# Baixar arquivo
get /remote/path/file.txt

# Enviar arquivo
put localfile.txt /remote/path/

# Sincronizar diretório (via scp)
rsync -avz -e ssh localdir/ user@server:/remote/
