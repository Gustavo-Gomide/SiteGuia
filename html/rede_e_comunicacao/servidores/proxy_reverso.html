Proxy Reverso

Conceitos, diferença de forward proxy, casos de uso e configuração com Nginx

Um proxy reverso intercepta requisições dos clientes e as encaminha para um ou mais servidores de backend — ficando invisível ao cliente. É o padrão dominante na arquitetura de aplicações web modernas.

Forward Proxy vs Reverse Proxy

CaracterísticaForward ProxyReverse Proxy
PosiçãoNa frente do clienteNa frente do servidor
Quem configuraO cliente — proxy da empresa, VPNO administrador do servidor
VisibilidadeServidor não sabe o IP real do clienteCliente não sabe o IP real do servidor
Caso de usoFiltro de conteúdo, anonimato, bypassTLS termination, load balancing, cache
ExemplosSquid, 3proxy, Tor, PrivoxyNginx, Apache, HAProxy, Traefik, Cloudflare

Casos de Uso

TLS Termination

Decifra HTTPS e encaminha HTTP para o backend. Centraliza certificados.

Load Balancing

Distribui requisições entre múltiplas instâncias do backend sem mudar o cliente.

Cache

Armazena respostas em memória/disco, reduzindo carga no app e latência.

Ocultar Backend

O cliente nunca conhece IP/porta reais do servidor de aplicação.

Compressão

Comprime respostas com gzip/brotli antes de enviar ao cliente.

Rate Limiting

Limita requisições por IP para proteção contra abuso e DDoS de aplicação.

WAF

Filtra requisições maliciosas (SQLi, XSS) antes de chegarem ao backend.

Header Manipulation

Injeta X-Request-ID, remove X-Powered-By, adiciona HSTS, etc.

Configuração Nginx — Proxy Reverso Completo

/etc/nginx/sites-available/app
upstream app_backend {
    server 10.0.1.10:8000;
    server 10.0.1.11:8000;
    keepalive 32;
}

server {
    listen 443 ssl http2;
    server_name app.example.com;
    server_tokens off;

    ssl_certificate     /etc/letsencrypt/live/app.example.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/app.example.com/privkey.pem;

    add_header Strict-Transport-Security "max-age=31536000" always;
    add_header X-Content-Type-Options    "nosniff" always;

    proxy_connect_timeout 10s;
    proxy_read_timeout    60s;

    location / {
        proxy_pass         http://app_backend;
        proxy_http_version 1.1;
        proxy_set_header Host              $host;
        proxy_set_header X-Real-IP         $remote_addr;
        proxy_set_header X-Forwarded-For   $proxy_add_x_forwarded_for;
        proxy_set_header X-Forwarded-Proto $scheme;
        proxy_hide_header X-Powered-By;
        proxy_set_header Upgrade    $http_upgrade;
        proxy_set_header Connection "upgrade";
    }
}

Cabeçalhos de Encaminhamento

CabeçalhoValorUso
X-Real-IPIP do cliente originalIP direto do cliente (sem cadeia de proxies)
X-Forwarded-ForclientIP, proxy1IPCadeia de IPs — o primeiro é o cliente original
X-Forwarded-ProtohttpsProtocolo original (o backend recebe HTTP puro)
X-Forwarded-Hostexample.comHost original da requisição
Forwarded (RFC 7239)for=1.2.3.4;proto=httpsVersão padronizada que unifica todos os X-Forwarded-*

Segurança: Nunca confie cegamente no X-Forwarded-For recebido pelo backend — um cliente malicioso pode falsificar esse header. Valide que a requisição veio do proxy esperado.

Cache no Proxy Reverso

nginx.conf — proxy cache
# http{} context
proxy_cache_path /var/cache/nginx
    levels=1:2
    keys_zone=app_cache:10m
    max_size=1g
    inactive=60m
    use_temp_path=off;

server {
    location / {
        proxy_cache           app_cache;
        proxy_cache_valid     200 302  10m;
        proxy_cache_valid     404      1m;
        proxy_cache_use_stale error timeout updating;
        add_header X-Cache-Status $upstream_cache_status;
        proxy_pass http://app_backend;
    }

    location /api/user {
        proxy_no_cache    1;
        proxy_cache_bypass 1;  # nunca cachear APis autenticadas
        proxy_pass http://app_backend;
    }
}

Aprofunde-se nas tecnologias que implementam proxy reverso: Nginx e Load Balancer.
