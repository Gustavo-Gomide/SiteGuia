Introdução a Servidores Web

Arquitetura cliente-servidor, modelos de processamento e visão geral das principais soluções

Um servidor web é um software que aceita requisições HTTP/HTTPS de clientes (navegadores, apps, CLI) e retorna respostas — HTML, JSON, arquivos estáticos ou dados de APIs. Entender como ele funciona internamente é essencial para construir e operar aplicações web confiáveis.

Modelo Cliente-Servidor

Toda aplicação web opera sobre o modelo cliente-servidor: o cliente inicia a comunicação e o servidor responde. Na prática, a maioria das aplicações modernas adiciona camadas intermediárias entre os dois extremos.

Cliente

Inicia a comunicação enviando uma requisição (ex.: GET /index.html HTTP/1.1). Pode ser um navegador, app mobile, curl, script Python, etc.

Servidor Web

Recebe a requisição, processa (serve arquivo estático ou delega para o app server) e devolve uma resposta com código de status, cabeçalhos e corpo.

App Server

Executa a lógica de negócios (Python / Node / Java / PHP). O servidor web atua como proxy reverso na frente do app server, repassando requisições dinâmicas via proxy_pass, fastcgi_pass, uwsgi, etc.

Banco de Dados

O app server consulta o SGBD (PostgreSQL, MySQL, MongoDB…) para persistir ou recuperar dados. Nunca exposto diretamente à internet.

Modelos de Processamento

A arquitetura interna do servidor define como ele lida com múltiplas conexões simultâneas. A escolha do modelo impacta diretamente a escalabilidade, o consumo de memória e o tipo de carga ideal.

Process-per-request (Prefork)

Um processo do SO separado por conexão. Isolamento máximo — uma falha não afeta outras conexões. Alto consumo de memória em escala. Usado pelo Apache com mod_php.

Ideal: aplicações PHP legadas com bibliotecas não thread-safe.

Thread-per-request (Worker)

Uma thread por conexão dentro de poucos processos. Menor overhead que prefork, mas threads compartilham memória — exige código thread-safe.

Ideal: conteúdo dinâmico sem mod_php, cargas médias.

Event-driven / Async

Loop de eventos único (ou pool mínimo) gerencia milhares de conexões via epoll / kqueue sem bloquear. Baixíssimo consumo de memória.

Ideal: alta concorrência I/O-bound. Ex.: Nginx, Node.js, Caddy.

Hybrid (Multi-process + Async)

Múltiplos worker processes, cada um com loop de eventos interno. Combina isolamento de processo com eficiência assíncrona.

Ideal: produção de alta carga. Ex.: Nginx (master + workers), Apache event MPM.

Fluxo de uma Requisição HTTP Típica

Entender o caminho completo de uma requisição — do clique do usuário até a resposta na tela — é fundamental para depurar problemas de performance e segurança em produção.

① Cliente envia requisição

O navegador resolve o DNS de example.com, abre uma conexão TCP/TLS e envia a requisição HTTP:

GET /api/users HTTP/1.1
Host: example.com
Accept: application/json

② Nginx recebe e roteia

O proxy reverso aceita na porta 443, termina o TLS e encaminha a requisição para o app server via proxy_pass na rede interna:

proxy_pass http://127.0.0.1:8000;
proxy_set_header X-Real-IP $remote_addr;

③ App Server processa

Gunicorn / Node.js / PHP-FPM executa a lógica de negócio e consulta o banco:

SELECT id, nome FROM users
WHERE ativo = true
LIMIT 50;

④ Resposta volta ao cliente

DB retorna dados → app server serializa em JSON → Nginx adiciona cabeçalhos de cache e segurança → cliente recebe:

HTTP/1.1 200 OK
Content-Type: application/json

[{"id":1,"nome":"Ana"},...]

Conceitos Fundamentais

Além do roteamento básico de requisições, servidores web modernos fornecem funcionalidades transversais que simplificam a arquitetura da aplicação como um todo.

Virtual Hosting

Múltiplos domínios no mesmo IP/porta. O servidor diferencia pelo cabeçalho Host da requisição. Permite hospedar centenas de sites em um único servidor.

TLS Termination

O servidor web decifra HTTPS antes de encaminhar ao app server em HTTP simples na rede interna. Centraliza certificados e elimina carga de criptografia no app.

Keep-Alive / Multiplexing

Keep-Alive reutiliza a conexão TCP para múltiplas requisições, eliminando o overhead do 3-way handshake. HTTP/2 vai além: múltiplas streams paralelas na mesma conexão.

Logs

Access log registra toda requisição (IP, método, URL, status, bytes, user-agent). Error log registra falhas de configuração e erros de app. Essenciais para troubleshooting e auditoria de segurança.
