Nginx

Arquitetura event-driven, proxy reverso, upstream, SSL termination e rate limiting

O Nginx nasceu em 2004 como solução para o problema C10K — servir 10 000 conexões simultâneas. Sua arquitetura event-driven assíncrona o torna ideal para proxy reverso, load balancer, TLS termination e entrega de conteúdo estático em alta escala.

Arquitetura Master/Worker

O Nginx usa um processo master mais N worker processes (um por core de CPU por padrão). Cada worker é single-threaded e gerencia milhares de conexões simultâneas via epoll no Linux sem bloquear.

$ ps aux | grep nginx
root     12345  nginx: master process /usr/sbin/nginx
www-data 12346  nginx: worker process
www-data 12347  nginx: worker process
www-data 12348  nginx: cache manager process

Master Process

Roda como root — lê a configuração, abre portas privilegiadas (80, 443) e cria os worker processes. Não processa requisições diretamente.

Worker Processes

Rodam como www-data. Cada worker usa epoll() no Linux para gerenciar milhares de conexões simultâneas em um único loop de eventos.

Cache Manager/Loader

Processos auxiliares que gerenciam o cache em disco — carregam metadados ao iniciar e expiram entradas obsoletas em background.

Estrutura do nginx.conf

O nginx.conf é organizado em contextos aninhados: main → events → http → server → location. Cada diretiva é válida apenas no contexto correto.

/etc/nginx/nginx.conf
worker_processes  auto;
error_log  /var/log/nginx/error.log warn;

events {
    worker_connections 1024;
    use epoll;
    multi_accept on;
}

http {
    include       /etc/nginx/mime.types;
    default_type  application/octet-stream;

    log_format  main  '$remote_addr - $remote_user [$time_local] "$request" '
                      '$status $body_bytes_sent "$http_referer"';
    access_log  /var/log/nginx/access.log  main;

    sendfile        on;
    tcp_nopush      on;
    keepalive_timeout  65;
    gzip  on;
    gzip_types text/plain text/css application/json application/javascript;

    include /etc/nginx/sites-enabled/*;
}

Server Block e Location

O server block define um virtual host. O location mapeia padrões de URL para comportamentos — serve arquivos, faz proxy reverso ou aplica regras de cache e rate limiting.

Servidor Estático + HTTPS

sites-available/static
server {
    listen 80;
    server_name example.com;
    return 301 https://$host$request_uri;
}

server {
    listen 443 ssl http2;
    server_name example.com;
    root /var/www/html;

    ssl_certificate     /etc/letsencrypt/.../fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/.../privkey.pem;
    ssl_protocols       TLSv1.2 TLSv1.3;

    location / {
        try_files $uri $uri/ /index.html;
    }

    location ~* \.(css|js|woff2|png)$ {
        expires 6M;
        add_header Cache-Control "public, immutable";
    }
}

Proxy Reverso para App

sites-available/app
upstream backend {
    server 127.0.0.1:8000;
    server 127.0.0.1:8001;
    keepalive 32;
}

server {
    listen 443 ssl http2;
    server_name api.example.com;

    location / {
        proxy_pass         http://backend;
        proxy_http_version 1.1;
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_set_header   X-Forwarded-Proto $scheme;
        proxy_read_timeout 60s;
    }
}

FastCGI (PHP-FPM)

sites-available/php
server {
    root /var/www/php;

    location ~ \.php$ {
        include        fastcgi_params;
        fastcgi_pass   unix:/run/php/php8.2-fpm.sock;
        fastcgi_index  index.php;
        fastcgi_param  SCRIPT_FILENAME
                       $document_root$fastcgi_script_name;
    }

    location ~ /\.ht { deny all; }
}

Rate Limiting

O Nginx implementa rate limiting via limit_req_zone (no contexto http{}) e limit_req (nos location blocks). Fundamental para proteção contra brute force e abusos de API.

nginx.conf — http context
# Definir zona — no contexto http{}
limit_req_zone $binary_remote_addr zone=api:10m rate=10r/s;

server {
    location /api/ {
        limit_req zone=api burst=20 nodelay;
        proxy_pass http://backend;
    }

    location /login {
        limit_req zone=api burst=5;
        proxy_pass http://backend;
    }
}

Cabeçalhos de Segurança

Adicione estes cabeçalhos no bloco server ou em /etc/nginx/conf.d/security.conf para endurecimento básico em todos os virtual hosts. server_tokens off oculta a versão do Nginx no cabeçalho Server.

server block
server_tokens off;

add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
add_header X-Content-Type-Options    "nosniff" always;
add_header X-Frame-Options           "SAMEORIGIN" always;
add_header Referrer-Policy           "strict-origin-when-cross-origin" always;
add_header Content-Security-Policy   "default-src 'self'" always;

Comandos de Gerenciamento

Sempre valide a configuração com nginx -t antes de qualquer reload. Um erro de sintaxe em produção sem testes prévios pode derrubar o serviço.

Bash
nginx -t                  # testar config (SEMPRE antes de reload!)
nginx -s reload           # reload graceful
systemctl restart nginx   # restart completo
nginx -T 2>/dev/null | head -50  # dump config
tail -f /var/log/nginx/access.log
tail -f /var/log/nginx/error.log
nginx -s quit             # graceful shutdown
nginx -s reopen           # reopen log files
