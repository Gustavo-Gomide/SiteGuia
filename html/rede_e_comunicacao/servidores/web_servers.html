Arquiteturas de Processamento

Apache e Nginx representam as duas principais filosofias de arquitetura de servidores web: múltiplos processos/threads vs. loop de eventos assíncrono.

Apache — MPMs

MPMModeloProcessosThreads/processoQuando usar
prefork1 processo por connN1mod_php legado, bibliotecas não thread-safe
worker1 thread por connPoucosNConteúdo dinâmico, melhor performance que prefork
eventAsync + thread poolPoucosN (+ keep-alive thread)Alta concorrência, equivalente a Nginx

Nginx — Arquitetura Master/Worker

Processos Nginx
Master Process (root)
├── Worker 1  (CPU 0)  — event loop, aceita conexões, lê/escreve sem bloquear
├── Worker 2  (CPU 1)
├── Worker 3  (CPU 2)
└── Cache Manager / Cache Loader

Cada worker é single-threaded mas usa epoll (Linux) / kqueue (macOS/BSD) para gerenciar milhares de conexões simultâneas sem bloquear o processo. worker_processes auto define um worker por core de CPU.

Virtual Hosting — Configuração Básica

Virtual hosting permite servir múltiplos domínios a partir do mesmo servidor. Cada servidor tem sua própria sintaxe, mas o conceito é idêntico: diferenciar domínios pelo cabeçalho Host da requisição.

Apache — VirtualHost

/etc/apache2/sites-available/exemplo.conf
<VirtualHost *:443>
    ServerName  exemplo.com
    DocumentRoot /var/www/exemplo

    SSLEngine on
    SSLCertificateFile    /etc/ssl/certs/cert.pem
    SSLCertificateKeyFile /etc/ssl/private/key.pem

    <Directory /var/www/exemplo>
        AllowOverride All
        Require all granted
    </Directory>
</VirtualHost>

Nginx — server block

/etc/nginx/sites-available/exemplo
server {
    listen 443 ssl http2;
    server_name exemplo.com;
    root /var/www/exemplo;

    ssl_certificate     /etc/ssl/certs/cert.pem;
    ssl_certificate_key /etc/ssl/private/key.pem;

    location / {
        try_files $uri $uri/ =404;
    }
}

Caddy — Caddyfile

Caddyfile
exemplo.com {
    root * /var/www/exemplo
    file_server
    # TLS automático via Let's Encrypt!
}

api.exemplo.com {
    reverse_proxy localhost:8080
}

Qual Escolher?

Não existe bala de prata — a escolha ideal depende do stack, do volume de tráfego e das necessidades operacionais do projeto.

Use Apache quando…

  • Precisa de .htaccess por diretório
  • Herança de configs por usuário / shared hosting
  • Stack PHP legado com mod_php
  • Quantidades moderadas de tráfego

Use Nginx quando…

  • Proxy reverso / load balancer
  • Alta concorrência e baixo consumo de memória
  • Servir arquivos estáticos em grande escala
  • TLS termination para múltiplos backends

Use Caddy quando…

  • Quer HTTPS automático sem scripts extras
  • Projeto pessoal / startups / dev
  • HTTP/3 nativo em produção
  • Config simples, zero boilerplate

Use IIS quando…

  • Stack Windows + ASP.NET / .NET Core
  • Integração com Active Directory
  • Ambientes corporativos Microsoft
