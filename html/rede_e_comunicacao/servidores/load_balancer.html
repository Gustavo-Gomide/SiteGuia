Load Balancer

Algoritmos de distribuição, L4 vs L7, health checks, HAProxy, Nginx upstream e soluções cloud

Um load balancer distribui o tráfego entre múltiplos servidores de backend para maximizar throughput, minimizar latência e garantir alta disponibilidade. É a peça-chave na escalabilidade horizontal de aplicações web.

Algoritmos de Balanceamento

Round Robin

Distribuição em sequência circular. Melhor para backends homogêneos.

upstream backend {
    server 10.0.0.1:8000;
    server 10.0.0.2:8000;
    # padrão no Nginx
}

Weighted Round Robin

Servidores com maior peso recebem mais requisições. Melhor para backends com capacidades diferentes.

upstream backend {
    server 10.0.0.1:8000 weight=3;
    server 10.0.0.2:8000 weight=1;
}

Least Connections

Envia para o servidor com menos conexões ativas. Melhor para backends com tempo de resposta variável.

upstream backend {
    least_conn;
    server 10.0.0.1:8000;
    server 10.0.0.2:8000;
}

IP Hash

A mesma origem IP sempre vai para o mesmo backend (sticky sem cookie). Melhor para apps com sessão em memória local.

upstream backend {
    ip_hash;
    server 10.0.0.1:8000;
    server 10.0.0.2:8000;
}

Random Two Choices

Seleciona dois servidores aleatórios e encaminha para o de menor carga. Melhor para grandes clusters.

upstream backend {
    random two least_conn;
    server 10.0.0.1:8000;
    server 10.0.0.2:8000;
}

Backup / Failover

Servidor backup só recebe tráfego se todos os primários estiverem offline.

upstream backend {
    server 10.0.0.1:8000;
    server 10.0.0.2:8000;
    server 10.0.0.3:8000 backup;
}

Health Checks

Nginx — Passivo (OSS)

upstream backend {
    server 10.0.0.1:8000
        max_fails=3 fail_timeout=30s;
    server 10.0.0.2:8000
        max_fails=3 fail_timeout=30s;
}
# 3 falhas em 30s → marca como down
# Tenta novamente após 30s

HAProxy — Health Check Ativo

backend web_servers
    balance roundrobin
    option httpchk GET /health HTTP/1.1\r\nHost:\ localhost
    http-check expect status 200
    server web1 10.0.0.1:8000 check inter 5s rise 2 fall 3
    server web2 10.0.0.2:8000 check inter 5s rise 2 fall 3
# inter 5s = checar a cada 5s
# rise 2   = 2 OK → up
# fall 3   = 3 falhos → down

HAProxy — Configuração Completa

/etc/haproxy/haproxy.cfg
global
    log /dev/log local0
    maxconn 50000
    user haproxy
    group haproxy
    daemon

defaults
    mode    http
    log     global
    option  httplog
    option  forwardfor
    timeout connect 5s
    timeout client  30s
    timeout server  30s

frontend http_in
    bind *:80
    redirect scheme https code 301 if !{ ssl_fc }

frontend https_in
    bind *:443 ssl crt /etc/ssl/haproxy.pem
    default_backend web_servers
    acl is_api   path_beg /api/
    use_backend  api_servers if is_api

backend web_servers
    balance roundrobin
    option httpchk GET /health
    http-check expect status 200
    server web1 10.0.0.1:8000 check
    server web2 10.0.0.2:8000 check
    server web3 10.0.0.3:8000 check backup

backend api_servers
    balance leastconn
    server api1 10.0.1.1:9000 check
    server api2 10.0.1.2:9000 check

listen stats
    bind *:8080
    stats enable
    stats uri /haproxy_stats
    stats auth admin:senha_secreta

Sticky Sessions

Sticky Sessions garantem que o mesmo cliente sempre seja roteado para o mesmo backend. A solução recomendada é usar sessões centralizadas (Redis) para evitar a necessidade de sticky sessions — pois se um servidor falha, todos os usuários "stickados" perdem a sessão.
