Conceitos Básicos de Redes

Do bit ao pacote — entenda a linguagem comum que todo profissional de redes precisa dominar antes de qualquer outra coisa

Antes de mergulhar em protocolos e configurações, é essencial dominar o vocabulário fundamental das redes. Termos como largura de banda, latência, PDU, unicast e NAT aparecem em todo tipo de conversa técnica — de troubleshooting a design de infraestrutura. Esta página explica cada conceito com precisão, sem ambiguidades.

Métricas de Performance

Largura de Banda (Bandwidth)

É a capacidade máxima teórica de um link de rede — quanto dado pode ser transmitido por unidade de tempo. Medida em bits por segundo (bps, Kbps, Mbps, Gbps).

  • Um cabo Cat6 suporta até 1 Gbps — essa é sua largura de banda
  • Fibra óptica monomodo pode chegar a 400 Gbps (e além com WDM)
  • Não confunda: 100 Mbps de banda não significa 100 MB/s de download — bits ≠ bytes (divida por 8: 100 Mbps ÷ 8 ≈ 12,5 MB/s)
  • Largura de banda é compartilhada: 10 usuários em 100 Mbps = 10 Mbps esperado cada

Throughput (Vazão Real)

É a taxa de transferência efetivamente alcançada em condições reais. Sempre menor que a largura de banda teórica, descontados overhead de protocolos, colisões, retransmissões, congestionamento e limitações do hardware.

  • Link de 1 Gbps com 20% de overhead TCP/IP → throughput ≈ 800 Mbps
  • Ferramentas de medição: iperf3 (melhor), speedtest-cli
  • Em redes Wi-Fi, throughput pode ser 40–60% da banda nominal
  • O gargalo do throughput é o elo mais lento da cadeia (princípio de Amdahl)

Latência (RTT)

É o tempo de ida e volta (Round-Trip Time) de um pacote entre dois pontos. Medida em milissegundos (ms). Completamente independente da largura de banda.

  • Fibra SP→RJ: ~5 ms • SP→EUA: ~120 ms • Satélite GEO: ~600 ms
  • Latência alta arruína aplicações interativas (jogos, VoIP, RDP) mesmo com banda alta
  • Componentes: propagação (velocidade da luz), processamento (roteadores), fila, transmissão
  • Medição: ping (ICMP), traceroute / tracert por salto

Jitter e Perda de Pacotes

Jitter é a variação na latência ao longo do tempo. Perda de pacote é a porcentagem de pacotes que não chegam ao destino.

  • VoIP tolera até ~30ms de jitter e <1% de perda antes de ficar ininteligível
  • Videoconferência (Zoom, Meet): jitter <50ms e perda <2% para qualidade aceitável
  • TCP retransmite pacotes perdidos — visível como queda de throughput, não perda de dados
  • UDP não retransmite — perda = artefato visível/audível
  • Medição: mtr (My Traceroute), iperf3 --udp

PDUs — Unidades de Dados por Protocolo

PDU (Protocol Data Unit) é o nome dado ao bloco de dados em cada camada da pilha. O mesmo payload recebe nomes diferentes conforme desce pela pilha TCP/IP.

Tipos de Comunicação

Unicast

Comunicação de um para um: um pacote para um único destinatário identificado pelo IP/MAC específico. É o mais comum — 99% do tráfego normal da internet.

  • IPv4: qualquer endereço unicast (ex: 192.168.1.10)
  • IPv6: endereços começando com qualquer prefixo que não seja ff::/8
  • Exemplos: HTTP, SSH, SMTP, uma chamada de vídeo ponto-a-ponto

Broadcast

Comunicação de um para todos na mesma rede local. O pacote é enviado para um endereço especial e todos os dispositivos recebem e processam.

  • IPv4 broadcast limitado: 255.255.255.255 — todos na mesma sub-rede
  • IPv4 broadcast direcionado: 192.168.1.255/24 — todos na 192.168.1.0/24
  • Ethernet broadcast: FF:FF:FF:FF:FF:FF
  • Uso: ARP Request, DHCP Discover, NetBIOS
  • IPv6 não tem broadcast — substituído por multicast (FF02::1 = todos os nós)

Multicast

Comunicação de um para muitos selecionados: pacote enviado para um grupo de assinantes sem duplicar o tráfego em cada link.

  • IPv4 multicast: faixa 224.0.0.0/4 (224.x.x.x a 239.x.x.x)
  • Link-local: 224.0.0.1 (todos hosts), 224.0.0.5 (todos roteadores OSPF)
  • IPv6 multicast: prefixo FF00::/8
  • Protocolo de assinatura: IGMP (IPv4), MLD (IPv6)
  • Uso: streaming IPTV, videoconferência em escala, protocolos de roteamento, OSPF hello

Anycast

Comunicação de um para o mais próximo de muitos: o mesmo endereço IP é anunciado por múltiplos servidores e o roteamento direciona o cliente ao mais próximo.

  • Nativo no IPv6; também usado em IPv4 com BGP
  • Uso: DNS raiz (13 servidores raiz atendem o mundo via anycast), CDNs (Cloudflare 1.1.1.1, Google 8.8.8.8)
  • Benefícios: latência reduzida, resiliência a falhas, distribuição de carga geográfica

NAT — Network Address Translation

O NAT permite que múltiplos dispositivos com IPs privados compartilhem um único IP público. É a razão pela qual você tem 10+ dispositivos em casa com um único IP de operadora. Funciona no roteador/firewall, modificando os IPs e portas nos cabeçalhos IP/TCP/UDP em tempo real.

Tipos de NAT

  • NAT Estático (Static NAT): mapeamento 1:1 fixo entre IP privado e IP público. Usado para servidores que precisam ser acessíveis externamente.
  • NAT Dinâmico: mapeia IPs privados para um pool de IPs públicos. Conexões de entrada não são possíveis para IPs sem mapeamento ativo.
  • PAT / NAT Overload (masquerade): muitos IPs privados para um único IP público, diferenciados pela porta TCP/UDP. O mais comum em roteadores domésticos.
  • NAT64: tradução entre IPv6 e IPv4 para redes em transição.

DHCP — Configuração Automática de IP

O DHCP (Dynamic Host Configuration Protocol) distribui automaticamente configurações de rede para os clientes: IP, máscara de sub-rede, gateway padrão, servidores DNS e muito mais. Sem ele, cada dispositivo precisaria ser configurado manualmente.

Processo DORA — como o DHCP distribui IPs

4 mensagens UDP: Discover (67) Offer (68) Request (67) Acknowledge (68)

D — DHCP Discover

Quem envia: Cliente (novo dispositivo na rede)

Como: Broadcast (255.255.255.255, porta 67 UDP) — o cliente ainda não tem IP, então manda para todos.

Conteúdo: "Sou o MAC AA:BB:CC:DD:EE:FF, precisa de um IP. Tem algum servidor DHCP aqui?"

O — DHCP Offer

Quem envia: Servidor DHCP

Como: Unicast ou broadcast para o MAC do cliente (porta 68 UDP)

Conteúdo: "Oferço o IP 192.168.1.50 por 24h, gateway 192.168.1.1, DNS 8.8.8.8 e 8.8.4.4." O servidor reserva temporariamente o IP oferecido.

R — DHCP Request

Quem envia: Cliente

Como: Broadcast (ainda não confirmou que tem o IP)

Conteúdo: "Aceito a oferta do servidor 192.168.1.1 — quero o 192.168.1.50." O broadcast avisa outros servidores DHCP que a oferta deles não foi aceita, para liberarem a reserva.

A — DHCP Acknowledge

Quem envia: Servidor DHCP

Como: Unicast ou broadcast para o MAC

Conteúdo: "Confirmado! 192.168.1.50/24, gateway 192.168.1.1, DNS 8.8.8.8, lease de 86400 segundos (24h)." O cliente configura a interface e já pode comunicar na rede.

Renovação do Lease

Quando: Em 50% do tempo do lease (T1), o cliente tenta renovar com unicast para o mesmo servidor.

Se não responder, em 87,5% (T2) tenta broadcast. Se ainda assim falhar ao final do lease, o cliente perde o IP e reinicia o processo DORA do zero.

Comando útil: ipconfig /release + ipconfig /renew (Windows) ou dhclient -r && dhclient (Linux)

ARP — Resolvendo IP para MAC

Quando você tem o IP de destino mas está na mesma rede local, precisa do endereço MAC para montar o frame Ethernet. O ARP (Address Resolution Protocol) resolve essa tradução.

Processo ARP

Fluxo ARP — quem tem 192.168.1.1?
PC-A (192.168.1.10)  →  Broadcast Ethernet FF:FF:FF:FF:FF:FF
ARP Request: "Quem tem IP 192.168.1.1? Me diga AA:BB:CC:DD:EE:FF"

Gateway (192.168.1.1)  →  Unicast para AA:BB:CC:DD:EE:FF
ARP Reply: "Sou eu! Meu MAC é 11:22:33:44:55:66"

PC-A armazena em cache:
  192.168.1.1  →  11:22:33:44:55:66  (válido por ~20 min)

Visualizar cache ARP:
  Windows: arp -a
  Linux:   ip neigh show  ou  arp -n

ARP na prática — para fora da rede

Se o destino estiver em outra rede (IP diferente do prefixo local), o ARP não pergunta sobre o IP remoto. Em vez disso:

  1. PC verifica: IP destino está na minha rede? (aplica a máscara)
  2. Não está → envia o frame para o MAC do gateway padrão
  3. ARP resolve o MAC do gateway, não do destino remoto
  4. O IP destino no pacote permanece 93.184.216.34, mas o frame vai para o MAC do roteador local
ARP Gratuito (Gratuitous ARP): ao ganhar um IP, o host anuncia seu próprio mapeamento IP→MAC em broadcast. Serve para atualizar caches e detectar IP duplicado (se alguém responder, há conflito).

DNS — Resolução de Nomes

DNS (Domain Name System) é a "lista telefônica" da internet: converte nomes legíveis (www.exemplo.com) em endereços IP. É hierárquico, distribuído e usa cache para escalabilidade global.

Hierarquia DNS

Estrutura do nome www.exemplo.com.br
. (root)            ← Raiz: 13 clusters de servidores raiz (anycast)
└── .br             ← TLD: Top Level Domain (Registro.br)
    └── .com.br     ← SLD: Second Level Domain
        └── exemplo ← Domínio registrado (você controla)
            └── www ← Subdomínio (host / A record)

Processo de Resolução Recursiva

Resolvendo www.exemplo.com.br (sem cache)
1. Browser verifica cache local (TTL ainda válido?)
2. Pergunta ao resolver recursivo do provedor/DHCP (ex: 8.8.8.8)
3. Resolver pergunta ao servidor Raiz: "quem é .br?"
4. Raiz responde: "pergunte ao servidor TLD 200.160.0.10"
5. Resolver pergunta ao TLD .br: "quem é exemplo.com.br?"
6. TLD responde: "pergunte ao NS do domínio: ns1.exemplo.com.br"
7. Resolver pergunta ao NS autoritativo: "qual o IP de www?"
8. NS responde: "A record: 177.10.20.30, TTL=3600"
9. Resolver devolve 177.10.20.30 ao browser e guarda em cache
10. Browser abre conexão TCP para 177.10.20.30:443

Modelos de Comunicação

Cliente-Servidor

O modelo dominante na web: clientes solicitam, servidores respondem. Os papéis são assimétricos e fixos.

  • Servidor: sempre disponível, endereço fixo (IP + porta bem conhecida), processa requisições de muitos clientes simultâneos
  • Cliente: inicia conexão, endereço efêmero, geralmente atrás de NAT
  • Exemplos: navegador (cliente HTTP) ↔ Apache/Nginx (servidor), Outlook (cliente SMTP) ↔ Exchange (servidor)
  • Escalabilidade: load balancers distribuem clientes entre múltiplos servidores

Peer-to-Peer (P2P)

Cada nó é simultaneamente cliente e servidor. Sem servidor central — distribuição de carga e redundância inerentes.

  • Exemplos: BitTorrent, Bitcoin/blockchain, WebRTC (videoconferência ponto-a-ponto), IPFS
  • Vantagens: sem ponto único de falha, escala com o número de participantes
  • Desafios: descoberta de peers (geralmente usa um tracker ou DHT), NAT traversal (STUN/TURN/ICE), sem controle central
  • Híbrido: WhatsApp usa servidores para sinalização mas trafega mídia P2P via WebRTC

Ferramentas de Diagnóstico

  • ping / ping6 — latência e perda (ICMP)
  • traceroute / tracert — caminho e latência por salto
  • arp -a / ip neigh — cache ARP
  • nslookup / dig — consulta DNS manual
  • ipconfig /all (Win) / ip addr (Linux) — config da interface
  • iperf3 -s / iperf3 -c host — throughput entre dois pontos
  • mtr host — traceroute + ping contínuo (mostra jitter)
  • Wireshark — análise completa de pacotes em tempo real
