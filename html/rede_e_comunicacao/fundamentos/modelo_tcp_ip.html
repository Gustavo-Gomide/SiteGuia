Modelo TCP/IP

O modelo que realmente implementa a internet — 4 camadas práticas que definem como cada pacote trafega do seu computador até o mundo

Enquanto o OSI é um modelo de referência teórico, o TCP/IP (Transmission Control Protocol / Internet Protocol) é o modelo que a internet realmente usa. Desenvolvido pelo DARPA nos anos 1970 e formalizado na RFC 1122 (1989), ele organiza a comunicação em 4 camadas: Acesso à Rede, Internet, Transporte e Aplicação. Cada protocolo que você usa diariamente — HTTP, DNS, TLS, SMTP — vive e opera neste modelo.

TCP/IP vs OSI: O TCP/IP é anterior ao OSI e foi design-by-implementation (surgiu da ARPANET). O OSI foi design-by-committee (ISO). Na prática, o TCP/IP ganhou. O OSI sobreviveu como linguagem técnica para descrever onde cada protocolo opera.

As 4 Camadas do TCP/IP

Camada 4 — Aplicação

Equivale ao OSI: Camadas 5, 6 e 7 • PDU: Mensagem / Dados

É onde vivem os protocolos que os usuários interagem diretamente. Inclui a lógica de apresentação (codificação, criptografia TLS) e gerência de sessão. Toda a complexidade das camadas 5, 6 e 7 do OSI está aqui.

  • Web: HTTP/1.1, HTTP/2, HTTP/3 (sobre QUIC)
  • Segurança: TLS 1.2 / TLS 1.3 (antes era SSL)
  • DNS: resolução de nomes para IPs (UDP/TCP porta 53)
  • Email: SMTP (25/587), IMAP (143/993), POP3 (110/995)
  • Acesso remoto: SSH (22), RDP (3389), Telnet (23, inseguro)
  • Arquivos: FTP (21/20), SFTP, TFTP (69 UDP), SCP
  • Config automática: DHCP (67/68 UDP), NTP (123 UDP)
  • Gerência: SNMP (161 UDP), Syslog (514 UDP)
  • Streaming: RTP, RTSP, WebRTC

Camada 3 — Transporte

Equivale ao OSI: Camada 4 • PDU: Segmento (TCP) / Datagrama (UDP)

Responsável pela comunicação fim-a-fim entre processos. Enquanto o IP leva o pacote entre redes, o Transporte garante que chegue ao processo correto e (no TCP) que chegue de forma confiável.

  • TCP: orientado a conexão, handshake 3-way, ACK, retransmissão, controle de fluxo (janela deslizante), controle de congestionamento (slow start, AIMD)
  • UDP: connectionless, sem garantias, overhead mínimo. Ideal para DNS, DHCP, streaming, jogos online, VoIP
  • SCTP: combina confiabilidade do TCP com suporte a multistreaming; usado em telecomunicações (SS7 sobre IP)
  • QUIC: UDP + confiabilidade + criptografia integrada (TLS 1.3). Base do HTTP/3. Elimina o HOL blocking do TCP
  • Portas de 0 a 65535: <1024 são "well-known" (requerem privilégio root), 1024–49151 são registradas, >49151 são efêmeras (cliente)

Camada 2 — Internet

Equivale ao OSI: Camada 3 • PDU: Pacote / Datagrama IP

O coração do modelo: roteamento de pacotes entre redes distintas usando endereços IP lógicos. É aqui que a internetworking acontece — a capacidade de conectar redes heterogêneas.

  • IPv4: endereços de 32 bits (4,3 bilhões de endereços), header de 20 bytes mínimo, TTL, fragmentação
  • IPv6: endereços de 128 bits (340 undecilhões), header simplificado de 40 bytes fixos, sem fragmentação no roteador, autoconfiguração (SLAAC)
  • ICMP: mensagens de controle — Echo (ping), Destination Unreachable, Time Exceeded (base do traceroute)
  • ICMPv6: além do controle, inclui NDP (Neighbor Discovery Protocol — substitui ARP no IPv6)
  • ARP: resolve IP → MAC na mesma rede local (opera na fronteira camadas 2 e 3)
  • Protocolos de roteamento: OSPF (intra-AS, link-state), BGP-4 (inter-AS, path-vector, a "cola" da internet), RIP (legado, distance-vector)

Camada 1 — Acesso à Rede

Equivale ao OSI: Camadas 1 e 2 • PDU: Frame / Bits

Tudo que diz respeito ao meio físico e ao enlace de dados local. O TCP/IP não especifica essa camada — ela é propositalmente agnóstica para funcionar sobre qualquer tecnologia de enlace.

  • Ethernet (IEEE 802.3): par trançado (10Mbps a 400Gbps), endereçamento MAC, CSMA/CD (obsoleto em full-duplex), frame com CRC
  • Wi-Fi (IEEE 802.11): 802.11a/b/g/n/ac(Wi-Fi 5)/ax(Wi-Fi 6)/be(Wi-Fi 7), CSMA/CA, OFDM
  • PPP: ponto-a-ponto (links seriais, DSL antigo, VPN PPTP)
  • Fibra óptica: multimodo (até ~2 km) e monomodo (até 80+ km); padrões 10GBase-LR, 100GBase-SR4
  • Celular: LTE (4G), NR (5G) — o IP opera sobre a pilha de radiofrequência
  • Responsável por: endereçamento MAC, controle de acesso ao meio, detecção de erro (CRC), VLANs (802.1Q)

A Jornada de um Pacote

O que acontece quando você digita https://exemplo.com no navegador? Acompanhe o pacote descendo e subindo a pilha TCP/IP.

Passo a passo — requisição HTTPS

Da digitação na barra de endereço até o HTML chegar de volta ao navegador

1 — Resolução DNS

Camada: Aplicação (UDP/TCP porta 53)

O navegador não tem IP de exemplo.com em cache. Envia consulta DNS ao servidor configurado (ex: 8.8.8.8). Recebe 93.184.216.34 como resposta. Todo esse tráfego usa UDP datagrama — sem handshake.

2 — TCP 3-Way Handshake

Camada: Transporte (TCP porta 443)

Navegador envia SYN para 93.184.216.34:443. Servidor responde SYN-ACK. Navegador confirma com ACK. Conexão estabelecida — 1 RTT (round-trip time) consumido antes de qualquer dado HTTP.

3 — TLS Handshake

Camada: Aplicação (TLS 1.3)

Navegador e servidor negociam cipher suite, trocam chaves (ECDHE), autenticam certificado X.509 do servidor. TLS 1.3 faz isso em 1-RTT (vs 2-RTT no TLS 1.2). O canal agora é criptografado.

4 — Requisição HTTP

Camada: Aplicação (HTTP/2)

Navegador envia GET / HTTP/2 com headers (Host, Accept, User-Agent, Cookie...). O HTTP/2 usa frames binários e multiplexação — múltiplos recursos em paralelo pela mesma conexão TCP.

5 — Encapsulamento na pilha

Camadas: Transporte → Internet → Acesso

TCP encapsula em segmento (porta src efêmera + dst:443 + seq numbers). IP encapsula em pacote (IP src:10.0.0.5 + IP dst:93.184.216.34 + TTL:64). Ethernet encapsula em frame (MAC src + MAC do gateway + CRC).

6 — Roteamento

Camada: Internet (IP + BGP/OSPF)

Cada roteador no caminho lê o IP destino, consulta sua tabela de rotas e encaminha. TTL decrementa a cada salto. O frame Ethernet é substituído em cada salto (novo MAC src/dst). O pacote IP permanece intacto.

7 — Resposta HTTP

Camada: Aplicação

Servidor processa, monta 200 OK com o HTML. TCP garante entrega ordenada com ACKs. Se o HTML referencia CSS/JS/imagens, HTTP/2 multiplexação os serve em paralelo. Browser monta a página com os recursos recebidos.

Análise de Cabeçalhos

Cabeçalho IPv4 — 20 bytes mínimo

Campos do cabeçalho IP (RFC 791)
Versão (4b) | IHL (4b) | DSCP (6b) | ECN (2b) | Comprimento Total (16b)
Identificação (16b)             | Flags (3b) | Fragment Offset (13b)
TTL (8b)       | Protocolo (8b)             | Checksum (16b)
IP de Origem                 (32 bits)
IP de Destino                (32 bits)
Opções (variável, se IHL > 5)                     | Padding

Exemplos de campo Protocolo:
  6  = TCP  |  17 = UDP  |  1 = ICMP  |  89 = OSPF  |  50 = ESP (IPsec)

Cabeçalho TCP — 20 bytes mínimo

Campos do cabeçalho TCP (RFC 793)
Porta de Origem (16b)       | Porta de Destino (16b)
Número de Sequência                   (32 bits)
Número de Acknowledgment              (32 bits)
Data Offset (4b) | Reserved | Flags (9b) | Janela (16b)
Checksum (16b)              | Ponteiro Urgente (16b)
Opções (se Data Offset > 5)

Flags TCP:
  SYN  = inicia conexão
  ACK  = confirma recebimento
  FIN  = encerra conexão (graceful)
  RST  = encerra conexão (forçado/erro)
  PSH  = envia dados imediatamente sem buffer
  URG  = dados urgentes (raramente usado)

Cabeçalho IPv6 — 40 bytes fixos

Campos do cabeçalho IPv6 (RFC 8200)
Versão (4b) | Traffic Class (8b) | Flow Label (20b)
Payload Length (16b)    | Next Header (8b) | Hop Limit (8b)
Endereço de Origem               (128 bits / 4 linhas de 32b)
Endereço de Destino              (128 bits / 4 linhas de 32b)

Diferenças-chave vs IPv4:
  - Sem checksum no cabeçalho (responsabilidade das camadas superiores)
  - Sem fragmentação pelo roteador (host fragmenta se necessário)
  - "Next Header" substitui "Protocol" e suporta encadeamento de Extension Headers
  - Hop Limit substitui TTL (mesmo conceito, nome diferente)

TCP 3-Way Handshake — em detalhe

Estabelecimento e encerramento de conexão TCP
--- ESTABELECIMENTO (3-Way Handshake) ---
Cliente                           Servidor
   |--- SYN (seq=x) ------------->|   Cliente quer conectar
   |<-- SYN-ACK (seq=y, ack=x+1) -|   Servidor aceita, confirma
   |--- ACK (ack=y+1) ----------->|   Cliente confirma. Conexão aberta.

--- TRANSFERÊNCIA DE DADOS ---
   |--- [dados] (seq=x+1) ------->|
   |<-- ACK (ack=x+1+len) --------|

--- ENCERRAMENTO (4-way FIN) ---
   |--- FIN (seq=a) ------------->|   Cliente terminou
   |<-- ACK (ack=a+1) ------------|   Servidor confirma
   |<-- FIN (seq=b) --------------|   Servidor terminou
   |--- ACK (ack=b+1) ----------->|   Cliente confirma. Conexão fechada.
