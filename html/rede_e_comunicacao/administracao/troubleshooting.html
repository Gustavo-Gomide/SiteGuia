Troubleshooting de Redes

Metodologia OSI de baixo para cima, checklist por camada, ferramentas e casos comuns

Um bom troubleshooting de redes segue uma metodologia sistemática, eliminando camadas de baixo para cima (física → enlace → rede → transporte → aplicação). Diagnosticar a camada errada desperdicia tempo — confirme cada camada antes de avançar.

Metodologia OSI — Bottom-Up

CamadaO que verificarFerramentasSintomas típicos
L1 — Física Cabos, conectors, LEDs de link, módulos SFP, potência óptica Inspeção visual, cable tester, OTDR Interface down, sem link, erros CRC, alta BER
L2 — Enlace Status de interface, VLAN correta, duplex/speed, STP, MAC table ip link, ethtool, show interfaces, show mac address-table Interface up mas sem tráfego, loop, broadcast storm
L3 — Rede IP correto, máscara, gateway, rotas, ARP ip addr, ip route, ping, traceroute, arp Não alcance gateway, TTL exceeded, ICMP unreachable
L4 — Transporte Portas abertas, firewall, nat, timeout de conexão ss -tulpn, nc, telnet host port, nmap Conexão recusada, timeout, RST
L7 — Aplicação DNS resolve, serviço rodando, certificado TLS, proxy, autenticação dig, curl -v, openssl s_client, logs DNS NXDOMAIN, 404/500, TLS handshake fail, auth error

Checklist Passo a Passo

1. Físico

  • Cabo conectado? LED aceso?
  • Interface no switch não está desativada?
  • Velocidade/duplex não há mismatch?
ip link show eth0
ethtool eth0
# procurar: Speed, Duplex, Link detected: yes

2. Enlace / VLAN

  • Interface em VLAN correta?
  • IP obtido via DHCP?
ip addr show eth0
# deve ter inet com IP
# se vazio: IP não configurado

3. Rede / Gateway

# Ver gateway
ip route show default

# Testar gateway
ping -c 4 <GATEWAY_IP>

# Ver tabela ARP (gateway resolvido?)
ip neigh show

4. DNS

# Resolver DNS
dig google.com
nslookup google.com

# DNS configurado?
cat /etc/resolv.conf

# Testar resolver alternativo
dig @8.8.8.8 google.com

5. Conectividade IP

# Ping para IP (sem DNS)
ping -c 4 8.8.8.8

# Rastrear rota
traceroute 8.8.8.8

# Se sem ping mas com DNS → ICMP bloqueado no caminho

6. Porta / Serviço

# Porta aberta localmente?
ss -tulpn | grep :443

# Alcançar porta remota
telnet host 443
nc -zv host 443
nmap -p 443 host

7. Aplicação HTTP/TLS

curl -v https://example.com
# procurar: SSL handshake, HTTP status

openssl s_client -connect example.com:443
# procurar: certificado válido, TLS version

curl -I http://example.com
# Location header?

8. Logs

# System / kernel
journalctl -f
dmesg | tail -20

# Nginx / Apache
tail -f /var/log/nginx/error.log
tail -f /var/log/apache2/error.log

# Firewall
iptables -L -v -n
journalctl -k | grep "IN=" | tail -20

Comandos de Diagnóstico Rápido

Bash — one-liners úteis
# IP atual de todas interfaces
ip -brief addr
# eth0  UP  192.168.1.100/24

# Gateway padrão
ip route show default
# default via 192.168.1.1 dev eth0

# DNS configurado
resolvectl status | grep -i dns

# Qual processo está na porta 80?
ss -tulpn | grep :80
# ou
fuser 80/tcp

# Testar porta de forma rápida
nc -zv host 443 2>&1

# Capturar tráfego para arquivo
tcpdump -i eth0 -w /tmp/cap.pcap &
curl https://target
kill %1

# Ver erros de interface (CRC, drops)
ip -s link show eth0 | grep -A2 RX

# MTU path discovery
ping -M do -s 1472 8.8.8.8
# Se falhar: MTU do caminho < 1500
