Camada de Transporte

TCP — Transmission Control Protocol

Entrega confiável, ordenada e sem erros de dados entre processos em rede

O TCP (RFC 793) é o protocolo de transporte orientado a conexão. Garante que bytes chegam em ordem, sem duplicatas e sem perdas. Usa confirmações (ACK), retransmissão, janela deslizante e controle de congestionamento. É a base de HTTP(S), SSH, SMTP e a maioria das aplicações web.

Campos principais

  • Source/Destination Port (16b cada): identificam o processo remetente/destinatário
  • Sequence Number (32b): número do primeiro byte neste segmento; permite reordenamento
  • Acknowledgment Number (32b): próximo byte esperado (ACK cumulativo)
  • Data Offset (4b): tamanho do header em words de 32 bits (mínimo 5 = 20B)
  • Window (16b): tamanho da janela de recepção; controle de fluxo; ampliado com Window Scaling option (RFC 7323)
  • Checksum (16b): integridade do header + dados + pseudo-header IP

Flags TCP (control bits)

  • SYN: synchronize; inicia conexão; consumido no ISN (Initial Sequence Number)
  • ACK: acknowledgment number é válido
  • FIN: remetente não tem mais dados; inicia encerramento gracioso
  • RST: reset abrupto da conexão; nenhum FIN/ACK
  • PSH: push; receptor deve passar dados à aplicação imediatamente (sem buffer)
  • URG: dado urgente; apontado por Urgent Pointer (raro na prática)

Controle de Fluxo e Congestionamento

Janela Deslizante (Flow Control)

O receptor anuncia o receive window (rwnd) — quanto de buffer está disponível. O remetente não pode ter mais bytes in-flight do que rwnd. Se rwnd = 0, o remetente para e envia probes periódicos.

Controle de Congestionamento

  • Slow Start: cwnd começa em 1 MSS; dobra a cada RTT (exponencial) até o ssthresh
  • Congestion Avoidance: após ssthresh, aumenta 1 MSS por RTT (linear); até perda
  • Fast Retransmit: 3 ACKs duplicados → retransmite sem esperar timeout
  • Fast Recovery (Reno/CUBIC): após fast retransmit, divide ssthresh ao meio e retoma na metade (não volta ao slow start)
  • CUBIC (Linux padrão): função cúbica para crescer cwnd; melhor para redes de alta latência/largura de banda
  • BBR (Google): modelo de rede baseado em BDP (Bandwidth-Delay Product); não baseia em perda; preferído em CDNs e Google
