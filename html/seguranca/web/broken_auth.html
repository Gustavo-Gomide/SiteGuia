Broken Authentication — Falhas de Autenticação

Falhas em login, sessão e recuperação de senha — A07 do OWASP Top 10.

Broken Authentication (A07 no OWASP Top 10 2021) abrange qualquer fraqueza nos mecanismos de verificação de identidade: login, gerenciamento de sessão, recuperação de senha e autenticação multifator. Essas falhas permitem ao atacante assumir identidade de outros usuários — incluindo administradores.

Principais Ataques

Brute Force

Tentativa sistemática de senhas. Sem rate limiting ou bloqueio de conta, o atacante pode testar milhares de senhas por segundo.

# hydra — brute force de login:
hydra -l admin -P /wordlists/rockyou.txt \
  https-post-form "/login:user=^USER^&pass=^PASS^:Invalid credentials"

Credential Stuffing

Uso de listas de usuário:senha vazados de outros serviços. Exploração de reutilização de senhas. Diferente de brute force — usa credenciais reais vazadas.

Password Spraying

Testa uma senha comum (ex: Senha2024!) contra muitos usuários. Evita bloqueio por tentativas — 1 tentativa por conta.

Session Fixation

Atacante força um session ID conhecido antes do login. Quando a vítima autentica, o atacante usa o mesmo ID para acessar a sessão.

Session não regenerada pós-login Mesmo session ID antes e depois do login Viabiliza session fixation
Tokens em URL Session token no query string — URL em logs/referrer Vazamento em logs HTTP / referrer headers

Fluxos Inseguros de Recuperação de Senha

Perguntas de segurança

Respostas são públicas (data de nascimento, nome de pet) ou facilmente enumeráveis. Nunca use como único fator de recuperação.

Token de reset previsível

Links de reset com timestamp + email em MD5. Atacante consegue prever o token para qualquer conta.

Token de reset sem expiração

Link de reset enviado por email que nunca expira. Um email comprometido meses depois ainda permite troca de senha.

Padrão seguro de reset

  • Token aleatório, criptograficamente seguro (mínimo 128 bits)
  • Expiração curta (15–60 minutos)
  • Uso único — invalidar após uso
  • Invalidar tokens anteriores ao gerar novo

Bypass de MFA

# Técnicas comuns de bypass de MFA:

1. Response Manipulation (interceptar com Burp):
   POST /mfa/verify  { "code": "123456" }
   Resposta: {"success": false}  → mudar para {"success": true}

2. Skip do step de MFA:
   Login: POST /auth/login → redireciona para /mfa
   Acessar diretamente /dashboard sem passar por /mfa
   Aplicação não valida se MFA foi completado

3. Código de backup como vetor:
   Códigos de backup armazenados em plaintext no servidor
   Brute force de códigos de 8 dígitos sem rate limiting

4. SIM Swapping (SMS MFA):
   Engenharia social na operadora para transferir número
   Compromete qualquer conta usando SMS como 2FA

Mitigações

Rate Limiting e Account Lockout

Limite tentativas de login (ex: 5 por conta por minuto). Implemente lockout progressivo com captcha. Monitore e alerte para picos de falhas.

MFA robusto

Use TOTP (Google Authenticator, Authy) ou passkeys em vez de SMS. Valide o estado MFA no servidor a cada acesso a recursos protegidos.

Gestão de sessão segura

Regenerar session ID pós-login. Cookies HttpOnly + Secure + SameSite. Timeout de inatividade. Logout invalida server-side.

Verificação de credenciais vazadas

Integrar com Have I Been Pwned API no login ou cadastro para alertar/bloquear senhas comprovadamente vazadas.

Falhas de autenticação são frequentemente o caminho mais direto para comprometimento total de uma aplicação. Credential stuffing automatizado é responsável por bilhões de tentativas de login por dia na internet. Use MFA, monitore anomalias de login (geo-localização, device novo) e implemente detecção de comportamentos suspeitos.

"A autenticação é o portão de entrada — uma vulnerabilidade aqui invalida todo o resto do sistema de segurança."
