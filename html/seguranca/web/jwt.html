JWT — JSON Web Token

Estrutura do JWT

JWT — JSON Web Token

Token compacto em Base64URL para autenticação em APIs REST e OAuth 2.0.

Um JWT é uma string compacta em Base64URL com três partes separadas por ponto: Header.Payload.Signature. É amplamente usado como token de acesso em APIs REST e OAuth 2.0 / OpenID Connect.

# JWT completo (exemplo):
eyJhbGciOiJIUzI1NiIsInR5cCI6IkpXVCJ9.eyJzdWIiOiIxMjM0NTY3ODkwIiwibmFtZSI6IkpvaG4gRG9lIiwiaWF0IjoxNTE2MjM5MDIyfQ.SflKxwRJSMeKKF2QT4fwpMeJf36POk6yJV_adQssw5c

# Decodificado:

# Header (Base64URL decoded):
{ "alg": "HS256", "typ": "JWT" }

# Payload (Base64URL decoded):
{
  "sub": "1234567890",       # subject — ID do usuário
  "name": "John Doe",        # claim customizada
  "iat": 1516239022,         # issued at
  "exp": 1516242622,         # expiration
  "role": "user"             # claim de autorização
}

# Signature (HMAC-SHA256):
HMACSHA256(
  base64url(header) + "." + base64url(payload),
  secret_key
)

O payload de um JWT não é criptografado por padrão — é apenas codificado em Base64URL. Não armazene dados sensíveis no payload sem usar JWE (JSON Web Encryption). Qualquer um que interceptar o token pode ler o conteúdo.

Vulnerabilidades e Ataques

Ataque alg:none

O atacante modifica o header para "alg": "none" e remove a assinatura. Bibliotecas vulneráveis aceitam o token sem verificação.

# Header modificado:
{ "alg": "none", "typ": "JWT" }
# Payload com role escalado:
{ "sub": "123", "role": "admin" }
# Token: eyJ... . eyJ... .   ← assinatura vazia

Algorithm Confusion (RS256→HS256)

O servidor usa RS256 (chave pública/privada). O atacante muda o header para HS256 e assina o token com a chave pública do servidor (que é pública). Bibliotecas vulneráveis verificam o HMAC com a chave pública — aceita o token forjado.

Weak Secret (Brute Force)

Se o segredo HMAC for fraco, pode ser quebrado offline com hashcat:

hashcat -a 0 -m 16500 token.jwt /wordlists/rockyou.txt

JWT sem exp / sem validação

Token sem exp nunca expira. Se o servidor não valida expiração, tokens comprometidos são válidos para sempre.

Boas Práticas

# Python — verificação segura de JWT (PyJWT):
import jwt

try:
    payload = jwt.decode(
        token,
        public_key,          # chave pública RSA / segredo HMAC
        algorithms=["RS256"], # FIXE o algoritmo — não use alg do header
        options={
            "require": ["exp", "iat", "sub"],  # claims obrigatórias
            "verify_exp": True,
        },
        leeway=10,           # tolerância de 10s para clock skew
    )
except jwt.ExpiredSignatureError:
    # Token expirado — rejeitar
except jwt.InvalidAlgorithmError:
    # Algoritmo não permitido — rejeitar
except jwt.InvalidTokenError:
    # Qualquer outro erro — rejeitar

Nunca confie no alg do token

Sempre especifique explicitamente os algoritmos aceitos na verificação. A decisão de qual algoritmo usar é do servidor, não do cliente.

Expiração curta + refresh token

Access tokens com exp curto (15min–1h). Refresh tokens com revogação server-side. Permite invalidar acesso sem blacklist de access tokens.

Segredo HMAC forte

Mínimo 256 bits de entropia gerados com CSPRNG. Nunca use strings legíveis como segredo. Rotacione regularmente.

Armazenamento no cliente

Prefira HttpOnly cookie a localStorage para tokens de acesso. localStorage é acessível por XSS.

JWT é poderoso mas frequentemente mal implementado. O formato stateless (sem consulta ao banco) tem custo: revogação é difícil. Para cenários onde revogação imediata é crítica (logout, troca de senha), considere tokens opacos com validação server-side, ou mantenha uma blacklist de JTI (JWT ID) revogados.

"Um JWT mal implementado troca a proteção de sessão por uma vulnerabilidade estática que pode ser explorada offline."
