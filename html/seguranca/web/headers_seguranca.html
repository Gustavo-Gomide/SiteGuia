Headers de Segurança HTTP

Por que Headers de Segurança importam?

Headers de Segurança HTTP

Instruções HTTP que protegem contra XSS, clickjacking e downgrade attacks.

Headers HTTP de segurança são instruções do servidor ao browser sobre como processar e proteger o conteúdo da página. São uma camada de defesa adicional — baratos de implementar e eficazes contra XSS, clickjacking, sniffing de MIME type e downgrade attacks. Use securityheaders.com para auditar qualquer site.

Headers Essenciais

HeaderFunçãoValor Recomendado
Content-Security-Policy Define origens permitidas para scripts, estilos, imagens, frames. Principal defesa contra XSS. default-src 'self'; script-src 'self'; object-src 'none'
Strict-Transport-Security Força HTTPS; impede downgrade para HTTP. Browser recusa conexões HTTP após primeiro acesso. max-age=63072000; includeSubDomains; preload
X-Frame-Options Impede que a página seja embutida em iframe — mitiga clickjacking. DENY ou SAMEORIGIN
X-Content-Type-Options Impede MIME sniffing — browser respeita o Content-Type declarado. nosniff
Referrer-Policy Controla quanta informação da URL atual é enviada no header Referer. strict-origin-when-cross-origin
Permissions-Policy Controla acesso a APIs do browser (câmera, microfone, geolocalização) por origem. camera=(), microphone=(), geolocation=()
X-XSS-Protection Legado (filtro XSS do IE). Desabilitar em browsers modernos — pode introduzir vulnerabilidades. 0 (desabilitar)

Content Security Policy — Detalhes

# CSP para aplicação típica com recursos externos:
Content-Security-Policy:
  default-src 'self';
  script-src 'self' https://cdn.jsdelivr.net 'nonce-{RANDOM_NONCE}';
  style-src 'self' https://fonts.googleapis.com;
  font-src 'self' https://fonts.gstatic.com;
  img-src 'self' data: https:;
  connect-src 'self' https://api.minhaapp.com;
  frame-src 'none';
  object-src 'none';
  base-uri 'self';
  form-action 'self';
  upgrade-insecure-requests;

# Explicação:
# nonce-{RANDOM_NONCE} — cada resposta gera nonce único:
# <script nonce="abc123"> ... </script>   ← inline script permitido
# base-uri 'self'  — impede injeção de <base href="...">
# form-action 'self'  — formulários só submetem para mesma origem
# upgrade-insecure-requests  — auto-upgrade http: → https:

Evite estas diretivas inseguras

  • 'unsafe-inline' — permite qualquer script/style inline
  • 'unsafe-eval' — permite eval(), Function(), setTimeout(string)
  • script-src * — permite scripts de qualquer origem
  • data: em script-src — permite injeção via data URI

CSP Report-Only para transição

Use Content-Security-Policy-Report-Only com report-uri /csp-violations para coletar violações sem bloquear. Permite ajustar antes de ativar em modo enforce.

HSTS — HTTP Strict Transport Security

# Header HSTS completo:
Strict-Transport-Security: max-age=63072000; includeSubDomains; preload

# max-age=63072000 = 2 anos em segundos (mínimo para preload list)
# includeSubDomains = aplica HSTS em todos os subdomínios
# preload = envia o domínio para ser incluído na lista HSTS pré-embutida dos browsers

# Processo de preload:
# 1. Configure o header acima
# 2. Envie para https://hstspreload.org
# 3. Browsers downstream incluem seu domínio na lista hard-coded
# CUIDADO: preload é irreversível a curto prazo — certifique que HTTPS funciona em TODOS subdomínios

X-Frame-Options e Clickjacking

Ataque de Clickjacking

Atacante embute sua página em um iframe transparente sobre conteúdo enganoso. A vítima clica no que parece ser um botão inocente mas na verdade clica na sua interface autenticada.

<!-- evil.com -->
<iframe src="https://banco.com/transferir"
  style="opacity:0; position:absolute; top:0; left:0">
</iframe>
<button style="...">Clique para ganhar prêmio!</button>

Proteção via CSP frame-ancestors

Proteção moderna: substitui X-Frame-Options. Mais flexível — permite whitelist de origens.

# Bloquear todos os framing:
Content-Security-Policy: frame-ancestors 'none';

# Permitir apenas mesmo domínio:
Content-Security-Policy: frame-ancestors 'self';

# Permitir origem específica:
Content-Security-Policy: frame-ancestors https://dashboard.empresa.com;

Configuração em Servidores

# Nginx — adicionar em server ou location block:
add_header Strict-Transport-Security "max-age=63072000; includeSubDomains; preload" always;
add_header X-Frame-Options "DENY" always;
add_header X-Content-Type-Options "nosniff" always;
add_header Referrer-Policy "strict-origin-when-cross-origin" always;
add_header Permissions-Policy "camera=(), microphone=(), geolocation=()" always;
add_header Content-Security-Policy "default-src 'self'; object-src 'none'" always;

# Apache — .htaccess ou httpd.conf:
Header always set Strict-Transport-Security "max-age=63072000; includeSubDomains"
Header always set X-Frame-Options "DENY"
Header always set X-Content-Type-Options "nosniff"
Header always set Referrer-Policy "strict-origin-when-cross-origin"

# Express.js (Node) — usando helmet:
const helmet = require('helmet');
app.use(helmet());  // configura todos os headers principais automaticamente

Implementar headers de segurança leva menos de 1 hora e elimina classes inteiras de ataques. Audite seu site em securityheaders.com e observatory.mozilla.org. Remova também headers que revelam informações desnecessárias: Server: Apache/2.4.1, X-Powered-By: PHP/8.1 — essas informações ajudam atacantes a escolher exploits.

"Headers de segurança são a linha de defesa no protocolo HTTP — configure todos antes do primeiro deploy."
