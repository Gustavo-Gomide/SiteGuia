CSRF — Cross-Site Request Forgery

O que é CSRF?

CSRF — Cross-Site Request Forgery

Força o browser da vítima a enviar requisições autenticadas sem seu consentimento.

Em um ataque CSRF, o atacante engana o navegador da vítima para enviar requisições autenticadas a outro site sem o consentimento do usuário. O browser inclui automaticamente cookies de sessão em toda requisição — o servidor não consegue distinguir uma requisição legítima de uma forjada se não houver proteção específica.

Como o Ataque Funciona

1. Vítima autenticada

Usuário está logado em banco.com com cookie de sessão válido no browser.

2. Visita a página maliciosa

Usuário acessa evil.com controlado pelo atacante (link em email, fórum, etc.).

3. Requisição forjada

A página do atacante contém HTML/JS que dispara uma requisição para banco.com — ex: transferência de dinheiro.

4. Browser envia com cookies

O browser inclui automaticamente o cookie de sessão. O servidor autentica a requisição como legítima e executa a ação.

<!-- Exemplo: página evil.com forjando transferência bancária -->
<html>
  <body onload="document.forms[0].submit()">
    <form action="https://banco.com/transferir" method="POST">
      <input type="hidden" name="conta_destino" value="444455556666">
      <input type="hidden" name="valor" value="5000">
    </form>
  </body>
</html>
<!-- Submete automaticamente assim que a vítima abre a página -->

Mecanismos de Proteção

MecanismoComo funcionaRobustez
Token Anti-CSRF (Synchronizer Token) Token único por sessão/formulário inserido no form; servidor valida se bate com o da sessão Alta — atacante não tem acesso ao token
Double Submit Cookie Token idêntico no cookie e no body; servidor compara os dois Média-alta — vulnerável se subdomínio comprometido
SameSite Cookie Strict: bloqueia cookie em todas cross-site; Lax: permite GET mas bloqueia POST cross-site Alta (Strict) / Média (Lax)
Verificação de Origin/Referer Servidor verifica se Origin header corresponde ao domínio esperado Média — pode ser suprimido; depende de implementação
Custom Request Headers Exige header customizado (X-Requested-With) — só JS pode enviar com fetch/XHR Média — contorna form submission simples

SameSite Cookie em Detalhe

# Configuração do cookie de sessão:
Set-Cookie: sessionid=abc123; HttpOnly; Secure; SameSite=Lax

# SameSite=Strict:
# Cookie NUNCA é enviado em requisições cross-site
# Problema: usuário clicando link externo para o site perde a sessão

# SameSite=Lax (padrão moderno):
# Cookie NÃO é enviado em POST/PUT/DELETE cross-site
# Cookie É enviado em navegação de topo via GET (link normal)

# SameSite=None:
# Cookie sempre enviado — precisa de Secure
# Use apenas para casos específicos como embeds de terceiros

Browsers modernos (Chrome 80+) definem SameSite=Lax como padrão para cookies sem atributo explícito. Isso reduziu drasticamente a superfície de ataque CSRF, mas não elimina completamente — ainda é necessário token CSRF para casos do tipo GET que causem estado.

Bypass de CSRF Tokens

Remover o token do pedido

Algumas aplicações só validam se o token estiver presente — se removido, aceitam. Teste enviando a requisição sem o parâmetro CSRF.

Token de outro usuário

Se o servidor não verifica que o token pertence à sessão atual (só se é válido em geral), o atacante usa seu próprio token na requisição da vítima.

XSS + CSRF

XSS pode ler o token anti-CSRF do DOM e usá-lo para montar requisição forjada. XSS bypassa completamente proteção CSRF.

Subdomínio comprometido

Double Submit Cookie: se atacante controla subdomínio (evil.app.com), pode setar cookie no domínio pai .app.com para injetar token falso.

CSRF é especialmente perigoso quando combinado com ações de alto impacto (mudança de email/senha, transferências financeiras). A melhor defesa atual é a combinação de tokens CSRF robustos + SameSite=Strict/Lax nos cookies de sessão. APIs JSON com autenticação via Authorization header (Bearer token) não sofrem CSRF, pois browsers não enviam esse header automaticamente.

"CSRF abusa da confiança que o servidor tem no browser — tokens anti-CSRF provam que a requisição veio de dentro da aplicação."
