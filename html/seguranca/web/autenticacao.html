Autenticação Web

Fatores de autenticação, MFA, passkeys e protocolos modernos de identidade.

Autenticação prova que você é quem diz ser. Os fatores de autenticação são classificados em três categorias fundamentais. Combinar dois ou mais é a base do MFA — Multi-Factor Authentication.

FatorTipoExemplosFraqueza
Conhecimento Algo que você sabe Senha, PIN, pergunta secreta Phishing, vazamentos, brute force
Posse Algo que você tem OTP (SMS/TOTP), chave física (YubiKey), smartphone SIM swap, roubo do dispositivo, phishing em tempo real
Inerência Algo que você é Impressão digital, reconhecimento facial, iris Spoofing biométrico; imutável se vazado
Localização Onde você está IP geolocalizado, rede corporativa VPN/proxy; fator adicional, não standalone

MFA — Multi-Factor Authentication

SMS / Email OTP

Código de uso único enviado por SMS ou email. Conveniente mas vulnerável a SIM swap, ataques de interceptação SS7 e phishing.

Nível: Básico

TOTP (Time-based OTP)

Apps como Google Authenticator / Authy geram códigos a cada 30s baseados em HMAC-SHA1 + timestamp (RFC 6238). Não depende de rede.

Nível: Bom

FIDO2 / WebAuthn

Criptografia de chave pública. A chave privada nunca sai do dispositivo. Resistente a phishing — a chave é vinculada ao domínio. Hardware keys (YubiKey) ou biometria do dispositivo.

Nível: Excelente

Push Notification

App de autenticação envia notificação ao smartphone para aprovar/rejeitar. Vulnerável a MFA fatigue attack (bombardear aprovações até o usuário aceitar).

Nível: Bom

Autenticação Passwordless e Passkeys

Passkeys são a implementação consumer de FIDO2. Vinculam a autenticação ao dispositivo e ao domínio — eliminam phishing de senha por design. Suportadas nativamente em iOS 16+, Android 9+, Windows 11 e todos os browsers modernos.

// Registro de Passkey (WebAuthn API — simplificado):
const credential = await navigator.credentials.create({
  publicKey: {
    challenge: serverChallenge,   // bytes aleatórios do servidor
    rp: { name: "Minha App", id: "minhaapp.com" },
    user: { id: userId, name: "user@email.com", displayName: "User" },
    pubKeyCredParams: [{ alg: -7, type: "public-key" }],  // ES256
    authenticatorSelection: {
      residentKey: "required",         // passkey salva no dispositivo
      userVerification: "required"     // exige biometria/PIN local
    }
  }
});
// Servidor armazena a chave pública e o credential ID

// Login:
const assertion = await navigator.credentials.get({
  publicKey: { challenge: serverChallenge, rpId: "minhaapp.com" }
});
// Servidor verifica assinatura com a chave pública armazenada

SSO — Single Sign-On

Como funciona

O usuário autentica uma vez em um Identity Provider (IdP) central. As aplicações (Service Providers) confiam no IdP e recebem tokens/asserções de identidade sem exigir nova senha.

Protocolos

  • SAML 2.0 — XML, enterprise, federação
  • OpenID Connect / OAuth 2.0 — JSON/JWT, web/mobile
  • Kerberos — redes Windows/Active Directory

Benefícios

  • Uma senha gerenciada centralmente
  • MFA aplicado no IdP — proteção para todas as apps
  • Revogação centralizada de acesso

Riscos

  • Single point of failure — IdP comprometido = tudo comprometido
  • Misconfigurações no trust de SP
  • Privilege escalation via claims/roles no token

Boas Práticas de Autenticação

PráticaDetalhe
Hash de senhas com algoritmo corretoUse bcrypt (cost ≥ 12), Argon2id ou scrypt. Nunca MD5, SHA1 ou SHA256 puros sem salt
Comprimento mínimo de senhaNIST 800-63B recomenda mínimo 8 caracteres; melhor 12+ sem regras complexas de composição
Verificar senhas vazadasIntegrar HIBP API; bloquear ou alertar uso de senhas em lists de breach
Rate limiting em endpoints de authLogin, reset de senha, verificação de MFA — todos precisam ter throttle por IP e por conta
Mensagens de erro genéricas"Credenciais inválidas" — não revelar se email existe ou não (user enumeration)
Expiração e rotação de sessãoSessões expiram por inatividade; regenerar session ID após login; invalidar no logout

A tendência atual aponta para o fim das senhas. Passkeys oferecem experiência de usuário melhor (nenhuma senha para lembrar) e segurança muito superior. Se sua aplicação precisa de senhas hoje, combine com TOTP ou FIDO2. O risco de senha fraca ou reutilizada nunca será zero enquanto senhas existirem.

"A melhor senha é aquela que não existe — passkeys são o futuro da autenticação segura."
