O que é OAuth 2.0?

OAuth 2.0 — Autorização Delegada

Acesso a recursos em nome do usuário sem expor credenciais.

OAuth 2.0 é um protocolo de autorização — permite que aplicações obtenham acesso limitado a recursos em nome do usuário, sem expor credenciais. "Faça login com Google/GitHub" usa OAuth 2.0 (+ OpenID Connect para identidade). O protocolo define flows (grant types) para diferentes contextos.

PapelDescrição
Resource OwnerO usuário dono dos dados
ClientA aplicação que quer acesso (ex: app de terceiro)
Authorization ServerEmite tokens após autenticação do usuário (ex: Google, Auth0)
Resource ServerAPI que protege os dados — aceita access tokens

Flows (Grant Types)

Authorization Code + PKCE

Flow recomendado para web apps e SPAs. O código de autorização é trocado por token no backend. PKCE (Proof Key for Code Exchange) protege contra interceptação do código.

1. App gera code_verifier (aleatório) e code_challenge = SHA256(verifier)
2. Redireciona: GET /authorize?response_type=code
   &client_id=...&redirect_uri=...&code_challenge=...
3. Usuário autentica → server retorna ?code=AUTH_CODE
4. App troca: POST /token { code, code_verifier, client_id, redirect_uri }
5. Recebe access_token (+ refresh_token)

Client Credentials

Comunicação machine-to-machine sem usuário. A aplicação se autentica com client_id + client_secret e recebe token de acesso.

POST /token
{ "grant_type": "client_credentials",
  "client_id": "...", "client_secret": "..." }
→ { "access_token": "...", "expires_in": 3600 }

Implicit Flow (Depreciado)

Retornava o access_token diretamente na URL (#fragment). Vulnerável a vazamento via referer/historico/XSS. Substituído por Authorization Code + PKCE.

Resource Owner Password

App recebe usuário/senha diretamente e troca por token. Exige confiança total na aplicação — contradiz o objetivo do OAuth. Evite sempre que possível.

Vulnerabilidades Comuns

VulnerabilidadeComo explorarMitigação
redirect_uri manipulation Mudar para domínio do atacante: auth code redirecionado para evil.com Whitelist exata de redirect_uri; não aceitar substrings ou wildcards
state ausente / fraco State não vinculado à sessão — CSRF no fluxo OAuth; account takeover State criptograficamente aleatório, vinculado à sessão do usuário
Auth code reutilizável Código não invalidado após uso — pode ser reutilizado se interceptado Invalidar código após primeiro uso; expiração curta (< 1 min)
Token leakage via Referer Access token em URL → vazamento para sites externos via Referer header Nunca colocar tokens em URL; usar Authorization header
Scope escalation Modificar scope na requisição para obter permissões não concedidas Validar scopes server-side; nunca confiar no scope do request sem validar

OpenID Connect (OIDC)

OpenID Connect é uma camada de identidade sobre OAuth 2.0. Enquanto OAuth fornece autorização, OIDC adiciona autenticação: o Authorization Server emite um ID Token (JWT) com informações do usuário (claims: sub, email, name, etc.).

# ID Token (JWT decodificado):
{
  "iss": "https://accounts.google.com",  # issuer
  "sub": "110169484474386276334",         # unique user ID
  "aud": "client_id_da_aplicacao",        # audience — deve bater com seu client_id
  "exp": 1516246110,                      # expiração
  "iat": 1516242510,
  "email": "usuario@gmail.com",
  "email_verified": true,
  "name": "Fulano de Tal"
}

# Validação obrigatória no servidor:
# 1. Verificar assinatura com chave pública do IdP (JWKS endpoint)
# 2. Verificar iss matches o IdP esperado
# 3. Verificar aud matches seu client_id
# 4. Verificar exp não expirou

OAuth 2.0 tem muitas partes móveis — uma misconfiguração pode resultar em account takeover. Ao implementar "login social", valide SEMPRE o ID Token server-side. Nunca confie nos dados vindos do cliente sem verificar a assinatura. PortSwigger Web Security Academy tem labs excelentes de OAuth para praticar.
