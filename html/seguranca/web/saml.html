O que é SAML?

SAML 2.0 — Security Assertion Markup Language

Padrão XML para SSO corporativo — Identity Provider, Service Provider e asserções.

SAML 2.0 é um padrão XML para federação de identidade em ambientes corporativos. Permite SSO entre organizações diferentes — o usuário autentica no Identity Provider (IdP) e recebe asserções XML assinadas que o Service Provider (SP) valida. É amplamente usado em ambientes enterprise com Okta, Azure AD, ADFS.

ComponentePapelExemplo
Identity Provider (IdP)Autentica o usuário e emite asserçõesOkta, Azure AD, ADFS, Google Workspace
Service Provider (SP)Confia no IdP; fornece o serviçoSalesforce, Jira, aplicação interna
SAML AssertionDocumento XML assinado com atributos do usuárioemail, grupos, roles, validade
SAML ResponseWrapper que contém uma ou mais asserçõesEnviado por POST do browser (SSO POST binding)

Fluxo SP-Initiated SSO

1. Usuário acessa o SP

Acessa app.empresa.com sem sessão.

2. SP redireciona ao IdP

SP gera SAMLRequest (AuthnRequest XML, Base64+deflate) e redireciona o browser para o IdP.

3. Usuário autentica no IdP

Login com credenciais corporativas. O IdP gera uma SAMLResponse com asserção assinada.

4. SP valida e cria sessão

Browser faz POST da SAMLResponse para o SP. SP valida assinatura XML, extrai atributos e cria a sessão.

Vulnerabilidades

XML Signature Wrapping (XSW)

Ataque mais sofisticado contra SAML. A assinatura cobre um elemento, mas o SP processa outro elemento não assinado injetado pelo atacante.

<!-- Estrutura original assinada: -->
<Response>
  <Assertion ID="real" Subject="atacante@evil.com">
    <Signature>...cobre ID=real...</Signature>
  </Assertion>
</Response>

<!-- XSW — injetar asserção não assinada antes: -->
<Response>
  <Assertion ID="evil" Subject="admin@empresa.com">
    <!-- sem assinatura -- SP lê este se for vulnerável -->
  </Assertion>
  <Assertion ID="real" Subject="atacante@evil.com">
    <Signature>...assinatura válida...</Signature>
  </Assertion>
</Response>

XXE — XML External Entity

A SAMLResponse é XML. Se o parser XML não tem XXE desabilitado, o atacante pode injetar entidades externas para ler arquivos locais do servidor SP ou fazer SSRF.

<!DOCTYPE foo [
  <!ENTITY xxe SYSTEM "file:///etc/passwd">
]>
<samlp:Response>
  <saml:Issuer>&xxe;</saml:Issuer>
  ...

Signature Exclusion

SP aceita asserções sem assinatura. Atacante remove a tag <Signature> completamente e modifica atributos como email/role livremente.

Comment Injection

Inserir comentário XML dentro de valores para bypassar comparação de strings: adm<!--comment-->in. Alguns parsers normalizam antes de comparar, outros não.

Defesas e Configuração Segura

Validar assinatura corretamente

Verificar que a assinatura cobre o elemento processado. Usar biblioteca SAML bem mantida (python3-saml, OneLogin). Não implementar SAML manualmente.

Desabilitar XXE

No parser XML: desabilitar FEATURE_EXTERNAL_GENERAL_ENTITIES e FEATURE_EXTERNAL_PARAMETER_ENTITIES. Em Java: factory.setFeature(XMLConstants.FEATURE_SECURE_PROCESSING, true).

Validar NotBefore / NotOnOrAfter

Asserções SAML têm janela de tempo. Validar estritamente + clock skew pequeno. Asserções expiradas ou futuras devem ser rejeitadas.

Validar Audience

O campo <AudienceRestriction> especifica para qual SP a asserção é válida. Validar que corresponde ao seu SP — evita replay em outro SP.

SAML é complexo e a maioria das vulnerabilidades de alto impacto vem de implementações incorretas de validação XML. Se possível, migre para OpenID Connect (OIDC) que usa JWT — mais simples, mais bem suportado e menos propenso a ataques de parsing XML. Em ambientes enterprise, SAML ainda é necessário para integração com sistemas legados.

"XML como base de segurança traz toda a complexidade e surface de ataque do formato — valide com rigor matemático."
