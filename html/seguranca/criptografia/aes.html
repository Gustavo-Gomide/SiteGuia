AES — Advanced Encryption Standard

Contexto e Origem

AES — Advanced Encryption Standard

O padrão de criptografia simétrica mais usado — blocos, modos e implementações.

AES é o algoritmo de criptografia simétrica mais usado no mundo. Estabelecido como padrão pelo NIST em 2001 (FIPS 197), substituiu o DES depois que este foi quebrado por força bruta.

Em 1997, o NIST abriu concurso internacional para novo padrão. Cinco finalistas foram selecionados e submetidos a intensa análise pública por 3 anos. O algoritmo Rijndael, criado pelos criptógrafos belgas Joan Daemen e Vincent Rijmen, venceu e foi nomeado AES.

Especificações técnicas

Tamanho de bloco: 128 bits (fixo)

Tamanhos de chave: 128, 192 ou 256 bits

Rodadas: 10 (AES-128), 12 (AES-192), 14 (AES-256)

Estrutura: Rede de substituição-permutação (SPN)

Segurança conhecida: Sem ataque prático conhecido — o melhor ataque teórico requer 2¹²⁶·¹ operações para AES-128

Como o AES Funciona

Cada rodada do AES aplica 4 transformações sobre o estado (matriz 4×4 de bytes — 128 bits):

1. SubBytes

Substituição não linear byte a byte usando uma S-Box (lookup table). Cria confusão — garante que a relação entre chave e ciphertext seja complexa.

2. ShiftRows

Rotação das linhas da matriz de estado. Linha 0: sem deslocamento; Linha 1: 1 byte; Linha 2: 2 bytes; Linha 3: 3 bytes. Garante difusão horizontal.

3. MixColumns

Multiplicação matricial no campo finito GF(2⁸). Cada coluna é transformada — garante difusão vertical. Não aplicado na última rodada.

4. AddRoundKey

XOR do estado com a subchave da rodada (derivada do Key Schedule). Introduz o segredo da chave no processo. Aplicado em todas as rodadas + rodada inicial.

Modos de Operação Recomendados

AES-GCM ( Preferido)

Galois/Counter Mode — combina cifração CTR + autenticação GHASH. Provê AEAD (confidencialidade + integridade em uma operação). Padrão no TLS 1.3. Requer IV de 96 bits único por operação.

# Python — AES-256-GCM:
from cryptography.hazmat.primitives.ciphers.aead import AESGCM
import os

key = os.urandom(32)  # 256 bits
nonce = os.urandom(12)  # 96 bits

aesgcm = AESGCM(key)
ct = aesgcm.encrypt(nonce, b"mensagem", b"aad")
pt = aesgcm.decrypt(nonce, ct, b"aad")

AES-CTR

Converte bloco em stream cipher. Paralelizável. Sem autenticação — use com HMAC separado (Encrypt-then-MAC). Útil quando GCM não está disponível.

AES-ECB (Nunca usar)

Blocos iguais → ciphertext igual. Revela padrões do plaintext. O famoso "Penguin do Linux" demonstra visualmente o problema.

AES-CBC (Evitar)

Vulnerável a ataques de padding oracle (POODLE, BEAST, Lucky13). Se necessário, use apenas com HMAC verificado antes de decifrar (Encrypt-then-MAC).

AES-128 vs. AES-256

Qual usar? A resposta depende do contexto:

Critério AES-128 AES-256
Segurança clássica128 bits — mais que suficiente256 bits — margem extra
Segurança pós-quântica (Grover)Reduz para ~64 bits efetivosReduz para ~128 bits — seguro
VelocidadeMais rápido (~25% menos rodadas)Mais lento
Aplicação recomendadaGeral, TLSDados com longevidade alta, governo, PQC-preparado

Para a maioria das aplicações, AES-128-GCM é suficiente e mais rápido. Para dados que precisam de sigilo por 20+ anos, use AES-256.

AES-NI — Aceleração por Hardware

Praticamente todos os processadores modernos (Intel desde 2010, AMD Ryzen, Apple Silicon) incluem instruções dedicadas AES-NI que executam cada rodada em um único ciclo de clock.

# Verificar suporte AES-NI no Linux:
grep -o 'aes' /proc/cpuinfo | head -1

# Benchmark AES com e sem AES-NI (OpenSSL):
openssl speed -evp aes-256-gcm

Com AES-NI, AES-256-GCM atinge facilmente 10–20 GB/s em CPUs modernas — eliminando qualquer overhead prático para cifração de dados.

Use AES-256-GCM para novas implementações. Nunca ECB, evite CBC sem autenticação. Veja também Criptografia Simétrica para o contexto geral e Funções Hash para proteção de integridade.
