Tríade CIA — Os Pilares da Segurança da Informação

Todo controle de segurança serve para preservar ao menos um dos três atributos.

A tríade CIA (Confidencialidade, Integridade, Disponibilidade) é o modelo conceitual mais importante da segurança. Todo controle, política e tecnologia de segurança serve para preservar pelo menos um desses três atributos.

Criado nos anos 1970 e formalizado em padrões como ISO/IEC 27001 e NIST SP 800-53, o modelo é agnóstico a tecnologia — se aplica a papel, banco de dados, API ou sistema operacional. Saber qual pilar está em risco é o primeiro passo para escolher a defesa correta.

Definição formal

Confidencialidade: garantia de que a informação é acessível apenas por entidades autorizadas.
Integridade: garantia de que a informação e os sistemas não foram alterados de maneira não autorizada.
Disponibilidade: garantia de que sistemas e dados estão acessíveis e operacionais quando necessários.

Confidencialidade

Confidencialidade significa que somente quem precisa saber, sabe. Não é sigilo absoluto — é controle de acesso granular baseado em necessidade e autorização.

Mecanismos de controle

  • Criptografia em trânsito (TLS/HTTPS) e em repouso (AES-256)
  • Controle de acesso baseado em papéis (RBAC)
  • Autenticação multifator (MFA)
  • Classificação de dados (público, interno, confidencial, secreto)
  • DLP — Data Loss Prevention

Ameaças comuns

  • Sniffing e interceptação de tráfego não criptografado
  • Credential theft — phishing, keyloggers
  • Vazamento por insider (funcionário mal-intencionado ou negligente)
  • SQL Injection para exfiltrar banco de dados
  • Shoulder surfing — espionagem visual

Exemplos de violação

  • Breach do LinkedIn (2012): 167M hashes de senha vazados
  • Equifax (2017): 147M registros expostos por falha em Apache Struts
  • Senha enviada em texto plano por e-mail

Princípio do Menor Privilégio

O mecanismo mais fundamental para preservar confidencialidade: conceder a cada usuário ou processo apenas o acesso mínimo necessário para executar sua função. Se um processo só precisa ler um arquivo, ele nunca deve ter permissão de escrita.

Controle Tipo Exemplo
Criptografia AES-256 Técnico / Preventivo Disco criptografado com BitLocker ou LUKS
ACL (Access Control List) Técnico / Preventivo Permissões de arquivo no Linux (chmod)
Política de classificação Administrativo Documentos marcados como "Confidencial"
DLP Técnico / Detectivo Bloquear envio de CPF por e-mail

Integridade

Integridade garante que os dados são confiáveis: não foram modificados, corrompidos ou substituídos por uma entidade não autorizada — nem em trânsito, nem em armazenamento.

Mecanismos de controle

  • Hashing criptográfico (SHA-256, SHA-3) — fingerprint imutável
  • Assinaturas digitais — autenticidade + integridade juntas
  • MACs (Message Authentication Codes) — HMAC-SHA256
  • Checksums para validar downloads
  • Controle de versão e logs de auditoria imutáveis

Ameaças comuns

  • Man-in-the-middle (MitM) — alteração de pacotes em trânsito
  • SQL Injection com UPDATE/DELETE
  • Malware que modifica arquivos de sistema
  • Race conditions em aplicações multithreaded
  • Bit rot — degradação silenciosa de mídias de armazenamento

Exemplos de violação

  • SolarWinds (2020): binário de atualização modificado na fonte
  • Defacement de sites: conteúdo substituído por invasores
  • Falha de HMAC: autenticação por cookie forjável

Como o hashing protege a integridade

Um hash criptográfico é uma função de mão única: qualquer alteração no dado — mesmo um único bit — produz um hash completamente diferente. Não é possível reverter o hash para obter o dado original. Por isso, hashes são usados para verificar se um arquivo foi alterado:

# Gerar SHA-256 de um arquivo:
sha256sum arquivo.iso
# Saída: a1b2c3d4...  arquivo.iso

# Verificar integridade após download:
# Compare o hash gerado com o hash publicado pelo fabricante
# Se divergirem, o arquivo foi corrompido ou adulterado

Disponibilidade

Disponibilidade garante que um sistema entrega seu serviço quando necessário. Um sistema perfeitamente seguro e íntegro que fica fora do ar quando necessário falhou em seu propósito.

Mecanismos de controle

  • Redundância: servidores em cluster, load balancers
  • Backup regular com testes de restauração (RTO/RPO)
  • CDN — distribuição geográfica de conteúdo
  • Proteção anti-DDoS (Cloudflare, AWS Shield)
  • Plano de continuidade de negócios (BCP) e DR

Ameaças comuns

  • DDoS — inundação de tráfego para saturar recursos
  • Ransomware — criptografa dados e exige resgate
  • Falhas de hardware não previstas (SPOF)
  • Erros humanos em produção
  • Desastres naturais sem recuperação planejada

Exemplos de violação

  • WannaCry (2017): hospitais no Reino Unido paralisados por ransomware
  • GitHub DDoS (2018): 1,35 Tbps — maior ataque registrado até então
  • Outage AWS us-east-1 (2021): impacto em centenas de serviços

SLA e métricas de disponibilidade

Disponibilidade é frequentemente expressa em "noves" — percentual de uptime anual:

Nível Uptime Downtime anual permitido
99%Dois noves~87,6 horas
99,9%Três noves~8,7 horas
99,99%Quatro noves~52 minutos
99,999%Cinco noves~5 minutos

Expansões do Modelo — IAAA e Parkerian Hexad

A tríade CIA é o mínimo. Frameworks modernos adicionam atributos que cobrem casos não contemplados:

Autenticidade

Quem enviou é realmente quem diz ser. Garantida por assinaturas digitais, certificados X.509 e protocolos de autenticação (OAuth, SAML).

Não-repúdio

O autor não pode negar ter executado uma ação. Essencial em transações financeiras e contratos digitais. Implementado via assinatura digital com chave privada.

Responsabilização

Toda ação é atribuída a um sujeito identificado. Logs de auditoria, trilhas de acesso e sistemas como SIEM garantem accountability.

Utilidade

Conceito do Parkerian Hexad: um dado precisa estar em formato utilizável. Dados criptografados sem chave disponível têm disponibilidade mas não têm utilidade.

Conflitos entre os Pilares

Na prática, os três pilares estão em tensão constante. A gestão de segurança consiste em equilibrá-los de acordo com o contexto de negócio:

Cenário Pilar priorizado Pilar sacrificado Justificativa
Sistema bancário Integridade + Confidencialidade Disponibilidade (em manutenção) Melhor offline do que com dados corrompidos
UTI hospitalar Disponibilidade Confidencialidade (acesso emergencial) Vidas dependem do sistema estar no ar
Repositório público Disponibilidade + Integridade Confidencialidade (dados públicos) Acesso aberto é o objetivo
Backup offline cifrado Confidencialidade + Integridade Disponibilidade imediata Segurança > velocidade de recuperação

A tríade CIA não é checklist — é uma lente analítica. Ao avaliar qualquer risco ou implementar qualquer controle, pergunte: "Qual pilar isso afeta e qual o trade-off?"

"Segurança não existe no vácuo — existe no contexto de um negócio. O pilar mais importante é aquele que, se violado, causa mais dano à organização."
