A Relação entre Ameaça, Vulnerabilidade e Risco

Ameaças e Vulnerabilidades

A diferença entre ameaça, vulnerabilidade, risco e exploit — e por que importa.

Confundir ameaça, vulnerabilidade e risco é erro comum. Entender cada conceito é pré-requisito para qualquer análise defensiva.

Vulnerabilidade

Fraqueza intrínseca de um ativo — software, processo, pessoa ou hardware. Existe independente de ser explorada. Ex: versão desatualizada de OpenSSL com CVE conhecido.

Ameaça

Agente ou evento com potencial de explorar uma vulnerabilidade. Pode ser intencional (hacker, insider) ou acidental (desastre, erro humano).

Risco

Probabilidade de uma ameaça explorar uma vulnerabilidade × impacto resultante. É o único dos três que pode ser gerenciado ativamente.

Ataque

Ação deliberada de uma ameaça explorando uma vulnerabilidade. É o risco materializado — o evento que a segurança busca prevenir ou mitigar.

Fórmula fundamental do risco

Risco = Probabilidade × Impacto

Uma vulnerabilidade crítica em sistema sem conexão à internet tem baixo risco. Uma vulnerabilidade média em sistema exposto ao público pode ter risco alto. O contexto é tudo.

Tipos de Vulnerabilidades

Vulnerabilidades surgem em diferentes camadas de um sistema. Conhecer os tipos ajuda a direcionar testes e controles:

Software

Bugs de programação: buffer overflow, use-after-free, format string, race conditions. A maioria dos CVEs pertence a esta categoria.

Configuração

Sistemas corretamente instalados mas mal configurados: senhas padrão, serviços desnecessários expostos, permissões excessivas.

Design

Falhas na arquitetura do sistema que não podem ser corrigidas sem redesign. Ex: protocolo sem criptografia nativa (Telnet, FTP).

Humana

Engenharia social, falta de treinamento, senhas fracas. Estudos apontam que 82% das violações envolvem fator humano (Verizon DBIR 2023).

Física

Falta de controle de acesso físico: servidores sem lock, USB livre, telas visíveis a terceiros (shoulder surfing).

Criptografia fraca

Uso de algoritmos obsoletos (MD5, SHA-1 para assinaturas, DES), chaves curtas ou protocolos inseguros (SSLv3, TLS 1.0).

CVSS — Classificando a Severidade

O Common Vulnerability Scoring System (CVSS) é o padrão global para classificar vulnerabilidades com pontuação de 0 a 10. A versão atual é CVSS v3.1.

Score Severidade Ação recomendada
0.0NenhumaNenhuma ação necessária
0.1 – 3.9BaixaCorrigir no próximo ciclo de manutenção
4.0 – 6.9MédiaCorrigir dentro de 30 dias
7.0 – 8.9AltaCorrigir dentro de 7 dias
9.0 – 10.0CríticaCorrigir imediatamente (emergência)

O CVSS avalia três grupos de métricas:

Base Score

Características intrínsecas: vetor de ataque (rede/local), complexidade, privilégios necessários, interação humana, impacto em CIA.

Temporal Score

Modificadores no tempo: disponibilidade de exploit, nível de remediação oficial, confiança no relato.

Environmental Score

Ajuste para o contexto organizacional específico: importância do sistema, controles mitigatórios existentes.

CVE, NVD e CWE

O ecossistema de gestão de vulnerabilidades possui bases de dados públicas fundamentais:

CVE

Common Vulnerabilities and Exposures — identificador único para cada vulnerabilidade pública. Formato: CVE-ANO-NÚMERO. Ex: CVE-2021-44228 (Log4Shell). Catálogo em cve.mitre.org.

NVD

National Vulnerability Database — banco do NIST que enriquece CVEs com score CVSS, CPE (produtos afetados) e links de patch. Consulta em nvd.nist.gov.

CWE

Common Weakness Enumeration — catálogo de categorias de fraquezas de software. Enquanto CVE é uma instância, CWE é a classe. Ex: CWE-89 = SQL Injection, CWE-79 = XSS.

Threat Modeling — Modelagem de Ameaças

Threat modeling é o processo estruturado de identificar ameaças antes que sistemas sejam comprometidos. É praticado desde a fase de design de sistemas (shift-left security).

STRIDE

Framework da Microsoft: Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege. Cada letra mapeia para um pilar CIA ou propriedade de segurança.

PASTA

Process for Attack Simulation and Threat Analysis — 7 etapas centradas no risco de negócio. Mais adequado para contextos corporativos com múltiplos stakeholders.

DREAD

Modelo de pontuação: Damage, Reproducibility, Exploitability, Affected Users, Discoverability. Produz score numérico para priorização.

Attack Trees

Representação hierárquica de como um atacante pode atingir um objetivo (raiz) com sub-objetivos (galhos). Útil para visualizar combinações de ataques.

As quatro perguntas do threat modeling

1. O que estamos construindo? — Diagrama do sistema (DFD)

2. O que pode dar errado? — Identificar ameaças (STRIDE, brainstorm)

3. O que faremos sobre isso? — Mitigações e controles por ameaça

4. Fizemos um bom trabalho? — Revisão, validação, documentação

Estratégias de Mitigação

Toda vulnerabilidade pode ser tratada segundo quatro estratégias (em ordem de preferência):

Remediação

Eliminar a vulnerabilidade completamente: aplicar patch, corrigir código, remover serviço desnecessário. Ideal mas nem sempre viável.

Mitigação

Reduzir probabilidade ou impacto sem eliminar a vulnerabilidade: WAF na frente de sistema legado desatualizado, segmentação de rede, monitoramento intensificado.

Transferência

Passar o risco para terceiros: seguro cibernético, terceirização do serviço, SLA contratual com fornecedor.

Aceitação

Decisão consciente de não agir — apenas quando o custo de mitigação supera o impacto potencial. Deve ser documentada e aprovada.

Gerenciar ameaças e vulnerabilidades é um processo contínuo, não uma tarefa única. Novos CVEs surgem diariamente; novos vetores de ataque emergem com cada mudança de tecnologia. A resposta é um programa maduro de Gestão de Riscos e monitoramento contínuo.

"Um atacante só precisa ter razão uma vez. Um defensor precisa ter razão sempre. Por isso, conhecer as ameaças é requisito inegociável."
