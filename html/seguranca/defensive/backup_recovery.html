Por que backup é segurança?

Backup e Recovery

O último controle de segurança — sem backup testado, não há recuperação garantida.

Backup não é apenas "boas práticas de TI" — é o último controle de segurança. Quando um ransomware criptografa todos os dados ou um atacante destrói sistemas intencionalmente, a única coisa que garante a sobrevivência do negócio é uma cópia íntegra e testada dos dados.

Um backup não testado não é um backup. Empresas descobrem que seus backups estavam corrompidos, incompletos ou sem os dados críticos exatamente no momento em que mais precisam. Testar restauração periodicamente é não-negociável.

Regra 3-2-1 (e variações)

3 — Três cópias dos dados

O dado original + 2 backups. Se um backup falhar na restauração, há outro. Nunca confie em uma única cópia.

2 — Dois tipos de mídia diferentes

Ex: disco local (NAS) + cloud. Um falha de hardware (HD queimado) não destrói todas as cópias simultâneamente.

1 — Uma cópia offsite

Fora do local físico principal. Se o datacenter pegar fogo ou for inundado, a cópia offsite sobrevive. Cloud = offsite automaticamente.

Regra 3-2-1-1-0

Variação moderna contra ransomware:

  • 3 cópias
  • 2 tipos de mídia
  • 1 offsite
  • 1 air-gapped ou imutável (offline ou WORM)
  • 0 erros nos testes de restauração

Por que imutável?

Ransomware moderno apaga backups conectados antes de criptografar. Um backup imutável (WORM — Write Once Read Many) ou air-gapped (desconectado da rede) não pode ser deletado ou criptografado pelo ransomware. AWS S3 Object Lock, Azure Immutable Blob, fita magnética offline são exemplos.

RTO e RPO

Métrica Significado Exemplo
RTO — Recovery Time Objective Tempo máximo aceitável para restaurar sistema após incidente Banco: RTO de 4h; blog pessoal: RTO de 48h
RPO — Recovery Point Objective Quantidade máxima aceitável de dados perdidos (medida em tempo) RPO de 1h = aceita perder até 1h de transações; RPO de 0 = zero perda

RTO e RPO são definidos pelo negócio, não pela TI. Um sistema de pagamentos tem RTO de minutos e RPO de segundos. Um portal de documentos internos pode ter RTO de dias. O backup deve ser projetado para atingir esses objetivos.

Ferramentas e Soluções

Veeam Backup

Líder de mercado para backup de VMs (VMware/Hyper-V). Suporta regra 3-2-1, imutabilidade, testes automáticos de restauração (SureBackup). Padrão corporativo.

Restic + Rclone

Combo open source popular. Restic faz backups incrementais criptografados para qualquer destino. Rclone sincroniza para dezenas de providers de cloud. Gratuito e muito capaz.

Cloud nativa

AWS Backup, Azure Backup, GCP Cloud Backup — integradas nativamente com outros serviços. Suporte a immutable backups. Ideais para workloads já em cloud.

Bacula / Amanda

Soluções open source enterprise para backup em fita e disco. Amplamente usadas em ambientes Linux/UNIX corporativos com necessidade de retenção longa.

# Backup incremental criptografado com Restic:

# Inicializar repositório (local ou S3):
restic -r s3:s3.amazonaws.com/meu-bucket init

# Fazer backup:
restic -r s3:s3.amazonaws.com/meu-bucket backup /dados

# Listar snapshots:
restic -r s3:s3.amazonaws.com/meu-bucket snapshots

# Restaurar snapshot específico:
restic -r s3:s3.amazonaws.com/meu-bucket restore abc123 --target /restore

Backup íntegro + teste regular de restauração = diferença entre recuperar de um ransomware em horas e pagar resgate ou fechar a empresa. Todo plano de continuidade começa com uma estratégia de backup bem definida. Veja também Resposta a Incidentes para o contexto completo de recuperação.

"Você não tem backup — você tem um processo de backup. A diferença é ter testado a restauração."
