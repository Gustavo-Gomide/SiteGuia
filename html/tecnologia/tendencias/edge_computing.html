Edge Computing — Visão Geral

Edge Computing

Processamento na borda da rede — quando enviar dados à nuvem é rápido demais, caro demais ou arriscado demais

Edge Computing é o paradigma de computação em que o processamento ocorre próximo à fonte dos dados — no dispositivo, na torre de celular, no gateway IoT ou no servidor da fábrica — em vez de ser enviado para um datacenter centralizado na nuvem. A motivação é tripla: latência (a física limita quanto velocidade a fibra óptica pode oferecer — 300 km de distância adicionam ~1 ms de atraso irreducível), largura de banda (câmeras 4K/8K geram terabytes/dia; enviar tudo para a nuvem é inviável), e privacidade/regulação (dados de saúde ou industriais muitas vezes não podem sair do país ou da instalação).

Cloud, Fog e Edge — Hierarquia Computacional

Cloud Computing

Datacenters centralizados (AWS, Azure, GCP) com recursos quase ilimitados. Latência de ida-e-volta (RTT) tipicamente 50–200 ms. Ideal para analytics em lote, treinamento de modelos ML, armazenamento de longo prazo.

Fog Computing

Camada intermediária entre dispositivos e cloud, introduzida pela Cisco (2014). Processa dados em gateways regionais ou servidores de borda localizados em pontos de agregação (ex: base station, subestação, campus). Latência: 10–50 ms.

Edge Computing

Processamento no ou próximo ao dispositivo final: câmera, sensor, robô, veículo, smartphone. Latência <10 ms, às vezes <1 ms (on-device). Poder de processamento limitado, mas suficiente para inferência (não treinamento) de modelos ML otimizados.

Casos de Uso Industriais

Manufatura e Indústria 4.0

Controle de qualidade visual por câmeras com IA na linha de produção: <5 ms para rejeitar peças defeituosas (impossível via cloud). PLCs e robôs com controle em malha fechada via edge server local. Downtime = prejuízo imediato: a disponibilidade offline é crítica.

Veículos Autônomos

Decisões de frenagem de emergência devem ocorrer em <100 ms. Processamento on-board (NVIDIA Drive, Qualcomm Snapdragon) + RSUs (Road Side Units) na infraestrutura. LiDAR gera ~20 GB/s — enviar à nuvem em tempo real é fisicamente inviável.

Saúde e Cirurgia Remota

Monitoramento contínuo de UTI: edge gateway no hospital processa vitais localmente, envia apenas anomalias para cloud. Cirurgia robótica remota requer <2 ms end-to-end — apenas edge local o garante.

Smart Cities e Infraestrutura

Câmeras de tráfego com visão computacional on-board: detecção de acidentes sem enviar vídeo para datacenter central. Medidores inteligentes com edge analytics: detecção local de fraudes e anomalias.

Varejo e Experiência do Cliente

Lojas sem caixa (Amazon Go): câmeras + sensores de peso processados localmente em <50 ms para detectar produtos retirados das prateleiras. Personalização em tempo real sem latência de API.

Principais Plataformas

  • AWS Greengrass: Lambda na borda, sincronização com AWS IoT Core.
  • Azure IoT Edge: módulos containerizados (Docker), suporte a ONNX para ML inference.
  • Google Distributed Cloud Edge: hardware gerenciado por Google on-premises.
  • Cloudflare Workers: edge serverless em 300+ PoPs globais, ideal para low-latency APIs.
  • NVIDIA Jetson: plataforma de hardware para AI na borda (Jetson Orin até 275 TOPS).
Exemplo — Inferência ML na Borda com ONNX Runtime (Python)
import onnxruntime as ort
import numpy as np
import cv2

# Carregar modelo otimizado para edge (ONNX, quantizado int8)
session = ort.InferenceSession(
    "modelo_deteccao.onnx",
    providers=["CPUExecutionProvider"]  # ou "CUDAExecutionProvider"
)

# Capturar frame da câmera local
cap = cv2.VideoCapture(0)
ret, frame = cap.read()

# Pré-processar
input_data = cv2.resize(frame, (640, 640)).astype(np.float32) / 255.0
input_data = np.expand_dims(input_data.transpose(2, 0, 1), axis=0)

# Inferência local (sem cloud) — tipicamente <10 ms em Jetson Orin
outputs = session.run(None, {"images": input_data})
print(f"Detecções: {outputs[0].shape}")
