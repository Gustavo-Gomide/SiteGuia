Paralelismo — Fazer Mais ao Mesmo Tempo

Aproveitando multiplos nucleos e GPUs para acelerar computacoes intensas.

Em 2005, Intel e AMD lancaram seus primeiros processadores dual-core — e a Lei de Moore mudou de direcao. Em vez de CPUs mais rapidas, passamos a ter mais nucleos. Programas que ignoram esse fato rodam em apenas um dos 16, 32 ou 64 nucleos disponíveis. Um modelo de ML que treina em 8 horas num unico core pode terminar em 30 minutos com paralelizacao correta. Paralelismo e a tecnica que transforma o hardware moderno em vantagem computacional real.

1. Concorrencia vs. Paralelismo

A distincao e fundamental e frequentemente confundida. Concorrencia e sobre estrutura e responsividade; paralelismo e sobre execucao fisica simultanea e velocidade.

Rob Pike: "Concurrency is about dealing with lots of things at once. Parallelism is about doing lots of things at once." Um servidor web que atende 1.000 conexoes em um unico core via event loop e concorrente, nao paralelo. Quatro workers treinando um modelo de ML — cada um em um core diferente — sao paralelos.

Dimensao Concorrencia Paralelismo
Definicao Multiplas tarefas em progresso sobrepostas no tempo Multiplas tarefas executando fisicamente ao mesmo tempo
Requer multi-core? Nao (funciona em single-core via context switch) Sim (multiplos CPUs, cores ou GPUs)
Objetivo principal Responsividade, throughput de I/O Velocidade — menor tempo de execucao (latencia)
Tipo de tarefa I/O-bound (rede, disco, banco de dados) CPU-bound (calculo numerico, rendering, ML, compressao)
Risco principal Race conditions, deadlock, starvation Overhead de sincronizacao, load imbalance, Lei de Amdahl
Exemplos Servidor web, GUI, event loop, consumers de fila Treinamento de ML, rendering 3D, simulacoes numericas, encoding de video

2. Como Funciona o Paralelismo

Paralelismo e implementado dividindo o trabalho entre recursos de computacao fisicamente independentes. Existem tres estrategias classicas de divisao:

diagrama — os tres tipos de paralelismo
-- 1. Paralelismo de Dados: mesma operacao, dados diferentes
Array [1, 2, 3, 4, 5, 6, 7, 8]
  Core 1: [1, 2]  * 2  →  [2, 4]
  Core 2: [3, 4]  * 2  →  [6, 8]
  Core 3: [5, 6]  * 2  →  [10, 12]
  Core 4: [7, 8]  * 2  →  [14, 16]
-- Resultado: [2, 4, 6, 8, 10, 12, 14, 16]

-- 2. Paralelismo de Tarefas: operacoes diferentes, simultaneas
  Core 1: comprimir imagem A
  Core 2: comprimir imagem B
  Core 3: gerar thumbnail de A
  Core 4: gerar thumbnail de B

-- 3. Paralelismo de Pipeline: estagios sobrepostos (linha de montagem)
Frame 1:  [Captura] → [Decodifica] → [Filtra] → [Exibe]
Frame 2:             [Captura]     → [Decodifica] → [Filtra] → ...
-- sempre 4 frames em estagios diferentes simultaneamente

Paralelismo de Dados

A mesma operacao e aplicada a particoes diferentes de um dataset simultaneamente. Ideal para arrays, matrizes e tensores. E o modelo dominante em GPUs (SIMT: Single Instruction Multiple Thread) e em NumPy (vetorizacao SIMD) e Apache Spark (RDDs particionados entre workers).

Paralelismo de Tarefas

Diferentes tarefas independentes executam simultaneamente em cores distintos. Requer que as tarefas nao tenham dependencias entre si. Implementado com thread pools, ForkJoinPool (Java), ProcessPoolExecutor (Python) e OpenMP sections.

Paralelismo de Pipeline

O processamento e dividido em estagios sequenciais que executam sobrepostos — como uma linha de montagem. Enquanto o estagio 1 processa o item N+1, o estagio 2 processa o item N. Usado em processamento de video, compiladores e nos proprios pipelines de instrucao dos processadores modernos.

3. Lei de Amdahl — O Limite do Paralelismo

A Lei de Amdahl (1967) define o ganho maximo de velocidade possivel ao paralelizar um programa. E a lei mais importante — e mais ignorada — em computacao paralela.

matematica — formula e suas implicacoes praticas
-- Speedup maximo com N processadores:
-- S(N) = 1 / (1 - P + P/N)
-- P = fracao paralelizavel do programa (0 a 1)
-- N = numero de processadores

-- Programa com 90% paralelizavel (P = 0.9):
N=1:   S = 1 / (0.1 + 0.9/1)    =  1.0x
N=2:   S = 1 / (0.1 + 0.9/2)    =  1.82x
N=4:   S = 1 / (0.1 + 0.9/4)    =  3.08x
N=8:   S = 1 / (0.1 + 0.9/8)    =  4.71x
N=16:  S = 1 / (0.1 + 0.9/16)   =  6.40x
N=∞:   S = 1 / 0.1               = 10.0x  ← limite absoluto!

-- INSIGHT: dobrar os cores NAO dobra a velocidade.
-- Os 10% sequenciais limitam o speedup a 10x, nao importa quantos cores.
-- Reduzir a parte sequencial e mais valioso que adicionar mais cores.

O Que a Lei Revela

Um programa com apenas 5% de codigo sequencial so pode escalar ate 20x, independente de quantos cores sejam adicionados. A parte sequencial inclui: I/O serial, sincronizacao entre workers, operacoes de reducao (merge de resultados) e codigo de coordenacao entre tarefas.

Lei de Gustafson — A Contrapartida

Gustafson (1988) reformulou o problema: em vez de escalar o mesmo problema, use mais processadores para resolver problemas maiores no mesmo tempo. "Scaled speedup": o que seria impossível processar em 1 hora num core fica possível num core-hora distribuído. MapReduce e o modelo de Gustafson em acao.

4. Paralelismo em CPU — OpenMP e Fork/Join

Para programas CPU-bound em C, C++ e Fortran, OpenMP e o padrao da industria. Diretivas de compilador transformam loops sequenciais em paralelos com minimas alteracoes ao codigo.

c — OpenMP: paralelizar um loop com uma diretiva
// Compilar: gcc -fopenmp programa.c -o programa
#include <omp.h>

int main() {
    double soma = 0.0;
    int n = 1000000;
    double arr[n];  /* preencher array... */

    // OpenMP divide o loop entre os cores disponíveis
    // reduction(+:soma): cada thread calcula parcial, runtime combina
    #pragma omp parallel for reduction(+:soma) schedule(static)
    for (int i = 0; i < n; i++) {
        soma += arr[i];
    }
    // Thread-safe automaticamente pela clausula reduction
    printf("Soma: %f\n", soma);
    return 0;
}
java — ForkJoinPool e parallel streams
import java.util.Arrays;
import java.util.stream.IntStream;

// Parallel Stream: usa ForkJoinPool common pool automaticamente
double soma = IntStream.range(0, 1_000_000)
    .parallel()                   // ativa paralelismo
    .mapToDouble(i -> arr[i])
    .sum();

// ForkJoinPool customizado para controlar o grau de paralelismo
ForkJoinPool pool = new ForkJoinPool(8);  // 8 threads
double resultado = pool.submit(() ->
    Arrays.stream(arr).parallel().sum()
).get();

5. Paralelismo Massivo em GPU

Uma GPU moderna tem 10.000+ nucleos CUDA versus 16–64 de uma CPU topo. Cada nucleo e mais simples, mas todos executam a mesma instrucao em dados diferentes — modelo SIMT (Single Instruction, Multiple Threads). Para operacoes massivamente paralelas sobre arrays, a GPU supera a CPU em 10–100x.

cuda c++ — adicao de vetores: um elemento por thread
// __global__: funcao executada na GPU, chamada pela CPU
__global__ void vetorAdd(float* A, float* B, float* C, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    // Cada thread calcula seu indice e processa UM elemento
    if (i < n) C[i] = A[i] + B[i];
}

int main() {
    int n = 1024 * 1024;  // 1 milhao de elementos
    float *d_A, *d_B, *d_C;
    cudaMalloc(&d_A, n * sizeof(float));
    cudaMalloc(&d_B, n * sizeof(float));
    cudaMalloc(&d_C, n * sizeof(float));

    // Lançar 1024 blocos de 1024 threads = 1.048.576 threads simultaneas
    vetorAdd<<<1024, 1024>>>(d_A, d_B, d_C, n);

    // Na CPU sequencial: 1M iteracoes. Na GPU: todas ao mesmo tempo.
    // CUDA cuida do escalonamento nos cores automaticamente.
}

Quando GPU vence CPU: operacoes sobre arrays grandes e regulares (produto de matrizes, convolucoes, reducoes). Quando CPU vence GPU: algoritmos com muitas ramificacoes condicionais (if/else por elemento), acesso irregular a memoria (arvores, grafos com ponteiros) e tarefas de baixa latencia — a transferencia de dados CPU⇆GPU tem overhead de centenas de microsegundos.

6. Quando Paralelizar

Ideal Para Paralelizar

  • Treinamento de modelos ML/DL: operacoes de matriz que ja expressam paralelismo de dados
  • Processamento de imagens e video: pixels sao independentes entre si
  • Simulacoes fisicas e numericas (CFD, Monte Carlo, meteorologia, financas quantitativas)
  • Compressao e criptografia em lote: cada bloco e independente
  • Renderizacao 3D: cada raio de luz e independente (ray tracing embarassingly parallel)
  • Processamento de grandes datasets (Spark, MapReduce) em clusters

Algoritmos Naturalmente Paralelizaveis

  • Embarassingly parallel: sem dependencias entre trabalhos (batch de imagens)
  • Map/filter sobre colecoes de elementos independentes
  • Produto de matrizes e operacoes de algebra linear (BLAS)
  • Busca em espaco de estados: diferentes ramos da arvore em paralelo
  • Geracao de numeros pseudoaleatorios em lote com seeds independentes

Nao Paralelizar Quando

  • A tarefa e I/O-bound: use concorrencia (async/await) em vez de threads de CPU
  • O dataset e pequeno: overhead de criacao de threads supera o ganho da paralelizacao
  • O algoritmo tem dependencias fortes entre passos (recorrencias: LSTM puro, Fibonacci)
  • A parte paralelizavel e inferior a 50% (Lei de Amdahl limita o ganho a 2x)

Armadilhas do Paralelismo

  • False sharing: threads em cores diferentes escrevem em posicoes adjacentes da mesma cache line, causando invalidades de cache constantes
  • Load imbalance: um worker recebe muito mais trabalho que os outros — os demais ficam ociosos esperando
  • Memory bandwidth: GPUs e CPUs multi-core sao frequentemente limitadas pela banda de memoria, nao pelo numero de cores
  • Overhead de sincronizacao: barreiras e reducoes frequentes anulam o ganho do paralelismo

7. Topicos Avancados

MPI — Paralelismo em Clusters Distribuidos

Message Passing Interface e o padrao para paralelismo em supercomputacao. Cada processo tem sua propria memoria; comunicacao exclusivamente via mensagens (MPI_Send, MPI_Bcast, MPI_Reduce). Roda em milhares de nos de forma coordenada. Usado em simulacoes de dinamica de fluidos, previsao do tempo e modelagem climatica em supercomputadores.

SIMD — Paralelismo em Nivel de Instrucao

Single Instruction Multiple Data: a CPU processa multiplos valores com uma unica instrucao via registradores vetoriais largos (SSE: 128 bits, AVX: 256 bits, AVX-512: 512 bits). Auto-vetorizacao de compiladores com -O2. NumPy, OpenBLAS e o proprio GCC exploram SIMD para operacoes de array sem escrever uma linha de codigo SIMD explicita.

GPU em Deep Learning — Tensor Cores

GPUs modernas (NVIDIA H100, A100) tem Tensor Cores especializados em multiplicacoes de matrizes FP16/BF16/INT8. PyTorch e TensorFlow compilam operacoes para cuBLAS e cuDNN automaticamente. O GPT-3 foi treinado em 10.000 GPUs A100 com paralelismo de dados, pipeline e tensor combinados.

MapReduce e Apache Spark

MapReduce (Google, 2004) formalizou o processamento paralelo distribuido em map (transformar dados em paralelo) e reduce (agregar resultados). Apache Spark evoluiu o modelo com DAGs em memoria, Spark ML e Structured Streaming. Processa petabytes em centenas de nos com tolerancia a falhas automatica via lineage.

Paralelize o gargalo, nao o codigo todo. Profiling antes de paralelizar e obrigatorio: perf (Linux), gprof (GCC), cProfile (Python), Nsight Compute (CUDA/GPU). Em 90% dos casos, 10% do codigo consome 90% do tempo. Paralelizar a parte errada e trabalho inutil que adiciona complexidade e sincronizacao sem beneficio mensuravel de performance.

8. Ecossistema e Ferramentas

OpenMP

API de paralelismo para C, C++ e Fortran com diretivas de compilador (#pragma omp). Suportada por GCC, Clang e MSVC. Modelo fork-join: threads criadas na entrada da regiao paralela, destruidas ao final. Ideal para loops numericos, simulacoes cientificas e HPC sem precisar reescrever o codigo.

MPI (Open MPI / MPICH)

Padrao para computacao distribuida em clusters. Processos completamente isolados comunicam-se via passagem de mensagens. Open MPI e MPICH sao as implementacoes open-source dominantes. Roda em supercomputadores com dezenas de milhares de nos. Binding para Python: mpi4py.

NVIDIA CUDA / cuBLAS / cuDNN

CUDA e a plataforma de programacao GPU da NVIDIA. cuBLAS fornece operacoes de algebra linear otimizadas para GPU (GEMM, GEMV). cuDNN implementa primitivos de deep learning (convolucoes, batch norm, multi-head attention). PyTorch e TensorFlow usam cuBLAS/cuDNN de forma transparente via .to('cuda').

Apache Spark

Motor de processamento distribuido para big data. DataFrames sao particionados entre workers e processados em paralelo. Lazy evaluation: operacoes sao otimizadas por Catalyst antes de executar. Versoes recentes incluem Spark Connect, Delta Lake e integracao com Pandas via Arrow.

Python — multiprocessing e Ray

multiprocessing.Pool contorna o GIL com processos separados. Ray e um framework distribuido que escala de um laptop a um cluster AWS com a mesma API — ideal para RL, hyperparameter search e servico de modelos. Joblib e popular para paralelismo de sklearn.

Java — ForkJoinPool e Parallel Streams

ForkJoinPool implementa work-stealing: threads ociosas "roubam" tarefas de threads sobrecarregadas, mantendo balanceamento automatico. Arrays.parallelSort(), parallelStream() e RecursiveTask sao os pontos de entrada mais comuns para paralelismo de dados em Java.
