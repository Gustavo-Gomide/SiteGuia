Programacao Concorrente

Concorrencia — Multiplas Tarefas, Uma Vez

Gerenciando complexidade quando varias coisas precisam acontecer ao mesmo tempo.

Em 2009, Ryan Dahl mostrou que o Node.js conseguia lidar com 10.000 conexoes simultaneas em uma unica thread — algo que servidores Java com uma thread por conexao custavam centenas de MB para fazer. No mesmo ano, o Google lancou Go com goroutines, tornando concorrencia acessivel a qualquer programador. O problema da concorrencia nunca foi velocidade: foi domar o caos do estado compartilhado.

1. O Que E Concorrencia

Concorrencia e a capacidade de um sistema lidar com multiplas tarefas em sobreposicao de tempo — nao necessariamente executando-as simultaneamente. A distincao e fundamental: concorrencia e sobre estrutura, paralelismo e sobre execucao.

Rob Pike (criador do Go): "Concurrency is about dealing with lots of things at once. Parallelism is about doing lots of things at once. Concurrency is about structure, parallelism is about execution." Um sistema pode ser concorrente sem ser paralelo — como um servidor Node.js em uma unica CPU que atende centenas de conexoes ao mesmo tempo via event loop.

2. Os Tres Modelos de Execucao

Todo sistema concorrente e construido sobre um de tres primitivos fundamentais. Entender o custo e o isolamento de cada um e essencial para escolher o modelo certo.

diagrama — hierarquia de custo e isolamento
-- Processo: isolamento maximo, custo maximo
Process A  [memoria propria ~50 MB]  PID 1001
Process B  [memoria propria ~50 MB]  PID 1002
-- Comunicacao via IPC (pipes, sockets, shared memory)
-- Uma falha nao corrompe outros processos

-- Thread: memoria compartilhada, custo medio
Process A  [heap compartilhado: 50 MB]
  ├─ Thread 1  [stack: ~2 MB]
  ├─ Thread 2  [stack: ~2 MB]
  └─ Thread 3  [stack: ~2 MB]
-- Comunicacao via variaveis compartilhadas (cuidado: race conditions!)

-- Corrotina/Goroutine: custo minimo, eschedula em nivel de usuario
OS Thread 1  [heap compartilhado]
  goroutine 1  (~2 KB stack)
  goroutine 2  (~2 KB stack)
  ...
  goroutine 10000  (~2 KB stack)
-- Comunicacao via channels (Go) ou await (Python/JS)

Processo

Unidade de isolamento maxima do SO. Cada processo tem seu proprio espaco de memoria, descritores de arquivo e estado — uma falha nao corrompe outros processos. Comunicacao via IPC (pipes, sockets, shared memory). Custo de criacao alto (~ms).

Exemplos: Nginx (multi-process), Chrome (processo por aba), Python multiprocessing.

Thread

Unidade de execucao dentro de um processo. Threads compartilham heap — isso e tanto vantagem (sem custo de copia de dados) quanto perigo (race conditions). O kernel as escalona nos cores disponíveis. Custo de criacao medio (~microsegundos, ~2 MB de stack).

Exemplos: Java threads, pthreads (C/C++), Python threading.

Corrotina / Goroutine / Task

Concorrencia cooperativa em nivel de usuario — o scheduler e parte da linguagem, nao do SO. Custo minimo (~2 KB de stack vs ~2 MB de OS thread). Uma unica OS thread roda milhares de corrotinas via event loop ou M:N scheduling.

Exemplos: Go goroutines, Python asyncio Tasks, JavaScript async/await, Erlang processes.

3. Race Condition e Deadlock

Compartilhar estado entre threads e a fonte de dois dos bugs mais dificeis de reproduzir e depurar na computacao: race conditions e deadlocks.

Race Condition — O Resultado Depende do Timing

python — race condition classica em contador
import threading

counter = 0

def incrementar():
    global counter
    for _ in range(100_000):
        counter += 1  # NAO e atomico!
        # CPU faz: LOAD counter, ADD 1, STORE counter
        # Outra thread pode STORE entre LOAD e ADD

threads = [threading.Thread(target=incrementar) for _ in range(5)]
for t in threads: t.start()
for t in threads: t.join()

print(counter)  # Esperado: 500000. Real: ~283471 (varia!)

Deadlock — Espera Circular

python — deadlock: thread 1 e 2 esperam uma pela outra
lock_a = threading.Lock()
lock_b = threading.Lock()

def thread_1():
    with lock_a:          # Thread 1 adquire lock_a
        time.sleep(0.1)
        with lock_b:      # Thread 1 espera por lock_b
            pass          # mas Thread 2 ja segura lock_b!

def thread_2():
    with lock_b:          # Thread 2 adquire lock_b
        time.sleep(0.1)
        with lock_a:      # Thread 2 espera por lock_a
            pass          # DEADLOCK: espera circular infinita

Race conditions sao corrigidas com sincronizacao (Mutex, operacoes atomicas). Deadlocks sao evitados com ordenacao consistente de locks (sempre adquira na mesma ordem) ou com modelos sem lock compartilhado (Actor Model, channels do Go). Ambos desaparecem com imutabilidade — se ninguem pode alterar o estado, nao ha o que proteger.

4. Mecanismos de Sincronizacao

Quando threads precisam acessar estado compartilhado, sincronizacao garante exclusividade. Cada primitivo tem seu caso de uso e suas armadilhas.

python — Mutex (Lock) corrigindo a race condition
lock = threading.Lock()
counter = 0

def incrementar_seguro():
    global counter
    for _ in range(100_000):
        with lock:      # adquire lock, entra na secao critica
            counter += 1
        # lock liberado automaticamente ao sair do bloco with

# Agora counter sera sempre 500000 — race condition eliminada
go — channels como alternativa ao mutex (modelo CSP)
// comunicar via channel, nao via variavel compartilhada
ch   := make(chan int, 100)
done := make(chan bool)
counter := 0

// Goroutine "dona" do estado — a unica que toca counter
go func() {
    for v := range ch { counter += v }
    done <- true
}()

// Outros workers enviam incrementos via channel, nao modificam counter
for i := 0; i < 100_000; i++ { ch <- 1 }
close(ch)
<-done

// "Do not communicate by sharing memory;
// share memory by communicating." — Go Proverbs
java — AtomicInteger: lock-free com instrucao CAS da CPU
import java.util.concurrent.atomic.AtomicInteger;

AtomicInteger counter = new AtomicInteger(0);

// compare-and-swap (CAS): atomico em nivel de hardware
// Instrucao x86: LOCK XADD — nao usa lock do SO
for (int i = 0; i < 100_000; i++) {
    counter.incrementAndGet();  // atomico, sem mutex
}

// Sem mutex = sem risco de deadlock
// Sem context switch = menor overhead
// Ver tambem: AtomicLong, AtomicReference, LongAdder

5. Quando Usar Concorrencia

Ideal Para

  • Servidores web e APIs: cada requisicao em goroutine/thread/task independente
  • I/O-bound tasks: leitura de arquivos, chamadas de rede, queries ao banco
  • Interfaces graficas: thread separada para nao travar a UI durante processamento
  • Consumidores de filas: multiplas workers processando Kafka ou RabbitMQ em paralelo
  • Web scrapers: paginas baixadas simultaneamente sem esperar uma de cada vez

Modelos Que Simplificam

  • Actor Model (Erlang/Elixir/Akka): sem estado compartilhado — mensagens sao copiadas
  • Event Loop (Node.js, asyncio): single-thread, sem race conditions por definicao
  • Channels (Go): comunicacao segura entre goroutines sem mutex explicito
  • Imutabilidade (Haskell, Clojure, Rust &T): sem estado mutavel, Zero sincronizacao necessaria

Evitar Quando

  • A tarefa e puramente CPU-bound e sequencial — concorrencia adiciona overhead sem ganho
  • O codigo precisa de estado compartilhado mutavel complexo entre muitas threads
  • A equipe nao tem experiencia com sincronizacao e debugging de race conditions
  • O algoritmo nao e naturalmente decomponivel em tarefas independentes

Armadilhas Comuns

  • Python GIL: threads nao paralelizam CPU-bound (use multiprocessing para isso)
  • Thread-per-request em alta escala: 10 mil conexoes = 10 GB so em stacks de threads
  • Locks aninhados sem ordem definida: receita garantida de deadlock eventual
  • Erros em goroutines/threads filhas sem tratamento: falhas silenciosas sao as piores

6. Topicos Avancados

Actor Model — Erlang, Elixir e Akka

Cada actor e uma unidade independente com seu proprio estado, mailbox e comportamento. Atores comunicam-se exclusivamente por mensagens assíncronas — nunca compartilham memoria. Elimina race conditions por design. Erlang usa o modelo desde 1986 para sistemas de telecom com 99.9999999% de uptime (nove noves). O WhatsApp chegou a 2 milhoes de conexoes simultaneas por servidor com Erlang.

STM — Software Transactional Memory

Aplica o conceito de transacoes de banco de dados para memoria compartilhada. Operacoes sao enquadradas em transacoes que comutam atomicamente ou fazem rollback em conflito — sem locks explicitos. Disponivel nativamente em Haskell (STM) e Clojure (refs + dosync). Elimina deadlocks e simplifica drasticamente o codigo de sincronizacao.

Lock-Free e Wait-Free Algorithms

Usam instrucoes atomicas da CPU (CAS — Compare-And-Swap) em vez de locks de SO. Lock-free: pelo menos uma thread progride a cada ciclo. Wait-free: toda thread progride em tempo finito. Essenciais em high-frequency trading, sistemas de tempo real e estruturas como ConcurrentHashMap (Java) e canais do Go.

Reactive Programming — Streams Assincronos

Modela computacao como fluxos de eventos assincronos. Em vez de callbacks ou locks, operadores como map, filter e flatMap compoem pipelines. RxJava, Project Reactor (Spring WebFlux), RxJS e Kotlin Flow sao as implementacoes principais. Base dos sistemas de streaming moderno (backpressure, error propagation).

A lei dos Heisenbugs: bugs de concorrencia desaparecem quando voce adiciona print statements para depura-los — porque o print altera o timing exato. Use ferramentas especializadas: ThreadSanitizer (Go, C++, Rust), Helgrind (Valgrind para C/C++), asyncio debug mode (Python). Em producao, metricas de contencao de lock e histogramas de latencia revelam gargalos de sincronizacao antes que virem incidentes.

7. Ecossistema e Ferramentas

Java — ExecutorService e Virtual Threads

ExecutorService gerencia pools de threads com Executors.newFixedThreadPool(). Java 21 introduziu Virtual Threads (Project Loom): threads ultra-leves M:N, permitindo milhoes de threads concorrentes sem overhead de OS threads. CompletableFuture e ForkJoinPool completam o ecossistema.

Python — asyncio e concurrent.futures

asyncio implementa event loop single-thread para I/O-bound tasks com async/await. Para CPU-bound, use ProcessPoolExecutor (contorna o GIL). concurrent.futures oferece interface unificada para threads e processos.

Go — Goroutines e Channels

O modelo concorrente mais elegante da atualidade. Goroutines custam ~2 KB de stack (vs ~2 MB de OS thread). O runtime Go usa M:N scheduling distribuindo goroutines entre os cores. Channels implementam CSP (Communicating Sequential Processes). select, sync.WaitGroup e context completam o modelo.

Rust — Fearless Concurrency

O sistema de tipos do Rust garante que race conditions sejam erros de compilacao, nao bugs de runtime. Os traits Send e Sync marcam quais tipos podem traversar threads. O borrow checker impede aliasing mutavel. Tokio e o runtime async principal para servidores de alta performance.

Erlang / Elixir — OTP e Supervisores

A BEAM roda milhoes de processos leves com isolamento total. O framework OTP fornece supervisores que reiniciam processos falhos automaticamente — a base da filosofia "let it crash". Phoenix (Elixir) lida com 2 milhoes de conexoes WebSocket em hardware modesto. WhatsApp usou Erlang para 900 milhoes de usuarios com apenas 50 engenheiros.

Node.js — Event Loop Single-Threaded

Modelo I/O nao-bloqueante via event loop: uma unica thread processa eventos de uma fila enquanto operacoes I/O sao delegadas ao libuv (C). Sem race conditions em estado de aplicacao por definicao. Worker threads (desde Node 12) permitem CPU-bound em threads separadas. Streams e o modelo nativo para I/O de grande volume.
