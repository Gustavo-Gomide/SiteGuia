Classes e Objetos

O Molde e a Coisa

A classe é o projeto; o objeto é o que você constrói a partir dele.

Classes são o alicerce da Programação Orientada a Objetos. Elas definem a estrutura (atributos) e o comportamento (métodos) de um tipo de dado personalizado. Cada vez que você instancia uma classe obtém um objeto independente — um exemplar com seus próprios dados, mas compartilhando a mesma lógica definida no molde.

O que é uma Classe?

Uma classe é um tipo de dado definido pelo programador que agrupa dados (atributos) e funções que operam esses dados (métodos) em uma única unidade coesa. Pense em uma planta arquitetônica: o projeto descreve como construir uma casa — mas você pode erguer várias casas distintas a partir do mesmo projeto. Cada casa construída é um objeto (instância) da classe "casa".

Segundo Booch et al. (2007), uma classe é "uma descrição de um conjunto de objetos que compartilham os mesmos atributos, operações, relacionamentos e semântica". Em linguagens como Python, Java e C#, praticamente tudo que você manipula é um objeto de alguma classe.

Classe ≠ Objeto. A classe existe no código-fonte (tempo de compilação/definição); o objeto existe na memória durante a execução. Uma única classe pode originar milhões de objetos simultâneos, cada um com seus próprios dados mas compartilhando a mesma lógica.

Anatomia de uma Classe

Toda classe é composta por três elementos fundamentais: atributos (dados que o objeto possui), métodos (o que o objeto sabe fazer) e o construtor (código que inicializa o objeto no momento da criação).

carro.py — anatomia completa
class Carro:
    """Representa um veículo automotor."""

    # ── Atributo de CLASSE (compartilhado por todas as instâncias) ──
    total_fabricados: int = 0

    # ── Construtor: executado automaticamente ao criar o objeto ──
    def __init__(self, marca: str, modelo: str, ano: int) -> None:
        self.marca       = marca      # atributo de instância — público
        self.modelo      = modelo
        self.ano         = ano
        self._velocidade = 0          # _ = convenção "protegido"
        Carro.total_fabricados += 1

    # ── Método de instância: opera sobre self ──
    def acelerar(self, qtd: int) -> None:
        self._velocidade += qtd
        print(f"{self.modelo} → {self._velocidade} km/h")

    def frear(self, qtd: int) -> None:
        self._velocidade = max(0, self._velocidade - qtd)

    # ── Método de CLASSE: recebe cls, não self ──
    @classmethod
    def quantos_fabricados(cls) -> int:
        return cls.total_fabricados

# ── Uso ──
c1 = Carro("Toyota", "Corolla", 2022)
c2 = Carro("Honda", "Civic", 2023)
c1.acelerar(80)                        # Corolla → 80 km/h
print(Carro.quantos_fabricados())        # 2

Atributo de Classe

Pertence à classe em si, não a nenhum objeto específico. Alterá-lo afeta todas as instâncias (a menos que uma instância sobrescreva localmente). Ideal para contadores, constantes e configurações compartilhadas.

Atributo de Instância

Definido em __init__ via self.attr = valor. Cada objeto tem sua cópia independente — representa o estado individual do objeto.

Método de Instância

Recebe self — referência ao objeto chamador. Pode ler e modificar atributos de instância e de classe. É o tipo padrão em POO.

Método de Classe

Decorado com @classmethod. Recebe cls — referência à classe. Não acessa dados de instância. Útil como factory method (construtor alternativo).

Encapsulamento

Encapsulamento é o princípio de ocultar os detalhes internos de um objeto e expor apenas uma interface controlada. É como um controle remoto: você pressiona "aumentar volume" sem precisar saber como o circuito funciona internamente. O interior pode mudar completamente sem que nenhum código externo precise ser alterado.

Python usa convenções de nomenclatura em vez de modificadores de acesso como em Java ou C#. O efeito e a intenção são os mesmos:

Convenção Exemplo Visibilidade Quando usar
Sem prefixo self.nome Público — qualquer código acessa livremente Dados que fazem parte da interface pública do objeto
Um underscore _ self._velocidade Protegido — use com cuidado fora da classe Implementação interna que subclasses podem precisar acessar
Dois underscores __ self.__saldo Privado — Python aplica name mangling Dados críticos que jamais devem ser acessados externamente
conta_bancaria.py — @property como getter controlado
class ContaBancaria:
    def __init__(self, titular: str, saldo_inicial: float) -> None:
        self.titular   = titular        # público
        self.__saldo   = saldo_inicial  # privado → _ContaBancaria__saldo

    # Getter: lê __saldo sem expô-lo diretamente
    @property
    def saldo(self) -> float:
        return self.__saldo

    def depositar(self, valor: float) -> None:
        if valor <= 0:
            raise ValueError("Valor deve ser positivo")
        self.__saldo += valor

    def sacar(self, valor: float) -> None:
        if valor > self.__saldo:
            raise ValueError("Saldo insuficiente")
        self.__saldo -= valor

# Use correto: manipulação apenas via métodos
conta = ContaBancaria("Alice", 1000.0)
conta.depositar(500)
print(conta.saldo)      # 1500.0  — @property funciona como atributo
# conta.__saldo        → AttributeError: acesso direto bloqueado

O decorador @property transforma um método em um atributo de leitura — você mantém a sintaxe limpa obj.saldo enquanto executa lógica de controle por baixo. Para validar escritas, adicione @saldo.setter.

Métodos Especiais (Dunder Methods)

Python reconhece métodos com dois underscores em cada lado (dunder = double under) como ganchos do protocolo da linguagem. Ao implementá-los você ensina ao Python como seu objeto deve se comportar em operações nativas: impressão, comparação, aritmética, iteração, gerenciamento de contexto e muito mais.

vetor2d.py — dunder methods em prática
class Vetor2D:
    def __init__(self, x: float, y: float) -> None:
        self.x, self.y = x, y

    # Representação amigável: print(v)
    def __str__(self) -> str:
        return f"Vetor({self.x}, {self.y})"

    # Representação técnica: repr(v) e REPL
    def __repr__(self) -> str:
        return f"Vetor2D(x={self.x!r}, y={self.y!r})"

    # Adição: v1 + v2
    def __add__(self, outro: "Vetor2D") -> "Vetor2D":
        return Vetor2D(self.x + outro.x, self.y + outro.y)

    # Igualdade lógica: v1 == v2
    def __eq__(self, outro: object) -> bool:
        if not isinstance(outro, Vetor2D):
            return NotImplemented
        return self.x == outro.x and self.y == outro.y

    # Módulo: abs(v)
    def __abs__(self) -> float:
        return ((self.x ** 2) + (self.y ** 2)) ** 0.5

v1 = Vetor2D(3, 4)
v2 = Vetor2D(1, 2)
print(v1)           # Vetor(3, 4)
print(v1 + v2)      # Vetor(4, 6)
print(abs(v1))      # 5.0
print(v1 == v2)     # False

Tópicos Avançados

dataclasses — boilerplate zero

O módulo dataclasses (Python 3.7+) gera automaticamente __init__, __repr__ e __eq__ a partir de anotações de tipo. Para classes que são principalmente contêineres de dados, é a abordagem preferida — menos código, menos bugs.

dataclass_exemplo.py
from dataclasses import dataclass, field

@dataclass
class Produto:
    nome:  str
    preco: float
    tags:  list[str] = field(default_factory=list)  # lista mutável segura

    def com_desconto(self, pct: float) -> float:
        return self.preco * (1 - pct / 100)

p = Produto("Notebook", 3500.0, ["tech", "eletrônico"])
print(p)                   # Produto(nome='Notebook', preco=3500.0, tags=[...])
print(p.com_desconto(10))  # 3150.0

__slots__ — otimização de memória

Por padrão cada objeto Python armazena seus atributos em um dicionário interno (__dict__), o que é flexível mas consome memória. Declarar __slots__ substitui esse dicionário por descritores estáticos — reduzindo uso de memória em 30–60% para classes com milhares de instâncias simultâneas.

slots_exemplo.py
class Ponto:
    __slots__ = ("x", "y")  # apenas esses atributos são permitidos

    def __init__(self, x: float, y: float) -> None:
        self.x, self.y = x, y

p = Ponto(1.0, 2.0)
# p.z = 3.0  → AttributeError: z não está em __slots__
# p.__dict__ → AttributeError: __slots__ elimina o dicionário

Armadilha clássica: nunca use um objeto mutável (lista, dicionário) como valor padrão de argumento no construtor — ele é criado uma única vez e compartilhado entre todas as instâncias. Use None como padrão e inicialize dentro do __init__. Com dataclass, use field(default_factory=list).
