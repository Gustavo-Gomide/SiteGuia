Interfaces e Classes Abstratas

Programe para Contratos, Não para Implementações

Defina o "o quê" sem ditar o "como".

Uma interface é um contrato: ela declara quais operações um objeto deve oferecer, sem implementar nenhuma delas. Quem assina o contrato (implementa a interface) garante que qualquer código que depende do contrato funcionará corretamente — independentemente de qual implementação concreta esteja por baixo. É o princípio da inversão de dependência: código de alto nível depende de abstrações, não de detalhes.

Interface vs. Classe Abstrata

Em linguagens como Java e C# existe a palavra-chave interface explicitamente. Python unifica esse conceito na Classe Abstrata Base (ABC) do módulo abc. A distinção prática é:

Característica Interface pura Classe Abstrata
Atributos de instância Não possui Pode ter
Métodos concretos (com corpo) Não possui Pode ter (código reutilizável)
Métodos abstratos Todos os métodos Ao menos um
Instanciação direta Impossível Impossível
Herança múltipla Comum e incentivada Possível mas cuidadosa
Em Python ABC apenas com @abstractmethod ABC com mix de métodos concretos + abstratos

Python também oferece Protocol (PEP 544, Python 3.8+) para structural subtyping (duck typing estático) sem herança explícita. Veremos ambas abordagens nesta página.

ABC e @abstractmethod

O módulo abc (Abstract Base Classes) fornece a infraestrutura para criar classes abstratas em Python. Ao herdar de ABC e marcar métodos com @abstractmethod, Python impede a instanciação de qualquer classe que não implemente todos os métodos abstratos.

forma_geometrica.py — ABC como interface
from abc import ABC, abstractmethod
import math

class FormaGeometrica(ABC):
    """Contrato: toda forma deve calcular área e perímetro."""

    @abstractmethod
    def area(self) -> float:
        """Retorna a área da forma em unidades²."""

    @abstractmethod
    def perimetro(self) -> float:
        """Retorna o perímetro da forma em unidades."""

    # Método concreto na abstrata: disponível a todos
    def descricao(self) -> str:
        return (f"{type(self).__name__}: "
                f"área={self.area():.2f}, perímetro={self.perimetro():.2f}")


class Circulo(FormaGeometrica):
    def __init__(self, raio: float) -> None:
        self.raio = raio

    def area(self) -> float:
        return math.pi * self.raio ** 2

    def perimetro(self) -> float:
        return 2 * math.pi * self.raio


class Retangulo(FormaGeometrica):
    def __init__(self, largura: float, altura: float) -> None:
        self.largura, self.altura = largura, altura

    def area(self) -> float:
        return self.largura * self.altura

    def perimetro(self) -> float:
        return 2 * (self.largura + self.altura)


# FormaGeometrica()  → TypeError: Can't instantiate abstract class
formas = [Circulo(5), Retangulo(4, 6)]
for f in formas:
    print(f.descricao())
# Circulo: área=78.54, perímetro=31.42
# Retangulo: área=24.00, perímetro=20.00

@abstractmethod pode ser combinado com @property, @classmethod e @staticmethod para exigir que subclasses forneçam propriedades ou métodos de classe específicos.

Protocol — Duck Typing Estático

Protocol (PEP 544, Python 3.8+) permite que uma classe satisfaça uma interface sem herdar explicitamente dela — basta implementar os métodos corretos. É o duck typing elevado ao nível da verificação estática de tipos (mypy, pyright).

A diferença é importante: ABC usa herança nominal (você precisa declarar class MinhaClasse(MinhaABC)); Protocol usa subtipagem estrutural (se tem os métodos certos, satisfaz o protocolo automaticamente).

protocol_exemplo.py — subtipagem estrutural
from typing import Protocol

class Desenhavel(Protocol):
    """Protocolo: qualquer objeto com .desenhar() é Desenhavel."""
    def desenhar(self) -> None: ...


# Nenhuma herança explícita de Desenhavel!
class Sprite:
    def desenhar(self) -> None:
        print("Sprite desenhado")


class Texto:
    def desenhar(self) -> None:
        print("Texto renderizado")


def renderizar(elemento: Desenhavel) -> None:
    elemento.desenhar()     # type checker aceita Sprite e Texto


renderizar(Sprite())    # Sprite desenhado
renderizar(Texto())     # Texto renderizado
# isinstance(Sprite(), Desenhavel) → True  (Python 3.12+: runtime checkable)

Use ABC quando:

  • A hierarquia é intencional e você quer isinstance() confiável em tempo de execução.
  • A classe abstrata fornece implementações parciais reutilizáveis.
  • Você controla todas as implementações (biblioteca interna).

Use Protocol quando:

  • Você não quer forçar herança em código de terceiros.
  • O contrato é leve (1–3 métodos).
  • A verificação em tempo de execução é dispensável ou secundária.

Tópicos Avançados

Propriedades Abstratas com @property

Combine @property com @abstractmethod para exigir que subclasses exponham determinados dados como propriedades — garantindo a interface pública sem ditar como o dado é armazenado internamente.

abstract_property.py
from abc import ABC, abstractmethod

class Autenticavel(ABC):
    @property
    @abstractmethod
    def token(self) -> str: ...

    @abstractmethod
    def autenticar(self) -> bool: ...


class UsuarioJWT(Autenticavel):
    def __init__(self, jwt: str) -> None:
        self._jwt = jwt

    @property
    def token(self) -> str:
        return self._jwt

    def autenticar(self) -> bool:
        return len(self._jwt) > 0

__subclasshook__ — registro virtual

ABCs podem aceitar subclasses virtuais — classes que não herdam explicitamente mas que a ABC reconhece como implementações válidas via __subclasshook__. Isso é o que permite que isinstance([], Iterable) retorne True sem list herdar explicitamente de Iterable.

subclasshook_exemplo.py
from abc import ABC, abstractmethod

class Comparable(ABC):
    @abstractmethod
    def __lt__(self, outro): ...

    @classmethod
    def __subclasshook__(cls, C) -> bool:
        # Qualquer classe com __lt__ é considerada Comparable
        if cls is Comparable:
            return hasattr(C, "__lt__")
        return NotImplemented

print(isinstance(42, Comparable))     # True — int tem __lt__
print(isinstance("hi", Comparable))  # True — str tem __lt__

Princípio da Inversão de Dependência (DIP): módulos de alto nível não devem depender de módulos de baixo nível — ambos devem depender de abstrações. Ao depender de uma ABC ou Protocol em vez de uma classe concreta, seu código aceita qualquer implementação futura sem ser modificado.
