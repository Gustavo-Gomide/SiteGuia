Programação Orientada a Objetos

Introdução à POO

Pense em Objetos, Não em Passos

A mudança de mentalidade que transformou a forma como o mundo escreve software.

A Programação Orientada a Objetos (POO) é um paradigma de programação que organiza o software em torno de objetos — entidades que combinam dados e comportamento em uma única unidade coesa. Em vez de pensar em uma lista de instruções para o computador executar (paradigma imperativo/procedural), você pensa em quais entidades existem no seu domínho, quais dados elas possuem e quais ações elas realizam. É a forma dominante de desenvolver software profissional há mais de quatro décadas.

Um Pouco de História

A POO não surgiu de uma hora para outra — foi uma evolução gradual em resposta às crescentes dificuldades de manter programas grandes e complexos.

Do procedural ao orientado a objetos

Nos anos 1950 e 1960 os programas eram escritos em linguagens como FORTRAN e COBOL: sequências lineares de instruções com GOTO e variáveis globais. Conforme os programas cresciam, o código tornava-se um espaguete impossível de manter — o que ficou conhecido como a "crise do software" dos anos 1960.

Em 1967, a linguagem Simula 67 (criada por Ole-Johan Dahl e Kristen Nygaard para simular sistemas físicos) introduziu os conceitos de classe e objeto pela primeira vez. Alan Kay, inspirado em Simula, criou o Smalltalk nos anos 1970 no PARC da Xerox e cunhou o próprio termo "orientado a objetos". Em seu modelo, tudo é um objeto — até números e classes.

Nos anos 1980 e 1990 o paradigma explodiu com C++ (Bjarne Stroustrup, 1983), Eiffel (Bertrand Meyer, 1986) e depois Java (Sun, 1995) e Python (Guido van Rossum, 1991). A publicação do livro Design Patterns pelo Gang of Four (1994) consolidou as melhores práticas. Hoje linguagens como C#, Kotlin, Swift, Dart e inúmeras outras são primariamente orientadas a objetos.

O que é um Paradigma de Programação?

Um paradigma é uma forma de pensar e estruturar programas — uma filosofia de design. Não é uma linguagem específica; é uma abordagem mental. Existem vários paradigmas e a maioria das linguagens modernas suporta mais de um:

Paradigma Foco principal Exemplos de linguagens
Imperativo / Procedural Sequência de instruções; funções e procedimentos C, Pascal, BASIC
Orientado a Objetos Objetos com estado e comportamento; mensagens entre objetos Java, C++, Python, C#, Ruby
Funcional Funções puras; sem estado mutável; composição Haskell, Erlang, Clojure, F#
Lógico / Declarativo Descreve o problema; o sistema deduz a solução Prolog, SQL, Datalog
Multiparadigma Permite misturar paradigmas conforme o problema Python, Scala, Rust, Kotlin

Python é multiparadigma: você pode escrever código procedural, orientado a objetos ou funcional — e até misturá-los no mesmo projeto. A POO em Python não é obrigada, mas é a escolha natural para sistemas maiores e mais complexos.

Objetos e Classes — a Fundação

Dois conceitos são absolutamente centrais em POO — tudo mais deriva deles:

Objeto

Uma instância concreta que existe na memória durante a execução. Cada objeto possui:

  • Estado — dados armazenados nos seus atributos (self.nome, self.saldo).
  • Comportamento — ações que ele sabe executar (seus métodos).
  • Identidade — cada objeto é único na memória, mesmo que tenha os mesmos dados de outro.

Classe

O molde / projeto / blueprint que define como objetos desse tipo são construídos. A classe:

  • Declara quais atributos os objetos terão.
  • Define os métodos disponíveis.
  • Não ocupa espaço de dados — apenas o objeto instanciado ocupa.
  • Pode originar milhares de objetos independentes.
ola_poo.py — primeiro contato com classe e objeto
class Pessoa:
    """Molde para representar uma pessoa."""

    def __init__(self, nome: str, idade: int) -> None:
        self.nome  = nome   # dado (estado)
        self.idade = idade

    def saudacao(self) -> str:   # comportamento
        return f"Oi, sou {self.nome} e tenho {self.idade} anos."

    def aniversario(self) -> None:
        self.idade += 1           # muda o estado


# Criando objetos a partir do molde
p1 = Pessoa("Alice", 28)
p2 = Pessoa("Bob",   35)

print(p1.saudacao())    # Oi, sou Alice e tenho 28 anos.
print(p2.saudacao())    # Oi, sou Bob e tenho 35 anos.
p1.aniversario()
print(p1.idade)         # 29 — só p1 mudou; p2 não foi afetado

Os Quatro Pilares da POO

A POO repousa sobre quatro princípios fundamentais. Juntos eles conferem ao paradigma sua capacidade de produzir código reutilizável, extensível e manutenível. Cada pilar resolve um problema real que os desenvolvedores enfrentavam com programação procedural em larga escala.

1º Pilar — Encapsulamento

Encapsulamento é o princípio de agrupar dados e os métodos que operam sobre esses dados em uma unidade coesa (a classe), e de controlar o acesso ao estado interno do objeto. O mundo externo interage com o objeto apenas pela sua interface pública — sem precisar conhecer os detalhes da implementação.

A analogia perfeita é um controle remoto: você pressiona "ligar" sem saber nada sobre o circuito interno. O fabricante pode redesenhar o circuito completamente e você continua usando o mesmo botão. É exatamente isso que o encapsulamento garante no software.

Benefícios

  • Protege o estado interno de modificações acidentais
  • Permite mudar a implementação interna sem quebrar código que usa a classe
  • Facilita a validação de dados (aceitar só valores válidos)
  • Reduz o acoplamento entre partes do sistema

Em Python

  • self.attr — público
  • self._attr — "protegido" (convenção)
  • self.__attr — "privado" (name mangling)
  • @property — getter/setter controlado
encapsulamento.py — controle de acesso
class Termostato:
    def __init__(self, temp_inicial: float) -> None:
        self.__temperatura = temp_inicial  # privado

    @property
    def temperatura(self) -> float:
        return self.__temperatura

    @temperatura.setter
    def temperatura(self, valor: float) -> None:
        if not (10 <= valor <= 30):         # validação
            raise ValueError(f"Temperatura {valor}°C fora do intervalo [10, 30]")
        self.__temperatura = valor

t = Termostato(22.0)
t.temperatura = 25         # ok
t.temperatura = 50         # ValueError: fora do intervalo

2º Pilar — Herança

Herança permite que uma classe derive de outra, herdando automaticamente seus atributos e métodos. A subclasse pode estender (adicionar novos membros) ou sobrescrever (redefinir com comportamento especializado) o que ela herdou.

Herança modela a relação "é um" (is-a): um Cachorro é um Animal; uma ContaCorrente é uma ContaBancaria. O principal benefício é reutilizar código sem duplicação — escreva uma vez na superclasse, todas as subclasses herdam automaticamente.

heranca.py — exemplo introdutório
class Veiculo:
    def __init__(self, marca: str, velocidade_max: int) -> None:
        self.marca          = marca
        self.velocidade_max  = velocidade_max
        self._velocidade     = 0

    def acelerar(self, qtd: int) -> None:
        self._velocidade = min(self._velocidade + qtd, self.velocidade_max)


class Moto(Veiculo):            # Moto IS-A Veiculo
    def __init__(self, marca: str) -> None:
        super().__init__(marca, velocidade_max=180)
        self.tem_sidecar = False      # atributo exclusivo de Moto


class Caminhao(Veiculo):         # Caminhao IS-A Veiculo
    def __init__(self, marca: str, carga_max_t: float) -> None:
        super().__init__(marca, velocidade_max=110)
        self.carga_max_t = carga_max_t


m = Moto("Honda")
m.acelerar(60)                    # herdado de Veiculo
print(m._velocidade)               # 60

c = Caminhao("Volvo", 20.0)
c.acelerar(200)                   # limitado a 110
print(c._velocidade)               # 110

3º Pilar — Polimorfismo

Polimorfismo (do grego: muitas formas) é a capacidade de um mesmo nome de método executar comportamentos distintos dependendo do tipo do objeto que o recebe. É o que permite escrever código genérico que funciona com qualquer tipo que cumpra um contrato, sem ifs ou switches.

Em Python o polimorfismo é especialmente natural graças ao duck typing: se um objeto tem o método esperado, ele funciona — independentemente de onde vem na hierarquia de classes. Você não precisa declarar que implementa uma interface; basta tê-la.

polimorfismo.py — mesmo loop, comportamentos diferentes
class Gato:
    def falar(self) -> str: return "Miau"

class Cachorro:
    def falar(self) -> str: return "Au au"

class Papagaio:
    def falar(self) -> str: return "Olá!"

# Nenhuma herança comum — duck typing puro
animais = [Gato(), Cachorro(), Papagaio(), Gato()]
for a in animais:
    print(a.falar())    # Miau / Au au / Olá! / Miau
# O loop não muda ao adicionar novos tipos

Polimorfismo é o que torna o código aberto para extensão, fechado para modificação (Princípio Aberto/Fechado — OCP): você adiciona novos tipos sem alterar o código existente.

4º Pilar — Abstração

Abstração é o ato de modelar o essencial de uma entidade para determinado contexto, ignorando tudo que não é relevante. Quando você cria a classe Carro, você não modela cada átomo do motor — você modela apenas o que importa para o seu sistema: marca, velocidade, acelerar(), frear().

Abstração vai além de simplesmente "simplificar": ela define quais conceitos do domínio do problema existem como intidades no software. Em um sistema bancário você abstrai Conta, Transacao, Cliente. Em um jogo você abstrai Personagem, Mapa, Inimigo.

Em Python abstração formal é implementada com classes abstratas (ABC) e Protocols: você declara o contrato sem especificar a implementação.

abstracao.py — abstraindo um sistema de notificação
from abc import ABC, abstractmethod

# ABSTRAÇÃO: define o contrato "o que" sem ditar "como"
class Notificador(ABC):
    @abstractmethod
    def enviar(self, destinatario: str, mensagem: str) -> None: ...


# Implementações concretas — cada uma a seu modo
class Email(Notificador):
    def enviar(self, dest: str, msg: str) -> None:
        print(f"[E-MAIL] Para: {dest} | {msg}")

class SMS(Notificador):
    def enviar(self, dest: str, msg: str) -> None:
        print(f"[SMS] {dest}: {msg[:160]}")

class Push(Notificador):
    def enviar(self, dest: str, msg: str) -> None:
        print(f"[PUSH] → {dest}: {msg}")


# Código de alto nível depende APENAS da abstração
def alertar_usuario(notificador: Notificador, usuario: str) -> None:
    notificador.enviar(usuario, "Compra aprovada!")

alertar_usuario(Email(), "alice@ex.com")
alertar_usuario(SMS(),   "+5511...")
alertar_usuario(Push(),  "device:abc")

Os Princípios SOLID

A sigla SOLID reúne cinco princípios de design orientado a objetos formulados por Robert C. Martin que, quando seguidos, produzem código mais limpo, testável e resiliente a mudanças. São o passo natural após dominar os quatro pilares.

Letra Princípio Resumo
S Single Responsibility (SRP) Uma classe deve ter apenas uma razão para mudar
O Open/Closed (OCP) Aberto para extensão, fechado para modificação
L Liskov Substitution (LSP) Subclasses devem substituir superclasses sem quebrar o comportamento
I Interface Segregation (ISP) Interfaces pequenas e coesas; não force implementar o que não é necessário
D Dependency Inversion (DIP) Dependa de abstrações, não de implementações concretas

SOLID não é sobre seguir regras cegamente — é sobre entender por quê o código se torna difícil de manter e ter princípios que guiem as decisões de design. Você já viu DIP em ação no exemplo de Notificador acima.

Vantagens e Quando Usar POO

Vantagens

  • Modelagem natural — entidades do mundo real viram classes
  • Reutilização — herança e composição eliminam duplicação
  • Manutenibilidade — encapsulamento isola mudanças
  • Extensibilidade — polimorfismo permite adicionar tipos novos sem alterar código existente
  • Colaboração — interfaces bem definidas permitem que times trabalhem em paralelo

Limitações (use com consciência)

  • Overhead de abstração — para scripts simples, POO é burocracia desnecessária
  • Hierarquias frágeis — herança profunda demais cria acoplamento oculto
  • Difícil de testar sem disciplina — estado mutável complica testes unitários
  • Não é bala de prata — dados + transformações puras são mais simples com funcional

Não force POO. Python é multiparadigma: use classes quando elas modelam entidades com estado e comportamento que persistem ao longo do programa. Para transformações de dados simples e pipelines, funções e compreensões de lista são mais claras.
