Herança

Herança

Herdar, Especializar, Reutilizar

Não escreva o que já existe — herde e adapte.

Herança é o mecanismo pelo qual uma classe (subclasse) adquire automaticamente os atributos e métodos de outra (superclasse). É a forma mais direta de reutilização de código em POO e a base para polimorfismo baseado em subtipo. Usada corretamente reduz duplicação; usada errada cria hierarquias frágeis difíceis de manter.

O que é Herança?

Herança modela a relação "é um" (is-a): se Cachorro herda de Animal, um cachorro é um animal. Isso significa que a subclasse pode ser usada em qualquer lugar onde a superclasse é esperada — o chamado Princípio de Substituição de Liskov (LSP).

A subclasse herda tudo que é público e protegido da superclasse. Ela pode estender adicionando novos atributos/métodos, ou sobrescrever (override) métodos existentes para fornecer comportamento especializado.

Sintaxe e super()

Em Python basta passar a superclasse entre parênteses na definição da subclasse. Para reutilizar o construtor da superclasse, use super().__init__() — isso garante que toda inicialização da hierarquia seja executada na ordem correta, mesmo em herança múltipla.

heranca_basica.py
class Animal:
    def __init__(self, nome: str) -> None:
        self.nome = nome

    def falar(self) -> str:
        return f"{self.nome} faz algum som"


class Cachorro(Animal):           # herda de Animal
    def __init__(self, nome: str, raca: str) -> None:
        super().__init__(nome)       # delega a Animal.__init__
        self.raca = raca

    # Override: substitui o falar() da superclasse
    def falar(self) -> str:
        return f"{self.nome} late: Au au!"

    # Extend: método novo que só Cachorro tem
    def buscar(self, item: str) -> str:
        return f"{self.nome} buscou: {item}"


rex = Cachorro("Rex", "Labrador")
print(rex.nome)          # Rex       ← herdado de Animal
print(rex.falar())       # Rex late: Au au!  ← override
print(rex.buscar("bola")) # Rex buscou: bola ← método novo
print(isinstance(rex, Animal))  # True ← Cachorro IS-A Animal

Sempre chame super().__init__() ao sobrescrever o construtor — não fazê-lo pode deixar a superclasse não inicializada e causar erros sutis em tempo de execução.

Tipos de Herança

Python suporta todos os tipos de herança da teoria POO. Entender cada tipo ajuda a escolher a estrutura certa para cada problema.

Simples

Uma subclasse herda de UMA única superclasse. Forma mais comum e mais fácil de entender. Cria uma cadeia linear de especializações.

Veiculo → Carro → Esportivo

Múltipla

Uma subclasse herda de MAIS de uma superclasse. Python suporta; Java e C# não. Poderoso mas perigoso — use mixins para controlar a complexidade.

class Anfibio(Veiculo, Barco): ...

Multinível

Uma cadeia de heranças: A → B → C. A subclasse herda de uma superclasse que ela mesma é subclasse de outra. MRO garante a ordem correta de resolução.

Animal → Mamífero → Cachorro

Mixin

Classes pequenas e focadas que adicionam uma única capacidade. São usadas em herança múltipla para compor comportamentos sem criar hierarquias profundas.

class Admin(Usuario, LogMixin): ...
mixin_exemplo.py — composição por herança múltipla
class LogMixin:
    """Mixin: qualquer classe que herde ganha log automático."""
    def log(self, msg: str) -> None:
        print(f"[LOG] {type(self).__name__}: {msg}")


class SerializeMixin:
    """Mixin: serializa atributos para dicionário."""
    def to_dict(self) -> dict:
        return vars(self)


class Usuario:
    def __init__(self, nome: str, email: str) -> None:
        self.nome  = nome
        self.email = email


class Admin(LogMixin, SerializeMixin, Usuario):
    def __init__(self, nome: str, email: str, nivel: int) -> None:
        super().__init__(nome, email)
        self.nivel = nivel


admin = Admin("Ana", "ana@ex.com", 3)
admin.log("Usuário criado")  # [LOG] Admin: Usuário criado
print(admin.to_dict())         # {'nome': 'Ana', 'email': '...', 'nivel': 3}

Ordem de Resolução de Métodos (MRO)

Quando uma subclasse tem múltiplos antepassados, o Python precisa decidir qual implementação de um método usar quando há conflito. Ele segue o algoritmo C3 Linearization — uma topological sort que garante que:

  • Subclasses sempre precedem superclasses.
  • A ordem de declaração na definição da classe é respeitada.
  • A hierarquia nunca é violada (uma classe não pode aparecer antes de suas subclasses).
mro_exemplo.py — problema do diamante
class A:
    def metodo(self) -> str:
        return "A"

class B(A):
    def metodo(self) -> str:
        return "B → " + super().metodo()

class C(A):
    def metodo(self) -> str:
        return "C → " + super().metodo()

class D(B, C):   # herança em diamante: D → B → C → A
    pass

d = D()
print(d.metodo())       # B → C → A
print(D.__mro__)        # [D, B, C, A, object]

Use Classe.__mro__ ou help(Classe) para inspecionar a MRO e entender exatamente qual método será chamado. Em hierarquias complexas, isso evita bugs muito difíceis de rastrear.

Quando Usar Herança (e quando não usar)

A regra de ouro é o Princípio de Substituição de Liskov (LSP): uma subclasse deve poder substituir a superclasse sem quebrar o comportamento esperado do programa. Se a subclasse precisa remover funcionalidades da superclasse para fazer sentido, a herança está errada — use composição.

Use herança quando:

  • A relação é genuinamente "é um" — Cachorro é um Animal.
  • Você quer reutilizar código comum e especializar comportamento.
  • Subtipos devem ser intercambiáveis com o tipo base (LSP).
  • Hierarquias rasas (1-2 níveis) — profundidade demais torna o código frágil.

Prefira composição quando:

  • A relação é "tem um" — Carro tem um Motor.
  • Você quer combinar comportamentos de fontes distintas sem criar hierarquia.
  • A subclasse precisaria desabilitar ou lançar exceção em métodos da superclasse.
  • A herança é usada apenas para reaproveitar código (sem relação conceitual real).

Prefira composição sobre herança — um princípio do GoF (1994). Herança cria acoplamento forte entre classes: mudanças na superclasse podem quebrar subclasses silenciosamente. Composição é mais flexível e testável.
