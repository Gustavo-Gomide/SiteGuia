OrientDB

Banco multi-modelo híbrido: Orientação a Objetos + Documentos JSON + Grafos nativos — consultados com SQL estendido.

O OrientDB foi um dos pioneiros no conceito de banco multi-modelo. Ele armazena dados como documentos JSON orientados a objetos, conecta-os por meio de relacionamentos de grafo com ponteiros físicos diretos (RIDs), e permite consultá-los com uma extensão natural do SQL — sem precisar aprender uma linguagem completamente nova.

1. O Que É o OrientDB?

Pense no OrientDB como um banco que se recusa a colocar dados em caixas separadas. Um Vértice é ao mesmo tempo um documento JSON com campos livres, um nó de grafo com arestas conectadas, e uma instância de uma Classe que pode herdar de outra. Isso é possível graças ao sistema de Record ID (RID).

Record ID (RID): cada documento/vértice/aresta no OrientDB recebe um identificador físico no formato #cluster:position — por exemplo, #15:1234. Esse RID é um ponteiro direto para a posição no arquivo de dados. Navegar uma aresta de grafo é uma operação O(1) — nenhum lookup de índice necessário, nenhum JOIN.

2. Como Funciona por Dentro

O OrientDB organiza dados em Clusters (partições físicas de armazenamento) e Classes (tipos/schemas). Cada Classe tem um ou mais Clusters associados. Vértices estendem a Class especial V, arestas estendem E.

arquitetura-orientdb.txt
Classes (schema OO):
  V (Vertex base)
    └─► Pessoa       (#15:*)   ← vértice com dados de pessoa
    └─► Empresa      (#16:*)   ← vértice com dados de empresa

  E (Edge base)
    └─► Trabalha     (#20:*)   ← aresta Pessoa→Empresa com propriedades
    └─► Conhece      (#21:*)   ← aresta Pessoa→Pessoa

Armazenamento de uma aresta:
#20:0 = {
  _class: "Trabalha",
  _from:  "#15:0",   ← RID de Ana em Pessoa
  _to:    "#16:0",   ← RID de TechCorp em Empresa
  cargo:  "Engenheira",
  desde:  2021
}

Navegação de grafo = dereferência de ponteiro (O(1)):
  Ana (#15:0) → out(Trabalha) → TechCorp (#16:0)
  Sem índice. Sem JOIN. Zero lookup extra.

MVCC e Transações

O OrientDB usa MVCC (Multi-Version Concurrency Control) via campo @version em cada registro. Escritas concorrentes incrementam a versão — conflitos são detectados sem locks. Transações ACID são suportadas em modo local (single-server); em modo distribuído, o nível de isolamento é eventually consistent.

Modos de Schema

Full Schema: todos os campos declarados na Classe, validação rígida. Schema-Less: campos livres sem declaração prévia. Schema-Mixed: campos obrigatórios declarados + campos extras livres — o modo mais comum em produção, equilibra estrutura e flexibilidade.

Distributed (Multi-Master)

OrientDB suporta replicação multi-master via protocolo Hazelcast. Cada nó pode receber escritas. O algoritmo de consenso garante que conflitos sejam resolvidos determinísticamente por versão e timestamp — mas performance de writes distribuídos é inferior a sistemas baseados em Raft (como etcd ou CockroachDB).

3. OrientDB SQL — SQL com Travessia de Grafo

A linguagem de query do OrientDB é um superconjunto do SQL padrão. Quem já conhece SQL aprende em horas: há SELECT, INSERT, UPDATE, DELETE, subqueries e JOINs. Mas também há TRAVERSE, MATCH e funções de grafo nativas como out(), in(), both().

orientdb-sql.sql
-- 1. Criar Classes com herança OO
CREATE CLASS Pessoa EXTENDS V;
CREATE CLASS Gerente EXTENDS Pessoa;  -- herda todos os campos de Pessoa
CREATE CLASS Empresa EXTENDS V;
CREATE CLASS Trabalha EXTENDS E;
CREATE CLASS Conhece EXTENDS E;

-- 2. Inserir vértices (documentos + nós de grafo)
INSERT INTO Pessoa (nome, cidade, idade) VALUES ("Ana", "São Paulo", 32);
INSERT INTO Empresa (nome, setor) VALUES ("TechCorp", "Software");

-- 3. Criar aresta com propriedades
CREATE EDGE Trabalha
  FROM (SELECT FROM Pessoa WHERE nome = "Ana")
  TO   (SELECT FROM Empresa WHERE nome = "TechCorp")
  SET cargo = "Engenheira Sênior", desde = 2021;

-- 4. Navegar grafo com out()/in() — O(1) por RID
SELECT expand(out("Trabalha")) FROM Pessoa WHERE nome = "Ana";

-- 5. TRAVERSE — busca em profundidade ilimitada
TRAVERSE out("Conhece") FROM (SELECT FROM Pessoa WHERE nome = "Ana")
WHILE $depth <= 3;   -- amigos até 3 graus de separação

-- 6. MATCH — pattern matching tipo Cypher, mas em SQL
MATCH {class: Pessoa, as: p1, where: (nome = "Ana")}
  .out("Conhece") {class: Pessoa, as: p2}
  .out("Trabalha") {class: Empresa, as: emp}
RETURN p1.nome, p2.nome, emp.nome;

-- 7. SQL padrão continua funcionando
SELECT * FROM Pessoa WHERE cidade = "São Paulo" ORDER BY nome;

-- 8. Herança: query em Pessoa retorna também Gerentes (polimorfismo)
SELECT * FROM Pessoa;  -- inclui todos os Gerentes automaticamente

4. Quando Usar (e Quando Não Usar)

Use OrientDB quando...

  • Domínio hierárquico complexo: herança de classes é nativa — modelar Funcionário → Gerente → Diretor sem tabelas de join intermediárias.
  • Equipe familiarizada com SQL: curva de aprendizado drasticamente menor que Cypher (Neo4j) ou AQL (ArangoDB).
  • Grafos moderados com rich documents: cada vértice/aresta pode ter dezenas de campos livres — perfeito para entidades complexas com relacionamentos.
  • Aplicações Java: driver Java maduro, integração com Spring Data, e API de objetos Java que mapeia diretamente para classes OrientDB.
  • Prototipagem rápida: schema-mixed permite começar schema-less e adicionar constraints conforme o modelo estabiliza.

Evite OrientDB quando...

  • Grafos em escala de bilhões de relacionamentos: Neo4j tem engine nativa mais otimizada para traversals massivos.
  • Alta ingestão de dados (write-heavy): Cassandra e HBase são ordens de magnitude mais eficientes para escrita contínua de alto volume.
  • Projeto crítico de produção: OrientDB tem histórico instável de releases e menor comunidade — avalie ArangoDB como alternativa mais ativa.
  • Queries analíticas complexas: sem suporte nativo a window functions, CTEs avançadas ou OLAP — use PostgreSQL ou Spark.

5. Tópicos Avançados

Índices e Performance

OrientDB suporta vários tipos de índice: UNIQUE (hash), NOTUNIQUE, FULLTEXT (Lucene integrado), DICTIONARY e SPATIAL (R-Tree para geolocalização). Índices compostos em múltiplos campos são suportados e funcionam com o query planner para eliminar full scans.

Gremlin e Tinkerpop

OrientDB implementa o padrão Apache TinkerPop. Isso significa que você pode usar a linguagem Gremlin para traversals de grafo — útil para migrar de outros bancos TinkerPop (JanusGraph, Amazon Neptune) ou usar frameworks como JUNG para análise de grafos em Java.

Sharding e Clusters por Classe

Cada Classe pode ter múltiplos Clusters (shards físicos). O OrientDB usa round-robin por default — ou você implementa uma ClusterSelectionStrategy customizada para controlar em qual cluster cada registro é armazenado. Útil para separar dados por região geográfica ou tenant em multitenancy.

Reactive e Live Queries

Live Queries permitem subscrever a uma query SQL e receber eventos (INSERT, UPDATE, DELETE) em tempo real via WebSocket ou hook Java. Útil para dashboards e sistemas de notificação que precisam reagir a mudanças específicas no banco sem polling.

Status do Projeto: O OrientDB passou por mudanças de propriedade e períodos instáveis de desenvolvimento. Em 2023/2024, a comunidade migrou para forks e o projeto principal teve desenvolvimento reduzido. Para novos projetos que precisam de multi-modelo com grafo, avalie ArangoDB (mais ativo, melhor suporte) antes de escolher o OrientDB.

6. Ecossistema

OrientDB Studio

Interface web built-in em localhost:2480. Visualização de grafos, editor de queries SQL, browser de classes e clusters, monitoramento de performance e gerenciamento de usuários.

Java API (ODocument / ODatabaseSession)

API Java nativa com suporte a Object-Database Mapping (OOM). Anote POJOs com @OClass e @OProperty para mapeamento automático — similar ao JPA mas para grafo + documento.

Apache TinkerPop / Gremlin

Implementação completa do TinkerPop 3. Use Gremlin Server para expor o grafo OrientDB como endpoint Gremlin padrão — intercambiável com JanusGraph, Neptune e outros bancos TinkerPop.

Spring Data OrientDB

Integração com Spring Framework via Repository pattern. Mapeamento de entidades, suporte a @Query com SQL OrientDB estendido, e transações gerenciadas pelo Spring.

Lucene Full-Text Index

Lucene integrado como provider de índice full-text. Permite queries como SELECT FROM Artigo WHERE SEARCH_INDEX("texto", "inteligência artificial") = true — busca vetorial e textual sem configuração extra.

ETL e Export

OrientDB ETL Framework importa de CSV, JSON, RDBMS (via JDBC) e MongoDB. Útil para migração incremental de dados relacionais para o modelo de grafo sem scripts manuais.
