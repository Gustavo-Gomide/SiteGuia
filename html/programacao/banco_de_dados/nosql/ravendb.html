RavenDB

O NoSQL transacional: ACID completo, auto-indexação inteligente e LINQ nativo para .NET.

O RavenDB é um banco de dados de documentos JSON open-source, construído inteiramente em C# para a plataforma .NET. Ele se destaca por oferecer algo genuinamente raro no mundo NoSQL: transações ACID completas por padrão, em multi-documentos e multi-coleções, usando o motor de armazenamento proprietário Voron (B+Tree com MVCC).

1. O Que É o RavenDB?

Pense no RavenDB como um banco de documentos que não abre mão da garantia de consistência que os desenvolvedores .NET já esperam do SQL Server. A filosofia do criador Oren Eini (Ayende Rahien) é clara: "desenvolvedores não deveriam ter que escolher entre escalabilidade e corretude".

O modelo de dados é baseado em documentos JSON com coleções (equivalentes a tabelas sem schema fixo), ligações entre documentos via document includes, e relacionamentos opcionais por referência. Mas o diferencial real está em duas decisões arquiteturais: a engine Voron e o sistema de índices automáticos.

Engine Voron: armazenamento B+Tree proprietário implementado em C#. Usa Memory-Mapped Files para leituras ultra-rápidas, write-ahead log (WAL) para durabilidade, e MVCC (Multi-Version Concurrency Control) para isolamento. Cada transação vê um snapshot consistente do banco — leitores nunca bloqueiam escritores e vice-versa.

2. Como Funciona por Dentro

O fluxo de uma operação no RavenDB segue um modelo session-based: a aplicação abre uma Session, faz várias operações em memória (unit-of-work pattern), e chama SaveChanges() para commitar tudo em uma única transação ACID.

arquitetura-ravendb.txt
Session (Unit of Work):
  session.Store(pedido)     <- tracked in memory
  session.Store(produto)    <- tracked in memory
  produto.Estoque -= qtd    <- change detected
  session.SaveChanges()     <- ONE atomic transaction

Voron Storage Engine:
  WAL (Write-Ahead Log)    <- journal sequencial, durabilidade
  B+ Tree                  <- estrutura principal de dados
  MVCC Snapshots           <- leitores isolados sem locks
  Lucene Index             <- full-text search e índices estáticos

Auto-Index Engine:
  Query chega -> Query Optimizer
    índice existente? -> usa índice
    índice ausente?   -> cria Auto-Index em background
                      -> responde com IsStale: true na 1a execução
                      -> índice pronto em segundos

Document Includes

Em vez de fazer N+1 queries para carregar documentos relacionados, o RavenDB suporta Include — carrega o documento principal e seus relacionamentos em um único round-trip ao servidor. Exemplo: carregar um Pedido e automaticamente incluir o Produto e o Cliente referenciados, tudo em uma Request.

Change Tracking

A Session do RavenDB rastreia automaticamente todas as mudanças feitas nos objetos carregados. Você não precisa chamar um método "update" — basta modificar a propriedade do objeto e chamar SaveChanges(). O RavenDB detecta o diff e gera apenas o patch mínimo necessário.

Cluster com Rachis (Raft)

Em cluster, o RavenDB usa Rachis — implementação própria do algoritmo Raft para consensus distribuído. Garante que escritas são confirmadas por maioria dos nós antes de retornar success. Suporta topologias: replicação de banco, replicação de coleção por nó (pull replication) e clusters geo-distribuídos.

3. RQL e LINQ — Duas Formas de Consultar

O RavenDB oferece duas linguagens de query intercambiáveis: RQL (Raven Query Language), uma sintaxe SQL-like para queries diretas, e integração nativa com LINQ para desenvolvedores C# que preferem type-safety em compile time.

ravendb-queries.cs
using (var session = store.OpenSession())
{
    // 1. CRUD básico com ACID
    var pedido = new Pedido {
        ProdutoId = "produtos/42",
        Quantidade = 3,
        Status = "Pendente",
        Timestamp = DateTime.UtcNow
    };
    session.Store(pedido);

    // Transação multi-documento — ou salva tudo, ou nada
    var produto = session.Load<Produto>("produtos/42");
    produto.Estoque -= pedido.Quantidade;
    session.SaveChanges();  // COMMIT ACID

    // 2. LINQ — type-safe, sem strings de query
    var pedidosPendentes = session
        .Query<Pedido>()
        .Where(p => p.Status == "Pendente"
                   && p.Timestamp > DateTime.UtcNow.AddDays(-7))
        .OrderByDescending(p => p.Timestamp)
        .Take(50)
        .ToList();

    // 3. Include — carrega Pedido + Produto em um único round-trip
    var pedidoComProduto = session
        .Include<Pedido>(p => p.ProdutoId)
        .Load<Pedido>("pedidos/1-A");
    var produtoRef = session.Load<Produto>(pedidoComProduto.ProdutoId);

    // 4. RQL — SQL-like direto
    var resultados = session.Advanced
        .RawQuery<Pedido>(@"from Pedidos
                             where Status = 'Pendente'
                             and search(Notas, 'urgente')
                             order by Timestamp desc
                             limit 20")
        .ToList();
}

4. Sistema de Indices

O RavenDB monitora continuamente as queries que chegam. Quando detecta uma query sem cobertura de índice, o Auto-Index Engine cria automaticamente um índice otimizado em background — sem intervenção do DBA. Você pode também criar índices estáticos (Map/Reduce) para cenários avancados, com full-text search via Lucene integrado.

Auto-Indexes

Criados automaticamente quando uma query chega sem índice adequado. Começam como "stale" na primeira execução e ficam totalmente atualizados em segundos. O RavenDB também remove auto-indexes não utilizados para economizar recursos.

Static Map/Reduce Indexes

Indices definidos explicitamente pelo desenvolvedor. A parte Map seleciona e transforma documentos; a parte Reduce agrega resultados — similar ao paradigma MapReduce, mas executado em tempo real a medida que documentos mudam.

Full-Text Search (Lucene)

Todo índice pode ter campos com analisadores Lucene — tokenização, stemming, ngrams. A query RQL usa a função search(campo, 'termo') para full-text search nativo, sem precisar de uma instância separada de Elasticsearch.

Time-Series e Counters

RavenDB tem suporte nativo a Time-Series (dados temporais) e Counters — incremento/decremento atômico sem lock, ideal para contadores de visualizações e métricas de alta frequência.

Stale Indexes: Índices são atualizados de forma assíncrona. Por padrão, queries retornam resultados "stale" imediatamente. Para leitura consistente pós-escrita, use WaitForNonStaleResults() --- isso adiciona latencia.

5. Quando Usar e Quando Evitar

Use RavenDB quando...

  • Ecossistema .NET/C#: LINQ type-safe e drivers NuGet oficiais.
  • ACID multi-documento obrigatório: pedidos + estoque + log em uma transação.
  • Schema flexível com garantias: documentos JSON evoluem sem migrations.
  • Full-text search nativo: Lucene embutido, sem Elasticsearch separado.
  • Offline-first .NET/Xamarin: replicação automática com clientes móveis.

Evite RavenDB quando...

  • Ecossistema Python/Node: drivers fora do .NET têm experiência inferior.
  • JOINs complexos: dados altamente normalizados ficam melhor em SQL.
  • Escala extrema de writes: Cassandra ou DynamoDB têm throughput superior.
  • Grafo denso: Neo4j é muito mais expressivo para traversal de grafos.

6. Topicos Avancados

Data Subscriptions

Mecanismo de change feed tipado. Define-se um filtro e o RavenDB entrega novos documentos correspondentes com backpressure e checkpointing. Ideal para pipelines assíncronos.

ETL / CDC Integrado

ETL nativo para SQL Server, Elasticsearch ou outro RavenDB. Script de transformação configurado no Studio propaga mudanças em near-real-time sem Debezium ou Kafka Connect.

Optimistic Concurrency

Cada documento recebe um Change Vector (clock lógico distribuído). Com UseOptimisticConcurrency = true, escritas com Change Vector desatualizado são rejeitadas.

Smuggler — Backup

Backup incremental para S3, Azure Blob e armazenamento local. Exporta .ravendbdump portáveis. CLI (raven.smuggler) ou Studio.

7. Ecossistema e Ferramentas

RavenDB Studio

Interface web embutida na porta 8080. Navegue documentos, crie índices, execute RQL e monitore performance — zero dependência externa.

RavenDB Cloud

Serviço gerenciado em AWS, Azure e GCP. Clusters automáticos, SSL/TLS por padrão e tier gratuito para desenvolvimento.

Drivers Oficiais

.NET/C# (completo com LINQ), Java, Python, Go, Node.js e Ruby. Todos os drivers fora do .NET acessam via HTTP/REST.

ASP.NET / Blazor

NuGet RavenDB.Client registra store e session no DI do ASP.NET Core com escopo por request HTTP automático.

Smuggler CLI

Backup e migração por linha de comando. S3, Azure Blob e GCS. Integra em pipelines CI/CD para snapshots programáticos.

AI / Vector Search

Versões recentes: embeddings como campos e pesquisa por similaridade. Permite RAG (Retrieval Augmented Generation) sem infraestrutura adicional.
