MongoDB

O banco de documentos BSON mais popular do mundo — schema flexível, escala horizontal, stack MERN.

O MongoDB armazena dados como documentos BSON (Binary JSON) em coleções sem schema fixo. Criado em 2009 pela 10gen (hoje MongoDB, Inc.), é o banco NoSQL mais popular do mundo e base da stack MERN. Sua proposta central: eliminar a impedância entre o modelo de objetos da aplicação e o modelo relacional do banco — dados são armazenados da forma que a aplicação já os usa.

1. O Que É o MongoDB?

Em um banco SQL, um perfil de usuário com múltiplos endereços exige três tabelas e JOINs. No MongoDB, o mesmo dado é um único documento JSON com um array de endereços aninhados — lido em uma operação, sem transações distribuídas.

Kristina Chodorow, autora do livro-referência da O'Reilly sobre MongoDB, resume: "Schema-less nao significa sem schema — significa que o schema esta no codigo da aplicacao, nao no banco." Isso é poder e responsabilidade ao mesmo tempo.

BSON (Binary JSON): superconjunto binário do JSON com tipos extras — Date, ObjectId, Decimal128, Binary. O ObjectId de 12 bytes é a chave primária padrão: 4 bytes de timestamp + 5 bytes de ID de máquina + 3 bytes de contador — unicidade global sem coordenação central.

2. Como Funciona por Dentro

O MongoDB usa WiredTiger como engine de armazenamento padrão desde a v3.2. WiredTiger usa compressão Snappy por padrão, MVCC para controle de concorrência (snapshot isolation) e um WAL (Write-Ahead Log) para durabilidade. Leituras e escritas concorrentes ocorrem sem bloquear umas às outras.

arquitetura-mongodb.txt
Replica Set (Alta Disponibilidade):
  PRIMARY   <- recebe todas as escritas e leituras padrão
  SECONDARY <- réplica assíncrona do oplog do Primary
  SECONDARY <- réplica (pode servir leituras com readPreference)
  Arbiter   <- apenas participa de eleições (sem dados)

  Failover automático via protocolo Raft modificado (<10s)
  O oplog registra cada escrita em formato idempotente.

Sharding (Escala Horizontal):
  mongos (router)  <- recebe queries da aplicacao
    Shard 1        <- chunk range A-M
    Shard 2        <- chunk range N-Z
    Config Servers <- metadados dos chunks (3 nós Replica Set)

Aggregation Pipeline:
  $match -> $unwind -> $group -> $sort -> $project -> $limit
  Executado no servidor — sem mover dados para o cliente

WiredTiger + MVCC

Cada escritor cria uma nova versão do documento (MVCC). Leitores veem um snapshot consistente sem locks. Compressão Snappy reduz I/O em 60–80%. Journaling por WAL garante durabilidade mesmo após crash abrupto do processo.

oplog e Change Streams

O oplog é uma coleção capped que registra cada operação de escrita de forma idempotente. Secondaries consomem o oplog continuamente. Change Streams expõem o oplog como CDC (Change Data Capture) para a aplicação — ideal para event-driven architecture sem polling.

Aggregation Pipeline

Processa documentos em estágios encadeados no servidor: $match → $group → $sort → $project. Suporta joins com $lookup, grafos com $graphLookup e facets multi-nível.

3. MQL — MongoDB Query Language

O MongoDB usa uma API baseada em JSON (nao SQL). Operadores começam com $. O Aggregation Pipeline permite transformações complexas equivalentes a GROUP BY + HAVING + JOIN do SQL, mas com maior expressividade para dados aninhados.

mongodb-mql.js
// 1. CRUD básico
db.pedidos.insertOne({
  clienteId: ObjectId("64a1..."),
  itens: [
    { produto: "Teclado", qtd: 1, preco: 299.90 },
    { produto: "Mouse",   qtd: 2, preco: 89.90  }
  ],
  status: "pendente", criadoEm: new Date()
});

// 2. Query com operadores compostos
db.pedidos.find({
  status: { $in: ["pendente", "processando"] },
  "itens.preco": { $gte: 100 },
  criadoEm: { $gte: new Date("2025-01-01") }
});

// 3. Update atômico ($inc, $push, $set na mesma operação)
db.estoque.updateOne(
  { produtoId: "teclado-99", quantidade: { $gte: 1 } },
  {
    $inc:  { quantidade: -1 },
    $push: { historico: { data: new Date(), op: "saida" } },
    $set:  { ultimaAtualizacao: new Date() }
  }
);

// 4. Aggregation Pipeline
db.pedidos.aggregate([
  { $match:   { status: "entregue" } },
  { $unwind:  "$itens" },
  { $group: {
      _id:     "$itens.produto",
      total:   { $sum: { $multiply: ["$itens.qtd", "$itens.preco"] } },
      pedidos: { $sum: 1 }
  }},
  { $sort:   { total: -1 } },
  { $limit:  10 }
]);

4. Schema Design — Embedding vs Referencing

A decisão mais importante no MongoDB nao é qual índice criar — é como modelar os dados. A regra geral: dados que são acessados juntos devem ser armazenados juntos (embedding). Dados com relacionamentos muitos-para-muitos ou que crescem sem limite devem ser referenciados.

Embedding (Subdocumentos)

Armazena dados relacionados dentro do mesmo documento. Leitura em um único round-trip. Use quando: dados sempre acessados juntos (ex: endereços de usuário); relação 1:N com N limitado; sem necessidade de consultar o subdocumento isoladamente.

Referencing (DBRef)

Armazena um ObjectId que aponta para outro documento, resolvido com $lookup. Use quando: dados crescem sem limite; mesmo dado referenciado de muitos documentos; entidade tem existência independente.

Padrao Bucket

Para séries temporais: em vez de um documento por evento (milhões de docs minúsculos), agrupe eventos em "buckets" — um documento por hora com array de leituras. Reduz overhead de índice e melhora compressão. Base do suporte nativo a Time Series (MongoDB 5+).

Padrao Outlier

Para dados com distribuição desigual (ex: celebridade com 50M seguidores): um campo flag has_extras: true redireciona para documentos de overflow. O documento principal permanece dentro do limite de 16MB e o acesso ao caso comum é rápido.

5. Quando Usar e Quando Evitar

Use MongoDB quando...

  • Schema evolui rapidamente: MVPs, startups, catálogos de produtos com atributos variáveis por categoria.
  • Dados hierárquicos naturais: documentos com arrays aninhados eliminam JOINs e round-trips.
  • Stack JavaScript/TypeScript: mesmo formato JSON do frontend ao banco sem conversão de impedância.
  • Escala global: Sharding com Zone Sharding e Atlas Global Clusters com dados geo-particionados.
  • Search + dados operacionais: Atlas Search (Lucene integrado) sem precisar de Elasticsearch separado.

Evite MongoDB quando...

  • Dados altamente relacionais: múltiplos $lookup encadeados = JOINs lentos; use PostgreSQL.
  • Transações financeiras críticas: ACID v4.0 tem overhead real (~3-4x); prefira PostgreSQL para dados financeiros.
  • Relatórios OLAP complexos: Aggregation Pipeline nao substitui um data warehouse columnar.
  • Documento cresce sem controle: limite real de 16MB por documento — use padrão Outlier ou modele diferente.

6. Topicos Avancados

Change Streams (CDC)

Baseados no oplog, Change Streams permitem subscrever a mudanças em coleções, bancos ou clusters inteiros em tempo real. Equivalente ao CDC — útil para event-driven architecture, sincronização de caches e pipelines de dados sem polling. Suportam resume tokens para retomar após reconexão.

ACID Multi-Documento (v4.0+)

Transações ACID em Replica Sets (v4.2+ em clusters shardados). Usam snapshot isolation. O overhead é real (~3–4x mais lento que operações simples) — use somente quando consistência cruzada entre coleções é crítica e nao há como resolver com embedding.

Atlas Vector Search

Armazene embeddings como campos e pesquise por similaridade (ANN — Approximate Nearest Neighbor) com o algoritmo HNSW. Permite RAG (Retrieval Augmented Generation) com LLMs sem infraestrutura separada — busca semântica integrada aos dados operacionais.

Time Series Collections

Coleções especializadas para séries temporais (MongoDB 5.0+). Usam internamente o padrão Bucket com compressão específica para dados numéricos temporais — arquivamento automático, granularidade configurável (segundos/minutos/horas) e queries de janela temporal nativas.

Armadilha do Schema Livre: sem validacao no banco ($jsonSchema validator) ou na aplicação (Mongoose/Zod), documentos heterogêneos na mesma coleção tornam queries inconsistentes e refatorações perigosas. Defina validators desde o início do projeto.

7. Ecossistema e Ferramentas

MongoDB Atlas

DBaaS gerenciado em AWS, Azure e GCP. Atlas Search, Vector Search, Data Lake e Charts integrados. Backups automáticos, auto-scaling e tier gratuito M0 com 512MB.

MongoDB Compass

GUI oficial para visualizar documentos, criar índices, analisar performance com explain plans e construir aggregation pipelines visualmente — sem escrever código.

Mongoose (Node.js ODM)

ODM (Object Document Mapper) para Node.js. Define schemas, validators e métodos na aplicação. Essencial para projetos TypeScript com type-safety nos documentos MongoDB.

Drivers Oficiais

Node.js, Python (PyMongo), Java, C#/.NET, Go, Rust, PHP e Ruby. Todos suportam transações, Change Streams e o Aggregation Pipeline completo com type stubs.

Atlas Search (Lucene)

Full-text search com relevância, facets, autocomplete e sinônimos diretamente no MongoDB Atlas, alimentado por Apache Lucene — sem precisar manter Elasticsearch separado.

Realm / Device Sync

Banco de dados mobile/edge com sincronizacao automática para MongoDB Atlas. Suporta offline-first em iOS, Android e React Native com resolução automática de conflitos.
