Apache Cassandra

Banco de família de colunas masterless — do básico ao avançado

O Cassandra foi criado pelo Facebook em 2007 para resolver o problema da Inbox Search — bilhões de mensagens indexadas em tempo real. Em 2008 foi aberto ao público e hoje é usado pela Netflix, Apple, Instagram, Uber e Spotify. Sua proposta central: um banco que nunca cai, não tem nó primário nem ponto único de falha, e escreve na velocidade de um log sequencial em disco.

O que é o Apache Cassandra?

O Cassandra é um banco de dados NoSQL de família de colunas (column-family), distribuído, masterless e tolerante a falhas. Ele combina o modelo de dados do Google Bigtable (famílias de colunas esparsas) com a arquitetura de distribuição do Amazon Dynamo (consistent hashing, sem nó primário) — conforme descrito pelos criadores Avinash Lakshman e Prashant Malik no paper original de 2010.

Pense no Cassandra como um fichário infinito e distribuído: os dados são organizados em linhas identificadas por uma partition key, e cada linha pode ter centenas de colunas diferentes. Esse fichário vive simultaneamente em 3, 5 ou 7 gavetas (nós) ao mesmo tempo — se uma gaveta pegar fogo, as outras continuam servindo os dados sem interrupção.

Segundo Eben Hewitt (Cassandra: The Definitive Guide, 2022), "Cassandra was designed to handle write-heavy workloads at Internet scale, where no single node is a bottleneck and no single failure takes the system down."

Para iniciantes: o Cassandra não é um banco de propósito geral. Ele brilha em escritas massivas e contínuas (logs, IoT, séries temporais) e é péssimo para queries ad-hoc e JOINs complexos. Escolha Cassandra quando você sabe exatamente quais queries vai executar antes de modelar os dados.

Como Funciona Internamente

Arquitetura em anel e Consistent Hashing

Todos os nós do Cassandra formam um anel lógico. Quando um dado chega, o hash da partition key determina em qual posição do anel ele pertence — e os nós responsáveis por aquela faixa do anel ficam com ele. Não há nó primário nem coordenador fixo: qualquer nó pode receber qualquer requisição e faz o roteamento internamente.

Protocolo Gossip

A cada segundo, cada nó troca informações de estado com até 3 vizinhos aleatórios. Em poucos rounds, todo o cluster sabe que um nó nasceu, morreu ou ficou lento — sem nenhum servidor central de coordenação.

Replication Factor

Cada dado é replicado em N nós consecutivos no anel (tipicamente 3). RF=3 significa: o sistema tolera 2 falhas simultâneas sem perder nenhum dado e sem interrupção de serviço.

Consistência Configurável

Por query, você escolhe o nível de consistência: ONE (1 nó confirma, mais rápido), QUORUM (maioria confirma, mais seguro), ALL (todos confirmam, mais forte).

LSM Tree: por que escritas são O(1)

O segredo da performance de escrita do Cassandra é a Log-Structured Merge Tree (LSM Tree). Em vez de atualizar dados diretamente no disco (operação cara e lenta), o Cassandra segue este fluxo:

fluxo-escrita-cassandra.txt
── Passo 1: Commit Log (WAL) ────────────────────────────────
Escrita → append no Commit Log (arquivo sequencial em disco)
          → operação sequencial = ultra-rápida mesmo em HDD
          → garante durabilidade: se o nó reiniciar, replay do log

── Passo 2: Memtable (memória RAM) ──────────────────────────
Dado → carregado na Memtable (estrutura em memória ordenada)
     → leituras recentes resolvidas aqui — sem I/O de disco

── Passo 3: Flush para SSTable (disco) ──────────────────────
Memtable cheia → flush para SSTable (Sorted String Table)
           → SSTable é imutável: nunca é modificada após gravação
           → SSTable velhas são mescladas em background (compaction)

── Resultado: Escrita é sempre O(1) sequential I/O ──────────
Sem UPDATE in-place. Sem random I/O. Sem btree rebalancing.
Cada escrita é um append — o mais rápido possível em qualquer disco.

UPDATE e DELETE no Cassandra são, internamente, novas escritas. Um UPDATE insere um novo registro com timestamp maior. Um DELETE insere um marcador especial chamado tombstone. A reconciliação acontece durante a leitura e na compactação — nunca em tempo de escrita.

CQL — Cassandra Query Language

O CQL tem sintaxe propositalmente parecida com SQL para reduzir a curva de aprendizado. Mas o modelo de dados é completamente diferente: tudo gira em torno da partition key e das clustering keys. A partition key determina em qual nó o dado vive; as clustering keys determinam a ordem dentro da partição.

logs-acesso.cql
-- 1. Criar keyspace (equivalente ao database/schema no SQL)
CREATE KEYSPACE app
  WITH replication = {
    'class': 'NetworkTopologyStrategy',
    'datacenter1': 3   -- RF=3 no datacenter1
  };

-- 2. Criar tabela modelada para a query: "logs de um usuário, por data DESC"
--    PRIMARY KEY ((partition_key), clustering_key1, clustering_key2)
CREATE TABLE app.logs_acesso (
  usuario_id   UUID,        -- partition key: distribui dados pelo cluster
  data_hora    TIMESTAMP,   -- clustering key: ordena dentro da partição
  ip_origem    TEXT,
  acao         TEXT,
  detalhes     MAP<TEXT, TEXT>,  -- tipo collection nativo do CQL
  PRIMARY KEY  ((usuario_id), data_hora)
) WITH CLUSTERING ORDER BY (data_hora DESC)
  AND default_time_to_live = 2592000;  -- TTL: 30 dias automático

-- 3. Inserir — append-only, sempre O(1)
INSERT INTO app.logs_acesso (usuario_id, data_hora, ip_origem, acao)
VALUES (uuid(), toTimestamp(now()), '192.168.1.1', 'LOGIN')
USING TTL 86400;  -- TTL por linha: 24h

-- 4. Busca eficiente: WHERE na partition key é obrigatório
SELECT data_hora, ip_origem, acao
FROM app.logs_acesso
WHERE usuario_id = a1b2c3d4-e5f6-7890-abcd-ef1234567890
  AND data_hora > '2024-01-01 00:00:00'
LIMIT 100;

Tipos nativos do CQL

Tipo Descrição Exemplo
UUID / TIMEUUID ID único universal; TIMEUUID inclui timestamp uuid(), now()
LIST<T> Lista ordenada de valores do tipo T ['a', 'b', 'c']
SET<T> Conjunto sem duplicatas {'tag1', 'tag2'}
MAP<K,V> Dicionário chave-valor {'chave': 'valor'}
FROZEN<T> Valor imutável serializado (UDT, List, etc.) Usado em chaves primárias compostas

Modelagem por Query — Sem JOINs

Regra fundamental: o Cassandra não tem JOINs, subqueries nem transações multi-partição. Você não projeta tabelas pela normalização dos dados — projeta pela query que precisa responder. Duplicar dados é intencional e esperado. Uma tabela por padrão de acesso é a regra.

Suponha que você precisa de duas queries distintas: buscar logs por usuario_id E por ip_origem. Em SQL, um único SELECT com índice resolve. No Cassandra, você cria duas tabelas — uma para cada padrão de acesso:

modelagem-por-query.cql
-- Query 1: "logs de um usuário específico"
-- Partition key = usuario_id (concentra dados do usuário no mesmo nó)
CREATE TABLE logs_por_usuario (
  usuario_id UUID,
  data_hora  TIMESTAMP,
  ip_origem  TEXT,
  acao       TEXT,
  PRIMARY KEY ((usuario_id), data_hora)
) WITH CLUSTERING ORDER BY (data_hora DESC);

-- Query 2: "logs de um IP específico" → tabela separada, dados duplicados
-- Partition key = ip_origem (concentra logs do mesmo IP no mesmo nó)
CREATE TABLE logs_por_ip (
  ip_origem  TEXT,
  data_hora  TIMESTAMP,
  usuario_id UUID,
  acao       TEXT,
  PRIMARY KEY ((ip_origem), data_hora)
) WITH CLUSTERING ORDER BY (data_hora DESC);

-- A aplicação insere em AMBAS as tabelas a cada evento — duplicação intencional

Casos de Uso

Use Cassandra quando...

  • Séries temporais com escrita contínua (IoT, métricas, logs)
  • Zero downtime é obrigatório — tolerância a falhas extrema
  • Escrita é muito mais frequente do que leitura
  • Volume de dados cresce horizontalmente sem limite previsível
  • Queries são conhecidas antecipadamente e simples por partição
  • Replicação multi-datacenter ou multi-região é necessária

Evite Cassandra quando...

  • Queries ad-hoc e JOINs complexos são o uso principal
  • Transações ACID multi-linha são frequentes
  • O volume de dados cabe confortavelmente em um único servidor
  • Os padrões de acesso mudam frequentemente (requer reescrita de tabelas)
  • A equipe não tem experiência com modelagem orientada a query

Avançado: Tunable Consistency e Compaction

Tunable Consistency — o equilíbrio CAP na sua mão

O Cassandra permite calibrar o trade-off entre consistência e disponibilidade por query. A fórmula que garante consistência forte (leitura sempre atualizada) é: W + R > RF, onde W é o nível de escrita, R o de leitura e RF o Replication Factor.

Nível de Escrita Nível de Leitura Resultado (RF=3) Uso típico
ONE ONE Consistência eventual — possível ler dado desatualizado Logs, eventos, telemetria
QUORUM QUORUM Consistência forte — W+R=4 > RF=3 Perfis de usuário, catálogos
ALL ALL MAX consistência, MIN disponibilidade Operações críticas raras

Compaction Strategies

Como as SSTables são imutáveis e se acumulam no disco, o Cassandra precisa mesclá-las periodicamente em background (compaction). A estratégia de compaction impacta diretamente a performance de leitura e o uso de disco:

STCS (SizeTieredCompaction)

Padrão. Mescla SSTables de tamanho similar. Ótimo para workloads de escrita pesada. Problema: amplificação de espaço em disco durante a compaction (precisa de 2x o espaço temporariamente).

LCS (LeveledCompaction)

Organiza SSTables em níveis de tamanho fixo. Ótimo para leitura pesada — leituras examinam menos SSTables. Usa mais I/O de escrita durante compaction. Recomendado para dados que são lidos com frequência.

TWCS (TimeWindowCompaction)

Ideal para séries temporais com TTL. Agrupa SSTables por janela de tempo. Quando o TTL de uma janela expira, a SSTable inteira é descartada — sem tombstone bloat, sem compaction cara.

Lightweight Transactions (LWT)

Para casos que exigem operações condicionais ("inserir apenas se não existe"), o Cassandra oferece LWT baseado no protocolo Paxos:

lwt-cassandra.cql
-- INSERT IF NOT EXISTS: garante unicidade sem race condition
INSERT INTO usuarios (email, nome, criado_em)
VALUES ('ana@ex.com', 'Ana', toTimestamp(now()))
IF NOT EXISTS;  -- retorna [applied]=true se inseriu, false se já existia

-- UPDATE condicional: atualiza só se o valor atual corresponde
UPDATE usuarios SET saldo = 500
WHERE usuario_id = abc123
IF saldo = 450;  -- CAS: Compare-And-Swap

LWT tem custo alto: usa 4 roundtrips de rede (Paxos) em vez de 1. Use apenas quando a operação condicional for estritamente necessária. Para a grande maioria dos casos de uso do Cassandra (logs, events, time series), LWT é desnecessário.

Ecossistema Cassandra

DataStax Astra

Cassandra gerenciado como serviço (DBaaS). Elimina operação de cluster. Tier gratuito disponível. Compatível 100% com drivers Apache Cassandra padrão.

cqlsh

Shell interativo oficial para CQL. Conecta ao cluster, executa queries, importa/exporta dados. Ferramenta básica para qualquer desenvolvedor Cassandra.

Drivers Oficiais

Drivers nativos para Java (DataStax Java Driver), Python (cassandra-driver), Node.js (cassandra-driver) e Go. Todos com suporte a async, connection pooling e retry policies configuráveis.

Spring Data Cassandra

Integração Spring Boot com Cassandra. Mapeamento de entidades Java para tabelas CQL via annotations (@Table, @PrimaryKeyColumn), repositories e queries derivadas de nome de método.

Cassandra + Kafka

Padrão comum em streaming: Kafka ingere eventos em tempo real, Cassandra os persiste para consultas históricas. O Kafka Connect possui conector oficial para Cassandra (Sink Connector).

nodetool

Ferramenta de administração de cluster: verificar status dos nós (nodetool status), disparar compaction (nodetool compact), reparar réplicas (nodetool repair).
