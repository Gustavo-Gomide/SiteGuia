ArangoDB

Multi-Model nativo: documentos, grafos e chave-valor em uma única engine — uma linguagem para tudo.

O ArangoDB é pioneiro no conceito de banco multi-modelo nativo. Diferente de soluções que adaptam uma engine para simular outros modelos, o ArangoDB foi construído do zero para suportar Chave-Valor, Documentos e Grafos simultaneamente, consultados por uma linguagem unificada: AQL (ArangoDB Query Language). Tudo isso sobre uma única engine de armazenamento: RocksDB.

1. O Que É o ArangoDB?

Imagine o ArangoDB como um canivete suíço de bancos de dados: uma única instalação resolve três problemas que normalmente exigiriam três bancos diferentes — um document store para dados flexíveis, um graph database para relacionamentos complexos, e um key-value store para acesso O(1) por identificador.

A sacada arquitetural é que os três modelos não são simulações uns dos outros — eles coexistem no mesmo motor (RocksDB) e no mesmo namespace. Um documento pode ser ao mesmo tempo o vértice de um grafo. Uma query AQL pode cruzar dados de uma coleção de documentos com traversais de grafo em um único statement.

Engine única — RocksDB: desde a versão 3.4, o ArangoDB usa exclusivamente o RocksDB (a mesma engine do Cassandra, MySQL 8 e TiKV) como storage. Uma LSM-tree otimizada para SSDs que garante writes rápidos e compactação eficiente em disco — e elimina a duplicação de dados que existia com a engine MMFiles anterior.

2. Como Funciona por Dentro

O fluxo de uma query AQL passa por um pipeline de otimização antes de executar. O ArangoDB usa um query planner que escolhe automaticamente entre acesso por índice, traversal de grafo nativo ou full collection scan dependendo do shape da query.

arquitetura-arangodb.txt
AQL Query Parser
      │
      ▼
AST (Abstract Syntax Tree)
      │
      ▼
Query Optimizer   ← regras de otimização (index scan, sort elimination, etc.)
      │
      ├─► Index Scan        (hash, skiplist, persistent, geo, fulltext)
      ├─► Traversal Engine  (DFS/BFS nativo sobre Edge Collections)
      └─► ArangoSearch      (inverted index com BM25 e phrase search)
      │
      ▼
RocksDB (LSM-tree, WAL, block cache, compression)
      │
      ▼
Resultado → Cliente (HTTP/VelocyStream)

Coleções e Documentos

Toda coleção no ArangoDB armazena documentos JSON com três campos automáticos: _id (coleção/chave), _key (identificador único na coleção) e _rev (revisão para MVCC). Não há schema — cada documento pode ter campos diferentes.

Edge Collections

Arestas são documentos especiais com _from e _to apontando para _id de vértices em qualquer coleção. Como são documentos normais, podem ter atributos arbitrários (peso, timestamp, tipo de relacionamento) — grafos com properties ricas sem nenhuma configuração extra.

Named Graphs e Smart Graphs

Named Graphs agrupam vertex collections + edge collections com transações ACID cobrindo todos. Smart Graphs (Enterprise) particionam o grafo garantindo que vértices relacionados fiquem no mesmo shard — eliminando comunicação cross-shard em traversals e acelerando queries em grafos distribuídos.

3. AQL — Uma Linguagem para Três Modelos

AQL é uma linguagem procedural e declarativa ao mesmo tempo — inspirada em SQL mas com primitivos nativos para grafos e documentos. Com as palavras-chave FOR, FILTER, SORT, COLLECT e RETURN você expressa qualquer query, cruzando modelos em um único statement.

arangodb-aql.js
// 1. Consulta de documentos — usuários adultos com índice persistente em "idade"
FOR u IN usuarios
  FILTER u.idade >= 18 AND u.cidade == "São Paulo"
  SORT u.nome ASC
  LIMIT 20
  RETURN { nome: u.nome, email: u.email }

// 2. Traversal de grafo — amigos de amigos (profundidade 1 a 2)
FOR v, e, p IN 1..2 OUTBOUND "usuarios/ana_silva"
  GRAPH "rede_social"
  FILTER v.ativo == true
  RETURN { nome: v.nome, profundidade: LENGTH(p.edges) }

// 3. Query multi-modelo: documentos + grafo em um só statement
// Encontrar todos os pedidos de amigos de um usuário
FOR amigo IN 1..1 OUTBOUND "usuarios/ana_silva" GRAPH "rede_social"
  FOR pedido IN pedidos
    FILTER pedido.usuario_id == amigo._id
      AND pedido.status == "entregue"
    RETURN { amigo: amigo.nome, produto: pedido.produto }

// 4. Upsert atômico — cria ou atualiza documento
UPSERT { email: "ana@example.com" }
  INSERT { email: "ana@example.com", nome: "Ana", criado: DATE_NOW() }
  UPDATE { nome: "Ana Silva", atualizado: DATE_NOW() }
IN usuarios

// 5. Acesso KV direto por _key — O(1)
RETURN DOCUMENT("usuarios/ana_silva")

4. Foxx Microservices

Lógica de Negócio Dentro do Banco

O Foxx é um framework que permite escrever microserviços JavaScript (V8) dentro do próprio banco de dados. Você cria endpoints REST que executam diretamente no servidor ArangoDB — zero latência de rede entre aplicação e banco para operações complexas.

foxx-routes.js — executado dentro do ArangoDB
const { db } = require('@arangodb');
const router = require('@arangodb/foxx/router')();

// GET /api/usuario/:id com traversal de grafo embutido
router.get('/usuario/:id/amigos', (req, res) => {
  const amigos = db._query(
    `FOR v IN 1..1 OUTBOUND @start GRAPH "rede_social" RETURN v.nome`,
    { start: `usuarios/${req.pathParams.id}` }
  ).toArray();
  res.json({ amigos });
});

module.context.use('/api', router);
// Acesso: GET http://arango:8529/_db/mydb/api/usuario/ana_silva/amigos

Foxx é ideal para lógica repetitiva que exige múltiplas operações no banco — você concentra tudo em um único round-trip HTTP em vez de N chamadas separadas da aplicação.

5. Quando Usar (e Quando Não Usar)

Use ArangoDB quando...

  • Multi-modelo genuíno: seu sistema precisa de documentos E grafos E buscas — ArangoDB elimina a necessidade de gerenciar 3 bancos separados.
  • Redes sociais e recomendações: traversals BFS/DFS em grafos com atributos ricos nas arestas sem custo de JOIN.
  • Fraud detection e supply chain: análise de conectividade entre entidades com queries que misturam filtros de documento e paths de grafo.
  • APIs flexíveis: schema-less com Foxx permite evoluir a API sem migrações pesadas.
  • Datasets médios a grandes: funciona bem de GBs a dezenas de TBs com cluster ArangoDB.

Evite ArangoDB quando...

  • Grafos massivos em escala web: para grafos com bilhões de vértices em produção crítica, Neo4j ou TigerGraph têm engines mais especializadas.
  • Alta escrita de time-series: Cassandra ou HBase são mais eficientes para ingestão de centenas de milhares de eventos/seg.
  • ACID multi-shard pesado: transações distribuídas funcionam mas têm overhead maior que sistemas OLTP dedicados.
  • Equipe MongoDB experiente: se o problema é puramente documental, MongoDB tem ecossistema e tooling mais maduro.

6. Tópicos Avançados

ArangoSearch — Busca Full-Text Nativa

Views do ArangoSearch criam índices invertidos sobre coleções de documentos. Suportam BM25, TF-IDF, phrase matching e stemming multi-idioma. Queries AQL usam SEARCH para combinar filtros estruturados com ranking textual — sem Elasticsearch separado.

Pregel — Grafos Distribuídos

Implementação do modelo Pregel do Google para computação distribuída em grafos. Algoritmos prontos: PageRank, Betweenness Centrality, Connected Components, Louvain para detecção de comunidades. Executa em paralelo sobre todos os shards do cluster sem mover dados para um nó coordenador.

Smart Graphs (Enterprise)

Em um cluster shardado, uma traversal normal pode exigir comunicação entre múltiplos shards para cada hop do grafo. Smart Graphs garantem que vértices com o mesmo smart attribute (ex: comunidade, região) fiquem no mesmo shard — traversals locais sem network roundtrips, ordens de magnitude mais rápido.

Transações ACID Multi-Coleção

ArangoDB fornece transações ACID que cruzam múltiplas coleções via API de Stream Transactions (JavaScript ou REST). Permitem operações atômicas em documentos + arestas — ex: criar usuário e suas arestas de relacionamento em uma única transação que faz rollback se qualquer parte falhar.

Sharding de Grafos — Armadilha Comum: Por padrão, documentos são shardados por _key de forma aleatória. Em um grafo distribuído, isso significa que quase todo traversal faz chamadas cross-shard — latência multiplicada por número de hops. Use Smart Graphs em produção ou mantenha o grafo em um único servidor até que o volume justifique a complexidade de sharding.

7. Ecossistema

ArangoDB Cloud (ArangoGraph)

DBaaS gerenciado disponível em AWS, GCP e Azure. Zero-ops: backups automáticos, scaling vertical/horizontal, monitoramento e patches de segurança incluídos.

Web Interface

Console web built-in em localhost:8529. Permite executar queries AQL, visualizar grafos interativamente (D3.js), gerenciar índices e monitorar estatísticas do cluster.

Drivers Oficiais

Clientes nativos para JavaScript/Node.js (arangojs), Python (python-arango), Java (arangodb-java-driver), Go e PHP. Suporte a async/await e connection pooling.

ArangoSearch BM25

Motor de busca full-text integrado com ranking BM25/TF-IDF, Ngram tokenizer, stemming (Snowball) e similarity search com vetores (ArangoDB 3.10+). Substitui Elasticsearch para casos de uso de busca acoplados ao grafo.

arangodump / arangorestore

Ferramentas de linha de comando para backup e restore de bancos completos ou coleções individuais. Formato JSON + envelopes de metadados — portável entre versões e clusters.

Spring Data ArangoDB

Integração com o ecossistema Spring. Repository pattern, mapeamento de entidades Java para documentos, e suporte a queries AQL anotadas com @Query — familiar para desenvolvedores JPA.
