Apache HBase

Banco de dados de família de colunas sobre HDFS — bilhões de linhas, petabytes de dados.

O HBase é um banco de dados NoSQL distribuído de família de colunas, modelado sobre o Google Bigtable (Chang et al., 2006). Foi projetado para acesso aleatório em tempo real a tabelas gigantescas — bilhões de linhas por milhões de colunas — rodando sobre clusters Hadoop com hardware comum de prateleira.

1. O Que É o HBase?

Pense no HBase como uma planilha infinita e distribuída: linhas identificadas por uma Row Key única, colunas agrupadas em Column Families e cada célula versionada no tempo. Mas diferente de uma planilha, uma tabela HBase pode ter 10 bilhões de linhas e ser consultada em milissegundos — porque os dados de cada Row Key estão sempre colocalizados no mesmo servidor (Region).

Origem: Chang et al. publicaram "Bigtable: A Distributed Storage System for Structured Data" no OSDI 2006. O HBase foi criado em 2007 para replicar esse design sobre o ecossistema open-source Hadoop/HDFS.

O modelo de dados é intencionalemnte simples: quatro dimensões identificam cada valor — (Row Key, Column Family, Column Qualifier, Timestamp) → valor em bytes. Não há tipos, não há schema para colunas — qualquer número de qualificadores pode ser adicionado livremente dentro de uma Column Family.

2. Como Funciona por Dentro

O fluxo de escrita do HBase segue a arquitetura LSM (Log-Structured Merge-Tree), a mesma usada pelo Cassandra e pelo LevelDB. Entender esse fluxo é essencial para projetar corretamente o compactação e a performance de leitura.

pipeline de escrita e leitura — HBase
ESCRITA (Write Path):
Client
  └─► WAL (Write-Ahead Log no HDFS)     ← durabilidade: se o RegionServer cair,
  └─► MemStore (RAM, por Region)          o WAL garante que nenhum dado se perde
        │
        ├─ [quando MemStore atinge ~128 MB]
        ▼
      HFile (imutável, no HDFS)           ← flush: MemStore vira HFile no HDFS
        │
        ├─ [N HFiles acumulados]
        ▼
      Compaction                          ← Minor: mescla poucos HFiles
                                             Major: mescla TODOS + apaga tombstones

LEITURA (Read Path) — "merge view":
Client →  BlockCache (RAM, LRU)        ← L1: dados quentes em memória
          └─ miss ─► MemStore            ← L2: dados ainda não flushed
          └─ miss ─► HFiles (HDFS)        ← L3: dados persistidos (com BloomFilter)
                      └── merge → versão mais recente retornada

RegionServer e Regions

A tabela é dividida em Regions — faixas contíguas de Row Keys. Cada Region é atendida por um RegionServer. Quando uma Region cresce muito (~10 GB), o HBase a divide automaticamente em duas (Region Split). O HMaster coordena qual servidor cuida de qual Region.

WAL e Durabilidade

Toda escrita vai primeiro ao Write-Ahead Log (arquivo sequencial no HDFS) antes de entrar no MemStore em RAM. Se o RegionServer cair, o WAL é reproduzido para reconstruir o estado perdido — zero perda de dados, mesmo com falha de hardware.

BloomFilter e BlockCache

Cada HFile carrega um BloomFilter que responde em O(1): "esta Row Key existe neste HFile?". Isso evita leituras desnecessárias de disco. O BlockCache (padrão: BucketCache off-heap) mantém blocos de dados quentes em memória, evitando repetir acesso ao HDFS.

3. Modelagem de Row Keys — A Decisão Arquitetural

No HBase, a Row Key é o único índice real. Toda consulta eficiente passa por ela. Projetar mal a Row Key é o erro número um de quem começa com HBase — e pode significar a diferença entre 5 ms e 30 segundos de latência.

Hot Region (anti-pattern crítico): Se sua Row Key começa com um valor sequencial (timestamp crescente, ID auto-increment), todas as escritas irão para a mesma Region — o servidor fica sobrecarregado enquanto os outros ficam ociosos. Isso se chama hot spotting.

Anti-Pattern: Timestamp como Prefixo

2024-01-15T10:30:00:sensor001 — escritas sempre no mesmo "fim" da tabela. A Region que armazena os keys mais recentes fica quente e todas as outras ficam frias. Nunca use timestamp crescente como prefixo de Row Key.

Salting

Adicione um prefixo aleatório (salt) de 1 byte calculado como hash(key) % N. Distribui escritas uniformemente por N Regions. Exemplo: 3:sensor001:2024-01-15T10:30:00. O trade-off: leitura de um range completo exige N Scans paralelos.

Reverse Timestamp

Para séries temporais onde você sempre quer os N registros mais recentes: use Long.MAX_VALUE - timestamp como prefixo. O dado mais novo vai para o início da tabela. Um Scan retorna os mais recentes primeiro, sem inversão posterior.

Composite Key

Combine entidades para colocar dados relacionados no mesmo Range — CLIENTE_101:PEDIDO:2024-01-15 garante que todos os pedidos de um cliente ficam em Regions próximas. Um único Scan retorna todo o histórico de pedidos de um cliente com IO mínimo.

hbase-shell.sh — operações com Row Key bem projetada
# Criar tabela: Column Families são definidas na criação
create 'eventos',
  {NAME => 'meta', VERSIONS => 1, COMPRESSION => 'SNAPPY'},
  {NAME => 'payload', VERSIONS => 3, BLOOMFILTER => 'ROW'}

# Inserir — reverse timestamp como prefixo evita hotspot
put 'eventos', 'sensor_42:9223370451775808:2024-01-15', 'payload:temp', '23.5'
put 'eventos', 'sensor_42:9223370451775808:2024-01-15', 'meta:unidade', 'celsius'

# Get por Row Key exata — O(log N)
get 'eventos', 'sensor_42:9223370451775808:2024-01-15'

# Scan por range (todos os eventos do sensor_42 mais recentes primeiro)
scan 'eventos', {
  STARTROW => 'sensor_42:',
  ENDROW   => 'sensor_42:~',
  LIMIT    => 100,
  FILTER   => "SingleColumnValueFilter('payload','temp',>,'binary:20.0')"
}

# Deletar — insere tombstone (apagado fisicamente na Major Compaction)
delete 'eventos', 'sensor_42:9223370451775808:2024-01-15', 'payload:temp'

4. Quando Usar (e Quando Não Usar)

Use HBase quando...

  • Volume massivo: bilhões de linhas com acesso aleatório por Row Key em milissegundos.
  • Séries temporais IoT: ingestão de milhares de eventos/segundo (sensores, telemetria, logs de máquina).
  • Clickstream / comportamento: histórico completo de ações de usuários com prefixo USER_ID:timestamp.
  • Integração Hadoop: quando o dado já está no HDFS e você precisa de acesso online (Spark + HBase, Hive + HBase).
  • Web crawling / índice invertido: URLs como Row Key, conteúdo e metadados em Column Families — o caso de uso original do Bigtable.
  • Dados esparsos: colunas que existem apenas em algumas linhas sem desperdício de storage (NULL não ocupa espaço).

Evite HBase quando...

  • Consultas ad-hoc complexas: "WHERE cidade = 'SP' AND idade > 30" exige full Scan — use PostgreSQL ou Elasticsearch.
  • Dados pequenos: overhead de um cluster HBase/HDFS não justifica tabelas com menos de 100 milhões de linhas.
  • Baixa latência crítica (< 1 ms): Redis ou Memcached são a escolha certa para cache ultra-rápido.
  • Equipe sem expertise Hadoop: operação de cluster HBase requer conhecimento de tuning de JVM, HDFS e ZooKeeper.
  • ACID completo: HBase garante atomicidade apenas dentro de uma linha (single-row), não multi-row.

5. Tópicos Avançados

Apache Phoenix — SQL Sobre HBase

Camada SQL completa que compila queries para operações nativas do HBase. Suporta SELECT, JOIN, índices secundários, e se integra ao JDBC/ODBC. Latência tipicamente abaixo de 10 ms para queries simples. Usado extensivamente no operacional do Salesforce, eBay e Bloomberg.

Coprocessors

Código Java executado dentro do RegionServer, colocado junto ao dado — análogo a stored procedures. Observer Coprocessors reagem a eventos (pre/post Get, pre/post Put). Endpoint Coprocessors implementam agregações distribuídas (SUM, COUNT) sem mover dados para o cliente.

Compaction e Tombstones

Deleções no HBase são marcas de lápide (tombstones) — o dado não é removido imediatamente. Durante a Major Compaction, todos os HFiles de uma Region são mesclados e tombstones + versões expiradas são fisicamente apagados. Programe Major Compactions em horários de baixo tráfego para evitar impacto de I/O em produção.

Replication e High Availability

HBase suporta replicação assíncrona entre clusters para DR (Disaster Recovery). O HMaster ativo/standby (via ZooKeeper) garante que a coordenação de Regions nunca fique sem responsável. ZooKeeper armazena o catálogo de Regions (hbase:meta) e a eleição de líderes.

Armadilha de produção — Region Server GC Pause: RegionServers rodam na JVM. Um GC pause de > 15-30 segundos pode fazer o ZooKeeper considerar o servidor morto e iniciar uma reassignação de Regions — causando downtime temporário. Use G1GC ou ZGC com heap de 8–16 GB máximo, e prefira mover dados quentes para BlockCache off-heap (BucketCache) para reduzir pressão de GC.

6. Ecossistema

Apache Phoenix

SQL sobre HBase com índices secundários, JOINs e JDBC. Abstrai completamente o modelo de Row Key — o desenvolvedor escreve SQL, o Phoenix compila para Scans e Get eficientes.

Apache Spark + HBase

Conector nativo HBase-Spark lê Regions em paralelo como RDD/DataFrame. Ideal para ETL massivo: processar petabytes no Spark e persistir resultados de volta ao HBase.

OpenTSDB

Banco de séries temporais construído sobre HBase. Row Key engineered para distribuição eficiente: métrica:timestamp:tags. Processa bilhões de pontos de dados por dia.

Cloudera CDP / HDP

Distribuições empresariais Hadoop com HBase gerenciado, monitoramento, segurança Kerberos, e integração com HDFS, YARN, Kafka e Spark em um único cluster.

HBase REST Gateway

Servidor REST que expõe tabelas HBase via HTTP/JSON. Permite acesso de qualquer linguagem sem cliente HBase nativo — útil para integração com microserviços não-JVM.

Apache Hive + HBase

O Hive Storage Handler mapeia tabelas HBase como tabelas Hive. Permite queries HiveQL analíticas sobre dados operacionais do HBase sem mover dados — Lambda Architecture simplificada.
