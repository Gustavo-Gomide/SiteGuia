Apache CouchDB

O banco da Web Sincronizada — JSON, HTTP nativo e replicação offline-first

O Apache CouchDB é um banco de dados NoSQL de documentos JSON pioneiro, lançado em 2005. Diferente do MongoDB, foi construído do zero para o mundo da conectividade intermitente: sincronizar dados entre sede e filiais com internet instável, ou rodar completamente offline no celular e sincronizar quando a rede voltar. Sua arquitetura MVCC (Multi-Version Concurrency Control) garante que duas escritas simultâneas nunca corrompam o banco.

O que é o CouchDB?

O CouchDB é um banco de documentos JSON com três características que o tornam único: API HTTP nativa (você usa curl para tudo), MVCC sem locks (leituras nunca bloqueiam escritas) e replicação Master-Master entre qualquer número de instâncias.

Pense no CouchDB como um banco de dados com a filosofia da Web: tudo é um recurso com uma URL, tudo é acessado com HTTP CRUD (GET/PUT/POST/DELETE), e o banco pode ser acessado diretamente pelo navegador sem intermediário. Não existe protocolo proprietário, não existe driver obrigatório — se você conhece HTTP, você já sabe como falar com o CouchDB.

Segundo Anderson, Lenhardt e Slater (CouchDB: The Definitive Guide, 2010), "CouchDB is built for the web. It stores your data as JSON documents, which can be queried and indexed using JavaScript. It replicates without central coordination — just peer to peer."

Para iniciantes: a grande ideia do CouchDB é que o banco não tem um "master" obrigatório. Duas pessoas podem editar o mesmo documento offline em dispositivos diferentes, e o CouchDB resolve o conflito automaticamente ao sincronizar — guardando as duas versões e permitindo resolução manual. É exatamente como o Git resolve conflitos de merge.

Como Funciona: MVCC e Revision Tree

O mecanismo central do CouchDB é o MVCC (Multi-Version Concurrency Control). Cada vez que um documento é modificado, o CouchDB não sobrescreve — ele cria uma nova revisão, identificada pelo campo _rev. Cada documento tem uma árvore de revisões que o CouchDB mantém internamente.

mvcc-arvore.txt
── Árbol de revisões de um documento ─────────────────────────

_rev: "1-abc"  ← versão original
       │
       ├── "2-xyz"  ← edição normal (atualização sequencial)
       │         │
       │         └── "3-def"  ← versão atual ✓ (Winner)
       │
       └── "2-qrs"  ← edição feita offline em outro nó
                         ↑ CONFLITO detectado após replicação
                         ↑ CouchDB guarda as duas, app resolve

── Resolução: app escolhe o winner, guarda no _rev atual ─────
── Revisões antigas são compactadas periodicamente ───────────

Para atualizar um documento, você deve passar o _rev atual. Se outra escrita aconteceu antes, o CouchDB retorna 409 Conflict — o banco nunca perde dados silenciosamente.

A API HTTP — O Banco é RESTful

Toda comunicação com o CouchDB é via HTTP puro. O painel web Fauxton (/_utils) permite gerenciar tudo visualmente. Não existe driver obrigatório — qualquer linguagem que faça HTTP funciona.

couchdb-api.http
// ─── Criar banco ───────────────────────────────────────────
PUT  http://localhost:5984/filmes
// → {"ok": true}

// ─── Criar documento com ID explícito ─────────────────────
PUT  http://localhost:5984/filmes/matrix
{ "titulo": "Matrix", "ano": 1999, "genero": "Sci-Fi" }
// → {"ok": true, "id": "matrix", "rev": "1-a2b3c4d5..."}

// ─── Ler documento ─────────────────────────────────────────
GET  http://localhost:5984/filmes/matrix
// → {"_id": "matrix", "_rev": "1-a2b3c4d5...", "titulo": "Matrix", ...}

// ─── Atualizar (OBRIGATÓRIO passar _rev atual) ─────────────
PUT  http://localhost:5984/filmes/matrix
{ "_rev": "1-a2b3c4d5...", "titulo": "Matrix", "nota": 9.5 }
// → {"rev": "2-b3c4d5e6..."}
// Sem _rev → 409 Conflict (proteção contra perda de dados)

// ─── Deletar (soft delete com _deleted: true) ──────────────
DELETE http://localhost:5984/filmes/matrix?rev=2-b3c4d5e6...

// ─── Bulk update (múltiplos docs em 1 request) ─────────────
POST http://localhost:5984/filmes/_bulk_docs
{ "docs": [
    { "titulo": "Inception", "ano": 2010 },
    { "titulo": "Interstellar", "ano": 2014 }
]}

Replicação Master-Master e PouchDB

A killer feature do CouchDB é o Protocolo de Replicação CouchDB: qualquer nó CouchDB pode sincronizar com qualquer outro nó, em qualquer direção, sem coordenação central. O protocolo é determinístico — replay múltiplas vezes sem duplicatas.

PouchDB no Cliente

Implementação JavaScript do protocolo CouchDB que roda no navegador (IndexedDB) ou Node.js/React Native. O app salva localmente primeiro, sincroniza depois. Funciona 100% offline — usuário nem percebe a desconexão.

CouchDB no Servidor

Recebe as sincronizações de todos os clientes PouchDB, resolve conflitos e distribui de volta. Pode replicar para outros servidores CouchDB em outras regiões — formando um cluster multi-master global.

Resolução de Conflitos

CouchDB detecta automaticamente conflitos após replicação e elege um "winner" deterministicamente (por _rev hash). A outra versão fica acessível para resolução manual pela aplicação — sem perda de dados.

pouchdb-sync.js
import PouchDB from 'pouchdb';

const local  = new PouchDB('pedidos_local');
const remote = new PouchDB('https://couch.empresa.com/pedidos');

// ─── Sincronização bidirecional contínua ─────────────────────
const sync = local.sync(remote, {
  live:  true,  // mantém conexão sempre ativa
  retry: true   // tenta reconectar automaticamente se cair
})
.on('change',   info => console.log('Sync:', info.change.docs.length, 'docs'))
.on('paused',   ()   => console.log('Sem conexão — modo offline'))
.on('active',   ()   => console.log('Reconectado — sincronizando...'))
.on('conflict', err  => resolverConflito(err))
.on('error',    err  => console.error(err));

// ─── Escrita local (funciona offline) ────────────────────────
await local.put({
  _id: `pedido_${Date.now()}`,
  produto: "Botas Trail",
  qtd: 2,
  status: "pendente"
});
// Se offline: salvo localmente. Ao reconectar: sincroniza automaticamente.

Consultas: Views MapReduce e Mango

O CouchDB tem dois sistemas de consulta. O original, Views MapReduce, usa JavaScript para definir índices permanentes. O moderno, Mango Query (inspirado no MongoDB), permite queries declarativas em JSON.

couchdb-consultas.json
// ─── MapReduce View (Design Document) ─────────────────────
{
  "_id": "_design/filmes",
  "views": {
    "por_genero": {
      "map": "function(doc) {
        if (doc.genero && doc.titulo) {
          emit(doc.genero, { titulo: doc.titulo, ano: doc.ano });
        }
      }",
      "reduce": "_count"  // conta filmes por gênero
    }
  }
}

// Consultar: GET /filmes/_design/filmes/_view/por_genero?key="Sci-Fi"

// ─── Mango Query (mais simples, similar ao MongoDB) ────────
POST /filmes/_find
{
  "selector": {
    "genero": "Sci-Fi",
    "ano":    { "$gte": 1990 }
  },
  "fields":  ["titulo", "ano"],
  "sort":    [{ "ano": "desc" }],
  "limit":   20
}

Full document scan: queries Mango sem índice (_index) fazem scan em todos os documentos — inaceitável em produção com volume alto. Sempre crie um índice Mango antes de executar uma query: POST /_index {"index": {"fields": ["genero", "ano"]}}

Casos de Uso

Use CouchDB quando...

  • App mobile/desktop que precisa funcionar offline
  • Sincronização de dados entre unidades geográficas com internet instável
  • Sistemas onde qualquer nó deve receber escritas (sem líder único)
  • Formulários de campo preenchidos offline (inspeções, survey, coleta)
  • Banco de documentos simples acessado via HTTP sem driver

Evite CouchDB quando...

  • Queries ad-hoc complexas e frequentes (use MongoDB)
  • Transações ACID entre múltiplos documentos
  • Escala horizontal massiva com alto write throughput (use Cassandra)
  • Ecossistema cloud centralizado já na AWS/GCP (use DynamoDB/Firestore)

Avançado: Clustering com CouchDB 3.x

O CouchDB 3.x introduziu o modo cluster nativo (antes era pelo Cloudant/BigCouch). No modo cluster, os dados são distribuídos usando um anel de shards consistente — similar ao Cassandra — com replicação configurável.

setup-cluster.http
// ─── Criar banco com sharding e replicação explícitos ─────
PUT /pedidos?q=8&n=3
//   q=8 → 8 shards (partições) distribuídos pelo cluster
//   n=3 → cada shard tem 3 réplicas em nós diferentes

// ─── Monitorar status do cluster ──────────────────────────
GET /_cluster_setup
GET /_membership     // lista todos os nós do cluster
GET /_scheduler/jobs  // jobs de replicação em andamento

// ─── Iniciar replicação contínua entre dois servidores ────
POST /_replicator
{
  "source":     "http://couch1.empresa.com/pedidos",
  "target":     "http://couch2.empresa.com/pedidos",
  "continuous": true,
  "create_target": true
}

Ecossistema CouchDB

PouchDB

Implementação JavaScript do CouchDB para browser e Node.js. Usa IndexedDB ou LevelDB como storage local. Sincroniza com qualquer servidor compatível com o protocolo CouchDB.

Fauxton (UI)

Interface web administrativa em /_utils. Permite criar bancos, visualizar documentos, criar views, gerenciar usuários, replicações e monitorar o cluster — sem linha de comando.

IBM Cloudant

Serviço gerenciado baseado em CouchDB pela IBM Cloud. Oferece escalabilidade automática, geo-replication multi-região e plano free. API 100% compatível com CouchDB.

nano (Driver Node.js)

Biblioteca Node.js oficial para CouchDB. API assíncrona com Promises. Simples porque mapeia diretamente os endpoints HTTP — sem abstração mágica nem ORM.
