Firebase

Backend as a Service do Google — banco, autenticação e hospedagem sem gerenciar servidor

O Firebase é uma plataforma BaaS (Backend as a Service) do Google, presente em mais de 2,5 milhões de aplicativos. Ele oferece autenticação (Google, GitHub, Apple, e-mail), dois bancos NoSQL distintos, Storage, Hosting, Cloud Functions e Analytics — tudo integrado sob uma única conta. Um desenvolvedor frontend pode lançar um produto completo em produção sem jamais gerenciar um servidor.

O que é Backend as a Service?

O modelo BaaS inverte o desenvolvimento tradicional. Em vez de criar uma API REST, escrever middleware de autenticação, gerenciar certificados SSL e fazer deploy manual — você conecta o frontend diretamente à plataforma usando um SDK. O Firebase é o SDK; a plataforma Google Cloud é o servidor.

Pense como a diferença entre construir uma casa tijolo a tijolo versus comprar uma casa modular pré-fabricada: a casa modular é menos customizável, mas você se muda em dias. O Firebase é a casa modular do backend — pronto, escalável, com segurança embutida, mas com os trade-offs que isso implica.

Segundo Moroney (The Definitive Guide to Firebase, 2017), "Firebase moves the backend from a separate server to a set of cloud services that your application code calls directly from the client, secured by rules rather than server-side code."

Para iniciantes: Firebase não substitui um backend complexo em todos os casos. Ele é excelente para MVPs, apps mobile, dashboards e projetos com equipe pequena. Para lógica de negócio complexa, use Cloud Functions para manter o código server-side seguro.

Como Funciona Internamente

O Firestore é construído sobre a mesma infraestrutura do Cloud Bigtable e do Cloud Datastore do Google. Cada documento é replicado em múltiplas regiões automaticamente. A sincronização em tempo real usa WebSockets (web) ou gRPC (mobile) — o servidor mantém conexões persistentes e envia diffs quando os dados mudam.

fluxo-interno.txt
── Fluxo de uma escrita no Cloud Firestore ───────────────────

Cliente  ──SDK──▶  Firestore SDK (local)
                      │
                      ├─ 1. Aplica escrita localmente (cache)
                      │     ↳ Listener dispara imediatamente (UI otimista)
                      │
                      ├─ 2. Envia via WebSocket/gRPC
                      │
                      ▼
             Google Cloud Firestore
                      │
                      ├─ 3. Avalia Security Rules
                      │     ↳ Se negado → retorna erro, reverte cache local
                      │
                      ├─ 4. Persiste em múltiplas réplicas (strong consistency)
                      │
                      └─ 5. Notifica todos os listeners activos via WebSocket
                               ↳ Todos os clientes atualizam em ~100ms

O modo offline do Firestore armazena os dados localmente em IndexedDB (web) ou LevelDB (Android/iOS). Quando o dispositivo fica sem conexão, as leituras e escritas continuam funcionando localmente — ao reconectar, o SDK sincroniza automaticamente resolvendo conflitos por "last-write-wins" por padrão.

Dois Bancos NoSQL: Realtime Database vs Firestore

O Firebase oferece dois bancos com filosofias distintas. O Realtime Database (2011) é o original — simples e ultra-rápido para sincronização. O Cloud Firestore (2017) é o sucessor — queries poderosas, melhor estrutura e multi-região.

Critério Realtime Database Cloud Firestore
Modelo de dadosÁrvore JSON gigante — um nó únicoCollections → Documents → Subcollections
QueriesApenas 1 filtro por query; sem paginação flexívelFiltros compostos, ordenação, paginação com cursores
EscalabilidadeUm banco por região; gargalo em gravaçõesHorizontal automático, multi-região nativo
ConsistênciaEventual (sincronização por região)Strong consistency para leitura após escrita
CustoPor GB/mês armazenado + conexõesPor operação (leitura/escrita/exclusão)
OfflineLimitadoCompleto (IndexedDB / LevelDB)
TransaçõesNão suportaACID com transações e batched writes
Ideal paraChats simples, placar de jogos em tempo realApps de produção, catálogos, perfis de usuários

Para novos projetos, o Google recomenda Cloud Firestore. O Realtime Database é mantido em produção, mas novas funcionalidades são adicionadas apenas ao Firestore.

Estrutura de Dados: Collections e Documents

O Firestore organiza dados em coleções que contêm documentos JSON. Documentos podem ter subcoleções, criando uma hierarquia. Não existe "tabela de usuários" — existe uma coleção users com documentos onde cada documento é um usuário.

firestore-sdk.js
import { getFirestore, collection, doc,
         addDoc, getDoc, query, where,
         orderBy, limit, getDocs,
         onSnapshot, runTransaction } from "firebase/firestore";

const db = getFirestore();

// ─── Criar documento com ID automático ────────────────────────
const docRef = await addDoc(collection(db, "produtos"), {
  nome:       "Tênis Runner Pro",
  preco:      299.90,
  categorias: ["esporte", "corrida"],
  estoque:    150,
  criadoEm:   serverTimestamp()
});

// ─── Query composta com filtro + ordenação + paginação ─────────
const q = query(
  collection(db, "produtos"),
  where("preco", "<=", 500),
  where("estoque", ">", 0),
  orderBy("preco", "asc"),
  limit(20)
);
const snapshot = await getDocs(q);
snapshot.forEach(d => console.log(d.id, d.data()));

// ─── Transação ACID: transferência de estoque ─────────────────
await runTransaction(db, async (t) => {
  const origem  = await t.get(doc(db, "depositos", "sp"));
  const destino = await t.get(doc(db, "depositos", "rj"));
  const qtd = 10;
  t.update(doc(db, "depositos", "sp"), { estoque: origem.data().estoque - qtd });
  t.update(doc(db, "depositos", "rj"), { estoque: destino.data().estoque + qtd });
});

Security Rules — O Backend Declarativo

Sem servidor, como garantir que um usuário não leia os dados de outro? A resposta do Firebase são as Security Rules: uma linguagem declarativa avaliada pelo servidor antes de qualquer leitura ou escrita. São a principal diferença entre um app Firebase seguro e um vulnerável.

firestore.rules
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    // Perfis: usuário só lê/escreve o próprio documento
    match /users/{userId} {
      allow read, write: if request.auth != null
                         && request.auth.uid == userId;
    }

    // Produtos: qualquer um lê; só admins escrevem
    match /produtos/{prodId} {
      allow read: if true;
      allow write: if request.auth.token.admin == true;
    }

    // Pedidos: criar se autenticado e userId bate; ler apenas o próprio
    match /pedidos/{pedidoId} {
      allow create: if request.auth != null
                    && request.resource.data.userId == request.auth.uid;
      allow read: if request.auth.uid == resource.data.userId;
      allow update, delete: if false; // imutável após criação
    }
  }
}

Erro crítico mais comum: deixar as rules abertas em dev e esquecer de fechar antes do deploy. allow read, write: if true; significa que qualquer pessoa na internet pode ler e deletar todos os seus dados. Sempre defina rules restritivas antes do primeiro deploy público.

Casos de Uso

Use Firebase quando...

  • MVP ou protótipo com time pequeno (1–3 devs)
  • App mobile com autenticação social e sync offline
  • Chat em tempo real, live feed, colaboração
  • Dashboard analítico com atualizações ao vivo
  • Jogos casuais com placar global em tempo real
  • App que precisa ir a produção rapidamente

Evite Firebase quando...

  • Lógica de negócio complexa requer backend robusto
  • Queries SQL complexas (JOINs, aggregations pesadas)
  • Volume de leituras/escritas muito alto (custo por operação)
  • Dados altamente relacionados (use SQL ou Neo4j)
  • Regulamentações que exigem dados fora do Google Cloud

Avançado: Cloud Functions e Triggers

Quando a lógica não cabe nas Security Rules, Cloud Functions permite executar código Node.js no servidor do Google — serverless, com billing por invocação.

cloud-functions.js
const functions = require("firebase-functions");
const admin    = require("firebase-admin");
admin.initializeApp();

// Trigger: ao criar um pedido, enviar e-mail + atualizar estoque
exports.onPedidoCriado = functions.firestore
  .document("pedidos/{pedidoId}")
  .onCreate(async (snap, context) => {
    const pedido = snap.data();
    const db = admin.firestore();

    // Decrementar estoque atomicamente
    await db.runTransaction(async (t) => {
      const prodRef = db.doc(`produtos/${pedido.produtoId}`);
      const prod    = await t.get(prodRef);
      if (prod.data().estoque < pedido.quantidade) {
        throw new Error("Estoque insuficiente");
      }
      t.update(prodRef, {
        estoque: admin.firestore.FieldValue.increment(-pedido.quantidade)
      });
    });

    // Enviar e-mail via callable externo
    await sendEmail(pedido.email, "Pedido confirmado!");
  });

// Callable Function: validação server-side chamada pelo cliente
exports.calcularFrete = functions.https.onCall(async (data, context) => {
  if (!context.auth) throw new functions.https.HttpsError("unauthenticated", "Login obrigatório");
  const frete = await calcularFreteAPI(data.cep, data.peso);
  return { frete };
});

Cold Starts: Cloud Functions escala para zero quando não há tráfego. A primeira invocação após um período inativo demora de 1 a 5 segundos (cold start). Para funções críticas de latência, use minimum instances para manter ao menos uma instância quente — mas isso gera custo contínuo.

Ecossistema Firebase

Firebase Authentication

Login pronto para Google, Apple, GitHub, Facebook, e-mail/senha e SMS OTP. Gerencia tokens JWT, renovação automática, revogação de sessão e integração com Custom Claims para roles/permissões.

Firebase Storage

Armazenamento de arquivos (imagens, vídeos, PDFs) sobre o Google Cloud Storage. Security Rules idênticas ao Firestore. Upload resumível com suporte nativo a pausar e retomar em mobile.

Firebase Hosting

Hospedagem global com CDN automática, SSL grátis, deploy atômico com rollback instantâneo e suporte a rewrites para SPAs (React, Vue, Angular) e Cloud Functions.

Remote Config

Altere comportamentos e configurações do app em produção sem publicar nova versão. Suporte a condições (por plataforma, versão, segmento de usuários) e integração com A/B Testing.

Firebase Emulator Suite

Emuladores locais para Firestore, Realtime DB, Auth, Storage, Functions e Hosting. Permite desenvolvimento e testes completos offline sem afetar dados de produção — CLI: firebase emulators:start.

Crashlytics + Analytics

Rastreamento de crashes em tempo real com stack trace completo e agrupamento por similaridade. Google Analytics integrado com funis, cohort analysis e integração com BigQuery para queries customizadas.
