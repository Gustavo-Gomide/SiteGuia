Amazon DynamoDB

NoSQL serverless da AWS — latência de milissegundos em qualquer escala.

O Amazon DynamoDB é um banco de dados de chave-valor e documentos totalmente gerenciado pela AWS. Sem servidores para provisionar, sem patches, sem tuning — você define a tabela e a Amazon cuida de replicação, backups e escalabilidade automática. Foi o sistema que inspirou o famoso paper "Dynamo" de 2007, que moldou toda uma geração de bancos NoSQL.

O que é o DynamoDB?

O DynamoDB é um banco de dados chave-valor + documento criado para resolver um problema real da Amazon: em 2004, durante o pico de vendas do Black Friday, os bancos relacionais travavam sob carga. A solução foi projetar um sistema que priorizasse disponibilidade e latência previsível em vez de consistência forte.

Pense em um armário de fichários de biblioteca: cada gaveta tem uma etiqueta (partition key) e dentro dela os documentos estão ordenados por número (sort key). Para recuperar um arquivo você vai direto à gaveta correta — não precisa varrer a biblioteca inteira. Essa navegação direta é o que garante latência de P99 abaixo de 10 ms, independente de terabytes de dados.

DeCandia et al., 2007: "Dynamo sacrifices consistency under certain failure scenarios and uses object versioning and application-assisted conflict resolution in a novel manner." — Dynamo: Amazon's Highly Available Key-value Store, SOSP 2007.

DynamoDB não é um banco relacional disfarçado. Tentar modelar dados relacionais (com JOINs em múltiplas tabelas) no DynamoDB é o caminho mais rápido para uma fatura AWS alta e uma performance ruim. Single-Table Design é a abordagem correta — aprenda antes de começar.

Como o DynamoDB Funciona

O DynamoDB distribui dados usando consistent hashing: a partition key é hashada com MD5 e o resultado determina qual nó físico (partition) armazena o dado. Cada partition suporta até 10 GB e 3.000 RCU + 1.000 WCU. Quando o limite é atingido, o serviço divide a partition automaticamente — sem downtime, sem migração manual.

dynamo-flow.txt
# Fluxo de uma escrita no DynamoDB
# ─────────────────────────────────────────────────────────

Cliente  ──PUT Item──►  Request Router
                              │
                     hash(PK) → partition_id
                              │
                    Partition Leader  ──►  Replica 1
                              │               ──►  Replica 2
                              │
                    WAL (Write-Ahead Log) gravado
                    ACK devolvido ao cliente
                              │
                    DynamoDB Streams  ──►  Lambda / Kinesis
                    (evento de mudança gerado para cada write)

# Consistência por padrão: eventual (lê de qualquer réplica)
# Strongly Consistent Read: lê apenas do leader (+50% custo)

Consistent Hashing

Cada partition key é hashada para determinar qual nó armazena o dado. A distribuição uniforme de partitions garante que nenhum nó fique sobrecarregado — desde que a chave seja bem escolhida. Uma PK ruim (ex: status "ativo/inativo") concentra tudo em 2 nós: hotspot partition.

WAL + Replicação

Cada escrita é gravada no Write-Ahead Log de 3 réplicas antes do ACK ao cliente. Isso garante durabilidade mesmo se um nó falhar imediatamente após a confirmação. A replicação é síncrona para o líder e assíncrona para as réplicas secundárias.

DynamoDB Streams

Toda mudança na tabela (INSERT, MODIFY, REMOVE) gera um evento ordenado em um stream com TTL de 24h. Use com AWS Lambda para arquitetura event-driven: atualizar cache, disparar notificações, replicar para Elasticsearch — sem polling, sem mensageria separada.

Partition Key, Sort Key e Single-Table Design

A modelagem no DynamoDB começa pela pergunta: "Quais queries minha aplicação precisa fazer?" — ao contrário do SQL, onde você modela os dados e depois escreve as queries. Cada item tem obrigatoriamente uma Partition Key (PK) e, opcionalmente, uma Sort Key (SK). Juntas, elas formam a chave primária composta.

PK (Partition Key) SK (Sort Key) Entidade / Significado
CLIENTE#101PERFILDados do cliente (email, nome, endereço)
CLIENTE#101PEDIDO#2024-01-15Pedido específico — range por data
CLIENTE#101PEDIDO#2024-03-22Outro pedido do mesmo cliente
PRODUTO#XYZESTOQUEQuantidade em estoque
PRODUTO#XYZREVIEW#5stars#uuidReviews ordenadas por nota
SESSAO#abc123METADATASessão de usuário + TTL automático

Esse padrão é chamado de Single-Table Design: múltiplas entidades convivem na mesma tabela, diferenciadas por prefixos na PK e SK. O benefício: uma única chamada Query(PK="CLIENTE#101") retorna o perfil + todos os pedidos em uma única roundtrip à AWS — sem JOINs, sem múltiplas queries.

dynamodb-sdk.js (AWS SDK v3)
import { DynamoDBClient } from "@aws-sdk/client-dynamodb";
import { DynamoDBDocumentClient, PutCommand, QueryCommand, TransactWriteCommand } from "@aws-sdk/lib-dynamodb";

const client = DynamoDBDocumentClient.from(new DynamoDBClient({ region: "us-east-1" }));

// 1. Inserir item (marshalling automático com DocumentClient)
await client.send(new PutCommand({
  TableName: "Loja",
  Item: {
    PK: "CLIENTE#101",
    SK: "PEDIDO#2024-01-15",
    total: 299.90,
    status: "Processando",
    TTL: Math.floor(Date.now() / 1000) + 7776000  // expira em 90 dias
  },
  ConditionExpression: "attribute_not_exists(PK)"  // evita sobrescrever
}));

// 2. Query por PK + SK begins_with (um cliente e todos os pedidos)
const { Items } = await client.send(new QueryCommand({
  TableName: "Loja",
  KeyConditionExpression: "PK = :pk AND begins_with(SK, :prefix)",
  ExpressionAttributeValues: { ":pk": "CLIENTE#101", ":prefix": "PEDIDO#" },
  ScanIndexForward: false,  // ordem decrescente (mais recentes primeiro)
  Limit: 10
}));

// 3. Transação ACID: aprovar pedido + decrementar estoque atomicamente
await client.send(new TransactWriteCommand({
  TransactItems: [
    { Update: { TableName: "Loja", Key: { PK: "CLIENTE#101", SK: "PEDIDO#2024-01-15" },
        UpdateExpression: "SET #s = :aprovado",
        ExpressionAttributeNames: { "#s": "status" },
        ExpressionAttributeValues: { ":aprovado": "Aprovado" } } },
    { Update: { TableName: "Loja", Key: { PK: "PRODUTO#XYZ", SK: "ESTOQUE" },
        UpdateExpression: "SET quantidade = quantidade - :um",
        ConditionExpression: "quantidade > :zero",
        ExpressionAttributeValues: { ":um": 1, ":zero": 0 } } }
  ]
}));

Quando Usar (e Quando Evitar)

Use DynamoDB quando

  • Precisa de latência de leitura/escrita abaixo de 10 ms com P99 garantido.
  • A aplicação já está na AWS e quer integração nativa com Lambda, API Gateway, EventBridge.
  • Workload de chave-valor clara: sessões, carrinho de compras, preferências de usuário.
  • Scale horizontal automático sem gerenciar shards: de zero a milhões de req/s.
  • Precisa de TTL automático em itens (sessões, cache, eventos temporários).
  • Arquitetura event-driven com DynamoDB Streams + Lambda.

Evite DynamoDB quando

  • A aplicação precisa de queries ad-hoc com WHERE em múltiplos campos — use PostgreSQL.
  • O modelo de dados é naturalmente relacional com muitos JOINs — o custo de modelagem é alto.
  • O volume de dados é menor que 10 GB e o tráfego é baixo — um RDS tb.micro é suficiente e mais barato.
  • Precisa de transações distribuídas cross-table complexas — DynamoDB limita TransactItems a 25 ações.
  • A equipe está fora do ecossistema AWS — Cassandra ou ScyllaDB são alternativas multi-cloud.

Avançado: GSIs, DAX e Capacidade

Após dominar o Single-Table Design, os próximos tópicos que fazem a diferença em produção são os índices secundários, o cache em memória e o planejamento de capacidade.

GSI — Global Secondary Index

Permite criar uma segunda "visão" da tabela com PK e SK diferentes. Exemplo: tabela principal com PK=CLIENTE, GSI com PK=STATUS para buscar todos os pedidos pendentes. Cada GSI replica os dados e tem seu próprio throughput — use com moderação.

LSI — Local Secondary Index

Índice alternativo com a mesma PK mas SK diferente. Deve ser definido na criação da tabela (não pode ser adicionado depois). Útil para ordenar itens da mesma partição por múltiplos critérios simultaneamente. Limite: 5 LSIs por tabela.

DAX — DynamoDB Accelerator

Cache in-memory totalmente compatível com DynamoDB. Reduz latência de milissegundos para microssegundos sem mudar uma linha de código (troca apenas o endpoint). Ideal para leituras repetitivas de dados hot (rankings, configurações, catálogos).

Modos de Capacidade

On-Demand: paga por request (RRU/WRU). Ideal para workloads imprevisíveis. Provisionado + Auto-Scaling: define RCU/WCU, mais previsível e até 7× mais barato para cargas estáveis. Fórmula: 1 WCU = 1 escrita/s até 1 KB; 1 RCU = 1 leitura forte/s até 4 KB.

dynamodb-gsi.js (query em GSI)
// GSI: "GSI1" com PK=status, SK=createdAt — buscar pedidos pendentes recentes
const { Items } = await client.send(new QueryCommand({
  TableName: "Loja",
  IndexName: "GSI1",
  KeyConditionExpression: "#st = :pendente AND createdAt > :data",
  ExpressionAttributeNames: { "#st": "status" },
  ExpressionAttributeValues: {
    ":pendente": "Pendente",
    ":data":    "2024-01-01"
  },
  ScanIndexForward: false
}));

Hot Partition: se todos os requests baterem na mesma partition key (ex: PK = "hoje" num sistema de logs), você ultrapassará o limite de 3.000 RCU + 1.000 WCU por partition e receberá erros ProvisionedThroughputExceededException. Soluções: adicionar sufixo aleatório à PK (write sharding) + GSI com agregação, ou usar Kinesis Data Streams diretamente para ingestão de alto volume.

Ecossistema DynamoDB

DynamoDB Local

Versão emulada que roda localmente em JAR ou Docker. Ideal para testes unitários e CI/CD sem custos de AWS. Comportamento idêntico ao serviço real, incluindo streams e GSIs.

NoSQL Workbench

GUI oficial da AWS para visualizar tabelas, modelar Single-Table Design com o Data Modeler e executar queries. Disponível para Windows, Mac e Linux. Essencial para aprender a modelagem.

PartiQL

Linguagem SQL-compatível da AWS que funciona sobre DynamoDB. Permite usar SELECT * FROM Loja WHERE PK = 'CLIENTE#101' em vez da API nativa. Útil para queries exploratórias — não recomendado para produção de alta performance.

boto3 e SDKs

SDKs oficiais para Python (boto3), JavaScript/TypeScript (v3), Java (SDK v2), Go, .NET e mais. O DynamoDBDocumentClient (JS) e o DynamoDBMapper (Java) simplificam o marshalling de tipos.

AWS CDK / Terraform

Infrastructure as Code para provisionar tabelas, GSIs, Auto-Scaling policies e Streams. O CDK permite definir tudo em TypeScript/Python com validação em tempo de build.

DynamoDB + Lambda

Padrão de arquitetura serverless: API Gateway → Lambda → DynamoDB. Com Streams, Lambda reage a mudanças para atualizar caches, enviar notificações e manter projeções secundárias em Elasticsearch ou Redshift — CQRS sem servidor dedicado.
