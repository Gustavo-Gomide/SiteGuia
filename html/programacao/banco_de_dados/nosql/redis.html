Redis

In-Memory Data Platform — latencia sub-milissegundo, 10+ estruturas de dados, eventos em tempo real.

O Redis (Remote Dictionary Server) armazena dados inteiramente em RAM. Criado por Salvatore Sanfilippo em 2009, opera em 0.1–0.5ms de latencia — 100 a 10.000 vezes mais rapido que bancos em disco. Nao e apenas um cache: e uma plataforma de dados em tempo real com Strings, Lists, Sets, Sorted Sets, Hashes, Streams, HyperLogLog e busca vetorial via Redis Stack.

1. O Que E o Redis?

Enquanto bancos tradicionais precisam buscar dados em disco (5–10ms por operacao), o Redis opera inteiramente em RAM. A diferenca nao e de otimizacao — e de arquitetura:

Hierarquia de latencia: CPU L1 cache ~0.5ns → RAM (Redis) ~100ns → SSD NVMe ~100µs → HDD ~10ms. O Redis e 100x mais rapido que SSD e 100.000x mais rapido que HDD para leituras simples. Isso e uma diferenca de ordem de grandeza, nao de percentual.

Salvatore Sanfilippo projetou o Redis com uma filosofia clara: "Use Redis para o que so funciona com velocidade de memoria; use um banco persistente para o que precisa de durabilidade." A combinacao Redis + PostgreSQL (ou Redis + MongoDB) e uma das arquiteturas mais comuns em sistemas de alta performance — Redis como camada de dados quentes, banco relacional para dados frios e criticos.

2. Como Funciona por Dentro

O Redis e single-threaded para operacoes de dados (I/O multiplexado via epoll/kqueue). Isso elimina locks e context-switching — cada operacao e atomica por natureza. Desde a v6.0, o I/O de rede usa threads multiplas, mantendo operacoes em memoria no thread principal.

arquitetura-redis.txt
Event Loop (Single-Thread de dados):
  epoll/kqueue I/O multiplexado
  |-> aceita 100K+ conexoes simultaneas
  |-> processa um comando por vez (atomicidade garantida)
  |-> INCR nunca tem race condition

Persistencia (Opcoes):
  RDB (Snapshot)         <- dump binario do estado em memoria
    SAVE 900 1           <- salva se 1 mudanca em 900s
    SAVE 60 10000        <- salva se 10K mudancas em 60s
    Pro: arquivo compacto, restore rapido
    Contra: pode perder dados desde o ultimo snapshot

  AOF (Append-Only File) <- log de cada operacao de escrita
    appendfsync always   <- fsync a cada write (duravel, lento)
    appendfsync everysec <- fsync por segundo (balanceado)
    Pro: RPO proximo de zero
    Contra: arquivo cresce, requer BGREWRITEAOF periodico

  RDB + AOF              <- combinacao recomendada em producao

Atomicidade Nativa

Toda operacao Redis e atomica. INCR incrementa sem race condition. Para operacoes compostas, MULTI/EXEC agrupa comandos em transacao (todos ou nenhum). Scripts Lua via EVAL executam como unidade atomica, permitindo logica complexa (compare-and-swap) sem round-trips extras.

Keyspace e TTL

Toda chave pode ter TTL (Time-To-Live) em segundos ou milissegundos. Expiracao e lazy (remove na proxima leitura) + periodica em background (amostra 20 keys/ciclo). Keyspace Notifications emitem eventos quando chaves expiram, sao modificadas ou deletadas — cache com invalidacao reativa sem polling.

Sentinel vs Cluster

Sentinel: monitora 1 master + N replicas e executa failover automatico sem sharding — ideal para ate ~100 GB de dados. Cluster: sharding automatico em 16.384 hash slots — escala horizontal ilimitada. Minimo 3 masters com 3 replicas para tolerancia a falhas de nos.

3. Estruturas de Dados

Redis nao e apenas chave-string. Cada tipo tem comandos especializados e complexidade de tempo garantida (O(1), O(log N)). Escolher a estrutura certa elimina codigo de aplicacao inteiro.

redis-estruturas.redis
# STRING — texto, numero, JSON, blob binario (max 512MB)
SET    sessao:abc '{"userId":42,"role":"admin"}' EX 3600
INCR   visitas:home                           # atomico, O(1)
SET    lock:recurso:1 "thread-7" NX EX 30   # distributed lock

# LIST — duplamente encadeada — O(1) nas extremidades
RPUSH  fila:emails "job:1001" "job:1002"
LPOP   fila:emails                             # FIFO dequeue
BRPOP  fila:emails 0                           # blocking pop

# SET — conjunto sem duplicatas
SADD   tags:post:99 "nosql" "redis" "cache"
SINTER tags:post:99 tags:post:100              # intersecao
SUNION tags:post:99 tags:post:100              # uniao

# SORTED SET — membros com score float, ordenados
ZADD      ranking 2300 "carlos"  1500 "ana"
ZREVRANGE ranking 0 9 WITHSCORES             # top 10
ZADD      agenda 1735700000 "tarefa:99"       # timestamp como score

# HASH — campos nomeados dentro de uma chave
HSET    user:42 nome "Ana" idade 28 plano "premium"
HGET    user:42 nome
HINCRBY user:42 pontos 100                    # atomico

# STREAM — log imutavel com grupos de consumidores (Kafka-like)
XADD       eventos "*" tipo "compra" userId 42 valor 299.90
XREADGROUP GROUP processadores p1 COUNT 10 STREAMS eventos ">"
XACK       eventos processadores "1735700000-0"

# HYPERLOGLOG — contagem de unicos com 0.81% erro, 12KB fixo
PFADD   visitantes:2024-01-15 "user:1" "user:2" "user:1"
PFCOUNT visitantes:2024-01-15               # ~2 usuarios unicos

4. Padroes de Uso

Cache-Aside (Lazy Loading)

A aplicacao busca no Redis primeiro. Se nao encontrar (cache miss), busca no banco principal, armazena no Redis com TTL e retorna. Problema: thundering herd — multiplos requests concorrentes causam cache miss simultaneo. Solucao: mutex com SET ... NX EX: apenas o primeiro thread busca no banco; os outros aguardam o cache ser preenchido.

Rate Limiting

Use INCR + EXPIRE para limitar requisicoes por janela fixa de tempo. Para janelas deslizantes precisas, use o algoritmo Token Bucket com Sorted Sets (score = timestamp) — remove entradas antigas e conta o que resta.

Pub/Sub e Fan-out

PUBLISH / SUBSCRIBE entregam mensagens em tempo real a todos os subscribers. Sem persistencia — se o subscriber desconectar, perde mensagens. Use Streams para replay, grupos de consumidores, acknowledgment (delivery garantido) e reprocessamento de falhas.

Distributed Lock (Redlock)

O algoritmo Redlock usa N instancias Redis independentes para exclusao mutua distribuida. Para locks simples em um unico no, SET ... NX EX e suficiente. Para garantias fortes em sistemas criticos, prefira ZooKeeper ou etcd (Kleppmann detalhou limitacoes do Redlock extensamente em 2016).

5. Quando Usar e Quando Evitar

Use Redis quando...

  • Latencia sub-milissegundo obrigatoria: sessoes, tokens JWT, configuracoes hot-path.
  • Rankings em tempo real: Sorted Sets com ZADD/ZREVRANGE em O(log N).
  • Filas de trabalho: BRPOP com multiplos workers consumers sem polling.
  • Contadores de alta frequencia: INCR atomico sem coordenacao externa.
  • Pub/Sub em tempo real: notificacoes, chat, dashboards ao vivo sem polling.

Evite Redis quando...

  • Dados maiores que a RAM: Redis e in-memory; ao atingir o limite, evicta dados ou rejeita escritas.
  • Queries por conteudo: Redis nao tem linguagem de query — use MongoDB ou PostgreSQL.
  • Durabilidade absoluta: para dados criticos sem tolerancia a perda, use banco ACID dedicado.
  • Dados relacionais: sem suporte a JOINs ou schema estruturado.

6. Topicos Avancados

Redis Cluster

Sharding automatico em 16.384 hash slots distribuidos entre nos. Cada no e responsavel por uma faixa de slots. Hash tags ({user:42}) garantem que chaves relacionadas vao ao mesmo slot. Limitacao: comandos multi-chave (MGET, SUNION) so funcionam se todas as chaves estiverem no mesmo slot hash.

Lua Scripting

Scripts Lua executam como unidade atomica no servidor Redis. Permitem compare-and-swap, transacoes condicionais e logica complexa sem round-trips extras. O Redis 7+ introduziu Redis Functions como substituto moderno do EVAL — com namespace, persistencia entre reinicializacoes e carregamento dinamico.

Redis Stack

Modulos oficiais empacotados: RedisSearch (full-text + busca vetorial ANN com HNSW), RedisJSON (documentos JSON nativos com JSON Path), RedisTimeSeries (series temporais com downsampling) e RedisBloom (Bloom/Cuckoo Filters para membership probabilistico).

Redis 7+ e RESP3

Redis 7 trouxe Redis Functions, ACL v2 (permissoes por padrao de chave) e o protocolo RESP3 com tipos ricos (mapa, double, null tipado, blob-error) — eliminando parsing manual de strings no cliente para determinar tipo de retorno.

Limite de Memoria e Eviction: configure sempre maxmemory e a politica de eviction (allkeys-lru para cache puro, volatile-lru para dados mistos). Sem maxmemory, o Redis consome toda a RAM disponivel e o OOM Killer do Linux pode terminar o processo sem aviso.

7. Ecossistema e Ferramentas

Redis Cloud

DBaaS gerenciado pela Redis Ltd. Active-Active geo-replication com CRDTs, backups automaticos e SLA de 99.999%. Tier gratuito com 30MB para desenvolvimento.

Redis Insight

GUI oficial gratuita para inspecionar chaves, analisar uso de memoria por tipo, profiler de queries lentas via SLOWLOG e visualizacao de comandos em tempo real (MONITOR).

Clientes Populares

ioredis (Node.js), redis-py (Python), Lettuce / Jedis (Java), StackExchange.Redis (C#). Todos suportam cluster, pipelining de comandos e SSL/TLS.

BullMQ (Node.js)

Biblioteca de filas de trabalho sobre Redis. Suporta jobs com prioridade, retry com backoff exponencial, rate limiting, workers concorrentes e dashboard UI integrado via Arena ou BullMQ UI.

Celery (Python)

Sistema de filas e tasks assincronas mais popular do Python, com Redis como broker padrao. Usado para background jobs, agendamento periodico (beat) e workflows com chains e chords.

Valkey

Fork open-source do Redis criado pela Linux Foundation em 2024 apos mudanca de licenca para BSL 1.1. 100% compativel com Redis 7.2. Suportado por AWS, Google Cloud e Oracle como alternativa.
