Elasticsearch

Motor de busca distribuído — busca full-text, relevância e análise de logs em tempo real

O Elasticsearch é um motor de busca e análise distribuído construído sobre o Apache Lucene, em produção em Netflix, Wikipedia, GitHub, Uber, LinkedIn e Microsoft. Ele não é um banco de dados convencional — é uma engine que indexa documentos JSON e os torna pesquisáveis em milissegundos com relevância, fuzzy search, autocompletar e análise estatística, suportando petabytes de dados em centenas de servidores.

O que é o Elasticsearch?

O Elasticsearch é um banco de dados NoSQL do tipo full-text search que combina armazenamento de documentos JSON com indexação invertida automática. Cada documento indexado se torna instantaneamente pesquisável — sem precisar definir índices manualmente.

A analogia perfeita é o índice de um livro: em vez de percorrer todas as páginas procurando uma palavra (o que um LIKE '%palavra%' faz em SQL), o índice invertido vai diretamente à lista de páginas que contêm aquela palavra. O Elasticsearch é esse índice — mas para bilhões de documentos, completamente distribuído.

Segundo Hinman, Robinson e Lee (Elasticsearch in Action, 2021), "Elasticsearch is a near real-time search and analytics engine. It processes data as JSON documents and makes them searchable through a distributed inverted index maintained across a cluster of nodes."

Para iniciantes: Elastic não é um substituto de PostgreSQL para seu banco transacional. O padrão correto é usar os dois juntos: PostgreSQL ou MongoDB como source of truth para escrita/leitura transacional, e Elasticsearch como motor de busca para as queries de texto, filtros facetados e análise.

Como Funciona: Lucene e o Índice Invertido

Ao indexar um documento, o Elasticsearch aplica uma cadeia de transformações chamada análise de texto antes de armazenar no índice invertido:

pipeline-analise.txt
── Pipeline de análise do texto "Os Melhores Smartphones iPhone" ─

1. Character Filter  → remove HTML, normaliza acentos
   "Os Melhores Smartphones iPhone"

2. Tokenizer         → divide em tokens
   ["Os", "Melhores", "Smartphones", "iPhone"]

3. Token Filters     → lowercase + stopwords + stemming
   ["melhores", "smartphone", "iphon"]
   //      ↑ stopword "os" removida
   //      ↑ stemming: "smartphones" → "smartphone"

4. Índice Invertido  → mapeia tokens para documentos:
   "smartphone"  →  [ Doc1(freq:2), Doc5(freq:1), Doc9(freq:3) ]
   "iphon"       →  [ Doc1(freq:1), Doc7(freq:2) ]
   "melhor"      →  [ Doc1(freq:1), Doc3(freq:1) ]

5. BM25 Scoring      → relevância = TF × IDF × field norms
   // Score alto = termo aparece frequentemente no doc (TF)
   // Score alto = termo raro no corpus inteiro (IDF)

Arquitetura Distribuída

Índice e Shards

Um índice Elasticsearch é dividido em shards primários (unidades de armazenamento Lucene) distribuídos pelo cluster. Cada shard é um índice Lucene completo e independente. Por default: 1 shard primário (ES 7.x+).

Réplicas

Cada shard primário tem cópias (replica shards) em outros nodes. Réplicas servem leituras em paralelo — quanto mais réplicas, maior o throughput de leitura. Se o node primário cair, uma réplica promovida vira primário automaticamente.

Near Real-Time

Documentos indexados ficam pesquisáveis em ~1 segundo (o intervalo de refresh). Internamente, novos documentos ficam em buffer na memória (in-memory buffer) antes de serem escritos em segmentos Lucene no disco.

Critério SQL LIKE '%termo%' Elasticsearch

Query DSL: A Linguagem de Busca

Toda comunicação com o Elasticsearch é via HTTP + JSON. O Query DSL é a linguagem de consulta — mais expressiva que SQL para busca textual, com composição de queries por booleanos.

elastic-queries.http
// ─── Indexar documento com mapeamento explícito ─────────────
PUT /produtos
{
  "mappings": {
    "properties": {
      "nome":      { "type": "text", "analyzer": "portuguese" },
      "preco":     { "type": "float" },
      "categoria": { "type": "keyword" },
      "estoque":   { "type": "integer" }
    }
  }
}

// ─── Query booleana: full-text + filtros + sorting ──────────
POST /produtos/_search
{
  "query": {
    "bool": {
      "must":   [{ "match": { "nome": "smartphone" } }],
      "filter": [
        { "range":  { "preco":   { "lte": 3000 } } },
        { "range":  { "estoque": { "gt": 0    } } },
        { "term":   { "categoria": "eletronicos" } }
      ]
    }
  },
  "sort":      [{ "preco": { "order": "asc" } }],
  "highlight": { "fields": { "nome": {} } },  // destaca o match
  "size": 20, "from": 0   // paginação
}

// ─── Fuzzy: tolerante a erros de digitação ──────────────────
GET /produtos/_search
{ "query": { "fuzzy": { "nome": { "value": "ipbone", "fuzziness": "AUTO" } } } }

// ─── Aggregations: analytics sobre os resultados ────────────
POST /produtos/_search
{
  "size": 0,
  "aggs": {
    "por_categoria": { "terms": { "field": "categoria" } },
    "preco_medio":   { "avg":   { "field": "preco" } }
  }
}

Casos de Uso

Use Elasticsearch quando...

  • Busca full-text no catálogo de produtos de e-commerce
  • Observabilidade: indexar e pesquisar logs de aplicação (ELK)
  • Busca em documentos, contratos, wikis corporativos
  • Autocompletar e sugestões em tempo real
  • Análise de métricas e dashboards com Kibana
  • Detecção de anomalias em séries temporais de logs

Evite Elasticsearch quando...

  • Source of truth transacional (use PostgreSQL/MongoDB)
  • Queries JOIN complexas e relacionamentos entre entidades
  • Dados financeiros que exigem ACID rigoroso
  • Writes muito frequentes no mesmo documento (custo de reindex)
  • Equipe sem capacidade de operar cluster Lucene em produção

Avançado: Otimizações e Arquitetura

Mappings: keyword vs text

A decisão mais crítica no Elasticsearch é o tipo de campo. Usar text onde deveria ser keyword causa bugs sutis — aggregations e sorting falham silenciosamente com campos text.

mapping-tipos.json
// text: analisado, pesquisável por partes — bom para descrições
"descricao": { "type": "text", "analyzer": "portuguese" }

// keyword: valor exato, não analisado — bom para filtros e aggregations
"status": { "type": "keyword" }

// multi-field: texto + keyword juntos no mesmo campo
"nome": {
  "type": "text", "analyzer": "portuguese",
  "fields": {
    "raw": { "type": "keyword" }  // nome.raw para sort/aggregation
  }
}

// dense_vector: para buscas semânticas (KNN / Vector Search)
"embedding": { "type": "dense_vector", "dims": 1536 }

Mapping explosion: o Elasticsearch cria campos dinamicamente por padrão. Em logs não estruturados com centenas de campos únicos, isso pode criar million-field mappings que degradam o cluster inteiro. Defina "dynamic": "strict" no mapping de produção e declare todos os campos explicitamente.

Ecossistema: ELK Stack e Além

Kibana

Interface visual do Elastic Stack: dashboards interativos, gráficos de séries temporais, mapas geoespaciais, alertas, Machine Learning e Discover (busca ad-hoc em logs). Acesso em localhost:5601.

Logstash e Beats

Logstash: pipeline ETL para ingestão e transformação flexível de logs (input → filter → output). Beats: agentes leves por categoria — Filebeat (logs), Metricbeat (métricas), Packetbeat (rede).

Elastic Cloud / OpenSearch

Elastic Cloud: cluster gerenciado na AWS/GCP/Azure. OpenSearch (fork AWS): alternativa open-source mantida pela Amazon após mudança de licença do Elastic em 2021.

Vector Search (KNN)

Desde ES 8.0, suporte nativo a busca por vetores de alta dimensão (ANN / KNN) via HNSW. Permite buscas semânticas com embeddings de modelos de linguagem (OpenAI, BERT, Sentence Transformers) — searchando por significado, não apenas palavras.
