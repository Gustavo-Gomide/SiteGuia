Neo4j

Banco de dados de grafos — do básico ao avançado

O Neo4j é o líder mundial em bancos de dados de grafos, com mais de 1.000 empresas usando em produção incluindo eBay, NASA, Airbus, UBS e LinkedIn. Enquanto bancos relacionais guardam dados em tabelas e resolvem relacionamentos com JOINs, o Neo4j armazena os relacionamentos como cidadãos de primeira classe — com ponteiros físicos diretos entre nós. Não há JOINs em tempo de query: navegar milhões de relacionamentos custa menos do que um segundo.

O que é um Banco de Grafos?

Um banco de grafos armazena dados como um conjunto de nós (entidades) conectados por arestas (relacionamentos), onde ambos podem ter propriedades chave-valor. O modelo é chamado de Property Graph.

Pense como um quadro branco com post-its coloridos (nós) e setas desenhadas entre eles (relacionamentos). Cada post-it tem anotações (propriedades) e cada seta tem um rótulo descritivo e suas próprias anotações. Isso é exatamente como o Neo4j armazena e consulta dados — a representação visual é idêntica ao modelo interno.

Segundo Robinson, Webber e Eifrem (Graph Databases, 2015), "in a relational database, relationships are first-class citizens only in the conceptual model. In a graph database, they're first-class citizens in the storage model too." É essa diferença que torna o Neo4j revolucionariamente mais eficiente para dados conectados.

Para iniciantes: o problema central que o Neo4j resolve é o "JOIN exponencial". Em SQL, consultar "amigos de amigos de amigos" exige múltiplos self-JOINs que ficam proibitivamente lentos com 5+ níveis de profundidade. No Neo4j, a mesma query de N níveis executa em milissegundos — o custo é proporcional ao número de nós visitados, não ao tamanho total do banco.

Como Funciona Internamente

Native Graph Storage

A diferença fundamental do Neo4j em relação a bancos que "emulam" grafos em tabelas é o armazenamento nativo de grafos. Cada nó armazena um ponteiro direto para o início de sua lista de relacionamentos. Cada relacionamento armazena ponteiros diretos para o nó de origem, nó de destino e o próximo relacionamento de cada nó.

estrutura-interna.txt
── Estrutura de um Nó no Neo4j (armazenamento nativo) ────────
Node Record:
  inUse: true
  firstRelationshipId: → rel#42     ← ponteiro para 1º relacionamento
  firstPropertyId:    → prop#17    ← ponteiro para 1ª propriedade
  labels: ["Pessoa"]

── Estrutura de um Relacionamento ────────────────────────────
Relationship Record #42:
  inUse: true
  firstNode:  → node#5    ← nó de origem
  secondNode: → node#12   ← nó de destino
  type: "AMIGO_DE"
  nextRelForFirstNode:  → rel#47   ← próximo rel do nó origem
  nextRelForSecondNode: → rel#38   ← próximo rel do nó destino

── Resultado: traversal segue ponteiros em memória ──────────
── Custo = O(relacionamentos visitados), NÃO O(banco inteiro) ──

Por que é mais rápido que SQL para grafos

SQL: JOIN varredura de tabela

Para "amigos de amigos" de Ana em SQL: JOIN amizades a1 ON ... JOIN amizades a2 ON a1.amigo_id = a2.pessoa_id. Com 1 milhão de amizades na tabela, cada JOIN varre 1M de linhas. 3 níveis de profundidade = 1M³ de operações potenciais.

Neo4j: traversal por ponteiro

Para "amigos de amigos" de Ana no Neo4j: o engine segue ponteiros físicos diretos. Ana tem 300 amigos → cada um tem 300 amigos = 90.000 traversals máximos. O banco inteiro poderia ter 100M de nós — o resultado seria o mesmo em milissegundos.

O Modelo Property Graph

O modelo usado pelo Neo4j é o Property Graph, composto por quatro elementos fundamentais:

Nós (Nodes)

Representam entidades do domínio: Pessoas, Filmes, Produtos, Cidades. Cada nó tem um ou mais labels (tipo/categoria) e propriedades chave-valor.

(:Pessoa {nome: "Ana", idade: 28})

Relacionamentos (Relationships)

Setas direcionadas entre nós com tipo obrigatório e propriedades opcionais. Armazenados como ponteiros diretos — acesso O(1) a partir do nó.

-[:AMIGO_DE {desde: 2020}]->

Labels

Classificam nós em grupos. Um nó pode ter múltiplos labels. Funcionam como índices categoriais — query por label é eficiente sem varrer todos os nós.

(:Pessoa:Funcionario {nome: "Carlos"})

Propriedades

Pares chave-valor em nós e relacionamentos. Tipos suportados: string, number, boolean, date, lista. Podem ser indexados para queries eficientes.

{nome: "Ana", pontos: 1500, ativo: true}

Cypher — A Linguagem em ASCII Art

O Cypher foi projetado com um princípio único: a sintaxe da query deve se parecer visualmente com o grafo que ela descreve. Você "desenha" o padrão que quer encontrar usando parênteses para nós e setas para relacionamentos.

cypher-exemplos.cypher
// 1. Criar nós e relacionamento
CREATE (keanu:Pessoa {nome: "Keanu Reeves", nasc: 1964})
       -[:ATUOU_EM {papel: "Neo", ano: 1999}]->
       (matrix:Filme {titulo: "Matrix", bilheteria: 463500000});

// 2. MATCH: encontrar padrão no grafo
//    "quem atuou no Matrix?"
MATCH (p:Pessoa)-[r:ATUOU_EM]->(f:Filme)
WHERE f.titulo = "Matrix"
RETURN p.nome, r.papel, f.titulo;

// 3. Traversal de N níveis — impossível em SQL sem recursão
//    "amigos de amigos de Ana (2 níveis de profundidade)"
MATCH (ana:Pessoa {nome: "Ana"})-[:AMIGO_DE*2]->(sugestoes)
WHERE NOT (ana)-[:AMIGO_DE]->(sugestoes)  // excluir amigos diretos
RETURN DISTINCT sugestoes.nome;

// 4. Caminho mais curto entre dois nós (shortest path)
MATCH p = shortestPath(
  (origem:Pessoa {nome: "Ana"})-[:AMIGO_DE*]->(destino:Pessoa {nome: "Carlos"})
)
RETURN p, length(p) AS graus_de_separacao;

// 5. Aggregation: filmes mais populares por atores em comum
MATCH (p:Pessoa)-[:ATUOU_EM]->(f:Filme)
RETURN f.titulo, count(p) AS total_atores
ORDER BY total_atores DESC LIMIT 10;

SQL vs Cypher: amigos de amigos

Em SQL, encontrar amigos de amigos exige um self-JOIN: SELECT DISTINCT a2.amigo_id FROM amizades a1 JOIN amizades a2 ON a1.amigo_id = a2.pessoa_id WHERE a1.pessoa_id = 5. Para 5 níveis de profundidade, são 5 JOINs aninhados — performance cai exponencialmente. Em Cypher: MATCH (:Pessoa)-[:AMIGO_DE*5]->(dest) RETURN dest — O custo é proporcional ao grafo visitado, não ao banco inteiro.

Casos de Uso

Use Neo4j quando...

  • Redes sociais — sugestões de amigos, propagação de conteúdo
  • Detecção de fraude — padrões suspeitos em N graus de separação
  • Sistemas de recomendação — "quem comprou X também comprou Y"
  • Knowledge graphs — ontologias, grafos de conhecimento corporativo
  • Grafo de dependências — pacotes npm, módulos, microserviços
  • Roteamento e logística — menor caminho em redes complexas
  • IAM — hierarquias de permissões e roles complexas

Evite Neo4j quando...

  • Dados são essencialmente tabulares e relações são simples
  • Queries são principalmente por chave simples (use chave-valor)
  • Volume de escrita é extremamente alto e contínuo (use Cassandra)
  • Transações ACID complexas envolvendo muitas entidades
  • A equipe não tem familiaridade com modelagem de grafos

Avançado: Algoritmos de Grafos e GDS

Graph Data Science (GDS) Library

O Neo4j oferece a biblioteca Graph Data Science (GDS) com mais de 60 algoritmos de grafos prontos para uso em produção. Os algoritmos são categorizados em quatro grupos principais:

Centralidade

Identificam os nós mais influentes no grafo. PageRank (ranking de influência como o Google), Betweenness Centrality (nós que conectam comunidades), Degree Centrality (nós com mais conexões).

Detecção de Comunidades

Encontram clusters naturais no grafo. Louvain (modularity optimization — rápido e escalável), Label Propagation (comunidades por propagação de rótulos), Weakly Connected Components (subgrafos conectados).

Caminhos e Traversal

Dijkstra e A* para menor caminho ponderado, All Pairs Shortest Path para distâncias entre todos os nós, Minimum Spanning Tree para redes de infraestrutura.

Similaridade

Medem proximidade entre nós baseada nas conexões. Node Similarity (Jaccard/Overlap para recomendações), K-Nearest Neighbors para sistemas de recomendação baseados em grafos.

gds-pagerank.cypher
// Projetar um subgrafo em memória para análise
CALL gds.graph.project(
  'rede-social',
  'Pessoa',
  'SEGUE'
);

// Executar PageRank — identificar as pessoas mais influentes
CALL gds.pageRank.stream('rede-social',  {
  maxIterations: 20,
  dampingFactor: 0.85
})
YIELD nodeId, score
RETURN gds.util.asNode(nodeId).nome AS pessoa, score
ORDER BY score DESC LIMIT 10;

// Detecção de comunidades com Louvain
CALL gds.louvain.stream('rede-social')
YIELD nodeId, communityId
RETURN communityId, count(*) AS tamanho
ORDER BY tamanho DESC LIMIT 5;

Índices e Performance

Para queries eficientes em grandes grafos, o Neo4j oferece três tipos de índice:

indices-neo4j.cypher
// Índice de propriedade — busca por valor exato ou range
CREATE INDEX pessoa_nome FOR (p:Pessoa) ON (p.nome);

// Índice composto — queries com múltiplos filtros
CREATE INDEX filme_titulo_ano FOR (f:Filme) ON (f.titulo, f.ano);

// Full-text index — busca por texto livre em propriedades
CREATE FULLTEXT INDEX busca_filmes
  FOR (f:Filme) ON EACH [f.titulo, f.sinopse];

CALL db.index.fulltext.queryNodes("busca_filmes", "inteligência artificial")
YIELD node, score
RETURN node.titulo, score ORDER BY score DESC;

Supernós (Dense Nodes): nós com centenas de milhares de relacionamentos (ex.: uma celebridade com 10M de seguidores) criam gargalos de performance — toda query que passa por eles varre a lista gigante de relacionamentos. Solução: use relationship type indexing ou divida o supernó em subgrupos por critério (ex.: seguidores por região).

Ecossistema Neo4j

Neo4j Aura

Banco de grafos gerenciado na nuvem (DBaaS). Tier gratuito disponível com 200k nós e 400k relacionamentos. Sem necessidade de gerenciar cluster ou infraestrutura.

Neo4j Bloom

Interface visual de exploração de grafos. Permite navegar, filtrar e visualizar o grafo sem escrever Cypher — ideal para analistas de negócio e stakeholders não técnicos.

Drivers Oficiais

Drivers nativos para Java, Python, JavaScript, Go e .NET. Suportam Bolt Protocol (protocolo binário eficiente), connection pooling, retry automático e transações assíncronas.

Spring Data Neo4j

Integração Spring Boot com mapeamento de entidades via @Node, @Relationship e repositories. Queries derivadas de nome de método e @Query para Cypher customizado.

LangChain + Neo4j

Integração com LLMs para GraphRAG (Retrieval-Augmented Generation sobre grafos de conhecimento). Permite que modelos de linguagem consultem grafos para enriquecer respostas com dados estruturados relacionais.

APOC Library

Awesome Procedures on Cypher: mais de 450 funções e procedures para manipulação de dados, integração com sistemas externos, importação de CSV/JSON, triggers, e utilitários de grafo avançados.
