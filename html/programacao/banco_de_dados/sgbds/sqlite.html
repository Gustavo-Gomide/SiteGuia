SQLite

SQLite — Pequeno, Rápido e Sem Servidor

SQLite — O Banco Más Implantado do Mundo

O banco de dados mais usado — em cada celular, navegador e Smart TV.

O SQLite é uma biblioteca C que lida diretamente com um arquivo de banco de dados no disco — sem servidor, sem configuração, sem instalação.

O que é SQLite?

O SQLite não é um banco de dados cliente-servidor como MySQL ou PostgreSQL. Ele é uma biblioteca C que lê e escreve diretamente em um único arquivo no disco, sem servidor, sem configuração e sem instalação. Pense nele como um mecanismo de persistência embutido diretamente na sua aplicação — como um arquivo texto, mas com SQL completo.

Segundo a documentação oficial: "SQLite is probably the most widely deployed and used database engine in the world." Está presente em cada Android, iPhone, Chrome, Windows e praticamente toda Smart TV.

O SQLite foi criado por D. Richard Hipp em 2000 para um projet de controle de navios da Marinha dos EUA que não podia depender de um servidor de banco de dados externo.

Arquitetura Serverless

A diferença fundamental do SQLite para os outros SGBDs é que não existe um processo separado rodando em segundo plano. A "conexão" com o banco é apenas abrir um arquivo — como um fopen().

Sem Instalação

Não existe "instalar o SQLite" ou gerenciar um serviço. Você inclui a biblioteca no projeto (Python já inclui sqlite3 na stdlib) e o banco cria o arquivo automaticamente ao abrir a conexão.

Arquivo Único

O banco inteiro (tabelas, índices, dados, metadados) reside em um único arquivo .db ou .sqlite. Fazer backup é só copiar o arquivo. Migrar é só mover o arquivo.

Cross-platform

Um arquivo SQLite criado no Windows pode ser aberto no macOS, Linux, Android ou iOS sem nenhuma conversão. O formato binário é portátil e amplamente documentado como padrão aberto.

Zero-configuration

Não há arquivos de configuração, parâmetros de conexão, usuários ou senhas a gerenciar. A segurança de acesso é delegada ao sistema de arquivos do SO quem pode ler o arquivo, pode ler o banco.

Tipagem Dinâmica — Type Affinity

O SQLite é famoso (e às vezes criticado) por não impor tipos estritamente. Em vez de tipos fixos, usa um sistema de afinidade de tipos o banco prefere um tipo mas aceita outros sem erro. Versões recentes introduziram tabelas STRICT para quem prefere rigor total.

SQL
-- Comportamento padrão: tipagem flexível
CREATE TABLE exemplo (
    id    INTEGER PRIMARY KEY,
    valor INTEGER
);

-- SQLite aceita texto numa coluna inteira sem erro!
INSERT INTO exemplo (valor) VALUES ('Texto num campo Inteiro');
SELECT * FROM exemplo;
-- Resultado: 1 | 'Texto num campo Inteiro'

-- Tabela STRICT: tipos são impostos (SQLite 3.37+)
CREATE TABLE strict_demo (
    id    INTEGER PRIMARY KEY,
    valor INTEGER
) STRICT;

-- Agora esta linha causa erro:
INSERT INTO strict_demo (valor) VALUES ('Texto');
-- Runtime error: datatype mismatch

Recomendação: para novos projetos, use tabelas STRICT (SQLite 3.37+). Isso elimina bugs sutis de tipo que são difíceis de detectar em testes mas aparecem em produção com dados inesperados.

Uso com Python

Python inclui o módulo sqlite3 na biblioteca padrão sem nenhum pacote extra a instalar.

Python
import sqlite3

# Conexão  cria o arquivo se não existir
conn = sqlite3.connect("meu_banco.db")
cursor = conn.cursor()

# Criar tabela
cursor.execute("""\n  CREATE TABLE IF NOT EXISTS usuarios (\n    id    INTEGER PRIMARY KEY AUTOINCREMENT,\n    nome  TEXT NOT NULL,\n    email TEXT UNIQUE NOT NULL\n  )\n""")

# Inserir  parameterized query é segura contra SQL Injection
cursor.execute(
    "INSERT INTO usuarios (nome, email) VALUES (?, ?)",
    ("Alice", "alice@exemplo.com")
)
conn.commit()

# Consultar
cursor.execute("SELECT * FROM usuarios")
for row in cursor.fetchall():
    print(row)

# Banco in-memory  ideal para testes: não persiste em disco
conn_mem = sqlite3.connect(":memory:")

# Context manager: commit automático ou rollback em exceção
with sqlite3.connect("meu_banco.db") as con:
    con.execute("UPDATE usuarios SET nome = ? WHERE id = ?", ("Alice S.", 1))

conn.close()

Quando Usar SQLite

Mobile (Android / iOS)

É o padrão nativo para persistência local em aplicativos móveis. O Android SDK inclui APIs de alto nível sobre SQLite, e o iOS usa SQLite internamente no Core Data.

Desktop e IoT

Aplicativos desktop (Electron, wxPython, Qt) e dispositivos embarcados (IoT) usam SQLite para configurações e dados locais. Roda em dispositivos com menos de 1 MB de RAM disponível.

Testes Automatizados

O banco in-memory (:memory:) é perfeito para testes: não persiste estado entre execuções, inicia em microssegundos e não requer nenhum servidor ou configuração Docker.

Prototipagem e Sites Pequenos

Comece com SQLite para não perder tempo configurando MySQL/Docker nas primeiras semanas do projeto. Websites com menos de ~100 k acessos/dia e baixa concorrência de escrita funcionam muito bem.

Limitação crítica — concorrência de escrita: O SQLite permite múltiplas leituras simultâneas (WAL mode), mas apenas uma escrita por vez. Se dois processos tentarem gravar simultaneamente, um aguardará o desbloqueamento. Para aplicações de alto volume de escrita ou múltiplos servidores, use PostgreSQL ou MySQL.

SQLite vs SGBDs Tradicionais

Critério SQLite MySQL / PostgreSQL
Configuração Zero — apenas um arquivo Servidor, usuários, portas, conexões
Concorrência de escrita Uma por vez (WAL melhora, mas não elimina) Múltiplas escritas simultâneas
Escala Até ~100 GB / baixa concorrência Petabytes / alta concorrência
Backup Copiar o arquivo .db pg_dump, mysqldump, backups lógicos/físicos
Acesso remoto Não — arquivo local somente Sim TCP/IP, rede, cloud
Uso ideal Mobile, desktop, IoT, testes, protótipos APIs, SaaS, sistemas multiusuário, produção
