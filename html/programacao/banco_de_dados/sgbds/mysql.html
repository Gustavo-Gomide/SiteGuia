MySQL: O SGBD Mais Usado do Mundo

Um banco relacional popular em aplicações web, com forte ecossistema e operação simples.

O MySQL se consolidou como banco relacional de referência para a web por combinar facilidade operacional, desempenho e ampla adoção no ecossistema LAMP.

O que é MySQL?

O MySQL é um sistema gerenciador de banco de dados relacional open-source criado em 1995, com foco histórico em aplicações web e deploy simples. Em termos práticos, ele oferece uma base sólida para armazenamento transacional, consultas SQL e integração com praticamente qualquer stack de desenvolvimento.

Hoje mantido pela Oracle (após aquisição da Sun Microsystems em 2010), o MySQL mantém edição comunitária robusta e permanece como padrão em hospedagens compartilhadas e aplicações legadas de grande escala.

O nome "MySQL" combina "My" (nome da filha de Michael Widenius) com SQL; posteriormente, Widenius criou o fork MariaDB.

História: Da Suécia ao Oracle

Linha do Tempo

  • 1994–1995: Desenvolvimento inicial por Michael Widenius e David Axmark na TcX DataKonsult AB.
  • 1995: MySQL 1.0 lançado — focado em velocidade máxima para aplicações web.
  • 2000: MySQL AB fundada; código tornado GPL. Início da explosão da pilha LAMP.
  • 2001–2004: Adoção massiva com WordPress, phpBB, Joomla — todos usam MySQL.
  • 2008: Sun Microsystems adquire a MySQL AB por US$ 1 bilhão.
  • 2010: Oracle adquire a Sun — e com ela o MySQL. Monty preocupado faz fork: MariaDB.
  • 2013: MySQL 5.6 — melhorias significativas em InnoDB, replicação e full-text.
  • 2018: MySQL 8.0 — window functions, CTEs, roles, JSON aprimorado, UTF8MB4 como padrão.

Storage Engines — A Arquitetura Plugável

Uma característica única do MySQL é sua arquitetura de storage engines plugáveis: cada tabela pode usar um mecanismo de armazenamento diferente, cada um otimizado para um tipo de carga.

MyISAM Não Não Por tabela Leitura intensiva, dados legados

Use InnoDB em praticamente todos os casos. MyISAM era popular antes do MySQL 5.5, mas perdeu sentido após o InnoDB se tornar padrão. O único cenário onde MyISAM ainda aparece é em sistemas legados muito antigos.

Características Principais

Leituras Rápidas

Historicamente otimizado para cargas de leitura intensa. Sites de conteúdo como blogs e portais de notícias se beneficiam muito da velocidade de SELECT simples com InnoDB.

Suporte Universal de Hospedagem

Praticamente todo plano de hospedagem compartilhada no mundo inclui MySQL. É a escolha obrigatória quando você precisa de banco em ambiente de hosting popular.

Replicação Simples

Configurar replicação primary-replica no MySQL é notavelmente simples. Binary logs e GTID (Global Transaction Identifiers) permitem escalar leitura horizontalmente com pouco esforço.

MySQL Workbench

Ferramenta visual oficial gratuita para modelagem ER, administração do servidor, desenvolvimento de queries e monitoramento de performance. Uma das melhores IDEs de banco visuais disponíveis.

Ecossistema Massivo

Drivers nativos para todas as linguagens populares: PHP (mysqli/PDO), Python (mysql-connector), Java (JDBC), Node.js, .NET, Go, Ruby. Documentação extensa e ativa comunidade.

JSON desde MySQL 5.7

Suporte nativo a colunas JSON com funções como JSON_EXTRACT(), JSON_SET() e índices em expressões JSON geradas (virtual columns).

Código Típico MySQL

Exemplos que mostram o MySQL em ação — desde criação de tabelas até consultas avançadas.

SQL
-- Criar tabela com InnoDB (explícito, mas é o padrão)
CREATE TABLE usuarios (
    id       INT PRIMARY KEY AUTO_INCREMENT,
    nome     VARCHAR(100) NOT NULL,
    email    VARCHAR(255) UNIQUE NOT NULL,
    criado_em DATETIME DEFAULT NOW(),
    ativo    TINYINT(1) DEFAULT 1
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci;

-- Criar tabela de pedidos com FK
CREATE TABLE pedidos (
    id          INT PRIMARY KEY AUTO_INCREMENT,
    usuario_id  INT NOT NULL,
    total       DECIMAL(10,2) NOT NULL,
    status      ENUM('pendente','pago','cancelado') DEFAULT 'pendente',
    criado_em   DATETIME DEFAULT NOW(),
    CONSTRAINT fk_usuario FOREIGN KEY (usuario_id)
        REFERENCES usuarios(id) ON DELETE CASCADE
) ENGINE=InnoDB;

-- INSERT com múltiplas linhas (otimizado)
INSERT INTO usuarios (nome, email) VALUES
('Alice Silva', 'alice@exemplo.com'),
('Bob Santos', 'bob@exemplo.com'),
('Clara Mendes', 'clara@exemplo.com');

-- SELECT com JOIN e filtro
SELECT
    u.nome,
    u.email,
    COUNT(p.id) AS total_pedidos,
    SUM(p.total) AS valor_total
FROM usuarios u
LEFT JOIN pedidos p ON p.usuario_id = u.id AND p.status = 'pago'
WHERE u.ativo = 1
GROUP BY u.id, u.nome, u.email
ORDER BY valor_total DESC
LIMIT 10;

Exemplo de stored procedure com tratamento de saída:

SQL
DELIMITER $$

CREATE PROCEDURE buscar_usuario_por_email(
    IN  p_email   VARCHAR(255),
    OUT p_nome    VARCHAR(100),
    OUT p_encontrado TINYINT
)
BEGIN
    DECLARE v_count INT DEFAULT 0;

    SELECT COUNT(*) INTO v_count
    FROM usuarios
    WHERE email = p_email AND ativo = 1;

    IF v_count > 0 THEN
        SELECT nome INTO p_nome
        FROM usuarios
        WHERE email = p_email;
        SET p_encontrado = 1;
    ELSE
        SET p_nome = NULL;
        SET p_encontrado = 0;
    END IF;
END$$

DELIMITER ;

-- Chamar a procedure
CALL buscar_usuario_por_email('alice@exemplo.com', @nome, @encontrado);
SELECT @nome, @encontrado;

Replicação — Escalando Leituras

A replicação do MySQL foi um dos fatores que impulsionou sua adoção em larga escala. A ideia é simples: um servidor primary (antes chamado de master) grava todos os dados, e um ou mais servidores replica (antes chamados de slave) recebem cópias em tempo real para servir leituras.

Binary Log — O Mecanismo Central

Todo dado gravado no primary é registrado no binary log (binlog). As réplicas se conectam ao primary, leem esse log e aplicam as mesmas operações. Isso garante consistência eventual entre todos os servidores.

GTID — Global Transaction Identifiers

Introduzido no MySQL 5.6, o GTID atribui um identificador único a cada transação no primary. Isso simplifica drasticamente o failover: a réplica sabe exatamente qual transação aplicar a seguir, sem depender de posições de byte no binlog.

SQL
-- No servidor PRIMARY: verificar status do binlog
SHOW MASTER STATUS;

-- Na RÉPLICA: configurar conexão com o primary
CHANGE REPLICATION SOURCE TO
    SOURCE_HOST = '192.168.1.10',
    SOURCE_USER = 'replicacao',
    SOURCE_PASSWORD = 'senha_segura',
    SOURCE_AUTO_POSITION = 1;  -- Usa GTID automaticamente

START REPLICA;

-- Verificar status da replicação
SHOW REPLICA STATUS\G

Replicação no MySQL é assíncrona por padrão — a réplica pode ficar alguns milissegundos (ou segundos em sobrecarga) atrás do primary. Para leitura de dados críticos (ex: saldo bancário), sempre leia do primary, não das réplicas.

Vale a Pena? Estado Atual do MySQL

A pergunta que todo desenvolvedor faz: com a Oracle no controle, ainda vale a pena usar MySQL? A resposta honesta é: depende do contexto.

Prefira MySQL quando...

  • O projeto usa WordPress, Drupal, Joomla ou similar
  • Hospedagem compartilhada é um requisito
  • A equipe já tem expertise consolidada em MySQL
  • Precisa do MySQL Workbench como ferramenta visual
  • Integração com aplicações PHP legadas
  • Requisito de compatibilidade com sistemas existentes

Considere PostgreSQL quando...

  • Queries complexas com múltiplos JOINs e window functions
  • Integridade de dados é crítica (financeiro, saúde)
  • Precisa de JSONB com indexação real
  • Conformidade total com padrões SQL é necessária
  • Quer uma licença sem qualquer dependência da Oracle

MySQL Community vs Enterprise: A versão Community (GPL) é gratuita e suficiente para a maioria dos projetos. A versão Enterprise (paga) adiciona ferramentas de backup, monitoramento avançado e suporte oficial da Oracle — relevante apenas para empresas com contratos corporativos.
