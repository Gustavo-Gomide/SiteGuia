MariaDB — O Irmão Livre do MySQL

Open-source de verdade, alto desempenho e compatível com MySQL.

O MariaDB é um fork do MySQL criado pelo seu próprio inventor para garantir que o banco permanecesse livre para sempre — e hoje é padrão em distribuições Linux como Debian e Fedora.

O que é MariaDB?

O MariaDB é um sistema gerenciador de banco de dados relacional (SGBDR) open-source, criado em 2009 por Michael Widenius — o mesmo criador do MySQL — como resposta à aquisição do MySQL pela Oracle. É um substituto direto do MySQL: usa a mesma porta (3306), os mesmos drivers e a mesma sintaxe SQL.

Segundo a MariaDB Foundation (2024): "MariaDB is one of the most popular database servers in the world. It’s made by the original developers of MySQL and guaranteed to stay open source."

O nome é homenagem à Maria, filha mais nova de Monty Widenius — assim como MySQL foi batizado em honor à filha mais velha, My.

Origem — Uma História com Três Nomes

Para entender o MariaDB, precisamos voltar à história do MySQL:

Linha do Tempo

Ano Evento
1994 Michael "Monty" Widenius e David Axmark criam o MySQL na Suécia. O nome "MySQL" é combinação de "My" (nome da filha de Monty) + SQL.
2001 Fundação da MySQL AB para comercializar o banco. O projeto cresce e se torna o banco web mais usado do mundo.
2008 Sun Microsystems adquire a MySQL AB por US$ 1 bilhão. Monty já demonstra preocupação com o futuro Open-Source.
2010 Oracle adquire a Sun Microsystems. Monty teme que o MySQL seja tornando pago ou descontinuado e faz o fork: MariaDB, batizado em homenagem à filha mais nova Maria.
2012 MariaDB Foundation fundada para garantir independência. Debian, Ubuntu e Fedora adotam MariaDB como padrão em vez do MySQL.
2019+ MariaDB Corporation lança o MariaDB SkySQL (Cloud) e o ColumnStore para analytics. Versões 10.x adicionam features ausentes no MySQL.

Curiosidade: O nome das filhas de Monty Widenius virou um meme de banco de dados. MySQL = My (filha mais velha), MariaDB = Maria (filha caçula). Existe também um motor de armazenamento chamado Aria — que também é um nome de família.

Drop-in Replacement — Substituição Transparente

A característica central do MariaDB é ser um substituto direto (drop-in replacement) do MySQL. Isso significa que você pode trocar um pelo outro sem alterar seu código de aplicação.

Mesma Porta

Ambos usam a porta 3306 por padrão. Nenhuma mudança de configuração de rede necessária.

Mesmos Drivers

Os conectores JDBC, ODBC, PHP PDO e Node.js escritos para MySQL funcionam com MariaDB sem modificação.

Mesmo SQL

Todos os comandos DDL, DML, DCL que você escreve para MySQL compilam e executam no MariaDB.

Arquivos Compatíveis

É possível fazer dump com mysqldump num servidor MySQL e restaurar no MariaDB (e vice-versa nas versões alinhadas).

Atenção: A compatibilidade é alta nas versões alinhadas, mas o MariaDB 10.x divergiu do MySQL 8.x em algumas áreas (como o sistema de roles e JSON). Para migrações entre versões muito diferentes, teste antes em ambiente de homologação.

Embora compatíveis, o MariaDB acumulou recursos exclusivos ao longo dos anos:

Storage Engines Exclusivos

Aria Engine

O Aria é o substituto moderno do MyISAM. Ao contrário do MyISAM, o Aria é crash-safe: se o servidor desligar abruptamente, ele não deixará tabelas corrompidas. É ideal para tabelas temporárias internas do próprio MariaDB.

Spider Engine — Sharding Nativo

O Spider permite criar tabelas particionadas em múltiplos servidores remotos. Do ponto de vista da aplicação, parece uma tabela normal, mas os dados estão espalhados por vários nós — um recurso equivalente ao sharding do MongoDB.

SQL
-- Criar tabela Spider que aponta para servidor remoto
CREATE TABLE pedidos (
  id INT PRIMARY KEY,
  valor DECIMAL(10,2)
) ENGINE=Spider
  COMMENT='wrapper "mysql://usuario:senha@servidor-remoto:3306/loja", table "pedidos"';

ColumnStore — Analytics Colunar

O ColumnStore armazena dados por coluna em vez de por linha. Isso é ideal para consultas analíticas (OLAP) que leem poucas colunas em bilhões de linhas. É o mesmo princípio do Amazon Redshift e Google BigQuery.

Temporal Tables — Versionamento de Dados

Um recurso avançado do MariaDB 10.3+ é suporte a System-Versioned Tables, definidas no padrão SQL:2011. Elas guardam automaticamente o histórico de todas as alterações em uma linha, permitindo consultas no passado.

SQL
-- Criar tabela com versionamento automático
CREATE TABLE funcionarios (
  id      INT PRIMARY KEY AUTO_INCREMENT,
  nome    VARCHAR(100),
  salario DECIMAL(10,2)
) WITH SYSTEM VERSIONING;

-- Atualizar dados normalmente
UPDATE funcionarios SET salario = 8500.00 WHERE id = 1;
UPDATE funcionarios SET salario = 9200.00 WHERE id = 1;

-- Consultar histórico completo da linha
SELECT nome, salario, ROW_START, ROW_END
FROM funcionarios FOR SYSTEM_TIME ALL
WHERE id = 1;

-- Consultar como estava em uma data específica
SELECT * FROM funcionarios
FOR SYSTEM_TIME AS OF '2025-01-01 12:00:00'
WHERE id = 1;

Caso de Uso: Auditoria de dados financeiros, histórico de preços, registros médicos e qualquer tabela onde você precise responder "qual era o valor disso em uma data X?". Sem necessidade de criar tabelas de log separadas.

Galera Cluster — Alta Disponibilidade

O Galera Cluster é uma solução de replicação síncrona multi-master integrada ao MariaDB. Ao contrário da replicação tradicional Master-Slave (onde só um nó aceita escrita), no Galera todos os nós aceitam leituras e escritas.

Como funciona

  • Cada nó tem uma cópia completa dos dados.
  • Uma transação só confirma (COMMIT) quando todos os nós a aceitam (replicação síncrona).
  • Se um nó cair, os demais continuam operando automaticamente.
  • O nó que caiu se re-sincroniza automaticamente ao voltar (State Snapshot Transfer).

Galera vs replicação MySQL padrão: A replicação padrão é assíncrona — pode haver lag entre master e slaves, causando perda de dados em falhas. O Galera é síncrono: zero perda de dados garantida, com o custo de maior latência de escrita.

Exemplos de Código

SQL Básico — 100% compatível com MySQL

SQL
-- SELECT da versão
SELECT VERSION(); -- Retorna ex: 10.11.7-MariaDB

-- CRIAR banco e tabela
CREATE DATABASE loja CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;
USE loja;

CREATE TABLE produtos (
  id          INT UNSIGNED PRIMARY KEY AUTO_INCREMENT,
  nome        VARCHAR(200) NOT NULL,
  preco       DECIMAL(10,2) NOT NULL DEFAULT 0.00,
  estoque     INT NOT NULL DEFAULT 0,
  criado_em   DATETIME DEFAULT CURRENT_TIMESTAMP,
  atualizado  DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP
) ENGINE=InnoDB;

-- DML padrão
INSERT INTO produtos (nome, preco, estoque)
VALUES ('Teclado Mecânico', 349.90, 50),
       ('Mouse Gamer', 199.90, 120);

SELECT * FROM produtos WHERE preco < 300 ORDER BY preco DESC;

UPDATE produtos SET estoque = estoque - 1 WHERE id = 1;

DELETE FROM produtos WHERE estoque = 0;

Recurso exclusivo — Sequence Objects

SQL
-- MariaDB possui objetos SEQUENCE nativos (MySQL não tem)
CREATE SEQUENCE seq_pedido
  START WITH 10000
  INCREMENT BY 1
  MINVALUE 10000
  MAXVALUE 9999999
  NOCYCLE;

-- Usar a sequência
SELECT NEXT VALUE FOR seq_pedido;  -- Retorna 10000
SELECT NEXT VALUE FOR seq_pedido;  -- Retorna 10001

-- Inserir com sequência
INSERT INTO pedidos (id, descricao)
VALUES (NEXT VALUE FOR seq_pedido, 'Pedido do cliente X');

Instalação com Docker

A forma mais rápida de subir um MariaDB para desenvolvimento:

Shell / YAML
# Subir container MariaDB
docker run -d \
  --name mariadb-dev \
  -e MARIADB_ROOT_PASSWORD=senha123 \
  -e MARIADB_DATABASE=meubanco \
  -e MARIADB_USER=meuusuario \
  -e MARIADB_PASSWORD=minhasenha \
  -p 3306:3306 \
  mariadb:11

# Conectar via CLI
docker exec -it mariadb-dev mariadb -u meuusuario -p meubanco

# docker-compose.yml para projetos
version: '3.8'
services:
  db:
    image: mariadb:11
    restart: unless-stopped
    environment:
      MARIADB_ROOT_PASSWORD: root_secreto
      MARIADB_DATABASE: app_db
      MARIADB_USER: app_user
      MARIADB_PASSWORD: app_senha
    ports:
      - "3306:3306"
    volumes:
      - mariadb_data:/var/lib/mysql

volumes:
  mariadb_data:

Use MariaDB quando...

Você já tem um projeto MySQL e quer mais liberdade open-source, precisa do Galera Cluster integrado, ou quer features do SQL:2011 como Temporal Tables sem pagar licença Enterprise.

Use MySQL quando...

Seu stack já é MySQL 8.x e usa recursos novos como o novo sistema de roles ou JSON Path, e a equipe tem expertise e certificações MySQL específicas.

Use PostgreSQL quando...

Você precisa de JSONB avançado, extensões como PostGIS, conformidade máxima com padrões SQL, ou lida com cargas analíticas pesadas sem engines separadas.
