Segurança em Banco de Dados

Segurança — Defesa em Profundidade

Protegendo o bem mais valioso da sua empresa: os dados dos seus usuários.

Em 2021, um vazamento atingiu 530 milhões de contas do Facebook. Em 2017, a Equifax expôs dados financeiros de 147 milhões de americanos. Em ambos os casos, a falha nao foi tecnologia insuficiente — foi configuracao negligente. Seguranca em banco de dados e uma mentalidade de Defesa em Profundidade: camadas sobrepostas onde a falha de uma nao compromete o sistema inteiro.

1. Defesa em Profundidade

O modelo de seguranca em camadas garante que um atacante precise superar multiplos obstaculos independentes para chegar aos dados. Cada camada assume que a anterior pode falhar.

O modelo das camadas: Rede (firewall, VPC privada) → Sistema Operacional (hardening, patches) → Banco de Dados (autenticacao, autorizacao) → Aplicacao (Prepared Statements, validacao) → Dados (criptografia em repouso e transito).

Camada de Rede

O banco de dados nunca deve ser acessivel diretamente pela internet. Coloque-o em uma VPC privada ou rede interna. Aceite conexoes apenas da aplicacao (IP especifico ou security group).

Camada de Autenticacao

Cada usuario deve ter credenciais proprias. Proibido compartilhar o usuario root entre times. Use senhas longas e rotacione-as regularmente com ferramentas como HashiCorp Vault.

Camada de Autorizacao

Principio do Menor Privilégio: cada usuario tem acesso apenas ao que precisa. O usuario da API web nao pode fazer DROP TABLE. O usuario de relatorios nao pode fazer INSERT.

2. Autenticacao e Autorizacao

Autenticacao responde "quem voce e?" e Autorizacao responde "o que voce pode fazer?". Sao camadas distintas e independentes — passar pela autenticacao nao garante acesso a tudo.

postgresql — criar roles e conceder permissoes minimas
-- Criar role de aplicacao (sem login direto)
CREATE ROLE app_role;
GRANT SELECT, INSERT, UPDATE ON ALL TABLES IN SCHEMA public TO app_role;

-- Criar usuario de aplicacao vinculado a role
CREATE USER app_api WITH PASSWORD 's3nh@_F0rt3_R0t4c1on4da';
GRANT app_role TO app_api;

-- Criar usuario de relatorios (somente leitura)
CREATE USER app_reports WITH PASSWORD 'outr4_S3nh4';
GRANT SELECT ON ALL TABLES IN SCHEMA public TO app_reports;

-- REVOGAR o que nao deve ser permitido
REVOKE DELETE ON pedidos FROM app_api;
REVOKE INSERT, UPDATE, DELETE ON ALL TABLES IN SCHEMA public FROM app_reports;
postgresql — Row-Level Security (RLS): vendedor ve apenas seus pedidos
-- Habilitar RLS na tabela
ALTER TABLE pedidos ENABLE ROW LEVEL SECURITY;

-- Criar politica: cada vendedor ve apenas os proprios pedidos
CREATE POLICY ver_proprios_pedidos ON pedidos
  FOR SELECT
  USING (vendedor_id = current_setting('myapp.current_user_id')::int);

-- A aplicacao define o contexto antes da query
SET LOCAL myapp.current_user_id = '42';
SELECT * FROM pedidos;  -- retorna apenas pedidos do vendedor 42

3. SQL Injection — A Ameaca #1

SQL Injection ocorre quando input do usuario e concatenado diretamente em uma query SQL. E a vulnerabilidade mais comum e mais devastadora contra bancos de dados — presente no OWASP Top 10 desde sua criacao. A defesa e simples e absoluta: Prepared Statements.

Vulneravel — Concatenacao Direta

java — codigo vulneravel
// Input do atacante: ' OR '1'='1' --
String query = "SELECT * FROM users"
  + " WHERE login = '" + userInput + "'";

// Query resultante no banco:
// SELECT * FROM users
// WHERE login = '' OR '1'='1' --'
// Retorna TODOS os usuarios. Login ignorado.

Seguro — Prepared Statements

java — codigo seguro com JDBC
// O placeholder ? e compilado ANTES do input
String sql = "SELECT * FROM users WHERE login = ?";
PreparedStatement ps = conn.prepareStatement(sql);
ps.setString(1, userInput);

// Se o atacante passar: ' OR '1'='1' --
// O banco busca literalmente esse texto como login.
// Nao e interpretado como SQL. Ataque neutralizado.

Regra absoluta: Nunca concatene strings para montar queries com input do usuario. ORMs modernos (Hibernate, SQLAlchemy, ActiveRecord, Prisma) usam Prepared Statements por padrao — mas atencao a queries nativas (raw(), execute()) que podem contornar essa protecao.

4. Criptografia — Em Transito e Em Repouso

Se o atacante bypassar todas as camadas anteriores e roubar o hardware fisico ou interceptar a rede — o que ele consegue ler? Com criptografia correta: nada util.

Em Transito — TLS/SSL

Toda comunicacao entre aplicacao e banco deve usar TLS 1.2+. Protege contra ataques man-in-the-middle em redes corporativas ou interceptacao de trafego.

pg_hba.conf — exigir TLS para conexoes externas
# Conexoes locais: sem TLS (loopback seguro)
local   all  all               peer
host    all  all  127.0.0.1/32  md5

# Conexoes externas: TLS obrigatorio
hostssl all  all  0.0.0.0/0     scram-sha-256

Em Repouso — TDE (Transparent Data Encryption)

O arquivo fisico do banco em disco fica criptografado com uma chave mestre. Um HD roubado e inutil sem a chave. Disponivel de forma nativa no SQL Server, Oracle e em cloud providers.

sql-server — habilitar TDE
-- Criar chave master e certificado
CREATE MASTER KEY ENCRYPTION BY PASSWORD = 'S3nh4_D0_Cofre!';
CREATE CERTIFICATE TDE_Cert WITH SUBJECT = 'TDE Certificate';

-- Criar chave de criptografia do banco vinculada ao certificado
CREATE DATABASE ENCRYPTION KEY
  WITH ALGORITHM = AES_256
  ENCRYPTION BY SERVER CERTIFICATE TDE_Cert;

-- Ativar criptografia
ALTER DATABASE loja SET ENCRYPTION ON;

Colunas Sensiveis — Criptografia a Nivel de Campo

Para dados ultra-sensiveis (CPF, cartao, senha), criptografe a nivel de coluna na aplicacao. Mesmo um DBA com acesso total ao banco nao consegue ler o valor sem a chave da aplicacao.

postgresql — criptografia de coluna com pgcrypto
-- Extensao de criptografia do PostgreSQL
CREATE EXTENSION pgcrypto;

-- Inserir dado criptografado
INSERT INTO clientes (nome, cpf_enc) VALUES (
  'Maria Silva',
  pgp_sym_encrypt('123.456.789-00', 'chave_secreta_app')
);

-- Descriptografar na leitura
SELECT nome, pgp_sym_decrypt(cpf_enc, 'chave_secreta_app') AS cpf
FROM clientes;

5. Auditoria — Quem Fez o Que e Quando

Seguranca nao e apenas prevencao — e tambem deteccao e rastreabilidade. Auditoria registra todas as acoes criticas para que breaches sejam detectados rapidamente e investigacoes sejam possiveis. Regulamentacoes como LGPD e GDPR exigem logs de acesso a dados pessoais.

postgresql.conf — ativar logging de seguranca
# Logar todas as conexoes e desconexoes
log_connections    = on
log_disconnections = on

# Logar todos os comandos DDL (CREATE/DROP/ALTER)
log_statement = 'ddl'

# Logar queries que demoram mais de 1 segundo (possivel exfiltration scan)
log_min_duration_statement = 1000

# Logar o usuario e host de cada query
log_line_prefix = '%t [%p]: [%l-1] user=%u,db=%d,app=%a,client=%h '
pgaudit — auditoria granular por objeto
-- pgAudit: auditoria detalhada para conformidade (PCI-DSS, LGPD)
-- postgresql.conf:
-- shared_preload_libraries = 'pgaudit'
-- pgaudit.log = 'write, ddl'

-- Auditar acoes especificas em tabela sensivelm
SELECT pgaudit.set_config('write', TRUE, FALSE);

-- O log vai registrar: usuario, timestamp, query, objeto acessado, resultado
-- Formato: AUDIT: SESSION,1,1,WRITE,INSERT,TABLE,public.clientes,INSERT INTO...

6. Quando Aplicar Cada Controle

Prepared Statements

  • Toda query que recebe input do usuario
  • APIs REST e GraphQL com parametros
  • Forms de busca, login e cadastro
  • Queries nativas em ORMs
  • Sempre. Sem excecao.

TLS na Conexao

  • Aplicacao em servidor separado do banco
  • Banco em cloud (AWS RDS, Azure, GCP)
  • Equipes acessando banco remotamente
  • Qualquer dado pessoal ou financeiro (LGPD/PCI)

Row-Level Security

  • Aplicacoes multi-tenant (varios clientes no mesmo banco)
  • Sistemas onde vendedores/agentes so veem seus proprios dados
  • Portais de clientes com isolamento por usuario
  • Relatorios com restricao por departamento

Criptografia de Coluna e Desnecessaria Quando

  • Os dados nao sao pessoais nem sensiveis
  • O banco ja esta em rede totalmente isolada sem acesso externo
  • TDE (disco) ja cobre o requisito regulatorio
  • A performance e critica e o campo e indexado e buscado constantemente

Usuario Root na Aplicacao

  • Nunca em producao
  • Nunca em staging com dados reais
  • Nunca em containers sem segredo gerenciado
  • Nunca hardcoded em codigo-fonte ou .env no repositorio

Auditoria Pode Ser Reduzida Quando

  • Ambiente de desenvolvimento local sem dados reais
  • Bancos de teste com dados sinteticos anonimizados
  • Performance e critica e logs de query aumentam latencia
  • Mas: mantenha sempre log de DDL e de conexoes em todo ambiente

7. Topicos Avancados

pgAudit — Auditoria para Conformidade

Extensao oficial do PostgreSQL para auditoria granular. Gera logs estruturados compatíveis com PCI-DSS, HIPAA e LGPD. Registra objeto acessado, usuario, timestamp e resultado de cada operacao — diferente do log padrao que e orientado a sessao.

HashiCorp Vault — Gerenciamento de Segredos

Elimina credenciais de banco hardcoded em aplicacoes. O Vault gera credenciais temporarias (dynamic secrets) com TTL de minutos. A aplicacao solicita credenciais ao Vault a cada conexao — se o credencial vazar, ele ja expirou.

DAM — Database Activity Monitoring

Ferramentas como IBM Guardium e Imperva monitoram em tempo real todo o trafego SQL, detectam anomalias (exfiltracao em massa, acesso fora do horario) e geram alertas automaticos. Essencial em ambientes regulados com dados de saude ou financeiros.

LGPD e GDPR — Implicacoes para DBA

Ambas as leis exigem: pseudonimizacao de dados pessoais, direito ao esquecimento (DELETE auditavel), logs de acesso retidos por no minimo 5 anos, e criptografia para dados sensiveis. O DBA e co-responsavel pela conformidade.

Segredos nunca em codigo-fonte. Credenciais de banco em repositorios Git sao a causa #1 de breaches em startups. Use variaveis de ambiente com gestao externa (Vault, AWS Secrets Manager, Azure Key Vault) e nunca faca commit de arquivos .env de producao. Ferramentas como git-secrets e truffleHog escaneiam historico de commits em busca de segredos expostos.

8. Ecossistema e Ferramentas

pgAudit

Extensao para PostgreSQL que implementa auditoria de sessao e objeto conforme padroes PCI-DSS e HIPAA. Integra com sistemas de SIEM via syslog. Amplamente usado em producao em ambientes financeiros e de saude.

HashiCorp Vault

Gerenciador de segredos open-source lider de mercado. Suporta dynamic secrets para PostgreSQL, MySQL, MongoDB e outros. Rotacionamento automatico de senha, PKI integrada e politicas de acesso baseadas em identidade (RBAC/ABAC).

IBM Guardium / Imperva

Plataformas enterprise de Database Activity Monitoring (DAM). Monitoram trafego SQL em nivel de rede (sem agente no banco). Detectam exfiltracao, escalonamento de privilegio e acesso fora do baseline comportamental.

OWASP — Recursos de Seguranca

A OWASP publica o SQL Injection Prevention Cheat Sheet, o Database Security Cheat Sheet e o OWASP Top 10. Sao referencias gratuitas e atualizadas sobre as ameacas mais criticas e como mitiga-las.

fail2ban

Monitora logs do sistema e do banco de dados. Apos N tentativas de autenticacao falhas em X segundos, bloqueia automaticamente o IP do atacante via firewall. Eficaz contra ataques de forca bruta e credential stuffing.

AWS Secrets Manager / Azure Key Vault

Servicos gerenciados de cloud para armazenar e rotacionar credenciais de banco automaticamente. Integram-se com IAM roles para acesso sem senha hardcoded. Rotacionam a senha do banco sem downtime via Lambda trigger.
