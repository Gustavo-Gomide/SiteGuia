Plano de Execucao SQL

Plano de Execucao

O mapa que o banco usa para encontrar seus dados — aprenda a le-lo antes de otimizar qualquer coisa.

Voce escreve SQL declarativo: diz ao banco O QUE quer, nao COMO buscar. O otimizador gera um plano de execucao com a rota de menor custo estimado. Entender esse plano e o pre-requisito para qualquer otimizacao de performance — sem ele, voce esta adivinhar no escuro. Com EXPLAIN ANALYZE voce ve custo, linhas estimadas vs reais, tempo por no e onde o banco esta desperdicando I/O.

1. O Otimizador de Consultas

SQL e uma linguagem declarativa — voce descreve o resultado que quer, nao o algoritmo para obtê-lo. O otimizador e o componente do SGBD que converte essa descricao num plano de execucao fisico eficiente.

ciclo-do-otimizador.txt
Ciclo de vida de uma query no PostgreSQL:

  1. PARSER           ->  Verifica sintaxe, cria AST (arvore sintatica)
  2. ANALYZER         ->  Resolve nomes de tabelas e colunas
  3. REWRITER         ->  Expande views, aplica regras de rewrite
  4. PLANNER          ->  Gera multiplos planos candidatos
     |                    Estima custo de cada um usando:
     |                      - pg_statistic: histogramas, NDV (valores distintos)
     |                      - Fator de correlacao de colunas
     |                      - Parametros de custo de I/O e CPU
     v
  5. EXECUTOR         ->  Executa o plano de menor custo estimado

Estatisticas do Catálogo

O otimizador usa estatisticas armazenadas em pg_statistic (PostgreSQL) ou information_schema.TABLE_STATISTICS (MySQL) para estimar quantas linhas cada operacao vai processar. Estatisticas desatualizadas = estimativas erradas = planos ruins. Execute ANALYZE ou deixe o autovacuum fazê-lo automaticamente.

Planner de Custo

O custo nao e tempo em segundos — e uma unidade relativa de esforco (I/O + CPU). O PostgreSQL usa seq_page_cost = 1.0 como base. Um Index Scan tem custo de acesso aleatorio ao disco (random_page_cost = 4.0 por default). O otimizador escolhe o plano de menor custo total estimado.

Quando o Otimizador Erra

O otimizador pode errar quando: (1) estatisticas desatualizadas — estimativas de cardinalidade erradas; (2) correlacoes entre colunas ignoradas por padrao; (3) muitos JOINs — o espaco de planos cresce exponencialmente. Nesses casos, EXPLAIN ANALYZE mostra a diferenca entre linhas estimadas e linhas reais — um signal direto de problema estatistico.

2. O Comando EXPLAIN

explain-comandos.sql
-- PostgreSQL: plano estimado (nao executa a query)
EXPLAIN SELECT * FROM pedidos WHERE total > 100;

-- PostgreSQL: plano + execucao real (CUIDADO em producao para queries pesadas)
EXPLAIN ANALYZE SELECT * FROM pedidos WHERE total > 100;

-- PostgreSQL: formato JSON (para ferramentas como explain.depesz.com)
EXPLAIN (ANALYZE, FORMAT JSON, BUFFERS)
SELECT p.id, c.nome
FROM pedidos p JOIN clientes c ON p.cliente_id = c.id
WHERE p.data_criacao >= '2024-01-01';

-- MySQL: EXPLAIN + EXPLAIN ANALYZE (MySQL 8.0.18+)
EXPLAIN FORMAT=JSON SELECT * FROM pedidos WHERE total > 100;
EXPLAIN ANALYZE SELECT * FROM pedidos WHERE total > 100;

-- SQL Server: exibir plano de execucao textual
SET SHOWPLAN_ALL ON; GO
SELECT * FROM pedidos WHERE total > 100; GO
SET SHOWPLAN_ALL OFF; GO

EXPLAIN vs EXPLAIN ANALYZE: EXPLAIN mostra estimativas sem executar — seguro em producao. EXPLAIN ANALYZE executa a query de fato e retorna tempos reais. Para queries de escrita (INSERT/UPDATE/DELETE), EXPLAIN ANALYZE executa a modificacao — envolva em uma transacao e faca rollback se precisar analisar sem alterar dados.

3. Lendo o Plano de Execucao

O plano e uma arvore de operacoes. Leitura: do no mais profundo (mais indentado) para o no raiz. O no raiz e a operacao final (resultado ao cliente). Cada no filho alimenta o pai.

explain-analyze-output.txt
EXPLAIN ANALYZE
SELECT p.id, c.nome FROM pedidos p JOIN clientes c ON p.cliente_id = c.id
WHERE p.total > 100;

Resultado:
Hash Join
  (cost=150.00..320.00 rows=1200 width=20)   <- estimativas do planner
  (actual time=5.2..12.8 rows=1187 loops=1)  <- execucao real
  Hash Cond: (p.cliente_id = c.id)
  -> Seq Scan on pedidos  p
       (cost=0..125.00 rows=1200 width=16)
       (actual time=0.05..4.1  rows=1187 loops=1)
       Filter: (total > 100)
       Rows Removed by Filter: 313
  -> Hash
       (cost=10.00..10.00 rows=1000 width=12)
       Buckets: 1024  Batches: 1  Memory Usage: 53kB
       -> Seq Scan on clientes  c
            (cost=0..10.00 rows=1000 width=12)
            (actual time=0.01..0.4 rows=1000 loops=1)

Planning Time: 0.3 ms
Execution Time: 12.9 ms

Interpretacao:
- rows estimado (1200) ~= real (1187): estatisticas OK
- Seq Scan em clientes: tabela pequena (1000 linhas), aceitavel
- Seq Scan em pedidos com Filter: crieria criando indice em pedidos.total

4. Dicionario do EXPLAIN

Seq Scan / Full Table Scan

O banco le todas as paginas da tabela sequencialmente para encontrar as linhas que satisfazem o filtro. E inevitavel quando: (1) nao ha indice na coluna filtrada; (2) a query retorna mais de 20-30% da tabela (nesse caso, o banco prefere Seq Scan ao I/O aleatorio do Index Scan).

Index Seek vs Index Scan

Index Seek (PostgreSQL: Index Scan com range preciso): vai direto ao no folha correto da arvore B-Tree. Leitura em O(log n). Melhor caso.
Index Scan (completo): le o indice de ponta a ponta. Melhor que Seq Scan em tabelas grandes, pior que Seek.

Hash Join vs Nested Loop

Hash Join: constroi hash table da menor relacao em memoria; eficiente para tabelas grandes sem indice no JOIN. Nested Loop: para cada linha da tabela outer, busca na inner — eficiente quando a inner e pequena OU quando ha indice no predicado de JOIN.

Custo e Rows Estimados

cost=X..Y: X = custo de startup (ate retornar a 1a linha), Y = custo total estimado. Se rows estimado difere muito do actual rows, as estatisticas estao desatualizadas — execute ANALYZE tabela para corrigir.

5. Quando Analisar o Plano

Analise o plano quando...

  • Query esta aparecendo no slow query log / pg_stat_statements com tempo elevado.
  • Voce criou um indice e quer confirmar que o banco esta usando-o.
  • Linhas retornadas pelo plano diferem muito do numero real (estatisticas desatualizadas).
  • Antes de colocar em producao uma query nova em tabelas grandes (> 10M linhas).
  • Apos mudancas no schema (adicionar/remover indice, mudar tipos de coluna).

Nao precisa analisar quando...

  • Tabela tem menos de 100.000 linhas — o banco e rapido o suficiente de qualquer forma.
  • Query ja esta com tempo medido abaixo do SLA — nao ha problema a resolver.
  • Seq Scan em tabela pequena — e o comportamento esperado e otimo.
  • O gargalo esta na aplicacao ou na rede, nao no banco.

6. Topicos Avancados

Partition Pruning no Plano

Em tabelas particionadas, o EXPLAIN deve mostrar apenas as particoes relevantes — nao todas. Se voce ve todas as particoes no plano, o pruning nao esta funcionando (verifique se a funcao na coluna de particao esta impedindo o pruning).

Parallel Query

PostgreSQL 9.6+ pode executar Seq Scans e Hash Joins em multiplos workers paralelos. O plano mostra nos Gather e Parallel Seq Scan. Configuravel via max_parallel_workers_per_gather. Nao ajuda em queries com muitos INDEXes pequenos.

JIT Compilation

PostgreSQL 11+ usa LLVM para compilar expressoes de filtro em codigo nativo (JIT). O EXPLAIN ANALYZE mostra JIT: Functions: N / Inlining: true. Beneficio em queries com muita aritmetica ou filtros complexos em grandes volumes; overhead em queries simples curtas.

Bitmap Heap Scan

Hibrido entre Index Scan e Seq Scan. O PostgreSQL usa o indice para construir um bitmap das paginas com linhas relevantes, depois le essas paginas em ordem sequencial. Mais eficiente que Index Scan quando a query retorna muitas linhas espalhadas em varias paginas.

Nao force planos com hints de query (ex: /*+ INDEX(tabela idx) */ no MySQL/Oracle). Hints "consertam" o sintoma mas escondem o problema real (estatisticas desatualizadas, indice faltando, modelo de dados errado). Prefira corrigir as estatisticas ou criar o indice correto — o otimizador vai adaptar automaticamente quando os dados mudam.

7. Ferramentas de Analise

explain.depesz.com

Visualizador online de planos PostgreSQL. Cole a saida do EXPLAIN ANALYZE e veja nos custosos destacados em vermelho. Exibe percentual de tempo por no, rows estimado vs real, buffers usados. Gratuito e sem necessidade de conta.

pgMustard

Ferramenta SaaS que analisa planos PostgreSQL e gera texto explicativo em linguagem natural: "Este no esta tomando 78% do tempo total. Considere adicionar um indice na coluna X." Tier gratuito disponivel.

MySQL EXPLAIN FORMAT=JSON

O MySQL 8+ expoe informacoes muito mais ricas no formato JSON que no formato tabulado padrao — incluindo custo por operacao, loops, buffer usage e use de indices. Combine com EXPLAIN ANALYZE para ver tempos reais.

SQL Server Execution Plan (SSMS)

O SQL Server Management Studio exibe planos graficos com setas proporcionais ao volume de dados. Nós em vermelho destacam avisos (Missing Index, Implicit Conversion). O botao "Include Actual Execution Plan" equivale ao EXPLAIN ANALYZE.

auto_explain (PostgreSQL)

Extensao que loga automaticamente o EXPLAIN ANALYZE de queries que excedem um threshold de tempo. Configuravel por sessao ou globalmente. Essencial para capturar planos ruins em producao sem instrumentacao manual.

pg_stat_statements

Extensao PostgreSQL que acumula estatisticas de execucao de todas as queries (tempo total, chamadas, I/O de buffer). Primeiro passo para identificar as queries que merecem EXPLAIN ANALYZE — foca no que realmente importa em producao.
