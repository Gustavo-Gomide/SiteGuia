Replicacao de Banco de Dados

Replicacao

Primary-Replica, WAL streaming, Replication Lag — alta disponibilidade sem downtime.

Em 2016, o GitLab perdeu 300 GB de dados de producao em um incidente. As replicas existiam, mas nao estavam configuradas para failover automatico. Replicacao resolve dois problemas fundamentais: disponibilidade (se o servidor principal falhar, a replica assume) e escala de leitura (distribuir milhoes de SELECTs entre multiplos nos). Entender como o dado viaja do primario para a replica — e o que pode dar errado nessa viagem — e critico para qualquer sistema de producao.

1. Dois Propositos, Uma Tecnologia

Replicacao e RAID sao coisas diferentes. RAID protege contra falha de disco no mesmo servidor. Replicacao distribui dados para outros servidores — sobrevivendo a falha total do hardware, incendio no datacenter ou erro humano que corrompe todo o servidor.

Proposito O Problema Como a Replicacao Ajuda
Alta Disponibilidade Primary falha (hardware, rede, SO). Aplicacao para. Failover: replica e promovida a primary em segundos. Aplicacao reconecta.
Escala de Leitura Primary nao suporta o volume de SELECTs. CPU a 95%. Direcionar leituras para replicas. Primary lida so com escritas.
Backup Quente Backup tradicional (pg_dump) precisa parar ou e lento. Tirar snapshot da replica sem impactar o primary em producao.
Geografico (Multi-Region) Usuarios na Europa sofrem latencia no servidor nos EUA. Replica na Europa serve leituras locais com baixa latencia.

2. Como Funciona a Replicacao

Todo banco relacional usa um log de transacoes para garantir durabilidade (o D do ACID). O PostgreSQL chama de WAL (Write-Ahead Log), o MySQL chama de binlog. A replicacao funciona enviando esse log para os nos replica, que reexecutam as operacoes.

replicacao-fluxo.txt
Fluxo de dados na replicacao Primary-Replica:

  Cliente --(INSERT/UPDATE/DELETE)--> [Primary]
                                           |
                                     1. Grava no WAL (disco)
                                     2. Confirma ao cliente (commit OK)
                                     3. Envia WAL para replicas
                                           |
                                           v
                                    [WAL Sender] ----TCP----> [WAL Receiver na Replica]
                                                                      |
                                                               4. Replica aplica WAL
                                                               5. Mantém copia ativa

  Cliente --(SELECT)--> [Replica] -- responde com dado atualizado

Streaming Replication

Modo padrao do PostgreSQL. O WAL e enviado em tempo real via TCP/IP assim que cada registro e confirmado. A replica aplica continuamente. Lag tipico: milissegundos. Suporta replicacao fisica (copia exata de bytes) ou logica (operacoes SQL).

Logical Replication

Replica operacoes logicas (INSERT/UPDATE/DELETE decodificadas) em vez de bytes fisicos. Permite replicar apenas tabelas especificas; replica para versoes diferentes do PostgreSQL; replica para ferramentas de CDC (Change Data Capture) como Debezium. Disponivel nativamente no PostgreSQL 10+.

Cascading Replication

Replica encadeia: Primary → Replica1 → Replica2 → Replica3. A Replica1 retransmite o WAL para as demais, aliviando a carga de rede no Primary. Util quando ha muitas replicas ou replicas em regioes geograficas distantes.

3. Arquitetura Primary-Replica

postgresql.conf / recovery.conf
-- PRIMARY (postgresql.conf)
wal_level = replica            -- habilita WAL para replicacao
max_wal_senders = 5            -- max de replicas conectadas
wal_keep_size = 256            -- MB de WAL mantidos para replicas atrasadas

-- PRIMARY (pg_hba.conf) -- permite conexao da replica
host replication replicador 10.0.0.2/32 scram-sha-256

-- REPLICA (postgresql.conf) -- PostgreSQL 12+
primary_conninfo = 'host=10.0.0.1 port=5432 user=replicador'
primary_slot_name = 'replica_slot_1'

-- REPLICA: criar o arquivo de sinal para modo standby
touch /var/lib/postgresql/data/standby.signal

-- INICIAR a replica com o backup base do primary:
pg_basebackup -h 10.0.0.1 -U replicador -D /var/lib/postgresql/data --wal-method=stream

4. Sincrono vs Assincrono

O tradeoff central da replicacao: velocidade vs segurança de dados.

Assincrono (Padrao)

O Primary grava o WAL localmente, confirma ao cliente e depois transmite para a replica. Latencia de escrita: minima (sem espera de rede). Risco: se o primary falhar antes de transmitir o WAL, as ultimas transacoes sao perdidas. Aceitavel para a maioria das aplicacoes com RPO de segundos tolerado.

Sincrono

O Primary espera a confirmacao da replica antes de responder ao cliente. Zero perda de dados (RPO = 0). Custo: latencia de escrita aumenta pela RTT de rede (adicional de 1-5ms em LAN, 30-100ms entre regioes). Se a replica cair, o primary fica bloqueado esperando — necessita configuracao de fallback (synchronous_standby_names = 'ANY 1 (...)').

5. Replication Lag

Na replicacao assincrona, existe atraso entre o dado chegar no primary e aparecer na replica. Esse atraso e o Replication Lag. Normalmente milissegundos, mas pode crescer sob carga pesada ou rede congestionada — e causa um fenomeno classico de bugs de producao.

Read-After-Write Bug: Usuario clica "Salvar perfil" (WRITE no primary) → pagina recarrega (READ na replica) → usuario ve dados antigos e pensa que nao salvou → clica "Salvar" novamente → duplicata. Isso e o replication lag em acao.

Mitigacao: Read-After-Write

Apos uma escrita do proprio usuario, redirecionar a proxima leitura desse usuario para o primary por um curto periodo (ex: 1 segundo). Implementado no router de banco (ProxySQL, PgBouncer) ou na camada de aplicacao com token de sessao.

Mitigacao: Sticky Sessions

Fixar o usuario sempre na mesma replica durante a sessao. Simples de implementar. Trade-off: reduz o balanceamento de carga — uma replica pode ficar mais carregada que outras se um usuario fizer muitas queries.

Monitoramento do Lag

PostgreSQL: SELECT * FROM pg_stat_replication; e SELECT now() - pg_last_xact_replay_timestamp() AS lag; na replica. Alertar quando lag ultrapassar 5s. Quando alto, desviar leituras criticas automaticamente para o primary.

6. Quando Usar Replicacao

Use replicacao quando...

  • SLA exige alta disponibilidade — downtime maximo de minutos, nao horas.
  • Leituras excedem a capacidade de um unico servidor (read-heavy workload).
  • Voce precisa de backup quente sem impactar a producao.
  • Sistema distribui usuarios geograficamente — latencia de rede na leitura importa.
  • Auditoria / analytics pesados: executar na replica sem impactar operacoes transacionais.

Nao adiciona valor quando...

  • Workload e write-heavy — replicacao nao escala escritas, apenas leituras.
  • Aplicacao nao distingue reads e writes — sem separacao de conexoes, replicas ficam ociosas.
  • Volume baixo — um unico servidor com bom hardware e suficiente.
  • Sem plano de failover testado — replica que nunca foi usada pode ter lag enorme.

7. Topicos Avancados

Patroni (Failover Automatico)

Patroni e um agente de alta disponibilidade para PostgreSQL. Monitora o primary via DCS (etcd, Consul, ZooKeeper). Se o primary falhar, inicia eleicao automatica e promove a mejor replica. Reconfgura automaticamente as demais replicas para o novo primary. Padrao de fato para PostgreSQL em producao.

Galera Cluster (Multi-Primary)

Replicacao sincrona multi-primary para MySQL/MariaDB. Todos os nos aceitam escritas simultaneamente. Certifica cada write em todos os nos antes de confirmar — zero lag, zero perda de dados. Custo: latencia de escrita proporcional ao RTT entre nos. Muito usado em setores financeiros que exigem RPO = 0.

Logical Replication Slots (PG)

Replication slots garantem que o WAL nao seja descartado no primary enquanto a replica nao consumiu. Perigo: se a replica ficar muito atrasada ou morrer, o slot acumula WAL indefinidamente — pode encher o disco do primary e derrubar todos os servicos. Sempre configure max_slot_wal_keep_size.

Active-Active (Bi-Directional)

Ambos os nos aceitam escritas e replicam entre si. Muito complexo: exige resolucao de conflitos quando dois nos modificam o mesmo registro simultaneamente. Implementacoes: Bucardo, pglogical (PostgreSQL), MySQL Group Replication. Evite sem conhecimento profundo dos trade-offs.

Replica nao e substituto de Backup. Se voce executar DROP TABLE producao no primary, essa operacao sera replicada para todas as replicas em milissegundos. Replicacao protege contra falha de hardware — nao contra erro humano. Voce ainda precisa de backup com PITR.

8. Ferramentas

Patroni

Gerenciador de HA para PostgreSQL. Failover automatico, reconfiguração de replicas, controle via DCS. Integra com Kubernetes (operator). Mantido pela Zalando, amplamente adotado em producao.

repmgr

Ferramenta de gerenciamento de replicacao e failover para PostgreSQL. Mais simples que Patroni. Comandos: repmgr primary register, repmgr standby follow, repmgr standby promote. Bom para setups menores.

ProxySQL

Proxy SQL para MySQL/MariaDB. Faz roteamento automatico: escritas para primary, leituras para replicas. Monitora o lag das replicas e exclui replicas atrasadas do pool de leitura automaticamente. Essencial para MySQL com multiplas replicas.

Galera Cluster

Replicacao sincrona multi-primary para MySQL/MariaDB/Percona. Usada no Percona XtraDB Cluster e MariaDB Galera Cluster. Nao requer configuracao manual de failover — todos os nos sao equivalentes. Latencia de escrita e o trade-off.

AWS RDS Read Replicas

Replicas gerenciadas para RDS MySQL, PostgreSQL e Aurora. Criacao com um clique; failover automatico no Aurora (Multi-AZ). Replicas podem estar em regioes diferentes. Aurora Global Database oferece lag sub-segundo entre regioes.

Debezium (CDC)

Plataforma de Change Data Capture que usa logical replication (WAL do PG, binlog do MySQL). Transforma mudancas do banco em eventos para Kafka. Fundamental para arquiteturas event-driven e sincronizacao entre sistemas.
