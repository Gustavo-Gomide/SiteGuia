Sharding — Dividir para Conquistar

Distribuindo dados entre multiplos servidores quando um servidor nao e mais suficiente.

Em 2010, o Instagram enfrentou seu primeiro gargalo: um unico servidor PostgreSQL com 250 mil usuarios cadastrados. Seis meses depois, eram 14 milhoes. A solucao foi sharding — dividir a tabela de usuarios em dezenas de shards independentes. Cada foto, cada follow, cada like roteado automaticamente para o servidor correto. Sharding e a tecnica que permite escalar bancos de dados para bilhoes de registros quando todas as outras opcoes ja foram esgotadas.

1. Scale Up vs. Scale Out

Antes de recorrer ao sharding, entenda as duas estrategias de crescimento e quando cada uma faz sentido.

Dimensao Scale Up (Vertical) Scale Out / Sharding (Horizontal)
Como funciona Troca o servidor por um mais poderoso (mais RAM, CPU, SSD NVMe) Adiciona mais servidores e divide os dados entre eles
Limite Teto fisico e economico (servidor de 4 TB RAM custa > R$ 5 milhoes) Quase ilimitado — adicione quantos shards quiser
Complexidade Baixissima — o banco funciona normalmente Altissima — JOINs, transacoes e resharding sao problemas novos
Downtime Necessario para trocar hardware Adicionar shards pode ser feito com zero downtime (com suporte do sistema)
Quando usar Ate ~10 TB de dados e ~100k queries/s Apos esgotar Scale Up, particao, indices e caching
Exemplos RDS r6g.16xlarge (AWS), base de dados de ERP corporativo Instagram, Discord, Shopify — bilhoes de registros

Sharding nao substitui Replicacao: replicacao copia os mesmos dados para multiplos servidores (para leitura e alta disponibilidade). Sharding divide os dados — cada servidor tem um subconjunto unico. Em producao de grande escala, ambas as tecnicas sao usadas simultaneamente: cada shard tem suas proprias replicas.

2. Como Funciona o Sharding

O sistema de sharding tem tres componentes: o Router que recebe as queries, a logica de roteamento que determina qual shard contem os dados, e os shards em si (servidores independentes com subconjunto dos dados).

diagrama — fluxo de uma query em ambiente shardado
-- Aplicacao envia query com shard key
App  →  Router (mongos / Vitess / middleware)

                    Router consulta mapa de shards
                         |
          +--------------+--------------+
          |              |              |
       Shard A        Shard B        Shard C
    (users 0-333)  (users 334-666)  (users 667-999)
     servidor-1     servidor-2      servidor-3

-- SELECT * FROM users WHERE id = 1500
-- Router: 1500 % 3 = 0  →  Shard A (servidor-1)
-- Consulta vai APENAS para servidor-1. Servidores 2 e 3 nao sao tocados.

Hash Sharding

A shard key passa por uma funcao hash (id % N ou consistent hashing). Garante distribuicao uniforme dos dados. Ideal para IDs numericos ou UUIDs. Problema: range queries viram full scatter (vao a todos os shards).

Range Sharding

Os dados sao divididos por intervalos da shard key (A-I, J-Q, R-Z ou datas por ano). Excelente para range queries (buscar pedidos de janeiro a marco em um unico shard). Risco: hotspot se um intervalo receber muito mais dados que outros.

Directory-Based Sharding

Um servico separado mantem um mapa explicito de qual registro pertence a qual shard. Oferece maximo de flexibilidade para mover dados entre shards sem reescrever as regras. Custo: o servico de diretorio vira um ponto unico de falha.

3. A Logica do Corte — Escolhendo a Shard Key

A shard key e a decisao mais critica e mais permanente em um sistema shardado. Uma escolha errada e quase impossivel de reverter sem reescrever o sistema inteiro.

diagrama — hash mod vs. consistent hashing
-- Hash Simples: shard = id % N_shards
user_id = 1500, shards = 3:  1500 % 3 = 0  →  Shard A
user_id = 1501, shards = 3:  1501 % 3 = 1  →  Shard B
user_id = 1502, shards = 3:  1502 % 3 = 2  →  Shard C

-- PROBLEMA ao adicionar shard: N muda de 3 para 4
-- user_id=1500: 1500 % 4 = 0  →  ok
-- user_id=1501: 1501 % 4 = 1  →  ok
-- user_id=1502: 1502 % 4 = 2  →  ok
-- user_id=1503: 1503 % 4 = 3  →  NOVO SHARD (dado estava no shard 0!)
-- Adicionar um shard com hash simples invalida 75% dos mapeamentos!

-- Consistent Hashing: resolve o problema
-- Organiza shards em um anel logico (0 a 2^32)
-- Adicionar novo shard move apenas 1/N dos dados (em vez de quase todos)
-- Usado por DynamoDB, Cassandra, Riak, Redis Cluster

Boas Shard Keys

  • user_id: distribuicao uniforme, isolamento de tenant natural
  • tenant_id: ideal para SaaS multi-tenant (cada empresa em um shard)
  • UUID v4: alta cardinalidade, distribuicao aleatoria uniforme
  • hash(email): uniforme e evita hotspot de IDs sequenciais

Mas Shard Keys — Exemplos de Erros Reais

  • timestamp: todos os writes vao para o shard mais recente (hotspot)
  • status (ativo/inativo): apenas 2 valores — impossivel distribuir
  • pais: Brasil pode ter 10x mais usuarios que Bolivia (desequilibrio)
  • ID sequencial sem hash: writes sempre no ultimo shard (hotspot)

4. O Pesadelo do JOIN Distribuído

A operacao mais natural em SQL — o JOIN — se torna o problema mais complicado em reambientes shardados. Entender essa limitacao e fundamental antes de decidir pelo sharding.

sql — um JOIN simples se torna um problema distribuido
-- Em banco monolitico: simples e eficiente
SELECT u.nome, COUNT(p.id) AS total_pedidos
FROM users u
JOIN pedidos p ON u.id = p.user_id
WHERE u.id = 1500;

-- Com sharding por user_id:
-- users onde id=1500  →  Shard A (servidor-1)
-- pedidos onde user_id=1500  →  tambem Shard A (se shardado pela mesma key)
-- Funciona! Ambas as tabelas estao no mesmo shard.

-- PROBLEMA: JOIN sem a shard key:
SELECT u.nome, SUM(p.valor) AS total
FROM users u
JOIN pedidos p ON u.id = p.user_id
WHERE p.status = 'pendente';  -- sem filtro por user_id!

-- O sistema precisa consultar TODOS os shards em paralelo,
-- juntar os resultados em memoria na aplicacao,
-- e retornar o resultado final. Latencia x N shards.

Alternativas antes do Sharding

  • Indices compostos e parciais: eliminam 90% dos problemas de leitura
  • Particionamento local: mesma maquina, muito mais veloz
  • Read Replicas: escala leitura sem dividir dados
  • Caching (Redis): queries frequentes servidas sem tocar o banco
  • Scale Up: dobrar RAM frequentemente resolve por anos

Sinais de Que Voce Precisa de Sharding

  • Dataset acima de 10-50 TB que nao cabe em um unico servidor
  • Acima de 100.000 writes por segundo no banco principal
  • Scale Up chegou ao teto de hardware disponivel e custo-beneficio
  • Requisito regulatorio de isolamento geografico de dados (GDPR)
  • Particionamento e indices ja foram otimizados ao maximo

5. Quando Shardear

Use Sharding Quando

  • Dataset nao cabe em um unico servidor mesmo com Scale Up
  • Escala de escrita onde um unico no nao aguenta (redes sociais, IoT)
  • Isolamento geografico obrigatorio por lei (dados brasileiros no Brasil)
  • Multi-tenancy com isolamento forte entre clientes enterprise
  • Voce tem equipe de infraestrutura dedicada para gerenciar a complexidade

Sharding e Natural Para

  • Redes sociais: cada usuario e seus dados no mesmo shard
  • E-commerce de alta escala: pedidos shardados por seller_id
  • Sistemas de mensagens: conversas shardadas por conversation_id
  • Jogos online: partidas e jogadores shardados por regiao
  • IoT: series temporais shardadas por device_id

Nao Shardear Quando

  • Dataset cabe confortavelmente em um servidor com margem de crescimento
  • Aplicacao faz muitos JOINs entre tabelas que seriam divididas em shards diferentes
  • Equipe pequena sem experiencia em sistemas distribuidos
  • Indices e particao local resolvem o problema de performance
  • O sistema precisa de transacoes ACID que cruzem multiplas entidades

Sharding Complica Muito

  • Transacoes distribuidas (2PC vira necessario, latencia explode)
  • Resharding ao adicionar servidores (mover dados com zero downtime e complexo)
  • Queries globais sem shard key (full scatter scan em todos os shards)
  • Debugging: um erro pode estar em qualquer um dos N servidores
  • Backup e disaster recovery multiplicados por N shards

6. Topicos Avancados

Vitess — Sharding para MySQL na Escala do YouTube

Criado pelo YouTube e hoje usado pelo Slack, GitHub e PlanetScale. O Vitess adiciona uma camada de roteamento transparente sobre MySQL, gerenciando resharding online, failover automatico e connection pooling. O MySQL nao sabe que esta sendo shardado.

Citus — PostgreSQL Distribuido

Extensao open-source que transforma PostgreSQL em um banco distribuido. Os dados sao distribuidos como shards entre workers enquanto um no coordenador recebe todas as queries e as paraleliza. A Microsoft oferece Citus como servico gerenciado no Azure (Hyperscale).

MongoDB Sharding — Mongos como Router

MongoDB tem sharding nativo. O processo mongos e o router que recebe queries dos clientes e as encaminha para o shard correto com base no shard key definido por colecao. Config Servers armazenam os metadados do mapeamento de chunks para shards.

CockroachDB — Sharding Automatico e Transparente

Banco de dados distribuido com SQL nativo que faz sharding automaticamente sem que o desenvolvedor precise configurar nada. Os dados sao divididos em ranges de 512 MB que sao movidos entre nos automaticamente. Implementa consenso Raft para consistencia forte mesmo em falhas parciais.

O serio problema dos Cross-Shard JOINs. Qualquer query que nao inclua a shard key no WHERE vira um scatter-gather: o router envia a query para TODOS os shards em paralelo, coleta os resultados e os une. Para relatórios analiticos e aggregations globais, isso pode ser 10-100x mais lento que um banco monolitico com indices corretos. Considere um data warehouse separado (BigQuery, Redshift, Snowflake) para queries analiticas em sistemas shardados.

7. Ecossistema e Ferramentas

Vitess

Camada de sharding para MySQL com resharding online, failover automatico e connection pooling. Open-source, CNCF graduated. Usado em producao pelo YouTube desde 2011 e pelo Slack, GitHub e Square. Oferecido como servico gerenciado pelo PlanetScale.

Citus (Microsoft)

Extensao PostgreSQL que distribui tabelas em shards entre workers. Suporta queries SQL completas, incluindo JOINs distribuidos (com overhead). Disponivel como Azure Cosmos DB for PostgreSQL (antigo Hyperscale Citus) totalmente gerenciado.

MongoDB Atlas Sharding

Sharding nativo do MongoDB com interface visual no Atlas. Permite escolher shard key por colecao, monitorar distribuicao de chunks e fazer resharding com zero downtime na versao 6.0+. Compound shard keys (multi-campo) permitem evitar hotspots.

PlanetScale

Banco MySQL gerenciado em cloud com Vitess como backend. Sharding horizontal transparente, deployments sem schema lock, e branching de banco de dados similar ao Git. Usado por Booking.com, Figma e Square para escala de bilhoes de linhas.

CockroachDB

Banco distribuido com sharding completamente automatico. O desenvolvedor trabalha com SQL padrão sem se preocupar com shard keys. Forte consistencia via Raft e distribuicao geografica nativa. Versao community open-source disponivel.

Amazon Aurora + DynamoDB

Aurora oferece storage distribuido ate 128 TB sem sharding de aplicacao. Para ultrapasse disso, DynamoDB implementa consistent hashing nativo com particionamento automatico transparente — praticamente infinito — com throughput provisionado por tabela.
