Backup e Recuperacao de Bancos de Dados

Backup e Recuperacao

RPO, RTO, PITR e a Regra 3-2-1-1-0 para sobreviver a qualquer desastre.

Existem dois tipos de DBAs: os que ja perderam dados e os que vao perder. Em 2012, o Ma.gnolia perdeu irreversivelmente todos os dados de usuarios por ausencia de backups. Em 2017, a GitLab deletou acidentalmente 300 GB de producao — e os backups estavam quebrados. Estrategia de backup nao e sobre copiar arquivos: e sobre garantir que a empresa sobreviva a um desastre, seja ransomware, corrupcao silenciosa ou erro humano.

1. O Vocabulario do Desastre

Antes de configurar qualquer estrategia de backup, o negocio precisa definir tres metricas criticas. Sem elas, qualquer decisao tecnica e arbitraria.

Metrica Definicao Exemplo Pratico
RPO — Recovery Point Objective Quanto de dado (no tempo) podemos perder? Define a frequencia minima dos backups. RPO = 1h: backup de hora em hora. Se o disco falhou as 14h50, perdemos dados desde 14h.
RTO — Recovery Time Objective Em quanto tempo o sistema deve voltar ao ar apos um desastre? RTO = 4h: equipe tem 4h para restaurar antes do impacto financeiro ser inaceitavel.
MTD — Maximum Tolerable Downtime Tempo maximo absoluto de indisponibilidade antes de danos irreversiveis ao negocio. E-commerce na Black Friday: MTD = 2h. Hospital critico: MTD = 15 min.

Regra de ouro: Quanto menores forem o RPO e RTO, mais caro e complexo sera o sistema de backup. RPO = 0 e RTO = 0 exigem replicacao sincrona em tempo real — com custo financeiro e de latencia significativos.

2. Tipos de Backup

Nao e viavel copiar dezenas de terabytes todos os dias. A industria desenvolveu tres estrategias complementares que equilibram custo, velocidade e complexidade de restauracao.

Full (Completo)

Copia completa de todos os dados — base para todos os outros tipos. Geralmente agendado semanalmente (ex: domingo de madrugada). O mais simples de restaurar: um unico arquivo.

Diferencial

Copia tudo que mudou desde o ultimo backup Full. Cresce progressivamente durante a semana. Restauracao exige o ultimo Full + o ultimo Diferencial — apenas 2 arquivos, previsivel.

Incremental

Copia apenas o que mudou desde o ultimo backup de qualquer tipo. Arquivo diario minimo, mas a restauracao exige o Full + toda a cadeia de incrementais — RPO minimo, maior complexidade.

3. Point-in-Time Recovery (PITR)

O PITR e a "maquina do tempo" dos bancos de dados. Permite restaurar o banco para qualquer momento especifico no passado — nao apenas para o instante do ultimo backup. Funciona gravando continuamente o log de transacoes (WAL no PostgreSQL, Redo Log no MySQL/Oracle).

Cenario real: Um desenvolvedor executou UPDATE pedidos SET status = 'cancelado' sem clausula WHERE as 14h35. Com PITR habilitado, voce restaura para 14h34:59 em ambiente separado e extrai os dados corretos — sem perder o dia inteiro de trabalho.

pitr-postgresql.sh
# 1. Backup base
pg_basebackup -h localhost -U replicador -D /backup/base -Ft -z -P

# 2. Arquivar WALs (postgresql.conf)
archive_mode = on
archive_command = 'cp %p /backup/wal/%f'

# 3. Restaurar para ponto especifico (PostgreSQL 12+)
restore_command = 'cp /backup/wal/%f %p'
recovery_target_time = '2026-03-04 14:34:59'
recovery_target_action = 'promote'

touch /var/lib/postgresql/data/recovery.signal
pg_ctl start -D /var/lib/postgresql/data
pitr-mysql.sh
# MySQL/MariaDB — habilitar binary logs (my.cnf)
[mysqld]
log_bin    = /var/log/mysql/mysql-bin.log
binlog_format = ROW

# Restaurar backup + aplicar binlogs ate o ponto desejado
mysqlbinlog --stop-datetime="2026-03-04 14:34:59" \
  /var/log/mysql/mysql-bin.000001 | mysql -u root -p

4. A Regra 3-2-1-1-0

Desenvolvida na fotografia digital e adotada como padrao de seguranca da informacao, a Regra 3-2-1 e o minimo aceitavel para qualquer sistema critico. A versao moderna acrescenta duas exigencias fundamentais: imutabilidade e verificacao.

3 Copias dos Dados

O dado original + dois backups independentes. Se uma copia estiver corrompida, voce ainda tem outras duas. Nunca dependa de uma unica copia, nem mesmo do dado original.

2 Midias Diferentes

Exemplos: disco local + nuvem, SSD + fita magnetica. Se ambas as copias ficam no mesmo hardware, uma unica falha pode eliminar ambas simultaneamente.

1 Copia Offsite

Pelo menos uma copia fora do ambiente principal, geograficamente separada. Incendio, inundacao, roubo ou ransomware na rede local nao afetam o backup offsite.

+1 Imutavel e 0 Erros

+1 copia imutavel (WORM/air-gapped — nao pode ser deletada por ransomware). 0 erros verificados: todo backup deve ser validado automaticamente e com restore drill periodico. O "0 erros" e a parte mais negligenciada e a mais importante.

5. Quando Usar Cada Estrategia

Use PITR quando...

  • RPO minimo — perder qualquer transacao e inaceitavel (financeiro, saude).
  • Equipe precisa se recuperar de erros humanos (UPDATE/DELETE sem WHERE).
  • Auditoria exige prova do estado do banco em qualquer ponto no tempo.
  • Dados de producao sao dificeis de recriar e tem altissimo valor de negocio.

Full + Incremental quando...

  • Volume de dados grande torna backups full diarios proibitivos em custo e janela de tempo.
  • RPO de algumas horas e toleravel pelos SLAs do negocio.
  • Equipe tem automacao para gerenciar cadeias de incrementais.

Nunca faca isso...

  • Backup no mesmo servidor dos dados — sem protecao contra falha de hardware.
  • Backup sem verificacao periodica — corrupcao silenciosa e descoberta so no desastre.
  • Estrategia sem restore drill documentado e praticado regularmente.
  • Backup configurado uma unica vez e esquecido — politicas caducam com o negocio.

6. Topicos Avancados

Backups Imutaveis (WORM)

Write Once Read Many — arquivos que nao podem ser modificados ou deletados apos criacao. S3 Object Lock, Azure Immutable Blob Storage e fitas LTO WORM sao imunes a ransomware: mesmo com credenciais de admin, o atacante nao pode deletar dentro do periodo de retencao configurado.

Restore Drill

Simulacao periodica (mensal ou trimestral) de recuperacao completa em ambiente isolado. Mede o tempo real de restauracao, valida a integridade dos dados e confirma que o RTO definido e atingivel. Um backup nunca testado nao e backup — e uma esperanca.

Criptografia de Backups

Backups em repouso devem ser criptografados com AES-256. pgBackRest tem criptografia nativa. WAL-G integra com AWS KMS e GCP KMS. As chaves devem ser gerenciadas separadamente dos dados: backup criptografado sem a chave e inutil para atacante e para voce.

Backup Logico vs Fisico

Logico (pg_dump, mysqldump): exporta SQL portavel, permite restauracao seletiva de tabelas. Fisico (pg_basebackup, XtraBackup): copia arquivos brutos, muito mais rapido para grandes volumes, mas restrito ao mesmo SGBD e versao major.

Corrupcao Silenciosa: backups corrompidos sao descobertos apenas na restauracao — geralmente durante o proprio desastre. Use pgbackrest verify, checksums do S3 e validacao de hash apos cada backup. Ferramentas como pgBackRest armazenam checksums de cada arquivo WAL para detectar corrupcao automaticamente antes que voce precise restaurar.

7. Ferramentas de Backup

pgBackRest

Estado da arte para PostgreSQL self-managed. Backups paralelos, compressao, criptografia AES-256, integracao com S3/GCS/Azure Blob, PITR completo e verificacao de integridade nativa com pgbackrest verify.

WAL-G

Alternativa moderna ao pgBackRest, tambem para MongoDB e MySQL. Nativo para cloud — uploads paralelos para S3, GCS, Azure Blob ou Yandex Object Storage. Integra com AWS KMS para criptografia gerenciada.

Percona XtraBackup

Hot backup para MySQL e MariaDB sem travar as tabelas. Suporta backup incremental nativo, criptografia e streaming direto para S3. Essencial para bases MySQL acima de alguns GBs onde mysqldump seria proibitivamente lento.

AWS RDS / Aurora

Backups automaticos diarios com retencao de 1–35 dias. PITR com granularidade de 5 minutos habilitado por padrao. Snapshots manuais retidos indefinidamente. Aurora Global faz replicacao para segunda regiao com RPO de segundos.

Azure SQL / GCP Cloud SQL

Azure: full semanal + diferencial a cada 12h + log a cada 5–10 min, retencao ate 35 dias, geo-redundant backup opcional. GCP: backups automaticos diarios com PITR habilitavel e exportacao para Cloud Storage em SQL ou CSV.

Barman

Backup and Recovery Manager para PostgreSQL, mantido pela EDB (2ndQuadrant). Gerenciamento centralizado de multiplos servidores PostgreSQL, catalogos de backups, retention policies e integracao com Ansible e Terraform.
