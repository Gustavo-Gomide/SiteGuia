Otimizacao de Consultas

Do SELECT* ao plano de execucao — escrevendo SQL que respeita o otimizador e o disco.

Uma consulta mal escrita pode derrubar um servidor inteiro. Otimizacao nao e sobre "adicionar mais memoria" — e sobre escrever SQL que le o minimo de dados possivel do disco. Um Full Table Scan em uma tabela de 500 milhoes de linhas pode levar minutos; o mesmo dado com um indice correto retorna em milissegundos. Entender o otimizador e deixar de adivinhar.

1. O Otimizador de Consultas

Quando voce escreve SQL, voce diz ao banco O QUE quer — nao COMO buscar. O Otimizador baseado em custo e o cerebro que decide a melhor rota estimando I/O de disco, uso de CPU e numero de linhas processadas para cada plano possivel.

ciclo-de-execucao.txt
Ciclo de vida de uma query SQL:

  Cliente envia SQL
       |
       v
  [Parser]          <- verifica sintaxe, cria AST (arvore sintatica)
       |
       v
  [Rewriter]        <- expande views, aplica regras
       |
       v
  [Planner/Optimizer] <- gera N planos possiveis, estima custo de cada um
       |                  usa estatisticas (pg_statistic / information_schema)
       v
  [Executor]        <- executa o plano de menor custo estimado
       |
       v
  Resultado ao cliente

Estatisticas do Otimizador

O otimizador usa estatisticas como contagem de linhas, distribuicao de valores (histograma) e cardinalidade de colunas para estimar o custo de cada plano. No PostgreSQL, ANALYZE atualiza essas estatisticas. Estatisticas desatualizadas geram planos ruins — por isso o autovacuum as atualiza automaticamente.

Custo Baseado (CBO)

O otimizador compara planos por custo estimado (unidade arbitraria de CPU + I/O). Ele pode errar quando as estatisticas estao desatualizadas, quando ha muitos JOINs com correlacoes entre colunas ou quando a distribuicao de dados e muito nao uniforme. EXPLAIN ANALYZE mostra o custo estimado vs real.

Cache de Planos

Bancos como PostgreSQL e MySQL cachear planos compilados para prepared statements. Um plano gerado para uma distribuicao de dados pode ser subotimo para outra. PostgreSQL 14+ usa planos genericos vs customizados automaticamente. No SQL Server, plans sao cached por hash da query — parametros literais diferentes = planos diferentes.

2. Tipos de Indice

O indice e a estrutura de dados auxiliar que permite ao banco encontrar linhas sem ler a tabela inteira. Escolher o tipo errado de indice pode ser pior do que nao ter indice.

indices-postgresql.sql
-- B-Tree (padrao): igualdade, range, ORDER BY
CREATE INDEX idx_pedidos_data ON pedidos (data_criacao);
-- Suporta: =, <, >, BETWEEN, LIKE 'prefix%', IS NULL

-- Hash: APENAS igualdade — mais rapido que B-Tree para = exato
CREATE INDEX idx_sessao_token ON sessoes USING HASH (token);
-- NAO suporta range queries nem ORDER BY

-- Partial: indexa apenas subconjunto das linhas
CREATE INDEX idx_pedidos_pendentes ON pedidos (cliente_id)
  WHERE status = 'pendente';
-- Indice 100x menor, mais rapido, zero custo para linhas 'entregue'

-- Composito: colunas na ordem certa do WHERE/ORDER BY
CREATE INDEX idx_pedidos_comp ON pedidos (cliente_id, data_criacao DESC);
-- Serve para: WHERE cliente_id = ? ORDER BY data_criacao DESC
-- NAO serve para: WHERE so data_criacao (leftmost prefix rule)

-- Covering Index: inclui colunas extras para evitar heap fetch
CREATE INDEX idx_cov ON pedidos (cliente_id) INCLUDE (status, total);
-- Query retorna sem nem tocar na tabela principal (Index Only Scan)

-- GIN: full-text search e arrays
CREATE INDEX idx_descricao_fts ON produtos USING GIN (to_tsvector('portuguese', descricao));

3. Escrever Queries Eficientes

O Pecado do SELECT *

sql
-- Lento: traz foto BLOB, historico e tudo
SELECT * FROM usuarios;

-- Rapido: pode usar Covering Index
SELECT nome, email FROM usuarios;

SARGable (Index Friendly)

sql
-- NAO SARGable: funcao na coluna cega o indice
WHERE YEAR(data_venda) = 2023

-- SARGable: range direto no indice
WHERE data_venda BETWEEN
  '2023-01-01' AND '2023-12-31'

LIKE com coringa no inicio

sql
-- Sem indice: coringa no inicio
WHERE nome LIKE '%Silva'

-- Com indice: prefixo conhecido
WHERE nome LIKE 'Silva%'

Problema N+1

sql/pseudocodigo
-- N+1: 1 query de lista + N queries no loop
SELECT * FROM posts LIMIT 100; -- query 1
for each post: SELECT * FROM autores ... -- 100x!

-- Solucao: JOIN ou eager loading
SELECT p.titulo, a.nome
FROM posts p JOIN autores a ON p.autor_id = a.id
LIMIT 100;

4. Quando Otimizar e Quando Nao Otimizar

Otimize quando...

  • Query aparece no pg_stat_statements / slow query log com tempo > 1s.
  • EXPLAIN mostra Full Table Scan / Seq Scan em tabelas grandes.
  • Numero de linhas estimado pelo planner difere muito do real (estatisticas desatualizadas).
  • Problema N+1 diagnosticado — ORM fazendo centenas de queries por request.
  • Query e executada milhares de vezes por minuto em producao.

Nao otimize quando...

  • Query roda 1x por dia em batch — custo/beneficio nao justifica.
  • Tabela tem menos de 10.000 linhas — Full Scan pode ser mais rapido que indice.
  • Indice seria em coluna de baixa cardinalidade (ex: booleano, status com 3 valores).
  • A query nao e o gargalo — o problema esta no codigo da aplicacao ou na rede.

5. Topicos Avancados

pg_stat_statements

Extensao do PostgreSQL que acumula estatisticas de todas as queries executadas: tempo total, chamadas, linhas retornadas, I/O de buffer. Essencial para encontrar as queries mais custosas em producao. SELECT * FROM pg_stat_statements ORDER BY total_exec_time DESC LIMIT 20;

Autovacuum e Estatisticas

O PostgreSQL usa MVCC — linhas antigas (dead tuples) acumulam e degradam performance. O autovacuum as limpa e atualiza estatisticas automaticamente. Sinais de problema: table bloat (tabela muito maior que o esperado), planos ruins (estatisticas velhas), sequencias longas de Index Scan virando Seq Scan.

Connection Pooling

Abrir/fechar conexoes com o banco e caro (fork de processo no PostgreSQL). PgBouncer e ProxySQL mantem um pool de conexoes persistentes e dao ao cliente uma conexao do pool. Reduz overhead de conexao e previne o banco de ficar sem conexoes disponiveis sob carga.

Parallel Query e JIT

PostgreSQL 9.6+ executa queries em multiplos workers paralelos para Seq Scans e Hash Joins grandes. JIT (Just-In-Time compilation) compila expressoes de filtro em codigo nativo via LLVM — beneficio em queries com muita aritmetica ou filtros complexos em grandes volumes de dados.

Excesso de Indices: cada indice adicional tem custo em toda operacao de INSERT, UPDATE e DELETE — o banco deve atualizar todos os indices afetados. Banco com 15 indices por tabela pode ter writes 5x mais lentos do que o necessario. Use pg_stat_user_indexes para identificar indices nunca usados e remova-os.

6. Ferramentas de Performance

EXPLAIN / EXPLAIN ANALYZE

Comando nativo de todos os principais SGBDs. EXPLAIN mostra o plano estimado; EXPLAIN ANALYZE executa e mostra tempos reais, linhas reais vs estimadas e buffers usados. Primeiro passo obrigatorio em qualquer investigacao de performance.

pganalyze

SaaS de monitoramento continuo de performance para PostgreSQL. Detecta queries lentas, sugere indices automaticamente, monitora bloat de tabelas, lock waits e replication lag. Tem tier gratuito para um servidor.

pt-query-digest (Percona)

Analisa o slow query log do MySQL e gera relatorio agregado das queries mais custosas: tempo total, percentis, fingerprint normalizada. Parte do Percona Toolkit — essencial para auditorias de performance em MySQL/MariaDB.

explain.depesz.com

Visualizador online de planos de execucao do PostgreSQL. Cole a saida do EXPLAIN ANALYZE e veja os nos mais custosos destacados em vermelho/amarelo. Facilita enormemente a leitura de planos complexos com muitos JOINs.

PgBouncer

Connection pooler leve para PostgreSQL. Tres modos: session (compatibilidade total), transaction (mais eficiente) e statement. Essencial para aplicacoes com muitas conexoes curtas — reduz connextion overhead e previne esgotamento de conexoes do banco.

auto_explain

Extensao do PostgreSQL que loga automaticamente o plano EXPLAIN ANALYZE de queries que excedem um threshold de tempo configuravel. Permite diagnosticar queries lentas em producao sem instrumentacao manual nas aplicacoes.
