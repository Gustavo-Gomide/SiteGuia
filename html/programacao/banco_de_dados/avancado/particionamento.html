Particionamento de Dados

Particionamento

Como dividir tabelas de bilhoes de linhas sem matar o banco — e sem mudar uma linha de SQL.

Uma tabela de logs com 2 bilhoes de registros ocupa 800 GB. Uma query de auditoria WHERE data_log BETWEEN '2024-01-01' AND '2024-03-31' sem particao faz Full Scan de 800 GB. Com particionamento por mes, o banco le apenas as 3 particoes relevantes — menos de 2 GB. Particionamento e sobre eliminar dados irrelevantes antes de qualquer indice entrar em cena. E a tecnica mais poderosa para gerenciar dados em escala que um engenheiro de banco precisa dominar.

1. Particao vs Tabela vs Sharding

Confusao comum: Particionamento e sharding sao coisas diferentes. Particionamento divide uma tabela dentro de um unico servidor — transparente para a aplicacao, o mesmo SQL funciona sem mudancas. Sharding distribui os dados por multiplos servidores e exige mudancas na camada de aplicacao.

Criterio Tabela Normal Particionada Sharding
Localizacao Um arquivo de dados N arquivos, 1 servidor N servidores
Transparencia SQL Total Total (mesmo SQL) Parcial (depende do router)
Escala Disco do servidor Disco do servidor Multi-servidor (horizontal)
Complexidade Nenhuma Media (DDL) Alta (infra + codigo)
Quando usar Tabelas < 100M linhas 100M–10B linhas > 10B linhas ou write bottleneck

2. Partition Pruning

O mecanismo central que torna o particionamento util. Quando o otimizador consegue determinar pela clausula WHERE quais particoes sao relevantes, ele simplesmente ignora todas as outras — sem leitura, sem I/O, sem CPU. Isso e o partition pruning.

partition-pruning.txt
Tabela: logs_evento (particionada por RANGE em data_log, particao por mes)
Particoes: logs_2023_01, logs_2023_02, ..., logs_2024_12

SELECT COUNT(*) FROM logs_evento
WHERE data_log BETWEEN '2024-01-01' AND '2024-03-31';

Sem particao: Seq Scan em 2 bilhoes de linhas (800 GB)
Com particao: Pruning elimina 21 particoes, le apenas 3 (logs_2024_01..03)
Resultado: 150ms vs 8min

EXPLAIN mostra o pruning em acao (PostgreSQL):
Append
  -> Seq Scan on logs_2024_01  (cost=0..45321.00)
  -> Seq Scan on logs_2024_02  (cost=0..44218.00)
  -> Seq Scan on logs_2024_03  (cost=0..41534.00)
As outras 21 particoes nem aparecem no plano.

Pruning Estatico

Ocorre quando a clausula WHERE usa uma constante literal (WHERE data = '2024-01-15'). O otimizador determina as particoes relevantes em tempo de planejamento — antes de executar. Melhor caso: plano gerado ja sem as particoes desnecessarias.

Pruning Dinamico

Ocorre com parametros de prepared statements (WHERE data = $1). O otimizador nao sabe o valor em tempo de planejamento, entao o pruning acontece em tempo de execucao. PostgreSQL 11+ e MySQL 8+ suportam dynamic pruning automaticamente.

Armadilhas do Pruning

Funcao na coluna de particao desabilita o pruning: WHERE YEAR(data_log) = 2024 faz Full Scan. Use sempre a coluna diretamente: WHERE data_log BETWEEN '2024-01-01' AND '2024-12-31'.

3. Como Criar Particoes

particoes-postgresql.sql
-- PostgreSQL 10+ (declarative partitioning)
CREATE TABLE logs_evento (
  id          BIGSERIAL,
  data_log    DATE NOT NULL,
  tipo        VARCHAR(50),
  descricao   TEXT
) PARTITION BY RANGE (data_log);

-- Criando particoes manualmente
CREATE TABLE logs_2024_01 PARTITION OF logs_evento
  FOR VALUES FROM ('2024-01-01') TO ('2024-02-01');

CREATE TABLE logs_2024_02 PARTITION OF logs_evento
  FOR VALUES FROM ('2024-02-01') TO ('2024-03-01');

-- Indices sao criados por particao (PostgreSQL cria automaticamente)
CREATE INDEX ON logs_evento (data_log);
-- Cria um indice em cada particao individualmente
particoes-mysql.sql
-- MySQL 8+ — particionamento por RANGE com TO_DAYS
CREATE TABLE logs_evento (
  id          BIGINT UNSIGNED AUTO_INCREMENT,
  data_log    DATE NOT NULL,
  tipo        VARCHAR(50),
  descricao   TEXT,
  PRIMARY KEY (id, data_log)  -- coluna de particao deve estar na PK
)
PARTITION BY RANGE (TO_DAYS(data_log)) (
  PARTITION p2024_01 VALUES LESS THAN (TO_DAYS('2024-02-01')),
  PARTITION p2024_02 VALUES LESS THAN (TO_DAYS('2024-03-01')),
  PARTITION p2024_03 VALUES LESS THAN (TO_DAYS('2024-04-01')),
  PARTITION p_futuro VALUES LESS THAN MAXVALUE
);

4. Estrategias de Divisao

RANGE (Intervalo)

sql
-- Ideal para dados temporais
PARTITION BY RANGE (data_criacao)
-- Cada particao = 1 mes/ano/dia
-- Pruning excelente em range queries
-- Suporta DROP PARTITION (instantaneo)

LIST (Lista de Valores)

sql
-- Ideal para categorias fixas
PARTITION BY LIST (regiao)
  PARTITION norte VALUES IN ('AM', 'PA')
  PARTITION sul   VALUES IN ('RS', 'SC')
-- Pruning por valor exato
-- Bom para multi-tenancy

HASH (Distribuicao Uniforme)

sql
-- Distribui uniformemente
PARTITION BY HASH (cliente_id)
  PARTITIONS 8
-- NAO suporta pruning por range
-- Bom para balancear I/O
-- Sem suporte a DROP PARTITION

RANGE-HASH Composto

sql
-- PostgreSQL: sub-particoes
PARTITION BY RANGE (ano)
  SUBPARTITION BY HASH (regiao_id)
-- Range pruning + distribuicao
-- Complexidade mais alta
-- Casos de uso muito grandes

5. O DELETE Instantaneo

A vantagem mais subestimada do particionamento por RANGE temporal: deletar um mes inteiro de dados antigos leva milissegundos em vez de horas — e nao gera nenhuma fragmentacao de indice.

delete-vs-drop.sql
-- SEM particao: deletar 1 mes de logs (250 milhoes de linhas)
DELETE FROM logs_evento
WHERE data_log < '2023-01-01';
-- Tempo: 4-8 horas, lock na tabela, fragmenta indices, enorme redo log

-- COM particao: drop atomico de uma particao inteira
ALTER TABLE logs_evento DETACH PARTITION logs_2022_12;
DROP TABLE logs_2022_12;
-- Tempo: < 100ms, zero fragmentacao, zero redo log, zero impacto em outras particoes

-- MySQL equivalente:
ALTER TABLE logs_evento DROP PARTITION p2022_12;
-- Instantaneo tambem

Politica de retencao automatica: Com partition pruning, um job agendado pode fazer ALTER TABLE ... DROP PARTITION toda primeira madrugada do mes para manter apenas os ultimos 12 meses — sem impacto operational, sem janela de manutencao.

6. Quando Particionar

Particione quando...

  • Tabela tem mais de 100 milhoes de linhas e queries sempre filtram por um campo de data ou categoria.
  • Voce precisa de politica de retencao — deletar dados antigos periodicamente.
  • Tabela tem hot data (ultimos 3 meses) e cold data (historico) — particoes podem estar em tablespaces de storage diferente.
  • Manutenção como VACUUM / ANALYZE demora demais em tabela monolitica.
  • Necessidade de paralelismo de queries — cada particao pode ser processada em paralelo.

Nao particione quando...

  • Tabela tem menos de 50 milhoes de linhas — overhead de particao supera o beneficio.
  • Queries raramente filtram pela coluna de particao — voce so adiciona complexidade sem pruning.
  • Coluna de particao tem baixa cardinalidade (ex: booleano, status com 3 valores).
  • JOINs frequentes entre particoes — pode ser mais lento que uma tabela unica com bom indice.

7. Topicos Avancados

Indices em Particoes

No PostgreSQL, indices sao locais — criados por particao. Um indice global em tabela particionada e chamado de partitioned index. Indices locais sao mais eficientes para queries com pruning; indices globais permitem unicidade global (ex: PK que inclua a coluna de particao).

Particoes Default

No PostgreSQL 11+, voce pode criar uma particao DEFAULT que recebe linhas que nao se encaixam em nenhuma particao definida. Essencial para evitar erros em inserts com valores nao previstos. Porem, a existencia da particao default degrada o pruning em algumas situacoes — meça antes de usar.

Partition Exchange (MySQL)

MySQL permite carregar dados em uma tabela comum (sem lock) e depois trocar atomicamente com uma particao via ALTER TABLE ... EXCHANGE PARTITION. Tecnica usada para bulk-loading em producao — sem downtime, sem lock na tabela principal durante o carregamento.

Tablespace por Particao

PostgreSQL e Oracle permitem criar particoes em tablespaces diferentes. Dados quentes (ultimos 3 meses) em SSD NVMe; dados frios (ano passado) em HDD mais barato; dados historicos (2+ anos) em armazenamento por objeto (ex: S3 via FDW). Tiered storage nativo sem mover a tabela.

Armadilha de JOINs: JOINs entre tabelas particionadas por chaves diferentes podem resultar em Partition Wise Join (eficiente) ou em um produto cartesiano de planos (catastrofico). Sempre verifique o EXPLAIN de queries com JOINs envolvendo tabelas particionadas antes de ir para producao.

8. Ferramentas e Referencias

pg_partman

Extensao PostgreSQL que automatiza a criacao de novas particoes (ex: criar a particao do proximo mes automaticamente) e a remocao de particoes antigas segundo politica configuravel. Essencial para tabelas de series temporais em producao.

MySQL Partitioning Guide

Documentacao oficial do MySQL sobre particionamento. Cobre RANGE, LIST, HASH, KEY e subparticionamento. Inclui limitacoes importantes: sem foreign keys em tabelas particionadas, coluna de particao deve estar na PRIMARY KEY.

TimescaleDB

Extensao PostgreSQL open-source especializada em time-series que usa particionamento por tempo (hypertables) automaticamente. Adiciona compressao nativa, continuous aggregates e data tiering. Transparente para SQL — o mesmo codigo funciona.

Citus (PostgreSQL)

Extensao que transforma PostgreSQL em banco distribuido usando sharding + particionamento. As particoes sao distribuidas entre nos workers automaticamente. Mantido pela Microsoft, disponivel como Azure Cosmos DB for PostgreSQL.

AWS RDS Partitioning

RDS Aurora MySQL e Aurora PostgreSQL suportam particionamento nativo com as mesmas sintaxes das versoes open-source. Aurora adiciona automatic failover sem perder particoes — storage separado do compute.

EXPLAIN ANALYZE

Principal ferramenta para validar se o pruning esta funcionando. No plano de execucao, particoes nao usadas nao aparecem. Se voce ve todas as particoes no plano, sua query nao esta aproveitando o particionamento.
