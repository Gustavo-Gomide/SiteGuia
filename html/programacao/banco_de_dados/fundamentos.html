Fundamentos de Banco de Dados

Fundamentos de Banco de Dados Relacionais e Transacionais

Uma trilha progressiva do básico ao avançado: persistência, modelagem, MER, relacional, normalização, desnormalização e ACID.

Este módulo reúne os principais conceitos para entender como dados são armazenados, organizados e mantidos consistentes. A ideia é ir do conceito (o que o sistema representa) até a implementação (tabelas, chaves e transações).

Conceitos Fundamentais

Um Banco de Dados (BD) é uma coleção organizada de informações estruturadas armazenadas eletronicamente. Seu objetivo principal é resolver o problema da Persistência: garantir que os dados sobrevivam após o encerramento do programa ou desligamento da máquina, ao contrário da memória RAM.

Segundo Silberschatz, Korth e Sudarshan, um sistema de banco de dados combina dados persistentes, mecanismos de consulta e controle de concorrência para manter consistência e disponibilidade em escala.

Dados vs. Informação

Embora usados como sinônimos, na computação eles são distintos:

Dados

São fatos brutos, números ou caracteres sem contexto.

2500
"Maria"
true
-- Sozinhos, isso não significa nada.

Informação

São dados processados, organizados e com contexto.

-- Com contexto:
O salário de Maria é R$ 2.500 e ela está ativa.
-- Agora tem valor para tomada de decisão.

O que é um SGBD?

O Sistema Gerenciador de Banco de Dados (SGBD / DBMS) é o software que interage com o usuário e com o armazenamento físico dos dados.

A Analogia da Biblioteca

Imagine uma biblioteca gigante.
O Banco de Dados são as estantes cheias de livros.
O SGBD é o bibliotecário.

Você (o usuário) não sai correndo pelas estantes bagunçando tudo. Você pede ao bibliotecário: "Quero o livro X". Ele sabe onde está, verifica permissões e te entrega. O SGBD garante organização, segurança e acesso eficiente.

Funções vitais do SGBD

CRUD

Criar, ler, atualizar e deletar dados de forma segura e eficiente.

Segurança

Controlar quem pode ver ou alterar quais dados até o nível de coluna.

Concorrência

Garantir que duas pessoas não alterem o mesmo registro ao mesmo tempo de forma conflitante.

Backup e Recuperação

Recuperar dados após falhas de hardware, erros humanos ou desastres.

Tipos de Dados

Dados Estruturados

Seguem formato rígido e pré-definido pense em planilhas ou tabelas SQL.

-- SQL (Tabela)
| ID | Nome | Idade |
|----|------|-------|
| 01 | Ana  |  25   |
| 02 | João |  30   |

Dados Semi/Não Estruturados

Não têm formato fixo textos, imagens ou documentos flexíveis como JSON.

// NoSQL (JSON/Documento)

  "nome": "Ana",
  "skills": ["Java", "SQL"],
  "ativo": true
}

Modelagem de Dados

Modelagem de Dados é o processo de criar uma representação visual de um sistema de informação. É como desenhar a planta de uma casa antes de assentar os tijolos o objetivo é comunicar regras de negócio e definir a estrutura antes de escrever SQL.

Os 3 Níveis de Abstração

A modelagem desce do mais abstrato (humano) para o mais concreto (máquina):

Modelo Conceitual

Público: cliente / gestor de negócios.
Foca no "O quê". Representa entidades e relacionamentos sem detalhes técnicos. Usa o Diagrama Entidade-Relacionamento (DER). 

[Cliente] --- COMPRA --- [Produto]

Modelo Lógico

Público: desenvolvedores / arquitetos.
Foca no "Como". Define tabelas, colunas, tipos genéricos, PKs e FKs. 

Cliente (ID_Cliente PK, Nome)
Produto (ID_Produto PK, Preço)
Pedido  (ID_Pedido, ID_Cliente FK)

Modelo Físico

Público: DBA / SGBD.
É a implementação final scripts SQL com tipos específicos do banco escolhido. 

CREATE TABLE clientes (
  id   INT AUTO_INCREMENT PRIMARY KEY,
  nome VARCHAR(100) NOT NULL
);

Por que não pular etapas?

Sem Modelagem

  • Dados redundantes (repetidos sem necessidade).
  • Inconsistência cliente com 2 endereços diferentes em lugares diferentes.
  • Dificuldade de manutenção e colunas sem significado claro.

Com Modelagem

  • Visão clara do negócio antes do código.
  • Padronização de nomes e estrutura.
  • Documentação viva do sistema para novos desenvolvedores.

Modelo Entidade-Relacionamento (MER)

O MER é o padrão mais usado para modelagem conceitual. Ele ignora detalhes técnicos e foca em descrever quais objetos existem no sistema e como eles interagem. O desenho resultante é o Diagrama Entidade-Relacionamento (DER).

Os Blocos de Construção

Entidade

Representa um objeto do mundo real ou conceito (Pessoa, Carro, Pedido). É o "substantivo" do domínio.

Representada por retângulo.

Atributo

Representa uma característica da entidade (Nome, Preço, Data). É o "adjetivo" que descreve a entidade.

Representado por elipse.

Relacionamento

Representa a associação entre entidades (compra, pertence, realiza). É o "verbo" do modelo.

Representado por losango.

Cardinalidade

Define o número de ocorrências em um relacionamento responde "Quantos?"

1 : 1 Um para Um

Cada item se liga a apenas um do outro lado. Ex: um motorista possui uma única CNH.

1 : N Um para Muitos

O mais comum. Um item se liga a vários, mas os vários se ligam a um. Ex: uma mãe tem vários filhos, mas cada filho tem uma única mãe biológica.

N : N Muitos para Muitos

Vários de um lado se conectam com vários do outro. Ex: um aluno cursa várias matérias e uma matéria tem vários alunos.

Leitura do diagrama: "Um Cliente (1) faz vários Pedidos (N), mas cada Pedido pertence a apenas um Cliente."

Modelo Relacional

O Modelo Relacional é a implementação técnica do modelo conceitual. Aqui, losangos e retângulos viram Tabelas. A conexão entre tabelas acontece por meio das Chaves.

Anatomia de uma Tabela

-- Tabela: Usuarios
+----+------+--------------+
| id | nome | email        |
+----+------+--------------+   linhas (tuplas)
|  1 | Ana  | ana@email   |
|  2 | Beto | beto@email  |
+----+------+--------------+
  
colunas (atributos)
-- id = Chave Primária (PK): identifica unicamente cada linha

Chave Primária (PK)

Identificador único de cada linha. Não se repete e não pode ser nulo.
Ex: CPF, ID numérico autoincremental, número de matrícula.

Chave Estrangeira (FK)

Coluna que guarda a PK de outra tabela, criando o elo de ligação entre as duas.
Ex: em Pedidos, a coluna cliente_id referencia Clientes.id.

Regras de Tradução MER Relacional

Relacionamento 1 : N

Exemplo: um Cliente (1) tem vários Pedidos (N).
Regra: a tabela do lado "N" (Pedidos) ganha uma coluna FK apontando para o lado "1" (Clientes). 

-- Clientes
id | nome
 1 | Ana

-- Pedidos (FK aponta para Clientes)
id  | cliente_id | total
100 |          1 | R$ 50

Relacionamento N : N

Exemplo: um Aluno cursa várias Matérias e uma Matéria tem vários Alunos.
Regra: criar uma Tabela Associativa contendo as FKs das duas tabelas. 

-- Tabela Associativa: Matriculas
aluno_id (FK) | materia_id (FK)
            1 |              5
            1 |              8
            2 |              5

Normalização

A Normalização é um conjunto de regras (Formas Normais) aplicadas às tabelas para evitar dois vilões: redundância (dados repetidos) e anomalias (erros lógicos ao inserir, atualizar ou deletar dados).

Por que Normalizar?

Se uma tabela mistura [Cliente, Endereço, Pedido] tudo junto, surgem problemas:

Anomalia de Inserção

Não consigo cadastrar um cliente novo se ele não tiver feito nenhum pedido ainda.

Anomalia de Atualização

Se o cliente muda de endereço, atualizo várias linhas. Se esquecer uma, o banco fica inconsistente.

Anomalia de Exclusão

Se apago o único pedido de um cliente, posso perder os dados cadastrais do próprio cliente junto.

Primeira Forma Normal (1FN)

Regra Atomicidade: cada célula deve conter apenas um valor (não listas, não múltiplos valores).

Violação da 1FN

Cliente | Telefones
Ana     | 9999-1111, 8888-2222
                 dois valores em uma célula!

Em conformidade com 1FN

Cliente | Telefone
Ana     | 9999-1111
Ana     | 8888-2222
-- Uma linha por valor atômico.

Segunda Forma Normal (2FN)

Regra Dependência Funcional Total: todo atributo não-chave deve depender da chave primária inteira. Aplica-se quando a PK é composta (2+ colunas). 

--  Violação 2FN: PK composta = (ID_Pedido + ID_Produto)
| ID_Ped | ID_Prod | Qtd | Nome_Produto |
|    10  |    50   |  2  | Mouse        |
-- "Nome_Produto" depende só de ID_Produto, não da PK inteira!

--  Solução: mover Nome_Produto para tabela Produtos
Produtos: (ID_Produto PK, Nome_Produto)
Itens_Pedido: (ID_Pedido, ID_Produto, Qtd)   ✔ 2FN

Terceira Forma Normal (3FN)

Regra Sem Dependência Transitiva: atributos normais não devem depender de outros atributos normais (só da chave primária). 

--  Violação 3FN
| ID_Entrega | CEP   | Cidade    |
|         1  | 01000 | São Paulo |
-- "Cidade" depende de "CEP" que depende de "ID_Entrega" — dependência transitiva!

--  Solução: criar tabela de CEPs
CEPs: (CEP PK, Cidade, Estado)
Entregas: (ID_Entrega PK, CEP FK, ...)   ✔ 3FN

Desnormalização

A Desnormalização é o processo de adicionar redundância intencional. Em vez de dados espalhados em várias tabelas normalizadas, duplicamos informações em uma estrutura única para reduzir o custo de leitura.

Atenção: desnormalizar é estratégia consciente, não bagunça. Só faz sentido com um motivo claro de performance.

Em sistemas transacionais, desnormalização sem governança costuma gerar divergência entre cópias do mesmo dado. Defina política de atualização e auditoria antes de aplicar esse padrão.

O Custo dos JOINs

Em bancos normalizados, juntar tabelas tem custo. Exemplo: montar um recibo de pedido completo.

Normalizado Consulta cara

-- Visita várias tabelas com JOINs
SELECT ...
FROM Pedidos P
JOIN Clientes C ON P.cli_id = C.id
JOIN Enderecos E ON C.end_id = E.id
JOIN Itens I ON P.id = I.ped_id;
-- Melhor para escrita e integridade

Desnormalizado Leitura rápida

-- Dados já "prontos" na tabela
SELECT *
FROM Pedidos_Completo
WHERE id = 10;
-- Melhor para leitura e relatórios

O Preço da Velocidade

Benefícios

  • Velocidade de leitura (SELECT) muito maior.
  • Queries mais simples menos JOINs.
  • Menos uso de CPU por consulta.

Custos

  • Escrita (UPDATE) mais lenta atualiza várias cópias.
  • Risco de inconsistência se uma cópia ficar desatualizada.
  • Maior uso de espaço em disco.

Onde Usar Cada Estratégia

OLTP Online Transaction Processing

Sistemas do dia a dia: caixa, login, cadastro.
Foco: escrita rápida e integridade dos dados.
Estratégia: mais normalizado.

OLAP Online Analytical Processing

Relatórios, dashboards, data warehouses.
Foco: leitura massiva e agregação histórica.
Estratégia: mais desnormalizado.

Propriedades ACID

Para um banco ser confiável, ele precisa suportar transações. Uma transação é uma sequência de operações tratada como um bloco único. O acrônimo ACID define as regras que essas transações devem obedecer.

O Exemplo de Ouro: Transferência Bancária

Transferir R$ 100 não é uma operação; são pelo menos duas:

-- Passo 1: débito
UPDATE conta SET saldo = saldo - 100 WHERE id = 1;

-- Passo 2: crédito
UPDATE conta SET saldo = saldo + 100 WHERE id = 2;

-- O que acontece se a luz acabar entre os dois passos?

A Atomicidade

"Tudo ou nada."
Se tudo funcionar, confirma (COMMIT). Se algo falhar, desfaz tudo (ROLLBACK). Nunca fica na metade.

C Consistência

"Respeite as regras."
O banco deve ir de um estado válido para outro, respeitando todas as constraints (chaves, tipos, CHECK), nunca ficando em estado inválido.

I Isolamento

"Não espie."
Transações concorrentes não devem interferir uma na outra. Cada transação enxerga os dados como se fosse a única rodando (locks, MVCC).

D Durabilidade

"Gravado na pedra."
Após um COMMIT, o resultado é persistido no disco (via WAL) mesmo que o sistema caia imediatamente depois. Não se perde. 

-- Transação completa em SQL
START TRANSACTION;

UPDATE conta SET saldo = saldo - 100 WHERE id = 1;  -- débito
UPDATE conta SET saldo = saldo + 100 WHERE id = 2;  -- crédito

-- Se chegou aqui sem erros: confirma tudo (Atomicidade + Durabilidade)
COMMIT;

-- Se algo deu errado: desfaz tudo automaticamente (Atomicidade)
-- ROLLBACK;

-- Verificar saldos (Consistência + Isolamento)
SELECT id, saldo FROM conta WHERE id IN (1, 2);
