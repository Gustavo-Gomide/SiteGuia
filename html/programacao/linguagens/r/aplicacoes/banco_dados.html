R para Banco de Dados

DBI como interface unificada, dbplyr para SQL gerado automaticamente, pool para connection pooling, e conectores para PostgreSQL, MongoDB, Redis e Apache Spark.

R tem uma das arquiteturas mais elegantes para acesso a bancos de dados: a camada DBI abstrai completamente o driver específico, enquanto o dbplyr permite escrever manipulações de dados em dplyr e traduzir automaticamente para SQL otimizado — sem escrever SQL manualmente.

DBI — Interface Unificada

O pacote DBI define uma interface padrão para conexão com bancos de dados relacionais em R. Qualquer driver (RPostgres, RSQLite, RMySQL, odbc) implementa a mesma API, tornando o código portável entre backends.

dbi-basico.R — conexão, query e fechamento
library("DBI")

# ---- SQLite (banco local, sem servidor) ----
library("RSQLite")
con_sqlite <- dbConnect(SQLite(), dbname = "meu_banco.db")

# ---- PostgreSQL ----
library("RPostgres")
con_pg <- dbConnect(
  Postgres(),
  host     = Sys.getenv("PG_HOST"),
  dbname   = Sys.getenv("PG_DB"),
  user     = Sys.getenv("PG_USER"),
  password = Sys.getenv("PG_PASS"),
  port     = 5432L
)

# ---- Operações DBI padronizadas ----
dbListTables(con_pg)                          # listar tabelas
dbExistsTable(con_pg, "vendas")               # verificar existência

# Query segura com parâmetros (previne SQL injection)
resultado <- dbGetQuery(con_pg,
  "SELECT * FROM vendas WHERE regiao = $1 AND data >= $2",
  params = list("Sul", "2024-01-01")
)

# Escrita de tabela
dbWriteTable(con_pg, "resultado_analise", df_resultado, overwrite = TRUE)

# Transação explícita
dbWithTransaction(con_pg, {
  dbExecute(con_pg, "UPDATE clientes SET ativo = FALSE WHERE ultimo_acesso < $1",
            params = list(Sys.Date() - 365))
})

# Fechar conexão — SEMPRE ao final
dbDisconnect(con_pg)

Nunca interpole variáveis diretamente na string SQL. Use sempre parâmetros params = list(...) no dbGetQuery(). Interpolação com paste() é vulnerável a SQL injection, mesmo em análises internas.

dbplyr — dplyr sobre Banco de Dados

dbplyr permite usar a sintaxe familiar do dplyr em tabelas de banco de dados. As operações são preguiçosas — acumulam-se sem executar — e são traduzidas para SQL apenas quando você solicita os dados com collect().

dbplyr-workflow.R
library("dplyr")
library("dbplyr")
library("DBI")

# Referência lazy para tabela no banco
vendas_db <- tbl(con_pg, "vendas")
clientes_db <- tbl(con_pg, "clientes")

# Pipeline dplyr — NÃO executa ainda
query <- vendas_db |>
  filter(data >= as.Date("2024-01-01")) |>
  inner_join(clientes_db, by = "cliente_id") |>
  group_by(regiao, mes = month(data)) |>
  summarise(total = sum(valor, na.rm = TRUE), .groups = "drop") |>
  arrange(desc(total))

# Ver SQL gerado (sem executar)
show_query(query)
# <SQL>
# SELECT regiao, EXTRACT(MONTH FROM data) AS mes, SUM(valor) AS total
# FROM vendas
# INNER JOIN clientes ON vendas.cliente_id = clientes.cliente_id
# WHERE data >= '2024-01-01'
# GROUP BY regiao, mes
# ORDER BY total DESC

# Executar e trazer para R
resultado <- collect(query)   # tibble em memória

# Escrever resultado de volta ao banco
copy_to(con_pg, resultado, "vendas_mensais", overwrite = TRUE)

pool — Connection Pooling

Em aplicações Shiny ou APIs Plumber que servem múltiplos usuários simultâneos, abrir e fechar conexões por request é ineficiente. O pacote pool mantém um pool de conexões persistentes e as empresta sob demanda.

pool-shiny.R
library("pool")
library("RPostgres")

# Criado uma vez na inicialização do app
pool <- dbPool(
  drv       = Postgres(),
  host      = Sys.getenv("PG_HOST"),
  dbname    = Sys.getenv("PG_DB"),
  user      = Sys.getenv("PG_USER"),
  password  = Sys.getenv("PG_PASS"),
  minSize   = 2,   # conexões mantidas abertas
  maxSize   = 10   # máximo de conexões simultâneas
)

# onStop — fechar o pool quando o app encerrar
onStop(function() poolClose(pool))

# Usar o pool como se fosse uma conexão DBI normal
dados <- dbGetQuery(pool, "SELECT * FROM vendas LIMIT 100")

# dbplyr com pool
tbl(pool, "vendas") |>
  filter(valor > 1000) |>
  collect()

NoSQL: MongoDB e Redis

nosql.R — mongolite e redux
# mongolite — driver MongoDB para R
library("mongolite")
col <- mongo(collection = "usuarios", db = "minha_base",
             url = Sys.getenv("MONGO_URI"))

# Inserir documentos
col$insert(data.frame(nome = "Ana", nota = 9.2))

# Query com filtro MongoDB
resultado <- col$find('{"nota": {"$gt": 8}}')
# data.frame com documentos onde nota > 8

# Agregação
col$aggregate('[
  {"$group": {"_id": "$regiao", "media": {"$avg": "$nota"}}},
  {"$sort": {"media": -1}}
]')

# Update
col$update('{"nome": "Ana"}', '{"$set": {"ativo": true}}')

# redis com redux
library("redux")
r <- hiredis(host = "localhost", port = 6379L)
r$SET("chave", "valor")
r$GET("chave")      # "valor"
r$EXPIRE("chave", 3600L)   # TTL de 1 hora
# Cache de resultados pesados:
chave_cache <- digest::digest(params)
if (is.null(r$GET(chave_cache))) {
  resultado <- calcular_pesado(params)
  r$SET(chave_cache, serialize(resultado, NULL))
}

sparklyr — Apache Spark via R

sparklyr.R
library("sparklyr")
library("dplyr")

# Instalar Spark local (para desenvolvimento)
spark_install(version = "3.5")

# Conectar ao Spark
sc <- spark_connect(master = "local")   # ou "yarn", "mesos", URL do cluster

# Copiar dados R para Spark
vendas_spark <- copy_to(sc, vendas_df, "vendas", overwrite = TRUE)

# Usar dplyr — traduzido para Spark SQL
resultado <- vendas_spark |>
  filter(valor > 1000) |>
  group_by(regiao) |>
  summarise(total = sum(valor)) |>
  collect()   # traz resultado para R

# Ler Parquet do HDFS/S3
df_hdfs <- spark_read_parquet(sc, "vendas_parquet",
                               path = "s3://meu-bucket/vendas/")

# ML com sparklyr (ml_*)
modelo <- ml_random_forest(vendas_spark, mpg ~ ., type = "regression")

spark_disconnect(sc)
