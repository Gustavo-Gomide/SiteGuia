Lua - Dados e IA

Visao Geral

Lua em pipelines de dados e servicos de IA

Lua aparece em IA quando o foco e orquestracao leve, inferencia em edge, integracao com engines C/C++ e servicos de baixa latencia. Historicamente, teve papel central no Torch classico e hoje atua muito como camada de cola entre sistemas.

Mesmo quando o treinamento principal ocorre em Python, Lua pode ser excelente para preprocessamento, roteamento de inferencia, regras de negocio de ML e operacao em tempo real.

Conceitos Fundamentais

Uma boa analogia: em muitos projetos de IA, Lua e o "maestro" e nao necessariamente o "instrumentista" principal. Ele coordena ingestao, validacao, chamadas de modelo, cache e fallback.

Quatro blocos definem a arquitetura: coleta de dados, transformacao, inferencia e observabilidade. Quando esses blocos ficam desacoplados, voce ganha resiliencia e troca de modelo sem quebrar o sistema.

Onde Lua encaixa melhor

  • Edge inference com OpenResty (Nginx + LuaJIT).
  • Preprocessamento textual e regras de classificacao rapidas.
  • Automacao de ETL leve e validacao de esquema.
  • Sistemas embarcados com restricao de memoria.

Pipeline de Dados e Inferencia

Fluxo tipico: evento chega -> normalizacao -> feature extraction -> chamada de modelo -> pos-processamento -> log de decisao. Em Lua, corrotinas ajudam a manter alto throughput sem custo elevado de threads por requisicao.

Preprocessamento de texto e vetorizacao simplificada
local M = {}

function M.normalizar(txt)
  txt = txt:lower()
  txt = txt:gsub("[^%w%s]", " ")
  txt = txt:gsub("%s+", " ")
  return txt
end

function M.bag_of_words(txt)
  local freq = {}
  for token in txt:gmatch("%S+") do
    freq[token] = (freq[token] or 0) + 1
  end
  return freq
end

return M

Exemplos: API de Inferencia e Concorrencia

Exemplo de chamada HTTP para um modelo remoto, com timeout e fallback.

Client de inferencia com fallback
local http = require("resty.http")

local function inferir_sentimento(texto)
  local cli = http:new()
  cli:set_timeout(200)

  local res, err = cli:request_uri("http://modelo-interno/predict", {
    method = "POST",
    body = "{\"texto\":\"" .. texto .. "\"}",
    headers = { ["Content-Type"] = "application/json" }
  })

  if not res then
    -- fallback deterministico
    return { classe = "neutro", score = 0.0, erro = err }
  end

  return { classe = "positivo", score = 0.91, raw = res.body }
end

Para throughput maior, use corrotinas para processar lote de eventos sem bloquear o worker inteiro.

Corrotinas para pipeline concorrente
local fila = { "evento1", "evento2", "evento3" }

local worker = coroutine.create(function()
  for i, ev in ipairs(fila) do
    print("processando", i, ev)
    coroutine.yield()
  end
end)

while coroutine.status(worker) ~= "dead" do
  coroutine.resume(worker)
end

Casos Reais de Uso

  • Classificacao de requests em gateway (fraude, spam, risco).
  • Feature extraction para enviar a um servico de treino central.
  • A/B testing de regras de decisao com rollout gradual.
  • Inferencia em edge para reduzir custo de rede e latencia.
  • Sistema hibrido: Lua para roteamento + Python para treino pesado.

Padroes Avancados de MLOps com Lua

Boas praticas em producao

  • Versionar modelos por rota (`/v1/predict`, `/v2/predict`).
  • Logar feature e decisao para auditoria e retraining.
  • Aplicar circuit breaker quando o modelo remoto degrada.
  • Implementar cache de predicoes para entradas repetidas.
  • Criar sombra de trafego (shadow mode) para validar novo modelo.

Nunca coloque somente ML no caminho critico sem fallback. Em incidentes, uma regra simples e explicavel salva SLA.
