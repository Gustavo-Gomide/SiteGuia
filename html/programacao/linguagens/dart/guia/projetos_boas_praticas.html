Projetos e Boas Praticas em Dart

Dart idiomatico segue o guia Effective Dart publicado pelo Google, que define convencoes de nomenclatura, uso de tipos, documentacao e design de APIs. Para projetos Flutter maiores, o padrao Clean Architecture com camadas de dominio, dados e apresentacao promove testabilidade e manutencao. Esta secao cobre arquitetura, padroes de design naturais em Dart e os erros mais comuns a evitar.

A regra mais importante do Effective Dart: dart analyze deve passar com zero warnings em modo strict. Um codebase com warnings acumulados degradara rapidamente a qualidade e introduzira bugs silenciosos.

Effective Dart — Convencoes Fundamentais

Dart — Nomenclatura conforme Effective Dart
// CORRETO: UpperCamelCase para tipos
class    ProdutoService    {}
typedef  Predicado<T> = bool Function(T);
enum     StatusPedido     { pendente, aprovado, cancelado }

// CORRETO: lowerCamelCase para variaveis, funcoes, parametros
String   nomeCompleto;
void     calcularTotal() {}

// CORRETO: lowercase_com_underscore para arquivos, pacotes, imports
import   'package:minha_lib/produto_service.dart';

// CORRETO: _ para biblioteca privada (nao use __ duplo)
class  _Cache      {}
void   _parse()    {}

// ERRADO: usar SCREAMING_CAPS para constantes em Dart
// const TAXA_IMPOSTO = 0.1; // Java style — nao usar!
const  taxaImposto = 0.1;  // CORRETO em Dart

// CORRETO: doc comments com /// em vez de /**
/// Calcula o total com impostos inclusos.
///
/// Exemplo:
/// ```dart
/// final total = calcularComImposto(100.0);
/// ```
double calcularComImposto(double base) => base * (1 + taxaImposto);
Dart — Design de APIs idiomaticas
// Preferir getters a metodos get*() sem parametros
class Pedido {
  List<Item> get itens      => _itens.toList(); // imutavel externamente
  bool       get vazio      => _itens.isEmpty;
  int        get quantidade => _itens.length;
  double     get total     => _itens.fold(0, (s, i) => s + i.subtotal);
}

// Preferir nomes descritivos aos tipos
// Ruim: List<Usuario> getAll() — redundante o 'get' com getter
// Bom:
abstract class UsuarioRepository {
  Future<List<Usuario>> listar();              // verbo descritivo
  Future<Usuario?> buscarPorId(String id);  // encontrar ou null
  Future<void>     salvar(Usuario u);
  Future<void>     deletar(String id);
}

// Usar parametros nomeados para melhorar legibilidade na chamada
// Ruim: calcular(10, true, 2, false)
// Bom:
double calcular({
  required double valor,
  bool incluirImposto = false,
  int  parcelas      = 1,
  bool arredondar    = true,
}) { ... }

calcular(valor: 100, incluirImposto: true, parcelas: 3);

Arquitetura: Clean Architecture com Dart

Dart — Repository pattern com Result type (sem excecoes no dominio)
// lib/domain/entities/produto.dart — entidade pura (sem dependencias)
class Produto {
  final String id, nome;
  final double preco;
  const Produto({required this.id, required this.nome, required this.preco});
}

// lib/domain/repositories/produto_repository.dart — contrato
abstract interface class ProdutoRepository {
  Future<Result<List<Produto>>> listar();
  Future<Result<Produto>>       buscarPorId(String id);
}

// lib/domain/usecases/buscar_produto_usecase.dart
class BuscarProdutoUseCase {
  final ProdutoRepository _repo;
  const BuscarProdutoUseCase(this._repo);

  Future<Result<Produto>> execute(String id) async {
    if (id.isEmpty) return Result.failure('ID invalido');
    return _repo.buscarPorId(id);
  }
}

// lib/data/repositories/produto_repository_impl.dart — implementacao
class ProdutoRepositoryImpl implements ProdutoRepository {
  final ProdutoRemoteDataSource _remoto;
  final ProdutoLocalDataSource  _local;

  ProdutoRepositoryImpl(this._remoto, this._local);

  @override
  Future<Result<Produto>> buscarPorId(String id) async {
    try {
      final cached = await _local.buscar(id);
      if (cached != null) return Result.success(cached);
      final servidor = await _remoto.buscar(id);
      await _local.salvar(servidor);
      return Result.success(servidor);
    } on NotFoundException {
      return Result.failure('Produto nao encontrado');
    } catch (e) {
      return Result.failure('Erro de rede: $e');
    }
  }
}

// Result type simples (ou use package:fpdart/dartz)
sealed class Result<T> {
  const factory Result.success(T value)      = Success<T>;
  const factory Result.failure(String error) = Failure<T>;
}
class Success<T> extends Result<T> { const Success(this.value); final T value; }
class Failure<T> extends Result<T> { const Failure(this.error); final String error; }

Boas Praticas vs Anti-Padroes

Boas Praticas
  • Usar const para construtores e expressoes imutaveis em compile-time
  • Preferir final para todos os campos e variaveis locais que nao sao reatribuidas
  • Declarar tipos de retorno em todas as funcoes publicas
  • Usar sealed class + exhaustive switch para modelar estados finitos
  • Fechar StreamController e cancelar StreamSubscription no dispose
  • Usar Isolate.run() para processamento CPU-intensivo
  • Escrever testes para toda logica de dominio (use cases, repositories)
  • Expor apenas interfaces (abstract interface class) para inversao de dependencia
Anti-Padroes
  • Usar dynamic sem necessidade — perde null safety e checagem de tipos
  • Lancar excecoes como fluxo de controle normal — use Result ou Option
  • Realizar I/O em construtores de classe — use async factory ou inicializacao lazy
  • Compartilhar estado mutavel entre Isolates — viola o modelo de memoria isolado
  • Esquecer await em Futures (unawaited_futures lint captura isso)
  • Usar strings como IDs de tipo em vez de generics typesafe
  • Criar classes abstract com apenas metodos estaticos — use funcoes top-level
  • Misturar logica de UI com logica de negocios no mesmo arquivo

Design Patterns em Dart

Dart — Singleton, Factory, Observer e Command em Dart idiomatico
// Singleton — usando construtor factory com cache
class ConfiguracaoApp {
  static final _instance = ConfiguracaoApp._();
  ConfiguracaoApp._();
  factory ConfiguracaoApp() => _instance;

  String baseUrl = 'https://api.producao.com';
  bool   modoDebug = false;
}

// Factory Method — subtipo detereminado por data
abstract class Notificacao {
  void enviar(String mensagem);

  factory Notificacao(String canal) => switch (canal) {
    'email' => _EmailNotificacao(),
    'push'  => _PushNotificacao(),
    'sms'   => _SmsNotificacao(),
    _       => throw ArgumentError('Canal desconhecido: $canal'),
  };
}

// Observer via Stream (idiomatico Dart)
class EventBus {
  static final _ctrl = StreamController<Object>.broadcast();
  static Stream<T> on<T>() => _ctrl.stream.whereType<T>();
  static void emit(Object event) => _ctrl.add(event);
}

// Uso: ouvir so PedidoCriado
EventBus.on<PedidoCriado>().listen((e) => print('Pedido: ${e.id}'));
EventBus.emit(PedidoCriado('123'));

// Builder pattern — Dart usa named parameters em vez de Builder
final req = HttpRequest(
  url:    'https://api.com/dados',
  method: HttpMethod.post,
  headers: {'Authorization': 'Bearer token'},
  timeout: const Duration(seconds: 30),
);
