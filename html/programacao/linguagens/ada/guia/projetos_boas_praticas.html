Projetos Ada — Boas Práticas

Estruturar, Nomear e Organizar Projetos Ada Profissionais

Convenções de código, GPR files, testes unitários e padrões de qualidade industrial.

Escrever código Ada correto é apenas metade do trabalho. A outra metade é organizar projetos de forma que possam ser construídos, testados e mantidos por equipes ao longo de décadas — o horizonte de vida típico de um sistema Ada em aviação ou defesa. Esta página cobre as convenções de nomenclatura, a estrutura de arquivos com .gpr, testes com AUnit e os padrões de codificação da industria.

1. Convenções de Nomenclatura

Ada é case-insensitive mas a comunidade adota convenções fortes para manter a legibilidade:

2. GNAT Project Files (.gpr)

O GNAT Project File (.gpr) é o sistema de build do mundo Ada, compilado pela ferramenta gprbuild. Permite definir alvos de compilação, configurações por plataforma e dependências externas.

ada — default.gpr (projeto típico)
with "gnatcoll.gpr";  -- dependência externa

project Meu_Sistema is

   -- Modo: executável ou biblioteca
   for Main use ("src/main.adb");
   for Source_Dirs use ("src/**", "include");
   for Object_Dir use "obj";
   for Exec_Dir  use "bin";

   -- Configuração de compilação
   package Compiler is
      for Default_Switches ("Ada") use
         ("-gnatwa",   -- todos os warnings
          "-gnatVa",   -- todas as verificações de validade
          "-gnaty",    -- verificação de estilo
          "-O2",       -- otimização
          "-g");       -- informação de debug
   end Compiler;

   -- Linker
   package Linker is
      for Default_Switches ("Ada") use ("-lm");
   end Linker;

end Meu_Sistema;
terminal — build com gprbuild
# Build padrão
gprbuild -P default.gpr

# Build com modo específico (release)
gprbuild -P default.gpr -XBUILD=release

# Limpar artefatos de build
gprclean -P default.gpr

# Build paralelo (usa N threads)
gprbuild -P default.gpr -j4

4. Testes Unitários com AUnit

AUnit é o framework de testes unitários padrão para Ada, modelado no JUnit/xUnit. Inclui asserções, suites de teste e relatórios de cobertura via GNATcoverage.

ada — teste com AUnit
with AUnit.Assertions; use AUnit.Assertions;
with AUnit.Test_Cases; use AUnit.Test_Cases;
with Matematica;      use Matematica;

package body Teste_Matematica is

   procedure Testar_Quadrado (T : in out Test_Case'Class) is
   begin
      Assert (Quadrado (5) = 25,    "5² deve ser 25");
      Assert (Quadrado (0) = 0,     "0² deve ser 0");
      Assert (Quadrado (-3) = 9,   "(-3)² deve ser 9");
   end Testar_Quadrado;

   procedure Testar_Divisao_Por_Zero (T : in out Test_Case'Class) is
      Q, R : Integer;
   begin
      begin
         Dividir (Dividendo => 10, Divisor => 0,
                  Quociente => Q, Resto => R);
         Assert (False, "Esperava exceção Constraint_Error");
      exception
         when Constraint_Error => null;  -- correto!
      end;
   end Testar_Divisao_Por_Zero;

end Teste_Matematica;

5. Boas Práticas Ada

Use Subtipos com Range

Sempre que possível, defina o intervalo válido de valores com subtype X is Integer range 0 .. 100. O compilador e o runtime protegem contra valores fora do intervalo automaticamente.

Feche Blocos com Nomes

Use end Nome_Subprograma em vez de apenas end. Em programas grandes com múltiplos níveis de aninhamento, isso elimina ambiguidades e facilita a revisão de código.

Prefira in out a ponteiros

Ada permite modificar parâmetros com in out sem ponteiros. Use tipos de acesso (access) apenas quando a semântica de referência é genuinamente necessária (listas encadeadas, polimorfismo).

Ative Todos os Warnings

Use -gnatwa (all warnings) e -gnatVa (all validity checks). Em sistemas críticos, configure -gnatwae para tratar warnings como erros.

Nunca use pragma Suppress em sistemas críticos. Este pragma desativa verificações em tempo de execução (bounds checking, overflow). O guia DO-178C exige que verificações em tempo de execução estejam ativas ou que sua ausência seja justificada formalmente.
