Clojure: Bibliotecas e Frameworks

Pilhas modernas para backend, frontend e dados em Clojure

O ecossistema Clojure privilegia composicao. Em vez de um framework unico e monolitico, voce combina bibliotecas pequenas com contratos claros.

Escolher bibliotecas em Clojure e escolher arquitetura. A qualidade do sistema depende menos de "magia" e mais de fronteiras explicitas, dados simples e REPL-driven development.

Conceitos de selecao de stack

Pense em uma linha de montagem: cada estacao faz uma tarefa bem definida e entrega uma interface previsivel para a proxima. Em Clojure, Ring define a estacao HTTP, Reitit define roteamento e coercao, Integrant organiza ciclo de vida de componentes, e cada parte pode ser substituida sem reescrever o todo.

Criterios praticos

  • Baixo acoplamento entre camada HTTP, dominio e persistencia.
  • Interoperabilidade JVM sem adaptadores excessivos.
  • Observabilidade: logs estruturados, metricas e tracing.
  • Operacao simples em producao: startup previsivel e configuracao externa.

Como as pecas se encaixam

Ring modela requisicao e resposta como mapas imutaveis. Reitit decide o endpoint e aplica validacao/coercao. O dominio recebe dados ja normalizados, executa regras puras e delega I/O para adaptadores.

Categorias principais

Web API

Ring, Reitit, Pedestal, Muuntaja para HTTP, roteamento, middlewares e serializacao.

Persistencia

next.jdbc, HugSQL, HoneySQL, XTDB, Datomic para SQL, query-building e bancos orientados a tempo.

Estado e sistema

Integrant, Component, Mount para ciclo de vida de recursos e dependencia entre subsistemas.

Frontend

Reagent e Re-frame para SPA em ClojureScript com reatividade e arquitetura orientada a eventos.

Sintaxe e uso: backend tipico

(ns app.http
  (:require
    [reitit.ring :as ring]
    [muuntaja.core :as m]
    [reitit.ring.coercion :as rrc]
    [reitit.coercion.spec :as spec-coercion]))

(def router
  (ring/router
    [["/health"
      {:get {:handler (fn [_] {:status 200 :body {:ok? true}})}}]
     ["/users/:id"
      {:get {:parameters {:path {:id int?}}
             :handler (fn [req]
                        (let [id (get-in req [:parameters :path :id])]
                          {:status 200 :body {:id id}}))}}]]
    {:data {:muuntaja m/instance
            :coercion spec-coercion/coercion
            :middleware [rrc/coerce-request-middleware
                         rrc/coerce-response-middleware]}}))

(def app
  (ring/ring-handler router))

Esse arranjo separa infraestrutura de regras de negocio e reduz erros de contrato entre cliente e API com coercao automatica.

Quando usar e quando evitar

Use quando

  • Voce precisa evoluir arquitetura por composicao.
  • Ha alta concorrencia e necessidade de seguranca de estado.
  • O time trabalha com contratos de dados e testes de integracao.

Evite quando

  • O time depende fortemente de scaffolding magico de framework monolitico.
  • Nao existe maturidade para REPL e programacao funcional.
  • Ha pressao por stack mainstream sem espaco para treinamento.

Padroes avancados de ecossistema

Arquitetura orientada a dados + sistema declarativo

Integrant usa um mapa de configuracao para declarar componentes e dependencias. Isso permite boot parcial, testes por recorte de sistema e substituicao de adaptadores sem reescrever o dominio.

{:db/pool {:jdbc-url #env "JDBC_URL"}
 :http/server {:handler #ig/ref :http/handler :port 3000}
 :http/handler {:router #ig/ref :http/router :db #ig/ref :db/pool}}

Armadilha comum: misturar side effects (HTTP/DB) no nucleo de negocio. Em Clojure, mantenha regras puras e I/O nas bordas para maximizar testabilidade.
