C# — Projetos e Boas Praticas

Projetos e Boas Praticas em C#

Construcao de software .NET profissional: Clean Architecture para separacao de responsabilidades, DDD para modelagem do dominio, testes com xUnit/Moq, analise estatica com Roslyn analyzers, nullable reference types para eliminar NullReferenceExceptions, e convencoes de codigo adotadas na industria.

A maioria dos bugs em producao em C# vem de tres fontes: NullReferenceException, condicoes de corrida e SQL injection. As boas praticas modernas eliminam as duas primeiras em compile-time (nullable types + async correto) e a terceira via parametrizacao (EF Core / Dapper parametrizado).

Nullable Reference Types e Seguranca de Tipo

C# — nullable enable, null-safety patterns e required members
// Com <Nullable>enable</Nullable> no .csproj:
// string  = nao-nulavel (aviso se atribuir null)
// string? = nulavel (deve verificar antes de usar)

record Usuario
{
  public required string Nome    { get; init; }   // required: obrigatorio no construtor
  public required string Email   { get; init; }
  public string?          Apelido { get; init; }  // opcional
}

// Padroes seguros para nulabilidade
static string ObterNomeExibicao(Usuario? usuario)
{
  if (usuario is null)        return "Anonimo";
  return usuario.Apelido ?? usuario.Nome;  // ?? fallback
}

// Guard clauses com ArgumentNullException
class ServicoEmail(IEmailProvider provider)
{
  private readonly IEmailProvider _p = provider ?? throw new ArgumentNullException(nameof(provider));

  public void Enviar(string destinatario, string assunto, string corpo)
  {
    ArgumentException.ThrowIfNullOrWhiteSpace(destinatario);
    ArgumentException.ThrowIfNullOrWhiteSpace(assunto);
    _p.Send(destinatario, assunto, corpo);
  }
}

// Null-forgiving operator ! — use com cautela, apenas quando voce sabe
string? lido = Console.ReadLine();
string  garantido = lido!;  // suprime aviso — voce garante nao-nulo

Testes com xUnit e Moq

C# + xUnit + Moq — testes unitarios, mocking e dados parametrizados
using Xunit;
using Moq;
using FluentAssertions;

// Classe de teste — sem atributo de classe, apenas [Fact] e [Theory]
public class PedidoTests
{
  private readonly Mock<IProdutoRepository> _repoMock = new();
  private readonly Mock<IEventBus>         _busMock  = new();

  [Fact]
  public async Task CriarPedido_ProdutoValido_RetornaId()
  {
    // Arrange
    var produtoId = Guid.NewGuid();
    _repoMock.Setup(r => r.ExisteAsync(produtoId, It.IsAny<CancellationToken>()))
             .ReturnsAsync(true);
    var service = new PedidoService(_repoMock.Object, _busMock.Object);

    // Act
    var id = await service.CriarAsync(new CriarPedidoDto(produtoId, 2));

    // Assert
    id.Should().NotBeEmpty();
    _busMock.Verify(b => b.Publicar(It.IsAny<PedidoCriadoEvent>(), It.IsAny<CancellationToken>()),
                   Times.Once());
  }

  [Theory]
  [InlineData(0)]
  [InlineData(-1)]
  [InlineData(-999)]
  public async Task CriarPedido_QuantidadeInvalida_LancaValidacaoException(int qtd)
  {
    var service = new PedidoService(_repoMock.Object, _busMock.Object);
    var acao = () => service.CriarAsync(new CriarPedidoDto(Guid.NewGuid(), qtd));
    await acao.Should().ThrowAsync<ValidationException>();
  }
}

// Teste de integracao com WebApplicationFactory (testa endpoint HTTP real)
public class PedidosApiTests(WebApplicationFactory<Program> factory)
    : IClassFixture<WebApplicationFactory<Program>>
{
  [Fact]
  public async Task Post_PedidoValido_Retorna201()
  {
    var client = factory.CreateClient();
    var resp = await client.PostAsJsonAsync("/api/pedidos",
      new { ClienteId = Guid.NewGuid(), Itens = new[] { new { ProdutoId = Guid.NewGuid(), Qtd = 1 } } });
    resp.StatusCode.Should().Be(System.Net.HttpStatusCode.Created);
  }
}

Convencoes e Analise Estatica

Convencoes de Codigo

  • PascalCase: tipos, metodos, propriedades, eventos — PedidoService, CriarAsync()
  • camelCase: variaveis locais e parametros — clienteId, totalPedido
  • _camelCase: campos privados — _repository, _logger
  • IPrefixo: interfaces — IRepository, IEmailService
  • async suffix: metodos async — BuscarAsync(), CriarAsync()
  • file-scoped namespace: namespace MinhaApp.Services; no topo (C# 10+)
  • Expression body: metodos simples com => quando cabe em uma linha

Anti-Patterns Comuns

  • async void: exceto em handlers de eventos — excecoes sao perdidas silenciosamente
  • .Result / .Wait(): em codigo async causa deadlock em contextos sincronizados
  • catch (Exception) { } silencioso: nunca engolir excecoes sem log
  • new DbContext() em loop: DbContext nao e thread-safe e nao deve ser singleton
  • string.Format vs $"...": sempre prefira interpolacao — mais legivel e verificada em compile-time
  • Nao usar ConfigureAwait(false) em bibliotecas: pode causar deadlock em apps WinForms/WPF/ASP.NET
