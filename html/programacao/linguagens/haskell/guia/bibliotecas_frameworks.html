Visao Geral

Como escolher bibliotecas e frameworks em Haskell

Haskell oferece um ecossistema maduro para APIs, concorrencia, streaming, bancos de dados, validacao e interfaces. A chave e combinar abstrações fortes com bibliotecas pragmáticas para produzir software seguro e escalavel.

Nesta pagina, voce vai do mapa mental do ecossistema ate padroes avancados de arquitetura com foco em servidor, paralelismo, composicao de efeitos e manutencao de longo prazo.

Conceitos e Contexto

Pense nas bibliotecas Haskell como pecas de LEGO com encaixes rigorosos: tipos funcionam como o formato de cada peca. Quando os tipos "encaixam", a chance de erro em runtime cai drasticamente. Essa cultura de tipos fortes influenciou frameworks como Servant, Yesod e Persistent.

Historicamente, o ecossistema cresceu com foco academico e depois evoluiu para producao pesada em fintech, observabilidade e sistemas de baixo erro. Por isso, e comum ver documentacao com enfase em propriedades formais, leis de abstrações e comportamento previsivel sob carga.

Eixos de escolha

  • Produtividade inicial: Scotty e IHP reduzem tempo de setup.
  • Correcao por tipos: Servant e Opaleye elevam seguranca em contratos.
  • Performance: Warp, bytestring, text, stm e async.
  • Escalabilidade de codigo: ReaderT pattern, mtl/polysemy/effectful.

Como o Ecossistema se Organiza

Em projetos reais, voce normalmente combina camadas: HTTP server, roteamento, validacao, acesso a dados, observabilidade e execucao concorrente. O fluxo comum e: request HTTP - decodificacao - regra de negocio pura - IO controlado - resposta.

Esqueleto de composicao de camadas
-- Dominio puro (sem IO): regra testavel
module Domain.User where

newtype Email = Email String

validarEmail :: String -> Either String Email
validarEmail raw =
  if '@' `elem` raw
    then Right (Email raw)
    else Left "email invalido"

-- Camada de servidor: converte request/response e chama dominio
criarUsuarioHandler :: String -> IO String
criarUsuarioHandler raw =
  case validarEmail raw of
    Left err -> pure ("400: " ++ err)
    Right _  -> pure "201: usuario criado"

Esse modelo separa "o que" (dominio) de "como" (infra), facilitando testes, refatoracao e mudanca de framework sem reescrever regra de negocio.

Regra pratica

Para APIs modernas: Servant + Warp + aeson + persistent/esqueleto + async/stm + katip/co-log. Para prototipagem: Scotty. Para fullstack opinionado: IHP. Para app muito tipado end-to-end: Yesod.

Uso Pratico em API e Servidor

O exemplo abaixo mostra uma API tipada com Servant, um servidor Warp e um handler simples. O ganho principal e detectar inconsistencias de contrato em compilacao.

Servant + Warp: contrato de API tipado
{-# LANGUAGE DataKinds, TypeOperators #-}

import Network.Wai.Handler.Warp (run)
import Servant

type API =
       "health" :> Get '[JSON] String
  :<|> "users"  :> Capture "id" Int :> Get '[JSON] String

server :: Server API
server = health :<|> userById
  where
    health = pure "ok"
    userById uid = pure ("usuario #" ++ show uid)

main :: IO ()
main = do
  putStrLn "Servidor em :8080"
  run 8080 (serve (Proxy :: Proxy API) server)

Para bancos SQL, combine com Persistent/Esqueleto e mantenha transacoes no limite da camada de infraestrutura. Isso evita espalhar IO pela regra de negocio.

Concorrencia com async + STM
import Control.Concurrent.Async (mapConcurrently)
import Control.Concurrent.STM

processarLote :: [Int] -> IO [Int]
processarLote xs = mapConcurrently (
  x -> pure (x * 2)
  ) xs

contadorSeguro :: IO Int
contadorSeguro = do
  tv <- newTVarIO 0
  atomically $ do
    n <- readTVar tv
    writeTVar tv (n + 1)
  readTVarIO tv

Quando Usar e Quando Evitar

Excelente escolha

  • APIs criticas com contratos estaveis e validacao forte.
  • Sistemas com alta concorrencia e necessidade de previsibilidade.
  • Backends com longa vida util, foco em manutencao e baixo bug rate.
  • Times que valorizam testes de propriedade e modelagem de dominio.

Nao e ideal em todo cenario

  • Equipes sem disponibilidade para curva inicial funcional.
  • Prototipos de curtissimo prazo sem compromisso de manutencao.
  • Ambientes que exigem ecosistema especifico ausente em Haskell.
  • Apps GUI multiplataforma com forte dependencia de widgets nativos.

Nao confunda "tipagem forte" com lentidao no desenvolvimento. Em projetos longos, a economia de retrabalho tende a superar o custo inicial de aprendizagem.

Padroes Avancados

Em sistemas maiores, use padroes que isolam efeitos, controlam recursos e facilitam observabilidade: ReaderT pattern, bracket para lifecycle, structured logging, retry com politicas e filas internas com STM.

Checklist de producao

  • Timeout por operacao remota e cancelamento de async.
  • Pool de conexoes DB (resource-pool) e limites de concorrencia.
  • Metrica por endpoint e tracing distribuido.
  • Circuit breaker para dependencia externa instavel.
  • Testes property-based para invariantes de dominio.
Gestao de recurso com bracket
import Control.Exception (bracket)

abrir :: IO String
abrir = pure "conexao"

fechar :: String -> IO ()
fechar _ = putStrLn "conexao fechada"

usar :: String -> IO ()
usar c = putStrLn ("usando " ++ c)

main :: IO ()
main = bracket abrir fechar usar

Estrategia de Stack por Cenario

Em vez de escolher bibliotecas isoladas, defina um stack coerente por objetivo de negocio. Um stack de API transacional prioriza consistencia de contrato e banco; um stack de dados prioriza streaming e throughput; e um stack de CLI/tooling prioriza distribuicao simples e manutenibilidade.
