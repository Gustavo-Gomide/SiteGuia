Haskell — Projetos e Boas Práticas

Boas práticas em Haskell incluem o Newtype pattern para segurança de tipos sem custo em runtime, o AppM pattern com monad transformers para injeção de dependências, uso de Text em vez de String para performance, e estrutura de projetos com separação clara entre domínio e efeitos.

O princípio central: "make illegal states unrepresentable". Use o sistema de tipos para tornar estados inválidos impossíveis de expressar. Um campo que pode ou não existir é Maybe. Uma operação que pode falhar retorna Either Erro. Dois IDs que não deveriam ser trocados são newtype UserId = UserId Int e newtype ProductId = ProductId Int.

Newtype Pattern — Tipos Seguros sem Custo

Evitar confusão de IDs e primitivos com tipos distintos
{-# LANGUAGE GeneralizedNewtypeDeriving #-}
{-# LANGUAGE DerivingStrategies #-}

-- Newtype: zero runtime cost, máxima type safety
newtype UserId    = UserId    Int     deriving (Show, Eq, Ord)
newtype ProdutoId = ProdutoId Int     deriving (Show, Eq, Ord)
newtype Email     = Email     String  deriving (Show, Eq)
newtype Preco     = Preco     Double  deriving (Show, Num, Ord, Eq)

-- Agora estas funções são type-safe — impossível trocar os IDs
buscarPedidoDoUsuario :: UserId -> ProdutoId -> IO (Maybe Pedido)
buscarPedidoDoUsuario uid pid = undefined

-- Smart constructors — validação centralizada
mkEmail :: String -> Either String Email
mkEmail s
    | '@' `elem` s && '.' `elem` (dropWhile (/='@') s)
                 = Right (Email s)
    | otherwise  = Left $ "email inválido: " ++ s

mkPreco :: Double -> Either String Preco
mkPreco v
    | v >= 0    = Right (Preco v)
    | otherwise = Left "preço não pode ser negativo"

-- Usar com do-notation
criarUsuario :: String -> Double -> Either String (Email, Preco)
criarUsuario rawEmail precoBase = do
    email <- mkEmail rawEmail
    preco <- mkPreco precoBase
    return (email, preco)

-- Phantom types — segurança ainda maior
data Estado = Rascunho | Publicado | Arquivado

newtype Documento (estado :: Estado) = Documento String

publicar :: Documento 'Rascunho -> Documento 'Publicado
publicar (Documento conteudo) = Documento conteudo

-- arquivar só aceita documentos publicados
arquivar :: Documento 'Publicado -> Documento 'Arquivado
arquivar (Documento conteudo) = Documento conteudo

Repository Pattern e Injeção de Dependências

Type classes como interfaces + AppM para DI
-- Interface como type class
class Monad m => PedidoRepo m where
    salvarPedido  :: Pedido -> m PedidoId
    buscarPedido  :: PedidoId -> m (Maybe Pedido)
    listarPedidos :: UserId -> m [Pedido]

-- Implementação de produção (usa banco de dados)
newtype AppDB a = AppDB { runDB :: ReaderT Conexao IO a }
    deriving (Functor, Applicative, Monad, MonadReader Conexao, MonadIO)

instance PedidoRepo AppDB where
    salvarPedido pedido = do
        conn <- ask
        liftIO $ executarInsert conn pedido

    buscarPedido pid = do
        conn <- ask
        liftIO $ executarQuery conn pid

-- Implementação de teste (in-memory)
newtype AppTeste a = AppTeste { runTeste :: StateT (Map PedidoId Pedido) Identity a }
    deriving (Functor, Applicative, Monad)

instance PedidoRepo AppTeste where
    salvarPedido p = AppTeste $ do
        pid <- gets (PedidoId . Map.size)
        modify (Map.insert pid p)
        return pid
    buscarPedido pid = AppTeste $ gets (Map.lookup pid)
    listarPedidos uid = AppTeste $ gets (filter (\p -> pedidoUserId p == uid) . Map.elems)

-- Service layer — independente de implementação
processarPedido :: PedidoRepo m => UserId -> Pedido -> m (Either String PedidoId)
processarPedido uid pedido
    | pedidoValor pedido <= 0 = return $ Left "valor inválido"
    | otherwise = do
        pid <- salvarPedido pedido { pedidoUserId = uid }
        return $ Right pid

Decisoes de Arquitetura em Producao

Boas praticas tambem envolvem escolher conscientemente onde Haskell entrega mais valor. Em geral, ele e excelente no nucleo de regra de negocio, validacao de contratos, servicos de dados e processamento concorrente confiavel. Em camadas de UI nativa, costuma ser mais eficaz atuar por integracao.

Priorizar Haskell

  • API e servidor com requisitos de corretude.
  • Regras financeiras, compliance e trilha de auditoria.
  • Pipelines de dados reproduziveis e testaveis.

Integrar com outras stacks

  • Mobile/GUI com forte dependencia de SDK nativo.
  • Jogos com pipeline grafico especializado.
  • Embedded hard real-time de hardware restrito.

O pattern "make illegal states unrepresentable" atinge sua forma mais pura em Haskell: tipos de dados algébricos garantem que apenas estados válidos existem; phantom types adicionam restrições de fluxo em compile-time (só podemos publicar um Documento 'Rascunho); smart constructors centralizam validação. O resultado é código onde uma enorme classe de bugs simplesmente não pode existir — o compilador os rejeita antes mesmo de rodar.
