F# - Banco de Dados

Persistencia com F#

F# conversa com bancos relacionais e NoSQL usando toda a infraestrutura .NET, com vantagem extra na modelagem de dados e validacao em compilacao.

Com F#, seu modelo de dominio pode ser mais restrito que o schema do banco. Essa diferenca e saudavel: o banco guarda dados, o dominio protege regras.

Exemplo com Npgsql.FSharp

Consulta parametrizada e mapeamento seguro
open Npgsql.FSharp

type Cliente = {
    Id: int
    Nome: string
    Email: string
}

let conn = "Host=localhost;Database=app;Username=app;Password=segredo"

let buscarPorEmail email =
    conn
    |> Sql.connect
    |> Sql.query "select id, nome, email from clientes where email = @email"
    |> Sql.parameters [ "email", Sql.string email ]
    |> Sql.executeRow (fun read ->
        {
            Id = read.int "id"
            Nome = read.string "nome"
            Email = read.string "email"
        })

Uso de parametros evita SQL injection. O mapeamento explicito deixa clara a traducao banco-dominio.

Entity Framework Core em F#

EF Core funciona em F#, especialmente quando o time ja usa migracoes e padrao DbContext no ecossistema .NET.

Consulta simples com LINQ e Async
open Microsoft.EntityFrameworkCore

let buscarAtivos (ctx: AppDbContext) =
    task {
        return!
            ctx.Clientes
               .Where(fun c -> c.Ativo)
               .OrderBy(fun c -> c.Nome)
               .ToListAsync()
    }

Em F#, prefira projetar para DTOs/records no limite de aplicacao. Evita vazar entidades ORM para o dominio.

Transacoes e Concorrencia

  • Use transacoes curtas e com escopo explicito.
  • Adote chave de idempotencia em comandos externos.
  • Use controle otimista (versionamento) para evitar lost updates.
  • Nao misture I/O remoto demorado dentro da mesma transacao do banco.

Quando F# Brilha na Persistencia

Pontos fortes

  • Modelagem forte de comandos e eventos.
  • Validacao de dados antes de tocar no banco.
  • Composicao clara de fluxos com Result/Async.

Cuidados

  • Muitos samples de ORM estao em C# e exigem adaptacao.
  • Evite sobreengenharia funcional em CRUD simples.
  • Decisao de stack deve considerar maturidade do time.
