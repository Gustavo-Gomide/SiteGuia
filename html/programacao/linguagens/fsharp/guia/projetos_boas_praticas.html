F# - Projetos e Boas Praticas

Projetos F# Sustentaveis em Producao

F# escala muito bem quando o projeto trata dominio como primeira classe, define fronteiras claras e automatiza qualidade desde o primeiro commit.

A melhor pratica mais importante em F# e esta: modele estados validos no tipo. Isso reduz ramificacoes defensivas e converte erro de negocio em erro de compilacao.

Estrutura de Solucao

Estrutura recomendada para solucoes F#
/src
  /MeuProduto.Domain        // tipos, regras, funcoes puras
  /MeuProduto.Application   // casos de uso, orquestracao
  /MeuProduto.Infrastructure// banco, HTTP, fila, IO
  /MeuProduto.Api           // web host (Giraffe/Saturn)
/tests
  /MeuProduto.Domain.Tests
  /MeuProduto.Integration.Tests
/build
  build.fsx                 // FAKE

Separar responsabilidades permite testar o nucleo de negocio sem subir banco, fila ou servidor HTTP.

Modelagem de Dominio com Tipos

Evitar estados invalidos com DU e records
type Email = private Email of string

type PedidoStatus =
    | Criado
    | Pago of dataPagamento: System.DateTime
    | Cancelado of motivo: string

type Pedido = {
    Id: System.Guid
    ClienteEmail: Email
    Status: PedidoStatus
}

let cancelar motivo pedido =
    match pedido.Status with
    | Pago _ -> Error "pedido pago nao pode ser cancelado"
    | Cancelado _ -> Error "pedido ja cancelado"
    | Criado -> Ok { pedido with Status = Cancelado motivo }

Evite usar string solta para representar estados de dominio. Prefira tipos explicitos para impedir combinacoes invalidas.

Teste de Unidade e de Propriedade

FsCheck para validar invariantes
open FsCheck

let somaEhComutativa (a: int) (b: int) =
    (a + b) = (b + a)

Check.QuickThrowOnFailure somaEhComutativa

Estrategia recomendada

  • Unitarios no dominio (rapidos, sem I/O).
  • Integracao para adaptadores externos (banco, APIs, filas).
  • Contratos para endpoints publicos (OpenAPI e versionamento).

Logs, Metricas e Tracing

Projetos maduros em F# usam a pilha padrao do .NET para observabilidade: OpenTelemetry, Serilog e exporters para Prometheus/Grafana ou Azure Monitor.

Log estruturado com contexto de dominio
logger.LogInformation(
    "Pedido {PedidoId} mudou para status {Status}",
    pedido.Id,

CI/CD e Automacao

Pipeline de build com FAKE (conceitual)
Target.create "Restore" (fun _ -> DotNet.restore id "MinhaSolucao.sln")
Target.create "Build"   (fun _ -> DotNet.build id "MinhaSolucao.sln")
Target.create "Test"    (fun _ -> DotNet.test id "MinhaSolucao.sln")

open Fake.Core.TargetOperators
"Restore" ==> "Build" ==> "Test"
Target.runOrDefault "Test"

O beneficio de scriptar em F# e compartilhar tipos, utilitarios e convencoes entre automacao e produto.

Armadilhas Comuns

Mutabilidade sem criterio

Usar mutable em excesso reduz previsibilidade e dificulta paralelizacao.

Excecoes como fluxo

Para falhas esperadas, prefira Result e compose com bind.

Acoplamento ao framework

Nao coloque regras de negocio dentro do handler HTTP. Mantenha o dominio independente.
