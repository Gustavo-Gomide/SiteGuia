Lisp - Projetos e Boas Práticas

Engenharia de software em Common Lisp para código que continua vivo

Projetos Lisp sustentáveis não dependem de truques de macro nem de genialidade individual. Eles dependem de packages claros, sistemas bem descritos, testes reproduzíveis e disciplina no uso do runtime interativo.

O maior risco em Lisp não é “ter linguagem demais”; é usar liberdade demais sem construir fronteiras técnicas que mantenham o sistema legível para o próximo mês e para a próxima equipe.

Fundamentos de projeto

Um projeto Common Lisp maduro funciona como um laboratório bem organizado. Reagentes ficam identificados, instrumentos têm lugar fixo e procedimentos podem ser repetidos. Em software, isso significa packages explícitos, ASDF descrevendo a ordem de carregamento, separação entre domínio e infraestrutura, e um REPL usado para acelerar iteração, não para substituir processo.

Princípios que fazem diferença em Lisp
  • Packages pequenos e intencionais para limitar superfície pública.
  • Sistemas ASDF claros para que build e load não dependam de ordem manual.
  • Domínio separado de I/O para manter teste e raciocínio baratos.
  • Uso criterioso de macros apenas quando a abstração realmente melhora o modelo.
  • Condições e restarts quando falhas precisarem de recuperação estruturada.

Estrutura recomendada de diretórios

Organizar por domínio é mais útil do que espalhar arquivos em pastas genéricas como “utils”, “helpers” e “misc”. Em Lisp, essa disciplina é ainda mais importante porque a linguagem permite abstrações muito densas; sem fronteiras claras, a base degrada rapidamente.

Estrutura sugerida
meu-sistema/
	package.lisp
	meu-sistema.asd
	README.md
	src/
		core.lisp
		domain/
			pedido.lisp
			faturamento.lisp
		adapters/
			http.lisp
			db.lisp
			queue.lisp
	test/
		suite.lisp
		pedido-test.lisp
	scripts/
		dev.lisp
		build-image.lisp
Por que esse layout funciona
  • package.lisp concentra a API pública de cada módulo.
  • .asd documenta dependências e ordem de carga de forma declarativa.
  • src/domain mantém regra de negócio longe de detalhes operacionais.
  • test vira artefato de primeira classe, não uma coleção solta de scripts.

Fluxo de desenvolvimento na prática

O fluxo REPL-driven funciona melhor quando é apoiado por estrutura. Você edita um arquivo, recompila apenas a unidade alterada, roda testes rápidos, inspeciona objetos reais e ajusta o comportamento sem reiniciar o mundo inteiro. O benefício aparece quando esse ciclo é repetível por qualquer pessoa do time.

Lisp/Common Lisp
; scripts/dev.lisp
(ql:quickload '(:meu-sistema :fiveam))

(in-package #:cl-user)

(defun recarregar ()
	(asdf:load-system :meu-sistema :force t))

(defun testar ()
	(asdf:test-system :meu-sistema))

(defun reiniciar ()
	(recarregar)
	(testar))

Esse script reduz improviso. Em vez de cada desenvolvedor lembrar uma sequência diferente de comandos, o próprio projeto oferece um protocolo mínimo de trabalho.

Padrões de organização mais comuns

Biblioteca reutilizável

API pública pequena, documentação de exports, testes de contrato e portabilidade entre implementações.

Aplicação web

Separação entre rotas, handlers, domínio e persistência; inicialização explícita do servidor e shutdown controlado.

CLI ou automação

Entry point simples, tratamento claro de erro e geração previsível de imagem executável quando necessário.

Sistema com macros

Macros isoladas em módulos próprios, macroexpansão fácil de inspecionar e documentação do contrato sintático.

Uso prático: packages, testes e fronteiras

Lisp/Common Lisp
; package.lisp
(defpackage #:meu-sistema.domain.pedido
	(:use #:cl)
	(:export #:novo-pedido #:pedido-total))

(defpackage #:meu-sistema.adapters.db
	(:use #:cl)
	(:import-from #:meu-sistema.domain.pedido #:novo-pedido)
	(:export #:salvar-pedido))

; domain/pedido.lisp
(in-package #:meu-sistema.domain.pedido)

(defstruct pedido itens)

(defun pedido-total (pedido)
	(reduce #'+ (pedido-itens pedido) :initial-value 0))

; test/pedido-test.lisp
(fiveam:test total-de-pedido
	(fiveam:is (= 60
		(pedido-total (make-pedido :itens '(10 20 30))))))

Note a regra central: o domínio não conhece armazenamento, HTTP ou fila. Esse isolamento reduz o custo de teste, facilita refatoração e torna mais óbvio onde otimizações ou efeitos colaterais realmente pertencem.

Quando usar certas práticas e quando evitar exageros

Boas decisões frequentes
  • Manter packages pequenos e exports mínimos.
  • Encapsular boot do sistema em funções previsíveis.
  • Executar testes automatizados fora do REPL pessoal do desenvolvedor.
  • Tratar macro como ferramenta de modelagem, não de ornamentação.
Exageros que custam caro
  • Criar DSL para problema que uma função comum resolveria melhor.
  • Depender de estado global implícito espalhado por vários arquivos.
  • Esconder ordem de carga ou inicialização em scripts não documentados.
  • Testar apenas manualmente porque “o REPL já mostra tudo”.

Disciplina avançada: erros, observabilidade e performance

À medida que o sistema cresce, boas práticas em Lisp deixam de ser apenas organização de arquivos. Você precisa definir política de falhas, telemetria e critérios de otimização. Common Lisp oferece ferramentas fortes para isso, mas elas exigem intenção.

Três frentes de maturidade técnica
  • Condições e restarts: úteis quando a recuperação de falhas precisa ser negociada, não apenas abortada.
  • Observabilidade: logs com contexto, métricas de throughput e inspeção de estados anômalos.
  • Profiling orientado a evidência: medir alocação, despacho dinâmico e pontos quentes antes de declarar tipos ou baixar safety.
Lisp/Common Lisp
; exemplo de boundary com condition customizada
(define-condition pedido-invalido (error)
	((motivo :initarg :motivo :reader pedido-motivo)))

(defun validar-pedido (pedido)
	(unless (pedido-itens pedido)
		(error 'pedido-invalido :motivo "pedido sem itens")))

(declaim (optimize (speed 2) (safety 3) (debug 3)))

Armadilha clássica: otimizar cedo demais com declarações agressivas de tipo e (safety 0). Em Lisp, otimização útil começa por perfil real, não por intuição.

Ferramentas que sustentam boas práticas

ASDF

Transforma ordem de carga e definição do sistema em contrato versionável.

Quicklisp

Facilita dependências, mas não substitui curadoria arquitetural.

SLIME / SLY

Dão velocidade ao ciclo de feedback, desde que o projeto tenha comandos reproduzíveis.

FiveAM

Ajuda a transformar experiência interativa em garantia automática contra regressão.

Comparação com estilos de projeto mais convencionais

EstiloVantagemRisco
Projeto Lisp com domínio + adaptadoresAlta maleabilidade e testabilidadePrecisa de disciplina arquitetural
Projeto centrado em framework únicoOnboarding inicial rápidoMais acoplamento à convenção e ao ciclo do framework
Projeto “REPL-first” sem estruturaIteração local muito velozDificuldade de reproduzir ambiente e depurar regressões

O objetivo não é eliminar a experimentação, e sim convertê-la em um sistema que outras pessoas consigam carregar, entender, testar e evoluir.
