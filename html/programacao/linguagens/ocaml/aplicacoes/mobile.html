OCaml no desenvolvimento mobile

Estrategias praticas para compartilhar logica de negocio entre iOS e Android sem sacrificar experiencia nativa.

A abordagem mais eficaz hoje e usar OCaml como shared core (regras, validacao, algoritmos) e manter UI nativa em Swift/Kotlin ou camada web mobile quando apropriado.

Modelos de adocao em mobile

Shared Core

Biblioteca OCaml compilada para C e chamada por Swift/Kotlin.

Web shell

Core em OCaml + front com js_of_ocaml/Melange dentro de WebView controlada.

Backend-first

OCaml no backend e contratos tipados para apps mobile nativas.

Ferramentas internas

App utilitaria corporativa com logica pesada compartilhada.

Arquitetura de interoperacao

Exemplo: API C para validar payload

core.ml + stub C (conceitual)
(* core.ml *)
type cadastro = { email : string; idade : int }

let valida c =
  String.contains c.email '@' && c.idade >= 18

(* exportar via Ctypes/FFI para funcao C-friendly *)

// pseudo-C header gerado/manual
// int validar_cadastro(const char* email, int idade);

// Swift/Kotlin chama essa funcao e recebe 0/1.

O valor arquitetural esta em manter uma unica fonte de verdade para validacoes e politicas de negocio, sem duplicar logica entre plataformas.

Praticas avancadas em projetos mobile hibridos

  • Definir ABI estavel e versionada para evitar quebra entre app e core.
  • Isolar alocacao de memoria na fronteira FFI para evitar leaks.
  • Automatizar testes de contrato (fixtures iguais em OCaml, Swift e Kotlin).
  • Centralizar observabilidade com codigos de erro canonicamente mapeados.

O maior risco em mobile hibrido e depuracao de fronteira FFI. Mantenha interfaces pequenas e extremamente documentadas.
