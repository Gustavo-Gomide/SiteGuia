OCaml - Aplicacoes com Banco de Dados

Visao Geral

Persistencia de dados em OCaml

Como construir camada de dados confiavel com SQL, transacoes e tipos fortes, minimizando inconsistencias entre codigo e esquema.

Em OCaml, o maior ganho em banco de dados vem de modelar invariantes no tipo e limitar pontos de acesso SQL com contratos bem definidos.

Fundamentos: conexao, query e mapeamento

Pense na camada de dados como um tradutor simultaneo: de um lado, SQL e constraints; do outro, tipos de dominio. Quanto melhor o tradutor, menos ambiguidades em runtime.

Caqti e a escolha comum para acesso a PostgreSQL, MariaDB e SQLite com assinatura de tipos em cada consulta.

Pipeline de uma operacao no repositorio

Exemplo com Caqti e transacao

repositorio_usuario.ml
(* Repositorio com separacao entre erro de dominio e erro de infraestrutura. *)
open Lwt.Syntax

type usuario = { id : int; email : string; ativo : bool }

let inserir_usuario =
  Caqti_request.exec
    Caqti_type.(tup2 string bool)
    "INSERT INTO usuarios (email, ativo) VALUES (?, ?)"

let buscar_por_email =
  Caqti_request.find_opt
    Caqti_type.string
    Caqti_type.(tup3 int string bool)
    "SELECT id, email, ativo FROM usuarios WHERE email = ?"

let criar_usuario conn email =
  let* existe = Caqti_lwt.Connection.find_opt conn buscar_por_email email in
  match existe with
  | Ok (Some _) -> Lwt.return (Error "email ja cadastrado")
  | Ok None ->
      let* r = Caqti_lwt.Connection.exec conn inserir_usuario (email, true) in
      begin match r with
      | Ok () -> Lwt.return (Ok ())
      | Error e -> Lwt.return (Error (Caqti_error.show e))
      end
  | Error e -> Lwt.return (Error (Caqti_error.show e))

Esse padrao permite tratar erro de dominio (email duplicado) separado de erro infra (falha de conexao), algo crucial para APIs previsiveis.

Quando usar Caqti, PGOCaml ou binding direto

  • Caqti: melhor escolha geral para portabilidade e simplicidade.
  • PGOCaml: quando o projeto e 100% PostgreSQL e deseja DSL mais dedicada.
  • Bindings diretos (postgresql-ocaml, sqlite3-ocaml): quando precisa de recursos especificos de driver.

Topicos avancados

  • Pool de conexoes com limites por workload para evitar saturacao de banco.
  • Migracoes automatizadas (Goose/Flyway externo ou script versionado no CI).
  • Estrangulamento por transacao: granularidade minima para reduzir lock contention.
  • Idempotencia de comandos de escrita em filas/eventos.

Evite concatenacao de SQL com string. Mesmo em codigo interno, padronize parametros para seguranca e legibilidade.
