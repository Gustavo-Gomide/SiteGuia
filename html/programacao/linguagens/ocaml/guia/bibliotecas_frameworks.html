Bibliotecas e Frameworks em OCaml

Do backend web a sistemas concorrentes, OCaml tem um ecossistema menor que JavaScript, mas extremamente coeso e com foco em corretude.

Escolher boas bibliotecas em OCaml e mais sobre compatibilidade de modelos de concorrencia, ergonomia de tipos e estabilidade de API do que sobre "quantidade de pacotes".

Como pensar o ecossistema

Uma analogia util: em vez de uma cidade com milhares de ruas confusas, OCaml parece um campus compacto com predios bem conectados. Menos opcoes redundantes, mais composicao previsivel.

Regra pratica: comece pelo trio opam + dune + ocaml-lsp, depois escolha um stack principal por dominio: Dream (web), Caqti (dados), Eio ou Lwt (concorrencia), Alcotest (testes).

Stacks recomendados por categoria

Web e APIs

Dream para HTTP, rotas e middleware; Yojson para JSON; Caqti para persistencia.

Concorrencia

Lwt para ecossistema maduro; Eio para I/O estruturada e cancelamento mais explicito.

Infra e plataformas

MirageOS para unikernels, Bos para scripting robusto e Cmdliner para CLIs profissionais.

Dados e IA

Owl para algebra numerica, bindings como ocaml-torch para deep learning e interoperacao C quando necessario.

Exemplo pratico: API tipada com Dream e Caqti

main.ml
(* Exemplo simplificado de repositorio orientado a tipos. *)
open Lwt.Syntax

let consulta_usuario =
  Caqti_request.find_opt
    Caqti_type.int
    Caqti_type.(tup2 int string)
    "SELECT id, nome FROM usuarios WHERE id = ?"

let rota_usuario pool req =
  let id = Dream.param req "id" |> int_of_string in
  let* conn = pool () in
  match Caqti_lwt.Connection.find_opt conn consulta_usuario id with
  | Ok (Some (uid, nome)) -> Dream.json (Printf.sprintf {|{"id":%d,"nome":"%s"}|} uid nome)
  | Ok None -> Dream.respond ~status:`Not_Found "nao encontrado"
  | Error e -> Dream.respond ~status:`Internal_Server_Error (Caqti_error.show e)

A vantagem tecnica aqui e que tipos SQL, contrato HTTP e serializacao ficam no mesmo fluxo de compilacao, reduzindo discrepancias entre banco e API.

Criterios avancados de selecao

Checklist de arquitetura

  • Modelo de concorrencia unico por servico (evite misturar Lwt e Eio sem fronteira clara).
  • Bibliotecas com manutencao ativa e suporte a OCaml 5.
  • Superficie de API pequena e tipos publicos estaveis.
  • Observabilidade: logs estruturados e testes de integracao automatizados.

Armadilha comum: adotar muitas libs de baixo nivel cedo demais. Em OCaml, simplicidade de stack quase sempre melhora produtividade e depuracao.
