TypeScript — Projetos e Boas Praticas

Projetos e Boas Praticas

Estrutura de pastas para projetos escaláveis, monorepos com workspaces, path aliases, migracao para strict e padroes que eliminam erros silenciosos em bases de codigo grandes.

TypeScript sem disciplina reproduz os mesmos problemas do JavaScript. Strict mode, isolatedModules, type-only imports e path aliases sao os quatro pilares de um projeto TS de qualidade industrial.

Estrutura de Projeto

Shell — Estrutura recomendada de projeto TypeScript
# Projeto Node.js / API backend
projeto/
├─ src/
│  ├─ index.ts              # entry point
│  ├─ config.ts             # env vars tipadas
│  ├─ errors.ts             # hierarquia de erros customizada
│  ├─ routes/
│  │  ├─ index.ts           # barrel de rotas
│  │  ├─ usuarios.ts
│  │  └─ produtos.ts
│  ├─ services/             # logica de negocio (sem dependencias de framework)
│  ├─ repositories/         # acesso a dados (Prisma / SQL)
│  ├─ schemas/              # schemas Zod + tipos inferidos
│  ├─ middleware/
│  └─ utils/
├─ src/__tests__/           # testes colocalizados com src
├─ dist/                    # saida do build (gitignore)
├─ tsconfig.json
├─ tsconfig.build.json      # exclui __tests__ para o build de prod
├─ vitest.config.ts
├─ package.json
└─ .env.example
tsconfig.json — Configuracao com path aliases
{
  "compilerOptions": {
    "target":                          "ES2022",
    "module":                          "NodeNext",
    "moduleResolution":                "NodeNext",
    "outDir":                          "./dist",
    "rootDir":                         "./src",
    "declaration":                     true,
    "declarationMap":                  true,
    "sourceMap":                       true,

    /* Strict suite completa */
    "strict":                          true,
    "noUncheckedIndexedAccess":        true,
    "exactOptionalPropertyTypes":      true,
    "noImplicitOverride":              true,

    /* Importacoes */
    "isolatedModules":                 true,
    "verbatimModuleSyntax":            true,

    /* Path aliases */
    "baseUrl":                         ".",
    "paths": {
      "@/*":               ["src/*"],
      "@schemas/*":        ["src/schemas/*"],
      "@services/*":       ["src/services/*"],
      "@repositories/*":   ["src/repositories/*"]
    }
  },
  "include": ["src"],
  "exclude": ["node_modules", "dist"]
}

Path aliases (@/) sao resolvidos pelo compilador TypeScript, mas nao pelo Node.js em runtime. Para Node, use tsup com --external ou tsc-alias para reescrever os caminhos na saida. Para Vite / Vitest, configure resolve.alias no vite.config.ts.

Monorepo com Workspaces

Shell — Estrutura de monorepo npm/pnpm
monorepo/
├─ packages/
│  ├─ core/               # tipos e logica compartilhada
│  │  ├─ src/index.ts
│  │  ├─ tsconfig.json
│  │  └─ package.json     # name: "@minha-org/core"
│  ├─ api/                # servidor Fastify/Hono
│  │  ├─ src/
│  │  ├─ tsconfig.json    # references: [{ path: "../core" }]
│  │  └─ package.json     # deps: { "@minha-org/core": "workspace:*" }
│  └─ web/                # frontend Next.js / Vite
├─ tsconfig.base.json     # opcoes compartilhadas
└─ package.json           # workspaces: ["packages/*"]
TypeScript — Project References (build incremental)
// packages/core/tsconfig.json - pacote referenciado
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": {
    "composite": true,       /* obrigatorio para project references */
    "outDir": "./dist",
    "declaration": true
  },
  "include": ["src"]
}

// packages/api/tsconfig.json - pacote consumidor
{
  "extends": "../../tsconfig.base.json",
  "compilerOptions": { "outDir": "./dist" },
  "references": [
    { "path": "../core" }  /* resolve tipagem do core sem compilar tudo */
  ]
}

// Build raiz: tsc --build packages/api
// Recompila apenas o que mudou — essencial em monorepos grandes

Type-Only Imports e isolatedModules

TypeScript — import type e verbatimModuleSyntax
// SEM verbatimModuleSyntax - pode causar problemas com bundlers
import { Request, Response } from "express";  // ambiguo: valor ou tipo?

// COM verbatimModuleSyntax - obrigatorio ser explicito
import type { Request, Response, NextFunction } from "express";
import express from "express";  // importacao de valor - permanece no JS

// O compilador elimina 'import type' completamente do output JS
// Sem import type, bundlers como esbuild precisam saber o que e valor vs tipo

// Exportacao de tipo
export type { Usuario };          // nao gera codigo JS
export { UsuarioService };        // gera codigo JS (classe e valor)

// Re-export de tipos de outro modulo
export type { Produto } from "./produto.js";

// Barrel file (src/schemas/index.ts) - exportar tipos nao polui bundle
export type { UsuarioInput, UsuarioOutput } from "./usuario.js";
export type { ProdutoInput, ProdutoOutput } from "./produto.js";
export { UsuarioSchema, ProdutoSchema } from "./schemas.js"; // valores Zod

Migracao para Strict Mode

TypeScript — Estrategia de migracao incremental
// Passo 1: ativar strict progressivamente no tsconfig
// Semana 1-2: base sem any implicito
{
  "compilerOptions": {
    "noImplicitAny": true,      /* exige anotacao quando TS nao consegue inferir */
    "strictNullChecks": true    /* null e undefined sao tipos distintos */
  }
}
// Semana 3-4: strict completo
{ "compilerOptions": { "strict": true } }
// Semana 5: extras
{ "compilerOptions": { "noUncheckedIndexedAccess": true } }

// Passo 2: usar ts-migrate para gerar anotacoes automaticas
// npx @ts-migrate/ts-migrate-full ./src
// Gera $ExpectError e @ts-ignore temporarios para CI verde

// Passo 3: supressao temporaria em arquivos problematicos
// @ts-expect-error — preferido pois da erro quando o problema e corrigido
// @ts-ignore — silencia permanentemente (evitar)

// Arquivo com muitos erros: usando ts-expect-error cirurgicamente
const resultado = antigoModulo.calcular(
  // @ts-expect-error legado: retorna any, corrigir JIRA-123
  dadosBrutos,
);

// Passo 4: eliminar 'any' com tipos utilitarios
// any → unknown (force explicit narrowing)
function processar(dado: unknown): string {
  if (typeof dado === "string") return dado.toUpperCase();
  if (typeof dado === "number") return dado.toString();
  throw new Error(`Tipo nao suportado: ${typeof dado}`);
}

Padroes e Antipadroes

Boas Praticas

  • Use satisfies para validar literal sem perder precisao
  • Prefira unknown a any em boundaries externos
  • Use Branded Types para IDs de entidades distintas
  • Ative exactOptionalPropertyTypes para distinguir undefined de ausente
  • Type-only imports em todos os tipos (verbatimModuleSyntax)
  • Zod para validacao em runtime + tipos inferidos via z.infer
  • Re-exporte types em barrel files com export type
  • Declare retornos explicitos em funcoes publicas de API

Antipadroes

  • as AnyType forcado — use type guard ou unknown
  • Barrel files com re-export de valores pesados (todo o projeto em index.ts)
  • Ignorar @ts-ignore sem comentario de issue
  • Classes vazias so para agrupar tipos — use namespace ou module
  • Interfaceamentos desnecessarios (interface para tudo)
  • Overuse de generics com 4+ parametros de tipo
  • Nao usar const assertions em arrays literais usados como union
  • Comparar com == null sem strictNullChecks ativo
TypeScript — Padroes avancados para projetos grandes
// ---- Config tipada com Zod + satisfies ----
import { z } from "zod";

const EnvSchema = z.object({
  PORT:         z.coerce.number().default(3000),
  DATABASE_URL: z.string().url(),
  NODE_ENV:     z.enum(["development", "production", "test"]),
  JWT_SECRET:   z.string().min(32),
}).readonly();

export type Env = z.infer<typeof EnvSchema>;

const parsed = EnvSchema.safeParse(process.env);
if (!parsed.success) {
  console.error(parsed.error.flatten());
  process.exit(1);
}
export const env = parsed.data;

// ---- Resultado tipado (Result monad) ----
type Ok<T>  = { ok: true;  valor: T };
type Err<E> = { ok: false; erro: E };
type Result<T, E = Error> = Ok<T> | Err<E>;

function ok<T>(valor: T): Ok<T> { return { ok: true, valor }; }
function err<E>(erro: E): Err<E> { return { ok: false, erro }; }

async function buscarUsuario(id: string): Promise<Result<Usuario>> {
  try {
    const u = await db.usuario.findUniqueOrThrow({ where: { id } });
    return ok(u);
  } catch (e) {
    return err(new Error(`Usuario ${id} nao encontrado`));
  }
}

const resultado = await buscarUsuario("42");
if (resultado.ok) {
  console.log(resultado.valor.nome); // TS sabe que e Usuario
} else {
  console.error(resultado.erro.message); // TS sabe que e Error
}
