Kotlin — Projetos e Boas Práticas

Projetos Kotlin bem estruturados combinam arquitetura modular, princípios SOLID expressos naturalmente com data classes e interfaces, documentação KDoc e uma suíte de testes robusta com Kotest e MockK.

Boas práticas em Kotlin não são impostas pela linguagem — elas emergem quando usamos os recursos idiomáticos certos: extension functions em vez de herança, sealed classes para modelagem de domínio, e coroutines para concorrência limpa.

Arquitetura Modular

Projetos Kotlin grandes beneficiam-se de uma estrutura Gradle multi-módulo. Cada módulo tem responsabilidade única e pode ser compilado e testado de forma independente, reduzindo tempos de build incrementais.

Estrutura — Projeto multi-módulo
meu-projeto/
├── settings.gradle.kts        # inclui todos os módulos
├── build.gradle.kts           # configurações compartilhadas
├── domain/                    # entidades, regras de negócio
│   └── src/main/kotlin/
│       └── com/app/domain/
│           ├── User.kt
│           └── UserRepository.kt  (interface)
├── data/                      # implementações de repositórios, BD
│   └── src/main/kotlin/
│       └── com/app/data/
│           └── UserRepositoryImpl.kt
├── api/                       # endpoints HTTP (Ktor ou Spring)
│   └── src/main/kotlin/
│       └── com/app/api/
│           └── UserRoutes.kt
└── app/                       # entry point, injeção de dependências
    └── src/main/kotlin/
        └── Main.kt
Kotlin — settings.gradle.kts
rootProject.name = "meu-projeto"
include("domain", "data", "api", "app")

SOLID em Kotlin

Os princípios SOLID se expressam de forma muito concisa em Kotlin. Data classes eliminam boilerplate de SRP, interfaces pequenas implementam ISP, e extension functions permitem abrir comportamento sem modificar a classe original (OCP).

Kotlin — SOLID na prática
// SRP — cada data class tem responsabilidade única
data class Pedido(val id: Long, val itens: List<Item>, val status: StatusPedido)

// ISP — interfaces pequenas e focadas
interface PedidoLeitor { suspend fun buscar(id: Long): Pedido? }
interface PedidoEscritor { suspend fun salvar(p: Pedido): Pedido }

// DIP — domínio depende de interfaces, não implementações
class ProcessarPedidoUseCase(
    private val leitor: PedidoLeitor,
    private val escritor: PedidoEscritor
) {
    suspend fun executar(id: Long): Pedido {
        val pedido = leitor.buscar(id) ?: throw IllegalArgumentException("Pedido não encontrado")
        return escritor.salvar(pedido.copy(status = StatusPedido.PROCESSADO))
    }
}

// OCP — abrir comportamento via extension function sem alterar a classe
fun Pedido.valorTotal(): Double = itens.sumOf { it.preco * it.quantidade }

Documentação com KDoc

KDoc é o padrão de documentação do Kotlin, análogo ao Javadoc. Ferramentas como Dokka geram sites de documentação HTML ou Markdown a partir dos comentários KDoc do código-fonte.

Kotlin — KDoc
/**
 * Representa um usuário da plataforma.
 *
 * @property id identificador único no banco de dados
 * @property nome nome completo do usuário
 * @property email endereço de e-mail único
 */
data class User(val id: Long, val nome: String, val email: String)

/**
 * Calcula o desconto aplicável ao [valor] baseado na [categoria].
 *
 * @param valor preço original em reais
 * @param categoria tipo de desconto a aplicar
 * @return valor com desconto aplicado, nunca negativo
 * @throws IllegalArgumentException se [valor] for negativo
 */
fun calcularDesconto(valor: Double, categoria: String): Double {
    require(valor >= 0) { "Valor não pode ser negativo: $valor" }
    return when (categoria) {
        "vip" -> valor * 0.80
        "regular" -> valor * 0.95
        else -> valor
    }
}

Testes com Kotest e MockK

Kotest oferece DSLs expressivas para assertions (shouldBe, shouldThrow) e múltiplos estilos de spec (BehaviorSpec, StringSpec). MockK é o framework de mocking nativo para Kotlin, com suporte a coroutines via coEvery.

Kotlin — Kotest + MockK
import io.kotest.core.spec.style.StringSpec
import io.kotest.matchers.shouldBe
import io.mockk.*
import kotlinx.coroutines.test.runTest

class ProcessarPedidoTest : StringSpec({
    val leitor = mockk<PedidoLeitor>()
    val escritor = mockk<PedidoEscritor>()
    val useCase = ProcessarPedidoUseCase(leitor, escritor)

    "deve processar pedido existente" {
        runTest {
            val pedido = Pedido(1L, emptyList(), StatusPedido.ABERTO)
            coEvery { leitor.buscar(1L) } returns pedido
            coEvery { escritor.salvar(any()) } answers { firstArg() }

            val resultado = useCase.executar(1L)
            resultado.status shouldBe StatusPedido.PROCESSADO
            coVerify(exactly = 1) { escritor.salvar(any()) }
        }
    }

    "deve lançar exceção para pedido inexistente" {
        runTest {
            coEvery { leitor.buscar(99L) } returns null
            shouldThrow<IllegalArgumentException> { useCase.executar(99L) }
        }
    }
})
