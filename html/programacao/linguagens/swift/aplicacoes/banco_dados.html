Swift com banco de dados

Persistencia relacional com Fluent, PostgreSQL, modelagem tipada e migrations reproduziveis.

Em Swift servidor, acesso a dados nao precisa ser improvisado. O pacote Fluent oferece uma camada de modelos, queries e migracoes que organiza a evolucao do schema sem abandonar a clareza SQL nem o controle operacional.

Fluent e o papel do ORM

Pense no banco de dados como o arquivo mestre do sistema. Um ORM nao substitui esse arquivo; ele age como um tradutor disciplinado entre tabelas e estruturas da linguagem. Em Swift, o Fluent cumpre esse papel com modelos, relacionamentos e query builder.

O ponto forte do Fluent esta em integrar schema versionado, conexoes configuradas por driver e uma API coerente com o resto do ecossistema Vapor. O ponto de atencao e o mesmo de qualquer ORM: entender quando gerar queries simples, quando usar eager loading e quando recorrer a SQL manual.

PostgreSQL como banco principal

PostgreSQL e a combinacao mais comum com Swift servidor porque entrega consistencia ACID, tipos ricos, otima documentacao e ecossistema consolidado. Para APIs e backends transacionais, ele costuma ser a escolha padrao antes de considerar alternativas mais especializadas.

Por que a dupla funciona bem

  • Driver maduro para conexao com Vapor.
  • Boa compatibilidade com UUID, JSONB, datas e constraints.
  • Migrations reproduziveis facilitam CI, staging e rollback controlado.

Modelagem e queries com Fluent

Modelos representam entidades persistentes. Propriedades mapeiam colunas; wrappers como @ID, @Field e @Parent descrevem o contrato entre codigo e tabela.

swift - modelo Fluent com relacionamento e migration
import Fluent

final class Article: Model, Content {
    static let schema = "articles"

    @ID(key: .id)
    var id: UUID?

    @Field(key: "title")
    var title: String

    @Parent(key: "author_id")
    var author: User

    init() { }
}

struct CreateArticle: AsyncMigration {
    func prepare(on db: Database) async throws {
        try await db.schema("articles")
            .id()
            .field("title", .string, .required)
            .field("author_id", .uuid, .required, .references("users", "id"))
            .create()
    }
}

O ganho pedagogico aqui e claro: o leitor consegue ver schema e modelo na mesma linguagem, com menos salto mental entre camada de aplicacao e camada de persistencia.

Migrations e evolucao segura do schema

Migracao e um registro programavel da historia do banco. Em vez de editar tabelas manualmente em cada ambiente, a equipe descreve alteracoes como codigo versionado. Isso reduz divergencia entre desenvolvimento, homologacao e producao.

A migration deve ser tratada como artefato de release. Alterar uma migration antiga depois de publicada costuma quebrar reproducao de ambientes e dificulta auditoria.

Para grandes sistemas, vale separar migrations de schema e scripts de backfill. Uma muda a estrutura; a outra corrige ou recalcula dados. Misturar as duas tarefas torna rollback muito mais arriscado.

Desempenho, consistencia e limites do ORM

Fluent resolve boa parte do CRUD, mas um engenheiro senior precisa identificar quando o modelo abstrato atrapalha. Relatorios pesados, agregacoes complexas, janela analitica e tuning fino de indices quase sempre exigem SQL explicito e leitura de plano de execucao.

  • Use eager loading para evitar N+1 queries.
  • Prefira transacoes explicitas em operacoes multi-etapa.
  • Meça consultas lentas e revise indices antes de culpar a linguagem.
  • Quando necessario, combine Fluent com SQL bruto para pontos criticos.
