Perl na Web

De CGI legado a APIs modernas com PSGI/Plack, Mojolicious e Dancer2.

Embora Perl tenha ficado marcado pela era CGI, o ecossistema evoluiu com middleware padronizado, servidores persistentes, roteamento moderno e suporte robusto a APIs REST. Em ambientes corporativos, Perl continua relevante tanto para manter sistemas críticos quanto para construir serviços leves de integração.

PSGI e Plack: A Base Moderna

O padrão PSGI para Perl cumpre papel similar ao WSGI/Python e Rack/Ruby: separa aplicação de servidor. Com Plack, você conecta sua app a middlewares de log, compressão, autenticação e controle de erro.

app.psgi - app mínima PSGI
use strict;
use warnings;
use JSON qw(encode_json);

my $app = sub {
		my $env = shift;
		my $payload = { status => "ok", path => $env->{PATH_INFO} };

		return [
				200,
				[ "Content-Type" => "application/json; charset=utf-8" ],
				[ encode_json($payload) ]
		];
};

return $app;

Em produção, prefira servidores persistentes como Starman, uWSGI (plugin PSGI) ou Hypnotoad (Mojolicious), evitando o custo de inicialização por requisição típico do CGI clássico.

Mojolicious e Dancer2

Mojolicious

Framework full-stack com roteamento, WebSocket, templates, autenticação e servidor embutido. Excelente para APIs e apps em tempo real.

Dancer2

Microframework minimalista com sintaxe direta para rotas REST, ideal para serviços internos e prototipagem rápida.

Catalyst

Framework maduro para sistemas corporativos maiores, com arquitetura MVC e ecossistema de plugins consolidado.

api_mojo.pl - rota REST com Mojolicious::Lite
use Mojolicious::Lite -signatures;

get '/health' => sub ($c) {
	$c->render(json => { ok => JSON::PP::true, service => 'perl-api' });
};

post '/users' => sub ($c) {
	my $data = $c->req->json // {};
	return $c->render(status => 422, json => { erro => 'nome obrigatório' })
		unless $data->{nome};

	$c->render(status => 201, json => { id => 1, nome => $data->{nome} });
};

app->start;

Segurança, Performance e Deploy

Em web, o maior risco não é linguagem, e sim arquitetura: entrada não validada, sessão fraca, SQL sem bind, logs sem correlação, e deploy sem observabilidade. Em Perl, recomenda-se:

  • Validação de entrada com regras explícitas e mensagens coerentes para API.
  • Prepared statements com DBI/DBIx::Class para evitar SQL injection.
  • Hash de senhas com Argon2 ou bcrypt, nunca SHA simples.
  • Middleware de segurança (headers, limite de payload, rate limit).
  • Containerização e health checks para rollback rápido.

Manutenção de Legado CGI

Muitas empresas ainda têm sistemas CGI críticos. A estratégia recomendada é migração incremental: encapsular regras de negócio em módulos, criar camadas de teste, e expor endpoints gradualmente via PSGI/Mojolicious, mantendo compatibilidade operacional.

Plano prático: 1) extrair funções puras, 2) cobrir com testes (Test::More), 3) criar adapter PSGI, 4) ativar métricas, 5) migrar rotas por domínio de negócio.
