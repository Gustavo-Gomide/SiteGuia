Terraform e HCL: Introducao

Infraestrutura como Codigo com Terraform

Comecando em Terraform/HCL do jeito certo

Do conceito de IaC ate o ciclo de vida real de provisionamento em nuvem.

Terraform transforma infraestrutura em declaracoes versionaveis: voce descreve o estado desejado, e a ferramenta calcula como chegar nele com seguranca.

Conceitos Fundamentais

Imagine um canteiro de obras sem planta: cada engenheiro monta uma parte diferente, gerando inconsistencias. IaC (Infrastructure as Code) e a planta unica da obra. Em vez de criar recursos manualmente no painel da nuvem, voce descreve esses recursos em arquivos de texto, revisa com Git e aplica de forma repetivel.

O Terraform e uma ferramenta declarativa de IaC, e o HCL (HashiCorp Configuration Language) e a linguagem usada para expressar essa declaracao. O foco nao e o "como" detalhado de cada passo, mas o "o que" deve existir no final.

Declarativo nao significa "magico": o Terraform calcula um plano de execucao explicito antes de alterar qualquer ambiente.

Como o Terraform Funciona por Dentro

O mecanismo central envolve tres elementos: configuracao HCL, estado atual da infraestrutura (state) e estado desejado. O Terraform compara os dois mundos e gera um plano.

Pipeline mental

1) Ler configuracao -> 2) Consultar providers/API -> 3) Comparar com state -> 4) Propor mudancas -> 5) Aplicar de forma ordenada.

Providers, Recursos e State

O provider e o plugin que fala com a API real (AWS, Azure, GCP, Kubernetes, GitHub e muitos outros). Cada bloco resource representa um objeto gerenciado, como uma VM, uma VPC, um bucket ou uma role IAM.

O state e um arquivo que guarda o mapeamento entre o que esta no codigo e o que existe de fato. Sem state confiavel, o Terraform perde contexto e pode tentar recriar ou destruir recursos indevidamente.

State deve ficar em backend remoto com bloqueio (locking) para times, nunca em arquivo local compartilhado manualmente.

Workflow Plan e Apply na Pratica

Fluxo minimo recomendado para cada alteracao de infraestrutura.

bash - ciclo base terraform
# Inicializa providers e backend
terraform init

# Valida sintaxe e semantica local
terraform validate

# Mostra alteracoes propostas sem aplicar
terraform plan -out=tfplan

# Aplica o plano aprovado
terraform apply tfplan

Em equipes maduras, o plan roda no pull request e o apply ocorre apenas apos aprovacao. Isso reduz erros e cria trilha de auditoria.

hcl - exemplo minimo
terraform {
	required_providers {
		aws = {
			source  = "hashicorp/aws"
			version = "~> 5.0"
		}
	}
}

provider "aws" {
	region = "us-east-1"
}

resource "aws_s3_bucket" "logs" {
	bucket = "meu-bucket-logs-exemplo"
}

Quando Terraform Brilha (e Quando Nao)

Use Terraform para

  • Provisionar infraestrutura repetivel entre ambientes.
  • Padronizar redes, IAM e recursos basicos de plataforma.
  • Controlar mudancas criticas com revisao de codigo.

Evite Terraform para

  • Deploy de aplicacao a cada commit (prefira ferramentas de entrega de app).
  • Scripts imperativos de bootstrap complexo sem idempotencia.
  • Dados sensiveis hardcoded em arquivos HCL.
