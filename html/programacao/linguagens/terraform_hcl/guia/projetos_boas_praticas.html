Terraform e HCL: Projetos e Boas Praticas

Design de Projeto para Longo Prazo

Como evitar caos em repositorios de infraestrutura

Modularizacao, padroes de naming, segregacao por ambiente e politicas.

A maior parte dos problemas de Terraform em producao nao e de sintaxe, e de arquitetura de projeto.

Estrutura de Modulos e Repositorios

Separar modulos reutilizaveis de stacks por ambiente reduz acoplamento e melhora evolucao.

text - estrutura recomendada
# Layout sugerido para separar stacks e modulos
			terraform-live/
	envs/
		dev/
		stg/
		prod/
	modules/
		network/
		compute/
		observability/

Naming Convention e Tags Obrigatorias

Nomes consistentes facilitam operacao, custos, auditoria e troubleshooting.

hcl - padrao de tags
# Tags minimas para operacao e custos
			locals {
	common_tags = {
		Project     = var.project
		Environment = var.environment
		Owner       = "plataforma"
		ManagedBy   = "terraform"
		CostCenter  = var.cost_center
	}
}

Padrao pratico de nome: {projeto}-{ambiente}-{tipo}-{indice}.

Segregacao por Ambiente

Dev, staging e producao devem ter estados distintos e, idealmente, contas/projetos de nuvem distintos. Isso reduz blast radius de erro humano.

Separacao minima

Workspaces + backend remoto por chave distinta.

Separacao robusta

Conta cloud separada por ambiente, credenciais e politicas independentes.

State Remoto, Governanca e Politicas

State remoto com locking e versao e requisito operacional. Governanca entra com policy as code para impedir configuracoes perigosas.

hcl - backend remoto com seguranca
# Backend remoto em bucket dedicado
			terraform {
	backend "s3" {
		bucket  = "org-state-prod"
		key     = "prod/network/terraform.tfstate"
		region  = "us-east-1"
		encrypt = true
	}
}

terraform init

Nunca commite arquivos .tfstate no Git. Eles podem conter segredos e metadados sensiveis.
