Projetos e Boas Praticas Java

Arquitetura em camadas com separacao de responsabilidades, principios SOLID, Clean Code, tratamento defensivo de erros, seguranca com Spring Security e estrutura de pacotes para projetos reais.

Em Java empresarial, a estrutura do projeto e tao importante quanto o codigo. A separacao clara entre dominio, repositorios, servicos e controladores facilita testes, manutencao e a incorporacao de novos membros ao time.

Boas Praticas

  • Cada classe tem uma unica responsabilidade (SRP)
  • DTOs separados de entidades JPA — nunca exponha @Entity na API
  • Servicos recebem dependencias pelo construtor (injecao explicita)
  • Repositorios sao interfaces — nunca implemente persistencia no servico
  • Use Optional nos repositorios, nunca retorne null
  • Trate erros nas bordas (controller e servico), nao no repositorio

Anti-padroes

  • @Autowired em campo — dificulta testes unitarios
  • Logica de negocio dentro do @Controller
  • Entidade JPA como DTO de resposta (expoe schema do banco)
  • catch(Exception e) {} — engolir excecoes silenciosamente
  • Consultas JPQL gigantes dentro do servico em vez de no repositorio
  • Transactions em metodos privados (@Transactional ignorado por Spring AOP)

// DEPOIS: OCP + DIP — aberto para extensao, fechado para modificacao
public interface Calculavel {
  double area();
  double perimetro();
}

public class CalculadoraArea {
  private final List<Calculavel> formas;

  public CalculadoraArea(List<Calculavel> formas) { this.formas = formas; }

  public double totalArea() {
    return formas.stream().mapToDouble(Calculavel::area).sum();
  }
}

// Qualquer nova forma: basta implementar Calculavel — sem alterar CalculadoraArea
public record Hexagono(double lado) implements Calculavel {
  public double area()      { return (Math.sqrt(3) / 2) * lado * lado * 6; }
  public double perimetro() { return lado * 6; }
}

public boolean usuarioPodeRealizarPedido(Usuario usuario, Pedido pedido) {
  if (usuario == null)                  return false;
  if (!usuario.isAtivo())               return false;
  if (pedido.getItens().isEmpty())     return false;
  return true;
}

Seguranca — OWASP Top 10 em Java

Java + Spring Security — JWT stateless, senhas bcrypt, CORS
@Configuration
@EnableWebSecurity
public class SecurityConfig {

  @Bean
  public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
    return http
      .csrf(AbstractHttpConfigurer::disable)   // stateless JWT nao precisa de CSRF
      .cors(cors -> cors.configurationSource(corsSource()))
      .sessionManagement(sm -> sm.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
      .authorizeHttpRequests(auth -> auth
        .requestMatchers("/api/auth/**").permitAll()
        .requestMatchers(HttpMethod.GET, "/api/produtos/**").permitAll()
        .requestMatchers("/api/admin/**").hasRole("ADMIN")
        .anyRequest().authenticated())
      .addFilterBefore(jwtFilter(), UsernamePasswordAuthenticationFilter.class)
      .build();
  }

  // BCrypt com fator de custo 12 — resistente a brute-force
  @Bean
  public PasswordEncoder passwordEncoder() {
    return new BCryptPasswordEncoder(12);
  }

  @Bean
  public CorsConfigurationSource corsSource() {
    var config = new CorsConfiguration();
    config.setAllowedOrigins(List.of("https://minha-loja.com.br")); // nunca use *
    config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE"));
    config.setAllowedHeaders(List.of("Authorization", "Content-Type"));
    var source = new UrlBasedCorsConfigurationSource();
    source.registerCorsConfiguration("/**", config);
    return source;
  }
}

Segredos e variaveis de ambiente

Nunca coloque senhas, chaves de API ou connection strings diretamente no codigo ou no application.properties versionado. Use variaveis de ambiente (SPRING_DATASOURCE_URL) ou um gerenciador de segredos como AWS Secrets Manager ou HashiCorp Vault.

Arquitetura Evolutiva em Java

Praticas para crescer sem reescrever tudo

  • Separe dominio de infraestrutura para trocar banco, mensageria ou framework com menor impacto.
  • Crie modulos por contexto de negocio, nao por camada tecnica generica.
  • Mantenha contrato de API versionado e observavel para suportar evolucao incremental.
  • Refatore continuamente com metricas de acoplamento e complexidade, nao apenas por intuicao.
