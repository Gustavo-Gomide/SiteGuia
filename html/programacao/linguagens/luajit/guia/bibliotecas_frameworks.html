Escolhendo stack de producao com desempenho e manutencao

Esta pagina organiza as principais opcoes do ecossistema LuaJIT e mostra como combinar produtividade com throughput.

A melhor stack em LuaJIT nao e a com mais bibliotecas, e sim a que separa claramente orquestracao em Lua e aceleracao pontual em C.

Conceito central: camadas de biblioteca em LuaJIT

O ecossistema pode ser visto em tres camadas: bibliotecas puras Lua, frameworks orientados a runtime (como OpenResty) e aceleradores via FFI. Essa separacao facilita decidir onde a prioridade e clareza, onde e latencia e onde e portabilidade.

OpenResty: framework principal para web orientada a eventos

OpenResty integra Nginx e LuaJIT para executar logica de aplicacao no ciclo de eventos. Ele e dominante em gateways, API edge, autenticacao e roteamento com baixa latencia.

nginx.conf - endpoint com estado compartilhado e resposta Lua
# exemplo didatico simplificado
lua_shared_dict api_cache 64m;

server {
  listen 8080;
  location /health {
    content_by_lua_block {
      local dict = ngx.shared.api_cache
      local n = dict:incr("hits", 1, 0)
      ngx.say("ok hits=", n)
    }
  }
}

O ganho operacional vem de manter o caminho critico sem bloqueios desnecessarios e usar APIs assicronas da familia resty.*.

Tipos de bibliotecas e criterios de escolha

Bibliotecas puras Lua

Portabilidade alta, setup simples e menor risco de ABI.

Bibliotecas C via LuaRocks

Podem acelerar muito, mas exigem pipeline de build por sistema operacional.

Pacotes resty.*

Feitos para OpenResty, foco em I/O nao bloqueante e integracao com Nginx.

FFI wrappers proprios

Maior controle de desempenho, maior responsabilidade de seguranca e manutencao.

Sintaxe de uso: LuaRocks e modulos resty

Shell - instalacao reproducivel para LuaJIT (Lua 5.1 ABI)
# arvore local para isolar dependencias
luarocks --lua-version=5.1 --tree=.rocks install lua-cjson
luarocks --lua-version=5.1 --tree=.rocks install lua-resty-http
luarocks --lua-version=5.1 --tree=.rocks install lua-resty-redis
OpenResty - cliente HTTP assicrono com resty.http
local http = require("resty.http")
local cli = http.new()

local res, err = cli:request_uri("https://httpbin.org/get", {
  method = "GET",
  ssl_verify = true
})

if not res then
  ngx.log(ngx.ERR, "falha HTTP: ", err)
  return
end

ngx.say(res.status)

Quando integrar bibliotecas C via FFI

FFI e recomendado quando existe hotspot comprovado em CPU: hashing, compressao, parse de protocolo binario, transformacao vetorial e criptografia. Em operacoes dominadas por rede, o ganho costuma ser pequeno.

LuaJIT - wrapper didatico para SHA256
local ffi = require("ffi")

ffi.cdef[[
  unsigned char *SHA256(
    const unsigned char *d,
    size_t n,
    unsigned char *md
  );
]]

local crypto = ffi.load("crypto")
local out = ffi.new("unsigned char[32]")
local msg = "payload"

crypto.SHA256(msg, #msg, out)
print("hash calculado")

Sempre inclua fallback puro Lua para cenarios sem biblioteca C disponivel na maquina de deploy.

Comparacao de stacks e trade-offs

Stack Ponto forte Risco principal
OpenResty + resty.* Escala alta para HTTP e proxy inteligente Curva operacional do ecossistema Nginx
LuaJIT puro + LuaRocks Flexibilidade para CLI e servicos internos Mais trabalho para infraestrutura web completa
LuaJIT + FFI seletivo Performance maxima nos hotspots certos Maior custo de manutencao de ABI e build
