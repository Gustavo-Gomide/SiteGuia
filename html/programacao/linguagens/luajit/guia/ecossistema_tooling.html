LuaJIT - Ecossistema e Tooling

Ferramentas para otimizar com evidencia, nao com achismo

Esta secao cobre o conjunto minimo de instrumentos para diagnosticar e evoluir desempenho em LuaJIT.

Sem medir traces, GC e latencia, qualquer tuning de LuaJIT vira opiniao; com tooling adequado, vira processo tecnico reproduzivel.

Conceitos de observabilidade para runtime JIT

Observabilidade em LuaJIT combina tres perguntas: o que compilou, o que saiu do trace e qual foi o custo de cada caminho. Ferramentas de runtime respondem a primeira pergunta; profiling e benchmark respondem as outras duas.

Flags de runtime: -jv, -jdump e -jp

As flags oficiais do executavel ajudam a entender compilacao e comportamento de traces durante a carga. Use sempre em ambiente de laboratorio antes de aplicar mudancas em producao.

Shell - diagnostico inicial de comportamento JIT
# eventos de tracing em modo verboso
luajit -jv script.lua

# dump de bytecode/IR/ASM (dependendo do modo)
luajit -jdump script.lua

# profiler embutido com saida em arquivo
luajit -jp=G,profile.out script.lua

Compare resultados apenas com mesma entrada, mesma maquina e mesma versao de runtime.

Profiling de CPU, memoria e pausas de GC

jit.p

Profiler embutido para hotspots em runtime, util para iniciar diagnostico.

perf/eBPF

Correlaciona custo de usuario e kernel em Linux para entender syscalls e bloqueios.

Flame Graph

Ajuda a visualizar de forma hierarquica onde o tempo total e consumido.

Metrica de GC

Mostra impacto de churn de objetos e pausas no throughput.

LuaJIT - exemplo minimo com jit.p
local p = require("jit.p")
p.start("fi1", "profile.out")

local s = 0
for i = 1, 5e6 do
  s = s + (i * 0.125)
end

p.stop()
print(s)

Benchmark justo: protocolo minimo de confiabilidade

Checklist para evitar autoengano

  • Realizar warmup antes da medicao para permitir compilacao de traces.
  • Fixar dataset, afinidade de CPU e parametros de ambiente.
  • Separar benchmark de CPU puro e benchmark de I/O.
  • Executar muitas repeticoes e reportar mediana/p95.
  • Salvar baseline por commit para detectar regressao.
LuaJIT - harness simples com warmup
local clock = os.clock

local function bench(fn, rounds)
  for i = 1, 8 do fn() end
  local t0 = clock()
  for i = 1, rounds do fn() end
  return clock() - t0
end

local function work()
  local z = 0
  for i = 1, 100000 do
    z = z + i
  end
  return z
end

print("tempo=", bench(work, 50))

Inspecao de traces e leitura de sinais de regressao

Ao analisar dumps, procure por traces curtos, side exits repetitivos e uso de operacoes NYI no caminho quente. Esses sintomas indicam onde atuar: modelagem de dados, fluxo de controle ou substituicao de API.

Pseudo-log comentado de side exits
-- saida resumida de -jv
[TRACE 1 api.lua:42 loop]
[TRACE 1 side exit at api.lua:57]
[TRACE 2 api.lua:57 -> 42]

-- leitura tecnica:
-- 1) loop principal foi compilado;
-- 2) linha 57 quebra premissa com frequencia;
-- 3) runtime cria trace secundario para o desvio.

Side exit frequente costuma ser problema de modelagem de dados, nao de falta de flags.

Ferramentas complementares no ecossistema

  • OpenResty XRay para diagnostico de latencia e throughput em servicos OpenResty.
  • perf + FlameGraph para investigacao aprofundada em Linux.

  • LuaRocks com versoes pinadas para reproducibilidade entre ambientes.
