Projetos e Boas Práticas em C

Estrutura de projeto, convenções de nomenclatura, header guards, tratamento de erros, testes unitários e práticas de segurança para código C de qualidade profissional.

C dá liberdade total — o que significa que as convenções e disciplinas são inteiramente responsabilidade do programador. Projetos C bem estruturados têm layout de diretório consistente, headers com guardas adequados, gestão de erros sistemática e testes automatizados. Esta página consolida décadas de sabedoria do mundo real.

Estrutura de Diretórios

Não existe padrão obrigatório, mas há convenções amplamente adotadas que tornam projetos navegáveis por qualquer desenvolvedor C.

Shell — layout de projeto C recomendado
meu_projeto/

├── LICENSE

Headers: Guardas e #pragma once

Um header incluído múltiplas vezes — direta ou indiretamente — causa erros de definição dupla. Guardas de inclusão evitam isso.

C — include guard vs #pragma once
/* ===== Método 1: Include Guard (portável, ISO C) ===== */
/* include/meu_projeto/vetor.h */
#ifndef MEU_PROJETO_VETOR_H
#define MEU_PROJETO_VETOR_H

#include <stddef.h>  /* size_t */

typedef struct {
    int    *dados;
    size_t  tamanho;
    size_t  capacidade;
} Vetor;

/* Protótipos públicos */
Vetor *vetor_criar(size_t cap_inicial);
int    vetor_push(Vetor *v, int val);
void   vetor_destruir(Vetor **v);

#endif /* MEU_PROJETO_VETOR_H */


/* ===== Método 2: #pragma once (não-portável mas amplamente suportado) ===== */
#pragma once

#include <stddef.h>

typedef struct { /* ... */ } Vetor;

Separação de interface e implementação: O header (.h) deve conter apenas o necessário para o usuário da API: typedefs, structs públicas e protótipos. Implementação (.c) e tipos internos ficam no .c ou em um header privado (internal.h).

Tratamento de Erros

C não tem exceções — o tratamento de erros é explícito via valores de retorno, errno, ou estruturas de resultado. Isso força o programador a lidar com erros no ponto de detecção.

C — padrões de tratamento de erros
#include <stdio.h>
#include <stdlib.h>
#include <errno.h>
#include <string.h>

/* Padrão 1: retorno de código de erro */
typedef enum {
    OK = 0, ERR_MEMORIA, ERR_INVALIDO, ERR_IO
} Resultado;

Resultado processar(int n, int **out_array) {
    if (n <= 0) return ERR_INVALIDO;
    *out_array = malloc(n * sizeof(int));
    if (!*out_array) return ERR_MEMORIA;
    return OK;
}

/* Padrão 2: errno para funções de sistema */
void abrir_arquivo(const char *path) {
    FILE *f = fopen(path, "r");
    if (!f) {
        fprintf(stderr, "Erro ao abrir '%s': %s\n",
                path, strerror(errno));
        return;
    }
    /* ... usar f ... */
    fclose(f);
}

/* Padrão 3: goto para limpeza de recursos (Linux Kernel style) */
int operacao_complexa(void) {
    char *buf1 = malloc(1024);
    if (!buf1) return -1;

    char *buf2 = malloc(2048);
    if (!buf2) goto cleanup_buf1;

    FILE *f = fopen("config.txt", "r");
    if (!f) goto cleanup_bufs;

    /* … lógica principal … */
    fclose(f);
    free(buf2);
    free(buf1);
    return 0;

cleanup_bufs:  free(buf2);
cleanup_buf1:  free(buf1);
    return -1;
}

int main(void) {
    int *arr = NULL;
    Resultado r = processar(10, &arr);
    if (r != OK) {
        fprintf(stderr, "Erro: %d\n", r);
        return 1;
    }
    free(arr);
    return 0;
}

Testes com Unity

Unity é um framework de testes unitários minimalista para C — um único arquivo unity.c. Amplamente usado em sistemas embarcados (ThrowTheSwitch/Unity no GitHub).

C — testes unitários com Unity
#include "unity.h"
#include "vetor.h"

void setUp(void) {}     /* executado antes de cada teste */
void tearDown(void) {} /* executado após cada teste */

void test_vetor_criar(void) {
    Vetor *v = vetor_criar(4);
    TEST_ASSERT_NOT_NULL(v);
    TEST_ASSERT_EQUAL_INT(0, v->tamanho);
    TEST_ASSERT_EQUAL_INT(4, v->capacidade);
    vetor_destruir(&v);
}

void test_vetor_push_e_realloc(void) {
    Vetor *v = vetor_criar(2);
    TEST_ASSERT_EQUAL_INT(0, vetor_push(v, 10));
    TEST_ASSERT_EQUAL_INT(0, vetor_push(v, 20));
    TEST_ASSERT_EQUAL_INT(0, vetor_push(v, 30)); /* realloc aqui */
    TEST_ASSERT_EQUAL_INT(3, v->tamanho);
    TEST_ASSERT_EQUAL_INT(10, v->dados[0]);
    TEST_ASSERT_EQUAL_INT(30, v->dados[2]);
    vetor_destruir(&v);
    TEST_ASSERT_NULL(v);  /* destruir seta para NULL */
}

int main(void) {
    UNITY_BEGIN();
    RUN_TEST(test_vetor_criar);
    RUN_TEST(test_vetor_push_e_realloc);
    return UNITY_END();
}
/* gcc -o testes tests/test_vetor.c src/vetor.c third_party/unity/unity.c */

Boas Práticas Consolidadas

Faça sempre

  • Compile com -Wall -Wextra -Wpedantic e trate todos os warnings como erros (-Werror) em CI
  • Inicialize todas as variáveis na declaração
  • Verifique o retorno de todas as funções de alocação e I/O
  • Use const em parâmetros de ponteiro que não devem ser modificados
  • Use size_t para tamanhos e índices — nunca int negativo acidentalmente
  • Atribua NULL após free() para evitar dangling pointers
  • Prefira funções a macros sempre que possível
  • Use assert() para invariantes que nunca devem ser falsas

Evite

  • gets() — removida do padrão. Use fgets()
  • scanf("%s", buf) sem limite de tamanho — buffer overflow
  • Casts de void* desnecessários (em C, não precisa; em C++ precisa)
  • Variáveis globais mutáveis — dificultam testes e threading
  • Macros com efeitos colaterais: MAX(a++, b++) incrementa duas vezes
  • Misturar malloc com VLAs em código crítico
  • Strings hardcoded em UTF-8 sem lidar com encodings
  • Ignorar o valor de retorno de realloc — vaza o ponteiro original

CERT C Coding Standard: Para sistemas críticos ou de segurança, siga o CERT C Coding Standard (publicado pela Carnegie Mellon University). São mais de 100 regras cobrindo overflows, race conditions, gestão de memória e I/O seguro. O SEI CERT C Coding Standard está disponível gratuitamente online.
