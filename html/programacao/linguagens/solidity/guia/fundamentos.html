Solidity: Fundamentos

Base tecnica para contratos robustos

Domine os elementos fundamentais da linguagem

Tipos, localizacao de dados, funcoes, eventos e tratamento de erro sao o nucleo de qualquer contrato seguro.

Grande parte das vulnerabilidades em smart contracts nasce de fundamentos mal aplicados, nao de tecnicas avancadas.

Tipos primitivos e estruturados

Solidity oferece tipos de valor (`uint`, `int`, `bool`, `address`, `bytesN`) e tipos de referencia (`string`, `bytes`, arrays, mappings, structs). A escolha correta influencia custo de gas, legibilidade e seguranca.

Regra pratica

Prefira o tipo mais semantico e minimamente suficiente. Exemplo: usar `uint64` para timestamp e `address` para conta evita ambiguidades.

Storage, memory e calldata

`storage` persiste na blockchain e custa mais para escrita. `memory` e temporario durante a execucao. `calldata` e somente leitura para dados de entrada, geralmente mais barato em funcoes externas.

Solidity
pragma solidity ^0.8.24;

contract Cadastro {
		string[] private nomes;

		function adicionar(string calldata nome) external {
				nomes.push(nome);
		}

		function ler(uint256 i) external view returns (string memory) {
				return nomes[i];
		}
}

No exemplo, entrada em `calldata` reduz copia desnecessaria; retorno usa `memory` porque sai do contexto interno para o externo.

Funcoes, visibilidade e modifiers

Visibilidades (`public`, `external`, `internal`, `private`) definem fronteiras de acesso. Modifiers encapsulam pre-condicoes reutilizaveis, como controle de permissao.

Solidity
contract Cofre {
		address public dono;

		modifier apenasDono() {
				require(msg.sender == dono, "sem permissao");
				_;
		}

		constructor() { dono = msg.sender; }

		function acaoSensivel() external apenasDono {
				// logica administrativa
		}
}

Eventos, mappings, arrays e erros

Eventos criam trilha auditavel para indexadores e interfaces. `mapping` e ideal para busca O(1) por chave. `require`, `revert` e `assert` servem a cenarios diferentes.

  • require: valida entrada ou estado esperado do usuario.
  • revert: interrompe fluxo com motivo customizado.
  • assert: invariantes internas que nunca deveriam falhar.

Em producao, prefira erros customizados para reduzir gas e melhorar diagnstico de falhas.
