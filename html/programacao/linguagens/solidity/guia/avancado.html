Heranca, interfaces e libraries

Heranca evita duplicacao, interfaces formalizam contratos entre componentes e libraries encapsulam logica reutilizavel sem estado proprio persistente.

Solidity
// Interface padroniza integracao entre contratos
	interface IVoto {
		function votar(uint256 id) external;
}

library MathLib {
	// Evita duplicacao e mantem regras utilitarias em um ponto
		function min(uint256 a, uint256 b) internal pure returns (uint256) {
				return a < b ? a : b;
		}
}

Fallback, receive e low-level calls

`receive()` trata recebimento de ETH sem calldata. `fallback()` e executada quando nenhuma funcao bate com a assinatura ou quando ha calldata arbitraria. Chamadas low-level (`call`, `delegatecall`) exigem validacao rigorosa de retorno.

Risco classico

Ignorar o booleano de retorno de `call` pode mascarar falhas silenciosas e quebrar invariantes de negocio.

Inline assembly (Yul) e controle fino

Assembly permite otimizacoes de gas e acesso baixo nivel a memoria/storage, mas reduz legibilidade e aumenta superficie de erro. Use apenas em pontos criticos medidos por benchmark.

Toda rotin a em assembly deve vir acompanhada de testes direcionados, invariantes documentadas e revisao especializada.

Proxy patterns e upgrades

Contratos sao imutaveis por design. Para evolucao, usa-se proxy: um contrato delega chamadas para uma implementacao trocavel. Padroes comuns: Transparent Proxy e UUPS.

Solidity
// Trecho conceitual de delegacao
fallback() external payable {
		(bool ok, bytes memory data) = implementation.delegatecall(msg.data);
		require(ok, "delegatecall falhou");
}

O estado fica no proxy; a logica vive na implementacao. Esse detalhe e central para evitar colisoes de storage e vulnerabilidades de upgrade.
