Elixir — Projetos e Boas Práticas

Um projeto Elixir bem estruturado organiza código em contextos (boundaries funcionais), favorece funções puras testáveis, usa o with para pipelines com tratamento de erro, e aproveita o OTP para tolerância a falhas. A filosofia de design valoriza clareza e composabilidade sobre abstração prematura.

A maior diferença entre código Elixir iniciante e experiente é o uso correto de contextos (como praticado pelo Phoenix): cada domínio do negócio é encapsulado em um módulo de contexto com API pública clara, escondendo detalhes de implementação (Ecto queries, dependências externas) dos chamadores.

Contextos — Fronteiras de Domínio

lib/meu_app/contas/contas.ex — context module
defmodule MeuApp.Contas do
  @moduledoc """
  Contexto de Contas — gerencia usuários, autenticação e sessões.
  API pública: criar_usuario/1, autenticar/2, buscar_usuario/1
  """

  import Ecto.Query
  alias MeuApp.Repo
  alias MeuApp.Contas.Usuario

  @spec criar_usuario(map()) :: {:ok, Usuario.t()} | {:error, Ecto.Changeset.t()}
  def criar_usuario(attrs) do
    %Usuario{}
    |> Usuario.changeset(attrs)
    |> Repo.insert()
  end

  @spec autenticar(String.t(), String.t()) :: {:ok, Usuario.t()} | {:error, :invalido}
  def autenticar(email, senha) do
    usuario = Repo.get_by(Usuario, email: email)
    case Bcrypt.check_pass(usuario, senha) do
      {:ok, u} -> {:ok, u}
      _         -> {:error, :invalido}
    end
  end

  @spec listar_usuarios() :: list(Usuario.t())
  def listar_usuarios do
    from(u in Usuario, where: u.ativo == true, order_by: u.nome)
    |> Repo.all()
  end

  # Funções privadas ficam DENTRO do contexto, não expostas
  defp notificar_cadastro(%Usuario{} = u) do
    MeuApp.Notificacoes.enviar_boas_vindas(u.email)
  end
end

Tratamento de Erros — with e tagged tuples

Padrões seguros para pipelines que podem falhar
# ── Padrão {:ok, valor} / {:error, motivo} ───────────────────
# NUNCA usar exceções para controle de fluxo esperado

# Com with — elegante e seguro
def processar_pedido(params) do
  with
    {:ok, usuario}  <- Contas.buscar_usuario(params.usuario_id),
    {:ok, _}        <- validar_limite_credito(usuario, params.valor),
    {:ok, pedido}   <- Loja.criar_pedido(usuario, params),
    {:ok, _nota}    <- Fiscal.emitir_nota(pedido)
  do
    {:ok, pedido}
  else
    {:error, :usuario_nao_encontrado} -> {:error, "usuário inválido"}
    {:error, :limite_insuficiente}   -> {:error, "limite de crédito insuficiente"}
    {:error, changeset}               -> {:error, format_errors(changeset)}
  end
end

# Funções bang (!) — apenas quando falha é irrecuperável
# Caso contrário, usar a versão sem bang que retorna {:ok}/{:error}
Repo.get!(Usuario, id)          # lança Ecto.NoResultsError se não achar
Repo.get(Usuario, id)           # retorna nil (sem exceção)

# rescue — apenas para erros inesperados
def operacao_externa do
  resultado = chamar_api_externa()
  {:ok, resultado}
rescue
  e in [RuntimeError, ArgumentError] ->
    Logger.error("Falha inesperada: #{inspect(e)}")
    {:error, "serviço indisponível"}
end
Boas Práticas Elixir
  • Organizar em contextos (boundaries de domínio)
  • Retornar {:ok, valor} / {:error, motivo}
  • Usar with para pipelines com falha
  • Adicionar @spec em todas as funções públicas
  • Funções pequenas com responsabilidade única
  • mix format sempre antes do commit
  • Testes para cada cláusula de função
  • Usar Logger em vez de IO.puts em produção
Evitar
  • Usar exceções para fluxo esperado (use {:error})
  • Módulos "god" (contexto com 50+ funções)
  • Acesso direto ao Repo fora do contexto
  • Funções anônimas longas (extrair para funções nomeadas)
  • Estado global via :persistent_term sem necessidade
  • Ignorar warnings do compilador
  • Process.sleep em testes (uso de assert_receive)
  • Acoplamento direto entre contextos (usar eventos)

O padrão de contextos do Phoenix é uma das melhores ideias da comunidade Elixir: ao invés de expor os schemas Ecto diretamente para os controllers, cada domínio de negócio tem uma API pública explícita. Isso cria fronteiras claras, facilita testes (mock do contexto inteiro), e permite refatorar a implementação interna sem impactar os chamadores.
