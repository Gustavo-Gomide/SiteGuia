Um desenvolvedor PHP produtivo em 2024 vai muito além do código da aplicação: configura PHP-FPM com Nginx para alta performance, usa Xdebug para depuração precisa, executa PHPStan para garantir tipos em tempo de análise, formata código com PHP CS Fixer e containeriza tudo com Docker. Esta página cobre toda essa cadeia.

Modelos de Servidor PHP

A forma como o PHP é executado em produção impacta diretamente desempenho e escalabilidade. Existem três modelos principais:

Apache + mod_php

PHP roda embarcado no processo Apache. Cada worker Apache carrega o interpretador PHP. Simples de configurar, mas ineficiente — todo request ocupa um worker Apache completo, mesmo para arquivos estáticos.

Uso: hospedagem compartilhada, configurações legadas.

Nginx + PHP-FPM

Configuração recomendada para produção. Nginx é um servidor de alto desempenho assíncrono que delega requisições PHP via FastCGI para um pool de workers gerenciado pelo PHP-FPM. Arquivos estáticos são servidos pelo Nginx sem passar pelo PHP.

PM dinâmico ajusta workers automaticamente à carga.

Swoole / RoadRunner

Servidores coroutine-based que mantêm workers PHP em memória entre requisições — eliminando o overhead de inicialização por request. Ideal para APIs de alta carga. Swoole implementa coroutines; RoadRunner usa workers Go + PHP.

Laravel Octane usa ambos.

nginx — configuração PHP-FPM
server {
    listen 80;
    root /var/www/html/public;
    index index.php;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        fastcgi_pass   unix:/run/php/php8.3-fpm.sock;
        fastcgi_index  index.php;
        include        fastcgi_params;
        fastcgi_param  SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
    }
}
ini — php-fpm pool (www.conf)
[www]
user  = www-data
group = www-data
listen = /run/php/php8.3-fpm.sock

; Gerenciamento dinâmico de workers
pm = dynamic
pm.max_children   = 20
pm.start_servers  = 4
pm.min_spare_servers = 2
pm.max_spare_servers = 6
pm.max_requests   = 500  ; reinicia worker a cada 500 req (previne memory leaks)

OPcache — Bytecode Caching

OPcache armazena o bytecode compilado de scripts PHP em memória compartilhada. Sem ele, cada requisição recompila todos os arquivos PHP do zero. Com ele habilitado, o ganho de performance é de 3× a 5× em aplicações típicas.

ini — php.ini OPcache para produção
opcache.enable=1
opcache.memory_consumption=256       ; MB de RAM para bytecode
opcache.interned_strings_buffer=16
opcache.max_accelerated_files=20000
opcache.revalidate_freq=0           ; 0 = nunca revalida (produção)
opcache.validate_timestamps=0       ; não verifica timestamps (produção)
opcache.save_comments=1            ; necessário para Reflection/Attributes
opcache.jit_buffer_size=100M
opcache.jit=tracing

Com validate_timestamps=0, alterações nos arquivos PHP só serão detectadas após reiniciar PHP-FPM ou limpar o cache com opcache_reset(). Use somente em produção com deploy que reinicia o FPM.

Xdebug — Depuração e Cobertura de Código

Xdebug é a extensão de depuração padrão do PHP. Além de breakpoints e step-debug, gera relatórios de cobertura de código para PHPUnit e perfis de execução para análise de performance.

ini — configuração Xdebug 3
zend_extension=xdebug
xdebug.mode=debug,coverage            ; debug | coverage | profile | trace
xdebug.start_with_request=yes
xdebug.client_host=host.docker.internal ; para uso com Docker
xdebug.client_port=9003
xdebug.idekey=PHPSTORM

Nunca habilite Xdebug em produção. Ele reduz a performance em 50–70% e pode expor informações internas do servidor.

PHPStan — Análise Estática

PHPStan analisa o código PHP sem executá-lo, detectando erros de tipo, métodos inexistentes, variáveis não inicializadas e muito mais. Pode ser configurado em 10 níveis de rigor (0 a 9).

bash — uso do PHPStan
composer require --dev phpstan/phpstan

# Analisar o diretório src/ no nível máximo
./vendor/bin/phpstan analyse src --level=9
neon — phpstan.neon
parameters:
    level: 8
    paths:
        - src
        - tests
    ignoreErrors:
        - '#Call to an undefined method.*Eloquent.*#'
    bootstrapFiles:
        - vendor/autoload.php

PHP CS Fixer — Padronização de Código

PHP CS Fixer reformata automaticamente o código PHP para conformidade com os padrões PSR-1, PSR-2, PSR-12 e regras adicionais configuráveis.

php — .php-cs-fixer.php
$finder = PhpCsFixer\Finder::create()
    ->in([['src', 'tests']]);

return (new PhpCsFixer\Config())
    ->setRules([
        '@PSR12'                       => true,
        'strict_param'                 => true,
        'array_syntax'                 => ['syntax' => 'short'],
        'no_unused_imports'            => true,
        'ordered_imports'              => true,
        'declare_strict_types'         => true,
    ])
    ->setFinder($finder)
    ->setRiskyAllowed(true);

Docker com PHP

Containerizar PHP garante paridade entre ambientes de desenvolvimento, teste e produção.

dockerfile — produção
FROM php:8.3-fpm-alpine AS base

# Instalar extensões do sistema
RUN docker-php-ext-install pdo pdo_mysql opcache bcmath

# Copiar configuração de produção do OPcache
COPY docker/php/opcache.ini /usr/local/etc/php/conf.d/opcache.ini

FROM base AS builder
COPY --from=composer:2 /usr/bin/composer /usr/bin/composer
WORKDIR /app
COPY composer.* ./
RUN composer install --no-dev --optimize-autoloader --no-interaction

FROM base AS production
WORKDIR /app
COPY --from=builder /app/vendor ./vendor
COPY . .
RUN chown -R www-data:www-data /app/storage /app/bootstrap/cache

Gerenciamento de Versões PHP

Projetos diferentes podem exigir versões PHP distintas. Ferramentas de gerenciamento de versões resolvem isso:

phpenv

bash
phpenv install 8.3.0
phpenv global 8.3.0
phpenv local 8.2.0  # .php-version no diretório

asdf-php

bash
asdf plugin add php
asdf install php 8.3.0
asdf global php 8.3.0
