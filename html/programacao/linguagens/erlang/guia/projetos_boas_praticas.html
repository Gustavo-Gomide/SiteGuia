Erlang — Projetos e Boas Práticas

Projetos Erlang duráveis nascem de escolhas arquiteturais simples e repetíveis: processos pequenos, responsabilidades nítidas, supervisão explícita, releases reproduzíveis e instrumentação suficiente para entender o sistema enquanto ele ainda está saudável.

A grande boa prática de Erlang é aceitar a filosofia da plataforma. Se você tenta transformar a BEAM num runtime orientado a objetos ou num monólito de estado compartilhado, perde quase tudo que torna a linguagem valiosa.

Arquitetura orientada a aplicações OTP

Um projeto maduro em Erlang costuma ser dividido em aplicações OTP pequenas, cada uma com propósito claro. Em vez de um diretório central cheio de módulos utilitários, o ideal é pensar em limites operacionais: qual aplicação sobe primeiro, qual pode reiniciar sem corromper o resto e qual estado precisa ser reconstruído.

Supervisão e modelagem de estado

Em Erlang, o estado pertence ao processo. Isso simplifica concorrência, mas exige disciplina. O processo deve possuir apenas o estado que realmente controla; tudo o que pode ser derivado, recomputado ou recarregado não precisa viver indefinidamente na mailbox ou no heap.

supervisor e child_spec mínimos
init([]) ->
    Flags = #{strategy => one_for_one,
              intensity => 5,
              period => 10},
    Children = [
      #{id => cache_local,
        start => {cache_srv, start_link, []},
        restart => permanent,
        shutdown => 5000,
        type => worker,
        modules => [cache_srv]},
      #{id => http_listener,
        start => {web_listener, start_link, []},
        restart => permanent,
        shutdown => infinity,
        type => worker,
        modules => [web_listener]}
    ],
    {ok, {Flags, Children}}.

O ponto essencial é semântico: restart, shutdown e type não são burocracia. Eles documentam como o sistema deve se comportar diante de falhas reais.

Um erro comum é esconder demais a estratégia de restart. Se a árvore de supervisão não reflete o domínio, incidentes ficam difíceis de explicar e corrigir.

Releases, configuração e deploy

Projeto sério em Erlang precisa ser distribuível como release, não como uma coleção de comandos de shell. Releases fixam versão de ERTS, consolidam aplicações e tornam o ambiente previsível. Configuração sensível a ambiente deve ficar fora do código-fonte, preferencialmente em runtime.

Testes, tracing e operação

EUnit cobre lógica local; Common Test entra quando há múltiplos nós, protocolos ou integração com ambientes externos. Em sistemas distribuídos, teste útil é o que valida recuperação: reconexão, backoff, reprocessamento idempotente e comportamento sob timeout.

Na operação, comece com o básico bem feito: Logger estruturado, métricas de fila, contagem de processos, tempo de resposta de handlers, erros por rota e inspeção de mailbox. Ferramentas como Observer e Recon tornam isso visível sem reinventar um ecossistema inteiro.

Boas práticas que escalam em equipe

Adote

  • Behaviours como contrato explícito de processo
  • Árvores de supervisão legíveis pelo time inteiro
  • Estados pequenos e serializáveis quando possível
  • Releases versionados e rollback planejado

Evite

  • gen_server “faz-tudo” com estado inflado
  • Uso indiscriminado de processos sem protocolo claro
  • Dependências externas para problemas já resolvidos pelo OTP
  • Operação baseada só em logs textuais soltos
