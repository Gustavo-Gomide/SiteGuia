Erlang — Web

Erlang não é a escolha mais comum para CRUDs convencionais, mas é extraordinariamente forte em gateways HTTP, APIs com muitas conexões simultâneas, WebSockets, callbacks de telecom e serviços que precisam continuar respondendo mesmo quando partes do sistema falham.

Na web, Erlang rende melhor quando o problema central não é gerar páginas bonitas, e sim manter milhares ou milhões de conversas concorrentes com latência previsível e recuperação rápida.

Onde Erlang se encaixa bem

Imagine um restaurante com um único cozinheiro para tudo: pedidos, entrega, limpeza e caixa. Esse é o backend que concentra demais trabalho numa thread ou num pool pequeno. A BEAM adota o oposto: muitos processos leves, cada um cuidando de uma responsabilidade pequena. Para tráfego web, isso casa especialmente bem com conexões longas e workloads orientados a mensagem.

Excelente

WebSockets, chat, gateways, APIs de evento, callbacks, realtime e proxies.

Viável

REST APIs e backends internos com foco em disponibilidade e isolamento de falhas.

Menos natural

CMS, templating pesado e stacks front-end acopladas ao backend.

Fluxo de uma requisição na BEAM

Esse desenho importa porque cada camada pode falhar de forma localizada. Um handler malcomportado não precisa travar o listener inteiro; um pico de conexões afeta filas e schedulers, mas não impõe o mesmo modelo de bloqueio tradicional de servidores thread-per-connection.

Exemplo prático com Cowboy

Handler HTTP com resposta JSON
init(Req0, State) ->
    Body = jsx:encode(#{status => ok, linguagem => erlang}),
    Req = cowboy_req:reply(
      200,
      #{<<"content-type">> => <<"application/json">>},
      Body,
      Req0
    ),
    {ok, Req, State}.

start_link() ->
    Dispatch = cowboy_router:compile([
      {'_', [{<<"/health">>, health_handler, []}]}
    ]),
    cowboy:start_clear(http_listener,
      [{port, 8080}],
      #{env => #{dispatch => Dispatch}}).

O ponto didático aqui não é a sintaxe do JSON, e sim a separação de papéis: o listener aceita conexão, o router resolve rota, o handler responde, e todo o resto do comportamento de robustez continua sendo responsabilidade da aplicação OTP.

Quando usar e quando evitar

Use Erlang para web quando:

  • o número de conexões simultâneas importa mais que scaffolding rápido
  • o sistema precisa sobreviver a falhas parciais sem downtime amplo
  • WebSocket, fila e eventos são parte central do produto

Prefira outra stack quando:

  • o principal valor vem de ecossistema full-stack e produtividade de CRUD
  • o time depende fortemente de ORMs, geração de painel e plugins prontos
  • o backend é simples o bastante para não justificar a curva de BEAM

Erlang serve muito bem a camada de tráfego e coordenação. Se seu problema principal é rendering de frontend moderno, trate o BEAM como backend especializado, não como solução universal.
