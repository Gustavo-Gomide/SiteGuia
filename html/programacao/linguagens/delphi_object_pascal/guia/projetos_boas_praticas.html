Delphi — Projetos e Boas Práticas

Projetos Delphi profissionais seguem convenções de nomenclatura rigorosas, separam camadas (apresentação, negócio, dados) e usam interfaces para inversão de dependência. Spring4D traz IoC container; DUnitX fornece testes unitários; e a estrutura de diretórios organizada é essencial para escalar.

A convenção de prefixo T para tipos/classes é a mais conhecida de Delphi — TCliente, TPedido. Mas o sistema vai além: F para fields privados, I para interfaces, E para exceções, C para constantes. Seguir essas convenções torna o código imediatamente reconhecível por qualquer desenvolvedor Pascal.

Convenções de Nomenclatura

PrefixoUsoExemplo
TTipos: classes, records, enumsTCliente, TEnderecoRecord
IInterfacesIRepositorio, ILogger
EExceções (descendentes de Exception)EValidacaoError, EDbError
FFields privados de classeFNome, FIdade
AParâmetros de métodosANome, AValor
CConstantesCMaxTentativas, CVersionApp
GVariáveis globaisGAppConfig (evitar)
sem prefixoVariáveis locais, parâmetros sem Anome, total, i

Interfaces e Injeção de Dependência

Padrão Repository com interfaces — dependência invertida
type
  // Interface — contrato sem implementação
  IClienteRepo = interface
  ['{AAA-BBB-CCC}']
    function  Buscar(AId: Integer): TCliente;
    procedure Salvar(ACliente: TCliente);
    function  Listar: TArray<TCliente>;
  end;

  // Serviço depende da interface — não da implementação
  TClienteServico = class
  private
    FRepo: IClienteRepo;
  public
    constructor Create(ARepo: IClienteRepo);
    procedure RegistrarNovoCliente(const ANome, AEmail: string);
  end;

constructor TClienteServico.Create(ARepo: IClienteRepo);
begin
  FRepo := ARepo;   // injeção pelo construtor
end;

procedure TClienteServico.RegistrarNovoCliente(const ANome, AEmail: string);
var c: TCliente;
begin
  c := TCliente.Create;
  try
    c.Nome  := ANome;
    c.Email := AEmail;
    FRepo.Salvar(c);
  finally
    c.Free;
  end;
end;

// Em produção — implementação real
servico := TClienteServico.Create(TClienteRepoFireDAC.Create(conexao));

// Em teste — implementação fake
servico := TClienteServico.Create(TClienteRepoMemoria.Create);

Testes com DUnitX

DUnitX — testes unitários modernos em Delphi
unit TestCalculadora;

interface
uses
  DUnitX.TestFramework,
  App.Calculadora;

type
  [TestFixture]
  TTestCalculadora = class
  private
    FCalc: TCalculadora;
  public
    [Setup]    procedure Setup;
    [TearDown] procedure TearDown;

    [Test]
    procedure Soma_DoisInteiros_RetornaCorreto;

    [Test]
    [TestCase('Positivos',    '2,3,5')]
    [TestCase('ComNegativo',  '-1,1,0')]
    [TestCase('Zeros',        '0,0,0')]
    procedure Soma_Parametrizado(A, B, Esperado: Integer);

    [Test]
    [WillRaise(EDivByZero)]
    procedure Dividir_PorZero_LancaExcecao;
  end;

implementation

procedure TTestCalculadora.Setup;
begin
  FCalc := TCalculadora.Create;
end;

procedure TTestCalculadora.TearDown;
begin
  FCalc.Free;
end;

procedure TTestCalculadora.Soma_DoisInteiros_RetornaCorreto;
begin
  Assert.AreEqual(4, FCalc.Soma(2, 2));
end;

procedure TTestCalculadora.Soma_Parametrizado(A, B, Esperado: Integer);
begin
  Assert.AreEqual(Esperado, FCalc.Soma(A, B));
end;

procedure TTestCalculadora.Dividir_PorZero_LancaExcecao;
begin
  FCalc.Dividir(5, 0);  // deve lançar EDivByZero
end;

initialization
  TDUnitX.RegisterTestFixture(TTestCalculadora);
end.

A convenção [TestCase('Nome', 'a,b,esperado')] do DUnitX é o equivalente Delphi de parametrized tests. Um único método de teste com 10 [TestCase] attributes testa 10 cenários diferentes — muito mais legível do que 10 métodos separados. Use Assert.AreEqual, Assert.IsTrue, e Assert.WillRaise como fundação.
