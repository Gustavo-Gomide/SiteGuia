Lógica de Programação

A Arte de Ensinar o Computador a Pensar

Fundamentos universais para resolver problemas com clareza e precisão — antes de escrever qualquer código

Lógica de programação é a disciplina de construir sequências de passos bem definidos para resolver um problema. Ela existe antes de qualquer linguagem de programação — é a habilidade de pensar computacionalmente que determina a qualidade de todo o código que você vai escrever ao longo da carreira.

O que é Lógica de Programação?

Imagine que você precisa explicar para alguém que nunca cozinhou como fritar um ovo. Você não pode dar uma instrução vaga como "faça o ovo ficar pronto" — essa pessoa não sabe o que isso significa. Você precisa detalhar: acenda o fogão, coloque a frigideira, aguarde esquentar, quebre o ovo na borda, despeje o conteúdo, espere a clara solidificar, desligue o fogo. Cada passo é preciso, tem uma ordem e depende do anterior. Isso é, essencialmente, o que lógica de programação faz: transforma problemas do mundo real em sequências de instruções que uma máquina consiga executar.

Segundo Forbellone e Eberspächer (2005), lógica de programação é a técnica de encadear pensamentos para atingir determinado objetivo — organizar o raciocínio de forma coerente, ordenada e sem contradições.

Ponto central: lógica de programação é independente de linguagem. Você pode pensar logicamente em seu idioma, em pseudocódigo ou em fluxogramas, a linguagem de programação é apenas a forma de expressar essa lógica para uma máquina específica.

O computador não tem julgamento. Ele executa exatamente o que você mandou, nem mais, nem menos. Se a instrução estiver errada, o resultado estará errado. Isso exige que o programador seja absolutamente explícito ao descrever cada ação que a máquina deve realizar. Muitos iniciantes tentam aprender Python, JavaScript ou outra linguagem sem antes dominar a lógica. O resultado é que decoram a sintaxe, mas travam diante de qualquer problema novo porque não sabem como pensar para chegar à solução. A linguagem é apenas o vocabulário; a lógica é a gramática do pensamento computacional.

Algoritmos: o Coração da Lógica

Um algoritmo é uma sequência finita de passos bem definidos que resolve um problema ou executa uma tarefa. A palavra vem do nome do matemático árabe Al-Khwarizmi (século IX), cujas obras formalizaram os procedimentos de cálculo que usamos até hoje.

Para ser considerado um algoritmo válido, um conjunto de instruções precisa satisfazer três propriedades fundamentais, descritas por Knuth (1997) em The Art of Computer Programming:

Finitude

O algoritmo deve sempre terminar após um número finito de passos. Um processo que roda para sempre não é um algoritmo — é um loop infinito.

Definição

Cada passo deve ser definido com precisão e sem ambiguidade. "Faça algo útil" não é uma instrução válida; "some os valores de A e B e armazene em C" é.

Efetividade

Cada operação deve ser suficientemente básica para que possa, em princípio, ser realizada por um humano com papel e caneta em tempo finito.

Exemplo: encontrar o maior entre dois números

Antes de codificar, é comum representar um algoritmo em pseudocódigo — texto estruturado parecido com código mas escrito em linguagem natural — ou em fluxograma. Veja os dois para o mesmo problema:

maior_de_dois.pseudo
// Algoritmo: maior entre dois números
INÍCIO
  LEIA A
  LEIA B

  SE A > B ENTÃO
    ESCREVA "O maior é A"
  SENÃO SE B > A ENTÃO
    ESCREVA "O maior é B"
  SENÃO
    ESCREVA "São iguais"
  FIM SE
FIM

Fluxograma

Note o terceiro ramo "Iguais" — um bom algoritmo cobre todos os casos possíveis de entrada.

Variáveis e Tipos de Dados

Uma variável é um espaço nomeado na memória do computador onde você guarda um valor para usar depois. Pense em variáveis como caixas etiquetadas: a etiqueta é o nome da variável; o conteúdo da caixa é o valor. Se você tem uma caixa chamada idade e coloca 25 dentro, sempre que precisar da idade basta olhar dentro dessa caixa.

Declaração e uso de variáveis

variaveis.pseudo
// Declaração
DECLARE nome   : TEXTO
DECLARE idade  : INTEIRO
DECLARE altura : REAL
DECLARE ativo  : LÓGICO

// Atribuição
nome   ← "Ana"
idade  ← 28
altura ← 1.65
ativo  ← VERDADEIRO

Os quatro tipos primitivos fundamentais

Independentemente da linguagem, os dados se organizam em quatro tipos primitivos. Cada tipo define quais operações fazem sentido sobre aquele valor e quanto espaço ele ocupa na memória:

Inteiro

Números sem parte decimal: -10, 0, 42. Usado para contagens, índices, quantidades discretas.

qtd     ← 30
posicao ← -1

Real (Ponto Flutuante)

Números com parte decimal: 3.14, -0.5, 1.65. Usado para medidas, porcentagens, valores monetários.

preco ← 19.99
pi    ← 3.14159

Texto (String)

Sequências de caracteres delimitadas por aspas. "42" é texto, não número — não confunda os tipos.

nome ← "Ana"
msg  ← "Olá mundo"

Lógico (Booleano)

Só dois valores: VERDADEIRO ou FALSO. Fundamental para decisões. Batizado em homenagem ao matemático George Boole (1815–1864).

ativo  ← VERDADEIRO
logado ← FALSO

Armadilha comum: misturar tipos sem converter. Se você tentar somar o número 5 com o texto "3", a maioria das linguagens não fará 8 — pode gerar um erro ou, pior, produzir "53" silenciosamente. Sempre seja explícito sobre o tipo dos dados.

Constantes

Quando um valor não deve mudar durante toda a execução, usa-se uma constante. O nome em maiúsculas é convenção amplamente adotada para sinalizar que aquele valor é fixo: PI = 3.14159, VELOCIDADE_LUZ = 299792458. Usar constantes em vez de "números mágicos" espalhados pelo código melhora a leitura e a manutenção — se o valor mudar, você altera em um único lugar.

Operadores: Cálculos e Comparações

Operadores são símbolos que realizam operações sobre valores (chamados de operandos). Existem três categorias principais: aritméticos, relacionais e lógicos. A combinação deles permite construir expressões arbitrariamente complexas.

Operadores Aritméticos

OperadorOperaçãoExemploResultado
+Adição7 + 310
-Subtração7 - 34
*Multiplicação7 * 321
/Divisão real7 / 23.5
DIVDivisão inteira7 DIV 23
MODResto da divisão7 MOD 21

O operador MOD (módulo) merece destaque: retorna o resto da divisão inteira. É extremamente útil para verificar se um número é par (N MOD 2 = 0), para iterar circularmente sobre listas, ou para limitar valores a um intervalo.

Operadores Relacionais

Comparam dois valores e produzem sempre um resultado lógico: VERDADEIRO ou FALSO. São a base de toda tomada de decisão em um programa:

Precedência de operadores

Assim como na matemática (multiplicação antes da adição), operadores têm ordem de prioridade. Da maior para a menor: NÃO → aritméticos (*, /, MOD antes de +, -) → relacionais → E → OU. Use parênteses para garantir clareza:

precedencia.pseudo
// Ambíguo — qual E ou OU é avaliado primeiro?
SE A > 0 E B > 0 OU C > 0 ENTÃO ...

// Com parênteses — intenção explícita
SE (A > 0 E B > 0) OU C > 0 ENTÃO ...
// ou
SE A > 0 E (B > 0 OU C > 0) ENTÃO ...

Estruturas Condicionais

Um programa que executa sempre as mesmas instruções em sequência linear resolve muito poucos problemas. Na vida real, decisões dependem do contexto: se o saldo é suficiente, aprova a compra; se não é, rejeita. As estruturas condicionais são o mecanismo que permite tomar caminhos diferentes dependendo de uma condição avaliada em tempo de execução.

Condicional Simples — SE/ENTÃO

Executa um bloco somente se a condição for verdadeira. Se for falsa, pula inteiramente.

nota ← 7.5

SE nota >= 7 ENTÃO
  ESCREVA "Aprovado"
FIM SE

Condicional Composta — SE/SENÃO

Garante que sempre um dos dois blocos será executado — nunca os dois, nunca nenhum.

nota ← 5.0

SE nota >= 7 ENTÃO
  ESCREVA "Aprovado"
SENÃO
  ESCREVA "Reprovado"
FIM SE

Encadeada — SENÃO SE

Cada SENÃO SE só é avaliado se todos os anteriores foram falsos — eficiência garantida.

SE nota >= 9 ENTÃO
  ESCREVA "Excelente"
SENÃO SE nota >= 7 ENTÃO
  ESCREVA "Aprovado"
SENÃO SE nota >= 5 ENTÃO
  ESCREVA "Recuperação"
SENÃO
  ESCREVA "Reprovado"
FIM SE

Seleção Múltipla — ESCOLHA/CASO

Quando você precisa comparar uma única variável contra muitos valores específicos, ESCOLHA/CASO é mais limpo do que vários SENÃO SE encadeados. Equivale ao switch de linguagens como C, Java e JavaScript:

selecao_multipla.pseudo
dia ← 3

ESCOLHA dia
  CASO 1: ESCREVA "Segunda"
  CASO 2: ESCREVA "Terça"
  CASO 3: ESCREVA "Quarta"     // executado
  CASO 4: ESCREVA "Quinta"
  CASO 5: ESCREVA "Sexta"
  CASO CONTRÁRIO: ESCREVA "Final de semana"
FIM ESCOLHA

Estruturas de Repetição

Uma das maiores vantagens dos computadores é executar a mesma operação milhares ou milhões de vezes sem cansaço. As estruturas de repetição — também chamadas de laços ou loops — permitem que um bloco de código seja executado repetidamente enquanto uma condição for satisfeita ou por um número determinado de vezes.

Toda estrutura de repetição precisa de uma condição de parada. Sem ela, o programa entra em loop infinito e nunca termina — violando a propriedade de finitude dos algoritmos.

ENQUANTO — Pré-condição

Verifica a condição antes de cada execução. Se já for falsa na primeira verificação, o bloco nunca executa. Use quando você não sabe quantas vezes vai repetir.

i ← 1
ENQUANTO i <= 5 FAÇA
  ESCREVA i
  i ← i + 1  // obrigatório!
FIM ENQUANTO
// Saída: 1, 2, 3, 4, 5

REPITA/ATÉ — Pós-condição

Verifica a condição após cada execução. O bloco é executado ao menos uma vez. Ideal para validar entradas do usuário.

REPITA
  ESCREVA "Digite positivo:"
  LEIA n
ATÉ n > 0
// Sempre pede ao menos uma vez

PARA — Laço Contado

Quando você sabe exatamente quantas vezes vai repetir. Inicializa, testa e incrementa automaticamente — mais limpo e menos propenso a erros.

PARA i DE 1 ATÉ 10 PASSO 1 FAÇA
  ESCREVA i * i
FIM PARA
// 1,4,9,16,25,36,49,64,81,100

Fluxo interno do laço PARA

A linha tracejada indica o retorno ao teste após a atualização. O ciclo continua até o teste retornar FALSO.

Laços Aninhados

Um laço dentro de outro. Para cada iteração do externo, o interno completa todas as suas iterações:

aninhado.pseudo — tabuada de 1 a 3
PARA i DE 1 ATÉ 3 FAÇA           // laço externo: 3 iterações
  PARA j DE 1 ATÉ 10 FAÇA        // laço interno: 10 iterações cada
    ESCREVA i, " x ", j, " = ", i * j
  FIM PARA
FIM PARA
// Total: 3 x 10 = 30 iterações

Cuidado com a complexidade: laços aninhados multiplicam as iterações. Dois laços de 1.000 iterações cada, quando aninhados, resultam em 1.000.000 de operações. Esse é o primeiro contato com análise de eficiência de algoritmos — tema aprofundado em Estruturas de Dados.

Modularização e Funções

À medida que um programa cresce, escrevê-lo como um bloco único e contínuo se torna insustentável. A modularização é o princípio de dividir um problema grande em partes menores, cada uma com responsabilidade bem definida. Em programação, essas partes são chamadas de funções, procedimentos ou sub-rotinas.

Segundo o princípio Don't Repeat Yourself (DRY), formalizado por Hunt e Thomas (1999) em The Pragmatic Programmer: toda vez que você está copiando e colando código, está violando esse princípio e deve extrair uma função.

Procedimento

Executa uma série de ações mas não retorna um valor. Seu objetivo é realizar um efeito — como exibir algo na tela.

PROCEDIMENTO ExibirBV(nome: TEXTO)
  ESCREVA "Bem-vindo, " + nome
FIM PROCEDIMENTO

ExibirBV("Maria")

Função

Realiza cálculos e retorna um valor ao chamador para ser usado em outra parte do programa.

FUNÇÃO Media(a, b: REAL): REAL
  RETORNE (a + b) / 2
FIM FUNÇÃO

m ← Media(8.0, 6.5)  // 7.25

Parâmetros, argumentos e escopo

Parâmetros

Variáveis declaradas na definição da função — os "slots" que ela espera receber. Ex: a e b em Media(a, b).

Argumentos

Valores concretos passados na chamada. Ex: 8.0 e 6.5 em Media(8.0, 6.5).

Escopo Local

Variáveis declaradas dentro de uma função. Existem apenas enquanto ela executa — destruídas ao final.

Escopo Global

Variáveis declaradas fora de qualquer função. Visíveis em todo o programa. Devem ser minimizadas — criam dependências ocultas.

Vetores e Matrizes

Até aqui, cada variável guarda um único valor. Mas e quando precisamos armazenar a nota de 30 alunos? Criar 30 variáveis separadas seria impraticável. A solução são os vetores (ou arrays) — estruturas que guardam múltiplos valores do mesmo tipo em posições indexadas.

Pense em um vetor como uma fileira de caixas numeradas. Cada caixa tem um índice e você acessa qualquer uma especificando esse número. Em pseudocódigo, os índices normalmente começam em 1; em linguagens como C, JavaScript e Python, começam em 0 — atenção a essa diferença ao traduzir para código real.

Vetor notas — 6 elementos

O elemento [3] está destacado em neon — contém o valor 9.2. Acesso: notas[3].

vetores.pseudo
DECLARE idades[5]: INTEIRO

idades[1] ← 22
idades[2] ← 35
idades[3] ← 19
idades[4] ← 28
idades[5] ← 41

// Percorrendo com PARA — padrão fundamental
soma ← 0
PARA i DE 1 ATÉ 5 FAÇA
  soma ← soma + idades[i]
FIM PARA
ESCREVA "Média: ", soma / 5   // 29

Matrizes — Vetores Bidimensionais

Uma matriz é um vetor de vetores — uma tabela com linhas e colunas. Usada para representar grades, tabelas, imagens (como pixels) e dados bidimensionais. Você acessa um elemento fornecendo dois índices: linha e coluna.

matrizes.pseudo
DECLARE tabela[3,3]: INTEIRO

PARA lin DE 1 ATÉ 3 FAÇA
  PARA col DE 1 ATÉ 3 FAÇA
    tabela[lin, col] ← lin * col
  FIM PARA
FIM PARA

// Resultado:
// 1  2  3
// 2  4  6
// 3  6  9

Método de Resolução de Problemas

Ter conhecimento de variáveis, operadores e estruturas não é suficiente se você não souber como atacar um problema. George Pólya, em How to Solve It (1945), descreveu quatro etapas universais para resolver problemas — aplicáveis perfeitamente à programação:

1. Entender o Problema

O que exatamente é pedido? Quais são as entradas? Quais as saídas esperadas? Existe algum caso especial? A maioria dos erros vem de não entender completamente o enunciado antes de começar a codificar.

2. Planejar a Solução

Antes de escrever qualquer código, esboce o algoritmo em pseudocódigo ou papel. Divida o problema em subproblemas menores. Escolha as estruturas adequadas. Pense nos casos extremos.

3. Executar o Plano

Transforme o plano em código na linguagem escolhida. Implemente uma parte de cada vez, testando incrementalmente. Não tente resolver tudo de uma vez.

4. Revisar

O código funciona? Funciona para todos os casos, incluindo entradas inválidas? Existe uma solução mais simples ou eficiente? O código está legível para outro programador?

Tópicos Avançados

Uma vez dominados os fundamentos, surgem padrões e técnicas que aparecem repetidamente em problemas mais complexos. Eles ainda pertencem à lógica de programação — são formas mais sofisticadas de combinar o que já foi visto.

Recursividade

Uma função é recursiva quando chama a si mesma durante sua execução. É elegante para problemas com estrutura naturalmente recursiva — fatoriais, sequências de Fibonacci, árvores. Toda recursão precisa de uma condição de parada (caso base), sem a qual a função se chama indefinidamente até travar o programa.

fatorial_recursivo.pseudo
FUNÇÃO Fatorial(n: INTEIRO): INTEIRO
  SE n <= 1 ENTÃO
    RETORNE 1                    // caso base — para aqui
  SENÃO
    RETORNE n * Fatorial(n - 1)  // chamada recursiva
  FIM SE
FIM FUNÇÃO

// Fatorial(4) = 4 × Fatorial(3)
//            = 4 × 3 × Fatorial(2)
//            = 4 × 3 × 2 × Fatorial(1)
//            = 4 × 3 × 2 × 1 = 24

Busca Linear vs. Busca Binária

A busca linear percorre o vetor elemento por elemento — funciona em qualquer vetor, mas é lenta em vetores grandes: O(n). A busca binária divide o intervalo de busca pela metade a cada passo: O(log n). A restrição é que o vetor deve estar ordenado.

busca_binaria.pseudo
FUNÇÃO BuscaBinaria(v[], n: INTEIRO, alvo: INTEIRO): INTEIRO
  esq ← 1
  dir ← n

  ENQUANTO esq <= dir FAÇA
    meio ← (esq + dir) DIV 2

    SE v[meio] = alvo ENTÃO
      RETORNE meio            // encontrou
    SENÃO SE v[meio] < alvo ENTÃO
      esq ← meio + 1       // busca à direita
    SENÃO
      dir ← meio - 1       // busca à esquerda
    FIM SE
  FIM ENQUANTO

  RETORNE -1               // não encontrado
FIM FUNÇÃO

Bubble Sort — Ordenação por Bolha

Um dos algoritmos de ordenação mais simples. Compara pares de elementos adjacentes e os troca se estiverem na ordem errada. Repete até que nenhuma troca seja necessária. Complexidade O(n²) — simples para aprender, ineficiente para vetores grandes.

bubble_sort.pseudo
PROCEDIMENTO BubbleSort(v[], n: INTEIRO)
  PARA i DE 1 ATÉ n - 1 FAÇA
    PARA j DE 1 ATÉ n - i FAÇA
      SE v[j] > v[j+1] ENTÃO
        temp    ← v[j]    // guarda temporariamente
        v[j]    ← v[j+1] // move para a esquerda
        v[j+1] ← temp    // coloca no lugar
      FIM SE
    FIM PARA
  FIM PARA
FIM PROCEDIMENTO

Validação e Casos Extremos

Programas reais recebem entradas de usuários que podem ser inesperadas: números negativos onde só positivos fazem sentido, texto vazio, divisão por zero. Um bom algoritmo valida suas entradas antes de processá-las e define comportamento explícito para cada caso de erro.

validacao.pseudo
FUNÇÃO DividirSeguro(a, b: REAL): REAL
  SE b = 0 ENTÃO
    ESCREVA "Erro: divisão por zero não permitida"
    RETORNE 0
  FIM SE
  RETORNE a / b
FIM FUNÇÃO

Regra de ouro: sempre pergunte "o que acontece se a entrada for zero? Negativa? Vazia? O máximo possível?". Testar esses limites antes de considerar o algoritmo pronto é o que separa software robusto de software frágil.

Boas Práticas

Nomes Significativos

x é um nome ruim para uma variável que armazena a média das notas. media_notas é imediatamente compreensível. Nunca sacrifique clareza por brevidade.

Uma Responsabilidade por Função

Se uma função lê dados, processa e exibe resultados ao mesmo tempo, ela tem responsabilidades demais. Separe em funções menores, cada uma fazendo bem uma única coisa.

Comentários com Propósito

Não comente o quê o código faz (o código já diz). Comente o porquê — por que essa abordagem foi escolhida, quais premissas existem, quais armadilhas foram evitadas.

Evite Duplicação (DRY)

Se você copiou e colou um bloco de código, transforme-o em uma função. Código duplicado significa que qualquer correção precisa ser aplicada em múltiplos lugares — e você inevitavelmente esquecerá um.

Teste com Exemplos Concretos

Antes de afirmar que o algoritmo está correto, trace sua execução manualmente com dois ou três exemplos, incluindo casos extremos. Isso revela bugs que não são óbvios lendo o código.

Simplicidade Primeiro

A solução mais simples que funciona é geralmente a melhor. Otimizações prematuras tornam o código difícil de entender sem ganho real — faça funcionar, depois otimize se necessário.
