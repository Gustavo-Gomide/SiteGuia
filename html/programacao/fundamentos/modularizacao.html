Modularização

A arte de dividir sistemas complexos em partes pequenas, independentes e gerenciáveis

Imagine construir um carro fundindo uma única peça de metal: se o pneu furar, você troca o carro inteiro. Isso é um sistema monolitíco. Na programação, a Modularização garante que cada parte do código cuide de apenas uma responsabilidade.

1. Dividir para Conquistar

Em vez de escrever 10.000 linhas de código em um único arquivo, dividimos em 10 arquivos de 1.000 linhas. Cada arquivo é um Módulo.

O arquivo Main orquestra o sistema, mas a lógica pesada fica isolada nos módulos (caixas tracejadas).

2. A Regra de Ouro da Engenharia

Para saber se seu código está bem modularizado, usamos duas métricas inversas:

Alta Coesão (Bom)

Significa que as coisas dentro de um módulo fazem sentido juntas.

Ex: Um arquivo matematica.js deve ter apenas funções de soma, subtração, média. Se tiver uma função "enviarEmail", a coesão é baixa (ruim).

Baixo Acoplamento (Bom)

Significa que os módulos dependem pouco uns dos outros.

Ex: Se você mudar a senha do banco de dados, o módulo de "Cores da Tela" não deveria parar de funcionar. Eles devem ser desacoplados.

3. Do Monolito aos Módulos

Como transformamos um código bagunçado em código modular.

Código Monolitíco (Ruim)

Tudo misturado em um arquivo só.

// main.js (5000 linhas)

function conectarBanco() { ... }
function calcularImposto() { ... }
function mudarCorBotao() { ... }
function validarEmail() { ... }

// Difícil de ler e manter!

Código Modular (Bom)

Separado por responsabilidade.

// banco.js
export function conectar() { ... }

// financeiro.js
export function calcImposto() { ... }

// main.js (Principal)
import { conectar } from './banco';
import { calcImposto } from './financeiro';

conectar();

4. Por que modularizar?

  • Reutilização: Escreva a função de "Validar CPF" uma vez e use em 50 telas diferentes.
  • Manutenção: Se o cálculo do imposto mudar, você só precisa abrir o arquivo financeiro.js. O resto do sistema nem percebe.
  • Colaboração: O Programador A trabalha no módulo de Banco, enquanto o Programador B trabalha na Interface. Eles não atropelam o código um do outro.
  • Testabilidade: É muito mais fácil testar um módulo pequeno isolado do que testar um sistema inteiro de uma vez.

Pode ser feito em um arquivo só, separando apenas em funções diferentes, mas o ideal é separar em arquivos diferentes para garantir melhor organização e reaproveitamento além de facilitar a manutenção e encontrar uma parte específica, o que é mais demorado quando tudo está em um único arquivo grande.

Pense como se fosse um caderno:

  • 1 matéria: você encontra o caderno e a página que quer rapidamente, mas pega só uma matéria e se precisar de outra terá que procurar em outro caderno, em compensação é mais leve e ocupa menos espaço.
  • 10 matérias: teria que folhear várias páginas até achar a matéria e o que quer, mas já traz 10 matérias de uma vez — mais pesado e difícil de manusear, em compensação é mais prático para quem precisa de tudo.

Dito isso, se for um projeto pequeno pode ser feito em um único arquivo, mas se for maior o ideal é separar em módulos. Em todo caso, ao menos separe em funções bem nomeadas e adicione um comentário no início do arquivo como índice.

Prática leva à perfeição: comece a identificar responsabilidades no seu próximo projeto.

Código que escala é código modular. Divida para conquistar.
