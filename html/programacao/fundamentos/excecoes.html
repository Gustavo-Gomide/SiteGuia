Tratamento de Exceções

Como lidar com falhas sem travar o programa

Pensar em exceções é pensar em falhas possíveis. Em vez de deixar o programa quebrar no meio, você cria um caminho de segurança: detecta o erro, reage e, se possível, continua com uma mensagem clara.

1. O que é uma exceção?

Uma exceção é um evento de erro ou situação inesperada que interrompe o fluxo normal do programa, também pode ser algo esperado, mas que por algum motivo precisa ser tratado de forma especial.

Exceção vs erro

Em muitas linguagens, “exceção” é um objeto/estrutura lançado quando algo dá errado. “Erro” é o problema em si.
Em outras palavras uma exceção é nada mais que um erro "corrigido" em tempo de execução.

Exemplo: Tentar dividir um número por zero pode gerar um erro de divisão por zero, então pode criar uma exceção que dividirá o número por 1 em vez de zero e informar ou não o usuário sobre a correção automática e o erro.

Por que tratar?

Para evitar travamentos, melhorar mensagens, liberar recursos (arquivos/conexões) e manter o sistema confiável.

Exemplo: Você cria um loop para ler vários números digitados pelo usuário. Se ele digitar qualquer coisa em vez de um número em um dos loops ou não digitar nada, o programa vai quebrar e terá que ser reiniciado. Se você tratar a exceção, pode avisar o usuário que a entrada é inválida e pedir para digitar novamente, sem travar o programa.

2. Try / Catch / Finally

O padrão mais comum: tente executar, capture se falhar e finalize a limpeza.

Try

Onde o código pode dar erro (ler arquivo, converter texto para número, acessar rede, entrada do usuário).

Catch

Executa se ocorrer uma exceção no try. Aqui você trata ela e se quiser informa ao usuário.

Finally

Executa sempre. Ideal para liberar recursos: fechar arquivo, encerrar conexão, desfazer lock.
O clássico: "Faça, não importa o que aconteça".

Exemplo (pseudocódigo)

try { // Código arriscado var arquivo = abrirArquivo("dados.txt"); print(arquivo.ler()); } catch (erro) { // Plano de emergência print("Falha ao abrir arquivo."); } finally { // Limpeza (sempre) fecharArquivo(); }

3. Como o fluxo muda quando dá erro

Quando uma exceção acontece, o programa “salta” para o catch mais próximo que trate aquela falha.

O finally roda sempre.

4. Lançando exceções (throw)

Nem todo erro vem do sistema. Às vezes é uma regra sua: “idade não pode ser negativa”.

Validação

Quando a entrada não faz sentido, você pode interromper o fluxo e obrigar o código a tratar, tentar novamente ou avisar o usuário.

Exemplo (código)

function cadastrar(idade) { if (idade < 0) { throw new Error("Idade inválida"); } }

5. Erros comuns (exemplos clássicos)

  • Divisão por zero: tentar calcular 10 / 0.
  • Null/None: usar algo “vazio” como se tivesse valor.
  • Índice fora do limite: acessar posição que não existe em uma lista.
  • Falha de rede: servidor indisponível, timeout.
  • Falha de conversão: converter "abc" para número.

6. Boas práticas

Mensagens úteis

Mostre mensagens que expliquem o que o usuário pode fazer (“Verifique sua conexão”, “Digite um número”).

Trate o específico

Quando possível, trate erros específicos primeiro (ex: arquivo não encontrado) e evite “catch genérico” sem critério.

Sempre limpe recursos

Use finally (ou equivalente) para garantir fechamento de arquivos/conexões.

Dica: em linguagens que suportam, prefira o padrão de “gerenciamento automático” de recursos (ex: context manager, try-with-resources).

Agora você sabe como proteger seu código contra o inesperado.

Código robusto não é o que nunca falha — é o que sabe como se recuperar quando falha.
