Complexidade de Algoritmos

Complexidade de Tempo e Espaço

Como medir o custo de um algoritmo independente de hardware — da Notação Big O às classes de complexidade P, NP e NP-Completo.

Complexidade é a linguagem matemática para descrever como o custo de um algoritmo cresce em função do tamanho da entrada $n$. Em vez de medir em segundos (que dependem da máquina), medimos a ordem de crescimento: um algoritmo $O(n^2)$ sempre será superado por um $O(n \log n)$ para entradas grandes, independente de qual computador o executa. Compreender complexidade é fundamental para escrever código que escala.

Notação Big O

A Notação Big O foi formalizada pelo matemático alemão Paul Bachmann em 1894 e popularizada por Edmund Landau no início do século XX — por isso também chamada de notação de Bachmann-Landau. Descreve um limite superior assintótico: "no pior cenário, a quantidade de operações não cresce mais rápido que esta curva". Permite comparar algoritmos independentemente de hardware, linguagem ou constantes de implementação.

Definição formal: $f(n) = O(g(n))$ se existem constantes $c > 0$ e $n_0$ tais que $f(n) \leq c \cdot g(n)$ para todo $n \geq n_0$. Em palavras: $g(n)$ é um limite superior para $f(n)$ a partir de certo ponto.

Análise de Algoritmos

Para estimar complexidade, contamos operações fundamentais e vemos como elas crescem com n. A ideia é sair de "parece rápido" para "cresce assim".

Regra 1 — ignore constantes

3n + 4 → O(n). Multiplicadores não mudam a classe.

T(n) = 3n + 4 → O(n)

Regra 2 — fique com o dominante

O termo de maior crescimento absorve os menores.

n² + 50n + 10 → O(n²)

Regra 3 — loops sequenciais somam

Dois loops um depois do outro → O(n + n) = O(n).

for i: O(n) for j: O(n) Total: O(n)

Regra 4 — loops aninhados multiplicam

Um loop dentro do outro → O(n × n) = O(n²).

for i: for j: O(n × n) = O(n²)

Cuidado com entradas diferentes: se você percorre dois arrays independentes a e b, o custo pode ser O(a + b) sequencial ou O(a × b) aninhado — só vira "O(n²)" se ambos tiverem tamanho n.

Análise de Tempo

"Tempo" aqui não é cronômetro: é o número de passos em função de n. O mesmo algoritmo pode ter custos muito diferentes dependendo do cenário.

Ω (Ômega) — melhor caso

Limite inferior: o mínimo de operações que o algoritmo fará em qualquer entrada.

Exemplo: busca linear acha o elemento na 1ª posição → Ω(1).

Θ (Theta) — caso exato

Limite superior e inferior apertados: o algoritmo cresce exatamente assim.

Exemplo: Merge Sort — sempre Θ(n log n), sem surpresas.

O (Big O) — pior caso

Limite superior: garante que o custo não ultrapassa esse valor.

Exemplo: busca linear não encontra → O(n).

Complexidade amortizada

Às vezes uma operação rara é cara, mas acontece tão raramente que o custo se dilui ao longo de muitas operações. Formalizado por Robert Tarjan (1985).

Array dinâmico — exemplo clássico

Inserir no final é geralmente O(1). Quando o array enche, ele realoca e copia tudo: O(n). Mas isso acontece apenas após n inserções, então o custo médio por inserção é O(1) amortizado.

inserções: 1 2 3 4 5 6 7 8 ← ops normais O(1) cópia em n: O(n) ← rara, dilui o custo ───────────────────────────────────── custo médio por operação: O(1) amortizado

Análise de Espaço

Mede quanta memória extra um algoritmo usa além da entrada — variáveis auxiliares, estruturas temporárias e pilha de recursão.

O(1) — espaço constante

Só variáveis escalares, independente do tamanho da entrada.

total = 0 for x in arr: total += x # sem criar estruturas

O(n) — espaço linear

Cria uma estrutura proporcional ao tamanho da entrada.

out = [] for x in arr: out.append(x * 2) # novo array

O(log n) — recursão eficiente

Pilha de recursão com profundidade logarítmica (busca binária recursiva).

def busca(arr, l, r): mid = (l + r) // 2 return busca(arr, l, mid) # log n frames

Classes de Complexidade (P, NP, NP-Completo, NP-Hard)

Existe um nível além do Big O: problemas para os quais não conhecemos algoritmos eficientes — e talvez nunca conheçamos.

P — Polynomial

Problemas resolvíveis em tempo polinomial. Ordenação, caminhos mínimos, fluxo máximo.

NP — Nondeterministic Polynomial

Soluções verificáveis em tempo polinomial. Resolver pode ser muito mais caro.

NP-Completo

Os mais difíceis dentro de NP. Se um fosse resolvido em P, todos seriam. (Cook, 1971; Karp, 1972)

NP-Hard

Pelo menos tão difíceis quanto NP-Completo — mas podem nem estar em NP. Ex.: problema da parada.

P vs NP — a questão em aberto

A pergunta "P = NP?" é um dos 7 Problemas do Milênio do Clay Institute (US$ 1 000 000 de prêmio). A maioria dos pesquisadores acredita que P ≠ NP — e essa suposição sustenta toda a criptografia moderna.

P ⊆ NP (todo problema de P é também NP) NP-Completo ⊆ NP (são os mais difíceis dentro de NP) NP-Hard ⊇ NP-Completo (podem estar fora de NP) Questão em aberto: P =? NP

Na prática: para problemas NP-Completos, usamos heurísticas, aproximações e algoritmos exponenciais com podas (branch & bound). O conhecimento de complexidade ajuda a saber quando desistir do ótimo e buscar o suficientemente bom.
