Rubro-Negras: balanceamento por cor

A BST auto-balanceada que usa regras de coloração para garantir O(log n) com menos rotações que a AVL.

Usada no kernel Linux, no Java TreeMap e no C++ std::map, a árvore rubro-negra é a estrutura de busca balanceada mais difundida na prática. Seu segredo: codificar o balanceamento em dois bits de cor por nó em vez de calcular alturas.

O que é uma Árvore Rubro-Negra?

Uma árvore rubro-negra é uma BST onde cada nó é pintado de vermelho ou preto. As cinco propriedades abaixo, mantidas invariantemente, garantem que a árvore permaneça aproximadamente balanceada:

As 5 Propriedades

  1. Todo nó é vermelho ou preto.
  2. A raiz é preta.
  3. Toda folha (NIL) é preta.
  4. Se um nó é vermelho, ambos os seus filhos são pretos.
  5. Para cada nó, todos os caminhos simples daquele nó até as folhas descendentes contêm o mesmo número de nós pretos (altura-negra).

Segundo Cormen et al. (2022), uma árvore rubro-negra com n nós internos tem altura no máximo 2log(n+1).

Estrutura Visual

Inserção

Novo nó sempre entra como vermelho. Se viola regra 4 (pai vermelho), corrige com recoloração e/ou rotação. No máximo 2 rotações por inserção.

Remoção

A remoção é o caso mais complexo. Se o nó removido for preto, a altura-negra é violada e requer rebalanceamento. No máximo 3 rotações por remoção.

Recoloração

Antes de recorrer a rotações, verifica-se se pintar o tio de preto e o avô de vermelho resolve o desequilíbrio. É a operação mais barata.

Rubro-Negra vs AVL

Critério Árvore AVL Rubro-Negra
Balanceamento Mais rígido (FB 1) Menos rígido (altura 2log n)
Velocidade de busca Levemente mais rápida Ligeiramente mais lenta
Inserção/Remoção Mais rotações Menos rotações (máx. 3)
Uso prático Consultas intensas Uso geral (stdlib)
Overhead por nó Campo altura (int) 1 bit de cor

Complexidade

Operação Tempo Rotações máximas
Busca O(log n)

Espaço O(n)
