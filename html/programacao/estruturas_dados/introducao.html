Estruturas de Dados

"Algorithms + Data Structures = Programs" — Niklaus Wirth, 1976. Antes de escrever um único loop, a estrutura que você escolhe já define o custo do programa.

Uma estrutura de dados é uma forma de organizar, armazenar e operar sobre dados de modo que certas operações — busca, inserção, remoção, travessia — sejam realizadas com o menor custo possível para o problema em questão.

O que é uma Estrutura de Dados

Donald Knuth, em The Art of Computer Programming (1968), define estruturas de dados como os objetos matemáticos sobre os quais a programação opera. Wirth (1976) sintetizou: algoritmo + estrutura = programa. Hoje falamos também em Tipo Abstrato de Dados (TAD): a separação entre o que uma estrutura faz (interface) e como ela faz (implementação).

Estrutura de dados

Regras para representar dados e as operações sobre eles. Ex.: array, lista encadeada, fila, pilha, árvore, hash table, grafo.

Algoritmo

Passo a passo para resolver uma tarefa, geralmente usando uma ou mais estruturas. Ex.: busca binária, BFS/DFS, Merge Sort.

TAD — Tipo Abstrato de Dados

Separação entre interface (operações disponíveis) e implementação (como estão armazenados na RAM). Ex.: Pilha pode ser array ou lista.

Por que isso importa? Em muitos casos, a diferença entre "funciona" e "funciona bem" é apenas a troca de estrutura — de O(n) busca linear para O(1) hash map, por exemplo.

Para que Servem na Prática

Estruturas de dados aparecem em toda parte do software moderno — muitas vezes invisíveis, mas determinantes para a performance.

Bancos de dados

Índices B-tree permitem busca em disco em O(log n) mesmo com bilhões de registros. Hash index para buscas exatas em O(1).

Sistemas operacionais

Filas de prioridade para escalonamento de processos. Árvores de segmento para gerenciamento de memória virtual.

Buscadores

Índice invertido (hash de palavras → lista de documentos) permite buscar em bilhões de páginas em milissegundos.

Compiladores

Pilha para análise sintática e avaliação de expressões. Tabela hash para tabela de símbolos. Árvore sintática (AST).

Redes e GPS

Grafos com Dijkstra/A* para roteamento de pacotes e cálculo de rotas. Heap para fronteira de expansão.

Machine Learning

KD-trees e Ball Trees para k-nearest neighbors. Grafos para redes neurais. Matrices/tensores para álgebra linear.

Como Escolher uma Estrutura

Em vez de decorar nomes, pense nas operações que você mais faz e no custo de cada uma. Responda estas perguntas:

Guia de decisão

PerguntaEstrutura indicada
Preciso acessar por índice muito frequentemente?Array / vetor dinâmico
Insiro/removo muito no início ou em posições arbitrárias?Lista encadeada (ou deque)
Preciso do mínimo ou máximo rapidamente?Heap / priority queue
Busco por chave (id, nome, e-mail) com frequência?Hash map / dicionário
Os dados têm ordem e preciso de range queries?BST balanceada / Segment Tree
Os dados têm hierarquia (pai/filho)?Árvore genérica / Trie
Os dados têm relações arbitrárias (conexões em rede)?Grafo (lista de adjacência)

Regra de Knuth: "Premature optimization is the root of all evil." Escolha a estrutura mais simples que atenda o desempenho necessário. Se ficar lento, você mede e troca com base em dados reais.

Trade-offs Fundamentais

Nenhuma estrutura ganha em tudo. Quase sempre você troca alguma coisa por outra.

Tempo

O custo (Big O) das operações. Hash map ganha em busca O(1); array ganha em iteração por localidade de cache.

Operações favorecidas

Toda estrutura tem operações baratas e caras. Array: acesso O(1), inserção no meio O(n). Lista: inserção O(1), acesso O(n).

Pior caso vs caso médio

Hash map: O(1) médio, O(n) no pior caso (colisões). Quick Sort: O(n log n) médio, O(n²) pior caso.

Cache locality

Arrays têm elementos contíguos na RAM — cache-friendly. Listas encadeadas têm ponteiros espalhados — cache-unfriendly.

Memória: Stack e Heap

Toda estrutura de dados vive na memória do processo. Entender onde ela fica ajuda a prever seu comportamento.

Stack (Pilha de Chamadas)

Variáveis locais, parâmetros de função, frames de recursão. Alocação e desalocação automáticas — muito rápidas (LIFO).

def soma(n): total = 0 # ← stack for i in range(n): total += i return total

Heap (Memória Dinâmica)

Objetos criados em tempo de execução — listas, dicionários, instâncias de classe. Gerenciados pelo GC (Python) ou manualmente (C/C++).

dados = [] # ← heap for i in range(n): dados.append(i) # cresce no heap

Boas Práticas

Ao trabalhar com estruturas de dados

  • Comece simples: array ou lista na maioria dos casos. Só otimize quando medir um gargalo real.
  • Conheça as complexidades: saiba o Big O das operações principais antes de escolher.
  • Use as nativas da linguagem: list, dict, set em Python são altamente otimizados em C.
  • Pense no acesso mais frequente: a operação que acontece 90% do tempo deve ser a mais barata.
  • Documente a escolha: um comentário explicando "por que hash e não lista" poupa horas de manutenção futura.
