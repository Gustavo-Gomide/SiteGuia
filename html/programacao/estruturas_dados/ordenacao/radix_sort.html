Radix Sort: ordenar dígito por dígito

Aplica Counting Sort repetidamente, da posição menos significativa à mais significativa (LSD). Cada passagem ordena por um único dígito ao final de d passagens, o array inteiro está ordenado.

O Radix Sort parte de uma ideia contraintuitiva: para ordenar números de d dígitos, ordene-os primeiramente pelo dígito menos significativo, depois pelo segundo menos significativo, e assim por diante. Se cada ordenação por dígito for estável, o resultado final é correto. Isso produz complexidade O(dn) linear quando d é fixo ou cresce como log(n).

O que é o Radix Sort?

O Radix Sort tem origem nas máquinas de cartões perfurados IBM da década de 1880, usadas para processar o Censo dos EUA de 1890. Essas máquinas ordenavam cartões por uma coluna de cada vez exatamente a ideia do LSD Radix Sort. O algoritmo foi formalizado para computadores por Harold Seward e outros pesquisadores do MIT na década de 1950.

Existem duas variantes: LSD (Least Significant Digit) começa pelo dígito menos significativo e MSD (Most Significant Digit) começa pelo mais significativo. LSD é mais simples e stable; MSD pode ser implementado sem memória extra mas requer particionamento recursivo similar ao Quick Sort.

Analogia dos cartões de biblioteca: Imagine ordenar fichas de biblioteca por número de código (ex: 321, 142, 213). Primeiro agrupe por último dígito (1, 2, 3): fichas terminadas em 1 primeiro, depois 2, depois 3. Mantendo a ordem dentro de cada grupo, ordene pelo dígito do meio. Depois pelo primeiro. Ao final, as fichas estão em ordem crescente completa.

Por que LSD funciona?

A chave é a estabilidade do Counting Sort. Quando a passagem do dígito d+1 encontra dois números com o mesmo dígito d+1, a passagem do dígito d já os deixou na ordem relativa correta e como a ordenação é estável, essa ordem é preservada.

Base e número de passagens

Com base B e d dígitos, são d passagens de Counting Sort com k=B. O melhor valor de B para minimizar d(n+B) é aproximadamente B n, resultando em O(n) total. Na prática, B=256 (byte) é uma boa escolha para inteiros.

Implementação em Python

radix_sort.py
def radix_sort(arr: list) -> list:
    if not arr:
        return arr

    maximo = max(arr)
    exp = 1  # 1 = unidades, 10 = dezenas, 100 = centenas...

    while maximo // exp > 0:
        arr = _counting_por_digito(arr, exp)
        exp *= 10

    return arr


def _counting_por_digito(arr: list, exp: int) -> list:
    """Counting Sort estável pelo dígito na posição 'exp'."""
    n   = len(arr)
    output = [0] * n
    count  = [0] * 10  # base 10: dígitos 0-9

    for val in arr:
        digito = (val // exp) % 10
        count[digito] += 1

    for i in range(1, 10):
        count[i] += count[i - 1]

    for val in reversed(arr):  # reversed = estabilidade
        digito = (val // exp) % 10
        count[digito] -= 1
        output[count[digito]] = val

    return output


# Teste
if __name__ == "__main__":
    dados = [170, 45, 75, 90, 802, 24, 2, 66]
    resultado = radix_sort(dados)
    print(resultado)  # [2, 24, 45, 66, 75, 90, 170, 802]

Pontos importantes do código:

  • O loop while maximo // exp > 0 executa uma vez por dígito do número máximo d = floor(log10(max)) + 1 iterações.
  • (val // exp) % 10 extrai o dígito na posição atual: divida pelo valor posicional para deslocar, e o módulo 10 pega apenas o dígito menos significativo do resultado.
  • O array count tem sempre tamanho 10 (base decimal), independente do tamanho do input operações com k=10 são O(1) amortizado.
  • reversed(arr) na fase de distribuição garante estabilidade, mantendo a ordem relativa de elementos com o mesmo dígito.

Análise de Complexidade

CasoTempoEspaçoObservação
TodosO(d (n + k))O(n + k)d = dígitos, k = base (10, 256...)
Inteiros 32-bit com k=256O(4n) O(n)O(n)4 passagens com base 256

Comparação com Quick Sort: Para n inteiros de 32 bits com k=10, são d=10 passagens de Counting Sort: O(10n) = O(n). Quick Sort é O(n log n) O(20n) para n=1M. Radix Sort ganha teoricamente, mas na prática Quick Sort pode ser competitivo pela localidade de cache. Para n muito grande (bilhões de registros), Radix Sort vence.

Estabilidade: Estável (depende de usar Counting Sort estável em cada passagem).

Quando Usar

Grandes volumes de inteiros

Ordenação de IDs (CPF, RG, telefone), endereços IP, timestamps de alta resolução. Quando n é na casa de milhões e os dados são inteiros de tamanho fixo, Radix Sort é difícil de bater.

Strings de comprimento fixo

Para strings de comprimento fixo (como códigos postais ou ISBNs), cada posição de caractere é um "dígito" d passagens, k = tamanho do alfabeto (26 ou 256).

Dados heterogêneos ou variáveis

Strings de comprimento variável, objetos complexos, dados com valores negativos sem tratamento: Radix Sort requer adaptações não triviais. Quick Sort ou Tim Sort são mais flexíveis.
