Quick Sort

Quick Sort: particionar, conquistar, combinar

Escolhe um elemento pivô, reorganiza o array de modo que todos os menores fiquem à esquerda e os maiores à direita, e aplica recursão nas duas partições in-place e ultrarrápido na prática.

Na prática, o Quick Sort costuma ser o mais rápido entre os algoritmos de uso geral. Apesar de ter pior caso O(n²), com estratégias de pivô inteligentes esse pior caso é extremamente improvável. Sua grande vantagem é ser in-place (O(log n) de espaço para recursão) e ter constantes ocultas muito baixas, batendo o Merge Sort em memória principal por isso está na linha de base de bibliotecas padrão de C, C++ e Rust.

O que é o Quick Sort?

O Quick Sort foi inventado por Tony Hoare em 1959, quando ele tinha 25 anos e trabalhava como estudante de intercâmbio em Moscou. Ao tentar implementar uma tradução automática do russo para o inglês, precisou ordenar palavras rapidamente e criou o algoritmo. Hoare o descreveu formalmente em 1961 no Computer Journal.

A ideia central é o particionamento: escolher um elemento como pivô e rearranjar o array de modo que todos os elementos menores que o pivô fiquem à sua esquerda e todos os maiores fiquem à sua direita. O pivô então está em sua posição definitiva. Recursão nas duas metades completa a ordenação.

Analogia da divisão de grupo: Imagine organizar 30 alunos por altura. Você escolhe um aluno (o pivô), pede que todos mais baixos se movam para a esquerda e os mais altos para a direita. O pivô está agora em sua posição final. Você repete independentemente em cada grupo sem precisar olhar para o outro lado.

Partição de Lomuto vs Hoare

Lomuto (pivô = último elemento) é mais simples de implementar mas faz mais trocas. Hoare (dois ponteiros convergindo) é mais eficiente faz 3x menos trocas em média mas é mais difícil de implementar corretamente. A maioria dos sistemas usa Hoare ou variantes com 3-way partition.

Escolha do pivô

O pior caso O(n²) ocorre quando o pivô sempre é o menor ou maior elemento (arrays já ordenados com pivô = primeiro ou último). Estratégias comuns: pivô aleatório, mediana de três (início, meio, fim), ou Dual-Pivot (usado no Arrays.sort do Java para primitivos).

Implementação em Python

quick_sort.py
import random

def quick_sort(arr: list) -> list:
    if len(arr) <= 1:
        return arr

    # Pivô aleatório para evitar pior caso em arrays ordenados
    pivot = random.choice(arr)

    menores  = [x for x in arr if x < pivot]
    iguais   = [x for x in arr if x == pivot]
    maiores  = [x for x in arr if x > pivot]

    return quick_sort(menores) + iguais + quick_sort(maiores)


# Versão in-place com partição de Lomuto
def quick_sort_inplace(arr: list, baixo: int = 0, alto: int = -1) -> list:
    if alto == -1:
        alto = len(arr) - 1
    if baixo < alto:
        p = _lomuto(arr, baixo, alto)
        quick_sort_inplace(arr, baixo, p - 1)
        quick_sort_inplace(arr, p + 1, alto)
    return arr

def _lomuto(arr: list, baixo: int, alto: int) -> int:
    pivot = arr[alto]
    i = baixo - 1
    for j in range(baixo, alto):
        if arr[j] <= pivot:
            i += 1
            arr[i], arr[j] = arr[j], arr[i]
    arr[i + 1], arr[alto] = arr[alto], arr[i + 1]
    return i + 1


# Teste
if __name__ == "__main__":
    dados = [3, 6, 8, 10, 1, 2, 1]
    print(quick_sort(dados.copy()))       # [1, 1, 2, 3, 6, 8, 10]
    print(quick_sort_inplace(dados.copy())) # [1, 1, 2, 3, 6, 8, 10]

Pontos importantes do código:

  • A versão funcional com list comprehensions é clara e trata automaticamente duplicatas (lista iguais), mas cria O(n) memória auxiliar igual ao Merge Sort.
  • A versão in-place quick_sort_inplace usa O(log n) de pilha de chamadas recursivas (espaço de stack), muito mais eficiente.
  • O pivô aleatório (random.choice) elimina a possibilidade de pior caso determinístico torna O(n²) astronomicamente improvável.
  • A partição de Lomuto posiciona o pivô em seu lugar final e retorna seu índice p para que as chamadas recursivas não o incluam novamente.

Análise de Complexidade

CasoTempoEspaço (stack)Observação
MelhorO(n log n)O(log n)Pivô divide o array exatamente ao meio
MédioO(n log n)O(log n)Esperado com pivô aleatório
PiorO(n²)O(n)Pivô sempre mínimo/máximo (array ordenado + pivô extremo)

Por que O(n log n) em média? Se o pivô sempre divide o array em proporção (p, 1-p) onde 0 < p < 1 (qualquer proporção constante), a recursão tem O(log n) níveis. Com pivô aleatório, a probabilidade de sempre escolher o pior pivô é astronomicamente pequena a análise probabilística demonstra tempo esperado O(n log n).

Estabilidade: Instável. As trocas durante o particionamento podem inverter a ordem relativa de elementos iguais.

Quando Usar

Dados em memória principal

O Quick Sort tem excelente desempenho de cache (acessa memória sequencialmente durante o particionamento) e baixo overhead na prática costuma ser 2-3x mais rápido que Merge Sort para arrays em RAM.

Primitivos sem estabilidade necessária

A biblioteca padrão do Java usa Dual-Pivot Quick Sort para primitivos (int, long, etc.) exatamente por ser mais rápido que Merge Sort quando a estabilidade não é necessária.

Pior caso inaceitável

Se o pior caso O(n²) é inaceitável (sistemas críticos, inputs potencialmente adversariais), use Heap Sort ou Merge Sort. IntroSort (Quick + Heap fallback) combina o melhor dos dois.

Atenção com dados ordenados: Quick Sort com pivô = primeiro ou último elemento tem pior caso exatamente em arrays já ordenados ou quase-ordenados justamente o cenário mais comum em dados reais. Sempre use pivô aleatório ou mediana-de-três em produção.
