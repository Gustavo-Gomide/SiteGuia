Sleep Sort: o tempo como critério de ordem

Cada elemento "dorme" por uma duração proporcional ao seu valor. Quem acorda primeiro é o menor — a ordem emerge do tempo, não de comparações.

O Sleep Sort é uma curiosidade algorítmica, não um algoritmo prático. Para cada elemento n, uma thread dorme por n milissegundos e então emite o valor. Como a thread do elemento menor acorda primeiro, o output aparece em ordem crescente. Complexidade real: O(max_value) de tempo de parede — pior que qualquer comparativo clássico para grandes valores.

O que é o Sleep Sort?

Surgiu em 2011 em um post anônimo no imageboard 4chan (/prog/) como piada. Viralizou porque a ideia é genuinamente estranha: usar o escalonador do sistema operacional como "comparador".

O algoritmo não é correto garantido: se dois elementos são próximos, a granularidade do timer do OS pode fazer o maior chegar antes. Também não escala — para ordenar um array com o valor 1.000.000, você esperaria ~17 minutos.

NB: Sleep Sort viola todos os critérios formais de um bom algoritmo de ordenação — não é confiável, não é eficiente, não é determinístico. É estudado como curiosidade e para ilustrar que criatividade algorítmica tem limites práticos.

Como Funciona: Linha do Tempo

Para [4, 1, 3, 2]: cada thread dorme t ms e acorda emitindo o valor.

Implementação em Python

sleep_sort.py
import threading
import time


def sleep_sort(arr: list) -> list:
    resultado = []
    lock = threading.Lock()

    def add_after_sleep(n):
        time.sleep(n * 0.01)   # 10ms por unidade (ajustável)
        with lock:
            resultado.append(n)

    threads = []
    for elem in arr:
        t = threading.Thread(target=add_after_sleep, args=(elem,))
        threads.append(t)

    # Inicia todas as threads antes de qualquer uma terminar
    for t in threads:
        t.start()

    # Aguarda todas terminarem
    for t in threads:
        t.join()

    return resultado


if __name__ == "__main__":
    dados = [4, 1, 3, 2]
    # AVISO: resultado pode variar com granularidade do OS e carga do sistema
    print(sleep_sort(dados))  # usualmente [1, 2, 3, 4]

Pontos importantes do código:

  • O threading.Lock() garante que o resultado.append() não cause condições de corrida (race conditions) entre threads simultaneâneas.
  • O fator 0.01 (10ms/unidade) é ajustável — valores menores são mais rápidos mas mais suscetíveis a erros de granularidade do OS.
  • Todas as threads são iniciadas antes de qualquer join() — se você chamasse start() e join() no mesmo loop, executaria sequencialmente, tornando o algoritmo ainda mais lento.
  • No Windows, a resolução mínima do timer é ~15ms — elementos com diferença menor que isso podem sair fora de ordem.

Curiosidades

Origem em 2011

Postado anonimamente no 4chan /prog/ em 2011. O autor original apresentou como uma brincadeira ("This idea is just too funny not to share"). Virou meme de programação e gerou dezenas de implementações em linguagens diferentes.

Por que não funciona de verdade

Timers do OS têm resolução mínima (~1ms no Linux, ~15ms no Windows). Dois valores próximos (ex: 100 e 101) podem acordar na mesma janela de escalonamento. Com valores grandes, o tempo é absurdamente longo.

Não é um algoritmo real: O Sleep Sort é não-determinístico, depende do escalonador do SO, falha para valores negativos, e escala de forma catastrofa para valores grandes. Não use em nenhum contexto de produção. Seu único valor é humor e reflexão sobre o que constitui um algoritmo válido.
