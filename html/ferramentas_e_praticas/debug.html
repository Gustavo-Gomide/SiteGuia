Debug e Depuração

Debug — A Arte de Encontrar e Corrigir Defeitos

Depurar é mais do que encontrar erros: é raciocinar sistematicamente sobre o comportamento do software

Depuração é o processo de identificar, isolar e corrigir defeitos em um programa. Todo desenvolvedor — independentemente da experiência — passa a maior parte do tempo interpretando comportamentos inesperados. Saber depurar bem é tão importante quanto saber programar bem.

O termo "bug" foi popularizado por Grace Hopper (1947), quando sua equipe encontrou uma mariposa (bug) presa nos relés do computador Mark II, causando uma falha. O registro ficou no diário de bordo: "First actual case of bug being found."

Tipos de Defeitos

Sommerville (2018) classifica os defeitos de software em categorias que determinam as estratégias de detecção e correção. Compreender o tipo do defeito é o primeiro passo para encontrá-lo.

Erro de Sintaxe

Violação das regras gramaticais da linguagem. Detectado pelo compilador ou interpretador antes da execução. São os mais fáceis de resolver.

// JavaScript — ponto e vírgula e fechar a chave faltando (erro de sintaxe)
function soma(a, b) {
  return a + b   // SyntaxError em modo strict

Erro de Tempo de Execução (Runtime)

O código compila, mas falha durante a execução. Exemplos: divisão por zero, acesso a ponteiro nulo, índice fora de limites, recursão infinita causando stack overflow.

# Python — NullPointerError equivalente
usuario = None
print(usuario.nome)  # AttributeError: NoneType

Erro de Lógica

O programa executa sem falhas, mas produz resultado incorreto. São os mais difíceis de encontrar pois não geram mensagens de erro — a lógica simplesmente está errada.

// Lógica errada: condição invertida
if (idade < 18) {
  permitirAcesso(); // deveria ser >= 18!
}

Erro de Semântica

O código executa sem erro mas o significado está errado. Diferente do erro de lógica: aqui a estrutura está correta, mas o uso de APIs ou valores está semanticamente incorreto.

// Confundindo == com = em C
if (x = 5) { ... }  // Atribui 5 a x — sempre verdadeiro!

Condição de Corrida (Race Condition)

Em código concorrente ou assíncrono, o resultado depende da ordem não determinística de execução de threads/coroutines. Muito difícil de reproduzir.

Exemplo: uma função pega no banco de dados de um banco o valor de R$1000,00 e adiciona R$100,00 e outra função pega o mesmo valor (R$1000,00) e remove R$200,00, na lógica deveria dar R$900,00 porém a primeira devolve R$1100,00 e a segunda R$800,00 porque não há sincronização e pegaram o valor anterior, resultando em valor final incorreto de R$800,00 causando um prejuízo.

Vazamento de Memória (Memory Leak)

Memória alocada dinamicamente nunca é liberada. O programa lentamente consome mais RAM até travar — ou ser encerrado pelo SO.

Comum em C/C++ (esqueceu de free()). Em linguagens com GC (Java, Python), pode ocorrer com referências circulares ou coleções que crescem indefinidamente.

O Método Científico Aplicado ao Debug

Zeller (2009), em Why Programs Fail, propõe que a depuração deve seguir o método científico: formular hipóteses, testá-las e refutá-las sistematicamente. McConnell (2004) chama isso de "debugging por confirmação científica".


Checklist do Bom Depurador

  1. Reproduza o bug — se não consegue reproduzir, não pode corrigir com confiança. Minimize o caso de teste ao mínimo necessário.
  2. Entenda o comportamento esperado — qual deveria ser a saída correta? Verifique especificações, testes e documentação.
  3. Formule uma hipótese específica — "Acredito que o problema está na função X porque Y". Hipóteses vagas levam a buscas infrutíferas.
  4. Teste a hipótese minimamente — adicione um único ponto de inspeção, não altere múltiplas coisas de uma vez.
  5. Documente o que descobriu — anotar o processo revela padrões e evita repetir esforço.
  6. Adicione um teste de regressão — após corrigir, escreva um teste automatizado que falha sem o fix e passa com ele.

Depuradores — Conceitos Fundamentais

Um depurador (debugger) é um programa que permite controlar a execução de outro programa em tempo real: pausar, inspecionar estado, avançar passo a passo e modificar valores na memória.

Vocabulário Universal dos Depuradores

Breakpoint

Ponto de parada — o debugger pausa a execução naquele ponto do código.

Step Over (F10)

Executa a próxima linha inteira, sem entrar em funções chamadas.

Step Into (F11)

Entra na função que está sendo chamada na linha atual.

Step Out (Shift+F11)

Sai da função atual, voltando ao nível de chamada acima.

Continue (F5)

Retoma a execução até o próximo breakpoint.

Watch Expression

Monitora o valor de uma expressão continuamente durante a execução.

Call Stack

Pilha de chamadas — mostra quais funções chamaram a atual.

Conditional Breakpoint

Pausa somente quando uma condição é verdadeira (ex: i == 99).

Logpoint

Imprime uma mensagem sem pausar a execução — alternativa ao console.log espalhado.

GDB — GNU Debugger (C/C++)

O GDB (GNU Debugger) é o depurador padrão para programas em C e C++ no Linux/macOS. Desenvolvido pelo Projeto GNU desde 1986, é uma ferramenta de linha de comando poderosa que também é usada por IDEs como base (VSCode usa o GDB ou LLDB internamente).

Compilar com informações de debug e iniciar GDB
# -g inclui informações de debug (nomes de variáveis, linhas)
# -O0 desativa otimizações (facilita o debug)
gcc -g -O0 -o programa programa.c
gdb ./programa
Comando GDB Abreviação Descrição Exemplo
run r Inicia a execução do programa r arg1 arg2
break b Define breakpoint b main.c:42
next n Step over — avança uma linha n
step s Step into — entra na função s
continue c Continua até o próximo breakpoint c
print p Imprime valor de expressão/variável p *ptr
backtrace bt Exibe a call stack atual bt
info locals Lista variáveis locais da frame atual info locals
watch Para quando variável é modificada watch contador
display Exibe valor a cada passo display i
x Examina memória no endereço x/10x 0xabcd
quit q Sai do GDB q
Sessão GDB — exemplo prático
(gdb) b main          # breakpoint na função main
(gdb) r               # inicia o programa
Breakpoint 1, main () at programa.c:10
10          int x = 5;
(gdb) n               # avança uma linha
11          int y = x * 2;
(gdb) p x             # inspeciona variável x
$1 = 5
(gdb) b 20 if x > 10  # condicional: pausa na linha 20 se x > 10
(gdb) bt              # call stack
#0  main () at programa.c:11

Python — pdb e breakpoint()

O pdb (Python Debugger) é o depurador nativo do Python. Desde o Python 3.7, a função builtin breakpoint() é a forma idiomática de inserir um ponto de parada no código.

Usando pdb no código
def processar(dados):
    resultado = []
    for item in dados:
        breakpoint()  # pausa aqui
        valor = item['preco'] * 1.1
        resultado.append(valor)
    return resultado

Também pode ser iniciado pela linha de comando:
python3 -m pdb script.py

Comando pdb Ação
n next — avança uma linha (step over)
s step — entra na função (step into)
c continue — continua a execução
p expr imprime o valor da expressão
pp obj pretty-print do objeto
l lista o código ao redor da linha atual
bt backtrace — call stack
b 42 breakpoint na linha 42
q sai do pdb
qualquer expr avalia como Python no contexto atual

JavaScript — Chrome DevTools

O Chrome DevTools (e equivalentes no Firefox, Edge e Safari) oferece um depurador completo para JavaScript no browser. É acessado com F12 ou Ctrl+Shift+I.

Console

REPL JavaScript na página atual. Permite executar expressões, inspecionar o DOM, chamar funções e visualizar erros e logs.

// Técnicas de console além do .log
console.table(arrayDeObjetos);   // tabela formatada
console.group('Seção');          // agrupa logs
console.time('operação');        // inicia timer
console.timeEnd('operação');     // imprime duração
console.assert(cond, msg);       // log se falso

Sources (Depurador)

Permite definir breakpoints, step through, inspecionar variáveis, monitorar call stack e modificar código em tempo real.

Dica: debugger; no código JavaScript equivale a um breakpoint — pausa o DevTools automaticamente naquele ponto.

Network

Monitora todas as requisições HTTP(S) feitas pela página. Essencial para depurar APIs: verifica status codes, cabeçalhos, payloads e tempos de resposta.

Performance

Perfila a execução JavaScript, rendering e layout. Identifica funções lentas (bottlenecks) e frames que excedem 16ms (60 fps).

Debug no VSCode

O VSCode possui um sistema de depuração integrado universal — funciona com Node.js, Python, C/C++, Go, Java e outras linguagens por meio de extensões. A configuração fica em .vscode/launch.json.

.vscode/launch.json
{ "version": "0.2.0", "configurations": [ { "name": "Debug Node.js", "type": "node", "request": "launch", "program": "${workspaceFolder}/src/index.js", "env": { "NODE_ENV": "development" } }, { "name": "Debug Python", "type": "python", "request": "launch", "program": "${file}", "console": "integratedTerminal" }, { "name": "Debug C++ (GDB)", "type": "cppdbg", "request": "launch", "program": "${workspaceFolder}/build/programa", "MIMode": "gdb" } ] }

Estratégias de Depuração

Bisseção (Binary Search Debug)

Divida o código ao meio e determine em qual metade o bug está. Repita até isolar o mínimo reproduzível. Hunt & Thomas (1999) chamam isso de "select isn't broken" — o bug está no seu código, não na biblioteca. Isole.

Logging Estratégico

Use logs estruturados com níveis (DEBUG, INFO, WARN, ERROR). Log de entrada e saída de funções críticas revela o estado exato no momento da falha sem modificar a execução.

Testes de Unidade como Debug Tool

Escreva um teste que reproduz o bug antes de corrigi-lo. Quando o teste passar, o bug está corrigido — e permanece como proteção contra regressões.

Rubber Duck Debugging

Explique o problema em voz alta (ou por escrito) para um "pato de borracha" — sem necessidade de pessoa real. O ato de verbalizar o raciocínio frequentemente revela o defeito. Descrito formalmente por Hunt & Thomas (1999).

Diff Debugging

Compare uma versão que funciona com uma que não funciona usando git diff. Quando o bug foi introduzido? Use git bisect para uma busca binária automática no histórico de commits.

Análise de Core Dump

Em C/C++, quando o programa crasha, o SO pode gerar um arquivo de core dump com o estado da memória. O GDB consegue analisar: gdb ./programa core e bt mostra onde travou.
