Templates de Projeto

Inicie novos projetos em segundos com a estrutura, configurações e boas práticas que sua equipe definiu — sem copiar e colar

Templates de projeto (scaffolding) resolvem um problema recorrente: cada novo projeto começa recriando a mesma estrutura de diretórios, configurações de linter, CI/CD, Dockerfile, e arquivos de configuração. Templates automatizam essa criação, garantindo consistência e velocidade desde o primeiro commit.

Ferramentas de Scaffolding

cookiecutter

Ferramenta Python que gera projetos a partir de templates Jinja2 armazenados em Git. O padrão para templates Python e cientifico.

pip install cookiecutter
cookiecutter gh:audreyfeldman/cookiecutter-django

degit

Clona apenas o conteúdo de um repositório Git (sem o histórico), ideal para templates leves e criando “clones limpos”.

npx degit sveltejs/template meu-app
cd meu-app ; npm install

create-* (npm)

O npm init nome chama o pacote create-nome do npm. Cada framework tem o seu (create-react-app, create-vite, etc.).

npm create vite@latest
npm create astro@latest

GitHub Templates

Marque um repositório como template nas configurações do GitHub. Outros podem clicar em “Use this template” para criar um novo repositório com o mesmo conteúdo.

cookiecutter — Exemplo Completo

A estrutura de um template cookiecutter usa a variável {{cookiecutter.nome_variavel}} em nomes de arquivos e no conteúdo:

meu-template-python/
├── cookiecutter.json                 # variáveis do template
└── {{cookiecutter.project_slug}}/    # pasta raiz do projeto gerado
    ├── README.md
    ├── pyproject.toml
    ├── src/
    │   └── {{cookiecutter.project_slug}}/
    │       └── __init__.py
    └── tests/
// cookiecutter.json
{
  "project_name": "Meu Projeto Python",
  "project_slug": "{{ cookiecutter.project_name.lower().replace(' ', '_') }}",
  "python_version": ["3.11", "3.12"],
  "use_docker": ["y", "n"]
}

O que incluir em um Template

Estrutura mínima recomendada

  • README.md — com instruções de uso
  • Configuração de linter (.eslintrc, pyproject.toml)
  • Configuração de formatter (.prettierrc, ruff.toml)
  • .gitignore pré-configurado para a stack
  • .editorconfig para consistência entre editores
  • Workflow de CI (GitHub Actions / GitLab CI)
  • Dockerfile e docker-compose.yml se aplicável
  • Estrutura de pastas da arquitetura escolhida

Padronização de equipe

Templates de equipe devem refletir as decisões arquiteturais da organização: estrutura de pastas, convenções de nomeação, configurações de segurança, integrações com ferramentas internas.

Trate o template como um produto — versione, documente, e aceite PRs de melhoria da equipe.
