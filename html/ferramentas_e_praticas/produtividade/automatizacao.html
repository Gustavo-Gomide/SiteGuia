Ferramentas e Práticas

Automação de Tarefas — Elimine o Trabalho Repetitivo

Automação de Tarefas

Codifique seus processos repetitivos uma vez e deixe a máquina executar — builds, testes, deploys, formatação e qualidade automatizados

Um desenvolvedor produtivo é aquele que automatiza tarefas recorrentes. Cada processo manual é um candidato à automação: executar testes antes de commitar, formatar código ao salvar, fazer deploy após merge, enviar notificações quando uma tarefa termina. Esses ganhos, individualmente pequenos, são enormes em escala de meses e anos.

Regra prática para escolher a automação

Se a tarefa é local e reutilizável, use shell script. Se o objetivo é padronizar comandos do projeto, use Taskfile. Se a ação acontece antes do commit ou push, use hooks. Se precisa rodar em horários fixos, use cron.

O que é Automação de Tarefas?

Automação de tarefas é a prática de transformar ações repetitivas em instruções executáveis por uma máquina. Em vez de repetir o mesmo passo manualmente toda vez, você descreve o processo uma única vez e escolhe um gatilho para a execução.

Pense em uma linha de produção: o processo é definido com antecedência, a execução segue sempre a mesma ordem e o resultado final sai mais previsível. No desenvolvimento, isso reduz erro humano, acelera rotinas e libera tempo para trabalho que exige decisão real.

Automação boa não é a mais complexa. É a que remove repetição sem criar um novo problema de manutenção.

Como Funciona na Prática

Toda automação útil precisa responder a quatro perguntas: onde roda, o que dispara, o que executa e qual é a saída. Quando isso está claro, a pessoa entende onde encaixar a ferramenta no fluxo.

Gatilho

É o evento que inicia a automação. Pode ser um comando manual, um evento do Git, um agendamento no sistema operacional ou a chamada de outro script.

Ambiente

É o lugar onde a automação roda: terminal local, repositório do projeto, servidor de CI ou máquina com cron. Esse detalhe muda permissões, caminhos e dependências.

Execução

É o conjunto de comandos em si. Aqui entram shell scripts, Taskfile e hooks. A execução precisa ser previsível, curta e fácil de debugar.

Saída

É o efeito final: arquivos gerados, testes executados, commits bloqueados, relatórios emitidos ou backups criados. A saída tem que ser clara para o usuário confiar no processo.

Escolhendo a Ferramenta Certa

As ferramentas desta página resolvem problemas parecidos, mas entram em momentos diferentes do fluxo. Pensar no gatilho certo evita scripts excessivamente complexos e ajuda a manter o projeto simples.

Shell script

Bom para encadear comandos, preparar ambiente e automatizar rotinas locais. É a opção mais flexível, mas também a que mais exige disciplina para continuar legível.

Taskfile

Bom para padronizar comandos de projeto. Em vez de decorar scripts, a equipe executa um catálogo único como task build, task test e task dev.

Git hooks

Bom para garantir qualidade antes de mudar o histórico do repositório. Eles atuam no momento certo para prevenir erros, não para corrigir problemas depois.

Cron

Bom para rotinas periódicas, como backup, limpeza de logs e sincronização. Se a tarefa depende do relógio, cron é a ferramenta mais direta.

Casos de Uso Mais Comuns

Este guia só fica útil de verdade quando a pessoa consegue reconhecer a situação real em que cada ferramenta entra. Estes são os casos mais frequentes em projetos técnicos.

Preparar ambiente

Instalar dependências, criar ambiente virtual, baixar ferramentas e validar pré-requisitos. Normalmente isso começa em shell script e pode ser exposto no Taskfile.

Padronizar validação

Rodar lint, formatadores e testes antes do commit. Esse é o território natural de hooks, porque o problema é barrado antes de sair da máquina do autor.

Executar comandos frequentes

Build, teste, limpeza de artefatos e deploy local. Aqui o Taskfile ajuda a esconder a complexidade e deixa o comando oficial do projeto evidente.

Tarefas por agenda

Backup, relatórios, sincronização e limpeza periódica. Se a pergunta é “quando isso roda?”, cron resolve de forma direta.

Shell Scripts — A Base da Automação

Shell script é o ponto de partida porque ele conversa com o sistema operacional e com quase qualquer ferramenta de linha de comando. Use quando você precisa juntar etapas que já existem, como checar pré-requisitos, preparar diretórios, instalar dependências e chamar outros programas.

setup.sh — script de configuração de ambiente
#!/usr/bin/env bash set -euo pipefail # sair ao primeiro erro; var não definida = erro IFS=$'\n\t' LOG() { echo "[$(date '+%H:%M:%S')] $*"; } LOG "Configurando ambiente de desenvolvimento..." # Verificar pré-requisitos command -v python3 &>/dev/null || { echo "Python 3 não encontrado"; exit 1; } command -v node &>/dev/null || { echo "Node.js não encontrado"; exit 1; } # Criar ambiente virtual Python if [ ! -d ".venv" ]; then LOG "Criando venv..." python3 -m venv .venv fi source .venv/bin/activate # Instalar dependências LOG "Instalando dependências Python..." pip install -q -r requirements.txt LOG "Instalando dependências Node.js..." npm ci --silent LOG "Configurando git hooks..." npx husky install LOG "Pronto. Ambiente configurado com sucesso."

Quando o shell funciona melhor

Ele brilha em tarefas como provisionamento inicial, scripts de deploy, limpeza de arquivos temporários e automações pequenas que precisam ser executadas sem interface gráfica.

Onde colocar e como rodar

Onde colocar: normalmente em pastas como scripts/, bin/ ou tools/ dentro do projeto. Em tarefas de projeto, o ideal é versionar o arquivo junto com o código.

Como rodar: com bash scripts/setup.sh, sh scripts/setup.sh ou, se tiver permissão, ./scripts/setup.sh. Em CI, o pipeline chama o script diretamente.

Taskfile — Make Moderno

Taskfile (ou task) é uma alternativa moderna ao Make: sintaxe YAML legível, multi-plataforma (inclui Windows) e sem as peculiaridades do Makefile. A diferença prática é simples: ele organiza comandos de projeto em tarefas nomeadas e previsíveis.

Taskfile.yml
version: '3' vars: APP: meu-app tasks: default: desc: "Mostra as tarefas disponíveis" cmds: [task --list] build: desc: "Compila o projeto" cmds: - go build -o bin/{{.APP}} ./cmd/{{.APP}} test: desc: "Executa todos os testes" cmds: - go test ./... -v -race -coverprofile=coverage.out lint: desc: "Lint + format" cmds: - golangci-lint run - gofmt -w . docker:build: desc: "Build da imagem Docker" cmds: - docker build -t {{.APP}}:latest . dev: desc: "Inicia servidor de desenvolvimento" deps: [build] cmds: - air # live reload

Execute com task build, task test, etc. Lista de tarefas com task --list. Na prática, isso evita que cada pessoa invente seu próprio comando de build.

Quando o Taskfile é melhor que shell puro

Use quando o projeto tem vários comandos oficiais e você quer um ponto único de entrada. Isso melhora onboarding, reduz divergência entre equipes e deixa o fluxo visível dentro do repositório.

Onde colocar e como rodar

Onde colocar: o arquivo deve ficar na raiz do projeto como Taskfile.yml, para que toda a equipe encontre o mesmo catálogo de tarefas.

Como rodar: instale o binário task e execute comandos como task, task build, task dev e task --list.

Git Hooks — Automação no Fluxo de VCS

Hooks são automações orientadas por evento. Eles existem para reagir a ações do Git, como commit, push e merge, e são uma boa barreira de qualidade porque atuam antes do problema sair do ambiente local.

Hooks mais úteis

HookQuando executaUso comum
pre-commitAntes do commitLinting, formatação, testes unitários
commit-msgApós digitar a mensagemValidar formato do commit (Conventional Commits)
pre-pushAntes de fazer pushTestes de integração, segurança
post-mergeApós merge/pullReinstalar dependências

pre-commit — a ferramenta

pre-commit é um gerenciador de hooks Git. Define os hooks em .pre-commit-config.yaml e eles são compartilhados com a equipe. O valor real não é só automatizar, mas garantir que a mesma regra vale para todo mundo:

repos:
  - repo: https://github.com/pre-commit/pre-commit-hooks
    rev: v4.5.0
    hooks:
      - id: trailing-whitespace
      - id: end-of-file-fixer
      - id: check-yaml
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.1.6
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format

O que o pre-commit evita

Ele impede que pequenos erros escapem para o repositório: espaços sobrando, YAML inválido, arquivos sem quebra final e código fora do padrão. Em equipes grandes, isso economiza revisão manual e reduz ruído no histórico.

Onde colocar e como rodar

Onde colocar: o arquivo .pre-commit-config.yaml fica na raiz do repositório. Os hooks compartilhados ficam declarados ali; os hooks locais de Git vivem em .git/hooks/, mas não devem ser o único caminho em equipe.

Como rodar: instale com pre-commit install e ele passa a executar automaticamente em git commit. Também dá para testar manualmente com pre-commit run --all-files.

Cron — Tarefas Agendadas

Cron existe para tarefa recorrente baseada em tempo. Ele não substitui hooks nem Taskfile; ele resolve o problema de “rodar sempre às X horas” de forma simples e previsível.

crontab — sintaxe e exemplos
# Formato: minuto hora dia-do-mes mes dia-da-semana comando # min (0-59) | hora (0-23) | dom (1-31) | mês (1-12) | dow (0-7, dom=0 ou 7) # O cron é ideal quando a agenda é fixa e independente da ação do usuário. # Backup diário às 2h da manhã 0 2 * * * /usr/local/bin/backup.sh # Limpar logs toda segunda-feira às 3h30 30 3 * * 1 find /var/log/app -name "*.log" -mtime +30 -delete # Sincronizar dados a cada 15 minutos */15 * * * * /scripts/sync.sh >> /var/log/sync.log 2>&1 # Abrir o editor de crontab crontab -e # Ver crontabs atuais crontab -l

Onde colocar e como rodar

Onde colocar: no crontab do usuário com crontab -e ou, em servidores, em /etc/crontab e /etc/cron.d/ quando a administração do sistema exigir esse formato.

Como roda: o agendador do sistema chama o comando automaticamente no horário definido. Você valida com crontab -l e, se o script tiver saída, grava logs em arquivo para auditoria.

Quando cron faz sentido

Backup, limpeza de arquivos, exportação de relatórios e sincronização periódica são casos clássicos. Se você está tentando usar cron para reagir a eventos do aplicativo, provavelmente está escolhendo a ferramenta errada. Cron roda fora da aplicação, no sistema operacional, e por isso é ótimo para rotinas independentes do fluxo de código.

Tópicos Avançados

Depois que a automação funciona, o próximo desafio é torná-la confiável. O objetivo agora não é apenas executar, mas executar com clareza, repetibilidade e falha previsível.

Idempotência

Uma automação idempotente pode ser executada mais de uma vez sem causar estrago. Isso importa em scripts de setup e deploy, porque falhas intermediárias são comuns.

Falha explícita

Quando algo quebra, o script precisa parar e avisar com clareza. Silenciar erro para “seguir o fluxo” quase sempre transforma um problema pequeno em um problema grande.

Escopo correto

Shell prepara ambiente, Taskfile organiza comandos, hooks validam mudanças e cron agenda rotinas. Quando uma ferramenta tenta fazer tudo, a manutenção piora rápido.

Observabilidade

Logs claros, saída padronizada e mensagens úteis ajudam a diagnosticar problemas sem abrir o código inteiro. Automação sem rastreabilidade vira caixa-preta.

Se a automação altera estado crítico, documente o que ela faz e teste em ambiente controlado antes de colocá-la em produção.
