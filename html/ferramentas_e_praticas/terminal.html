Terminal e Shell

Terminal e Interface de Linha de Comando

A ferramenta mais poderosa e universal do desenvolvedor — dominar o shell é dominar o computador

O terminal é a interface que permite ao desenvolvedor interagir diretamente com o sistema operacional por meio de texto. Longe de ser uma tecnologia obsoleta, o shell é a ferramenta que une todas as outras: automatiza, orquestra, depura e entrega.

Conceitos Fundamentais

Antes de executar qualquer comando, é essencial diferenciar três conceitos frequentemente confundidos: terminal, shell e console.

Terminal (Emulador)

Programa gráfico que emula um terminal físico antigo. Fornece uma janela onde você interage com o shell. Exemplos: GNOME Terminal, Alacritty, Windows Terminal, iTerm2.

O terminal cuida apenas da exibição — renderiza texto, cores e fontes. Não interpreta comandos.

Shell

Programa que interpreta comandos e os executa. É a interface entre o usuário e o kernel do sistema operacional. Exemplos: Bash, Zsh, Fish, PowerShell, sh.

O shell lê o que você digita, interpreta a gramática do comando e chama as chamadas de sistema (syscalls) corretas.

Console

Originalmente, o dispositivo físico (teclado + monitor) conectado diretamente ao computador. Modernamente, o termo é usado intercambiavelmente com "terminal". Em sistemas Unix, /dev/console é o dispositivo de saída do kernel.

No contexto de browsers, o "Console" do DevTools é um REPL JavaScript — um ambiente shell especializado para a web.

Principais Shells

Diferentes shells oferecem diferentes capacidades. A escolha do shell impacta diretamente a produtividade e os scripts que você escreve.

Anatomia de um Comando

Todo comando shell segue uma estrutura gramatical bem definida. Compreender essa estrutura é o primeiro passo para usar o terminal com eficiência.

Estrutura de um Comando Shell

comando  [opções]  [argumentos]

Onde:

comando

Nome do programa ou função builtin a executar. Ex: ls, git, python3

opções (flags)

Modificam o comportamento. Curtas (-l) ou longas (--all). Podem ser combinadas: -la

argumentos

Os dados sobre os quais o comando opera. Ex: arquivos, caminhos, strings.

Exemplo Unix/Linux/macOS

bash
# Listar arquivos com detalhes e ocultos
ls -la /home/usuario

# cp: copiar  -r: recursivo  src/ dest/
cp -r src/ backup/

# Buscar padrão "-i" (case insensitive) em arquivo
grep -i "erro" app.log

Exemplo PowerShell (Windows)

powershell
# Get-ChildItem = ls; -Force mostra ocultos
Get-ChildItem -Force C:\Users\user

# Copy-Item recursivo
Copy-Item -Recurse src\ backup\

# Select-String = grep
Select-String -Pattern "erro" app.log

Pipes e Redirecionamento

O conceito de pipe é o mais poderoso da filosofia Unix. Descrito por Doug McIlroy (Bell Labs, 1972), permite compor programas simples em pipelines complexos: a saída de um programa é conectada diretamente à entrada do próximo.

Operadores de Redirecionamento

| — Pipe

Conecta stdout de um comando ao stdin do próximo.

> — Redireciona stdout

Grava saída em arquivo (sobrescreve).

>> — Acrescenta stdout

Acrescenta ao final do arquivo (não sobrescreve).

< — Redireciona stdin

Lê entrada de um arquivo.

2> — Redireciona stderr

Grava saída de erro em arquivo.

2>&1 — Merge stderr → stdout

Une stderr ao fluxo de stdout.

Exemplos práticos de pipes e redirecionamento
# Contar quantos processos do node estão rodando
ps aux | grep node | wc -l

# Top 5 arquivos mais grandes em megabytes
du -sh * | sort -rh | head -5

# Salvar lista de erros do log em arquivo separado
grep "ERROR" app.log > erros.txt

# Acrescentar ao arquivo de log existente
echo "Deploy concluído em $(date)" >> deploy.log

# Suprimir erros redirecionando stderr para /dev/null
find / -name "config.json" 2>/dev/null

# Salvar stdout e stderr no mesmo arquivo
npm run build > build.log 2>&1

# Pipeline complexo: IPs únicos e sua frequência no log
awk '{print $1}' access.log | sort | uniq -c | sort -rn | head -10

Variáveis de Ambiente

Variáveis de ambiente são pares chave-valor que configuram o comportamento de processos no sistema. Elas são passadas do processo pai para os filhos e são fundamentais para configurar aplicações sem modificar código-fonte.

Variáveis Importantes

$HOMEDiretório home do usuário atual
$USERNome do usuário logado
$SHELLShell padrão do usuário
$PWDDiretório atual de trabalho
$EDITOREditor de texto padrão
$NODE_ENVAmbiente Node.js: development / production
$?Código de saída do último comando (0 = sucesso)
$$PID do shell atual
Manipulando variáveis
# Exportar variável para subprocessos
export DATABASE_URL="postgres://localhost/db"

# Ler uma variável
echo $DATABASE_URL

# Adicionar diretório ao PATH
export PATH="$HOME/.local/bin:$PATH"

# Variável temporária (só para este comando)
NODE_ENV=production node app.js

# Verificar código de saída
git status
echo $?  # 0 = ok, 128 = não é repositório git

# Carregar variáveis de um arquivo .env
set -a; source .env; set +a

Boas práticas: nunca coloque segredos (API keys, senhas) diretamente no código. Use variáveis de ambiente e arquivos .env (não versionados no Git).

Shell Scripting Básico

Um shell script é um arquivo de texto contendo sequências de comandos shell. É a forma mais direta de automatizar tarefas repetitivas no sistema operacional. Scripts Bash são executáveis diretamente e amplamente suportados em qualquer sistema Linux/macOS.

deploy.sh 
#!/usr/bin/env bash
# Script de deploy simples — versão comentada para estudo
# Uso: ./deploy.sh [ambiente]

# --- Configuração ---
set -euo pipefail          # Aborta em erros, vars não definidas, pipes falhos
AMBIENTE=${1:-"staging"}  # Usa o 1º argumento ou "staging" como padrão
DATA=$(date +%Y%m%d_%H%M%S)

# --- Função de log ---
log() { echo "[$(date +'%H:%M:%S')] $*"; }

# --- Validação do ambiente ---
if [[ "$AMBIENTE" != "staging" && "$AMBIENTE" != "production" ]]; then
  echo "Ambiente inválido: $AMBIENTE" >&2
  exit 1
fi

log "Iniciando deploy para $AMBIENTE"

# --- Build ---
log "Instalando dependências..."
npm ci --silent

log "Executando testes..."
npm test

log "Gerando build de produção..."
npm run build

# --- Backup e deploy ---
BACKUP_DIR="backups/${AMBIENTE}_${DATA}"
mkdir -p "$BACKUP_DIR"
cp -r dist/ "$BACKUP_DIR/"

log "Deploy concluído! Backup em: $BACKUP_DIR"

Estruturas de Controle em Bash

controle.sh
# Condicional if/elif/else
if [[ $VAR -gt 10 ]]; then
  echo "maior que 10"
elif [[ $VAR -eq 10 ]]; then
  echo "igual a 10"
else
  echo "menor que 10"
fi

# Loop for clássico
for i in {1..5}; do
  echo "Iteração $i"
done

# Loop sobre arquivos
for arquivo in *.log; do
  echo "Processando: $arquivo"
done

# while com leitura de linha (padrão para processar arquivos)
while IFS= read -r linha; do
  echo "Linha: $linha"
done < arquivo.txt

# Operador ternário com &&/||
[[ -f config.json ]] && echo "existe" || echo "não existe"

Permissões de Arquivos

O sistema de permissões Unix é baseado em três princípios: quem (dono, grupo, outros), o quê (ler, escrever, executar). Compreender permissões é essencial para segurança e para tornar scripts executáveis.

Lendo a saída de ls -l

-rwxr-xr--  1  usuario  grupo  4096  Jan 10 14:22  script.sh
│├──┤├──┤├──┤
│  │    │    └── outros: leitura
│  │    └─────── grupo: leitura + execução
│  └──────────── dono: leitura + escrita + execução
└─────────────── tipo: - (arquivo), d (diretório), l (link)
r — read (4)

Permissão de leitura

w — write (2)

Permissão de escrita

x — execute (1)

Permissão de execução

- — ausente (0)

Permissão não concedida

chmod — alterando permissões
# Tornar script executável
chmod +x script.sh

# Permissão completa ao dono, leitura ao grupo/outros
chmod 744 script.sh     # rwxr--r--

# Permissão segura para arquivo de configuração (segredo)
chmod 600 .env          # rw------- (só o dono lê/escreve)

# Mudar dono do arquivo
chown usuario:grupo arquivo.txt

Ferramentas Modernas de Terminal

O ecossistema de ferramentas CLI evoluiu muito na última década. Alternativas modernas a comandos clássicos oferecem melhor ergonomia, cores e performance:

exa / eza — alternativa ao ls

Cores, ícones, exibição de Git status, suporte a árvore integrado. Escrito em Rust.

eza --tree --git-ignore --icons -L 2

bat — alternativa ao cat

Syntax highlighting, numeração de linhas, integração com paginador e Git. Escrito em Rust.

bat src/main.py

ripgrep (rg) — alternativa ao grep

Busca recursiva extremamente rápida, respeita .gitignore, suporte a Unicode. Amplamente usado pelo VSCode internamente.

rg -l "useState" src/

fd — alternativa ao find

Sintaxe simples, busca rápida, respeita .gitignore, colorido por padrão.

fd "\.test\.ts$" src/

fzf — fuzzy finder

Filtragem interativa em tempo real de qualquer lista. Integra com histórico (Ctrl+R), cd, git checkout.

git branch | fzf

tmux — multiplexador

Divide o terminal em painéis e janelas, mantém sessões ativas após desconexão (essencial em servidores remotos via SSH).

tmux new -s dev

Boas Práticas

Use set -euo pipefail em scripts

Garante que o script aborte em qualquer erro (-e), falhe em variáveis não definidas (-u) e detecte erros em pipes (-o pipefail). Padrão da comunidade para scripts robustos.

Sempre use aspas em variáveis

Variáveis sem aspas sofrem word splitting: rm $FILE pode apagar arquivos errados se $FILE contiver espaços. Use rm "$FILE".

Prefira [[ a [ no Bash

O operador [[ (duplo colchete) é mais seguro: suporta &&, ||, regex com =~ e não sofre de problemas com expansão de globbing.

Documente com comentários e usage

Scripts são mantidos por outras pessoas (incluindo você no futuro). Adicione um bloco de uso no início: # Uso: ./script.sh [opções] e comente seções não óbvias.

Verifique scripts com shellcheck

ShellCheck é o linter padrão para shell scripts. Detecta erros comuns, práticas inseguras e sugere correções. Disponível online em shellcheck.net.

Use man e --help

Todo comando Unix tem documentação embutida. man grep abre o manual completo. grep --help mostra um resumo rápido. Use antes de procurar na internet.
