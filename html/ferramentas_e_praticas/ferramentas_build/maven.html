Apache Maven — Convenção sobre Configuração

Apache Maven

O sistema de build Java baseado em "Convention over Configuration" — estrutura padronizada, ciclo de vida universal e o maior repositório de bibliotecas do mundo

Maven adota o princípio de Convenção sobre Configuração (Convention over Configuration): se você seguir a estrutura padrão do projeto, precisa de poquíssima configuração. O pom.xml descreve o projeto de forma declarativa — o Maven sabe exatamente como compilar, testar e empacotar.

História e Contexto

O Apache Maven foi criado por Jason van Zyl em 2002 como parte do projeto Turbine da Apache Software Foundation. A versão 2.0 (2005) estabeleceu o pom.xml e o Maven Central Repository — o maior repositório de artefatos Java do mundo, com mais de 500.000 artefatos.

O Maven Central (central.sonatype.com) hospeda virtualmente todas as bibliotecas Java/JVM open-source. Qualquer dependência declarada no pom.xml é baixada automaticamente de lá.

Estrutura de Diretórios Padrão

# Estrutura padrão Maven (Convention over Configuration)
projeto/
├── pom.xml                     # Project Object Model — configuração central
└── src/
    ├── main/
    │   ├── java/               # código-fonte Java
    │   │   └── com/empresa/app/
    │   └── resources/          # arquivos de configuração, templates
    └── test/
        ├── java/               # código de testes
        └── resources/          # fixtures de teste

Seguindo essa estrutura, o Maven sabe onde encontrar tudo — sem nenhuma configuração adicional.

pom.xml — Project Object Model

pom.xml — exemplo completo anotado 
<project xmlns="http://maven.apache.org/POM/4.0.0"
         xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
         xsi:schemaLocation="http://maven.apache.org/POM/4.0.0
                             http://maven.apache.org/xsd/maven-4.0.0.xsd">

  <modelVersion>4.0.0</modelVersion>

  <!-- Identificação única (coordenadas Maven) -->
  <groupId>com.empresa</groupId>
  <artifactId>meu-app</artifactId>
  <version>1.0.0-SNAPSHOT</version>
  <packaging>jar</packaging>

  <properties>
    <maven.compiler.source>17</maven.compiler.source>
    <maven.compiler.target>17</maven.compiler.target>
    <project.build.sourceEncoding>UTF-8</project.build.sourceEncoding>
  </properties>

  <dependencies>
    <!-- scope padrão: compile -->
    <dependency>
      <groupId>com.google.guava</groupId>
      <artifactId>guava</artifactId>
      <version>32.1.3-jre</version>
    </dependency>

    <!-- scope test: somente em testes -->
    <dependency>
      <groupId>org.junit.jupiter</groupId>
      <artifactId>junit-jupiter</artifactId>
      <version>5.10.0</version>
      <scope>test</scope>
    </dependency>
  </dependencies>

  <build>
    <plugins>
      <plugin>
        <groupId>org.apache.maven.plugins</groupId>
        <artifactId>maven-surefire-plugin</artifactId>
        <version>3.2.2</version>
      </plugin>
    </plugins>
  </build>
</project>

Ciclo de Vida do Build

Maven define três ciclos de vida built-in: default, clean e site. O ciclo default é o principal e possui as seguintes fases em ordem:

FaseDescriçãoPlugin padrão
validateValida o projeto e sua estrutura—
compileCompila o código-fontemaven-compiler-plugin
testExecuta testes unitáriosmaven-surefire-plugin
packageEmpacota (JAR, WAR, etc.)maven-jar-plugin
verifyVerifica critérios de qualidademaven-failsafe-plugin
installInstala no repositório local (~/.m2)maven-install-plugin
deployPublica no repositório remotomaven-deploy-plugin
Comandos Maven essenciais
# Compilar o projeto
mvn compile

# Executar testes
mvn test

# Empacotar em JAR
mvn package

# Limpar + compilar + testar + empacotar
mvn clean package

# Instalar no repositório local (para outros projetos usarem)
mvn install

# Pular testes (não recomendado em produção)
mvn package -DskipTests

# Ver árvore de dependências
mvn dependency:tree

# Verificar atualizações de dependências
mvn versions:display-dependency-updates
