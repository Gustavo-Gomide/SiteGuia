Gradle — Build Tool para a JVM

Gradle

Build tool flexível, incremental e altamente configurável para Java, Kotlin, Scala, Android e outras linguagens JVM

Gradle é um sistema de build baseado em tarefas (tasks) que combina a flexibilidade de um script com a estrutura de um modelo declarativo. Usa Groovy ou Kotlin como DSL — eliminando limitações de XML dos sistemas anteriores (Maven/Ant). É o sistema de build oficial do Android.

Contexto e Comparação

Por que Gradle?

  • Builds incrementais — recompila somente o que mudou (up-to-date checks)
  • Build cache — reutiliza resultados de builds anteriores, inclusive em CI
  • Configuração em código — Kotlin/Groovy DSL é mais expressivo e tipado que XML
  • Paralelismo — executa tasks independentes em paralelo automaticamente
  • Padrão Android — todo projeto Android Studio usa Gradle

Estrutura de um Projeto Gradle

# Estrutura típica de um projeto Gradle multi-módulo
projeto/
├── settings.gradle.kts      # configurações do projeto (nome, subprojetos)
├── build.gradle.kts         # build do projeto raiz
├── gradle/
│   └── wrapper/
│       ├── gradle-wrapper.jar
│       └── gradle-wrapper.properties  # versão do Gradle
├── gradlew                  # wrapper Unix (use sempre esse)
├── gradlew.bat              # wrapper Windows
├── app/
│   ├── build.gradle.kts     # build do módulo app
│   └── src/main/java/...
└── lib/
    └── build.gradle.kts

O Gradle Wrapper (gradlew) é o modo correto de usar Gradle em projetos — garante que todos usem a mesma versão sem instalar o Gradle globalmente.

build.gradle.kts — Kotlin DSL

build.gradle.kts — aplicação Java/Kotlin 
plugins {
    kotlin("jvm") version "1.9.22"
    application
}

group = "com.exemplo"
version = "1.0.0"

repositories {
    mavenCentral()
}

dependencies {
    // Dependência de implementação (runtime + compilação)
    implementation("org.jetbrains.kotlinx:kotlinx-coroutines-core:1.7.3")

    // Somente testes
    testImplementation(kotlin("test"))
    testImplementation("org.junit.jupiter:junit-jupiter:5.10.0")
}

application {
    mainClass = "com.exemplo.MainKt"
}

tasks.test {
    useJUnitPlatform()  // Usa JUnit 5
}

// Task personalizada
tasks.register("gerarDocs") {
    group = "documentation"
    description = "Gera a documentação do projeto"
    doLast {
        println("Gerando documentação...")
    }
}

Tasks — O Coração do Gradle

Tudo em Gradle é uma task. O build graph é um DAG de tasks com dependências declaradas. O Gradle executa na ordem correta, em paralelo quando possível.

Comandos Gradle essenciais
# Listar tasks disponíveis
./gradlew tasks

# Compilar
./gradlew compileKotlin

# Executar testes
./gradlew test

# Build completo (compila + testa + empacota)
./gradlew build

# Limpar artefatos
./gradlew clean

# Executar a aplicação
./gradlew run

# Gerar JAR executável
./gradlew jar

# Diagnóstico de build (profiling)
./gradlew build --profile

# Build offline (sem baixar dependências)
./gradlew build --offline

Gerenciamento de Dependências

Gradle usa configurações de dependência para determinar quando e como cada dependência é usada:

apiCompilação + runtimeSim (exposta ao consumidor)Tipos na API pública
compileOnlyApenas compilaçãoNãoAnotações processadas em build-time
runtimeOnlyApenas runtimeNãoDrivers JDBC, implementações
